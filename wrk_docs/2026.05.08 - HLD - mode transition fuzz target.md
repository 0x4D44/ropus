# Mode Transition Fuzz Target HLD

Date: 2026.05.08

Scope: Stream D of the 2026-05-08 fuzz expansion campaign. New differential
encoder fuzz target focused on SILK / HYBRID / CELT mode transitions across
multi-frame packets. Companion to existing
`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs`.

## Problem Statement

The 2026-05-07 overnight 8-hour campaign closed two real C-vs-Rust parity
bugs in the encoder, both surfaced by `fuzz_encode_multiframe`:

1. PLC short-frame `delay_buffer` contamination (commit `9b8c552`).
2. SILK LBRR `ec_prev_*` state mutation gap (commit `55e224c`).

Both bugs share one shape: the C reference mutates encoder state in place,
the Rust port takes `&State` and compensates at call sites, and one call
site missed the compensation. The bug only manifested when the missed
compensation site fired in a *different* frame from a write that lined up
with the C version's in-function mutation. In other words, both bugs
required cross-frame state carry to manifest. That defect class is
under-tested elsewhere.

`fuzz_encode_multiframe` already varies bitrate per-frame uniformly across
`[6000, 510000]` bps. That covers the surface, but it does not bias the
search toward the mode-decision boundaries where the encoder actually
flips between SILK_ONLY, HYBRID, and CELT_ONLY across consecutive frames.
The mode-decision logic in
`/home/md/language/ropus/ropus/src/opus/encoder.rs:1862-2118` is the
single largest concentration of cross-frame state-carry in the encoder:
`prev_mode` hysteresis, redundancy decisions, SILK re-init on
CELT_ONLY → SILK transition, stereo→mono transitions, and bandwidth
switching all key off the previous frame's chosen mode.

The hypothesis: there is at least one more `&State` parity gap on a mode
transition path that the uniformly distributed multiframe fuzzer is
unlikely to hit in reasonable wall-time, because it requires the
encoder to *change* mode between frame N and frame N+1, and the random
bitrate draw lands on either side of the boundary only by accident.

This HLD covers test-only fuzz infrastructure. It does not change codec
behavior.

## Stakeholder Value

The maintainer (Arthur) needs targeted fuzz coverage on the
mode-transition surface so the campaign can either (a) surface the next
multiframe-class parity bug or (b) provide an empirically clean signal
that the mode-transition logic survives byte-exact comparison under
adversarial intra-stream config drift.

Value:

- Surfaces the same defect class that produced both 2026-05-07 wins,
  but in the part of the codebase where that class is most concentrated
  (mode-decision + redundancy + SILK re-init + bandwidth coercion).
- Reuses the byte-exact differential oracle, so any finding is by
  construction a parity bug, not a quality regression.
- Does not retest saturated targets (8 of 10 from the previous campaign
  were saturated; `fuzz_encode_multiframe` was not, but its grammar is
  uniform).
- One implementation slice: ~1 new fuzz target file + small
  `c_reference.rs` extension. No codec source changes.

## Mode Transition Boundaries

Quantitative thresholds, cited from
`/home/md/language/ropus/ropus/src/opus/encoder.rs` and
`/home/md/language/ropus/reference/src/opus_encoder.c` (lines match across
both — the Rust mirror is faithful):

### Mode-decision threshold (SILK/HYBRID ↔ CELT_ONLY)

`encoder.rs:114-117` defines
`MODE_THRESHOLDS[mono|stereo][voice|music]`:

```
mono   voice = 64000, music = 10000
stereo voice = 44000, music = 10000
```

The actual threshold (`encoder.rs:1869-1876`) is interpolated:

```
mode_voice = (1-stereo_width)·64000 + stereo_width·44000   (mono → stereo)
mode_music = 10000   (both)
threshold  = mode_music + voice_est² · (mode_voice - mode_music) / 16384
```

with `voice_est ∈ [0, 128]`. Then:

- **VOIP application**: `threshold += 8000` (`encoder.rs:1878-1880`).
- **Hysteresis from prev_mode**: `+= 4000` if `prev_mode != CELT_ONLY`,
  `-= 4000` if `prev_mode == CELT_ONLY` (`encoder.rs:1881-1886`).

The decision is `equiv_rate >= threshold ⇒ CELT_ONLY` else `SILK_ONLY`
(`encoder.rs:1888-1892`). `equiv_rate` is roughly
`bitrate · (90+complexity)/100`, with a 1/12 CBR penalty if VBR=0
(`encoder.rs:711-740`). So at complexity=5, VBR=1, equiv ≈ 0.95·bitrate.

**Concrete kbps numbers for the SILK/HYBRID ↔ CELT_ONLY boundary**, all
at frame_rate=50 (20 ms), complexity=5, VBR=1, prev_mode != CELT_ONLY
(so threshold +4000 hysteresis), no VOIP boost:

| Channels | Signal     | voice_est | threshold (bps) | bitrate kbps at boundary |
|----------|------------|-----------|-----------------|--------------------------|
| mono     | pure music | 0         | 14000           | ~14.7                    |
| mono     | mid voice  | 64        | 27500           | ~28.9                    |
| mono     | pure voice | 128       | 68000           | ~71.6                    |
| stereo   | pure music | 0         | 14000           | ~14.7                    |
| stereo   | mid voice  | 64        | 22500           | ~23.7                    |
| stereo   | pure voice | 128       | 48000           | ~50.5                    |

Add ~8.4 kbps to bitrate for VOIP. Subtract 8000 from threshold (≈
8.4 kbps off bitrate) when prev_mode is CELT_ONLY (downward hysteresis).

The **dynamic range of interest** for this target is therefore
approximately **9 to 75 kbps**, with biased peaks around the per-config
boundary value.

### Bandwidth thresholds (NB ↔ MB ↔ WB ↔ SWB ↔ FB)

`encoder.rs:106-111`. MB is skipped (`encoder.rs:2020-2022`), so the
practical transitions are NB↔WB, WB↔SWB, SWB↔FB. Format is
`[thr, hys, thr, hys, thr, hys, thr, hys]` for the four crossings:

```
mono   voice: 9000, 700, 9000, 700, 13500, 1000, 14000, 2000
mono   music: 9000, 700, 9000, 700, 11000, 1000, 12000, 2000
stereo voice: 9000, 700, 9000, 700, 13500, 1000, 14000, 2000
stereo music: 9000, 700, 9000, 700, 11000, 1000, 12000, 2000
```

Effective transition equiv_rate values:

- **NB → WB**: ≈ 9 kbps equiv_rate (NB↔MB and MB↔WB both 9000).
- **WB → SWB**: 11 kbps (music) to 13.5 kbps (voice) equiv_rate.
- **SWB → FB**: 12 kbps (music) to 14 kbps (voice) equiv_rate.

### SILK ↔ HYBRID promotion rule

`encoder.rs:2110-2114`:

```
if mode == SILK_ONLY && bandwidth > WB { mode = HYBRID }
if mode == HYBRID    && bandwidth <= WB { mode = SILK_ONLY }
```

So the SILK/HYBRID split is *driven entirely by bandwidth*. HYBRID
exists only when the auto-mode landed on SILK and the bandwidth
gate raised it to SWB or FB. The SILK→HYBRID transition is therefore
the **WB→SWB bandwidth threshold**, i.e. ~11–13.5 kbps equiv_rate, at
sample rates where SWB/FB are reachable (sample_rate ≥ 24000 for SWB,
48000 for FB, capped further by `set_max_bandwidth`).

### Other forcings

- `application = RESTRICTED_LOWDELAY` ⇒ unconditionally CELT_ONLY
  (`encoder.rs:1864-1865`). Useful negative test only.
- `lfe = 1` ⇒ unconditionally CELT_ONLY at NB (`encoder.rs:1918-1919,
  2104-2107`). Out of scope (Stream C territory if any).
- `frame_size < fs/100` (i.e. < 10 ms) ⇒ forces CELT_ONLY
  (`encoder.rs:1915-1916`). 2.5 ms and 5 ms frames cannot be SILK or
  HYBRID. Out of scope.
- `set_force_mode(MODE_*)` overrides auto-selection
  (`encoder.rs:1910-1911, 3808-3818`). Critical lever for this target.
- FEC + high loss override forces SILK_ONLY at moderate-voice signals
  (`encoder.rs:1894-1900`). Already exercised by existing target.
- DTX + voice_est > 100 forces SILK_ONLY (`encoder.rs:1901-1904`).
  Already exercised.

### Setter constraints

- `set_application` (`encoder.rs:3842-3856`) **rejects** mid-stream
  changes — the gate at line 3849 returns `OPUS_BAD_ARG` if `first == 0`
  and the application differs. So the application is fixed at encoder
  construction; per-frame application changes are illegal and must
  not be attempted.
- `set_max_bandwidth` (`encoder.rs:3536-3542`), `set_bandwidth`
  (`encoder.rs:3522-3530`), `set_signal` (`encoder.rs:3548-3554`),
  `set_force_mode` (`encoder.rs:3808-3818`), and `set_force_channels`
  (`encoder.rs:3510-3516`) are all legal mid-stream and reset cleanly
  per CTL call.

## Considered Options

### Option 1: Extend `fuzz_encode_multiframe` in place

Add the new CTLs (max_bandwidth, signal, force_mode) and bias the
bitrate distribution toward boundaries, all in the existing target.

Pros:

- One fewer file.
- Reuses existing panic-capture, fingerprint, oracle, and decode-back
  infrastructure.

Cons:

- Conflicts with Stream A, which is broadening that exact target
  along an orthogonal axis (vbr_constraint, prediction_disabled,
  force_channels, etc.). Two streams editing the same `Arbitrary` impl
  is a merge nightmare and dilutes the byte-exact attribution if a
  bug surfaces.
- The boundary-biased bitrate distribution is a *different* sampling
  policy from the existing uniform one — running both in one target
  means coverage of either is half what it could be at fixed budget.
- Existing target's fingerprint shape would need to grow; existing
  corpus seeds would re-bind to new fields. Stream A is already
  proposing schema changes there.

Decision: reject. Keep the existing target as a uniform-grammar
baseline (Stream A's broadening is additive, not boundary-biased)
and put boundary-biased coverage in a fresh target.

### Option 2: New target `fuzz_encode_mode_transitions`

Single new target file, dedicated boundary-biased grammar, dedicated
oracle, optional small `c_reference.rs` extension.

Pros:

- Independent of Stream A's edits to `fuzz_encode_multiframe`.
- Boundary-biased Arbitrary impl can be tuned without disturbing
  the uniform baseline.
- Easy to attribute findings: any crash is mode-transition-shaped by
  construction.
- Same byte-exact oracle as the existing target — proven to land
  bugs in 2 minutes when one exists.

Cons:

- ~50% of the boilerplate (panic-capture, fingerprint, decode-back)
  duplicates the existing target. Mitigatable by lifting helpers,
  but not strictly needed for a single-slice deliverable.

Decision: accept. This is the cleanest fit for the multi-stream
parallel-implementation phase.

### Option 3: Force every frame's mode via `set_force_mode`, ignore auto

Drive transitions by selecting the mode directly per frame instead of
relying on the rate/bandwidth/signal logic to do it.

Pros:

- Maximally direct test of redundancy / SILK re-init / mode-flip code
  paths across consecutive frames.
- Skips the entire auto-decision logic, so failures localize to
  state-transition handlers, not mode-decision math.

Cons:

- Skips the auto-decision logic, so any bug *inside* that logic is
  invisible to this target. The auto path is where state hysteresis
  (`prev_mode`-based threshold adjustment) lives.
- The two 2026-05-07 bugs were both *state propagation* bugs, not
  *decision* bugs. Forcing mode would still reach them. But it would
  miss any threshold-arithmetic bug.

Decision: partial accept. Use a **two-mode grammar**: one bit selects
between "auto with boundary-biased bitrate" (covers auto-decision)
and "force-mode shuffle" (covers transition handlers directly). This
double-covers the surface for marginal extra Arbitrary cost.

### Option 4: Drive mode transitions through bandwidth/signal toggling

Instead of touching bitrate, vary `max_bandwidth` and `signal` per frame
to push the mode decision across the boundary.

Pros:

- Exercises the bandwidth-coercion path
  (`encoder.rs:2036-2039, 2110-2114`), which is the SILK/HYBRID lever.
- `signal_type` directly biases voice_est (`encoder.rs:1799-1801`,
  not yet read in detail) and shifts the threshold.

Cons:

- Without bitrate variation, signal/bandwidth changes alone may not
  cross the threshold often enough — they shift it but the bitrate
  lands wherever the random draw puts it.

Decision: accept as a **complementary** lever rather than a
replacement. Combine with bitrate biasing.

### Decision summary

Build a new fuzz target `fuzz_encode_mode_transitions` that:

- Per-frame draws bitrate from a boundary-biased distribution
  (Option 2 + 3 hybrid), with a small chance to use `set_force_mode`
  to drive transitions directly.
- Per-frame draws `max_bandwidth` and `signal` to lever the
  mode-decision threshold (Option 4).
- Holds `application` constant (legal constraint per
  `encoder.rs:3849`).
- Reuses `frame_duration::legal_frame_size_samples_per_channel` for
  the per-sequence frame size, restricting to frame durations that
  *can* select SILK or HYBRID (≥ 10 ms).

## Implementation Plan

### New target file

`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_mode_transitions.rs`

Skeleton (boundary-biased grammar):

```rust
#![no_main]
use libfuzzer_sys::arbitrary::{self, Arbitrary, Unstructured};
use libfuzzer_sys::fuzz_target;
use ropus::opus::encoder::{
    OpusEncoder, OPUS_APPLICATION_AUDIO, OPUS_APPLICATION_VOIP,
    OPUS_AUTO, OPUS_SIGNAL_MUSIC, OPUS_SIGNAL_VOICE,
    MODE_SILK_ONLY, MODE_HYBRID, MODE_CELT_ONLY,
};
use ropus::opus::decoder::OpusDecoder;

#[path = "c_reference.rs"]
mod c_reference;
#[path = "frame_duration.rs"]
mod frame_duration;

// Sample rates that can reach HYBRID/CELT; 8 kHz is included so the
// SILK_ONLY-NB ↔ CELT_ONLY-NB transition is reachable, but 12 kHz and
// up are needed for SWB/FB and therefore HYBRID.
const SAMPLE_RATES: [i32; 4] = [8000, 16000, 24000, 48000];

// Application is fixed for the whole sequence (set_application
// rejects mid-stream changes; encoder.rs:3849). Only AUDIO and VOIP —
// RESTRICTED_LOWDELAY pins to CELT_ONLY and would skip the transitions
// this target is built to find.
const APPLICATIONS: [i32; 2] = [
    OPUS_APPLICATION_AUDIO,
    OPUS_APPLICATION_VOIP,
];

const BANDWIDTHS: [i32; 4] = [
    c_reference::OPUS_BANDWIDTH_NARROWBAND,    // 1101
    c_reference::OPUS_BANDWIDTH_WIDEBAND,      // 1103
    c_reference::OPUS_BANDWIDTH_SUPERWIDEBAND, // 1104
    c_reference::OPUS_BANDWIDTH_FULLBAND,      // 1105
];
// MB intentionally absent from the picker — encoder skips it
// (encoder.rs:2020-2022).

const SIGNALS: [i32; 3] = [OPUS_AUTO, OPUS_SIGNAL_VOICE, OPUS_SIGNAL_MUSIC];

const FORCED_MODES: [i32; 4] = [
    OPUS_AUTO,
    MODE_SILK_ONLY,
    MODE_HYBRID,
    MODE_CELT_ONLY,
];

#[derive(Debug)]
struct ModeTransitionInput {
    sample_rate_idx: u8,
    channels_minus_one: u8,
    application_idx: u8,
    frame_duration_selector: u8, // 0..=5 mod, but Arbitrary picks ≥ 10 ms
    frames: Vec<FrameKnobs>,
}

#[derive(Debug)]
struct FrameKnobs {
    /// Boundary-biased bitrate selector. Decoded into a kbps value
    /// concentrated on the auto-decision boundary for the parent input's
    /// channels/signal/application.
    bitrate_selector: u16,
    max_bandwidth_idx: u8,
    signal_idx: u8,
    forced_mode_idx: u8,
    complexity: u8,
    vbr: bool,
    vbr_constraint: bool,
    fec: u8,
    dtx: bool,
    loss_perc: u8,
    pcm_bytes: Vec<u8>,
}
```

`vbr_constraint` is included as a per-frame knob (matches
`CModeTransitionConfig` and the C-side wrapper). Stream A is also
varying this dimension; co-varying it with mode transitions is
deliberate — CBR (`vbr=1`, `vbr_constraint=1`) applies the 1/12
equiv_rate penalty (`encoder.rs:711-740`), which shifts the
mode-decision boundary downward and is therefore directly relevant
to this target's coverage.

#### Arbitrary impl: bitrate biasing

The bitrate sampler is the heart of this target. The boundary depends on
several knobs that are themselves Arbitrary, so the bias has to be
*conditional* on the knobs already drawn.

```rust
fn boundary_biased_bitrate(
    raw: u16,
    channels: i32,
    application: i32,
    signal: i32,
    forced_mode: i32,
) -> i32 {
    // For forced modes the boundary is moot — the encoder bypasses
    // auto-decision. Use a wider [6 kbps, 96 kbps] range there to
    // cover the "narrow CELT vs wide CELT" rate-allocation transitions
    // and the hybrid SWB/FB rate split (encoder.rs:128-137 RATE_TABLE).
    if forced_mode != OPUS_AUTO {
        return 6000 + (raw as i32 % 90_001);
    }

    // Centre kbps for the auto-decision boundary (mode_thresholds
    // interpolation, encoder.rs:114-117 + 1869-1876). Conservative
    // mid-points; voice_est is unknown at fuzz-time so we hedge.
    let centre_bps: i32 = match (channels, signal) {
        (1, OPUS_SIGNAL_VOICE) => 60_000,                     // mono voice peak
        (1, OPUS_SIGNAL_MUSIC) => 14_000,                     // mono music peak
        (1, _)                 => 30_000,                     // mono auto
        (2, OPUS_SIGNAL_VOICE) => 44_000,                     // stereo voice peak
        (2, OPUS_SIGNAL_MUSIC) => 14_000,                     // stereo music peak
        (2, _)                 => 25_000,                     // stereo auto
        _                      => 30_000,
    };
    let centre = if application == OPUS_APPLICATION_VOIP {
        centre_bps + 8_000   // VOIP threshold boost (encoder.rs:1878-1880)
    } else {
        centre_bps
    };

    // 70 % of inputs land in [centre - 6 kbps, centre + 6 kbps].
    // 20 % land in the WB↔SWB bandwidth band (11–14 kbps) — the
    // SILK↔HYBRID transition.
    // 10 % land in the wide [6 kbps, 510 kbps] uniform range as a
    // safety net.
    let bucket = raw % 10;
    if bucket < 7 {
        let span = 12_000_i32;          // ±6 kbps
        let offset = (raw as i32 % span) - span / 2;
        (centre + offset).clamp(6_000, 510_000)
    } else if bucket < 9 {
        9_000 + (raw as i32 % 6_001)    // 9–15 kbps band
    } else {
        6_000 + (raw as i32 % 504_001)  // uniform fallback
    }
}
```

Key choices and their grounding:

- `centre_bps` values are pulled from the table in the
  "Mode Transition Boundaries" section above. Voice_est is not
  observable at fuzz-input-construction time; the biased centres are
  a heuristic that aims at the mid-voice (voice_est ≈ 64–96)
  boundary, where the threshold sits *inside* a 12 kbps window.
- The 20 % WB↔SWB band specifically targets the SILK→HYBRID
  promotion gate (`encoder.rs:2110-2114`), reachable only when
  bandwidth is auto-picked (`set_max_bandwidth >= SWB`).
- The 10 % uniform fallback preserves the property that *anything*
  reachable by the uniform multiframe target is still reachable here,
  just at lower probability. Insurance against threshold-arithmetic
  drift.

#### Arbitrary impl: frame-duration selection

Frame durations < 10 ms force CELT_ONLY (`encoder.rs:1915-1916`) and
make this target degenerate. Restrict to 10 / 20 / 40 / 60 ms by
masking the selector:

```rust
// frame_duration::legal_frame_duration_index gives 0..=5 for
// 2.5/5/10/20/40/60. Mask off the bottom two (2.5, 5 ms) which
// pin to CELT_ONLY.
let raw_selector: u8 = u.arbitrary()?;
let frame_duration_selector = 2 + (raw_selector % 4); // 2..=5
```

This lives only in this target, not in `frame_duration.rs`, because
it's a target-specific masking policy.

#### Arbitrary impl: per-frame knob distribution

```rust
impl FrameKnobs {
    fn arbitrary_for_shape<'a>(
        u: &mut Unstructured<'a>,
        pcm_bytes_needed: usize,
    ) -> arbitrary::Result<Self> {
        Ok(Self {
            bitrate_selector: u.arbitrary()?,
            max_bandwidth_idx: u.int_in_range(0..=3)?,
            signal_idx:        u.int_in_range(0..=2)?,
            // 60 % AUTO, 40 % forced — keeps auto-decision well-covered
            // while still hitting forced transitions.
            forced_mode_idx:   {
                let r: u8 = u.arbitrary()?;
                if r % 5 < 3 { 0 } else { 1 + (r % 3) }
            },
            // Skip cx ≥ 10 to dodge the analysis.c divergence
            // (same constraint as fuzz_encode_multiframe.rs:166-168).
            complexity: u.int_in_range(0..=9)?,
            vbr:        u.arbitrary()?,
            vbr_constraint: u.arbitrary()?,
            fec:        u.int_in_range(0..=2)?,
            dtx:        u.arbitrary()?,
            loss_perc:  u.int_in_range(0..=100)?,
            pcm_bytes:  u.bytes(pcm_bytes_needed)?.to_vec(),
        })
    }
}
```

Frame count: 2..=12 (slightly tighter than multiframe's 2..=16 — each
iteration spends more CPU on Arbitrary-decoding, and 12 frames is
enough cross-frame state to expose state-carry bugs).

#### Fuzz body

Mirrors `fuzz_encode_multiframe`'s flow with the additional CTLs
applied per frame on both Rust and C sides:

```rust
fuzz_target!(|input: ModeTransitionInput| {
    init_panic_capture();
    // ... fingerprint, PCM decode, etc. ...

    let mut rust_enc = OpusEncoder::new(sample_rate, channels, application)?;
    let mut rust_packets = Vec::with_capacity(input.frames.len());

    for fk in &input.frames {
        let bitrate = boundary_biased_bitrate(
            fk.bitrate_selector, channels, application,
            SIGNALS[fk.signal_idx as usize],
            FORCED_MODES[fk.forced_mode_idx as usize],
        );
        let max_bw = BANDWIDTHS[fk.max_bandwidth_idx as usize];
        let sig    = SIGNALS[fk.signal_idx as usize];
        let mode   = FORCED_MODES[fk.forced_mode_idx as usize];

        rust_enc.set_bitrate(bitrate);
        rust_enc.set_max_bandwidth(max_bw);
        rust_enc.set_signal(sig);
        rust_enc.set_force_mode(mode);
        rust_enc.set_complexity(fk.complexity as i32);
        rust_enc.set_vbr(if fk.vbr { 1 } else { 0 });
        rust_enc.set_vbr_constraint(if fk.vbr_constraint { 1 } else { 0 });
        rust_enc.set_inband_fec(fk.fec as i32);
        rust_enc.set_dtx(if fk.dtx { 1 } else { 0 });
        rust_enc.set_packet_loss_perc(fk.loss_perc as i32);

        let mut out = vec![0u8; 4000];
        let len = rust_enc.encode(pcm, frame_size, &mut out, 4000)?;
        out.truncate(len as usize);
        rust_packets.push(out);
    }

    // Build CModeTransitionConfig per frame and call the new
    // c_encode_mode_transitions wrapper (see c_reference.rs extension
    // below) — symmetric setter sequence on the C side.

    let c_packets = c_reference::c_encode_mode_transitions(
        &frame_refs, &c_cfgs, frame_size, sample_rate,
        channels, application,
    )?;

    // Same byte-exact frame-by-frame compare as fuzz_encode_multiframe.
    // Same parseability + decode-back invariants.
});
```

### `c_reference.rs` extension

This is non-trivial: the existing `CEncodeConfig` and
`c_encode_multiframe` do not cover `max_bandwidth`, `signal`, or
`force_mode`. Three options:

#### Option A: Extend `CEncodeConfig` and `c_encode_multiframe`

Pros: One C-side wrapper. Cons: Adds three fields used by only one
target; existing target's fingerprint policy doesn't need them.

#### Option B: Add a new struct `CModeTransitionConfig` and a new wrapper

Pros: Existing target untouched. Cons: One more wrapper; minor code
duplication inside the wrapper.

#### Option C: Reuse `CEncodeConfig`, hand the extra CTLs in via a parallel array

Pros: No struct changes. Cons: Awkward call shape.

**Decision: Option B.** The new struct and wrapper isolate the new
target completely and keep `apply_c_encoder_config` untouched (it has
been stable since 2026-04 and is used by every encode target). This
avoids any chance of regressing the saturated targets.

/// Extends `CEncodeConfig` with the three transition-driving CTLs.
#[derive(Clone, Copy, Debug)]
pub struct CModeTransitionConfig {
    pub bitrate: i32,
    pub complexity: i32,
    pub vbr: i32,
    pub vbr_constraint: i32,
    pub inband_fec: i32,
    pub dtx: i32,
    pub loss_perc: i32,
    pub max_bandwidth: i32,   // OPUS_BANDWIDTH_*
    pub signal: i32,          // OPUS_AUTO | OPUS_SIGNAL_VOICE | OPUS_SIGNAL_MUSIC
    pub forced_mode: i32,     // OPUS_AUTO | MODE_SILK_ONLY | MODE_HYBRID | MODE_CELT_ONLY

#[inline]
unsafe fn apply_c_mode_transition_config(enc: *mut OpusEncoder, cfg: &CModeTransitionConfig) {
    unsafe {
        // Order mirrors the Rust target's setter order. Bitrate must
        // come before VBR per the C reference's accepted ordering.
        opus_encoder_ctl(enc, OPUS_SET_BITRATE_REQUEST,           cfg.bitrate);
        opus_encoder_ctl(enc, OPUS_SET_VBR_REQUEST,               cfg.vbr);
        opus_encoder_ctl(enc, OPUS_SET_VBR_CONSTRAINT_REQUEST,    cfg.vbr_constraint);
        opus_encoder_ctl(enc, OPUS_SET_INBAND_FEC_REQUEST,        cfg.inband_fec);
        opus_encoder_ctl(enc, OPUS_SET_DTX_REQUEST,               cfg.dtx);
        opus_encoder_ctl(enc, OPUS_SET_PACKET_LOSS_PERC_REQUEST,  cfg.loss_perc);
        opus_encoder_ctl(enc, OPUS_SET_COMPLEXITY_REQUEST,        cfg.complexity);
        opus_encoder_ctl(enc, OPUS_SET_MAX_BANDWIDTH_REQUEST,     cfg.max_bandwidth);
        opus_encoder_ctl(enc, OPUS_SET_SIGNAL_REQUEST,            cfg.signal);
        // OPUS_SET_FORCE_MODE_REQUEST is not in opus_defines.h —
        // it's the private CTL OPUS_SET_FORCE_MODE = 11002 from

## Risks

- **Centre-bps heuristic drift.** The boundary depends on `voice_est`,
  which is set by the analysis stage from PCM content. The Arbitrary
  impl can't observe it, so the bias has to assume mid-voice. If the
  randomly-chosen PCM ends up extreme (pure silence, full-scale noise),
  the actual boundary may sit outside the ±6 kbps window. Mitigated by
  the 10 % uniform fallback. Risk: low.

- **Forced-mode + illegal frame_size combinations.** Forcing
  `MODE_SILK_ONLY` with a 5 ms frame is contradictory; the encoder's
  override at `encoder.rs:1915-1916` will silently coerce mode back to
  CELT_ONLY. That's fine for byte-exact compare (C does the same) but
  it dilutes coverage. Mitigated by restricting frame durations to
  ≥ 10 ms in the Arbitrary impl.

- **`OPUS_SET_FORCE_MODE` constant drift.** The private CTL number
  `11002` is in `opus_private.h`; if a future libopus version renumbers
  it, the C-side and Rust-side calls diverge silently. Mitigated by
  adding a build-time assertion or a one-line comment cite to the
  current `opus_private.h` line. Risk: low (Opus has not renumbered any
  CTL since 2014).

- **`get_mode()` returning the *current* finalized mode rather than the
  intent.** Confirmed at `encoder.rs:3866-3868` — the field is set at
  `encoder.rs:2118` before encode runs, so reading it post-encode gives
  the actual mode used. Validation plan stands.

- **Extension wrapper drift.** `apply_c_mode_transition_config` and
  the Rust target's per-frame setter sequence must apply the same CTLs
  in the same order. A drift here looks exactly like a parity bug in
  the codec but isn't one. Mitigated by keeping the order in one
  comment block in both files, plus a one-line `c_reference.rs`
  comment cross-referencing the Rust target.

- **Application is not a per-frame transition axis.**
  `set_application` rejects mid-stream changes
  (`encoder.rs:3849-3851`). Stream D fixes application at construction
  and calls this out in target docs. Risk: only if a future reviewer
  wonders why application isn't fuzzed per-frame; mitigated by the
  comment.

- **Per-iteration cost.** Expected throughput is comparable to
  `fuzz_encode_multiframe`'s campaign rate, roughly 0.5–2 runs/sec/worker
  (per `/home/md/language/ropus/wrk_journals/2026.05.07 - JRN - overnight 8h fuzz campaign.md`,
  the "Final per-target stats" row records `fuzz_encode_multiframe`
  accumulating 16 K total runs across 4 workers over ~8 hours, which
  is ~0.6–1 runs/sec/worker — `opus_encode` dominates per-iteration
  cost). The added "+3 CTLs per frame" is genuinely negligible against
  that baseline. The boundary-biased distribution reduces wasted runs
  that land on identical-mode sequences, so effective coverage per
  second should be at least on par with the baseline. Throughput is
  not predicted in this HLD; the smoke-validation step measures
  exec/sec empirically and that number is the anchor for the
  acceptance run.

- **Stream A merge conflict on `frame_duration.rs`.** Stream A may add
  helpers to `frame_duration.rs`. Stream D does not modify
  `frame_duration.rs` — only imports it. No conflict.

- **Stream C overlap (stereo).** HYBRID at high bitrates often goes
  stereo. Stream D will reach stereo SILK/HYBRID by chance through
  `channels=2`. That's fine and adds value rather than duplication —
  Stream C's *primary* axis is the stereo state machine
  (`hybrid_stereo_width_q14`, `silk_mode.to_mono`), Stream D's
  *primary* axis is mode flips. Some overlap is healthy; both targets
  will, ideally, find different bugs.

## Out of Scope

- Codec source modifications. The target is read-only with respect to
  `/home/md/language/ropus/ropus/src/`.
- Mid-stream `set_application` changes (illegal per
  `encoder.rs:3849`).
- Frame durations < 10 ms (forced to CELT_ONLY; degenerate for this
  target).
- Mediumband bandwidth (encoder skips it; not user-reachable through
  the auto path).
- LFE channel forcing (`encoder.rs:1918-1919`); orthogonal axis,
  Stream C territory.
- Multistream encoder. Stream D drives the mono/stereo `OpusEncoder`
  only; multistream mode-transition coverage is a separate question
  (existing `fuzz_multistream` target is saturated).
- Decoder side. The decoder consumes whatever mode the encoder
  produced — its mode-transition handling is exercised transitively
  by the decode-back invariant at the end of the target, but no
  separate decode-mode-transition fuzz is in scope.
- Fuzz corpus seeding. The Arbitrary impl is parseable from arbitrary
  bytes; no hand-curated seeds are needed for the first run. If the
  smoke-validation transition-coverage thresholds aren't met, seeds
  may be added in a follow-up.

## Key Decisions

- New target file `fuzz_encode_mode_transitions` rather than
  extending `fuzz_encode_multiframe` (avoids Stream A conflict,
  isolates boundary-biased grammar).
- New `CModeTransitionConfig` struct + new
  `c_encode_mode_transitions` wrapper in `c_reference.rs` rather than
  extending the existing `CEncodeConfig` (preserves stability of the
  existing wrapper used by saturated targets).
- Boundary bias = 70 % within ±6 kbps of a per-config centre, 20 % in
  the SILK→HYBRID bandwidth band (9–15 kbps), 10 % uniform [6 K, 510 K]
  safety net.
- Frame duration restricted to 10 / 20 / 40 / 60 ms; shorter durations
  pin to CELT_ONLY.
- Application held constant per sequence (legal constraint).
- `vbr_constraint` varied per-frame (CBR penalty shifts the mode
  threshold; co-varies cleanly with Stream A's broader use of the
  same axis on the multiframe target).
- Forced-mode + auto-mode mixed in one target (60/40 in favour of
  AUTO) to cover both auto-decision arithmetic and transition handlers.
- Smoke validation requires empirical confirmation that ≥ 30 % of
  sequences contain an actual mode flip and ≥ 5 % contain a
  SILK ↔ HYBRID transition.
- Use `OPUS_SET_FORCE_MODE_REQUEST = 11002` from
  `reference/celt/opus_private.h`; add a one-line comment cite.
- Ban use of `set_application` mid-stream; cite
  `encoder.rs:3849-3851` in the target docs.
- ~400–410 LoC total slice; one new file plus ~70 lines into
  `c_reference.rs`.
