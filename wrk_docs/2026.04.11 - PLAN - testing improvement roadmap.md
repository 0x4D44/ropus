## Phase 1: Branch Coverage for State Machines

**Goal:** Raise branch coverage in the six weakest files above 80%.
**Effort:** Medium. The harness already supports every configuration knob.
**ROI:** Highest of anything on this list.

### Actions

1. **Run `cargo llvm-cov --branch` with `--show-missing-lines`** for each weak file.
   Identify the specific uncovered branches — most will be mode transitions,
   error paths, or configuration combinations the current test matrix doesn't hit.

2. **Write targeted harness configurations** that force the uncovered paths:
   - DTX enable/disable transitions mid-stream
   - FEC with actual packet loss (not just FEC-enabled encode)
   - Bandwidth switching (NB -> WB -> FB) within a single stream
   - CBR vs VBR at extreme bitrates (6kbps, 510kbps)
   - PLC with bursty loss patterns (5 consecutive lost, recover, 3 lost)
   - Frame size transitions (2.5ms <-> 60ms)
   - SILK-only vs Hybrid vs CELT-only forced mode transitions

3. **Add coverage smoke tests** (`src/coverage_tests.rs`) for paths that can't
   be reached through the CLI harness — internal state machine edges, error
   returns from CTL calls, unusual channel/sample-rate combinations.

4. **Measure again.** If any file is still below 80% branch coverage, the
   remaining uncovered branches are likely dead code or unreachable — audit
   and annotate them.

### Definition of Done

- All six files above 80% branch coverage
- Or remaining uncovered branches documented as dead/unreachable

## Phase 2: Fuzz Testing Maturation

**Goal:** Move from "we ran it twice overnight" to a repeatable discipline.
**Effort:** Low-medium. Infrastructure exists; needs process.

### Actions

1. **Real-audio seed corpus.** Current seeds are synthetic (sine, noise, silence).
   Encode 10-20 diverse real audio clips through the C reference encoder, save
   the compressed packets as fuzz seeds. This gets past the shallow coverage
   plateau where structured random bytes mostly produce invalid configs.

2. **Multiframe fuzz target.** Current targets encode single frames. Add
   `fuzz_encode_multiframe` that feeds 5-10 sequential frames with prediction
   state carry-over. This exercises state accumulation bugs that single-frame
   fuzzing structurally cannot reach.

3. **Promote safety targets to differential.** The three `*_safety` targets
   only check for crashes. A crash-free decode that silently produces wrong
   samples is a correctness bug. Where SILK numerical divergence allows it
   (CELT-only mode, packet parsing), compare against C reference.

4. **Nightly automation.** Set up a scheduled script (cron, Task Scheduler, or
   GitHub Actions) that runs `tools/fuzz_run.sh --duration 3600` nightly and
   saves any new crashes to `tests/fuzz/crashes/`. Doesn't need to be fancy —
   a log file and non-zero exit on new crashes is enough.

5. **Crash-to-regression pipeline.** When a fuzz crash is found and fixed, the
   minimized input should be added as a unit test (not just kept in the crash
   corpus). This prevents the same class of bug from recurring after refactors.

### Definition of Done

- Real-audio seeds in corpus
- Multiframe fuzz target running
- At least one nightly fuzz job automated
- All fixed crashes have corresponding regression unit tests

## Phase 3: Property-Based Testing

**Goal:** Automated invariant checking with shrinking for packet format and API contracts.
**Effort:** Medium. Requires adding proptest dependency.

### Actions

1. **Add `proptest` as a dev-dependency.**

2. **Packet format invariants** — these are the highest-value properties:
   - `get_nb_frames(pkt) * get_samples_per_frame(pkt, fs) == get_nb_samples(pkt, fs)`
     for all valid TOC bytes and all 5 sample rates
   - `get_bandwidth(pkt)` maps monotonically to sample rate
   - `get_nb_channels(pkt)` is always 1 or 2
   - Encoding then calling `get_nb_frames` on the output returns expected count

3. **Decoder invariants:**
   - Output sample count equals `frame_size * channels * nb_frames` regardless of input
   - Decoding same packet twice with fresh state produces identical output (idempotency)
   - After N frames of PLC, output energy converges toward zero (not diverges)

4. **Encoder invariants:**
   - Output packet is always valid (parseable by `opus_packet_parse`)
   - CBR mode output length is deterministic for same frame size
   - Increasing bitrate never increases compressed size for CBR

5. **Energy conservation:**
   - Decoded energy bounded relative to input energy (catches gain explosion)
   - Encoding silence produces packets that decode to near-silence

### Definition of Done

- proptest running in `cargo test`
- At least 5 property tests covering packet format + decoder + encoder invariants

## Phase 4: Dead Code and Noise Cleanup

**Goal:** Remove mutation testing noise and coverage denominator inflation.
**Effort:** Low. Mostly deletion.

### Actions

1. **Remove dead functions** identified by mutation testing:
   - `types.rs`: ~20 unused wrapper functions
   - `celt_atan2p`, `celt_cos_norm32`, `opus_limit2_checkwithin1`
   - Verify with `cargo test` + grep for callers before removing

2. **Fix warnings** in `opus/multistream.rs` and `tests/harness/main.rs`.
   Warning noise lowers signal quality across all test output.

3. **Re-run mutation testing on cleaned modules** to get accurate kill rates
   without dead-code survivors inflating the denominator.

### Definition of Done

- Zero dead functions flagged by mutation testing
- `cargo build 2>&1 | grep warning` produces no output
- Updated mutation kill rates documented

## Phase 5: Mutation Testing Expansion

**Goal:** Extend mutation testing from math_ops pilot to all high-risk modules.
**Effort:** Medium. ~30 min per module for cargo-mutants run + analysis.

### Actions

1. **Priority order** (by risk, not alphabetical):
   - silk/common.rs (71 tests, core shared logic)
   - opus/decoder.rs (packet parsing, mode dispatch)
   - opus/encoder.rs (mode selection, rate control)
   - silk/decoder.rs (PLC/FEC state machine)
   - celt/decoder.rs (CELT decode pipeline)
   - celt/encoder.rs (CELT encode pipeline)

2. **For each module:**
   - Run `cargo mutants --file src/<module>.rs -j4 --timeout-multiplier 3`
   - Triage survivors: dead code vs equivalent vs real gap
   - Write targeted tests for real gaps
   - Target: >85% live-code kill rate

3. **Track results** in a mutation testing dashboard (simple markdown table
   appended to this document or a dedicated tracking file).

### Definition of Done

- All six priority modules assessed
- Live-code kill rate >85% for each, or survivors documented as equivalent/dead

## Phase 6: Exploratory Testing

**Goal:** Manual/scripted exploration of paths the systematic tests don't cover.
**Effort:** Low per session. Ongoing.

### Exploration Targets

1. **Mode transitions under load** — encode 10s audio, switch application
   mid-stream (VOIP -> AUDIO -> RESTRICTED_LOWDELAY), verify no crash/divergence.

2. **Bitrate extremes** — encode at 6kbps and 510kbps with all five sample
   rates, verify decode produces reasonable output (not silence, not explosion).

3. **Multistream edge cases** — unusual channel counts, mapping families,
   coupled vs uncoupled stream ratios.

4. **PLC stress patterns** — systematic loss patterns: bursty (5 lost, 2 good,
   3 lost), periodic (every 3rd packet), total blackout (20 consecutive lost).

5. **Frame size transitions** — switch between 2.5ms and 60ms frames within
   a single encoding session, verify state consistency.

6. **Repacketizer adversarial input** — hand-craft packets with maximum
   padding, zero-length frames, code-3 with inconsistent VBR lengths.

### How to Run

These should be scripted as harness invocations (not unit tests) so they can
be run interactively when investigating a specific area. Add any that find
bugs as permanent regression tests.

### Definition of Done

- Each target explored at least once
- Bugs found added as regression tests
- No-bug-found targets documented (negative results are still results)

## New Oracles to Implement

Beyond the C reference differential (which remains the primary oracle), these
oracles encode "what correct looks like" in forms machines can check at scale:

| Oracle | What It Catches | Where to Use |
|--------|----------------|--------------|
| Sample count invariant | Buffer overflows, frame accounting bugs | Decoder, all paths |
| Decode idempotency | State leakage between decode calls | Decoder, fresh-state |
| PLC convergence | Runaway state divergence | Decoder, PLC path |
| Energy conservation | Gain explosion, silence-to-noise bugs | Roundtrip |
| Silence preservation | Encoding artifacts on zero input | Encoder + decoder |
| Bitrate monotonicity | Rate control inversion | Encoder, statistical |
| Packet self-consistency | TOC/length/frame-count disagreement | Packet parsing |
| CBR determinism | Non-deterministic output | Encoder, CBR mode |

Implement these as assertions inside fuzz targets (not just unit tests) —
they multiply the value of every fuzz iteration.
