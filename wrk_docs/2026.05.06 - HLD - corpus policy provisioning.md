# HLD - Corpus Policy Provisioning (Phase 3.4)

Date: 2026.05.06
Status: design only — no source/manifest/script changes in this slice.
Predecessor: `wrk_docs/2026.05.06 - HLD - real world corpus gate.md` (gate plumbing landed in commit `d70730f`).

## Goal

Tighten the corpus policy contract so that any "real-world corpus claim" emitted by `full-test --release-preflight` is honestly falsifiable: no silent skips, pinned-or-non-green semantics for entries that promise determinism, and an empty/degenerate corpus must fail under release-preflight when the gate is claimed. We do not build a new gate; we sharpen the contract on the gate that already exists.

## Non-Goals

- No WebM/Matroska parsing. WebM is deferred explicitly and surfaces as a loud "container deferred" line; it never satisfies a claim.
- No WebRTC/SFU/browser/messaging captures committed. No third-party voice notes. No streaming-service extracts.
- No codec math changes. No encoder/decoder tuning. No SILK/CELT/Hybrid behavior touched.
- No expansion of required manifest entries beyond the two new ones listed under Behavioral Change 3.
- No automated license-compliance tooling. SOURCES.md is human curation, gated by Arthur's review on PR.
- No CI provider config. No new tools/ scripts. No network fetch added to the default gate.
- No change to `report-only` (default) or `--quick --release-preflight` claim language: those remain explicitly unclaimed.

## Supervisor decisions on the HLD's open questions

Resolved before implementation kicks off so the implementation agent isn't blocked:

1. **Edge-case fixture provenance:** drop `opusenc` entirely. Use a second FFmpeg-native opus entry that exercises a different rate corner (e.g. speech-48k-stereo). FFmpeg-only keeps the reproducibility story consistent across both pinned entries; `opusenc` writes a non-suppressible vendor tag from the linked libopus version, which would make any SHA256 pin unreproducible across builds.
2. **Expose pinned-match counter in JSON/HTML.** Yes, as a one-line meta field. Field name: `pinned_entries_matched` (renamed from the earlier `compared_pinned_entries` to read clearly alongside the existing `compared_required_entries`).

## Supervisor revisions (post-devil's-advocate)

Three blockers raised by the devil's-advocate review are addressed here. The rest of the HLD remains as written; only the items below are amended.

### Revision A — per-entry generation parameters

Today `corpus.rs:667-668` `generate_entry` hardcodes `-b:a 32k` and ignores the manifest's `generation_command` field. Without per-entry parameters, every SHA256 pin would be locked at 32k regardless of the manifest declaration.

Schema additions to `[[entries]]`:

```
bitrate_kbps = <u32>          # required for generated-non-reference; integer kbps
channels     = <u32>          # required for generated-non-reference; 1 or 2
```

`ManifestEntry` gains the fields. Parser at `full-test/src/corpus.rs:867-908` validates `channels in {1, 2}` and `bitrate_kbps in 6..=510`. `generate_entry` reads them and assembles the FFmpeg argv as:

```
ffmpeg -y -hide_banner -loglevel error
       -i <source_fixture>
       -c:a opus -strict -2
       -b:a {bitrate_kbps}k
       -ac {channels}
       <output_path>
```

`generation_command` remains a human-readable documentation field. It is **not** executed; it is shown in HTML/JSON for traceability only.

### Revision B — drop opusenc, second pinned entry is FFmpeg-native

Replace the entry described as `ogg-edge-preskip-or-output-gain` (path-a recipe via `opusenc`) with:

`ffmpeg-native-speech-48k-stereo` — exercises the speech / stereo / mid-rate path. Source: `tests/vectors/speech_48k_mono.wav` (with `channels = 2`, FFmpeg up-mixes implicitly via `-ac 2`); if a stereo speech vector already exists in the repo, prefer that. `class = "generated-non-reference"`, `container_support = "supported-ogg-family0"`, `required = true`, `max_size_bytes = 200000`, `oracle = "exact-pcm-parity-vs-c-reference"`, `bitrate_kbps = 48`, `channels = 2`, `expected_sha256 = "<TO BE LOCKED AT FIRST GENERATION>"`. Implementation agent verifies the source WAV exists during scoping; if no stereo speech WAV exists, use the mono speech WAV with `channels = 1` and a different bitrate (32 or 40) so the pin is structurally distinct from `ffmpeg-native-sine-32k`.

The manifest's three required entries become:

| id | bitrate | channels | source | pinned |
|---|---|---|---|---|
| `ffmpeg-native-sine-32k` | 32k | 1 | `48k_sine1k_loud.wav` | no (smoke) |
| `ffmpeg-native-sine-24k-mono` | 24k | 1 | `48k_sine1k_loud.wav` | yes |
| `ffmpeg-native-speech-48k-stereo` | 48k | 2 | speech WAV | yes |

The "ogg-edge" framing is dropped: the gate's oracle is per-packet PCM parity, which is independent of `OpusHead` field exotica. The real value of the second pinned entry is catching encoder drift in a different rate / source / channel-count corner. SHA256 mismatch failures in `corpus.rs` continue to attribute drift via the FFmpeg version line.

R2 in the original Risks section (Ogg edge fixture provenance) is superseded by this revision. SOURCES.md remains an empty template; no third-party committed media is admitted.

### Revision C — `CommandRunner` seam for testability

`corpus.rs` currently calls `std::process::Command::new` directly in `detect_ffmpeg`, `generate_entry`, and the `corpus_diff` invocation. HLD tests 8–11 cannot stub these without a seam.

Introduce a thin trait local to `corpus.rs`:

```rust
pub trait CommandRunner {
    fn run(&self, program: &str, args: &[String]) -> std::io::Result<std::process::Output>;
}

pub struct RealCommandRunner;

impl CommandRunner for RealCommandRunner {
    fn run(&self, program: &str, args: &[String]) -> std::io::Result<std::process::Output> {
        std::process::Command::new(program).args(args).output()
    }
}
```

Threading:

- `detect_ffmpeg(runner: &dyn CommandRunner)`, `generate_entry(runner: &dyn CommandRunner, ...)`, and the `corpus_diff` invocation in `run_generated_smoke_with_probe` all take a `&dyn CommandRunner`.
- The public function exposed to the rest of `full-test` keeps its existing signature; internally it constructs `RealCommandRunner` and forwards. No call-site churn outside `corpus.rs`.
- A new test-only helper `with_runner(runner: &dyn CommandRunner)` is exposed under `#[cfg(test)]` for tests 8–11.

Tests use a `FakeCommandRunner` that maps `(program, args)` to canned `Output` (status, stdout, stderr). For tests 8 and 9, the fake returns FFmpeg-version stdout for the version probe and a real-file path for `generate_entry` (the fake actually writes the bytes to a tempdir so the SHA256 helper has something to hash); for tests 10 and 11, the fake returns synthetic `corpus_diff` output with the desired exit code.

### Revision D — counter naming

The HLD originally introduced `compared_pinned_entries` next to the existing `compared_required_entries`. The names read confusingly. Final names:

- `compared_required_entries` (unchanged, existing) — count of required entries that were `decoded_and_compared` by `corpus_diff`.
- `pinned_entries_matched` (new) — count of required entries with `expected_sha256` set whose produced digest equalled the pinned digest.
- `decoded_and_compared` (unchanged, from `corpus_diff` summary line) — total files (required or not) that decoded successfully and were compared.

Acceptance criteria, JSON envelope, HTML render, and oracle matrix all use the final names. The HLD's earlier mentions of `compared_pinned_entries` are superseded by this revision.

### 1. `corpus_diff` exit-code split: distinguish "no candidates" from "all deferred"

File: `/home/md/language/ropus/harness/src/bin_inner/corpus_diff.rs`

Today (`harness/src/bin_inner/corpus_diff.rs:33-42`, `:369-377`, `:444-455`) the binary documents three exit codes: `0` pass, `1` mismatch/panic/no-decoded-audio/missing-arg, `2` directory-exists-but-empty.

Today, `.webm` files reach `harness/src/bin_inner/corpus_diff.rs:474-479` and emit a single line `  SKIP <name> (WebM/Matroska container not parsed here)`. That line is ambiguous with skips for malformed Ogg or family-3 channel-mapping. There is no per-file marker that says "this is a known-deferred container, by design."

Required changes:

a) Add a fourth exit code, `3 = "all candidates deferred"`. The existing exit-code mapping in `RunStats::exit_code` (`harness/src/bin_inner/corpus_diff.rs:369-377`) becomes:

   - `candidates == 0` → `2`
   - `candidates > 0 && deferred == candidates` → `3`
   - `mismatched > 0 || panicked > 0 || decoded_and_compared == 0` → `1`
   - else → `0`

   Track a new `deferred: usize` field on `RunStats` (alongside `skipped`), incremented only on the explicit deferred-container path. `skipped` continues to mean "structural skip" (malformed Ogg, family != 0, OpusTags missing, etc.).

b) For each deferred file, emit a single-line `  DEFER <relpath> reason=<container-deferred-reason>` to stdout. Include the file path (so a CI log can grep for unexpected deferrals) and a reason key that is stable across runs. For WebM the reason is `webm-matroska-container-deferred`. The line is in addition to inclusion in the `CORPUS_DIFF_SUMMARY` line.

c) Extend `CORPUS_DIFF_SUMMARY` to include a `deferred=<n>` field. The current line at `harness/src/bin_inner/corpus_diff.rs:387-397` becomes:

   ```
   CORPUS_DIFF_SUMMARY candidates=N decoded_and_compared=N zero_audio=N skipped=N deferred=N mismatched=N panicked=N
   ```

   Field ordering after `skipped` keeps existing parsers in `full-test/src/corpus.rs:794-820` partially compatible (they iterate split_whitespace key=value), but the parser must learn the new key (see Change 2c).

d) Remove the silent `.webm` skip path. The classifier in `harness/src/bin_inner/corpus_diff.rs:76-85` continues to admit `.webm`, but the dispatch at `:474-479` now routes through the deferred path with a per-file `DEFER` line and `stats.deferred += 1`. There is no other path through which a known-deferred container can be admitted silently.

Compatibility note: external CI consumers that gate on exit code `0` still see `0` in the green case. Exit code `2` semantics are unchanged. Exit code `3` is new; the README must be updated under Change 5.

### 2. `corpus.rs`: tighten `--release-preflight` non-quick semantics

File: `/home/md/language/ropus/full-test/src/corpus.rs`

Today the binary status logic at `full-test/src/corpus.rs:482-557` evaluates only `output.status.success()` plus the parsed `CORPUS_DIFF_SUMMARY decoded_and_compared` count. Exit code `2` is treated as a generic non-zero status string. SHA256 is computed for generated files (`:437`) but never compared to a pinned digest. FFmpeg version is captured (`:337-369`, `:101-104`) but only surfaced on tooling failure, not on parity failure.

Required changes:

a) Treat exit code `2` ("no candidates") as a hard fail when `Mode::GeneratedSmoke`. Today `corpus_diff` exit `2` cannot occur on the generated-smoke path because we always populate the temp dir before invoking it; but `manifest.entries` could theoretically be empty in a malformed manifest, or an upstream change could empty the directory. Encode the policy explicitly: when `Mode::GeneratedSmoke` and `output.status.code() == Some(2)`, push an issue `corpus_diff exited 2 (no candidates) under release-preflight; corpus claim cannot be satisfied by an empty directory` and set `Status::Fail`.

b) Treat exit code `3` ("all deferred") as a hard fail under `Mode::GeneratedSmoke`. Issue text: `corpus_diff exited 3 (all candidates deferred); a claimed corpus must include at least one non-deferred entry`.

c) Extend the manifest summary parser at `full-test/src/corpus.rs:794-820` to read the new `deferred` key. Add `pub deferred: usize` to `CorpusDiffSummary` (`:79-87`). Tolerate parsing summaries from older `corpus_diff` builds (treat absent key as `0`) so an in-flight bisect does not break.

d) On any release-preflight failure path that has an FFmpeg version captured, include the full FFmpeg version line in the issue text. Concretely, the existing failure issues (e.g. `:494-501`) gain a one-line suffix: ` (ffmpeg=<version>)`. This becomes essential once Change 3 lands and SHA256 mismatch failures need to attribute drift to a specific encoder build.

e) New SHA256 pinning enforcement. After `sha256_of(&output_path)` at `full-test/src/corpus.rs:437`, if the corresponding `ManifestEntry` has `expected_sha256 = Some("…")`:

   - if the produced SHA256 matches: continue, mark the entry as `pinned_match` (new entry-status string).
   - if it does not match: push an issue `pinned SHA256 mismatch for <id>: produced=<hex> expected=<hex> (ffmpeg=<version>)` and force `Status::Fail`.

   If `expected_sha256 = None`, behavior is unchanged: the entry runs but is treated as "smoke" only — see Change 3 and the count semantics at `full-test/src/corpus.rs:498-500`. An unpinned required entry continues to count toward `compared_required_entries` for backwards compatibility, but the `Outcome` gains a separate counter `compared_pinned_entries` so the claim language can distinguish "smoke" from "pinned".

f) Tighten the count comparison at `full-test/src/corpus.rs:494-501`. The release claim now requires `compared_pinned_entries >= pinned_required_entries`, where `pinned_required_entries` is the count of required entries whose manifest declares `expected_sha256`. The existing `decoded_and_compared >= required.len()` check stays as a precondition but is no longer the strongest statement.

g) Per supervisor decision 2: surface `compared_pinned_entries` in the JSON envelope (`full-test/src/report.rs:133-135` area) and HTML render (`full-test/src/html.rs:743-817` area) as a single meta field alongside the existing compared count. Naming: `compared_pinned_entries` in JSON, "Pinned entries matched" in HTML.

### 3. Manifest schema: add `expected_sha256`, plus two pinned required entries

File: `/home/md/language/ropus/tests/vectors/real_world/corpus_manifest.toml`

Today the manifest (`tests/vectors/real_world/corpus_manifest.toml`) has one required entry: `ffmpeg-native-sine-32k` (32 kbps, mono, sine source). The parser at `full-test/src/corpus.rs:867-908` rejects unknown keys; `expected_sha256` must be added explicitly to `EntryBuilder::set` and to `ManifestEntry`.

Schema diff:

```
[[entries]]
id = "<string>"
path = "<string>"
class = "<string>"                       # unchanged
encoder = "<string>"
source_fixture = "<string>"
generation_command = "<string>"
license = "<string>"
container_support = "<string>"
required = <bool>
max_size_bytes = <u64>
oracle = "<string>"
expected_sha256 = "<lowercase-hex-64>"   # NEW, optional
```

Parser changes (`full-test/src/corpus.rs:867-908`):

- Add `expected_sha256: Option<String>` to `EntryBuilder` and `ManifestEntry`.
- `EntryBuilder::set` learns the `expected_sha256` key. Validate the value: must be a quoted lowercase 64-hex-char string; reject otherwise with a line-numbered error.
- `EntryBuilder::finish` does not require the key; it remains `Option<String>`.

New entries to add (locked SHA256 values to be computed at first generation by the implementation agent — do **not** invent digests in this HLD):

1. `ffmpeg-native-sine-24k-mono` — exercises the low-end rate path. Source: `tests/vectors/48k_sine1k_loud.wav`. Generation: `ffmpeg -y -hide_banner -loglevel error -i tests/vectors/48k_sine1k_loud.wav -c:a opus -strict -2 -b:a 24k -ac 1 tests/vectors/real_world/ffmpeg-native-sine-24k-mono.opus`. `class = "generated-non-reference"`, `container_support = "supported-ogg-family0"`, `required = true`, `max_size_bytes = 60000`, `oracle = "exact-pcm-parity-vs-c-reference"`, `expected_sha256 = "<TO BE LOCKED AT FIRST GENERATION>"`.

2. `ogg-edge-preskip-or-output-gain` — one Ogg edge fixture with either a non-zero pre-skip OR a non-zero output_gain in `OpusHead`. Pinned with `expected_sha256`. Per supervisor decision 1, generate from a checked-in WAV via a documented recipe (path a). Likely recipe: `opusenc --comp 10 --bitrate 16` on a short crop of `tests/vectors/speech_48k_mono.wav` (or equivalent existing checked-in WAV) where the encoder's lookahead naturally yields a non-trivial pre-skip; OR a recipe that explicitly applies an output_gain via `opus-tools opusgain`. Implementation agent picks the simpler reproducible variant during implementation. `class = "generated-non-reference"`, `container_support = "supported-ogg-family0"`, `required = true`, `max_size_bytes = 80000`, `oracle = "exact-pcm-parity-vs-c-reference"`, `expected_sha256` locked at first generation.

   Note: `corpus_diff` does not currently *assert* anything about pre-skip or output_gain (they do not affect the per-packet PCM parity oracle — see `harness/src/bin_inner/corpus_diff.rs:166-179`). The point of pinning this fixture is to lock the bitstream that *carries* a non-default header field, so a future encoder change that flips those header fields silently is detectable via the SHA256 mismatch path.

The existing `ffmpeg-native-sine-32k` entry is left unchanged — no `expected_sha256` — because it is the historical "smoke" entry. Keeping it unpinned exercises the unpinned-but-runs path documented in Change 2e.

After this change the manifest contains exactly three required entries (one unpinned smoke + two pinned). No more.

## Risks and Mitigations

### R1. FFmpeg encoder non-determinism across versions

Risk: `expected_sha256` is sensitive to FFmpeg's native opus encoder output, which is gated by `-strict -2` and is allowed to change across FFmpeg releases. A maintainer with a different FFmpeg version will see a hard-fail SHA256 mismatch despite the decoder being correct.

Mitigation: The failure message under Change 2d/2e includes the FFmpeg version line, so the cause is immediate from the report. Document in the manifest comment block (and in `tests/vectors/real_world/README.md`) the exact `ffmpeg -version` first-line under which the digests were locked. The implementation agent must record that string verbatim alongside the digests. When a mismatch occurs, the maintainer either matches the FFmpeg version or re-locks both digests and updates the README's recorded version line — re-locking is a manifest-only change with Arthur signoff. We do not weaken the contract to a fuzzy comparison; pinned means pinned.

### R2. Ogg edge fixture provenance

Risk: The new `ogg-edge-preskip-or-output-gain` entry needs a fixture whose `OpusHead` carries non-default fields. If the implementation agent commits a third-party `.opus`, redistribution license must be vetted; if they generate it, the encoder pipeline must be reproducible.

Mitigation: Resolved by supervisor decision 1 — path (a) generate from existing checked-in WAV. SOURCES.md remains an empty template for this slice. Re-locking the digest is the only allowed path if the recipe needs to change later.

### R3. Ogg edge fixture flakiness under decoder changes

Risk: Even with the bitstream pinned via SHA256, future ropus changes might cause `corpus_diff` to drift its decoded PCM for a fixture that exercises a less-common header path. Locking the file to disk does not lock the decoder. A green slice today could become a permanent red blocker after an unrelated decoder refactor that legitimately fixes a bug in pre-skip or output_gain handling.

Mitigation: The corpus oracle is "ropus PCM equals C-reference PCM on the same input" (see `corpus.rs:19`). It is *relative*, not absolute. A decoder change that breaks parity is the gate doing its job; a decoder change that *fixes* both decoders identically continues to pass. The risk here is specifically a divergence between the two decoders triggered by an unusual header. That is exactly the signal we want — log the failure with the entry id and packet index (already supported by `corpus_diff`'s `Mismatch` reporting at `harness/src/bin_inner/corpus_diff.rs:140-147`) and let the maintainer decide whether to fix ropus or to re-pin against an updated reference. Do not preemptively weaken the oracle.

### R4. Manifest count contract is wider than the test set

Risk: An overzealous future contributor adds a fourth required entry without HLD signoff and quietly expands what "corpus claim" means.

Mitigation: The acceptance criteria explicitly cap the manifest at three required entries for this slice. Any fourth entry must come with its own HLD update. Code-side, the `corpus.rs` test `manifest_parser_loads_required_generated_entry` (`full-test/src/corpus.rs:978-986`) should be replaced/augmented with a test that asserts the *exact* set of required ids — making accidental additions visible.
