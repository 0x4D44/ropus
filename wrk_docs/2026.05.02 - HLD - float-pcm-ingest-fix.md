# Float-PCM ingest divergence fix

Author: Claude (lead engineer, 1M context)
Date: 2026-05-02
Status: HLD — pre-implementation. Awaiting supervisor sign-off.

## 1. Problem statement

The 24-hour fuzz campaign (2026-05-01 → 2026-05-02) surfaced two
medium-severity findings clustered on the float-PCM input path that
HLD V2 gap 11 added on 2026-05-01:

1. `tests/fuzz/known_failures/encode-float-lowdelay-8k-divergence/`
   (2 repros). Single-frame `encode_float` at
   `sr=8000, ch=1, application=RESTRICTED_LOWDELAY (2051),
   bitrate=65760, complexity=3`.
   - `crash-00.bin` (CBR, vbr=0, fec=0): Rust=164B, C=164B, **bytes
     differ**.
   - `crash-01-vbr-len-mismatch.bin` (VBR, vbr=1, fec=0): Rust=308B
     vs C=123B, **length divergence — Rust packs >2× the bytes**.
2. `tests/fuzz/known_failures/roundtrip-float-12k-cx7-vbr-divergence/`
   (2 repros). Single-frame `encode_float` at
   `sr=12000, ch=2, application=AUDIO (2049), bitrate=71281,
   complexity=7, vbr=true`.
   - Rust=289B vs C=179B, length divergence. The first ~13 bytes
     match (TOC + early VBR sub-frame size fields), then streams
     diverge.

The i16-PCM ingest path (`OpusEncoder::encode`) is byte-exact with the
C reference and has been since Stream A landed. Both findings only
fire when `encode_float` is the entry point. The campaign journal's
working hypothesis is therefore a shared root cause inside the
float-PCM ingest pipeline.

### Code I inspected

- `/home/md/language/ropus/ropus/src/types.rs:418-432`
  — `float2int(x)` and `float2int16(x)`. The Rust code uses
  `x.round_ties_even() as i32` after multiplying by `CELT_SIG_SCALE
  = 32768` and saturating to `[-32768, 32767]` in float, then
  truncating to `i16`.
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1540-1572`
  — `OpusEncoder::encode_float`: converts the f32 input via
  `float2int16` into a heap `Vec<i16>`, then calls
  `encode_native_with_analysis(&pcm16, frame_size, data,
  max_data_bytes, **24**, Some((pcm_bytes, analysis_frame_size,
  downmix_float)))`.
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1508-1535`
  — `OpusEncoder::encode` (i16 path): same call but with
  `lsb_depth=**16**` and `downmix_int`.
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1596-1621`
  — `encode_native_with_analysis`: `let lsb_depth =
  imin(lsb_depth, self.lsb_depth);` (line 1616). With the float
  caller passing 24 and `self.lsb_depth = 24` (default,
  `ropus/src/opus/encoder.rs:1285`), the effective `lsb_depth` is
  **24** for the float entry, **16** for the i16 entry.
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1631-1635`
  — analysis gate: `complexity >= 10 && fs >= 16000 && fs <= 48000
  && app != RESTRICTED_SILK`. **Both repros are below this gate**
  (cx=3 sr=8k, cx=7 sr=12k).
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:2146-2151`
  — propagation to CELT: `celt.ctl(CeltEncoderCtl::SetLsbDepth(
  lsb_depth));`.
- `/home/md/language/ropus/ropus/src/celt/encoder.rs:1378-1387`
  — `lsb_depth` enters CELT's `dynalloc_analysis` directly through
  the noise-floor formula:
  `noise_floor[i] = gconst(0.0625)*log_n[i] + gconst(0.5)
  + shl32(9 - lsb_depth, DB_SHIFT) - shl32(EMEANS[i], DB_SHIFT-4)
  + gconst(0.0062)*(i+5)*(i+5)`.
- `/home/md/language/ropus/reference/celt/arch.h:163-177` — under
  `FIXED_POINT && !ENABLE_RES24`, `MAX_ENCODING_DEPTH = 16`,
  `FLOAT2RES(a) = FLOAT2INT16(a)`, `opus_res = opus_int16`.
- `/home/md/language/ropus/harness/config.h:7-8` — confirms the C
  reference is built with `FIXED_POINT 1` and **without**
  `ENABLE_RES24`.
- `/home/md/language/ropus/reference/src/opus_encoder.c:2743-2766`
  — C `opus_encode_float` in the `FIXED_POINT, !ENABLE_RES24` branch:
  fills `in[i] = FLOAT2RES(pcm[i])` (i.e. `FLOAT2INT16`) into an
  `opus_res` (= `opus_int16`) buffer, then calls
  `opus_encode_native(st, in, frame_size, data, max_data_bytes,
  **MAX_ENCODING_DEPTH = 16**, ...)`.
- `/home/md/language/ropus/reference/src/opus_encoder.c:2667` — C
  `opus_encode` (i16) also passes `lsb_depth = 16`. Both i16 and
  float paths converge on lsb_depth=16 in the C build the harness
  links against.
- `/home/md/language/ropus/reference/celt/float_cast.h:64-156` — C
  `float2int(x)` on x86_64 GCC uses `_mm_cvt_ss2si`, which honours
  the SSE rounding mode (default: round-to-nearest-even).
  `FLOAT2INT16(x) = (opus_int16) float2int( MIN32(MAX32(x*32768,
  -32768), 32767) )`.
- `/home/md/language/ropus/reference/src/opus_encoder.c:1060-1077`
  — `is_digital_silence`: under `FIXED_POINT`, the `lsb_depth`
  parameter is explicitly cast to void; only the `!FIXED_POINT`
  branch uses it for the silence threshold. So the lsb_depth
  divergence does **not** propagate through `is_digital_silence`
  on the harness build — only through CELT.

### The unifying hypothesis

Rust's `encode_float` passes `lsb_depth = 24` (matching what the C
**float-build** does, where `MAX_ENCODING_DEPTH = 24`). But the C
reference under `FIXED_POINT && !ENABLE_RES24` — which is the build
the harness links — passes `lsb_depth = 16`. Rust effectively gives
CELT 8 dB more headroom in the noise-floor formula than the C
reference does, which shifts dynalloc decisions and bit allocation,
producing different bytes (and under VBR, different lengths).

This single line difference accounts for both findings:
- CBR same-length-different-bytes: bit allocation shifts within a
  fixed byte budget.
- VBR length divergence: the rate controller is free to spend more
  bytes when it sees a deeper effective dynamic range, so it does.

The same-bitrate-but-Rust-bigger pattern (308 vs 123, 289 vs 179)
is exactly what a "Rust thinks it has more signal information to
preserve" bit-allocation skew predicts.

## 2. Why it isn't the analysis-divergence class

Campaign 9 (2026-04-19) flagged a divergence class at
`complexity ≥ 10 ∧ sr ≥ 16000 ∧ application != RESTRICTED_SILK`,
caused by the missing analysis port (Stage 6).

- **8k LOWDELAY repros**: `complexity=3` (< 10) **and** `sr=8000`
  (< 16000). Both gates fail; analysis is skipped. The Rust gate
  (`ropus/src/opus/encoder.rs:1631-1635`) and the C gate
  (`reference/src/opus_encoder.c:1250` for FIXED_POINT) both
  evaluate false.
- **12k cx7 AUDIO repros**: `complexity=7` (< 10) **and** `sr=12000`
  (< 16000). Same reasoning. Analysis is skipped on both sides.
  (Note: `complexity ≥ 7` is the float-build gate at C line 1252,
  but the harness compiles with FIXED_POINT, so the active gate is
  `>= 10` per line 1250, and `tonality_analysis_reset` is called
  if `analysis.initialized`. The Rust mirror at line 1659-1661
  matches.)

Both repros are firmly below both gates. The campaign journal made
the same call (line 30-35 of
`tests/fuzz/known_failures/roundtrip-float-12k-cx7-vbr-divergence/README.md`).

## 3. Root-cause investigation plan

The hypothesis is concrete enough to be confirmed or refuted with a
single instrumented run. Plan: instrument the harness only (the
production code stays read-only until the root cause is confirmed),
dump the four most informative values, and compare side by side.

### Instrumentation (harness-only)

Add a debug binary at
`/home/md/language/ropus/harness/src/bin/trace_encode_float.rs`
that:

1. Loads a libFuzzer-format crash file from
   `tests/fuzz/known_failures/encode-float-lowdelay-8k-divergence/crash-00.bin`.
2. Reproduces the exact `(sample_rate, channels, application, bitrate,
   complexity, vbr, frame_size, pcm_f32)` tuple the fuzz harness
   would feed in. Reuses the existing decoding helpers in
   `tests/fuzz/fuzz_targets/c_reference.rs`.
3. Drives both Rust `OpusEncoder::encode_float` and the C
   `opus_encode_float` once each, dumping the following at
   matching points:

   | Point | Rust source | C source | Value |
   |---|---|---|---|
   | A | `encode_float` line 1556 | `opus_encoder.c:2761` | `pcm16[0..16]` after `float2int16` / `FLOAT2RES` |
   | B | `encode_native_with_analysis` line 1616 | `opus_encoder.c:1242` | `lsb_depth` after the `imin` clamp |
   | C | `encode_native_with_analysis` line 1617 | `opus_encoder.c:1246` | `is_silence` |
   | D | `encode_native_with_analysis` line 2149 | `opus_encoder.c:1678` | `lsb_depth` argument to `OPUS_SET_LSB_DEPTH` |
   | E | (CELT) celt/encoder.rs:1384 | `celt_encoder.c:1074` | `noise_floor[0..nbEBands]` |
   | F | encode return | encode return | byte-by-byte packet diff |

   Points A and C disambiguate "is this float→i16 conversion?" from
   "is this lsb_depth?". Point B is the smoking-gun delta if the
   hypothesis holds. Points D and E confirm the propagation into
   CELT. Point F gives the final answer.

   Each point must be exposed via a thin debug-only hook. The Rust
   side already has `pub(crate)` boundaries on `encode_native_with_analysis`
   so an `eprintln!`-equivalent shim wired through a `#[cfg(feature
   = "trace-encode")]` is sufficient. **No production code changes
   from this**; the trace points are gated behind a Cargo feature
   that lives in the harness crate's manifest only.

   The C side already has a working FFI through `harness/build.rs`.
   Add a `harness/debug_helper.c` entry that wraps
   `opus_encode_native` with a small instrumented prologue calling
   `fprintf(stderr, ...)` immediately before
   `OPUS_SET_LSB_DEPTH(lsb_depth)`. This is the exact pattern
   `harness/debug_silk_trace.c` already uses for SILK.

### Expected outcome

If the hypothesis is correct, point B will show `lsb_depth=24` on
the Rust side and `lsb_depth=16` on the C side. Points A, C are
expected identical. Point E will show all `noise_floor[i]` values
shifted by `(24-16) << DB_SHIFT = 8 << 24 = 2^27` (specifically:
Rust's `noise_floor` will be `8 << 24` lower than C's, since the C
formula is `+ SHL32(9 - lsb_depth, DB_SHIFT)` and Rust passes a
larger `lsb_depth` → smaller term → lower `noise_floor`).

If point A diverges (i.e. `pcm16` already differs), the hypothesis
is wrong and we need to dig into `float2int16` rounding. If point
A matches but B matches too, the divergence is something else
entirely (analysis hook? `compute_stereo_width`? — see the
hypothesis matrix below).

### Time budget

Instrumentation: ~2-3 hours (the harness already has the FFI
boilerplate). Trace run: ~10 minutes. Confirmation: same day.

## 4. Hypothesis matrix

| # | Candidate divergent point | Prior | Reasoning |
|---|---|---|---|
| H1 | **`lsb_depth = 24` passed by `encode_float` vs `16` by C** | **HIGH** | Confirmed the constants on both sides at the cited lines. C `MAX_ENCODING_DEPTH` is 16 in the harness build (`FIXED_POINT && !ENABLE_RES24`). Rust passes 24 unconditionally. CELT's noise_floor formula uses `lsb_depth` directly, so an 8-step delta produces 8 dB of dynalloc skew — easily large enough to redirect bit allocation across all sub-frames. Cleanly explains both the 8k CBR same-length-different-bytes case (re-allocation within a fixed budget) and the 12k VBR length divergence (rate controller spending more bytes given the deeper apparent depth). Sub-frames after the first ~13 bytes diverge in the 12k repro because that's where rate-allocation per band starts being applied, consistent with the noise_floor → dynalloc → rate path. |
| H2 | `float2int16` rounding mode (round-half-even vs C99 lrintf) | LOW | Rust uses `round_ties_even` after explicit float-domain saturation. C uses `_mm_cvt_ss2si` (default SSE rounding = round-to-nearest-even) on x86_64 GCC, or `lrintf` (also round-to-nearest-even by default) elsewhere. Both implementations produce identical i16 for any finite input that isn't an exact-half between two integers, and both produce ties-to-even for the half cases. The harness builds with GCC on x86_64. **Same rounding mode.** Furthermore, point A in the trace plan disambiguates this. |
| H3 | `float2int16` saturation behaviour outside [-1.0, 1.0] | LOW | Rust saturates in float domain BEFORE `as i32`, then casts to `i16`. C does the same (`MAX32`/`MIN32` are float-domain on float input — they're macro-expanded `MAX(a,b) = ((a)>(b)?(a):(b))` per `arch.h`). Both produce `±32767`/`-32768` for out-of-range input. **Same.** The NaN-handling sub-divergence is a separate finding (`nan-encode-float`, deferred — see campaign journal 23:58 BST entry). |
| H4 | Float-domain pre-emphasis or HP filter coefficients | LOW | The Rust HP filter (`silk_biquad_res`, `dc_reject` callers) operates on the **i16-converted** buffer, not the original float. There is no float-domain pre-emphasis or HP path in `encode_float` — the conversion happens at line 1556 and everything downstream consumes the same i16 PCM as the i16 entry point. The fb2k / `ropusenc` clients also encode after conversion, so this is structurally impossible without re-introducing a float pipeline. |
| H5 | Resampler tap differences | LOW | The Rust `encode_float` does **no** resampling. It converts f32→i16, then calls the same `encode_native_with_analysis` the i16 entry uses. Any internal Opus resampling (sr → silk_internal_sr) happens **after** the conversion, on i16 samples — identical for both entry points. The C path is the same (FLOAT2RES once, then opus_encode_native processes opus_res). README's resampler hypothesis is structurally implausible. |
| H6 | Complexity-derived encoder-config fires only on float entry | LOW | None of the complexity-gated decisions in `encode_native_with_analysis` (analysis, run_analysis params, equiv_rate, mode threshold) read the entry point or the `lsb_depth`. They read `silk_mode.complexity` directly. With both fuzz repros below the analysis gate, they aren't triggered. |
| H7 | Float-domain `downmix_float` analysis hook | LOW (for these repros) | The hook is only invoked when `run_analysis` runs, gated by `complexity >= 10 && sr >= 16000 && app != RESTRICTED_SILK`. **Both repros fail this gate** so `downmix_float` never executes in these traces. (`downmix_float` would still need scrutiny when fixing the analysis-divergence class above the gate, but that's a separate HLD scope.) |
| H8 | `is_digital_silence` lsb_depth threshold | LOW | Under `FIXED_POINT`, C's `is_digital_silence` casts `lsb_depth` to void (`reference/src/opus_encoder.c:1071`). Rust's mirror at `ropus/src/opus/encoder.rs:629` also ignores `_lsb_depth`. So even though the lsb_depth value is wrong on the Rust side, this code path doesn't notice. The fix to H1 won't change the silence detection for these repros. |
| H9 | `compute_frame_energy` numerical drift | LOW | Operates on i16 only. No float path. Bit-exact since Stream A. |
| H10 | Default `st->lsb_depth = 24` mismatch | NEGATIVE | Both Rust (`encoder.rs:1285`) and C (`opus_encoder.c:308`) initialize `st->lsb_depth = 24`. The clamp `imin(lsb_depth, self.lsb_depth)` happens correctly on both sides; the bug is what the **caller** passes, not what the encoder state holds. |

**Net**: H1 is the dominant prior. The trace plan in §3 is designed
to confirm or refute H1 in a single run. If H1 is confirmed, the
fix is mechanical (§5). If H1 is refuted, the trace points A, C,
E narrow the next investigation step without further harness
work.

## 5. Fix plan (after H1 confirmation)

The fix is a single-character semantic change in
`/home/md/language/ropus/ropus/src/opus/encoder.rs:1569`:

```
            24,
```

becomes

```
            16,
```

Mirroring C's `opus_encode_float` (`reference/src/opus_encoder.c:2762`)
which passes `MAX_ENCODING_DEPTH` — and `MAX_ENCODING_DEPTH = 16`
under `FIXED_POINT && !ENABLE_RES24`, which is the harness/production
build configuration of the C reference.

Because of the bit-exact requirement, "approximately right" is not
an option; the fix must produce byte-identical compressed output.
A simple integer-literal change matches the C constant exactly.

### Considered alternatives

- **Compute `MAX_ENCODING_DEPTH` from a constant**: introduce
  `pub const MAX_ENCODING_DEPTH: i32 = 16;` in `types.rs` and
  reference it at the call site. This is more idiomatic and
  documents the intent. Recommended, but the fix can land as a
  literal first and the constant follow up — they're trivially
  equivalent.
- **Always pass `self.lsb_depth` and let the encoder clamp**: this
  would silently lose the C semantic that `opus_encode` and
  `opus_encode_float` cap the depth at MAX_ENCODING_DEPTH
  regardless of `st->lsb_depth`. C explicitly does the clamp at
  the entry point, not inside `opus_encode_native`. Rejected — it
  would mask further drift if a caller ever raises `st->lsb_depth`.

### Order of operations

1. Land trace instrumentation (harness-only) and confirm H1.
2. Single-line fix in `encode_float`.
3. Add a debug assertion in `encode_native_with_analysis` that
   `lsb_depth <= 16` if we want to harden against future
   regressions. (Optional; bit-exact tests are the primary
   safeguard.)
4. Optionally introduce `MAX_ENCODING_DEPTH` as a named constant.

### What this does NOT fix

- The four other "real codec bug" findings from the campaign
  (multistream-encode-bytes-divergence, multiframe-cbr-cross-frame-
  divergence, multiframe-vbr-fec-dtx-divergence, the analysis-
  divergence class above the cx≥10 ∧ sr≥16k gate). Those are
  separate scopes, separate HLDs, separate PRs.
- The `nan-encode-float` finding. That's a sanitisation issue
  upstream of the conversion, addressed in a different one-liner
  per the campaign journal's 23:58 BST entry.
- The roundtrip-12k-cx7 `decode_float` half. The decoder side is
  bit-exact (range coder + CELT decode are verified to byte-level
  parity); only the encode side drives the divergence.

## 7. Blast radius / rollback

### External callers of `encode_float`

- `capi/src/encoder.rs:253` — `opus_encode_float` (the canonical
  C ABI shim). All foobar2000 / external libopus consumers go
  through this.
- `ropus-tools-core/src/commands/encode.rs:275, 298` —
  `ropusenc` (the production CLI tool) drives `encode_float` for
  any WAV with float PCM source.
- `ropus-fb2k/` — foobar2000 component. Uses `ropus-tools-core`
  encode internally per the 2026.04.19 fb2k implementation
  journal; same code path.
- `tests/fuzz/fuzz_targets/fuzz_encode.rs:182` and
  `fuzz_roundtrip.rs:179` — fuzz drivers, the diagnostic source
  of the findings.

`opus_multistream_encode_float` in `capi/src/ms_encoder.rs:430` is
returning `OPUS_UNIMPLEMENTED` on `ropus`, so multistream-float is
not a path that needs adjustment for this fix.

### User-visible behaviour change

After the fix:
- Float-PCM encodes will produce **byte-identical** compressed
  output to the C reference at this exact bitrate / app / sr /
  complexity / VBR combination, restoring the bit-exact guarantee
  the project advertises.
- The compressed output **changes** — at the byte level — relative
  to what `ropus` currently produces from float input. The output
  is still a valid Opus packet, decodable by any compliant
  decoder, but it will not match what a previous `ropus` build
  produced from the same float WAV.
- Audio quality is **expected to improve at low bitrates**: the
  current Rust `lsb_depth=24` causes CELT to over-allocate bits to
  perceived "low" bands (the noise_floor is artificially deeper),
  potentially starving real signal. The fix matches what the C
  reference has done all along, which is what the public Opus
  test vectors are tuned against.
- No API change. The `encode_float` signature is unchanged.

CLAUDE.md explicitly says "we are in a development sandbox; skip
ops sign-offs, migration choreography, or backwards-compatibility
guarantees unless explicitly requested." The byte-level output
change is therefore acceptable per the project's working compact —
bit-exactness with C is the contract, byte stability across Rust
versions is not.

### Rollback

The fix is a 1-line change in a hot, non-load-bearing part of
`encode_float`. Reverting is a single `git revert`. No state
migration, no on-disk format change, no allocator pinning. The
fuzz seed regression files added in step (2) of the test plan
serve as a permanent guard.

### Risk

- LOW for the codec itself: the change brings Rust into alignment
  with the well-tested C reference at this exact lever.
- LOW for downstream consumers: `ropusenc`, fb2k, the capi shim
  all delegate to `OpusEncoder::encode_float`. None reads
  `lsb_depth` directly (besides the runtime setter `set_lsb_depth`
  which writes `st->lsb_depth`, untouched here).
- MEDIUM-but-bounded for the test harness: the new differential
  test added in step (5) of the test plan will fail on any
  re-introduction of this bug, providing an immediate signal.

## 8. Open questions for supervisor review

1. **Constant introduction**: should the fix also introduce a
   public `MAX_ENCODING_DEPTH` constant in
   `ropus/src/types.rs` and reference it at the call site, or is
   the bare-literal `16` sufficient? My preference: introduce
   the constant, doc-comment it with the C-reference cross-link,
   and use it. This is one extra line plus the call-site change
   versus a single literal change. Trade: clarity vs. minimum
   diff.

2. **Coverage of `MAX_ENCODING_DEPTH = 24` future**: if `ropus`
   ever supports a 24-bit-encoding mode (`ENABLE_RES24`
