## Findings

### Issue: Quick release-preflight inventory can disagree with the shell source of truth

References:
- `/home/md/language/ropus/full-test/src/fuzz.rs:303`
- `/home/md/language/ropus/full-test/src/fuzz.rs:311`
- `/home/md/language/ropus/full-test/src/fuzz.rs:333`
- `/home/md/language/ropus/full-test/src/fuzz.rs:376`
- `/home/md/language/ropus/full-test/src/fuzz.rs:387`
- `/home/md/language/ropus/tools/fuzz_run.sh:175`
- `/home/md/language/ropus/tools/fuzz_run.sh:296`

`full-test --quick --release-preflight` uses a separate Rust parser and crash-file counter instead of the shell discovery/replay semantics. That creates two observable mismatches with the HLD:

- The Rust manifest parser accepts any line starting with `[[bin]]`, and treats any key starting with `name` inside that section as the target name. A malformed header such as `[[bin]]bad` or a key such as `name_suffix = "fuzz_decode"` can be accepted by quick inventory even though the shell parser is stricter and the HLD says malformed manifests must fail quick release-preflight.
- Quick inventory only counts immediate `*.bin` children in crash directories. The shell sanity path recursively finds committed crash bins with `find ... -type f -name '*.bin'`. A nested committed `.bin` under an undeclared crash directory would be missed by quick release-preflight but fail under full sanity.

Impact: quick release-preflight can report PASS/WARN for an inventory state that the full bounded sanity gate rejects. That weakens the quick preflight contract and makes the quick/full policy split harder to trust.

Suggested fix: either invoke/reuse `tools/fuzz_targets.sh` for quick target discovery, or make the Rust parser match its exact grammar and add explicit tests for malformed `[[bin]]` headers and `name*` keys. Make crash-bin inventory recursive, or deliberately align both shell and Rust on non-recursive semantics. Add a quick-preflight test for nested undeclared `.bin` files.

## Confirmations

- Default `full-test` remains non-hostile: `fuzz::selected_mode` returns `not_requested` whenever `--release-preflight` is absent, and `run_with_root` returns without probing cargo-fuzz, nightly, or crash corpora.
- `full-test --quick --release-preflight` is inventory-only: it does not construct or run the shell sanity command, and it does not build targets or replay crashes. Tool absence is reported as WARN rather than FAIL, subject to the inventory mismatch noted above.
- Full `full-test --release-preflight` runs `timeout 300 bash tools/fuzz_run.sh --sanity`; any nonzero exit becomes `Status::Fail`, and `banner::classify` maps that to a FAIL banner/exit.
- `tools/fuzz_run.sh --sanity` is a no-campaign path: it skips findings directory creation, builds selected targets, replays committed crash bins, emits per-target summary lines, and exits before the main fuzzing loop.
- Missing cargo-fuzz, missing nightly, target build failure, undeclared crash bins, and replay failure all exit nonzero in the full shell sanity path.
