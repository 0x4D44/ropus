# HLD: Performance Release Thresholds

Date: 2026.05.06

Scope: next testing-improvement slice for performance release thresholds and
benchmark gate honesty in `/home/md/language/ropus`.

## Problem Statement

Performance is currently visible but not release-gated in a way that supports a
strong claim.

`full-test` runs a native benchmark sweep unless `--quick` or
`--skip-benchmarks` is used. It reports Rust/C encode and decode ratios, and the
banner turns WARN when a row crashes or a ratio exceeds the hard-coded 1.15x
warning threshold. WARN exits 0 by design.

That is acceptable for local visibility, but it is not a release threshold. A
non-quick `--release-preflight` run can still finish with exit 0 when benchmark
rows are missing, crashed, or materially slower than the measured baseline, as
long as the rest of the runner is green. Conversely, changing the fixed 1.15x
constant into a hard release failure would be flaky: it is absolute versus C,
not per-vector versus measured ropus baseline, and one README baseline vector
already sits at 1.14x encode.

The next slice should therefore make the performance claim explicit before
tightening it: quick/default/release profiles must say exactly whether they
claim performance coverage, and release thresholds must be derived from measured
baseline data rather than a guessed constant.

## Stakeholders

- Arthur: needs a release signal that is honest enough to trust while keeping
  local development tolerable.
- Codec maintainers: need regressions surfaced without being blocked by normal
  benchmark noise.
- Future CI/release automation: needs machine-readable PASS/FAIL/WARN
  semantics for performance, not just a human-readable table.
- Users of ropus: care that performance claims in README/release notes are not
  stale or cherry-picked.
- The testing system itself: should not turn noisy microbenchmarks into false
  failures.

## Proposed Small Testing Slice

Recommended next slice: **performance gate contract and measured threshold
baseline**, not vector expansion.

This is the right next slice. The benchmark gate already exists and the oracle
matrix already admits it is smoke-only. Before adding more vectors, make the
existing lane honest and automatable.

### Behavior

Introduce an explicit benchmark profile/claim model:

| Profile | When | Performance claim | Banner/exit behavior |
|---|---|---|---|
| `not-claimed` | `--quick`, or default with `--skip-benchmarks` | No benchmark coverage. | PASS/FAIL ignores benchmark stage; report states no performance claim. |
| `observed-warn-only` | default full-test benchmark run | Ratios were measured and reported for available fixtures. | Existing WARN behavior remains exit 0. |
| `release-thresholded` | non-quick `--release-preflight` without `--skip-benchmarks` | Required benchmark smoke plus severe-regression threshold gate for the canonical vectors. | Benchmark build failure, row crash, zero gateable vectors, missing required fixture, or threshold breach becomes FAIL. |

Key point: quick release-preflight must explicitly make no performance claim.
It is a core smoke profile, not a performance gate.

### Data Model

Add a small tracked threshold spec, either as TOML or a Rust const table. TOML is
preferable because it separates measured policy data from runner logic:

`/home/md/language/ropus/full-test/bench_thresholds.toml`

Proposed fields:

- schema/version.
- source commit and measurement date.
- measurement command.
- required target profile notes, including `x86-64-v3`.
- per-vector rows:
  - label.
  - relative fixture path.
  - bitrate.
  - encode baseline ratio.
  - decode baseline ratio.
  - encode release-fail ratio.
  - decode release-fail ratio.
  - optional observed min/median/max from the baseline sweep.

The runner should report the threshold source and per-row threshold status in
JSON and HTML.

### First Threshold Source

Do not hard-code release failure thresholds from the README table directly.
Use the README/journal data as prior evidence only.

Baseline collection should run at least 5 clean sweeps on the intended release
machine/profile:

```bash
timeout 120 cargo build --release --bin ropus-compare
timeout 7200 bash tools/bench_sweep.sh --iters=30
```

Repeat the sweep 5 times, recording ratios. The implementation slice may add a
small parser or helper for this, but it should not turn into a general
benchmark framework.

## Threshold Policy

Use same-run Rust/C ratios, not absolute milliseconds. Absolute timing is too
machine-sensitive; the ratio cancels some host load and clock effects because C
and Rust are measured in the same invocation.

Use per-vector thresholds, not one global ratio.

Suggested initial release-fail threshold per operation:

```text
release_fail_ratio = max(
    observed_max_ratio + 0.05,
    baseline_median_ratio * 1.20,
    baseline_median_ratio + 4 * observed_mad
)
```

Where `observed_mad` is the median absolute deviation across the baseline
sweeps. If a simple implementation wants to avoid MAD initially, use:

```text
release_fail_ratio = max(observed_max_ratio + 0.05, baseline_median_ratio * 1.20)
```

Rationale:

- The gate catches severe regressions without chasing normal microbenchmark
  variance.
- A row currently near 1.14x versus C does not fail merely because it is slower
  than C.
- A faster-than-C row is not forced to remain faster than C; it gets a relative
  guard band.
- Thresholds are reviewable data and can be recalibrated intentionally.

Keep the existing 1.15x visual WARN as a local/reporting hint only, or rename it
to avoid implying release significance. The release threshold should be the
tracked per-vector value.

Required release-thresholded failures:

- `ropus-compare` release build fails.
- `ropus-compare` binary is missing after build.
- Any required vector fixture is missing.
- Any required vector crashes or lacks complete encode/decode timings.
- No gateable vectors produce timings.
- Any finite encode/decode ratio exceeds its per-vector release-fail threshold.

Non-release/default behavior:

- Preserve WARN exit 0 for local runs.
- Preserve fixture skip tolerance outside a claimed release threshold profile.
- Keep `--skip-benchmarks` useful for local iteration.

Open policy call: if `--release-preflight --skip-benchmarks` is allowed, the
runner must either FAIL because the release performance claim was explicitly
disabled, or downgrade the profile to `not-claimed` with very loud report
wording. My recommendation is FAIL for non-quick release-preflight; otherwise a
release command can silently omit performance.

```bash
timeout 7200 cargo run --release -p full-test -- --release-preflight --skip-coverage
```

Expected: release-thresholded performance claim, required benchmark vectors
present, threshold status emitted in HTML/JSON, non-zero exit on synthetic
threshold breach.

Synthetic tests should cover:

- threshold exactly equal to limit does not fail.
- threshold greater than limit fails only in release-thresholded profile.
- missing fixture is green in observed-warn-only but fail in
  release-thresholded.
- `build_failed` and crashed rows are WARN in default but FAIL in
  release-thresholded.
- quick release-preflight emits no performance claim.
- non-quick release-preflight plus `--skip-benchmarks` follows the signed-off
  policy.

## Risks

- Benchmark noise can still be real: thermal throttling, laptop power plans,
  background load, and C-reference runtime dispatch can move ratios.
- A severe-only threshold may miss medium regressions. That is intentional for
  release gating; WARN remains the human review signal.
- The 10-vector set is narrow. This slice should not pretend to cover PLC,
  FEC, DTX, multistream/projection, or real-world container performance.
- `x86-64-v3` is part of the current performance story. Thresholds should state
  that profile; generic x86_64/platform thresholds belong to the platform slice.
- Recalibrating thresholds can become churn if it is too easy. Treat threshold
  data changes like code changes: review the measurement log.
- The full release command may be long. Keep unit tests around policy pure and
  use full sweeps sparingly.

## Non-Goals

- Do not optimize codec code in this slice.
- Do not expand the benchmark vector suite in this slice.
- Do not introduce a full benchmark database or history service.
- Do not make default local runs fail on performance.
- Do not claim performance parity with C; the gate claims "no severe regression
  against the measured ropus baseline for the listed vectors."
- Do not gate absolute wall-clock milliseconds.
- Do not solve platform/sanitizer coverage here.

## Design Decisions Needing Supervisor Signoff

1. Proceed with this slice now as **performance claim contract plus measured
   threshold baseline**, leaving vector expansion to a later slice.
2. Non-quick `--release-preflight` should use a `release-thresholded`
   performance profile when benchmarks are not skipped.
3. Quick `--release-preflight` should explicitly make no performance claim.
4. Non-quick `--release-preflight --skip-benchmarks` should FAIL rather than
   quietly downgrade the release performance claim.
5. Thresholds should be per-vector/per-operation data generated from fresh
   baseline sweeps, not a global hard-coded 1.15x.
6. Release thresholds should use same-run Rust/C ratios, not absolute
   milliseconds and not previous-report comparison.
7. Initial release-fail policy should use a severe-regression guard band such
   as `max(observed_max + 0.05, median * 1.20)` and optionally MAD when the
   implementation cost stays small.
8. The existing 10 vectors are sufficient for this slice's first claim, as long
   as the claim is labeled thresholded smoke rather than broad performance
   coverage.
