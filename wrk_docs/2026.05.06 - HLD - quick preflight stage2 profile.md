# Quick Preflight Stage 2 Profile HLD

Date: 2026.05.06

Scope: design only. No source implementation in this slice.

## Problem Statement

The current uncommitted quick-preflight patch appends `--skip extensions` when
`options.quick && options.release_preflight`. Dalton reviewed that patch as
source-correct, but Herschel's diagnosis showed the user-facing smoke command
still times out because Stage 2 remains a broad workspace test run.

The known slow or unsuitable buckets for a quick release-preflight smoke are:

- `conformance`, especially `extensions`, `encode`, and `decode`.
- `ropus-harness-control` and `ropus-harness-deep-plc`, which are slow together
  and currently fail in the deep PLC SNR check.

Adding more libtest substring skips would make the command more fragile without
changing the underlying problem. The quick smoke path needs a narrower Stage 2
package profile, not a full workspace run with ad hoc test-name carve-outs.

## Stakeholder Value

Arthur and other agents need a bounded smoke command that gives useful release
asset and ordinary workspace test signal without spending minutes in exhaustive
conformance or deep harness suites.

The value is not full certification. It is fast confidence that the workspace
still builds, ordinary tests still execute, the runner still reports correctly,
and the release-preflight path is not stuck on known long-running suites. Full
conformance and deep PLC validation remain valuable, but they belong outside the
quick smoke profile.

## Considered Options

1. Keep the existing `--skip extensions` patch.
   - Reject as the final Stage 2 design. It removes one slow conformance test
     binary, but Stage 2 still runs slow conformance `encode`/`decode` and the
     deep/control PLC harness packages.

2. Add more libtest `--skip` filters such as `decode`, `encode`, or `tier2`.
   - Reject. `libtest --skip` is a global substring filter across all scheduled
     binaries. These names are too broad and would accidentally hide unrelated
     fast tests. It also leaves the command conceptually wrong: a full workspace
     run with exclusions by test name is not a clean smoke profile.

3. For `--quick --release-preflight`, use a package-level Stage 2 profile:

   ```text
   cargo test --workspace --exclude conformance --exclude ropus-harness-control --exclude ropus-harness-deep-plc --lib --tests --no-fail-fast -- --test-threads=1
   ```

   - Recommended. This excludes the known slow/unsuitable packages at Cargo's
     package-selection layer and keeps useful signal from the rest of the
     workspace.

4. Build an explicit allowlist of packages for quick-preflight Stage 2.
   - Defer. It is more exact, but it requires maintaining a list whenever
     workspace membership changes. The package-exclude profile is smaller and
     adequate for this implementation slice.

5. Build a sub-minute smoke list of a few named tests.
   - Defer. Herschel's measured profile still includes about 91 seconds for
     `ropus --lib`, so the recommended profile targets bounded-under-300s, not
     sub-minute execution. If Arthur wants sub-minute smoke, that is a different
     design with explicit named probes.

## Recommended Design

Replace the current quick-preflight `--skip extensions` policy with a
quick-preflight Stage 2 package profile.

The policy trigger remains internal:

```rust
options.quick && options.release_preflight
```

When false, keep the existing Stage 2 command shape unchanged.

When true and coverage is skipped, Stage 2 should render and execute:

```text
cargo test --workspace --exclude conformance --exclude ropus-harness-control --exclude ropus-harness-deep-plc --lib --tests --no-fail-fast -- --test-threads=1
```

If IETF vectors are unavailable, preserve the existing IETF-vector behavior by
appending the existing libtest skip:

```text
--skip testvector
```

That skip is independent of quick-preflight package selection. It handles a
missing fixture set and is paired with the existing synthetic failure so the
gate cannot pass green without required spec vectors.

### Existing Partial Patch Decision

Replace the existing `--skip extensions` partial patch with package-level
excludes for quick-preflight Stage 2.

Reasoning:

- Once `conformance` is excluded, `--skip extensions` is redundant for the
  target slow test.
- Keeping both would make the command harder to explain and would retain a
  global substring skip that can hide future unrelated tests named
  `extensions`.
- Package excludes express the actual policy: quick-preflight does not run the
  conformance package or deep/control PLC harness packages.
- The non-quick path must continue to run the full workspace Stage 2 command,
  including conformance and deep/control PLC packages.

Implementation should remove or repurpose the current `skip_slow_conformance`
helper/tests into a clearer `quick_preflight_stage2_profile` or equivalent
policy helper. It should remove command/header docs and CLI help language that
say quick-preflight filters only the slow extensions stress test.

### Coverage Decision

The current user smoke passes `--skip-coverage`; that is the priority path for
this implementation slice.

The design should also support coverage-enabled quick-preflight if feasible.
For coverage enabled, use the same package selection before the libtest
separator:

```text
cargo llvm-cov --json --ignore-run-fail --output-path <tmp>.json test --workspace --exclude conformance --exclude ropus-harness-control --exclude ropus-harness-deep-plc --lib --tests -- --test-threads=1
```

Do not block the slice on proving coverage-enabled runtime under 300 seconds.
The command builder should honor the chosen package profile for both
`skip_coverage=true` and `skip_coverage=false`; validation can prioritize the
skip-coverage smoke path and run a bounded coverage-enabled probe if practical.

### CLI Help

Yes, CLI help needs another revision beyond the current partial patch.

Replace the current help language:

```text
with --release-preflight, also filters the slow extensions stress test
```

with language that describes the package profile without over-promising exact
runtime, for example:

```text
--quick             Skip stages 1 and 4; with --release-preflight, use a narrower Stage 2 smoke profile.
```

No new public CLI flag is needed.


timeout:


If coverage-enabled runtime exceeds the timeout but the Stage 2 header shows the
correct package profile, report that as a runtime limitation rather than a
command-shape failure. The current acceptance priority is the explicit
`--skip-coverage` smoke.

## Risks And Tradeoffs

- The quick-preflight profile excludes all of `conformance`, not only slow
  conformance tests. This loses conformance signal in the smoke path by design.
  Full validation must remain the place for that signal.

- The profile excludes both PLC harness packages, including any fast tests they
  may contain. This is acceptable because one of the packages currently has a
  failing deep PLC threshold and both are outside the intended quick smoke
  value.

- The profile still includes `ropus --lib`, measured at about 91 seconds
  serially. The target is bounded smoke under the existing 300 second budget,
  not a sub-minute smoke.

- Package excludes are less brittle than libtest substring skips, but they are
  still policy. If new slow packages appear, this profile may need revision.

- Coverage-enabled quick-preflight may still be slower than the smoke budget.
  The implementation should honor the args for coverage-enabled mode, but
  acceptance should focus on the requested `--skip-coverage` command unless
  Arthur asks for coverage-enabled smoke to be equally bounded.

## Rollback Plan

Rollback is straightforward because the change is isolated to Stage 2 command
construction and help text.

1. Revert the quick-preflight Stage 2 package-profile policy and restore the
   previous full workspace Stage 2 command for all modes.
2. Restore the prior `--quick` help text if needed.
3. Keep the existing IETF-vector fallback behavior unchanged throughout.

Do not roll back by reintroducing broader libtest substring skips. If this
profile is rejected, choose either the old full workspace behavior or a new
explicit smoke allowlist design.
