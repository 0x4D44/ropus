## Why 98% Branch Coverage Is Not Attainable From Test Additions Alone

The remaining 619 uncovered branches fall into four categories that cannot be reached through the public API without changes to product code:

### 1. Defensive bounds checks in inner loops (~180 branches)

Examples:
- `celt/mdct.rs` L104, 132, 140 — `if idx < input_len { … } else { 0 }` slice-bounds guards. The else arm is only taken if the caller passes an undersized buffer, which is unsafe by construction for the documented API.
- `celt/bands.rs` L731/740/1136/1286/1543 — `N > 1` / `B > 1` band-sanity checks that cannot fire for legal mode configurations.
- `silk/decoder.rs` L705/717/975/1003/1029/1289/1302/1316/1324 — `celt_assert` / `debug_assert` equivalents.

These exist to harden the Rust port against corrupted state but are invariants under the codec's public contract. Covering them would require constructing deliberately malformed internal state — which the current plan explicitly forbids (no new `pub(crate)` in product code).

### 2. Short-circuit OR sub-branches (~130 branches)

Compound conditions like `if a || b || c { … }` instrument every sub-expression. If `a` is always true, the `b` and `c` sides are never evaluated. Examples:
- `silk/encoder.rs` L1734, L1782-1827, L2521, L3080 — inner VAD / autocorr / LTP preconditions.
- `opus/encoder.rs` L2112-2133 — multistream rate-adjustment branches.
- `celt/bands.rs` L749/751/1735/1737 — `compute_theta` short-circuit sides.

Many of these require very specific entropy-coder state or intermediate-value patterns that a reasonable-runtime test can't reliably force.

### 3. Mode-gated paths behind feature flags / unused options (~90 branches)

- `dnn/*.rs` branches behind `#[cfg(feature = "dnn")]` or code paths only reached via the `dnn` feature. The default test run does not enable it (and enabling it turns up two unrelated failing tests beyond the coverage-campaign scope).
- `opus/encoder.rs` L1160 — the `OPUS_APPLICATION_VOIP` OR-arm in `ms_set_application` is only reached via multistream callers that our tests drive with `RESTRICTED_LOWDELAY`.
- Several `kiss99` short-cycle paths and `silk_burg_analysis` saturation paths that require adversarial float-signal search beyond brute-force budget.

### 4. Assertion macros in test modules (~40 branches)

`assert!(a || (b && c))` patterns inside existing unit tests compile to branches. The branch instrumentation will mark the uncovered short-circuit side of any passing assertion as missed. These are test scaffolding — covering them would mean intentionally producing failing assertions.

### Unreachable total

Combining the four categories yields ≈440 branches that are not reachable from the public API under the coverage plan's "test-additions only" constraint. That leaves ≈180 branches in the "reachable but not yet covered" bucket, which would move the headline to roughly **92.7%** if completely closed.

## Recommendations to Reach 98% Branch Coverage

These are out of scope for the present campaign but list the options the user can choose from:

1. **Expose internal helpers for direct test.** Add `#[cfg(test)] pub(crate)` on a focused set of internal functions (VAD, autocorr, NSQ trellis, entropy-coder state builders) so tests can drive them with crafted state. Estimated closure: ~150 branches.

2. **Remove defensive branches where Rust type/slice safety already guarantees the invariant.** Many `if idx < input_len { uc!(...) } else { 0 }` guards can be replaced by direct slice indexing when the caller is documented to pass a correctly-sized buffer (the port uses `uc!` which is already unchecked in release builds). Estimated closure: ~120 branches.

3. **Enable and test the `dnn` feature cleanly.** Resolve the 2 failing DNN tests and run coverage with `--features dnn`. Estimated closure: ~50 branches.

4. **Delete dead `#[allow(dead_code)]` branches.** Constants like `OPUS_APPLICATION_RESTRICTED_SILK` (2052) are never reachable from the public API; the branches that guard against them are pure defensive noise. Estimated closure: ~30 branches.

5. **Replace compound `assert!(a || b)` patterns in existing tests** with separate `assert!` statements, or use `assert!(a || b, …)` with both cases exercised by separate test instances. Estimated closure: ~40 branches.

Combined, these would move branch coverage to roughly **98.3%**.
