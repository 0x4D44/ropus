# Testing Action Plan: mdopus Codec

Date: 2026-04-08

## Objective

Turn the current test toolbox into a disciplined verification program without bloating the workflow.

The goal is not "more tests everywhere". The goal is:

- protect the highest-risk codec behavior
- make important checks easy to run and hard to skip
- improve confidence in branches, robustness, and performance

## Guiding Principle

Keep the current strengths:

- fast `cargo test`
- C-reference differential harness
- deterministic fixture corpus

Do not replace those with a slower, noisier process. Build on them.

## Priority Order

1. Institutionalize the existing strongest checks
2. Close branch-heavy coverage gaps in top-level codec logic
3. Operationalize fuzzing
4. Make stress and performance checks repeatable
5. Add mutation testing only after the above is stable

## Phase 1: Make The Current Backbone Explicit

### 1. Create a single default validation path

Problem:

- The repo has strong commands, but no single obvious "run this and trust the result" path.

Action:

- Define a documented default validation sequence:
  - `cargo test`
  - `python tools/integrate.py test`
  - `cargo +nightly llvm-cov --branch --summary-only`

Deliverable:

- Add one repo-level script or documented command sequence that runs the default suite end to end.

Success signal:

- A developer can answer "what should I run before I trust a codec change?" with one command or one short sequence.

### 2. Promote the integration matrix to first-class status

Problem:

- `tools/integrate.py` is valuable, but it currently feels like a side tool rather than the core regression gate.

Action:

- Treat the 35-fixture x 4-bitrate encode+roundtrip matrix as the main regression job.
- Make sure the current result file is regenerated deliberately, not left stale.

Deliverable:

- A documented policy for when `tools/integrate.py test` must pass.

Success signal:

- The 280-case matrix becomes part of the normal change-validation path.

### 3. Clean warning noise in the validation path

Problem:

- Current test and harness runs emit persistent warnings, especially in `src/opus/multistream.rs` and `tests/harness/main.rs`.

Action:

- Remove the current unused imports, unused variables, and similar warning noise in the validation-critical files.

Success signal:

- `cargo test` and harness commands run without warning spam.

## Phase 2: Improve Coverage Where It Matters

### 4. Target the branch-heavy modules first

Problem:

- Remaining risk is concentrated in large control-flow files, not helpers.

Priority modules:

1. `C:\language\mdopus\src\dnn\lpcnet.rs`
2. `C:\language\mdopus\src\opus\encoder.rs`
3. `C:\language\mdopus\src\opus\repacketizer.rs`
4. `C:\language\mdopus\src\opus\multistream.rs`
5. `C:\language\mdopus\src\silk\encoder.rs`
6. `C:\language\mdopus\src\opus\decoder.rs`
7. `C:\language\mdopus\src\silk\decoder.rs`
8. `C:\language\mdopus\src\types.rs`

Action:

- Add tests that target branch behavior, not only happy-path outcomes.

Focus areas:

- mode transitions
- invalid and boundary CTL inputs
- packet layout errors
- PLC/FEC branch interactions
- multistream mapping/layout permutations
- DNN/PLC state transitions
- low-level numeric edge cases in shared helpers

Success signal:

- Branch coverage rises meaningfully in the worst files, not just total line coverage.

### 5. Add branch-driven coverage goals

Problem:

- Total line coverage is already respectable, so chasing only line percentage will waste effort.

Action:

- Track branch coverage as the primary coverage metric for new testing work.

Near-term target:

- Hold overall line coverage above 88%
- Raise overall branch coverage from about 73% to 78%+

Success signal:

- Coverage work reduces risk in control-flow-heavy files instead of padding helper modules.

## Phase 3: Operationalize Fuzzing

### 6. Add a checked-in fuzz seed corpus

Problem:

- Fuzz targets exist, but there is no visible seed corpus or crash corpus in `tests/fuzz`.

Action:

- Add small seed inputs for:
  - valid packets
  - truncated packets
  - PLC/FEC-relevant packets
  - repacketizer edge cases
  - multistream packet shapes

Success signal:

- Fuzzing has a repeatable starting point instead of cold-start random exploration only.

### 7. Strengthen fuzz invariants

Problem:

- Current fuzz targets are mostly crash-oriented and heavily centered on 48 kHz mono.

Action:

- Extend fuzz targets to cover:
  - more sample rates
  - mono and stereo
  - multistream packet paths where practical
  - differential checks against the C reference for selected targets
  - stronger invariants than "did not panic"

Candidate invariants:

- valid decode never overreads or panics
- repacketizer preserves packet validity rules
- packet parsing stays consistent across helper APIs
- encode+decode returns expected frame sizes when encoding succeeds

Success signal:

- Fuzzing starts finding semantic bugs, not only panic bugs.

### 8. Schedule fuzzing as an automated lane

Problem:

- Fuzzing that only exists locally will not stay exercised.

Action:

- Add a scheduled fuzz job outside the fast default path.
- Keep it time-boxed rather than trying to run endless fuzzing per change.

Suggested shape:

- nightly or scheduled fuzz run
- crash artifacts stored and reduced
- every fixed crash becomes a normal regression test

Success signal:

- Fuzzing produces a living regression corpus, not just a crate layout.

## Phase 4: Make Stress And Performance Real Gates

### 9. Upgrade soak testing from smoke to endurance

Problem:

- `longsoak` exists, but current use is short. The bundled full suite runs only a 5-second soak.

Action:

- Keep the short soak in fast validation.
- Add longer scheduled soak profiles, for example:
  - 5 seconds in fast suite
  - 5 minutes scheduled
  - 30+ minutes occasional deep run

Success signal:

- State accumulation bugs are more likely to show up before release rather than during debugging.

### 10. Turn `bench` into a tracked benchmark

Problem:

- Current performance numbers are useful but ephemeral.

Action:

- Standardize one benchmark matrix, for example:
  - mono 48 kHz sine
  - mono 48 kHz noise
  - stereo 48 kHz noise
  - at least one low-bitrate and one mid/high-bitrate case

- Record:
  - C encode/decode ms
  - Rust encode/decode ms
  - Rust/C slowdown factor

Success signal:

- Performance regressions become visible in trend data rather than anecdotes.

### 11. Set pragmatic performance thresholds

Problem:

- Without thresholds, performance results do not influence decisions.

Action:

- Start with guardrails, not aggressive goals.

Example:

- no regression worse than 10-15% on the benchmark set without explicit acceptance

Success signal:

- Performance becomes an engineering constraint instead of a post hoc observation.

## Phase 5: Add Mutation Testing Carefully

### 12. Pilot mutation testing on the highest-value modules

Problem:

- Mutation testing is currently absent.

Action:

- Run a pilot with a Rust mutation tool on a small, high-value slice first.

Best pilot candidates:

- `C:\language\mdopus\src\opus\repacketizer.rs`
- `C:\language\mdopus\src\opus\decoder.rs`
- `C:\language\mdopus\src\types.rs`

Why these:

- branch-heavy packet logic
- low-level helpers with wide blast radius
- existing tests are already substantial enough to make mutation results meaningful

Success signal:

- Surviving mutants reveal real test weaknesses and produce targeted follow-up tests.

### 13. Do not roll mutation testing into the fast path yet

Reason:

- It is expensive and noisy until the branch/fuzz/performance lanes are cleaner.

Policy:

- Keep mutation testing as a scheduled or on-demand quality lane first.

### P1

- Add branch-targeted tests for `opus/encoder.rs`

- Add checked-in fuzz seed corpus

- Add scheduled fuzz execution

### P3

- Add scheduled medium and long soak profiles
- Standardize benchmark matrix and thresholds

### P4

- Pilot mutation testing on a narrow slice

## What Not To Do

- Do not bloat the fast path with long soak, full fuzz, or mutation runs.
- Do not chase 100% line coverage.
- Do not spend effort padding already-strong helper modules while large entry-point files remain branch-weak.
- Do not replace the C-reference oracle with purely internal assertions. The oracle is one of the repo's best assets.

## Success Criteria After One Iteration

This action plan is working if the next audit can say:

- the default validation path is obvious and routinely used
- the 280-case integration matrix is part of the normal gate
- overall branch coverage is meaningfully higher
- `opus/encoder`, `repacketizer`, `multistream`, `silk/encoder`, and `lpcnet` have visibly better branch coverage
- fuzzing has a real seed corpus and automated schedule
- soak and benchmark results are repeatable and tracked
- mutation testing has at least been piloted once on high-value modules

If those are true, the project will have moved from "strong testing toolkit" to "strong testing program."
