# HLD: Platform And Sanitizer Breadth

Date: 2026.05.06

Scope: likely next testing-improvement slice after the performance release
threshold work in `/home/md/language/ropus`.

## Problem Statement

The current release gate is much more honest than it was at the start of the
testing-improvement effort: core comparison integrity, asset preflight,
manifest-discovered fuzz sanity, oracle-strength reporting, neural/DRED claim
policy, generated corpus coverage, and performance thresholds are now explicit.

The remaining release breadth gap is platform confidence. The workspace defaults
x86_64 builds to `target-cpu = "x86-64-v3"` in `.cargo/config.toml`. That is a
reasonable performance default, but it means the normal local gate is also the
optimized local target profile. The release story should show that the core
Rust codec still builds and runs under a generic x86_64 baseline, and it should
surface the existing sanitizer-backed fuzz sanity path as a named sanitizer
claim instead of leaving it implicit inside `cargo-fuzz` plumbing.

Exploration says platform/sanitizer breadth is the right next slice. It should
stay small: add one generic x86_64 smoke profile and make the existing fuzz
sanity sanitizer signal visible. Do not add CI-provider configuration or a broad
cross-platform matrix in this slice.

## Stakeholders

- Arthur: needs release reports that do not overclaim platform portability.
- Codec maintainers: need optimized x86-64-v3 performance without accidentally
  depending on it for correctness.
- Users on older x86_64 CPUs: need confidence that rebuilding with
  `target-cpu=x86-64` remains viable.
- Fuzz/test maintainers: need sanitizer status to be explicit, but not so
  brittle that normal local validation becomes toolchain archaeology.
- Future release automation: needs machine-readable claim status for platform
  and sanitizer breadth without assuming a specific CI provider.

## Proposed Small Testing Slice

Implement a bounded **platform/sanitizer breadth contract** inside `full-test`.

The slice should add one new reportable stage, or a small `breadth` module that
renders as a stage, with two lanes:

1. `generic-x86_64-smoke`
   - On x86_64 hosts, run the core Rust crate under generic x86_64 codegen by
     overriding the repo's `.cargo/config.toml` target-cpu setting.
   - Recommended first command:

     ```bash
     timeout 900 env \
       CARGO_TARGET_DIR=target/platform-breadth/x86_64-generic \
       RUSTFLAGS="-C target-cpu=x86-64" \
       cargo test -p ropus --lib -- --test-threads=1
     ```

   - This is intentionally scoped to `ropus --lib`. It catches illegal
     optimized-CPU assumptions in the core safe Rust codec without multiplying
     the entire workspace cost.

2. `asan-fuzz-sanity`
   - Do not add a second full-workspace ASan command in this first slice.
   - Treat the existing non-quick release `fuzz::Mode::FullSanity` path as the
     first sanitizer-backed gate because `cargo-fuzz` already owns nightly ASan,
     C-reference sanitizer flags, crash replay, and Windows runtime setup.
   - Add structured reporting that says this sanitizer claim is satisfied only
     when fuzz sanity actually ran and passed.

This is intentionally narrower than "platform matrix." It gives one practical
generic-codegen signal and one practical sanitizer signal while avoiding
provider-specific CI YAML, cross-compilation, and toolchain-flaky full-workspace
ASan.

## Platform And Sanitizer Policy

Use explicit claim states; never let unsupported or skipped work appear green.

Suggested status vocabulary:

- `not_claimed`: default/quick profile did not claim this lane.
- `not_applicable`: host cannot run the specific lane, such as generic x86_64
  smoke on a non-x86_64 host.
- `unsupported`: lane was requested but required tools are absent.
- `pass`: claimed lane ran and passed.
- `fail`: claimed lane ran and failed, or was claimed but could not run.

Profile policy:

| Profile | Generic x86_64 smoke | Sanitizer claim | Banner/exit behavior |
|---|---|---|---|
| Default `full-test` | `not_claimed`; optionally report host/config inventory. | `not_claimed`; fuzz sanity remains not requested. | No new failures. |
| Quick `--release-preflight` | `not_claimed`; quick stays core smoke only. | `not_claimed`; quick fuzz inventory is not sanitizer coverage. | No platform/sanitizer release claim. |
| Non-quick `--release-preflight` on x86_64 | Required `generic-x86_64-smoke`. | Required only through passed `fuzz::Mode::FullSanity`. | Claimed lane failure is banner FAIL. |
| Non-quick `--release-preflight` off x86_64 | `not_applicable`, no x86_64 claim. | Same fuzz-sanity policy if tools support the host. | Report honestly; no false x86_64 pass. |

Important boundaries:

- Generic x86_64 smoke proves only `ropus --lib` under generic x86_64 codegen.
  It does not prove all binaries, C ABI, foobar2000, or Windows.
- ASan fuzz sanity proves the fuzz target build/replay path under cargo-fuzz's
  sanitizer model. It does not prove full-workspace ASan or Miri.
- Missing tools fail only when the profile claims the lane. Default local runs
  should remain useful on machines without nightly/cargo-fuzz.

## Risks

- `RUSTFLAGS` override semantics must be verified. The implementation should
  keep command construction unit-tested and run the generic smoke once locally
  to prove `.cargo/config.toml` does not still force x86-64-v3.
- Running `ropus --lib` twice increases non-quick release-preflight duration.
  This is acceptable for a release profile but should not leak into default or
  quick runs.
- The generic x86_64 smoke is still a Linux-hosted signal here; it does not
  prove Windows, macOS, ARM, or WebAssembly.
- ASan fuzz sanity is a real sanitizer signal but not full-workspace sanitizer
  coverage. Reporting must say that explicitly.
- cargo-fuzz/nightly availability can be fragile. That is why sanitizer failures
  should be release-claim failures only when the sanitizer lane is claimed.
- Separate target dirs can leave extra build artifacts under `target/`; keep
  them under an ignored `target/platform-breadth/...` subtree.

## Non-Goals

- No `.github`, GitLab, Azure, or other CI provider configuration.
- No broad OS matrix.
- No Windows runner setup beyond preserving the existing fuzz runner's ASan
  support.
- No full-workspace ASan, MSan, TSan, UBSan, or Miri gate in this first slice.
- No new platform-specific SIMD implementation or performance tuning.
- No changes to codec behavior.
- No mutation testing revival; that remains Phase 4.3.
