# HLD — Acceptance oracles for Stages 6–8 (tiered)

**Supersedes:** the per-stage acceptance criteria for Stages 6, 7, 8 in
`2026.04.19 - HLD - deferred-work-closeout.md`. Everything else in that HLD
(scope, sizing, priority order) still stands.

**Document version:** v2 (2026-04-19, post-harness-config audit).
v1 claimed bit-exactness was infeasible for these stages. A subsequent
audit of `harness/build.rs`, `harness/config.h`, `reference/dnn/vec.h`,
and `reference/src/mlp.c` showed that claim was overstated. This v2
replaces v1 in full. The earlier framing is captured in the journal
for the record; the relaxed oracles are retained as tier-2 fallbacks.

## Post-audit finding

The audit changed the picture on three fronts:

1. **No FMA fusion in the current C build.** `harness/build.rs` passes
   no `-mfma`, `-mavx`, `-march=native`, or `-ffp-contract=*` flags.
   On MSVC (default `/fp:precise`) the compiler will not contract
   multiply-adds. On gcc/clang a single `-ffp-contract=off` flag
   guarantees the same. The C `mlp.c:38` macro
   `#define fmadd(a,b,c) ((a)*(b)+(c))` is semantically multiply-then-add,
   not a fused intrinsic.

2. **Scalar C DNN path matches our Rust pattern exactly.** `vec.h`
   scalar `sgemv`/`sparse_sgemv8x4` and `nnet_default.c` use the
   same `y[k] += w[k]*xj` accumulation order as our `ropus/src/dnn/`
   kernels. No FMA intrinsics. Same operation counts, same order.

3. **Analysis is currently gated off on the C side.** `config.h:23`
   sets `DISABLE_FLOAT_API 1`, which short-circuits
   `analysis.c`/`mlp.c`/`mlp_data.c`. Our existing bit-exact test
   suite works precisely because analysis is disabled symmetrically
   on both sides. Bringing analysis online means un-setting that
   define in tandem with wiring the Rust port.

Net: bit-exactness is plausible for Stages 6–8 under scalar-C build
conditions, provided we enforce `-ffp-contract=off`, keep our Rust
`mul_add`-free, and replicate the C algorithm (order, constants)
verbatim. The v1 framing that conflated scalar-C with SIMD-C was wrong.

**Do not add `f32::mul_add` to Rust DNN code.** That would create
drift against the scalar-C path we actually compare against. Our
10,318 lines of DNN are already `sum += a*b` throughout — leave them
that way.

## Strategy — tiered acceptance

Each stage attempts tier 1 first. If residual drift (measured, not
assumed) makes tier 1 unachievable, fall back to tier 2 with a
written justification recorded in the journal.

- **Tier 1: bit-exact.** Byte-exact encode + sample-exact decode vs C
  reference, under the scalar-C harness config. Same oracle the
  completed stages (0–5, 9) used.
- **Tier 2: tolerance-based.** Layer-appropriate measurements
  (decision agreement, PCM SNR, split-layer criteria for OSCE/DRED).
  These are fallbacks, not defaults.

Tier transitions are explicit and must survive a devils-advocate
review — "drift exists" is not sufficient; the journal must record
what the drift is, where it originates, why it cannot be eliminated,
and what residual confidence tier 2 provides.

## Measurement surfaces (tier-2 reference)

| Layer | What it captures | Oracle form | When used |
|---|---|---|---|
| Bitstream | encoded bytes | byte match | Tier 1 primary |
| PCM sample | decoded samples | sample match | Tier 1 primary |
| Numerical | MLP activations, feature vectors | rel err < N·10⁻⁴ | Tier 2 diagnostic |
| Decision | classifier verdict, mode choice | agreement rate | Tier 2 if tier 1 fails |
| Audio | decoded PCM under equivalence | SNR / spectral distance | Tier 2 if tier 1 fails |

Numerical-layer tolerances do not translate to bitstream-layer
tolerances because drift is non-linear at decision boundaries. That's
why the fallback hierarchy matters: if we can't hold bit-exact, the
next-best oracle is behavioural (decisions + audio), not numerical.

## Pre-work — harness config alignment

Before Stage 6 starts, in one commit:

1. Add `.flag_if_supported("-ffp-contract=off")` to `harness/build.rs`
   `cc::Build` configuration. Verifies MSVC behaviour is already
   correct; locks gcc/clang down explicitly.
2. Remove `#define DISABLE_FLOAT_API 1` from `harness/config.h`. This
   turns analysis back on for the C reference. Expected effect: the
   existing byte-exact encode tests will now have analysis running on
   the C side but no analysis on the Rust side — they will fail.
   **That failure is the expected starting state for Stage 6.** The
   Rust port restores them to green.
3. Do NOT touch `ropus/src/dnn/` — no `mul_add` edits, no structural
   changes. The audit confirmed Rust DNN math already matches the
   scalar-C path.

This is a 1-commit prep step, not a dedicated stage. Exits with: harness
builds, the existing test-suite has a known-red set of tests (encode
comparisons where analysis now diverges), tier 1 for Stage 6 is
actionable.

## Stage 6 — Analysis / tonality

**Scope (unchanged from prior HLD).** Port `reference/src/analysis.c`
(982 lines), `mlp.c` (131 lines), and the weights blob in `mlp_data.c`
(671 lines). Wire the tonality analyzer into `ropus/src/opus/encoder.rs`
mode/rate decision path.

**Tier 1 acceptance (attempt first).**

1. **Byte-exact encode.** The encode comparison tests that went red
   after the pre-work (step 2 above) return to green. This requires
   Rust analysis to produce bit-identical output to C analysis at
   every frame. Replicate `tansig_approx`, MLP matmul order, and state
   machine verbatim.
2. **No regressions.** Existing CELT-only, SILK-only, Hybrid byte-exact
   tests still pass. Analysis integration must not leak into paths
   that previously avoided it.

**Tier 2 acceptance (fallback if tier 1 demonstrably fails after
reasonable effort).**

1. **Decision agreement.** On a ≥5-minute mixed-content corpus,
   per-frame `speech_prob` and `music_prob` outputs agree with C on
   **>99% of frames** within 0.01 absolute probability.
2. **Bitrate stability.** Average bitrate over the corpus within
   **2% of C reference** at matching target bitrate.
3. **No regressions** (same as tier 1).

**Tooling.** Tier 1 uses the existing encode-comparison harness — no
new binary needed if it works. Tier 2 requires
`harness/src/bin/analysis_diff.rs` (WAV corpus → both encoders with
analysis → per-frame classifier dump → agreement/bitrate report).
Build tier 2 tooling only if we drop to tier 2.

## Stage 7 — DNN wiring (FARGAN PLC + adjacent CTLs)

**Scope.** Wire the existing (verified, 10,318-line) Rust DNN modules
into the codec hot path:

- `ropus/src/dnn/fargan.rs` → `ropus/src/opus/decoder.rs` PLC path
  (currently a no-op stub, per the comment at `decoder.rs:12`).
- `ropus/src/dnn/pitchdnn.rs` → SILK pitch analysis hints.
- `SET_DNN_BLOB`, `OSCE_BWE`, and related CTLs → routed through
  `capi/src/ctl.rs` + `ctl_shim.c`.

**Prerequisite.** Add `reference/dnn/nnet.c`, `nnet_default.c`, and
`fargan.c` (+ dependencies) to `harness/build.rs` **without** the
SIMD header variants (`vec_avx.h`, `vec_neon.h`). C compiles the
scalar path; Rust already runs the scalar path; `-ffp-contract=off`
from pre-work is in force. Also enable `ENABLE_DEEP_PLC` in
`config.h` and corresponding Rust wiring.

**Weights blob decision.** Embed as a `const` byte array in the
crate, gated behind a single Cargo feature (default-on). Keeps the
default build self-contained; advanced users can opt out for binary
size. No runtime file loading — that invites path bugs and ships
nothing we can't version-pin.

**Tier 1 acceptance (attempt first).**

1. **Byte-exact encode (with analysis + FARGAN both on).** The
   comparison tests pass bit-exactly under the enlarged C build.
2. **Sample-exact decode under packet loss.** Given an identical
   seeded packet-loss pattern, Rust-decoded PCM matches C-decoded PCM
   sample-for-sample.
3. **CTL conformance.** `SET_DNN_BLOB`, `OSCE_BWE`,
   `SET_COMPLEXITY` (decoder-side), and `SET_IGNORE_EXTENSIONS`
   round-trip SET → GET and match C shim behaviour.

**Tier 2 acceptance (fallback).**

1. **FARGAN PCM quality.** Rust-decoded PCM under the seeded
   packet-loss pattern has **SNR > 50 dB** against C-decoded PCM.
2. **CTL conformance** (same as tier 1).
3. **No regressions on lossless paths** (bit-exact still expected
   when no loss occurs).

**Amendment (2026.04.19, Stage 7b.3).** The original 60 dB gate was
unachievable by construction on the classical PLC path: this harness
links **float-mode** C (for DEEP_PLC compile-time weights) against
**fixed-point** Rust, and the two arithmetic kernels diverge by 1-2
LSB on every sample through the recursive LPC/LTP filters inside
`silk_PLC_conceal`. The `harness-control` crate measures that
ceiling directly by diffing C-fixed classical PLC against C-float
classical PLC on the same packet stream and seeded loss pattern; the
measured arithmetic-gap SNR is **42.33 dB**. Any gate above that is a
test of f32 rounding, not of concealment correctness.

The amended 50 dB gate sits ~10 dB above that ceiling — enough margin
to catch a real PLC regression (a state-carryover bug shows up as
<20 dB) without tripping on the inherent fixed/float floor. Ropus
with the stage 7b.3 PLC state-carryover fixes measures 51.79 dB,
beating the classical ceiling by ~9 dB because DEEP_PLC's neural
smoothing is more forgiving of small per-sample errors than the
classical LPC IIR path.

**Tooling.** Extend the decoder harness with
`--packet-loss-seed=<n> --loss-rate=<pct>`. Add a `pcm_snr` helper
only if we drop to tier 2.

**Explicit non-goal.** LPCNet wiring — deprecated upstream in favour
of FARGAN; the 3802-line Rust port can sit inert until someone asks.
Not in 1.0 scope.

## Stage 8 — OSCE + DRED (fresh ports)

**Scope.** Fresh Rust ports of OSCE (6 C files, neural post-filter)
and DRED (~12 C files, Deep REDundancy — in-band neural compression
of recent history, the Opus 1.5 headline feature).

**Prerequisite.** Add the OSCE and DRED C sources to
`harness/build.rs` under the scalar-only config, and enable
`ENABLE_OSCE` / `ENABLE_DRED` in `config.h`.

**Tier 1 acceptance (attempt first).**

1. **DRED encode byte-exact.** DRED redundancy frames from Rust match
   C byte-for-byte. The entropy coder is integer fixed-point — this
   should be the easiest tier-1 target of the three stages.
2. **OSCE post-filter sample-exact.** Post-filtered Rust output
   matches C sample-for-sample under the scalar build.
3. **DRED decode sample-exact.** Reconstructed PCM matches C
   reconstruction sample-for-sample on a seeded packet-loss pattern.
4. **No regressions** on non-OSCE / non-DRED paths.

**Tier 2 acceptance (fallback, per-layer — the entropy-coder layer
never leaves tier 1).**

1. **DRED encode byte-exact** (stays tier 1 — fixed-point integer).
2. **OSCE post-filter quality.** Post-filtered Rust PCM has
   **SNR > 60 dB** against post-filtered C PCM on a 6 kbps speech clip.
3. **DRED decode reconstruction.** Reconstructed PCM has
   **SNR > 50 dB** against C on a 5% loss / 1s gap scenario. Lower
   bar than OSCE because the synthesizer runs ~1 s of extrapolation.
4. **No regressions** (same as tier 1).

**Gate.** This stage is XL-scoped (2–3 weeks even under tier 2) and
requires explicit Arthur approval before start, per the original
HLD's stated gating.

## Out of scope

- Adding `f32::mul_add` to Rust DNN code (the audit confirms this
  would create drift, not close it).
- LPCNet wiring.
- Training, retraining, or finetuning any neural weights.
- Platform-specific SIMD for DNN paths (separate perf workstream).
- Runtime weight-blob loading from disk.
- Extending the harness to compile C's SIMD DNN paths (vec_avx.h,
  vec_neon.h). Scalar only — we validate against the deterministic
  path, not the platform-vectorized one.

## Risks

1. **Tier 1 may fail for subtler reasons than FMA.** Possibilities:
   compiler-driven reordering of commutative operations; differences
   in `f32` constant parsing (though both C and Rust use IEEE-754);
   an overlooked SIMD helper we accidentally compile; analysis state
   that depends on un-initialized memory in the C reference.
   Mitigation: the pre-work step makes the existing test-suite visibly
   red, so any tier-1 divergence shows up immediately and can be
   bisected before we've written the analysis port. If bisection
   points at something un-fixable, tier 2 is ready.

2. **Un-gating `DISABLE_FLOAT_API` may break fuzz targets** that
   assumed analysis was disabled. Mitigation: run the fuzz suite
   once after pre-work; update fuzz harness setup if anything
   regresses.

3. **The `-ffp-contract=off` flag might not be honoured by all
   supported compilers.** Mitigation: `flag_if_supported` is silent
   on unsupported compilers; we can always verify on Windows/MSVC
   (the primary dev target). Cross-platform check: CI.

4. **Tier 2 threshold is a judgement call if tier 1 fails and we
   land at, say, 97% decision agreement.** Mitigation: document
   disagreement frames manually; if they're genuinely acoustically
   equivalent (e.g., speech/music classifier flipping on 5% of
   ambiguous content), the threshold can be lowered with justification.
   If they're genuine port bugs, we fix them.

5. **DRED is still evolving upstream.** If xiph revises the 1.5 format
   before we ship Stage 8, we port twice. Mitigation: pin to a
   specific upstream commit before starting Stage 8.

## What this HLD does not change

- Stages 0–5, 9 are shipped against the original bit-exact oracle;
  that was correct for them. Nothing about those stages is revisited.
- Stage 10 (final plan-vs-implementation audit) still runs against
  the original HLD's scope — this HLD augments the acceptance
  criteria for Stages 6–8 that the audit will check. If a stage
  lands under tier 2, the audit records which tier and why.
- The priority order (6 → 7 → 8) stands.
