# HLD — Relaxed acceptance oracles for Stages 6–8

**Supersedes:** the per-stage acceptance criteria for Stages 6, 7, 8 in
`2026.04.19 - HLD - deferred-work-closeout.md`. Everything else in that HLD
(scope, sizing, priority order) still stands.

**Rationale for a second doc.** Stages 0–5, 9 shipped against the original
bit-exact oracle. Editing the original would rewrite the record those stages
were accepted under. This HLD is additive: it revises only what remains.

## Why the original oracle was wrong for these stages

The deferred-work HLD adopted *"byte-exact encode, sample-exact decode vs
the xiph/opus C reference"* as the universal oracle. That is correct for
pure fixed-point ports — Stages 0–5 and 9 all used it successfully.

It is infeasible for Stages 6–8 because each depends on float math whose
bit patterns cannot be reproduced in Rust without replicating the C
approximations verbatim:

- `tansig_approx`, `celt_exp2`, `celt_log2` use specific polynomial
  approximations with hand-picked coefficients and table lookups.
- MLP matmul in `reference/src/mlp.c` is evaluated with whatever the C
  compiler fuses — typically `vfmadd231ps` on x86. Rust does not fuse
  unless `f32::mul_add` is used explicitly.
- One FMA vs two discrete float ops differs by ≤1 ULP; a 16-unit MLP
  layer compounds that. Intermediate drift of ~1e-6 to 1e-4 is normal
  and acoustically inaudible, but it flips classifier boundaries at
  edge-case frames — which flips encoder decisions, which diverges
  the bitstream irrecoverably.

The bit-exact criterion was a drafting error. This HLD corrects it.

## Principles

1. **Oracle matches the signal.** Float-math outputs get tolerance-based
   oracles; integer control paths (state machines, entropy coding)
   keep bit-exactness where they still apply.
2. **Tighten before relaxing.** A `mul_add` audit of our DNN matmul
   kernels should come first. Closing the FMA gap may reduce drift by
   an order of magnitude at zero scope cost.
3. **PCM is the final test.** Numerical intermediates are proxies.
   What users perceive is the decoded audio.

## Measurement surfaces

| Layer | What it captures | Oracle form | Feasible? |
|---|---|---|---|
| Numerical | MLP activations, feature vectors | rel err < N·10⁻⁴ | Hard (FMA sensitivity) |
| Decision | classifier verdict, mode choice, rate target | agreement rate over corpus | Yes |
| Audio | decoded PCM | SNR or spectral distance | Yes |
| Bitstream | encoded bytes | byte match | No once decisions diverge |

Numerical-layer tolerances do **not** translate to bitstream-layer
tolerances because drift is non-linear at decision boundaries. Pick
the surface that matters for each stage, not the strictest one.

## Pre-work — `mul_add` audit

Before Stage 6 kicks off, audit `ropus/src/dnn/core.rs` matmul kernels
and any float-heavy paths in `dnn/fargan.rs`, `dnn/pitchdnn.rs`. Where
C fuses (it does, via the compiler's auto-FMA), switch to
`f32::mul_add` explicitly. Re-run existing DNN unit tests and measure
residual drift against stored golden vectors.

Expected outcome: module-level rel err drops from ~1e-4 to ~1e-6.
This is a ~1-hour investigation plus targeted edits, not a dedicated
stage. Its result informs the tolerance constants we pick below.

## Stage 6 — Analysis / tonality

**Scope (unchanged from prior HLD).** Port `reference/src/analysis.c`
(982 lines), `mlp.c` (131 lines), and the weights blob in `mlp_data.c`
(671 lines). Wire the tonality analyzer into `ropus/src/opus/encoder.rs`
mode/rate decision path.

**Acceptance criteria.**

1. **Decision agreement.** On a ≥5-minute mixed-content corpus (speech
   + music + silence + transients), per-frame `speech_prob` and
   `music_prob` outputs from Rust agree with C on **>99% of frames**
   within 0.01 absolute probability. Corpus populated by the operator
   from the scaffolding landed in Stage 3.
2. **Bitrate stability.** Average bitrate over the corpus is within
   **2% of C reference** at matching target bitrate (VBR, complexity 10,
   48 kHz stereo).
3. **No regressions.** CELT-only and SILK-only byte-exact tests at
   fixed bitrate (where analysis is off or has no effect) continue to
   pass bit-exactly. This catches accidental code paths where analysis
   output leaks into deterministic modes.

**Tooling.** One new harness binary: `harness/src/bin/analysis_diff.rs`.
Takes a WAV corpus, runs both encoders with analysis enabled, dumps
per-frame classifier outputs and bitrate stats, emits a report.

## Stage 7 — DNN wiring (FARGAN PLC + adjacent CTLs)

**Scope.** Wire the existing (verified, 10,318-line) Rust DNN modules
into the codec hot path. Specifically:

- `ropus/src/dnn/fargan.rs` → `ropus/src/opus/decoder.rs` PLC path
  (currently a no-op stub, per the comment at `decoder.rs:12`).
- `ropus/src/dnn/pitchdnn.rs` → SILK pitch analysis hints.
- `SET_DNN_BLOB`, `OSCE_BWE`, and related CTLs → routed through
  `capi/src/ctl.rs` + `ctl_shim.c`.

**Weights blob decision.** Embed as a `const` byte array in the
crate, gated behind a single Cargo feature (default-on). Keeps the
default build self-contained; advanced users can opt out for binary
size. No runtime file loading — that invites path bugs and ships
nothing we can't version-pin.

**Acceptance criteria.**

1. **FARGAN PCM quality.** Given a seeded packet-loss pattern (RNG
   seed fixed, 10% loss rate, 30s speech clip), Rust-decoded PCM has
   **SNR > 60 dB** against C-decoded PCM on the same input+loss pattern.
2. **CTL conformance.** `SET_DNN_BLOB`, `OSCE_BWE`, `SET_COMPLEXITY`
   (decoder-side), and `SET_IGNORE_EXTENSIONS` round-trip through
   SET → GET without error and match C shim behaviour.
3. **No regressions.** Lossless decode (no packet-loss) remains
   bit-exact vs C.

**Tooling.** Extend the existing decoder harness with a
`--packet-loss-seed=<n> --loss-rate=<pct>` flag. Add a `pcm_snr`
helper (~20 lines).

**Explicit non-goal.** LPCNet wiring — deprecated upstream in favour
of FARGAN, and the 3802-line Rust port can sit inert until someone
asks. Not in 1.0 scope.

## Stage 8 — OSCE + DRED (fresh ports)

**Scope.** Fresh Rust ports of OSCE (6 C files, neural post-filter
cleaning up low-bitrate speech artifacts) and DRED (~12 C files, Deep
REDundancy — in-band neural compression of recent history, the Opus
1.5 headline feature).

**Split acceptance by layer.** The DRED entropy coder is genuinely
integer fixed-point; only the neural synthesizer is float. Acceptance
criteria split accordingly.

1. **OSCE post-filter quality.** On a 6 kbps speech clip, post-filtered
   Rust output has **SNR > 60 dB** against post-filtered C output.
2. **DRED encode (fixed-point layer).** DRED redundancy frames emitted
   by the Rust encoder have **byte-exact lengths** matching C at the
   same input + bitrate + redundancy settings. The entropy coder is
   the same kind of code we already bit-exact-match in CELT.
3. **DRED decode reconstruction (neural layer).** On a 5% packet loss
   scenario with a 1-second gap, reconstructed PCM has **SNR > 50 dB**
   against C reconstruction. Lower bar than OSCE because the
   synthesizer runs 1s of extrapolation — more drift is expected.
4. **No regressions.** Non-OSCE / non-DRED paths remain bit-exact.

**Gate.** This stage is XL-scoped (2–3 weeks even with relaxed
oracles) and requires explicit Arthur approval before start, per the
original HLD's stated gating.

## Staging

| Order | Item | Exit condition |
|---|---|---|
| 0 | `mul_add` audit | DNN unit-test drift measured; tolerance constants chosen |
| 1 | Stage 6 (Analysis) | Decision ≥99%, bitrate ±2%, regressions clean |
| 2 | Stage 7 (FARGAN) | SNR ≥60 dB on PLC corpus, CTLs round-trip, regressions clean |
| 3 | Arthur checkpoint | Explicit go/no-go on Stage 8 |
| 4 | Stage 8 (OSCE+DRED) | Split criteria above all met |

## Out of scope

- Bit-exactness of any float intermediate (the whole point of this
  HLD).
- LPCNet wiring.
- Training, retraining, or finetuning any neural weights.
- Platform-specific SIMD for DNN paths (belongs to the separate perf
  workstream).
- Runtime weight-blob loading from disk.

## Risks

1. **Decision-agreement threshold is a judgement call.** 99% may be
   too strict on edge-case content. If we hit 97–98%, we have to
   decide: genuine port bug, or acceptable drift? Mitigation: review
   disagreement frames manually to classify; adjust threshold with
   written justification if needed.
2. **SNR > 60 dB assumes FMA fusion closes enough of the gap.** If
   drift is larger than expected, we may have to accept 40–50 dB and
   argue perceptual equivalence via PESQ or ViSQOL. Mitigation: run
   a perceptual metric as a sanity backstop early in Stage 7.
3. **DRED is still evolving upstream.** Opus 1.5 is recent; if xiph
   revises the format before we ship Stage 8, we port twice.
   Mitigation: check release cadence and pin to a specific upstream
   commit before starting Stage 8.
4. **Corpus bias.** A 5-minute corpus may not expose rare decision
   boundaries. Mitigation: include adversarial content (music-speech
   transitions, heavy silence, transients) deliberately.

## What this HLD does not change

- Stages 0–5, 9 are shipped against the original bit-exact oracle;
  that was correct for them. Nothing about those stages is revisited.
- Stage 10 (final plan-vs-implementation audit) still runs against
  the original HLD's scope — this HLD simply augments the acceptance
  criteria for Stages 6–8 that the audit will check.
- The priority order (6 → 7 → 8) stands.
