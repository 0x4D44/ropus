# HLD: Release Preflight And Oracle Matrix

Date: 2026-05-06

Status: Design only

Workstream: C - Release preflight and oracle matrix

## Problem Statement

The repository can currently produce useful green local validation while some of
the strongest oracle assets are absent or downgraded. That is acceptable for a
fresh-clone developer build, but it is not acceptable for release language such
as "conformance passes", "C-reference parity covered", or "DNN/DRED covered".

The minimum useful fix is not a bigger test runner. It is a small release
preflight that makes asset-backed claims explicit, plus a maintained oracle
matrix that labels the strongest assertion behind each validation lane. The
preflight should fail a release claim when required assets are missing, while
leaving default build ergonomics alone.

This design does not implement code.

## Proposed Release Preflight

Add a release-only preflight lane to `full-test`, not to default Cargo builds.
Default `cargo build` and ordinary `cargo test` should stay friendly to fresh
clones; release claims should be stricter.

Recommended user surface:

```text
cargo run --release -p full-test -- --release-preflight [existing flags]
```

For C1, `--release-preflight` has one concrete built-in profile: `core`.
There are no user-selectable claim profiles in the first implementation.

The C1 `core` profile means:

- Required: fixed-point C reference source is present.
- Required: upstream conformance C tests are present.
- Required: IETF vectors are present or provisioned by the existing
  `full-test` IETF provisioning path.
- Report-only: base DNN weights are present/missing.
- Report-only: DRED RDOVAE weights are present/missing.
- Report-only: float-mode deep PLC harness assets are present/missing.

This deliberately avoids a vague "selected claims" mechanism. In C1, the only
banner-failing preflight conditions are missing fixed-point C reference, missing
upstream conformance tests, or unavailable IETF vectors. Neural PLC, DNN, DRED,
and float-mode deep PLC are not claimed by the initial profile; their assets
should be visible in the report but must not affect the banner until a future
explicit neural/DRED profile exists.

Preflight should run in Stage 0 after normal setup capture and before Stage 1.
It should not fetch new assets except through the existing IETF provisioning
path, which already belongs to `full-test`. It should produce a structured
asset summary for the JSON envelope and HTML report, then participate in banner
classification as a hard FAIL when a C1 required asset is missing.

Minimal C1 checks:

| Asset key | Probe | C1 requirement | Banner effect |
|---|---|---|---|
| `fixed_reference` | `reference/celt/bands.c` | required | FAIL if missing |
| `conformance_sources` | `reference/tests/test_opus_api.c` | required | FAIL if missing |
| `ietf_vectors` | existing complete vector check in `full-test/src/ietf_vectors.rs` | required | existing Stage 2 synthetic failure if unavailable |
| `dnn_base_weights` | `reference/dnn/pitchdnn_data.c`, `reference/dnn/fargan_data.c`, `reference/dnn/plc_data.c` | optional/report-only | no banner effect |
| `dred_rdovae_weights` | `reference/dnn/dred_rdovae_enc_data.c`, `reference/dnn/dred_rdovae_dec_data.c` | optional/report-only | no banner effect |
| `float_deep_plc_assets` | `fixed_reference` plus `dnn_base_weights` | optional/report-only | no banner effect |

Status values should be:

| Status | Meaning | Release behavior |
|---|---|---|
| `present_required` | Required probe files exist before any provisioning. | Gateable. |
| `provisioned_required` | Existing provisioning supplied a required asset during this run. | Gateable; used for IETF vectors only in C1. |
| `missing_required` | Required probe failed. | FAIL under `--release-preflight`, except IETF uses the existing Stage 2 synthetic failure path. |
| `present_optional` | Optional/report-only probe files exist. | Render as available; no banner effect. |
| `missing_optional` | Optional/report-only probe failed. | Render as unavailable for stronger future claims; no banner effect in C1. |

Do not overload `build_failed`; a missing release asset is not a Rust compile
failure. It should be its own preflight result, so Stage 2 can still run and
produce useful signal.

IETF handling needs one special rule: do not add a second banner-failing
preflight error for missing IETF vectors. Existing code already converts
unavailable vectors into a synthetic Stage 2 failure at
`conformance::ietf_vectors::provisioning`. The C1 preflight table should still
show `missing_required` for `ietf_vectors`, but banner classification should
continue to fail through the existing Stage 2 result so the absence is not
double-counted in totals or failed-test lists.

## C1 And C2 Slices

### C1: Core Release Preflight And Matrix

C1 is the implementable slice after workstreams A/B finish.

Scope:

- Add `--release-preflight` as the C1 core profile described above.
- Add `full-test/src/preflight.rs` with pure path probes and status
  classification.
- Wire preflight into CLI, main/setup, JSON report, HTML report, and banner
  classification.
- Make `fixed_reference` and `conformance_sources` missing states fail the
  banner under `--release-preflight`.
- Reuse the existing IETF provisioning and synthetic Stage 2 failure path.
- Report `dnn_base_weights`, `dred_rdovae_weights`, and
  `float_deep_plc_assets` as optional present/missing.
- Add `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md` with the
  initial matrix rows and controlled oracle classes.

Out of scope for C1:

- New neural/DRED release profile flags.
- Hard-failing missing DNN/DRED weights.
- Matrix parsing or automatic drift detection.
- Changes to `harness/build.rs`, `tests/conformance/build.rs`, or
  `ropus/build.rs`.

### C2: Future Profiles And Automation

C2 can add explicit profiles once there is a release need and the corresponding
oracles are strong enough.

Candidate future CLI shape:

```text
cargo run --release -p full-test -- --release-preflight --profile core
cargo run --release -p full-test -- --release-preflight --profile neural-plc
cargo run --release -p full-test -- --release-preflight --profile dred
```

Alternative if profiles stay few:

```text
cargo run --release -p full-test -- --release-preflight --require-neural-plc
cargo run --release -p full-test -- --release-preflight --require-dred
```

Do not implement either shape in C1. C2 should decide based on real release
claims. A `neural-plc` profile would likely make `dnn_base_weights` and
`float_deep_plc_assets` required. A `dred` profile would likely make both
`dnn_base_weights` and `dred_rdovae_weights` required, but only after the DRED
oracle rows are no longer ignored/asset-skipped debt.

Preferred option: extend `full-test`.

- Add `--release-preflight` to `/home/md/language/ropus/full-test/src/cli.rs`.
- Add a small `/home/md/language/ropus/full-test/src/preflight.rs` module for
  pure probes, status classification, and unit tests.

  preflight after IETF provisioning.
- Extend `/home/md/language/ropus/full-test/src/report.rs` and
  `/home/md/language/ropus/full-test/src/html.rs` to show the preflight table.
- Extend `/home/md/language/ropus/full-test/src/banner.rs` so missing required
  C1 release assets cause FAIL without pretending Stage 2 failed to compile.

This is the smallest place because `full-test` already owns setup, IETF
provisioning, HTML/JSON reporting, and PASS/FAIL/WARN classification.

Alternative option: a dedicated binary such as `release-preflight`.

- Pro: simpler CLI and faster one-shot command.
- Con: duplicates workspace-root detection, reporting, asset status rendering,
  and future matrix plumbing already present in `full-test`.

I would only choose the dedicated binary if Arthur wants preflight to run without
building the full-test runner or without sharing its report format.

## Oracle Matrix Schema

Add a maintained document at:

`/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`

Schema:

| Column | Purpose |
|---|---|
| `subsystem` | `CELT`, `SILK`, `Opus`, `DNN`, `DRED`, `Conformance`, `Fuzz`, `Performance`, etc. |
| `command_or_target` | Exact command, test binary, fuzz target, or full-test stage. |
| `strongest_oracle` | One of the controlled oracle classes below. |
| `required_assets` | `fixed_reference`, `conformance_sources`, `ietf_vectors`, `dnn_base_weights`, `dred_rdovae_weights`, `float_deep_plc_assets`, corpus path, or `none`. |
| `asserts` | The correctness property actually checked. |
| `does_not_assert` | Explicitly names the gap. |
| `release_gate` | `required-core`, `optional-report`, `warn`, `not-gate`, or `debt`. |
| `preflight_key` | Asset key that release preflight should map to, or `none`. |
| `owner_or_followup` | Existing HLD, issue, journal, or person/team tag. |
| `last_reviewed` | Date the row was last verified against current tests. |

Controlled `strongest_oracle` classes:

- `exact-byte-parity`
- `exact-pcm-parity`
- `snr-gated-parity`
- `sample-count-only`
- `smoke-only`
- `ignored`
- `asset-skipped`

Initial rows should cover at least:

| Subsystem | Command/target | Strongest oracle | Required assets | Release gate |
|---|---|---|---|---|
| Conformance | `cargo test -p conformance -- --test-threads=1` | exact/reference API behavior by upstream test program | `fixed_reference`, `conformance_sources`, `ietf_vectors` for vector binary | required-core |
| Harness | `cargo run -p ropus-harness --bin ropus-compare -- unit range_coder` | exact-byte-parity | `fixed_reference` | required-core |
| Harness | `encode`, `decode`, `roundtrip` headline compare commands | exact-byte or exact-PCM parity once Phase 0.1/0.2 lands | `fixed_reference` plus fixture | required-core |
| IETF vectors | `cargo test -p conformance --test ietf_vectors -- --test-threads=1` | exact-PCM parity through upstream `opus_demo`/`opus_compare` | `fixed_reference`, `conformance_sources`, `ietf_vectors` | required-core |
| DNN PLC | `harness-deep-plc` tier-2 SNR tests | snr-gated-parity | `fixed_reference`, `dnn_base_weights`, `float_deep_plc_assets` | optional-report in C1; candidate C2 required neural profile |
| DRED | DRED payload-byte differential | ignored or asset-skipped today | `fixed_reference`, `dnn_base_weights`, `dred_rdovae_weights` | debt |
| Fuzz | shared fuzz oracle weakened classes | sample-count-only or exact-PCM depending class | target-specific | debt/warn |
| Performance | `full-test` bench stage | smoke/perf ratio, not correctness | `fixed_reference`, WAV fixtures | warn |

The matrix is documentation first. Do not automate it until it starts drifting in
practice. The only C1 machine link should be the `preflight_key` values matching
the `preflight.rs` asset keys.

## Risks And Tradeoffs

- Stricter release preflight can look like a regression to developers used to
  green fresh-clone builds. Mitigation: gate it behind `--release-preflight`,
  define C1 as the narrow core profile, and keep default Cargo behavior
  unchanged.
- Asset presence is not the same as oracle execution. Mitigation: preflight only
  says "gateable"; the matrix must still identify the command that proves the
  claim.
- DNN/DRED assets have multiple tiers. Base neural PLC weights and DRED RDOVAE
  weights should be separate optional keys in C1, or the report will imply
  stronger DRED coverage than exists.
- The matrix can drift because it is documentation. Mitigation: require
  `last_reviewed` and update it when adding/removing tests; automate only after
  drift becomes real.
- IETF vector provisioning uses network when vectors are missing. That behavior
  already exists in `full-test`; release preflight should reuse the result rather
  than add another fetch path.
- IETF absence can be double-counted if both preflight and Stage 2 add failures.
  Mitigation: C1 renders the IETF preflight status but relies on the existing
  synthetic Stage 2 failure for banner/test failure accounting.
- Build-script warnings are still useful for normal builds. Do not convert
  `harness/build.rs`, `tests/conformance/build.rs`, or `ropus/build.rs` back to
  hard panics; the stricter policy belongs in release validation.
- Future neural/DRED profiles could become premature. Mitigation: C2 should only
  add them when the oracle matrix has rows strong enough to justify hard release
  claims.
