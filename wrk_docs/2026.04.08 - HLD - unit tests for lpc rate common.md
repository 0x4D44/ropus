# HLD: Unit Tests for lpc.rs, rate.rs, silk/common.rs

**Date:** 2026-04-08
**Author:** Claude Opus 4.6 / Arthur
**Status:** Draft for review

## 1. Summary

Three core DSP modules (`celt/lpc.rs`, `celt/rate.rs`, `silk/common.rs`) have no
unit tests. They are covered indirectly by the 280-test integration sweep, but a
regression in any of them would surface only as cryptic integration failures
far from the root cause.

This HLD proposes adding `#[cfg(test)] mod tests { }` blocks to each file with
targeted unit tests. The goal is **fault isolation** -- when one of these modules
breaks, the failing unit test should name the exact function and failure mode.

Total estimated scope: ~75 test functions, ~1,200 lines of test code.

---

## 2. Module Analysis

### 2.1 `src/celt/lpc.rs` (354 lines, 4 functions)

This module implements LPC analysis and synthesis filtering for CELT. All four
functions are public, non-trivial, and performance-critical (called every frame
during encoding).

#### Functions inventory

| # | Function | Lines | Purpose | Complexity |
|---|----------|-------|---------|------------|
| 1 | `celt_lpc` | 20-104 | Levinson-Durbin LPC from autocorrelation (Q25 internal, Q12 output). Includes bandwidth expansion loop and fallback to A(z)=1. | **High** -- iterative recursion with convergence check and 10-iteration BW expansion. |
| 2 | `celt_autocorr` | 121-205 | Windowed autocorrelation with automatic normalization. Delegates bulk work to `celt_pitch_xcorr`. | **High** -- shift estimation, two-pass energy, normalization to [2^28, 2^30). |
| 3 | `celt_fir` | 227-263 | FIR filter (4-at-a-time unrolled + scalar tail). Input offset convention encodes filter memory. | **Medium** -- straightforward convolution but tricky indexing (memory at `x[0..ord]`, signal at `x[ord..]`). |
| 4 | `celt_iir` | 286-354 | IIR filter (4-at-a-time with inter-sample feedback corrections). Negated memory convention. | **High** -- unrolled feedback correction between the 4 outputs is a common source of bugs. |

#### Proposed test cases

**`celt_lpc`**

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_lpc_dc_signal` | ac = [N, N, N, ..., N] (flat autocorrelation of DC) | lpc_out[0] = 4096 (Q12 = 1.0), rest ~0 | First-order prediction of constant signal |
| `test_lpc_white_noise` | ac = [large, 0, 0, ..., 0] (delta autocorrelation) | lpc_out all zero (no prediction possible) | All-zero coefficients when signal is uncorrelated |
| `test_lpc_simple_sinusoid` | ac values for a pure cosine at known frequency | Two non-zero coefficients forming a conjugate pair | Mathematical property: sinusoid needs exactly 2nd-order LPC |
| `test_lpc_order_1_ar` | ac = [100, 50], p=1 | lpc_out[0] = -2048 (Q12 for r=-0.5) | Minimal case: first-order Levinson gives r = -ac[1]/ac[0] |
| `test_lpc_convergence_bailout` | ac where error drops below ac[0]>>10 quickly | Early termination; remaining coefficients are zero | 30dB gain bailout condition (line 50) |
| `test_lpc_bw_expansion_fallback` | Pathological ac producing huge Q25 coefficients that overflow Q12 | lpc_out[0] = 4096, rest 0 (A(z)=1 fallback) | 10-iteration BW expansion failure path (lines 93-98) |
| `test_lpc_zero_ac0` | ac[0] = 0 | All zeros output | Guard at line 27 |
| `test_lpc_max_order` | p = CELT_LPC_ORDER = 24, realistic ac | Stable Q12 coefficients | Full-order operation |

**`celt_fir`**

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_fir_identity` | num = [0,...,0,4096], x = signal padded with ord zeros | y = x (passthrough) | Unit impulse response |

**`celt_iir`**

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_iir_no_feedback` | den all zeros | y = x (passthrough) | Zero-denominator means no feedback |

| `test_iir_memory_update` | Short signal, check mem after call | mem[0] = last output sample | Memory update (lines 351-353) |
| `test_iir_n_not_multiple_of_4` | n=5, n=7 | Correct output | Scalar tail handles remainder |
| `test_iir_inter_sample_correction` | n=4, non-zero den[0..3] | Matches naive sample-by-sample IIR | The 4-sample feedback patch-up (lines 318-334) is the hardest part |

#### Edge cases and boundary conditions

- `celt_lpc`: p=0 (degenerate but should not crash), p=1 (minimal), p=24 (maximum)
- `celt_autocorr`: n=1 (minimum), lag=0, overlap=0 vs overlap=n/2
- `celt_fir`/`celt_iir`: n=0 (no-op), ord=4 (minimum for IIR), very large coefficients that test saturation in `sround16`

#### Priority ranking

1. **`celt_iir`** -- highest value. The inter-sample feedback correction is subtle and any error silently corrupts synthesis output. (Priority: P0)
2. **`celt_lpc`** -- the BW expansion and fallback logic has many branches. (Priority: P0)
3. **`celt_autocorr`** -- the shift normalization is error-prone. (Priority: P1)
4. **`celt_fir`** -- simpler than IIR but the memory convention is unusual. (Priority: P1)

### 2.2 `src/celt/rate.rs` (620 lines, 5 functions + tables)

This module handles CELT bit allocation. The two public entry points
(`clt_compute_allocation`, `interp_bits2pulses`) are large and stateful (they
read/write an entropy coder), making them harder to test in isolation. The helper
functions are simpler.

#### Functions inventory

| # | Function | Lines | Purpose | Complexity | Visibility |
|---|----------|-------|---------|------------|------------|
| 1 | `get_pulses` | 23-30 | Convert cache index to pulse count | **Low** -- two branches | `pub` |
| 2 | `bits2pulses` | 35-60 | Binary search: find pulse count fitting a bit budget | **Medium** -- binary search over cache | `pub` |
| 3 | `pulses2bits` | 64-73 | Cache lookup: bit cost for a pulse count | **Low** -- table lookup | `pub` |
| 4 | `celt_udiv` | 101-104 | Unsigned integer division (trivial wrapper) | **Trivial** | `fn` (private) |
| 5 | `interp_bits2pulses` | 121-420 | Core allocation: interpolate alloc vectors, decide skips, compute fine energy bits | **Very high** -- 300 lines, reads/writes EC coder | `fn` (private) |
| 6 | `clt_compute_allocation` | 462-620 | Top-level allocation: builds alloc vectors, calls `interp_bits2pulses` | **Very high** -- depends on CELTMode + EC coder | `pub` |

#### Proposed test cases

**`get_pulses`**

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_get_pulses_linear_range` | i = 0..7 | i | Below threshold, identity |
| `test_get_pulses_exponential_range` | i = 8 | (8+0)<<0 = 8 | Transition point |
| `test_get_pulses_i9` | i = 9 | (8+1)<<0 = 9 | Low bits extracted correctly |
| `test_get_pulses_i16` | i = 16 | (8+0)<<1 = 16 | Shift by 1 |
| `test_get_pulses_i24` | i = 24 | (8+0)<<2 = 32 | Shift by 2 |
| `test_get_pulses_monotonic` | i = 0..63 | Each result >= previous | Monotonicity property |

**`bits2pulses` and `pulses2bits`**

These depend on `CELTMode` (specifically `mode.cache.index` and `mode.cache.bits`).
The project provides `MODE_48000_960_120` as a static, so tests can use it.

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_bits2pulses_zero_budget` | bits=0, any band | 0 pulses | Minimum allocation |
| `test_bits2pulses_large_budget` | bits=very large, band 0 | Maximum pulses from cache | Saturation at cache maximum |
| `test_bits2pulses_roundtrip` | For each band/lm, q = bits2pulses(m, band, lm, pulses2bits(m, band, lm, q)) | q_out == q | Round-trip consistency |
| `test_pulses2bits_zero` | q=0, any band | 0 | Zero pulses = zero bits |
| `test_pulses2bits_monotonic` | q = 1..max for each band | Non-decreasing | More pulses always costs more bits |
| `test_bits2pulses_nearest` | Various bit budgets | Either lo or hi from cache, whichever is closer | Binary search finds nearest match |

**`interp_bits2pulses` and `clt_compute_allocation`**

These are hard to unit-test because they require a working entropy coder. However,
we can test them with a mock-free approach using the project's actual `RangeEncoder`
/ `RangeDecoder`:

| Test | Input | Expected | Validates |
|------|-------|----------|-----------|
| `test_allocation_encode_decode_symmetry` | Encode with known params, then decode the same bitstream | Decoded intensity, dual_stereo, coded_bands match encoded values | Encoder/decoder agreement |
| `test_allocation_total_bits_bounded` | Various total budgets | Sum of allocated bits <= total | Budget is not exceeded |
| `test_allocation_mono_no_stereo_params` | c=1 | intensity_rsv=0, dual_stereo_rsv=0 | Mono has no stereo overhead |
| `test_allocation_minimum_budget` | total=0 | All bands get 0 pulses, only fine energy | Graceful degradation |

#### Edge cases and boundary conditions

- `get_pulses`: i=0 (min), i=127 (if reachable -- check max cache entry)
- `bits2pulses`: bits < 0 (should not happen but could be defensive)
- Band indices: start=0, end=nb_ebands (full range); start=end-1 (single band)
- LM values: 0 (2.5ms), 1 (5ms), 2 (10ms), 3 (20ms)

#### Priority ranking

1. **`get_pulses`** -- trivial but foundational; every allocation depends on it. (Priority: P0)
2. **`bits2pulses` / `pulses2bits` round-trip** -- validates the cache is consistent. (Priority: P0)
3. **`clt_compute_allocation` encode/decode symmetry** -- validates the whole pipeline. (Priority: P1)
4. **`interp_bits2pulses` budget bound** -- harder to set up, lower marginal value. (Priority: P2)

### 2.3 `src/silk/common.rs` (1,523 lines, ~50 functions)

This is the largest of the three modules. It contains the shared SILK utility
functions: PRNG, math helpers, LPC analysis, NLSF processing, sorting,
resampling, and gain dequantization.

Many functions are trivial one-line wrappers around arithmetic (shifts, multiplies).
Testing every wrapper has low value. The prioritization below focuses on functions

## 3. Testing Approach

### 3.1 Deriving expected values

The FFI harness only exposes the public Opus API (encode/decode/repacketizer), not
the internal SILK or CELT functions. Therefore we **cannot** use the C reference as
a per-function test oracle directly.

Instead, expected values come from three sources:

**A. Mathematical properties (strongest)**
- Autocorrelation: R(0) >= |R(k)| for all k
- LPC Levinson-Durbin: 1st-order r = -ac[1]/ac[0]
- Sigmoid: s(0)=0.5, s(x)+s(-x)=1, monotonic
- Log/lin: log2(2^k) = k, roundtrip accuracy
- Sort: output is sorted, indices track original positions
- NLSF stabilize: output satisfies minimum spacing constraint
- Inverse prediction gain: identity filter has gain 1.0

**B. Hand-computed fixed-point values (reliable for simple cases)**
- `get_pulses(8)` = 8, `get_pulses(16)` = 16, etc.
- `silk_sigm_q15(0)` = `SIGM_LUT_POS_Q15[0]` = 16384
- `silk_lin2log(1)` = 0 (verified by tracing the C code)
- `silk_log2lin(0)` = 1, `silk_log2lin(128)` = 2

**C. Naive reference implementations (for complex functions)**
- Write a simple O(n^2) convolution to cross-check `celt_fir` and `celt_iir`
- Write sample-by-sample IIR to cross-check the 4-sample unrolled version
- Compute sum-of-squares with i64 to cross-check `silk_sum_sqr_shift`

### 3.2 Test structure

All tests live in `#[cfg(test)] mod tests { }` at the bottom of each file, using
`use super::*;` for access to private functions.

Tests follow the existing project convention (see `celt/math_ops.rs`, `celt/cwrs.rs`,
`types.rs`):
- Named `test_<function>_<scenario>`
- Use `assert_eq!` for exact values, `assert!` with inequality for properties
- Include error messages with context: `assert_eq!(result, expected, "failed for input {input}")`
- No external dependencies or test fixtures needed

For rate.rs tests that need a `CELTMode`, use the static `MODE_48000_960_120`.
For rate.rs tests that need an entropy coder, use `RangeEncoder`/`RangeDecoder`
from `celt::range_coder` (already in the crate, no FFI needed).

For silk/common.rs NLSF tests, use the static `SILK_NLSF_CB_NB_MB` (order=10)
and `SILK_NLSF_CB_WB` (order=16) from `silk::tables`.

## 5. Risks

### 5.1 No per-function C oracle

The FFI harness does not expose `celt_lpc`, `silk_NLSF2A`, etc. individually.
We cannot call the C reference function and compare output. This means:

- Expected values for complex functions (e.g. `silk_lpc_inverse_pred_gain`) must
  be hand-computed or derived from mathematical properties.
- If a test fails, we cannot immediately tell whether the bug is in the Rust code
  or in our expected value. Mitigation: use property-based checks (monotonicity,
  range, round-trip) rather than exact-value checks where possible.
- **Future option:** Add per-function FFI bindings in `tests/harness/bindings.rs`
  for the ~10 most critical internal functions. This would give us a true oracle
  but requires modifying the C build to export those symbols.

### 5.2 Internal dependencies

- `celt_fir` and `celt_iir` call `xcorr_kernel` from `celt/pitch.rs`. A bug in
  `xcorr_kernel` would show up as a failure in our LPC tests. Acceptable: the
  pitch module already has its own tests.
- `silk_nlsf2a` calls `silk_lpc_fit`, `silk_lpc_inverse_pred_gain`, and
  `silk_bwexpander_32`. Testing NLSF2A implicitly tests those too.
- `silk_gains_dequant` calls `silk_log2lin`. A bug there would cascade. But
  `silk_log2lin` is tested separately, so the root cause would be obvious.

### 5.3 Fixed-point precision subtleties

Several functions use wrapping arithmetic (`as u32`, `.wrapping_add()`,
`.wrapping_sub()`) by design. Test expected values must account for this.
The `silk_lpc_analysis_filter` accumulates in `u32` and casts back to `i32` --
tests should include values that exercise the wrapping path.

### 5.4 Non-deterministic ordering in `silk_nlsf_stabilize`

The stabilization loop is iterative and path-dependent. Different initial NLSF
configurations may require different numbers of iterations. Tests should focus on
**post-condition verification** (output satisfies spacing constraints) rather than
exact intermediate values.

### 5.5 `celt/rate.rs` setup complexity

`clt_compute_allocation` requires a fully initialized `CELTMode`, an entropy
coder with a buffer, and many output arrays. The test setup is verbose.
Mitigation: write a helper function that creates a default test context.
Alternatively, focus testing effort on the simpler `get_pulses`, `bits2pulses`,
`pulses2bits` functions which give high coverage of the cache logic with minimal
setup.

### 5.6 Large state space

`silk_resampler_down2_3` has 6 state elements and processes data in batches.
Testing all state transitions is infeasible. Stick to DC-gain and length tests
as smoke tests, relying on the integration sweep for full coverage.
