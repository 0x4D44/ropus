# Testing Gaps Review

Date: 2026.05.05

Scope: repository testing strategy and risk review for `/home/md/language/ropus`.

Method: four-agent review plus local validation. The agents split the work across
the comparison harness, Rust unit/integration coverage, fixtures/fuzz/CI, and a
devil's advocate pass. I then verified the highest-signal claims locally.

## Executive Summary

The repo has a serious test base. The `ropus` crate alone currently exposes
1866 library tests, property tests are present, the C-reference harness is deep,
the conformance crate wraps upstream xiph tests, IETF vectors are integrated,
and fuzz targets exist for several high-risk surfaces.

The problem is not "too few tests" in the simple sense. The problem is that
confidence is uneven and too easy to overstate:

- Some important gates are manual rather than enforced.
- Some advertised fuzz runners are stale relative to the actual fuzz manifest.
- Some asset-dependent tests silently skip their strongest paths.
- Some oracles intentionally degrade to weaker checks for malformed SILK/Hybrid
  recovery, high-rate Hybrid attacker-controlled decode, DRED, and neural paths.
- Some harness commands can print failures without returning a failing exit code.

My bottom line: the testing is broad, but the release-confidence story needs a
harder contract. We need to distinguish exact byte parity, exact PCM parity,
SNR-gated parity, sample-count-only, smoke-only, ignored, and asset-skipped
coverage. Collapsing those into one "green" status is where bugs lurk.

## What Is Strong

  `cargo +nightly llvm-cov --branch --summary-only -p ropus --lib -- --test-threads=1`
  with 1866 passed tests, 97.18% line coverage, and 87.40% branch coverage.
- The deterministic codec modules have many focused tests in
  `/home/md/language/ropus/ropus/src/**`, including mode transitions, CTL
  validation, packet parsing, PLC state, and C-reference pinning in several
  places.
- The conformance crate wraps upstream xiph test programs and IETF vector tests,
  documented at `/home/md/language/ropus/tests/conformance/README.md`.
- The harness has extensive diagnostic commands under
  `/home/md/language/ropus/harness/src/cli.rs`.
- `full-test` is a useful local gate: quality, tests/coverage, ambisonics,
  benchmark sweep, and HTML reporting.

## Highest-Risk Gaps

### 1. Harness Results Can Be Falsely Green

`compare_bytes` and `compare_samples` compare up to `max(len)` but substitute
missing values with zero. A missing zero tail can be reported as a match:
`/home/md/language/ropus/harness/src/cli.rs:123` and
`/home/md/language/ropus/harness/src/cli.rs:150`.

The headline `ropus-compare encode`, `decode`, and `roundtrip` commands print
comparison failures but do not return non-zero on mismatch:
`/home/md/language/ropus/harness/src/cli.rs:825`,
`/home/md/language/ropus/harness/src/cli.rs:1055`, and
`/home/md/language/ropus/harness/src/cli.rs:1101`.

This is a gate-integrity issue. A human reading output may notice, but a script
can pass unless it parses stdout.

Recommendation: make `CompareStats` length-aware and make non-diagnostic
comparison commands exit non-zero on mismatch or skip.

### 2. Oracle Assets Can Disappear

The harness stubs out when `reference/` is missing:
`/home/md/language/ropus/harness/build.rs:21` and
`/home/md/language/ropus/harness/src/main.rs:12`.

The conformance crate also compiles out when the upstream reference tests are
missing:
`/home/md/language/ropus/tests/conformance/build.rs:54`, and each test file is
guarded, for example `/home/md/language/ropus/tests/conformance/tests/api.rs:1`.

Fresh-clone build ergonomics are reasonable, but a release or CI gate should not
be allowed to report conformance if the reference tree is absent.

Recommendation: add a hard release preflight: reference C source present, IETF
vectors present/provisioned, DNN weights present when claiming neural coverage.

### 3. Fuzz Runners Are Stale

The fuzz manifest has 12 active bins:
`/home/md/language/ropus/tests/fuzz/Cargo.toml:22`.

The shell runners still list an older target set, including old names such as
`fuzz_repacketizer` and `fuzz_packet_parse`, while omitting newer targets such
as `fuzz_ctl_sequence`, `fuzz_dnn_blob`, `fuzz_multistream`, and
`fuzz_decode_plc_seq`:
`/home/md/language/ropus/tools/fuzz_cycle.sh:46` and
`/home/md/language/ropus/tools/fuzz_run.sh:97`.

The fuzz crate is also excluded from the top-level workspace:
`/home/md/language/ropus/Cargo.toml:16`.

Recommendation: generate the fuzz target list from `tests/fuzz/Cargo.toml` or
make one checked source of truth. Add a cheap CI step that at least runs
`cargo +nightly fuzz build --fuzz-dir tests/fuzz` and crash replay for committed
crash corpora.

### 4. Fuzz Oracles Are Deliberately Weakened In Important Places

The shared fuzz oracle classifies recovery/DTX and high-rate Hybrid
attacker-controlled decode into weaker classes:
`/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs:160`.

That is defensible. Malformed recovery paths are hard to compare, and the
current classifier prevents false positives. But it means those paths are not
covered by strong PCM parity.

Recommendation: track every weakened oracle class as named test debt. For each
sample-count-only path, state what correctness property is no longer asserted.

### 5. DRED/DNN Coverage Is Not As Strong As It Looks

The strongest DRED payload-byte differential encode test is ignored:
`/home/md/language/ropus/harness-deep-plc/tests/dred_encode_payload_diff.rs:150`.

The crate build emits an empty DNN weights blob when reference DNN assets are
absent:
`/home/md/language/ropus/ropus/build.rs:54`.

Several weight-dependent tests then return early instead of failing or being
reported as ignored, for example:
`/home/md/language/ropus/ropus/src/dnn/dred.rs:2519`,
`/home/md/language/ropus/ropus/src/dnn/dred.rs:2617`, and
`/home/md/language/ropus/ropus/src/opus/dred.rs:376`.

DRED PCM reconstruction is explicitly still a stub returning
`OPUS_UNIMPLEMENTED`: `/home/md/language/ropus/ropus/src/opus/dred.rs:218`.
Tests pin the stub behavior at `/home/md/language/ropus/ropus/src/opus/dred.rs:535`.

The scoped coverage run confirms the gap:

- `dnn/dred.rs`: 84.60% line, 50.00% branch.
- `opus/dred.rs`: 76.47% line, 36.84% branch.

Recommendation: unignore or replace DRED payload-byte parity with a passing
scoped gate. In release validation, treat missing weights as a failed preflight
for any claim about neural/DRED coverage.

### 6. PLC/Recovery Still Has Known Drift Windows

The full decode24 PLC parity test is ignored:
`/home/md/language/ropus/harness/tests/c_ref_differential.rs:1631`.

A narrower test asserts only the first 288 samples, with the comments
documenting remaining LSB-level drift beyond that:
`/home/md/language/ropus/harness/tests/c_ref_differential.rs:3388`.

This is exactly the kind of place where a broad "PLC is tested" claim becomes
misleading. PLC is tested, but not all PLC parity is restored.

Recommendation: keep the narrow regression, but add a visible debt item and a
tiered oracle for the remaining post-288-sample region.

### 7. Real-World Corpus Is Not Populated

`/home/md/language/ropus/tests/vectors/real_world/README.md:9` says the fetch
script only produces a reference-encoded baseline, not a non-reference encoder
diversity corpus. The same file documents that non-reference samples remain
open work at `/home/md/language/ropus/tests/vectors/real_world/README.md:20`.

The `corpus_diff` limitations are explicit: family 0 only, `.webm` skipped, no
seek, no FEC, no PLC:
`/home/md/language/ropus/tests/vectors/real_world/README.md:118`.

Recommendation: populate a small pinned corpus with files from FFmpeg native
Opus, WebRTC captures, messaging voice notes, and container edge cases. Gate
empty corpus as non-green.

### 8. Randomized Differential Coverage Is Too 20 ms-Centric

Deterministic sweeps cover many frame durations, but fuzz targets commonly
hard-code `sample_rate / 50` 20 ms frames. This misses randomized combinations
of 2.5/5/10/40/60 ms with bitrate extremes, VBR/CVBR, FEC, DTX, loss, and
state transitions.

Recommendation: add a structured fuzz input dimension for legal frame duration.
Start with `fuzz_encode_multiframe`, `fuzz_roundtrip`, and `fuzz_decode_plc_seq`.

### 9. Performance Coverage Is Narrow And Warn-Only

The canonical bench vector set has 10 vectors:
`/home/md/language/ropus/full-test/src/bench.rs:40`.

It misses long frames, FEC/DTX/PLC, multistream/projection, and container
decode. Benchmark crashes/regressions are tracked as WARN-level in the runner:
`/home/md/language/ropus/full-test/src/bench.rs:104`.

Recommendation: keep perf WARN-level for local iteration, but add a release
threshold for crashes and severe regressions in codec-critical vectors.

## Poorly Tested Or Ambiguous Areas

- DRED payload-byte encode parity and final PCM reconstruction.
- DNN weight-dependent behavior on bare checkouts.
- SILK/Hybrid malformed recovery and high-rate Hybrid attacker-controlled decode.
- Full-frame decode24 PLC parity after the first 288 samples.
- Real-world containers: pre-skip, gain, granule gaps, tags, pictures, seek,
  WebM, and non-reference encoders.
- Fuzzed non-20 ms frames and long-frame state transitions.
- Multistream/projection under packet loss, FEC, DTX, and runtime setter churn.
- Platform variation: the workspace pins x86_64 builds to x86-64-v3 in
  `/home/md/language/ropus/.cargo/config.toml:10`. That is a reasonable
  performance decision, but it should not be the only platform confidence
  story.
- Mutation testing is not currently a reliable gate: `.cargo/mutants.toml`
  skips known failing tests at `/home/md/language/ropus/.cargo/mutants.toml:7`,
  and `/home/md/language/ropus/tools/mutation_test.sh:24` still references
  stale `src/...` paths rather than `ropus/src/...`.

## Where Bugs Are Likely To Lurk

1. State accumulation across frames: VBR/CVBR, FEC, DTX, bandwidth, force-mode,
   force-channel, and application transitions.
2. Recovery paths: PLC, decode-FEC/LBRR, malformed packets that return `Ok`,
   and recovery after mode or sample-rate state changes.
3. DRED and neural paths: missing weights, unimplemented reconstruction, DRED
   payload byte layout, RDOVAE/FARGAN interactions.
4. Harness/gate plumbing: stale target lists, soft skips, non-failing CLI
   mismatches, absent reference assets.
5. Containers and end-user tools: Ogg/WebM edge cases, seeking, pre-skip,
   gain/tags/pictures, stdout/stdin CLI modes.
6. Platform assumptions: x86-64-v3, SIMD/vectorized operations, sanitizer/fuzz
   behavior on Windows vs Linux.

## Devil's Advocate

The big-picture concern is not that testing is absent. It is that the project
has enough tests to create a false sense of certainty unless every result is
labeled by oracle strength.

Hard questions:

- What is the actual shared release gate? There is no `.github` workflow in the
  repo, and `full-test` is a local report runner.
- When someone says "conformance passes", does that include `reference/`,
  provisioned IETF vectors, single-threaded execution, and the right C ABI shim?
- When someone says "DNN/DRED is covered", were weights present, or did tests
  early-return?
- When fuzzing says "no findings", did every active target run, or did stale
  target lists skip the newest targets?
- Are coverage tests proving correctness, or just exercising branches with
  output-shape assertions?

Assumptions to stop making:

- High line coverage means high codec confidence.
- A passing default `cargo test` means reference-backed conformance ran.
- Fuzz targets exist, therefore fuzz campaigns exercise them.
- SNR/sample-count gates are equivalent to bit-exact parity.
- DRED parse/process coverage implies DRED reconstruction coverage.

Different framing:

Publish a testing matrix by subsystem and oracle class:

| Subsystem | Current strongest oracle | Confidence |
|---|---|---|
| Core CELT/SILK encode/decode | C-reference differential + unit/property | High, with recovery exceptions |
| Repacketizer/packet parsing | Unit/property + conformance + fuzz | High |
| Official C ABI conformance | Upstream tests, asset-dependent | High only when `reference/` present |
| IETF vectors | Spec oracle, provisioned by `full-test` | High when vectors present/provisioned |
| Fuzz campaigns | Mixed differential/safety | Medium until runners are fixed |
| DRED encode payload | Ignored differential test | Low |
| DRED PCM reconstruction | Stub pinned as unimplemented | Not covered |
| DNN weights path | Asset-dependent early-return tests | Medium with assets, low without |
| Malformed SILK/Hybrid recovery | sample-count / weakened SNR classes | Low-to-medium |
| Real-world containers | mostly unpopulated corpus | Low |
| Performance | 10-vector warn-only sweep | Medium for those vectors only |

## Recommended Plan

### Immediate Fixes

1. Fix `CompareStats` length handling and make `ropus-compare
   encode|decode|roundtrip` fail on mismatch.
2. Fix fuzz runner target lists or generate them from `tests/fuzz/Cargo.toml`.
3. Add a release preflight that fails when `reference/`, IETF vectors, or DNN
   weights are missing for gates that claim those coverages.
4. Document every weak oracle class in one matrix in the README or a dedicated
   `wrk_docs` report.

### Next

1. Add a cheap CI workflow:
   `cargo test --workspace --lib --tests -- --test-threads=1`,
   `cargo +nightly fuzz build --fuzz-dir tests/fuzz`, conformance/IETF asset
   preflight, and crash replay.
2. Unignore or replace the DRED payload-byte test with a passing scoped parity
   gate.
3. Add fuzzed legal frame-duration variation across encode, roundtrip,
   multiframe, and PLC-sequence targets.
4. Populate `tests/vectors/real_world` and run `corpus_diff` as a real gate,
   with empty corpus treated as non-green.
5. Add focused tests for the remaining decode24 PLC drift window.

### Later

1. Add platform gates: generic x86_64 in addition to x86-64-v3; at least one
   sanitizer build where practical.
2. Revive mutation testing with corrected paths and no hidden baseline skips for
   release reporting.
3. Expand performance gates to long frames, DTX/FEC/PLC, multistream/projection,
   and real-world decode.
4. Replace coverage-only tests with differential or property oracles wherever
   the C reference or a stable invariant is available.
