# Build Time Benchmark

**Date:** 2026-05-02 (Machine A), 2026-05-03 (Machine B), 2026-05-03 (Machine C)

Both runs use the same protocol: cold builds with no prior `target/` artifacts. Debug build first (`cargo build`), then release (`cargo build --release`) — no artifacts shared between profiles.

## Observations

- Debug scales close to thread count on the A→B jump (7.8× speed-up vs 3.5× more threads), suggesting the debug build is still fairly parallel-friendly across the dependency graph.
- Release scales worse (5.0×) because LLVM optimisation runs single-threaded per crate; once the parallel front of crates narrows late in the build, extra threads sit idle.
- Machine C vs Machine A: same thread count (8), but the Ice Lake i7-1065G7 is ~1.6× faster for debug and ~1.2× faster for release. The debug gain tracks the IPC and clock-speed advantage of 10nm Ice Lake (higher single-thread throughput, ~3.9 GHz boost) over the older Kaby Lake HQ. The smaller release gain confirms that LLVM codegen is the bottleneck, not the frontend — extra IPC helps less when you're serialised on LLVM.
- The earlier prediction in this doc — "a modern 8-core/16-thread machine would likely cut cold release to ~50–60 s" — was conservative. A 14-core/28-thread part lands at 29.5 s, almost 2× faster than that estimate.
- Disk footprint is essentially identical across all three machines (debug ~1.3 GB, release ~0.5–0.6 GB); minor variance is toolchain/crate-version differences.
- Incremental builds after a single-file change remain sub-second for debug and a few seconds for release on all machines — cold-build time is the only meaningful axis where hardware matters.
