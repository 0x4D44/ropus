## Observations

- Debug scales close to thread count on the A→B jump (7.8× speed-up vs 3.5× more threads), suggesting the debug build is still fairly parallel-friendly across the dependency graph.
- Release scales worse (5.0×) because LLVM optimisation runs single-threaded per crate; once the parallel front of crates narrows late in the build, extra threads sit idle.
- Machine C vs Machine A: same thread count (8), but the Ice Lake i7-1065G7 is ~1.6× faster for debug and ~1.2× faster for release. The debug gain tracks the IPC and clock-speed advantage of 10nm Ice Lake (higher single-thread throughput, ~3.9 GHz boost) over the older Kaby Lake HQ. The smaller release gain confirms that LLVM codegen is the bottleneck, not the frontend — extra IPC helps less when you're serialised on LLVM.
- Machine D vs Machine B: similar tier of part (high-end desktop, 28–32 threads), but D loses on debug (14.3 s vs 11.3 s) and wins on release (25.0 s vs 29.5 s). The debug regression is most likely the WSL2-vs-native-Windows file-IO gap — Cargo writes thousands of small artifacts and rmeta files, and NTFS is meaningfully slower than ext4 on WSL2 at that workload. The release win flips because the LLVM tail benefits from the extra threads (32 vs 28) plus Zen 5's larger per-core L2/L3 (and the 3D V-Cache stack on the X3D CCD) keeping more codegen units resident.
- Release-stage scaling is now visibly thread-bound rather than IO-bound: D beats B by ~15 % on release despite losing on debug. Once you're past the dependency front-loading, more cores still help — just not as cleanly as the debug profile because each LLVM crate is itself single-threaded.
- The earlier prediction in this doc — "a modern 8-core/16-thread machine would likely cut cold release to ~50–60 s" — was conservative. The 14-core/28-thread part lands at 29.5 s and the 16-core/32-thread Zen 5 part at 25.0 s, both well under half that estimate.
- Machine E vs Machine C: same laptop tier, but E has 2.5× the thread count (20 vs 8) and a more recent uarch (Alder Lake P-cores vs Ice Lake). Debug gains 1.7× (below the thread ratio — front-end work doesn't fully parallelise) and release gains 2.8× (above the thread ratio, because LLVM tail benefits from both extra threads and higher single-core IPC on the P-cores). E lands roughly halfway between the 8-thread laptops and the 28–32-thread desktops, which matches its hardware position.
- Disk footprint is broadly similar across all five machines (debug ~0.9–1.3 GB, release ~0.4–0.6 GB); Machine E is noticeably smaller on both profiles despite compiling more crates (138 vs 121 on A/C/D), most likely a rustc 1.93 vs 1.95 artifact-size difference combined with NTFS cluster-size variance.
- Incremental builds after a single-file change remain sub-second for debug and a few seconds for release on all machines — cold-build time is the only meaningful axis where hardware matters.
