# Porting Opus to Rust in 24 Days (And What That Sentence Actually Means)

*Draft for Substack — ~3,400 words. Honest version, with the asterisks left in.*

---

On 5 April, at about 2 AM, a file called `VICTORY.md` landed in my journal folder. An agent had just finished a session that took the comparison harness from **122 of 280 configurations passing** to **212 of 280** — ninety extra green tests in a single night, against the xiph reference C codec, byte-for-byte. The journal was triumphant. It used the word "victory." It had bullet points.

There were then **298 more commits** before the project was what I would today call — hedging carefully — shippable.

I want to tell the story of those 24 days, because I think there's a useful article in it about what multi-agent AI coding can and cannot do on a real codebase. But I also want to tell it without the puffery. The VICTORY journal was premature. A lot of the intermediate claims were premature. If you read my other journals you will find the phrase "bit-exact" used before it was strictly true, and "all 26 modules ported" before several of them could encode stereo without a panic. The method worked. It didn't work the way the tweet-length summary implies.

So: what actually happened, what actually worked, and what I would do differently if I started again tomorrow.

## The claim, with asterisks

**ropus** is a port of [xiph/opus](https://github.com/xiph/opus) — the Opus audio codec, the one in every VoIP stack and most video conferencing products — from C to Rust. The starting point was ~60–80k lines of production C. The target was bit-exact output against the reference on encode and decode, at every configuration the harness could throw at it.

The first public commit (`de682e3`, 27 March) was a 53,852-line bulk import: "22/26 modules ported." Within 24 hours another commit added the remaining four and declared the implementation phase complete. So when I say "24 days," I am counting the clock from the moment the Rust modules *compiled* — not from the moment they were *correct*. Those are different clocks, and the gap between them was the project.

At the time of writing, on 19 April:

- All 26 modules are present.
- The comparison harness passes 280/280 configurations across bitrate, channel count, frame size, and complexity level.
- The 24 IETF RFC 6716 and 8251 conformance vectors pass.
- The xiph regression test (`opus_encode_regressions.c`), which I un-stubbed this morning, passes.
- A foobar2000 component (`ropus-fb2k`) decodes real-world `.opus` files at tier-2 SNR.
- DNN modules (LPCNet, FARGAN, PitchDNN, DRED range coder) are ported and wired into the decoder PLC path, but **the neural branch does not run** — the wiring gates off at every call site, because weights, buffer types, and FFT scalar conventions disagree between the fixed-point and DNN build configurations of upstream. DNN is plumbing-only.
- OSCE has zero lines of Rust. DRED has a range coder and an HLD. Encoder is classical-only.
- Performance parity with the C reference is achieved **at `target-cpu=x86-64-v3`** (AVX2/FMA3, roughly a 2013 Haswell baseline). At older targets, ropus is 1.19–1.27× slower on encode. The C reference does runtime CPU dispatch; we don't.

If any of that feels small, consider that the xiph codebase has been developed since 2007 by some of the best DSP engineers alive. If any of it feels big, consider how much I just had to caveat.

## The method: an oracle worth more than the codec

The whole project is built on one idea, and I want to be honest that everything else is a footnote to it.

**We had a differential oracle.** The C reference is open source. The C reference is correct. The C reference is compilable. So I did the obvious thing: I wrote a Rust harness (`harness/build.rs`) that uses the `cc` crate to build 89 xiph C files into a static library, links that library into the same Rust binary as my ropus code, and exposes one function: `compare(c_encode(x), rust_encode(x))`. Both implementations run in-process. Both receive the same byte-identical input. The harness returns the first offset at which the encoded packets diverge.

That turned a port into a bisection problem. Every bug has a specific offset. Every offset points to a specific frame. Every frame came from a specific codec path. And because the oracle lives a `memcpy` away instead of across a subprocess boundary, a full test cycle runs in ~100 ms. Over the course of the project, that sub-second feedback loop is the difference between "fix thirty bugs in a session" and "fix three."

I'll come back to why this matters, because the devil's advocate on my research team kept hammering one point and it's the right one: **the oracle is the project**. A codec port with a free executable specification is not a general AI-coding result. It's a particular result about what's possible when the thing you want to build has been built already in a language you can link against. Most ambitious projects do not have this luxury. Pretend otherwise and the method evaporates.

## What the first three weeks actually looked like

The early integration phase (roughly 29 March to 4 April) was an exercise in trace-fix loops. The pattern went: pick a failing config — usually silence at 48 kHz, because silence is the easiest thing to encode — find the first byte that diverges, add matched instrumentation at the same logical point in C and Rust, run both, diff the output, binary-search upstream or downstream until the divergence collapses to a single function, compare the two implementations line by line, fix, run the sweep, commit.

Eight of my commits in this period have the subject line "Trace-fix iteration 3." Not three iterations total — three was just the number I happened to put on the label, and then it stuck, and I kept using it. One of the fair criticisms from the devil's advocate is that 17 near-identical commit messages in a row is a signal that the coordinator was thrashing rather than progressing. I think that's half-right. Some of those commits landed real fixes and some landed partial fixes that had to be un-picked later. The bundling was sloppy.

The bugs that defined this phase:

**NSQ divergence.** The SILK noise-shaping quantizer was producing Q14 LPC predictions that were off by exactly 16 on the first voiced subframe. The symptom was in the quantizer. The cause was five layers upstream: `silk_quant_ltp_gains` was being called without the cross-correlation vector, so the VQ was optimising the wrong cost function, picking different codebook vectors, contaminating the history buffer, re-entering prediction. Six cascading fixes. A week of instrumentation.

**CELT prefilter.** The MDCT looked wrong on short blocks. The MDCT was fine. The prefilter upstream was skipping the comb filter whenever `gain1 == 0` — in C, the filter is always applied (to crossfade smoothly between gain transitions). We also weren't writing the prefilter memory buffer at all; silence passed because silence in gives silence out regardless of state. Six distinct bugs in `run_prefilter`. This is the bug that taught us that "passes silence" means nothing.

**The dynalloc session.** 4 April: I dispatched seven named agents in parallel (dynalloc-fixer, devil's-advocate, silk-investigator, spread-auditor, dynalloc-boost-auditor, trim-fixer, transient-fixer) against a cluster of related issues. They found 25+ bugs and flipped 14 tests in a day. They also stepped on each other's changes enough that the session notes include a plaintive "Need better isolation between sessions." The fix — git worktrees per agent — came four days later.

## The pivot

This is the part of the story I most want the article to get right, because it is the *only* part of the story that generalises beyond codec porting.

By 5 April, every DSP function we'd audited was bit-exact in isolation. Unit tests passed. Instrumented diffs showed matching values at every checkpoint we'd placed. And yet the sweep kept failing. The compositions were wrong even though the parts were right. That shouldn't have been possible.

I dispatched a devil's-advocate agent with the explicit brief of arguing against the current hypothesis. The current hypothesis was "there's still an arithmetic bug somewhere; we haven't found the right ±1 yet." The DA's counter-hypothesis: **the bugs aren't arithmetic. They're constants and state initialisation. Stop looking in hot functions. Look at the macros that produce the numbers they consume.**

That hypothesis cashed out within hours.

The largest single unlock of the project was a one-line change. The C macro `QCONST32(0.70710678f, 31)` — with a trailing `f` — evaluates in `f32`. My Rust `qconst32(0.70710678, 31)` evaluated in `f64`. The two produce `1518500224` and `1518500247` respectively, as `i32` Q31 values. A delta of 23 counts in Q31 is imperceptible in any individual sample but it rides through every Haar transform in the stereo encoder, which runs inside an iterative rate-distortion loop, which uses it as a decision threshold. Flipping that constant to `f32`-derived flipped **31 tests green** (`22d281e`). Six more `f32` vs `f64` bugs surfaced in the same sweep.

The next unlock was also a one-line change. `silk_find_lpc_fix` had a slice bound of `2*subfr_length - d` where the C reference used `2*subfr_length`. Twenty-one tests. The next after that: `lsb_depth` was stuck at 24 instead of 16 because the encoder never propagated it to CELT — sixteen tests and an 8 dB noise-floor error.

The lesson here is portable. When everything that should work does work and the composition still fails, the bug is not in the functions. It's in **the constants feeding them, the state between them, or the plumbing around them**. I would have arrived at this eventually. I arrived at it sooner because an agent whose job was specifically to disagree with me did disagree with me, and was right. That pattern — a designated skeptic with institutional cover to say "you're looking in the wrong place" — turned out to be the single most valuable piece of the multi-agent setup.

## What "multi-agent" actually looked like

I want to be careful here because the phrase attracts puffery. Let me describe what happened, not what I had planned.

I started with a coordinator (`tools/coordinator.py`) that encoded a five-phase pipeline: `document → HLD → test harness → implement → integrate`. The coordinator dispatched Claude (implementation) and Codex (adversarial review), ran modules in dependency waves, and kept state in a JSON file. It enforced gates via string-matching: if Codex's review contained "CRITICAL," re-dispatch Claude with the review appended. Up to 10 fix attempts per module.

The coordinator got the modules ported. It did not get them correct. From about 3 April onward, almost no useful work happened through `coordinator.py`. What replaced it was hand-orchestrated — me as team lead, dispatching agent teams for specific bugs — and a doc taxonomy that crystallised out of the `wrk_docs/` folder:

- **BRIEF** — I wrote these. Scope statement, acceptance criteria.
- **HLD** — the agent wrote a high-level design before any code. Often V1 then V2.
- **CR** (code review) + **DA** (devil's advocate) — parallel reviews with different postures. CR is structured (BLOCKER / IMPORTANT / NIT). DA is adversarial (REAL / POSSIBILITY).
- **PLN** — multi-stage plans for coverage or fuzz campaigns.
- **REPORT** — the post-mortem.
- **JRN** — the working journal, kept live during the session.

A typical mature-workflow session would run HLD → CR+DA in parallel → HLD V2 addressing the review findings → implement → CR again on the diff → journal close. The HLD V1 → V2 jumps were real content, not cosmetic revisions. One example I keep pointing to: a V1 HLD for the SILK rate-control fix put `nBitsExceeded` in the *risks* section. Overnight, a three-agent review team flagged a much worse problem I'd missed entirely — my encoder was passing a **zeroed second-channel buffer** to `quant_all_bands`, so every stereo test was doomed regardless of whatever rate control did. V2 reordered priorities: fix the zeroed Y buffer first. Same problem, completely different plan.

The team patterns I used most:

- **Parallel specialists.** 3–7 agents with distinct roles — implementer, reviewer, devil's advocate, domain investigator, feedback applier, verifier. Each gets one responsibility. Dispatch in parallel, aggregate findings, decide.
- **Worktrees for isolation.** After the dynalloc session taught me that shared working directories don't scale, each agent ran in its own git worktree on its own branch. Merge at the end, sequentially.
- **Supervisor mode, late.** By the final week, I was explicitly instructing Claude to *not code* and only dispatch sub-agent teams. This worked better than direct coding once the patterns were established.

## Where multi-agent made things worse

I owe this section honestly, because I've seen too many AI-development write-ups that narrate only the wins.

**Plan duplication.** On 8 April the `wrk_docs/` folder sprouted *six* near-duplicate coverage-improvement plans, written by different agents who didn't know the others existed. The symptom was a missing master-plan artifact; the cause was dispatching without first establishing a canonical doc. I eventually consolidated them by hand.

**Silent regressions from refactors.** The saddest one: commit `5abd1a9` was a "remove feature flags" refactor. It also silently deleted the DNN integration that had been wired in 11 days earlier. Nobody noticed for a week and a half, until a supervisor audit on 19 April found that the neural decoder path was not reachable. A multi-agent workflow should make this sort of regression impossible to miss — the DA should catch it, the reviewer should catch it, the test suite should catch it. In my case, none of them did.

**Test ping-pong.** During parallel sessions, one agent would land a state-initialization fix and flip 14 tests green, and another agent in a concurrent session would unknowingly revert it while fixing something else, and the sweep counter would oscillate 196 → 212 → 126 → 208 across commits. The worktree fix helped. It didn't eliminate the problem.

**The oracle was never in the mutation loop.** My agent running the mutation-testing pilot was honest enough to put this in the post-mortem: `cargo-mutants` only triggers `cargo test`, but my 280-configuration C-differential harness is a `[[bin]]`, not a test. So the 85.2% mutation score I was quoting in intermediate reports measured *unit-test* sensitivity while the actual correctness authority — the C oracle — sat outside the loop. An agent caught this. A human reviewing agent output could easily have missed it. I'm not sure I would have.

**Trace-fix thrash.** Those 17 iteration-3 commits look, from the outside, like the coordinator thrashing. Some of them were. Multi-agent parallelism can turn a sequential bug into a concurrent one, and concurrent bugs are harder. When the agents are stepping on each other, more parallelism makes the problem worse, not better.

**Cost.** I did not track token spend rigorously, but one journaled data point: a single overnight fuzz-campaign cycle consumed 465K tokens and 440 tool uses across 73 minutes of wall time. Across nine overnight campaigns and dozens of bug-fix sessions, the total was almost certainly substantial. My estimate for the project in API dollars is low thousands, not tens — but I haven't audited, and the reader should not trust my estimate. This is the part of the write-up I'm most uneasy about. If you're considering the method for your own project, assume it costs real money before assuming it saves time.

## The long tail

Every time I told myself the project was done, it wasn't.

VICTORY on 5 April was "280/280 on the harness." The IETF conformance vectors (24 of them) took another week. The xiph regression tests (historical crash repros) took another. On 18 April — two weeks after VICTORY — the official xiph `test_opus_encode.c` cross-check uncovered **five named bugs** in encoder bookkeeping (labeled K, M, N, O, P in my notes). The encoded bitstream was correct in every case. Both decoders agreed on it. But the encoder's internal `get_final_range()` value — which decoders cross-check against their own computed range — diverged, because I was clobbering a redundancy range at the wrong offset, or missing a single-line `redundant_rng = celt.rng;` before a state reset.

These are the worst kind of bugs. The bytes are right. The decoders don't notice. The only oracle that fails is a self-consistency check three layers inside the encoder. You can be byte-exact on outputs and still wrong on state, and if your harness doesn't test that specific self-check, you will ship a codec that silently fails someone else's test.

On 19 April — yesterday in this article's timeline, today as I write — **Bug L** in campaign 9 was a `usize` underflow panic in the freshly-ported analysis module. Re-entry into the analysis path meant re-entering a bug that had been latent for exactly as long as the analysis path had been disabled.

The uncomfortable truth: a port is never done. There is always another test suite, another fuzz target, another cross-check the original authors added in 2019 that you haven't found yet. What you have is a declining defect rate and a shipping decision. The shipping decision is a judgement call, not a boolean.

## What I would do differently

1. **Start the differential harness before I start the port.** I built it in week one. It should have been commit one. Every bug I found after VICTORY was a bug that bisection against the harness could have surfaced on the day the relevant code was written. Front-loading the oracle would have compressed this into something closer to ten days.

2. **Compile to the oracle from the first module, not the first architecture.** I ported bottom-up by dependency and waited to link against the C reference until the harness phase. I should have had a smoke-test differential wrapper around *each module* as it landed, even if trivial. Spend the week-one compile-and-link tax once; save it back twelve times over.

3. **Write V2 HLDs before implementing.** My best sessions did this. My worst didn't. A V1 HLD reviewed by a CR agent and a DA agent in parallel, revised to V2, and *only then* implemented, had a hit rate an order of magnitude better than "write a plan, implement it, review after." The overhead is real. The rework cost of skipping it is larger.

4. **Be suspicious of refactors.** My worst regression came from a "clean up feature flags" commit that silently deleted weeks of wiring. For any refactor touching integration surface, I would now require: a DA agent whose only job is "list what this commit could be silently deleting," and a test that reaches every branch it claims to preserve.

5. **Track cost from day one.** I don't know what the project cost me in tokens. Future me wants to know this. Future reader of this article deserves to know it too.

6. **Rename my commits.** Seventeen "Trace-fix iteration 3" commits is a tooling failure. Each one should say what it changed. The tooling pain (writing commit messages is work) is worth it on week two when you're trying to reconstruct what happened.

7. **Don't redefine the oracle without a flag day.** Stages 6–8 introduced a "tier 2" acceptance — SNR > 60 dB rather than bit-exact — because cross-platform floating-point determinism on DNN paths isn't reachable. That's a reasonable engineering call. But it was introduced quietly in an HLD, not promoted to a project-wide decision, and "bit-exact with C" now means different things in different stages. Next time, if the oracle has to change, it gets an ADR and a flag day.

## The question I had to ask myself

Before I posted this, I made myself answer one uncomfortable question.

**Would I still be writing this article if the xiph C reference did not exist?**

The honest answer is: I wouldn't have attempted the project. The differential oracle is not a nice-to-have; it is the spine of the method. Every bug I found, every trace-fix loop, every HLD V2, every devil's advocate redirect — all of them were made tractable by the fact that "correct" had an executable definition living one `cc` build away.

That matters for how to read the rest of the article. The headline a casual reader will take from "multi-agent AI ported 36k lines of production codec in 24 days" is that multi-agent AI has achieved a new level of autonomous competence. I think that reading is wrong. The sharper framing — the one I believe — is:

**Multi-agent AI, given an executable oracle and a skeptical human in the loop, can execute an ambitious translation against that oracle at a pace that would have been implausible two years ago.**

That's a narrower claim. It's still a consequential one. But the next person who reads this article and tries to apply the method to a problem *without* an oracle will find that most of the leverage is gone. Generative work without a way to check correctness at scale is a different problem, and I don't think the methods in this article solve it.

If you take one thing away: go find your oracle first. If there isn't one, build one. If you can't build one, pick a different problem.
