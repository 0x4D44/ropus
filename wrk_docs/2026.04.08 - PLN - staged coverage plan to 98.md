# Coverage Improvement Plan

Date: 2026-04-08

## Baseline

Measured from the current full repo branch-aware coverage run:

- lines: `91.45%`
- functions: `92.76%`
- regions: `92.09%`
- branches: `76.57%`

Distance from 98%:

- lines: `2,282` more covered lines
- functions: `101` more covered functions
- regions: `3,086` more covered regions
- branches: `1,224` more covered branches

## Strategy

Do not chase raw test count. Chase cold execution in the files that dominate missed lines and branches.

The stages below are ordered by expected return per hour, not by module prestige.

## Stage 0: Keep The Runner Honest

Goals:

- keep `cargo +nightly llvm-cov --branch --workspace --all-features` green
- fix any coverage-run regressions immediately
- regenerate JSON after each wave and re-rank hotspots from fresh data

Exit criteria:

- tests pass
- coverage report generates without manual cleanup

## Stage 1: Deterministic Medium-Cost Wins

Primary targets:

- `src\dnn\lpcnet.rs`
- `src\opus\multistream.rs`
- `src\opus\decoder.rs`
- `src\opus\repacketizer.rs`

Why:

- these files still contain hundreds of missed lines and meaningful branch debt
- they have better setup-to-payoff ratio than the largest encoder state machines
- unit tests can target packet parsing, layout validation, state resets, and helper transitions without destabilizing parity-sensitive paths

Success signal:

- line coverage moves noticeably
- branch coverage starts moving faster than test count

## Stage 2: Packet And Decoder Branch Closure

Primary targets:

- `src\opus\decoder.rs`
- `src\opus\repacketizer.rs`
- `src\silk\decoder.rs`
- `src\celt\decoder.rs`

Test themes:

- malformed packet shapes
- self-delimited and multi-frame boundary cases
- PLC/FEC fallbacks
- small-buffer and invalid-argument wrappers
- decoder state transitions that only occur on specific packet sequences

Success signal:

- branch coverage in these files climbs into the mid/high 80s

## Stage 3: Large Encoder State Machines

Primary targets:

- `src\silk\encoder.rs`
- `src\opus\encoder.rs`
- `src\opus\multistream.rs`

Why:

- these modules dominate the remaining uncovered totals
- 98% is not realistic without direct progress here

Test themes:

- bandwidth and mode transition edges
- bitrate, DTX, FEC, redundancy, and force-mode interactions
- low-budget and constrained frame-size branches
- initialization/reuse paths that toggle internal setup state

Success signal:

- repo-wide branch coverage keeps rising by material fractions of a point after each wave

## Stage 4: Residual High-Leverage Cleanup

Primary targets:

- `src\celt\encoder.rs`
- `src\celt\bands.rs`
- `src\dnn\lpcnet.rs`
- `src\silk\common.rs`

Why:

- these files are smaller than the big encoder modules but still hold enough missed branches to matter
- they are good parallel-agent work once the higher-value packet and encoder surfaces have been exercised

## Stage 5: Stop-And-Reassess Rule

Continue only while a wave produces material movement in one of the top hotspot files.

Pause and reassess if:

- a full wave buys only trivial movement
- new tests become brittle and assertion-heavy
- remaining misses require behavior-changing hooks instead of realistic black-box tests

## Immediate Next Wave

1. Add targeted tests in `dnn::lpcnet`, `opus::multistream`, `opus::decoder`, and `opus::repacketizer`.
2. Re-run full branch-aware coverage.
3. Re-rank hotspots from the new JSON.
4. Decide whether the next best return is another packet/control wave or an explicit push into `silk::encoder` and `opus::encoder`.
