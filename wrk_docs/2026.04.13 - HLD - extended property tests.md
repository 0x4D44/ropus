# HLD: Extended Property Tests — Stateful Sequences

**Date**: 2026-04-13
**Goal**: Add 4 property tests covering stateful sequence invariants that the existing single-frame property tests cannot catch.

## Context

Phase 3 delivered 14 proptest functions (7 packet format, 7 codec roundtrip). All test single-frame operations. This extension adds tests for multi-frame sequences, FEC recovery, mid-stream config transitions, and repacketizer integrity.

All changes go in the existing `src/property_tests_codec.rs`. No new files.

## New helpers (2 functions)

### `encode_sequence`
```rust
fn encode_sequence(
    enc: &mut OpusEncoder,
    frame_size: i32,
    channels: i32,
    n_frames: usize,
    pcm_seed: i32,
) -> Vec<Vec<u8>>
```
Encodes `n_frames` of patterned PCM (using `crate::coverage_tests::patterned_pcm_i16` with rotating seeds). Returns all packets.

### `decode_sequence`
```rust
fn decode_sequence(
    dec: &mut OpusDecoder,
    packets: &[Option<&[u8]>],
    frame_size: i32,
    channels: i32,
) -> Vec<Vec<i16>>
```
Decodes a sequence of packets. `None` entries trigger PLC (packet loss concealment). Returns all PCM frames.

## New property tests (4 functions)

### 1. `prop_sequence_invariants` (100 cases)

**What**: Encode 20 consecutive frames, decode all of them, check per-frame and aggregate invariants.

**Strategy**: `config_strategy()` x `bitrate in 16000..=128000` x `seed in 0..10000`

**Invariants checked**:
- Every packet is parseable (`get_nb_frames` succeeds, returns > 0)
- Every packet size is in 2..=1275 bytes
- Every decode returns exactly `frame_size` samples
- No frame-to-frame energy explosion: `energy[i] < 8 * max(energy[i-1], 100.0)` (allows transients but catches runaway divergence)

**Why**: State accumulation bugs (filter memory drift, reservoir overflow, prediction state corruption) only manifest after multiple frames. Single-frame tests structurally cannot catch these.

### 2. `prop_fec_recovery` (100 cases)

**What**: Encode 10 frames with FEC enabled, simulate loss of one frame, recover via FEC decode, verify graceful recovery.

**Strategy**: `config_strategy()` (filtered to SILK/Hybrid only — CELT doesn't produce FEC) x `bitrate in 16000..=64000` x `loss_frame in 3..=7` x `seed`

**Setup**:
- Encoder: `set_inband_fec(1)`, `set_packet_loss_perc(25)`
- Only use SILK-capable configs (fs <= 16000 or Hybrid). Filter out CELT-only and RESTRICTED_LOWDELAY configs since they don't produce FEC data.

**Sequence**:
1. Encode 10 frames
2. Decode frames 0..loss_frame normally
3. Skip frame `loss_frame` (don't decode — simulates loss)
4. Decode frame `loss_frame+1` with `decode_fec=true` (recovers lost frame)
5. Decode frame `loss_frame+1` again normally
6. Decode remaining frames normally

**Invariants**:
- FEC decode returns `Ok(frame_size)` — doesn't crash or error
- Normal decode after loss returns `Ok(frame_size)`
- Energy of FEC-recovered frame > 0 (not silence — we're recovering a non-silent frame)
- No decode after recovery produces energy > 10x the average pre-loss energy (no explosion)

**Why**: FEC is a critical reliability feature in real deployments. The code paths for FEC encoding (in SILK) and FEC decoding are complex and interact with prediction state. A bug here causes audible glitches under packet loss.

### 3. `prop_config_transition_no_crash` (100 cases)

**What**: Encode 5 frames, change one encoder parameter, encode 5 more frames, decode all 10, verify no crash or corruption.

**Strategy**: `config_strategy()` x `bitrate in 16000..=128000` x `transition_type in 0..=2` x `seed`

**Transitions** (selected by `transition_type`):
- 0: Bitrate change — `set_bitrate(new_bitrate)` where `new_bitrate` is current bitrate * 2 (capped at 256000) or / 2 (floored at 6000)
- 1: Bandwidth change — `set_bandwidth(OPUS_BANDWIDTH_NARROWBAND)` (downshift to NB)
- 2: VBR toggle — `set_vbr(1 - current_vbr)` (flip between VBR and CBR)

**Invariants**:
- All 10 encodes succeed (no error return)
- All 10 packets are parseable
- All 10 decodes return `Ok(frame_size)`
- Energy of frame 6 (first after transition) is bounded: < 10x energy of frame 5

**Why**: Real Opus deployments (WebRTC, VoIP) constantly adapt bitrate and bandwidth based on network conditions. The encoder must handle mid-stream transitions gracefully. Internal state (prediction filters, rate control reservoir) must absorb the change without crashing or producing garbage.

### 4. `prop_repacketizer_roundtrip` (100 cases)

**What**: Encode N single-frame packets, combine via repacketizer, verify combined packet is valid and decodable.

**Strategy**: `config_strategy()` x `bitrate in 16000..=128000` x `n_frames in 2..=6` x `seed`

**Sequence**:
1. Encode `n_frames` individual packets (each is a single-frame Opus packet)
2. Create `OpusRepacketizer::new()`, `cat()` each packet
3. Call `out()` to get combined packet
4. Parse combined packet: `get_nb_frames` should return `n_frames`
5. Decode the combined multi-frame packet
6. Decode each individual packet separately, concatenate PCM
7. Compare: combined decode == concatenated individual decodes (sample-exact)

**Invariants**:
- `cat()` returns `OPUS_OK` for each packet
- `out()` returns a positive byte count
- `get_nb_frames` on combined packet returns `n_frames`
- `get_nb_samples` on combined packet returns `n_frames * samples_per_frame`
- Decoded PCM from combined packet matches concatenated individual decodes

**Why**: The repacketizer is used in multistream encoding and by applications that batch frames. A bug in frame concatenation or TOC rewriting would produce packets that parse correctly but decode to wrong audio — a silent corruption that only shows up in production.

## What's NOT in scope

- Full state machine exploration (that's what fuzzing is for)
- Multistream surround (complex setup, low incremental value)
- Frame size transitions within a stream (valid Opus but rare in practice)
- Getter/setter round-trip consistency (trivial, low bug probability)
