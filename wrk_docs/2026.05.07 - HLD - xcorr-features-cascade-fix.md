## 1. Problem statement

After the six f32→f64 spotting fixes shipped today, 27 of 36 DRED LPCNet
features are bit-exact vs the C reference on the 440 Hz fixture. Residual
drift is confined to the two pitch-derived features:

| Feature | Locked bound today | Source per prior diagnostics |
|---------|-------------------:|------------------------------|
| 18 (`dnn_pitch`)  | 9.894371e-6 | xcorr_features cascade |
| 19 (`frame_corr`) | 8.34465e-7  | xcorr_features cascade |

These feed straight into the RDOVAE encoder. Closing them is the last
hop to all-36 bit-exactness on this fixture and removes the only source
of "noise floor" in DRED reconstruction.

## 2. Investigation findings

### Per-frame instrumentation

I added a temporary set of shims (`if_features`, `xcorr_features`,
`lp_buf`, `exc_buf`, `dnn_pitch`) to `harness-deep-plc/dred_encode_shim.c`
plus a paired `temp_xcorr_diag.rs` test, ran it on the 440 Hz fixture for
8 frames, then removed all instrumentation. Observed:

```
=== frame 0 (latents_fill=1) ===
  if_features  bit-exact
  xcorr_feats  first/max: idx=197 c=-2.6591e-1 r=-2.6591e-1 |d|=7.45e-7
  lp_buf       first/max: idx=247 c=-7.8998e+2 r=-7.8998e+2 |d|=3.36e-3
  exc_buf      first/max: idx=244 c=-1.8377e+2 r=-1.8377e+2 |d|=1.83e-3
  dnn_pitch    |d|=4.77e-6
=== frame 5 (latents_fill=6) ===
  ... lp_buf |d|=3.40e-3, dnn_pitch |d|=9.89e-6   <- max-drift frame
```

Pattern: `if_features` are bit-exact every frame; `lp_buf` and `exc_buf`
already show ~3e-3 absolute drift on the first frame at signal magnitude
~80, i.e. ~300 ULP relative. Drift then propagates into `xcorr_features`
(~1e-6 after normalisation by `ener_norm`) and into `frame_corr` via
`xx, yy, xy` = inner-prods over `lp_buf`.

### Data dependency walk

`if_features` (FFT-only) is bit-exact, ruling out the spectral path.
The drifting outputs all flow through one shared upstream step:
`celt_fir_f32` (`ropus/src/dnn/lpcnet.rs:1051-1059`), called at
`lpcnet.rs:1697-1703` to fill `lp_buf[PITCH_MAX_PERIOD..]` from
`x_fir = pitch_mem ++ aligned_in`. Everything downstream of the FIR
(pitch pre-filter, biquad, xcorr loop, pitchdnn, frame_corr) is either
scalar-in-order or a normalisation that masks but does not introduce
drift.

### Side-by-side FIR comparison

Rust `celt_fir_f32` (`ropus/src/dnn/lpcnet.rs:1051`):

```rust
for i in 0..n {
    let mut sum = x[ord + i];
    for j in 0..ord {
        sum += num[j] * x[ord + i - j - 1];   // num[0], num[1], ..., num[ord-1]
    }
    y[i] = sum;
}
```

C reference `celt_fir_c` (`reference/celt/celt_lpc.c:146-192`) for
float build (no `FIXED_POINT`) routes through `xcorr_kernel_c`
(`reference/celt/pitch.h:65-129`), with `rnum[k] = num[ord-1-k]`
pre-reversed (`celt_lpc.c:159-160`). For each output index `i`,
`sum[0]` accumulates as:

```
sum = x[i]                                    // matches Rust
sum += rnum[0]*x[i-ord]   = num[ord-1]*x[i-ord]
sum += rnum[1]*x[i-ord+1] = num[ord-2]*x[i-ord+1]
...
sum += rnum[ord-1]*x[i-1] = num[0]*x[i-1]
```

i.e. C accumulates **`num[ord-1]` first, `num[0]` last**. Rust
accumulates **`num[0]` first, `num[ord-1]` last**. Same operands,
**reversed summation order**. For `LPC_ORDER = 16` the chain is 17
adds long; in float arithmetic, reversing the order easily produces
~100 ULP drift on values of magnitude ~80, exactly matching the
observed `lp_buf` drift of ~3e-3.

The fixed-point Rust `celt_fir` (`ropus/src/celt/lpc.rs:227`) already
does it correctly: it reverses `num` into `rnum` and uses
`xcorr_kernel`. Only the float-domain helper diverges.

## 3. Root cause

Confidence: **high.**

`celt_fir_f32` at `ropus/src/dnn/lpcnet.rs:1051-1059` accumulates the
FIR in `num[0], num[1], ..., num[ord-1]` order. The C reference
(`reference/celt/celt_lpc.c:146-192` ↦ `reference/celt/pitch.h:65-129`,
selected via the float build's `xcorr_kernel` path) accumulates in
**reversed** order, `num[ord-1], num[ord-2], ..., num[0]`. The
operands and operations are identical; only the floating-point
addition order differs, which is non-associative in f32 and produces
~300 ULP drift on `lp_buf`. Everything downstream (pitch pre-filter,
biquad, xcorr, pitchdnn, frame_corr) faithfully propagates that drift.

This was not caught by the prior six fixes because they all targeted
f32-vs-f64 narrowing. Summation order is a different class of
divergence — the type signatures are identical, only the parenthesis
order differs.

## 4. Proposed fix

One change in `ropus/src/dnn/lpcnet.rs`:

```rust
fn celt_fir_f32(x: &[f32], num: &[f32], y: &mut [f32], n: usize, ord: usize) {
    // Match reference/celt/celt_lpc.c:159-189 — pre-reverse `num` into
    // `rnum` and accumulate as rnum[0]*x[i-ord] + rnum[1]*x[i-ord+1] +
    // ... + rnum[ord-1]*x[i-1] (i.e. num[ord-1] first). Float addition is
    // non-associative; the C summation order is part of the bit-exact
    // contract. The fixed-point sibling `celt::lpc::celt_fir` already
    // does this via `xcorr_kernel`.
    let mut rnum = vec![0.0f32; ord];
    for k in 0..ord {
        rnum[k] = num[ord - 1 - k];
    }
    for i in 0..n {
        let mut sum = x[ord + i];
        for j in 0..ord {
            sum += rnum[j] * x[i + j];
        }
        y[i] = sum;
    }
}
```

Notes:
- The 4-at-a-time unrolled `xcorr_kernel_c` reduction in C does not
  change the order in which any single `sum[k]` receives its
  accumulations relative to the scalar tail; sum[0] in the kernel and
  sum in the scalar tail both fold rnum[0..ord-1] left-to-right against
  the same y subsequence. Therefore the simple scalar form above is
  bit-exact with the C output for `FRAME_SIZE % 4 == 0` and
  `LPC_ORDER % 4 == 0` (both true here: 320 % 4, 16 % 4).
- No need to mirror the unrolled kernel itself — it matters only for
  vectorisation, not for numerics.
- The `vec!` allocation is cheap (16 floats per frame). If profiling
  ever flags it, swap to a stack `[f32; LPC_ORDER]`; out of scope here.

## 6. Risk analysis

`celt_fir_f32` is private to `dnn/lpcnet.rs` and called only from
`compute_frame_features` for the LPC residual stage feeding
`lp_buf`/`exc_buf`. It is **not** shared with FARGAN, OSCE, PitchDNN
inference, the encoder pitch detection, or SILK. The fixed-point
`celt::lpc::celt_fir` (used by CELT) is a separate function and
already correct.

Downstream impact: every consumer of `lp_buf`/`exc_buf` (pitch
pre-filter, biquad, `compute_pitchdnn` inputs, `frame_corr`) receives
bit-exact values, which is the desired outcome. The two consumers
that derive from `lp_buf` directly — frame_corr inner_prods at
`lpcnet.rs:1762-1777` and the pitchdnn xcorr conv — both gain
bit-exactness.

Regression vector: minimal. The change only reorders f32 additions
to match the C reference; the C reference is the oracle. No
production callers depend on the *current* (incorrect) summation
order — the locked bounds today exist precisely because of this
divergence.

## 7. Out of scope

- Vectorising `celt_fir_f32` to mirror the unrolled `xcorr_kernel_c` —
  the scalar form is bit-exact with the C output, so vectorisation is
  a perf concern, not correctness.
- Refactoring `celt_fir_f32` into the shared `celt::lpc` module — the
  fixed-point and float versions diverge in saturation/rounding
  behaviour, so unifying them is a separate cleanup.
- Tier-2 SNR tightening on DEEP_PLC, FARGAN, OSCE — those are upstream
  of DRED features and unaffected by this fix.
- The `dred_lpcnet_feature_drift` per-frame counter
  (`MAX_DRIFTING_FRAME_COUNT = 50`) — should naturally go to 0 after
  the fix; no separate work.
