## Brittle Assumptions

1. Hidden baseline skips in `/home/md/language/ropus/.cargo/mutants.toml:7-8`
   make mutation scores hard to interpret.

   The config currently adds:

   ```text
   --lib -- --skip dnn::lpcnet::tests::test_compute_frame_features
   --skip dnn::lpcnet::tests::test_plc_load_model
   --skip opus::encoder::tests::test_encode_multiframe_silk_special
   ```

   One skipped name appears stale as a test filter:
   `test_compute_frame_features` is now a method name/comment, not a matching
   test function found by `rg`. The other two are broad substring filters that
   match current tests such as `test_plc_load_model_invalid_blob_returns_error`,
   `test_plc_load_model_accepts_encoder_blob_but_not_fargan_blob`, and
   `test_encode_multiframe_silk_special_frame_sizes`. If those tests now pass,
   mutation runs still hide them.

2. The runner is unit-test-only by default.

   `/home/md/language/ropus/harness/Cargo.toml` exposes comparison tools such as
   `ropus-compare` and `corpus_diff` as binaries. They are not automatically
   exercised by `cargo mutants` when the test command is just `cargo test`.
   That repeats the caveat from the April pilot: mutation scores measure unit
   test sensitivity, not full C-reference oracle sensitivity.

3. The runner does not require `--package ropus`.

   In this workspace, source files from the codec crate are listed as
   `ropus/src/...`. A package-scoped invocation is clearer and avoids accidental
   future mutation of `full-test`, `harness`, CLIs, or helper crates.

4. Zero-mutant runs are not treated as failures.

   This is the most important immediate false-green risk. The current script can
   report a complete run with zero mutants because both inventory and execution
   paths accept the stale glob silently.

5. Result parsing assumes old textual output lines beginning exactly with
   `CAUGHT`, `MISSED`, `TIMEOUT`, and `UNVIABLE`.

   `cargo-mutants` supports JSON for `--list`, but the script parses run logs by
   grep. That may be acceptable for a local periodic tool, but the summary
   should include the raw `mutants.out` or tool report path rather than treating
   the grep summary as authoritative.

6. The default module set is high cost for a first revival.

   `ropus/src/silk/common.rs` alone lists 1199 mutants. The old default set is a
   broad overnight campaign, not a minimal implementation validation. A first
   revived lane should prove the contract on a smaller module.

7. The script's usage assumes executable permission.

   `timeout 30 ./tools/mutation_test.sh --list` fails with permission denied.
   Either make it executable in the implementation slice or document invocation
   as `bash tools/mutation_test.sh`.

## Recommended Minimal Implementation Scope

Implement one narrow periodic mutation-audit lane. Do not wire it into default
`full-test` or release preflight.

Minimum useful scope:

1. Update the runner to target the current crate layout.

   Normalize short paths to `ropus/src/<module>.rs`, accept full
   `ropus/src/...` paths, and pass `--package ropus` to `cargo mutants`.

2. Replace hidden global baseline skips with explicit baseline reporting.

   Recommended behavior:

   - Run or expose a baseline command before mutation.
   - If baseline fails, fail the mutation audit as `baseline_failed` and record
     failing tests.
   - If Arthur wants temporary skips retained for a local investigation, move
     them behind an explicit option such as `--allow-baseline-skips`, print them
     in the summary, and mark the result `skip-shaped`.

3. Add a zero-mutant guard.

   Any selected module that produces zero mutants should fail before a mutation
   run starts. This directly prevents the current stale-path false green.

4. Keep the first mutation audit unit-test-only, but label it honestly.

   The HLD can later design a harness-backed mutation wrapper, but the minimal
   implementation should first make the existing `cargo test` lane current and
   interpretable. The report must say `unit-test-only; C reference binaries not
   in loop`.

5. Scope the validation run to one small module.

   Recommended first target:

   ```text
   /home/md/language/ropus/ropus/src/celt/math_ops.rs
   ```

   Rationale: it has prior mutation baseline history, currently lists 230
   mutants, and is small enough for an audit without becoming an overnight
   campaign. `types.rs` is also feasible at 285 listed mutants. Avoid
   `silk/common.rs` for the implementation smoke because it currently lists
   1199 mutants.

6. Record output as an audit artifact, not a gate.

   Keep raw logs under `/home/md/language/ropus/logs/mutation-testing/`, and
   write a small markdown summary under `/home/md/language/ropus/wrk_docs/` only
   after a real scoped run has completed and survivors have been triaged.
