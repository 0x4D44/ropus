## Interpretation

The repo has three different coverage problems, and they should not be treated the same way:

1. `silk::encoder` and `opus::encoder` are the dominant debt. They are large, branch-heavy, and expensive to test because most uncovered paths are stateful control logic.
2. `dnn::lpcnet` is a line-heavy hole with some branch debt. It is a good candidate for lower-risk direct unit tests because much of the surface is deterministic helper logic and state transitions.
3. `opus::repacketizer`, `opus::decoder`, `opus::multistream`, `silk::decoder`, and `celt::bands` are branch-cleanup targets. They are not the largest files, but they have enough missed decision logic to matter.

The branch number is the limiting metric. The line number can still move with helper-heavy tests, but 98% overall will remain out of reach unless the branch-heavy codec orchestrators get direct attention.

## Remaining Friction

The run is green but not warning-free. Coverage builds still emit warnings, mostly in `src\opus\multistream.rs` and one in `src\celt\bands.rs`.

Important examples:

- unused imports in `src\opus\multistream.rs`
- unused and unnecessarily mutable locals in `src\opus\multistream.rs`
- unused helper `copy_channel_in_short` and unread field `application` in `src\opus\multistream.rs`
- unused local `sqrt_half` in `src\celt\bands.rs`

Those warnings do not block coverage collection, but they do violate the repo preference for warning-free builds.

## Conclusions

1. There are no current test failures blocking a branch-aware library coverage run.
2. The previous hard blockers were infrastructure: nightly requirement and a package build script that hard-required the missing C reference tree even for library-only coverage work.
3. The coverage problem is now honest and measurable. The repo is not “almost there”; it is thousands of lines and more than 1,600 branches away from 98%.
4. The next meaningful gains should focus on `silk::encoder`, `opus::encoder`, `dnn::lpcnet`, `opus::repacketizer`, `opus::decoder`, and `opus::multistream`, in that rough order.
