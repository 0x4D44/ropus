Current measured state:

Target:

Primary measurement command:

```powershell
mdtimeout 5400 cargo +nightly llvm-cov --workspace --all-features --branch --json --output-path target/llvm-cov/coverage-wave.json
```

## Stage 0: Stable Runner

Goal:

- keep branch-aware coverage repeatable and fast enough to iterate

Actions:

- always use `cargo +nightly llvm-cov --branch`
- write JSON summaries to `target/llvm-cov`
- treat any new test failure as a blocker and fix it before chasing more coverage

Exit criteria:

- coverage command runs cleanly without manual cleanup

## Stage 1: Deterministic Gap Recovery

Primary files:

- `src\dnn\lpcnet.rs`
- `src\opus\multistream.rs`
- `src\opus\decoder.rs`
- `src\opus\repacketizer.rs`

Why:

- these files still have meaningful uncovered line and branch debt
- most missing logic is deterministic and directly unit-testable
- they offer better return than diving into the largest encoders first

Test themes:

- model-load and reset edge cases
- packet shape and size parsing matrices
- invalid argument handling
- wrapper API branch selection
- small state transitions with deterministic fixtures

Success signal:

- branch coverage climbs meaningfully without touching parity-sensitive encode logic

## Stage 4: Fresh JSON Rerank

Goal:

- stop following stale assumptions and retarget from current coverage data

Actions after every meaningful test wave:

1. regenerate `target/llvm-cov/coverage-wave.json`
2. sort files by missed lines and missed branches
3. only attack files that still dominate totals

Decision rule:

- continue a file when new tests buy material movement
- stop when the next point would require brittle or behavior-changing scaffolding

The stated target is `98%`, but the repo should not fake progress.

If the remaining gap collapses into only large state-machine branches inside the encoders, reassess based on measured payoff rather than optimism. The honest metric is branch-aware JSON, not test count.
