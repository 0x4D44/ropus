# HLD: SILK Rate Control Loop Fix

## Problem

All 20+ @16kbps encode tests fail because the SILK encoder produces different gain indices from frame 2 onward. The `silk_gains_quant` function itself is correct — the divergence is in the `encode_frame` rate control loop, which iterates a different number of times between C and Rust, converging on different final gain multipliers.

## Root Cause Analysis

The investigation traced the divergence through gain debug prints:
- Frame 1: C makes 6 rate-control iterations, Rust makes 5. Both converge to `prev_ind=48` (same result despite different iteration counts).
- Frame 2: C makes 2 iterations, Rust makes 3. The extra Rust iteration uses different `gains_Q16` values, producing `prev_ind=48` (Rust) vs `prev_ind=37` (C from restore path).

Three independent issues feed into this:

### Issue A: TargetRate_bps is raw bitrate (most impactful)

**C** (enc_API.c:422-454) computes a per-frame target rate with 6 adjustments:
1. `nBits = bitRate * payloadSize_ms / 1000` (total packet bits)
2. Subtract LBRR bits used
3. Divide by `nFramesPerPacket`
4. Convert to bps (× 100 for 10ms or × 50 for 20ms)
5. Subtract bit reservoir decay (`nBitsExceeded * 1000 / 500`)
6. Subtract intra-packet balance for frame 2+

**Rust** (encoder.rs:7904-7906) passes `enc_control.bit_rate` directly — no adjustments.

For 16kbps mono with 20ms payloads and 1 frame/packet, the simple case gives `nBits = 16000*20/1000 = 320`, `TargetRate = 320*50 = 16000`. The adjustments (steps 2, 5, 6) are zero for the first packet with no LBRR. So the target matches initially, but diverges over time as `nBitsExceeded` accumulates in C.

### Issue B: Missing nBits re-read after damage control

When `iter == max_iter` and the damage control path fires (zero pulses, re-encode), C re-reads `nBits = ec_tell()` at line 297. Rust does not — it uses the stale pre-damage-control value. This can cause the `found_lower` restoration to trigger incorrectly.

### Issue C: gain_mult_lower/upper type mismatch

C uses `opus_int32` for `gainMult_lower` and `gainMult_upper`. Rust uses `i16`. The interpolation computation in the binary search may behave differently when values approach i16 boundaries, though typical values (64-1024) fit comfortably.

## Design

Fix all three issues in order of impact.

### Fix 1: TargetRate_bps computation

Replace the single-line target rate assignment in `silk_encode` (encoder.rs ~line 7904) with the full C computation:

```rust
// Total bits for packet
let mut n_bits_packet = silk_div32_16(
    silk_mul(enc_control.bit_rate, enc_control.payload_size_ms), 1000);

// Subtract LBRR bits (if not prefill)
if prefill_flag == 0 {
    n_bits_packet -= enc.n_bits_used_lbrr;
}

// Divide by frames per packet
n_bits_packet = silk_div32_16(n_bits_packet, n_frames_per_packet);

// Convert to bps
let mut target_rate = if enc_control.payload_size_ms == 10 {
    silk_smulbb(n_bits_packet, 100)
} else {
    silk_smulbb(n_bits_packet, 50)
};

// Subtract bit reservoir decay
target_rate -= silk_div32_16(
    silk_mul(enc.n_bits_exceeded, 1000), BITRESERVOIR_DECAY_TIME_MS);

// Intra-packet balance for frame 2+
if prefill_flag == 0 && enc.state_fxx[0].s_cmn.n_frames_encoded > 0 {
    let bits_balance = range_enc.tell() as i32
        - enc.n_bits_used_lbrr
        - n_bits_packet * enc.state_fxx[0].s_cmn.n_frames_encoded;
    target_rate -= silk_div32_16(
        silk_mul(bits_balance, 1000), BITRESERVOIR_DECAY_TIME_MS);
}

// Clamp
target_rate = target_rate.max(5000).min(enc_control.bit_rate);

silk_control_snr(&mut enc.state_fxx[0].s_cmn, target_rate);
```

Also add the `nBitsExceeded` update at packet completion (after the encoding loop, ~line 7982):

```rust
enc.n_bits_exceeded += (*n_bytes_out * 8) as i32;
enc.n_bits_exceeded -= silk_div32_16(
    silk_mul(enc_control.bit_rate, enc_control.payload_size_ms), 1000);
enc.n_bits_exceeded = enc.n_bits_exceeded.max(0).min(10000);
```

Constants needed: `BITRESERVOIR_DECAY_TIME_MS = 500` (from silk/define.h).

### Fix 2: nBits re-read after damage control

In `silk_encode_frame_fix` (encoder.rs ~line 7152), after the damage control re-encode block, add:

```rust
n_bits = range_enc.tell() as i32;
```

This is a single line matching C line 297: `nBits = ec_tell(psRangeEnc)`.

### Fix 3: gain_mult type widening

Change `gain_mult_lower` and `gain_mult_upper` from `i16` to `i32` in `silk_encode_frame_fix` (~line 7008). This matches C's `opus_int32` type for these variables. The `gain_mult_q8` stays `i16`.

## Testing

The @16kbps tests exercise SILK-only and hybrid modes. Fix 1 (TargetRate) is the most impactful — it directly affects the SNR target which controls how aggressively the encoder quantizes. With the correct target, the rate control loop should converge in the same number of iterations as C.

If Fix 1 alone doesn't resolve the iteration count difference, the nBits re-read (Fix 2) and type widening (Fix 3) should handle edge cases.

Expected result: most or all of the 20 @16kbps tests flip to PASS.

## Risks

1. **Missing LBRR encode**: The Rust port doesn't implement `silk_LBRR_encode_FIX`. Without LBRR, `n_bits_used_lbrr` is always 0 and `lbrr_flags` are always 0. This is correct for the current tests (LBRR is disabled by default), but the LBRR subtraction in the target rate computation should still use `enc.n_bits_used_lbrr` (which is 0) rather than hardcoding 0.

2. **silk_div32_16 / silk_mul availability**: These are SILK fixed-point math functions. Verify they exist in the Rust port. `silk_div32_16(a, b) = a / b` (16-bit divisor). `silk_mul(a, b) = a * b` (32-bit result). `silk_smulbb(a, b) = (i16(a) * i16(b))` (16×16→32).

3. **Multi-frame packets**: The tot_blocks/curr_block issue (per-frame maxBits splitting) only matters for 40ms+ payloads. The tests use 20ms payloads, so this can be deferred. Document it as a known limitation.
