## Recommended Next Slice

Start Phase 1.2, narrowly: add a bounded fuzz sanity gate that builds the manifest-discovered fuzz targets and replays committed crash corpora without starting fuzz campaigns.

Why this slice now:

- Phase 1.1 removed the stale-target-list blocker, so a fuzz build/replay gate can now trust the target set.
- It catches broken fuzz target code before overnight campaigns.
- It is a gate-plumbing improvement, not codec work, so it preserves the plan's order and avoids mixing DRED/PLC correctness work into infrastructure.
- It is small enough to keep honest: no long-running fuzzing, no new CI framework, no broad release-policy redesign.

Implementation shape I would start with:

1. Add a build-only mode to the existing manifest-driven fuzz runner, or a tiny shared helper command, so `cargo +nightly fuzz build --fuzz-dir tests/fuzz <target>` can run without launching fuzz campaigns.
2. Reuse the existing `--check-crashes` path for crash replay, but make the command shape bounded and documented.
3. Wire it into `full-test --release-preflight` as an explicit fuzz sanity lane, with a clear policy for missing `cargo-fuzz`/nightly before implementation starts.

Dependencies and blockers:

- Requires nightly Rust and `cargo-fuzz`; decide whether absence is FAIL under `--release-preflight` or a reported SKIP outside it.
- Full all-target fuzz builds may exceed quick-smoke expectations, so keep this out of `--quick --release-preflight` unless Arthur explicitly wants that cost.
- Crash replay depends on committed files under `/home/md/language/ropus/tests/fuzz/crashes`; empty crash directories should be a clean pass, not a false failure.

Suggested scoped validation for the implementation slice:

- `timeout 10s bash -n tools/fuzz_run.sh`
- `timeout 10s bash tools/fuzz_run.sh --list`
- one bounded build-only command for a single target first, then all targets only if the first path is clean
