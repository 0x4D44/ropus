## 1. Problem statement (post-V1 close-out)

After Cluster A V1 closed Bug A (setter asymmetry) and Bug B1 (multiframe stereo prefill), one
sub-class remains: **`multistream-encode-bytes-divergence`**, 5 repros under
`/home/md/language/ropus/tests/fuzz/known_failures/multistream-encode-bytes-divergence/`.

| Repro | sr/ch | family | bitrate | cx | vbr | Outcome at byte level |
|---|---|---|---|---|---|---|
| `crash-00.bin` | 8000/1 | 0 | 15472 | 6 | 0 | Rust=15B C=15B differ at byte 1 |
| `crash-01.bin` | 8000/1 | 0 | 15472 | 6 | 0 | Rust=15B C=15B differ at byte 2 |
| `crash-02-len-mismatch.bin` | 8000/1 | 0 | 62063 | 9 | 1 | Rust=15B vs C=13B differ at byte 2 |
| `crash-03-cx4-vbr1.bin` | 8000/1 | 0 | 15472 | 4 | 1 | Rust=12B C=12B differ at byte 2 |
| `crash-04-roundtrip-cx4-vbr1.bin` | 8000/1 | 0 | 15472 | 4 | 1 | Rust vs C differ on roundtrip-path bytes assert |

All are **mono SILK NB** through a single-stream multistream wrapper (`OpusMSEncoder::new_surround`,
ch=1, family=0 — effectively a thin pass-through to one `OpusEncoder`). The trigger is the
`fuzz_multistream` per-iteration setter shuffle (bitrate / complexity / vbr / fec / dtx /
loss_perc); a single `OpusMSEncoder::encode` call after the shuffle produces a divergent compressed
packet.

### 1.1 What V1 already established

Phase B's 7-boundary grid (after LBRR encoding, after HP cutoff, after target-rate computation,
after stereo LR→MS, after `silk_control_SNR`, **after `silk_encode_frame_Fxx`**, after bit-reservoir
update) was run on all 4 documented repros. **All 4 first diverge at boundary 6** — the post-encode
boundary inside the per-channel loop in `silk_Encode`. Boundaries 1–5 match identically across
implementations:

```
[  4] b5/ch 0 R: tell=    3 rng=20000000 tr=  5600  (matches C)
* [  5] b6/ch 0 R: tell=   95 rng=022ddd3a tr=  5600
       b6/ch 0 C: tell=   99 rng=2abc29a4 tr=  5600   <-- DIVERGE
```

Boundary 6 is the call site `silk_encode_frame_Fxx(...)` at C `enc_API.c:520` /
Rust `silk/encoder.rs:8341`. Both inputs (per b1..b5 trace) and the post-call observable
(`(ec_tell, rng)` at b6) are well-defined — the bug is **inside the function body**.

The inputs that match at b5 include: `target_rate=5600`, `n_bits_exceeded=0`, `curr_lbrr=0`,
`n_lbrr=0`, `mid_only=-1`, `prev_decode_only_middle=0`. So the rate-control / LBRR / stereo /
SNR-control inputs are all aligned at the call boundary.

### 1.2 The pre-encode state mismatches are NOT causal

`fuzz_repro_diff` reports four pre-encode state mismatches consistently across all 5 repros:

```
nsq_lag_prev:    C=0 R=100
last_gain_index: C=0 R=10
prev_gain_q16:   C=0 R=65536
prev_lag:        C=0 R=100
```

These look alarming but are **cosmetic** — both `silk_setup_fs` in Rust
(`ropus/src/silk/encoder.rs:1535-1540`) and C (`reference/silk/control_codec.c:254-259`)
unconditionally write `prev_lag=100, last_gain_index=10, lag_prev=100, prev_gain_q16=65536` inside
the `if (psEnc->sCmn.fs_kHz != fs_kHz)` block, which fires on *both* sides on the very first
encode (since memset zeros `fs_kHz`). The C dump is read **before** the Rust-vs-C `silk_Encode`
call begins, when C's `silk_setup_fs` has not yet run. Once both encoders enter
`silk_encode_frame_FIX`, the values are aligned. (Confirmed: stage 3b journal annotates these as
"pre-existing init-value diffs that don't propagate to bitstream output" for the multiframe class
that did get fixed — same applies here.)

V2 will **suppress** these four fields from the pre-encode mismatch report so the diagnostic noise
doesn't distract from the real signal.

### 1.3 The 66 post-encode NSQ-buffer field diffs are symptoms

After the divergent encode, 66 NSQ extended-state fields differ between Rust and C
(`nsq_slpc_q14[0..15]`, `nsq_sar2_q14[0..15]`, `nsq_sltp_shp_q14[0..31]`, plus `nsq_slf_ar_shp_q14`
and `nsq_sdiff_shp_q14`). These are downstream artefacts of running NSQ with divergent inputs —
they tell us *that* NSQ saw different inputs but not *which* input was first wrong. V2's mission
is to backwards-localise the first divergent input.

## 2. Why the V1 grid cannot resolve this

V1's Phase B was deliberately *coarse* — 7 boundaries spanning the entire `silk_Encode` body to
pin which of the function's seven major sections owns the bug. That answer is now in: section 6,
the `silk_encode_frame_Fxx` call. Going finer requires entering that function's 200-line body,
which V1 explicitly deferred (HLD V1 §4.2 anti-fix-creep rule).

`silk_encode_frame_FIX` (C: `reference/silk/fixed/encode_frame_FIX.c:85-389`,
Rust: `ropus/src/silk/encoder.rs:7036-7420`) decomposes into:

| # | C section | Range coder writes? | Functions called |
|---|---|---|---|
| F1 | LP variable cutoff | no | `silk_LP_variable_cutoff` |
| F2 | x_buf insertion | no | `memcpy` |
| F3 | Pitch lag analysis | no | `silk_find_pitch_lags_FIX` |
| F4 | Noise shape analysis | no | `silk_noise_shape_analysis_FIX` |
| F5 | LPC + LTP coef finding | no | `silk_find_pred_coefs_FIX` |
| F6 | Gain processing | no | `silk_process_gains_FIX` |
| F7 | LBRR encoding (low-rate redundancy) | conditional | `silk_LBRR_encode_FIX` (uses local NSQ, no entropy) |
| F8 | Rate-control loop (≤ 7 iterations): | YES (per iter) | |
|    |   8a. NSQ (pulses + indices fill) | no | `silk_NSQ` / `silk_NSQ_del_dec` |
|    |   8b. `silk_encode_indices` | YES | |
|    |   8c. `silk_encode_pulses` | YES | |
|    |   8d. Damage control (last iter only) | YES | re-encodes with zero pulses |
|    |   8e. Compute new gainMult, re-quantize gains | no | `silk_gains_quant` |
| F9 | x_buf shift, prev_lag/signal_type update | no | |

The only functions writing range-coded bits are F8b (`silk_encode_indices`) and F8c
(`silk_encode_pulses`), with possible F8d reruns. So either (a) F1–F7 + F8a produced different
*inputs* to F8b/c, or (b) F8b/c themselves diverged on identical inputs, or (c) the rate-control
loop took a different number of iterations / a different damage-control path.

This gives a cleanly-bounded hypothesis space.

## 3. Hypothesis matrix

Ranked by prior. Each hypothesis has a deterministic localisation strategy via the proposed
trace points in §4. **Priors revised after evidence review** — see §3.1.

| # | Hypothesis | Prior | Reasoning |
|---|---|---|---|
| **G2** | **NSQ (`silk_NSQ` / `silk_NSQ_del_dec`) produces different `pulses` / `indices` on identical inputs** | **High** | NSQ reads `pred_coef_q12, ltp_coef_q14, ar_q13, harm_shape_gain_q14, tilt_q14, lf_shp_q14, gains_q16, pitch_l, lambda_q10, ltp_scale_q14` from `s_enc_ctrl` plus the cross-frame `s_nsq` state. Single-character Q-format off-by-ones in NSQ's del-dec branch have surfaced before in this codebase (`wrk_journals/2026.04.03 - JRN - nsq-lpc-pred-divergence.md`). **Strongest fit for the observed divergence shapes** (§3.1): NSQ output divergence cascades into both `ec_tell` and `rng` deltas of arbitrary magnitude, matching the +0/+1/+4/+12 tell-spread we see. **Distinguishing**: hash NSQ inputs and outputs around F8a. |
| G1 | `silk_encode_indices` / `silk_encode_pulses` produce different bits on identical inputs | Medium-High | F8b/c are mostly mechanical mapping from indices+pulses to range-coded bits. The multistream setter shuffle reaches `signal_type`/`quant_offset_type`/`nb_subfr` combinations not strongly covered by `fuzz_encode`. Note: a *pure* entropy bug on identical pulses+indices typically preserves total `ec_tell` (the iCDF sums to a constant per state) — **so crash-01 (Δtell=0, Δrng≠0) is the strongest candidate for G1**, while crash-00/02/03 (Δtell≠0) lean G2. **Distinguishing**: hash `s_cmn.indices` and `s_cmn.pulses` immediately before F8b; if equal across Rust/C but bytes diverge after F8b, G1 owns that repro. |
| **G9** | **`silk_LBRR_encode_FIX` writes to `s_enc_ctrl.gains_q16` or `pulses_LBRR` with downstream effect** | **Medium** | LBRR is gated on `LBRR_enabled && speech_activity_q8 > LBRR_SPEECH_ACTIVITY_THRES_Q8 (=77)`. The setter shuffle toggles fec — exactly the input that activates this gate. The function temporarily overwrites `gains_q16` via `silk_gains_dequant`, then restores from a `temp_gains_q16` snapshot. **The boundary-6 trace's `cl=0,nl=0` does NOT rule this out** — those counters track outer-loop LBRR entropy bits (gated on `nFramesEncoded > 0`), so they're zero on these single-frame repros even when LBRR_encode_FIX fires internally. A Rust-side restore-bug would corrupt outer-scope gains and feed divergent NSQ inputs. **Distinguishing**: boundary 105 hashes `gains_q16` before and after F7 — should be unchanged on both sides. |
| G3 | `silk_process_gains_FIX` produces different gains | Medium | `s_enc_ctrl.gains_q16` is the dominant cross-implementation precision risk because it goes through `silk_lin2log` / `silk_log2lin` and `silk_LSHIFT_SAT32` chains. A 1-LSB drift propagates linearly through NSQ. **Distinguishing**: hash `gains_q16, gains_unq_q16, lambda_q10, last_gain_index_prev` after F6. |
| G4 | `silk_find_pred_coefs_FIX` (LPC + LTP) produces different coefficients | Medium | Burg-modified AR estimation and LTP correlation matrix inversion are the most numerically delicate steps. Past divergence (e.g. `wrk_journals/2026.04.05 - JRN - silk-16kbps-divergence.md`) localised here. **Distinguishing**: hash `pred_coef_q12, ltp_coef_q14, ar_q13, ltp_scale_q14, pitch_l` after F5. |
| G8 | Rate-control loop takes a different number of iterations or different damage-control path | Medium | The loop's break/continue decisions read `n_bits` (which depends on F8b/c output), `gainsID` (depends on `gains_quant`), `useCBR`, `iter`. A 1-bit difference in F8b/c output cascades into a different iter count, which is observable. **Distinguishing**: trace `(iter, gainsID, n_bits, found_lower, found_upper)` per loop body. Is downstream of G1/G2 mostly. |
| G5 | `silk_noise_shape_analysis_FIX` produces different shape coeffs | Low-Medium | `harm_shape_gain_q14, tilt_q14, lf_shp_q14` derive from the noise shape analysis. **Distinguishing**: hash these fields after F4. |
| G6 | `silk_find_pitch_lags_FIX` returns different pitch lag | Low | Pitch lag is a single integer per subframe — easy to verify and rarely diverges (the path is heavily exercised by `fuzz_encode`). **Distinguishing**: hash `pitch_l` (8 ints) after F3. |
| G7 | `silk_LP_variable_cutoff` produces different filtered samples in `input_buf[1..]` | Very low | LP-state is covered by `silk_extended_state` already; the existing harness would have caught this in V1. Verify-and-eliminate. |

The matrix is collapsible: G1 and G2 are the leaf cases; G3–G7 are the upstream "did the inputs to
NSQ match" question. G8 is downstream of G1/G2. G9 is a unique side-channel suspicion.

### 3.1 The 5 repros may not all share one root cause

The b6 first-divergence shapes vary substantially:

| Repro | Rust tell/rng | C tell/rng | Δtell | Hypothesis lean |
|---|---|---|---:|---|
| crash-00 | 95 / 0x022ddd3a | 99 / 0x2abc29a4 | +4 | G2/G3 (substantive divergence) |
| crash-01 | 110 / 0x051df9bc | 110 / 0x05101042 | 0 | **G1** (same total bits, different bytes — entropy-coder candidate) |
| crash-02 | 105 / 0x009f6090 | 93 / 0x084dda74 | +12 | G2 (large divergence; cx=9 → del-dec on; G2 dominant) |
| crash-03 | 87 / 0x03fbd68b | 88 / 0x012188b6 | +1 | G2 or G3 (small divergence) |
| crash-04 | (roundtrip-path) | (same shape as 03) | — | likely same as crash-03 |

A single bug can produce varying Δtell magnitudes (small early state error → varying NSQ pulse
patterns), so this is not proof of multiple bugs. But it's *suggestive*. **Phase C must run on
all 5 separately** and the close-out criterion is that *every* repro localises to a sub-boundary.
If different repros first-diverge at different sub-boundaries, treat them as separate bugs and
fix them in separate atomic commits (see §5.6).

## 4. Instrumentation plan — Phase C (extend trace grid into the function body)

### 4.1 Trace point grid (10 new boundaries)

Extend the existing `silk_trace::Tuple` schema with a typed variable-length payload (`payload:
Vec<i32>` + `payload_kind: u8` discriminator) that carries the **full state vectors** at each
boundary, not a digest. Per Q2 sign-off: diagnostic-by-default rather than hash-then-re-run.
Boundary 6 in the existing grid stays as-is (post-function); the 10 new boundaries are numbered
**6.1 .. 6.10** with `boundary_id` values 100–109, so the V1 7-tuple grid is unaffected.

| ID | Where | Payload (i32 vector) | Hypothesis |
|---|---|---|---|
| 100 | After F1 (`silk_LP_variable_cutoff`) | `input_buf[1..=frame_length]` (cast i16→i32, ≤320 elts at 8 kHz NB) | G7 |
| 101 | After F3 (`silk_find_pitch_lags_FIX`) | `pitch_l[0..nb_subfr]` ‖ `[lag_index, contour_index]` (≤6 elts) | G6 |
| 102 | After F4 (`silk_noise_shape_analysis_FIX`) | `harm_shape_gain_q14[0..nb_subfr]` ‖ `tilt_q14[0..nb_subfr]` ‖ `lf_shp_q14[0..nb_subfr]` ‖ `gains_unq_q16[0..nb_subfr]` ‖ `[harm_boost_smth, tilt_smth]` (≤18 elts) | G5 |
| 103 | After F5 (`silk_find_pred_coefs_FIX`) | `pred_coef_q12[0][0..max_lpc_order]` ‖ `pred_coef_q12[1][0..max_lpc_order]` ‖ `ltp_coef_q14[0..LTP_ORDER*nb_subfr]` ‖ `ar_q13[0..MAX_SHAPE_LPC_ORDER*nb_subfr]` ‖ `[ltp_scale_q14]` ‖ `nlsf_indices[0..max_lpc_order+1]` ‖ `ltp_index[0..nb_subfr]` ‖ `[ltp_scale_index, nlsf_interp_coef_q2]` (≤150 elts) | G4 |
| 104 | After F6 (`silk_process_gains_FIX`) | `gains_q16[0..nb_subfr]` ‖ `[lambda_q10, last_gain_index_prev]` ‖ `gains_indices[0..nb_subfr]` (≤10 elts) | G3 |

| 106 | Pre-F8a (entry to rate-control iter, once per iter) | `[iter, gainsID, gainsID_lower, gainsID_upper, found_lower, found_upper, gainMult_Q8]` (7 elts) | G8 |
| 107 | Post-F8a (`silk_NSQ` / `silk_NSQ_del_dec`, once per iter) | `[iter]` ‖ `pulses[0..frame_length]` (cast i8→i32, ≤320 elts) ‖ `[signal_type, quant_offset_type, rand_seed, lag_prev]` | G2 |
| 108 | Post-F8b (`silk_encode_indices`, once per iter) | `[iter, ec_tell, rng_lo, rng_hi]` (split u32 into two i32 halves) | G1a |
| 109 | Post-F8c (`silk_encode_pulses`, once per iter) | `[iter, ec_tell, rng_lo, rng_hi]` (a previous draft listed `n_bits` as a fifth column, but at this point `n_bits == ec_tell` exactly, so the column was dropped as redundant) | G1b |

Total per-encode: 6 setup boundaries + 4 per-iter × ≤ 7 iters = **34 tuples max**, but each
carries up to 320 i32s (boundaries 100/107). At 4 bytes/elt that's ~3.2 KB max per tuple, ~110 KB
per encode. Bump `DBG_SILK_TRACE_MAX` from 256 to 64 (we don't need the count, just the headroom)
or, simpler, keep it at 256 and rely on most tuples being short.

The first divergent (boundary_id, payload) pair resolves the hypothesis matrix:
- **First divergence at 101** → G6 (pitch lag).
- **First at 102** → G5 (noise shape).
- **First at 103** → G4 (pred coefs).
- **First at 104** → G3 (gains).
- **First at 105** → G9 (LBRR side-effect).
- **First at 107** → G2 (NSQ).
- **First at 108** → G1a (`silk_encode_indices` mis-encodes).
- **First at 109** → G1b (`silk_encode_pulses` mis-encodes).
- **First at 106** with G2/G1 already matching for prior iters → G8 (rate-control branching).

**Diff display**. `fuzz_repro_diff::print_phase_b_trace_diff` prints a per-boundary header
(`b{id} iter={iter}`) and then the payload as named columns (decoded by boundary_id) — e.g. for
boundary 104, print `gains_q16=[…] lambda_q10=… last_gain_index_prev=… gains_indices=[…]`. The

### 4.2 Implementation surface

C side: a new file `harness/silk_encode_frame_FIX_traced.c` — a copy of
`reference/silk/fixed/encode_frame_FIX.c` with both externals renamed
(`silk_encode_frame_FIX_traced`, `silk_encode_do_VAD_FIX_traced`) and `DBG_TRACE_F` macro
insertions at the 10 boundaries above. The companion `silk_enc_api_traced.c::silk_Encode_traced`
already exists; modify it to call the `_traced` variants. A new exported
`dbg_silk_trace_push_payload(int boundary_id, int iter, const int32_t *payload, int payload_len)`
in `harness/debug_silk_trace.c` writes one variable-length record per boundary.

Rust side: extend `ropus::silk_trace::Tuple` with `payload: Vec<i32>` and `iter: i32` fields,
gated behind the same `trace-silk-encode` feature. Add 10 `silk_trace::push_payload` call-sites
inside `silk_encode_frame_fix` (under `#[cfg(feature = "trace-silk-encode")]`) building each
boundary's payload per the §4.1 schema.

Read-back side: `harness/src/bindings.rs` declares `dbg_silk_trace_count_get`,
`dbg_silk_trace_read_payload(idx, out_buf, out_buf_len) -> (boundary_id, iter, len)` returning the
record at index `idx`. `fuzz_repro_diff::read_c_silk_trace` rebuilds a `Vec<Tuple>` from these.

`fuzz_repro_diff::print_phase_b_trace_diff` decodes each tuple's payload by boundary_id, prints
named columns, and highlights the first per-element divergence within the first divergent
boundary.

6. `harness/src/bin_inner/fuzz_repro_diff.rs` — extend the print path to decode each

7. `ropus/src/lib.rs` — extend `silk_trace::Tuple` with `iter: i32` and `payload: Vec<i32>`

8. `ropus/src/silk/encoder.rs` — add 10 `#[cfg(feature = "trace-silk-encode")] silk_trace::push`
   call-sites with named-init Tuple population per the §4.1 payload schema (full state vectors,
   not digests; per Q2 sign-off).

(8) is a production-source edit, but the call-sites are zero-cost when the feature is off (which
is the default). This mirrors the V1 stage 2b pattern. No behavioural change in the production
build path.

**Implementation gotcha (link-collision).** `reference/silk/fixed/encode_frame_FIX.c` defines two
externals: `silk_encode_frame_FIX` *and* `silk_encode_do_VAD_FIX`. A naive copy that renames only
the inner function will duplicate-define `silk_encode_do_VAD_FIX` at link time. Resolution
options:

- **(preferred)** Rename both functions in the traced file (`silk_encode_frame_FIX_traced`,
  `silk_encode_do_VAD_FIX_traced`) and have `silk_enc_api_traced.c` call the `_traced` variants.
- Or `#ifdef`-out the do_VAD definition in the traced copy via a `#define SKIP_DO_VAD` guard
  so the unmodified file owns it.

**Pre-flight check before extending `silk_trace::Tuple`.** `Tuple` currently has 10 named fields
(`encoder.rs` push call sites all use named-init `Tuple { boundary_id: …, channel: …, … }` per
my read of lines 8088–8420). Adding `payload: Vec<i32>` and `iter: i32` is binary-additive and
won't break those sites *if* none of them use positional init or `..default()` against a count
that we'd be changing. Verify with `grep -n "silk_trace::Tuple {" ropus/src` plus
`fuzz_repro_diff.rs:834-840` (the C-trace read-back constructor) before landing the field. Default
for `payload` is `Vec::new()`; default `iter` is `-1` (sentinel for "not in rate-control loop")
so the V1 7-tuple-grid pushes don't need to change.

## 5. Fix protocol

> No production-source edits to `silk/encoder.rs`'s function bodies (other than the trace pushes
> in §4.3 (8)) until Phase C confirms a hypothesis. Per V1 §4.2 anti-fix-creep rule.

### 5.1 If G1 (encode_indices / encode_pulses bug)

### 5.6 Anti-fix-creep rule (carried forward from V1 §4.2)

The implementation pass that follows Phase C will fix **only the field/function that the
payload-vector diff identifies as first divergent**. If multiple downstream payload vectors
diverge, the most-upstream one is the only one fixed in this pass. Subsequent regressions (if
any) drive separate fixes.

**Per-repro triage.** If different repros first-diverge at different sub-boundaries (per §3.1
risk), they are presumed-distinct bugs. Fix and commit them **separately** — one atomic commit
per root cause, one regression-seed migration per fix, one version bump per commit. Do not bundle
"the multistream NEW class fix" into a single commit if Phase C shows it's actually 2–3 bugs.
Each repro's regression seed migrates to `tests/fuzz/seeds/fuzz_multistream/` with a
distinguishing slug (e.g. `regression-cluster-a-newclass-nsq-NN.bin` vs `regression-cluster-a-newclass-entropy-NN.bin`).

`ropus` patch bump (0.12.4 → 0.12.5). `capi` unchanged unless the fix touches the FFI surface

## 8. Open questions for supervisor review

### Q1. Phase C scope — should we copy `encode_frame_FIX.c` or extend `silk_enc_api_traced.c`?

Option A (recommended): a **new** harness-only `silk_encode_frame_FIX_traced.c` next to
`silk_enc_api_traced.c`. Keeps the per-function trace concern locally scoped; no further edits to
the api-level traced file beyond re-pointing one call. Makes future tracing of *other* internal
functions a known-shape extension.

Option B: inline the new boundaries by macro-expanding `encode_frame_FIX.c` into the existing
`silk_enc_api_traced.c`. Smaller file count but couples two concerns.

Lean toward A. Confirm.

### Q2. Should the digest be a content hash or a structured tuple?

> **Closed**: option B (full vector). The §4.1 schema was rewritten before sign-off to carry the
> full per-boundary state vector inline (`payload: Vec<i32>`, no FNV-1a hash). Diagnostic-by-default
> beats hash-then-re-run for a 5-repro investigation. The §4.5 time budget and §4.3 implementation
> surface reflect option B; the discussion below is preserved for historical context.

Option A: 64-bit FNV-1a digest. Compact, easy to compare, matches existing tuple
shape. Cost: when the digest diverges, we have to re-run with verbose dumps to see the actual
field values. That re-run is a one-line CLI flag.

Option B (chosen): store the full vector inline (e.g. all 16 `pred_coef_q12` values as 16 i32
columns). Verbose; bloats the trace tuple. Diagnoses faster but harder to read.

### Q3. Should Phase C run on the closed multiframe regression seeds too?

Risk: zero — the trace is non-mutating. Benefit: validates that Phase C captures the same
"matches across all boundaries" signal we'd expect for a *fixed* class. Acts as a positive
control for the trace itself. Low cost; recommend yes.

### Q4. Should the V2 fix migrate the `crash-04-roundtrip-cx4-vbr1.bin` repro?

`crash-04` was added during campaign 2 as a roundtrip-side capture of the same root cause class
(per `wrk_journals/2026.05.03 - JRN - 24h-fuzz-campaign-2.md` 00:18 BST entry). The README
documents only crash-00..03. After fix, `crash-04` migrates to the same regression-seeds dir as
the others. Mention in the close-out journal that the README was implicitly extended by
campaign-2's monitor pipeline; not a scope question, just disclosure.

### Q5. Time budget — is 7.5 h realistic given V1's actual stage-2b cost?

V1 stage-2b spent ~4 h building the 7-boundary api-level trace and segfaulted on prefill. V2's
inner-function trace builds on now-tested infrastructure (the prefill NULL guard is in
`946433b`). 7.5 h budget is honest given that experience.

If Phase C produces a clean first-divergence call-out, fix authoring + validation + commit cycle
adds another 2–4 h. Total budget for cluster closure: ~12 h **under G1, G3-G9**. Under **G2
(NSQ)** add Phase D's +6 h, taking the worst case to ~18 h. Scope guardrail: if Phase C+D
combined have not produced a single-function localisation within 14 h, STOP and re-design.

### Q6. If the 5 repros split across hypotheses (§3.1 risk), do we triage them in one session or split?

If Phase C shows e.g. crash-01 → G1 and crash-00/02/03 → G2, two distinct fixes are needed. Three
options:

- **(a) one session, multiple commits** (recommended): same agent fixes both bugs in sequence,
  each as an atomic commit per §5.6. Keeps cluster context warm.
- **(b) split into two agent dispatches**: file the second bug for a follow-up session. Cleaner
  audit trail; loses ~1 h to re-loading context.
- **(c) prioritise by repro count**: fix the bug that closes 3 of 4 repros first; defer the
  outlier as a third sub-class if the time budget tightens.

Lean toward (a) under the 18 h cap; fall back to (b) if either fix's complexity surprises us.
Confirm preference.
