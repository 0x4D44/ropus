# HLD — FARGAN / DRED joint follow-up (decoder PCM reconstruction)

**Date.** 2026-04-19
**Status.** Open (orphan follow-up surfaced by Stage 10 adversarial audit).
**Related.** `wrk_docs/2026.04.19 - HLD - dred-port.md` (Stage 8),
`wrk_journals/2026.04.19 - JRN - stage7-dnn-wiring-supervisor.md` (Stage 7),
`wrk_journals/2026.04.19 - JRN - stage10-audit-stream-D.md` (gap surfaced).

## Mission

Fill in `OpusDREDDecoder::decode`, `decode24`, and `decode_float` with real
FARGAN-driven PCM reconstruction, mirroring the C path in
`reference/src/opus_decoder.c:1609-1691` (`opus_decoder_dred_decode{,24,_float}`).
These three methods currently return `Err(OPUS_UNIMPLEMENTED)`; that is the
last missing link in the Stage 6 + 7 + 8 pipeline.

## Context

- Stage 8.9 stubbed `OpusDREDDecoder::{decode,decode24,decode_float}` on the
  assumption that the Stage 7 follow-up would wire FARGAN into DRED
  reconstruction. Stage 7 closed on 7b.3 (SILK PLC state carryover) and
  shipped FARGAN for the **PLC** path only, not for DRED reconstruction.
- No HLD or journal currently tracks the remaining work. This document is
  the tracking seam.

## Scope

Three Rust methods — `i16`, `i32`, `f32` PCM variants — routing through
FARGAN synthesis from `OpusDred::fec_features` / latents / state produced
by `OpusDREDDecoder::process` (already shipped). The decision point is
whether to keep the methods on `OpusDREDDecoder` or relocate them onto
`OpusDecoder`:

- C places them on the classical decoder because FARGAN synthesis needs
  channel count, preemphasis state, and resampler state owned by the
  classical decoder.
- Stage 8.9 documented this at `ropus/src/opus/dred.rs:211-216` and left
  them on `OpusDREDDecoder` with an explicit note that the follow-up is
  free to relocate.

Recommendation: relocate to `OpusDecoder` when the body lands, matching C.

## Prerequisites

- Stage 7's FARGAN port is shipped and tested for PLC — see
  `ropus/src/dnn/fargan.rs` and the tier-2 SNR harness in
  `harness-deep-plc/tests/tier2_snr.rs`.
- DRED encoder + decoder-parse + decoder-process are shipped (Stage 8.3 /
  8.7 / 8.8). `OpusDred::fec_features` is populated after
  `OpusDREDDecoder::process`.
- The DRED path differs from the PLC path in that it drives FARGAN from
  DRED-recovered latents/state rather than from the classical-PLC
  state-carryover pipeline.

## Exit criterion

The three trip-wire tests in `ropus/src/opus/dred.rs` currently assert
`Err(OPUS_UNIMPLEMENTED)` for each `decode*` method. Exit requires each
of those tests to flip to asserting a successful PCM output with the
expected `frame_size` return value.

Optional tier-2 bar: PCM SNR of recovered vs original exceeds a
to-be-determined gate (expected 40+ dB, calibrated against a C-vs-C
float-mode control run in `harness-deep-plc` — same methodology as the
Stage 7b.3 DEEP_PLC gate).

## Stage 8.6 drift link

The `#[ignore]`d byte-exact DRED encoder test
(`dred_encode_silk_frame_bytes_match_c_reference` in
`harness-deep-plc/tests/dred_encode_payload_diff.rs:154`) is gated on
LPCNet `Ex[]`/FFT bit-exactness, not on FARGAN reconstruction. It may
become unblocked by this follow-up if the implementor addresses the
LPCNet drift as part of the same pass, but the two pieces of work are
logically separate and the FARGAN-DRED body can land without
un-ignoring that test.
