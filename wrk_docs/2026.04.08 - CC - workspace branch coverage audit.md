## Interpretation

There are three different coverage problems here:

1. `silk::encoder` and `opus::encoder` dominate the remaining debt. Any honest push to 98% must move those files.
2. `dnn::lpcnet`, `opus::multistream`, and `opus::decoder` still offer relatively cheaper line and branch gains through deterministic unit tests.
3. `opus::repacketizer` and `silk::decoder` are branch-heavy cleanup opportunities with bounded setup cost.

## Missing-Line Pattern Notes

The text report shows a few broad patterns:

- `src\dnn\lpcnet.rs` still has large cold blocks in mid-file helper/state logic and late file PLC / continuation paths.
- `src\opus\multistream.rs` still has several untouched mapping, packet fan-out, and projection decode spans.
- `src\opus\decoder.rs` retains cold packet-parse and decode-wrapper paths.
- `src\silk\encoder.rs` still contains long untouched control-flow regions; this is the biggest single target, but not the cheapest first move.

## Practical Conclusion

The repo does not currently have a coverage infrastructure problem. It has a targeted-test problem.

The next efficient work should start in:

1. `src\dnn\lpcnet.rs`
2. `src\opus\multistream.rs`
3. `src\opus\decoder.rs`
4. `src\opus\repacketizer.rs`
5. `src\silk\encoder.rs`
6. `src\opus\encoder.rs`

The first four can buy meaningful improvement without immediately paying the full complexity cost of the two encoder state machines.
