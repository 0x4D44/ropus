## Testing Principle

The testing campaign has mostly moved from "too many hidden weak spots" to
"which release claims remain too soft?" The next work should not start broad
codec debugging. It should choose the smallest testing contract that makes a
green or yellow report harder to misread.

Priority criteria:

- Does a current green/WARN result overclaim release confidence?
- Is the oracle already visible, or is the issue still hidden?
- Can the work be implemented as testing infrastructure rather than fragile
  codec parity repair?
- Can the gate be calibrated to avoid flaky local failures?

## Recommended Next Slice

Do Phase 4.1 next as a narrow release performance contract:

```text
Performance release gate and severe-regression thresholds
```

This should be an HLD-first slice. The goal is not to tune ropus or expand every
benchmark vector immediately. The goal is to make the performance lane's release
semantics precise:

1. Keep default `full-test` and `--quick` behavior WARN-only/no-performance-claim.
2. In non-quick release-oriented runs where benchmarks are not skipped, make the
   report explicitly claim a performance gate.
3. Make benchmark build failure, parser failure, crashed vectors, and calibrated
   severe Rust/C ratio regressions release-blocking for that performance claim.
4. Preserve the existing 1.15 ratio as a WARN threshold unless measurement says
   otherwise.
5. Do not pick a hard FAIL threshold blindly. The implementation agent should
   measure current ratios first, record the observed baseline, and set a
   conservative severe threshold only after that measurement. If variance is
   unclear, start with policy wiring and crash/build-failure hardening, then
   leave ratio hard-fail as a documented follow-up.
6. If `--skip-benchmarks` is used, the report must say performance coverage is
   not claimed rather than silently passing the release performance lane.

Why this next:

- The main correctness gates now have explicit oracle labels and release claim
  policy. Performance is the remaining full-test lane whose serious problems can
  still collapse to WARN with exit code 0.
- It is testing-infrastructure work, not codec implementation work.
- It is more bounded than platform/sanitizer work and more release-relevant than
  mutation testing.
- It avoids pretending that DRED/PLC exact parity can be solved by adding more
  test plumbing; those are now visible codec debts.
