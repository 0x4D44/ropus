# HLD: PitchDNN feat-18 residual drift

Author: investigator-agent  •  Date: 2026.05.07  •  Status: Draft for supervisor review

## 1. Problem statement

After the eight DRED parity fixes shipped through `da62d7a`, 34 of 36 DRED LPCNet
features are bit-exact against the xiph C reference on the 440 Hz fixture. Two
remain:

| feat | name        | max\|drift\| |
|------|-------------|--------------|
| 18   | `dnn_pitch` | 1.013279e-5  |
| 19   | `frame_corr`| 1.79e-7      |

Feat 18 is the focus. Test driver: `harness-deep-plc/tests/dred_lpcnet_feature_drift.rs`
(currently locks `MAX_ABS_FEATURE_DRIFT = 1.013279e-5`).

What is already ruled out (prior diagnostic in the supervisor brief and reconfirmed
this session by per-layer instrumentation):

- **Inputs** to `compute_pitchdnn` (`if_features`, `xcorr_features`) are
  bit-exact between C and Rust (sum, ssq, and per-element first-6 dump match
  to all printed digits).
- **Loaded weights** for layer 1 (`dense_if_upsampler_1`) are bit-exact:
  `len=5632 sum=-5.836683744 ssq=274.9119606`, identical first-8 floats.
- **Loaded biases** are bit-exact.

## 2. Investigation findings

### Network architecture (`reference/dnn/pitchdnn.c:12-56`)

PitchDNN is a small audio model:

```
if_features[88]  ─→ Dense(88→64) ─tanh─→ if1_out[64]
                                        └→ Dense(64→64) ─tanh─→ ds_in[224..288]
xcorr_features[224] ─→ pad+Conv2d(1ch→4ch,3×3) ─tanh─→ Conv2d(4ch→1ch,3×3) ─tanh─→ ds_in[0..224]
ds_in[288] ─→ Dense(288→64) ─tanh─→ ds_out[64]
ds_out[64] ─→ GRU(64,recurrent=64) ─→ gru_state[64]
gru_state[64] ─→ Dense(64→192) ─linear─→ logits[192]
soft-argmax(logits[0..180]) ─→ dnn_pitch  (= feat 18)
```

Activations are `tanh_approx`, sigmoid via tanh, `(f64).exp() as f32` in the
soft-argmax (already explicitly faithful — `pitchdnn.rs:387`).

### Per-layer instrumentation results (frame 0, fixture 48000hz_mono_sine440)

I instrumented both sides at every layer-output boundary, plus split layer 1
into pre-tanh (`if1_lin`) and post-tanh (`if1_out`). Cleaned up before
reporting; instrumentation is no longer in tree.

| tag       | C ssq          | Rust ssq         | first-element |
|-----------|----------------|------------------|---------------|
| `if_in`   | 8.168369914    | 8.1683699142     | bit-exact     |
| `xc_in`   | 0.04365915084  | 0.0436591508     | bit-exact     |
| `if1_lin` | 9.639134835    | 9.6391348346     | **bit-exact** (pre-tanh) |
| `if1_out` | 7.852916692    | 7.8534388445     | C=-0.461223572, R=-0.461181611 |

The **first divergence appears across the very first `tanh_approx` call**, with
input `if1_lin[0] = -0.498856246` (bit-exact on both sides) producing different
post-tanh values. From there, every downstream layer is contaminated and the
divergence propagates through 50 frames via the GRU recurrent state, peaking at
1.013e-5 on `dnn_pitch` at frame 5.

### Why does `tanh_approx` diverge?

C `tanh_approx` (`reference/dnn/vec.h:338-352`) and Rust `tanh_approx`
(`ropus/src/dnn/core.rs:235-256`) are both Padé-rational approximations with
identical f32 constants `N0..N2, D0..D2`.

But: **`vec.h:38` includes `vec_avx.h` whenever `__SSE2__` is defined**, which
on x86_64 Linux is the default baseline. So the C build used by the harness
takes the SSE/AVX path, **not** the scalar `tanh_approx` you read in `vec.h`.

The harness build flags (captured from cc-rs verbose output earlier this
session) are:

```
gcc -O0 -ffp-contract=off -m64 ... -DHAVE_CONFIG_H=1 -DOPUS_BUILD ...
```

with no `-mavx`, no `-mfma`. So in `vec_avx.h`:

- `__AVX__` is undefined → falls into the SSE2-only branch
  (`vec_avx.h:469-532` — `tanh4_approx`, `tanh_approx` wrapping it).
- `__FMA__` is undefined → `_mm_fmadd_ps` is the scalar fallback macro at
  `vec_avx.h:300-303` (`_mm_add_ps(_mm_mul_ps(a,b), c)`), so no FMA fusion.
  Fine — same effective arithmetic as Rust here.
- **`_mm_rcp_ps` IS used** (`vec_avx.h:486`):

```c
   X2 = _mm_mul_ps(X, X);
   num = _mm_fmadd_ps(_mm_fmadd_ps(N2, X2, N1), X2, N0);
   den = _mm_fmadd_ps(_mm_fmadd_ps(D2, X2, D1), X2, D0);
   num = _mm_mul_ps(num, X);
   den = _mm_rcp_ps(den);            // ← approximate reciprocal
   num = _mm_mul_ps(num, den);       // num * approx(1/den), not num/den
   return _mm_max_ps(min_out, _mm_min_ps(max_out, num));
```

`_mm_rcp_ps` is the SSE 12-bit-precision approximate-reciprocal intrinsic
(Intel SDM: max relative error `1.5 * 2^-12 ≈ 3.66e-4`). The C tanh therefore
deviates from a "true" f32 `num/den` by up to ~3.66e-4 relative, which on
typical pre-tanh values translates to ~1e-5 to 1e-4 absolute drift per element.

Rust `tanh_approx` does an honest f32 division `num * x / den`
(`core.rs:247`), no reciprocal approximation.

### Independent verification

A strict step-by-step f32 reproduction in Python of `tanh_approx(-0.498856246)`
yields **-0.46118161** — matching Rust **exactly to all printed digits**. The C
result -0.461223572 deviates by 4.2e-5, consistent with the `_mm_rcp_ps`
quantisation. This eliminates "Rust has a bug" — Rust matches the polynomial
spec; C deviates because of the SIMD reciprocal short-cut.

## 3. Root cause

**Confidence: high.** The residual feat-18 drift is caused by the C reference
using `_mm_rcp_ps` (12-bit-precision approximate reciprocal) inside its SSE2
`tanh4_approx` (and the analogous AVX `tanh8_approx`) at
`reference/dnn/vec_avx.h:486` and `:444`, while Rust performs an honest f32
division. Single-element drift in `tanh_approx` is within the documented
3.66e-4 relative-error budget of `_mm_rcp_ps`. The tiny per-frame error is then
amplified ~10× through six fully-connected/conv layers and a 64-cell GRU
recurrent loop before showing up as a 1e-5 drift in `dnn_pitch`.

This is **not the same shape as the prior eight fixes** (those were
f32-vs-f64-precision accidents in Rust). This time **C is the loose path**;
Rust is the spec-compliant one. The "fix" choices below reflect that
inversion.

## 4. Proposed fix

Three options. I recommend **(B)**; the choice has strategic implications for
how we treat upstream SIMD shortcuts.

### (A) Mimic `_mm_rcp_ps` inside Rust `tanh_approx`

Replace honest division at `core.rs:247` with the x86 intrinsic via
`core::arch::x86_64::_mm_rcp_ps` in an `unsafe` block.
**Pros:** bit-exact on x86_64 (expected new bound: 0.0). **Cons:** introduces
`unsafe` and an x86-only intrinsic; non-x86 hosts have no portable equivalent;
mirrors an *imprecision* in the reference; future xiph might add a
Newton-polish step and break our exact mimicry; `tanh_approx` is shared with
FARGAN/OSCE/lossgen — high blast radius.

### (B) Accept the residual; keep the locked bound (RECOMMENDED)

Update the comment in `dred_lpcnet_feature_drift.rs` to record the root cause
(reference's `_mm_rcp_ps` in SSE2 `tanh4_approx`) and the strategic decision
to hold `MAX_ABS_FEATURE_DRIFT = 1.013279e-5`. Sketch:

```rust
// Locked bound 1.013279e-5 (feat 18 dnn_pitch). Investigation
// `wrk_docs/2026.05.07 - HLD - pitchdnn-feat18-residual.md` traced the
// residual to the C reference's SSE2 tanh path
// (reference/dnn/vec_avx.h:486 — `_mm_rcp_ps` 12-bit reciprocal). Rust
// uses honest f32 division and is the more precise path. Reference-side
// artefact, NOT a Rust bug.
```

**Pros:** zero code change; bound already at observed value; residual is
provably reference-side. **Cons:** 34/36 features bit-exact, not 36/36 —
but `dnn_pitch` is mapped to integer pitch period via `floor(.5+...)`, so
a 1e-5 drift is orders of magnitude below the bin spacing. Acoustically
inaudible.

### (C) Disable C SIMD `tanh` via build flag

Predefine a preprocessor symbol (or `#undef __SSE2__`) in `harness-deep-plc/build.rs`
to force the scalar path in `vec.h:50+`.
**Pros:** 36/36 bit-exact against a "scalar reference". **Cons:** measures
against a non-default reference configuration that no xiph user ships.
Violates the Tier-1 spirit of "match what users actually run."

## 5. Verification strategy

`harness-deep-plc/tests/dred_lpcnet_feature_drift.rs` is the primary gate.

Expected outcomes:

- **(A)** new `MAX_ABS_FEATURE_DRIFT = 0.0` (or 5.96e-8 for feat 19 if its
  residual survives), `MAX_RMS_PER_FRAME_DRIFT = 0.0`,
  `MAX_DRIFTING_FRAME_COUNT = 0` (or 50 just for feat 19).
- **(B)** unchanged: `MAX_ABS_FEATURE_DRIFT = 1.013279e-5`,
  `MAX_RMS_PER_FRAME_DRIFT = 1.6888275e-6`. Just a comment update.
- **(C)** new `MAX_ABS_FEATURE_DRIFT` likely 0.0 — but the harness now runs
  a non-default C reference, which is a separate strategic concession.

In all cases also re-run `cargo test --workspace` to make sure no other
DNN-using tests regress (FARGAN, OSCE-related comparisons, `lossgen`).

## 6. Risk analysis

`tanh_approx` is shared by every DNN path: FARGAN, OSCE/LACE/NoLACE, lossgen,
DRED RDOVAE. Those currently match C at Tier-2 SNR ≥ 50/60 dB. Option A would
push x86_64 SNRs higher but introduce host-conditional numerical behaviour
(non-x86 fallback would mimic a specific Intel uarch — Skylake vs Alder Lake
`rcp_ps` LUTs differ in low bits). Option B touches zero codec code. Option C
touches only the harness build script.

## 7. Out of scope

Feat 19 residual (1.79e-7, last-bit noise). Non-x86 host bit-exactness
(NEON `tanh4_approx` has its own approximations, `vec_neon.h:86`). Tier-2
modules whose looser SNR bounds already absorb this `_mm_rcp_ps` artefact.

## 8. Acceptance criteria

For the supervisor to sign off:

1. Choose option A, B, or C.
2. If B (recommended): the only deliverable is a comment block in
   `dred_lpcnet_feature_drift.rs` (and possibly a CHANGELOG entry). No code

3. If A: new test gate is `max_abs_drift == 0.0` for all features except

   `unsafe` block reviewed; non-x86 fallback path explicitly documented and
   tested.
4. If C: harness build script change documented in
   `harness-deep-plc/build.rs`; HLD updated to record that the harness no
   longer compiles C reference in the upstream-default configuration.
