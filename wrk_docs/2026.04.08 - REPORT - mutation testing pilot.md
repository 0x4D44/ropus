# Mutation Testing Pilot Report

Date: 2026-04-08

## Executive Summary

Piloted `cargo-mutants` v27.0.0 on `math_ops.rs` (966 lines, 271 mutants). Kill rate improved from 74.5% to **85.2%** after writing 34 targeted tests. Excluding 3 dead-code functions (17 mutants with zero callers), the live-code kill rate is ~90.4%.

The main value was not the kill rate itself — it was the **prioritized list of gaps** that mutation testing forced us to audit function-by-function:
- **8 live production functions with zero test coverage** (now tested)
- **3 dead-code functions** that should be removed (20 more dead wrappers found in types.rs)
- **Newton iteration weak spot**: sign changes in correction steps survive because tolerances are too loose. The fix exists (FFI differential tests via existing harness infrastructure) but was out of scope for the pilot.

**Important caveat**: The C reference oracle (280-config comparison harness) was NOT in the mutation testing loop. It runs as a separate binary, not via `cargo test`. The unit test kill rate understates the project's actual correctness assurance.

## Tool Setup

- **Tool**: `cargo-mutants` v27.0.0
- **Config**: `.cargo/mutants.toml` with 3x timeout, 4 parallel jobs
- **Baseline exclusions**: 3 pre-existing failing tests (DNN + opus encoder) excluded from mutation runs
- **Test baseline**: 776 passing library tests in ~0.3s

### Critical finding: C reference oracle not exercised

The 280-configuration differential comparison harness (`tests/harness/main.rs`) is a `[[bin]]` target, not `#[test]`. `cargo-mutants` only runs `cargo test`, so the "exceptionally strong mutation detector" (C reference oracle) was **not in the mutation testing loop**. The pilot measured unit test suite sensitivity only.

## Supplementary: range_coder.rs

Ran mutation testing on range_coder.rs (849 mutants) as a secondary data point. Early results showed ~33% kill rate — dominated by untested accessor/debug functions (`debug_state` tuple replacements, `get_val`, `error`, `restore_snapshot`). Confirms that mutation testing on code with sparse unit tests (15 tests for 1292 lines) produces mostly "no test" noise. This validates the brief's phased approach: improve coverage first, then mutation-test.

## Key Insights

1. **Mutation testing is a prioritization tool, not a discovery tool.** The same gaps could have been found by reading the coverage report. But mutation testing forced a function-by-function audit that produced a *ranked* list of where tests matter most. The 8 live untested functions were in the coverage data but not flagged as urgent until mutations survived.

2. **Dead code inflated the failure count.** 17/40 survivors (42.5%) came from 3 functions with zero callers. The headline kill rate (85.2%) is honest; the live-code rate (90.4%) is the more actionable number, but both should be reported.

3. **Newton iteration is a genuine weak spot.** Sign changes in correction steps survive because tolerances are too loose. This is fixable — the FFI infrastructure for differential tests already exists in `debug_helper.c` — but was deferred as out of scope for the pilot. This is the most actionable follow-up.

4. **Equivalent mutants are ~20% of survivors.** Boundary comparisons (`<` vs `<=`) in fixed-point code often produce equivalent behavior because exact boundary values are handled by other code paths. Inherent limitation; not worth chasing.

5. **The C reference oracle gap matters.** Without the comparison harness in the mutation loop, we measured unit test sensitivity, not overall correctness assurance. The 34 new tests are fast regression guards — valuable, but not the source of truth. The source of truth remains the 280-config harness.

## Phase 2 Decision

**Recommendation: Proceed to Phase 2, conditionally.**

Phase 2 targets (fft.rs, mdct.rs, bands.rs, quant_bands.rs) have lower branch coverage (67-78%). Running mutation testing on low-coverage code will mostly produce "no test at all" survivors — things we already know from the coverage report. The pilot's real value came from "weak test" and "Newton iteration" categories, which only emerge when tests *exist* but are insufficient. Phase 2 will have a worse signal-to-noise ratio until coverage catches up.

That said, it's cheap to run (~30-45 minutes) and there's no harm in trying.

**Prerequisites (must be done first):**
1. Clean up dead code from Phase 1 (removes noise in future runs)
2. Branch coverage improvements substantially complete (another agent is working on this)
3. Add FFI differential tests for Newton iteration functions (rsqrt, rcp) using existing `debug_helper.c` infrastructure — this is ~30 minutes of work, not a research project
4. Consider `--test-tool` config to include comparison harness in mutation loop
