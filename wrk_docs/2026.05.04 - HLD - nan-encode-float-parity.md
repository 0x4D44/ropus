## Behavior Gap

C fixed-point `opus_encode_float` converts input samples before the native
encoder call:

- `/home/md/language/ropus/reference/src/opus_encoder.c:2744` enters the
  fixed-point `opus_encode_float` branch.
- `/home/md/language/ropus/reference/src/opus_encoder.c:2760` assigns
  `in[i] = FLOAT2RES(pcm[i])`.
- `/home/md/language/ropus/reference/celt/arch.h:173` maps `FLOAT2RES(a)` to
  `FLOAT2INT16(a)`.
- `/home/md/language/ropus/reference/celt/float_cast.h:150` defines
  `FLOAT2INT16` as:

```c
x = x*CELT_SIG_SCALE;
x = MAX32(x, -32768);
x = MIN32(x, 32767);
return (opus_int16)float2int(x);
```

`MAX32` and `MIN32` are C ternary macros at
`/home/md/language/ropus/reference/celt/arch.h:102`:

```c
#define MIN32(a,b) ((a) < (b) ? (a) : (b))
#define MAX32(a,b) ((a) > (b) ? (a) : (b))
```

For NaN, both ordered comparisons are false. Therefore:

1. `MAX32(NaN, -32768)` returns `-32768`.
2. `MIN32(-32768, 32767)` returns `-32768`.
3. C feeds `-32768` into the fixed-point encoder.

Rust currently does the clamp in the opposite shape:

- `/home/md/language/ropus/ropus/src/types.rs:441` defines `float2int16`.
- `/home/md/language/ropus/ropus/src/types.rs:443` clamps high first with
  `if x > 32767.0`, which leaves NaN unchanged.
- `/home/md/language/ropus/ropus/src/types.rs:444` clamps low with
  `if x < -32768.0`, which also leaves NaN unchanged.
- `/home/md/language/ropus/ropus/src/types.rs:435` then casts
  `x.round_ties_even() as i32`; Rust's float-to-int cast maps NaN to `0`.

So the actual gap is:

| Input class | C `FLOAT2INT16` | Rust `float2int16` today | Parity |
|---|---:|---:|---|
| finite in range | rounded ties-to-even | rounded ties-to-even | match |
| finite above range | `32767` | `32767` | match |
| finite below range | `-32768` | `-32768` | match |
| `+Inf` | `32767` | `32767` | match |
| `-Inf` | `-32768` | `-32768` | match |
| NaN | `-32768` | `0` | diverges |

The known-failure README says "NaN/Inf" because the fuzz workaround zeros both.
The inspected conversion logic narrows the true codec parity bug to NaN.

## Minimal Fix

Change only `/home/md/language/ropus/ropus/src/types.rs::float2int16` so its
float-domain clamp matches the C macro order and NaN branch behavior:

```rust
pub fn float2int16(x: f32) -> i16 {
    let x = x * CELT_SIG_SCALE;
    let x = if x > -32768.0 { x } else { -32768.0 };
    let x = if x < 32767.0 { x } else { 32767.0 };
    float2int(x) as i16
}
```

This is intentionally not a broader non-finite sanitizer. It preserves C's
existing behavior:

- NaN maps to `-32768` through false ordered comparisons.
- `+Inf` maps to `32767`.
- `-Inf` maps to `-32768`.
- finite values keep the current ties-to-even rounding behavior.

Because this changes public crate behavior for NaN input to `encode_float`, bump
`/home/md/language/ropus/ropus/Cargo.toml` from `0.12.5` to `0.12.6` under the
workspace version policy.

## Risks

- C's NaN result is not an Opus API guarantee; it is an artifact of the current
  fixed-point reference implementation's macro order. Matching it is still the
  right choice for this repository because byte-exact parity is the harness
  contract.
- The proposed fix covers `FLOAT2INT16` parity, not all possible non-finite
  float behavior. At higher complexity and sample rates, `encode_float` also
  passes the original `f32` buffer to `downmix_float` for analysis. This HLD
  deliberately stays below that gate.
- Removing the fuzz workaround increases float-path input space. That is good,
  but it may uncover unrelated analysis or safety issues if a target later
  raises complexity beyond the current differential cap.
- The direct unit test locks in NaN-to-`-32768`, which is surprising from a
  Rust API perspective. The test name and comment should explicitly cite the C
  macro behavior so this does not look like arbitrary sanitization.
- Promoting or deleting known-failure files can collide with concurrent fuzz
  campaign work. Coordinate before removing the known-failure directory.

## Non-goals

- Do not reject NaN or infinity in the public `encode_float` API.
- Do not sanitize non-finite PCM to zero in production code.
- Do not refactor `float2int`, `encode_float`, `downmix_float`, or the analysis
  path.
- Do not change finite sample rounding or saturation behavior.
- Do not address high-complexity float analysis divergence.
- Do not wire fuzzing into CI or change long-running fuzz campaign scripts.
