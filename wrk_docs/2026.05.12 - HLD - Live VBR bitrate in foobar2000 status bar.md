# HLD — Live VBR bitrate in foobar2000 status bar

**Date:** 2026-05-12
**Component:** `foo_ropus` + `ropus-fb2k`
**Author:** Claude (with Arthur)

## 1. Problem

foobar2000 displays a single fixed bitrate for `.opus` files (e.g. "160 kbps")
for the whole track. That figure is the *file-wide average* computed once at
open time in
`d:\language\ropus\ropus-fb2k\src\reader.rs:895` and pushed to fb2k via
`p_info.info_set_bitrate(kbps)` in
`d:\language\ropus\foo_ropus\src\input_ropus_opus.cpp:218`.

Opus is VBR: each 2.5–60 ms packet is independently sized. The instantaneous
bitrate genuinely varies during playback and the user should see that.
The shim never tells fb2k about per-packet variation, so the status bar is
frozen on the file-wide mean.

## 2. Goal

The fb2k status bar shows a live bitrate that:

- starts at the file-wide average (no flicker at track start),
- tracks real per-packet bitrate with ~1 s smoothing,
- updates without jitter at the integer-kbps display resolution,
- survives seeking cleanly.

That is the entire user-visible deliverable. Anything else is out of scope.

## 3. Non-goals

- No new codec-mode / profile readouts. Bitrate only.
- No persistent stats, history, or instrumentation.
- No environment variables, no feature flags, no settings UI.
- No backward-compat for the existing FFI signature — this is a dev-environment
  change; the only caller of `ropus_fb2k_decode_next` is the C++ shim in this
  repo and it changes in the same commit.
- No fb2k SDK version negotiation. We override one virtual that has been on
  `input_decoder` since fb2k 1.x.

## 4. Design

### 4.1 Where smoothing lives

The Rust side reports raw bytes-consumed. The C++ side does the EWMA and the
fb2k dynamic-info handshake. Rationale:

- Smoothing parameters and display thresholds are presentation concerns, tied
  to fb2k's poll cadence. They belong next to the fb2k SDK calls, not in the
  decoder.
- Keeps the FFI surface minimal: one extra out-pointer, no new functions.
- Reader stays stateless w.r.t. UI — easier to reason about, easier to test.

### 4.2 FFI change (`d:\language\ropus\ropus-fb2k\include\ropus_fb2k.h`)

`ropus_fb2k_decode_next` gains a fourth parameter:

```c
int ropus_fb2k_decode_next(RopusFb2kReader*,
                           float*    out_interleaved,
                           size_t    max_samples_per_ch,
                           uint64_t* out_bytes_consumed);
```

Semantics: `*out_bytes_consumed` is set to the sum of encoded packet payload
bytes consumed during this call (i.e. the bytes that map to the
samples-per-channel returned). It is set on every successful return,
including the EOF return where samples == 0 and bytes_consumed == 0.

`out_bytes_consumed` is required (non-null). Passing NULL returns
`ROPUS_FB2K_BAD_ARG`. We don't keep a nullable variant — the only caller is
the shim and it always wants the value.

### 4.3 Rust side (`d:\language\ropus\ropus-fb2k\src\reader.rs`)

`Reader::decode_next` accumulates `pkt.data.len()` into a local `u64` on every
loop iteration before the discard / kept logic, and returns it alongside the
sample count via an out-parameter or a `(samples, bytes)` tuple. The decode
loop already iterates across packets during seek pre-roll discard
(`reader.rs:507 continue`); summing across the loop is the natural shape.

`ropus_fb2k_decode_next` in `src/lib.rs:398` writes the accumulated value to
`*out_bytes_consumed` on success. On error returns the value is left
unmodified — the C++ side won't read it because samples < 0 short-circuits.

### 4.4 C++ side (`d:\language\ropus\foo_ropus\src\input_ropus_opus.cpp`)

SDK contract (verified in `foo_ropus\sdk\foobar2000\SDK\input.h:99-103`):
`get_dynamic_info` is called *after every successful* `run()`, on the same
thread. That gives us two simplifications:

- No dirty flag needed — we compute-and-compare inside the dynamic-info call.
- No threading guards needed — single-threaded against `decode_run`.

Add two members to `input_ropus_opus`:

```cpp
double m_smoothed_bps  = 0.0;   // EWMA, bits/sec
int    m_reported_kbps = -1;    // last value we returned via dynamic info
```

`decode_initialize`: seed `m_smoothed_bps` with the file-wide average
(`m_info.nominal_bitrate` if `> 0`, else 0.0). Reset `m_reported_kbps = -1`
so the first dynamic-info return after replay-restart re-asserts the value.

`decode_run`: after a successful decode with `samples > 0`, update the EWMA:

```cpp
const double inst_bps = static_cast<double>(bytes_consumed) * 8.0 *
                        static_cast<double>(srate) /
                        static_cast<double>(samples);
if (m_smoothed_bps <= 0.0) {
    // Cold start (unseekable stream / nominal_bitrate < 0). Seed directly
    // from the first real packet rather than ramping from zero, which
    // would otherwise flash 3 → 6 → 9 kbps in the status bar over the
    // first second of playback.
    m_smoothed_bps = inst_bps;
} else {
    constexpr double ALPHA = 0.02;   // ~1 s @ 20 ms packets
    m_smoothed_bps = ALPHA * inst_bps + (1.0 - ALPHA) * m_smoothed_bps;
}
```

`decode_get_dynamic_info` (new override on the class — `input_stubs` already
provides a default-false implementation that we replace):

```cpp
bool decode_get_dynamic_info(file_info& p_out, double& p_timestamp_delta) {
    const int kbps = static_cast<int>((m_smoothed_bps + 500.0) / 1000.0);
    if (kbps == m_reported_kbps || kbps <= 0) return false;
    p_out.info_set_bitrate(kbps);
    p_timestamp_delta = 0.0;   // applies as of the chunk we just delivered
    m_reported_kbps   = kbps;
    return true;
}
```

The wrapper template `input_singletrack_translator_t::decode_get_dynamic_info`
(`sdk/foobar2000/SDK/input_impl.h:392`) forwards by member name — no
`virtual` keyword needed; this is template-time dispatch.

Seek: do not touch `m_smoothed_bps` or `m_reported_kbps`. The EWMA naturally
re-converges over its ~1 s time constant; pre-roll bytes are correctly
counted by 4.3, so the brief post-seek transient is bounded and short.

### 4.5 Smoothing parameter

α = 0.02, chosen for ~50-packet time constant at 20 ms packets ≈ 1 second.
This is the only knob. It lives as a `constexpr double` next to the update
math — no setting, no env var, no rebuild flag. If we ever want to tune it,
we tune it in code and rebuild.

At packet durations other than 20 ms (Opus permits 2.5/5/10/40/60 ms), the
time constant scales with packet rate. Worst case is 60 ms packets where
τ ≈ 3 s — still acceptable for a status-bar readout. We will not weight by
packet duration; the extra arithmetic isn't worth the complexity for a
display value.

### 4.6 Hysteresis

None beyond "only push when rounded integer kbps changes". Because EWMA τ is
~1 s, the rounded value will drift smoothly rather than flicker on integer
boundaries. If real-world testing shows flicker we add a ±1 kbps dead zone
later — but until we see it, we don't pre-pay for it.

## 5. Testing

### 5.1 Rust unit test

In `ropus-fb2k/tests/roundtrip.rs`, one new test:

- Open a known short fixture (something already used by existing roundtrip
  tests), call `decode_next` in a loop until EOF, sum reported
  `bytes_consumed`.
- Assert that the sum equals the total encoded payload bytes the reader
  consumed end-to-end. Cross-check by reading the same file with the Ogg
  packet reader directly.

That validates the FFI plumbing. The EWMA math is three lines of C++ and
doesn't need its own test; integration evidence (5.3) covers it.

### 5.2 No new C++ unit tests

The C++ shim has no test harness today and we are not adding one for three
lines of arithmetic. Cost > value.

### 5.3 Manual smoke test

1. Build per `foo_ropus\README.md`.
2. Install the `.fb2k-component`.
3. Play a known-VBR `.opus` track. Confirm the status bar bitrate starts at
   the file average and then glides as the track plays.
4. Seek mid-track; confirm the readout stays sensible (no -1, no huge
   transient lasting more than ~2 s).
5. Play a near-CBR track; confirm the readout stays glued to the nominal.

## 7. Risks / open questions

- **fb2k poll cadence.** Resolved on review: SDK calls `get_dynamic_info`
  after every `run()`, same thread (`input.h:99`). No cadence mismatch and
  no dirty flag.
- **Override mechanism.** Resolved on review: `input_singletrack_translator_t`
  forwards `decode_get_dynamic_info` by member name; `input_stubs`'s
  default-false stub is shadowed by our member of the same signature
  (`input_impl.h:99-110, 392`).
- **Thread safety.** Resolved on review: same-thread guarantee. No atomics.
- **Cold-start ramp** (unseekable streams, `nominal_bitrate < 0`). Handled
  in 4.4 by special-casing the first packet to set smoothed = inst directly,
  bypassing the EWMA on the cold-start update only.
- **Test call-site churn.** 14 mechanical updates in `roundtrip.rs`. Tedious
  but a single search-and-replace pass. No helper wrapper — that would just
  add a layer of indirection over the FFI the tests are supposed to be
  validating.
