# HLD: Torture Test Improvements — Five Blind Spots

## Section 1: Overview

A devil's advocate review of the mdopus torture-test suite identified five
blind spots where the existing harness (`cmd_torture`, `cmd_transitions`,
`cmd_sweep`, `cmd_plc`, `cmd_longsoak` in
`C:\language\mdopus\tests\harness\main.rs`) could in principle pass while a
real bug is quietly hiding. This HLD specifies how each gap will be closed:

1. **Cross-decoding** — decode every packet with both decoders, whether it
   came from C or Rust, so an encoder bug that yields "valid but wrong" output
   cannot be masked by a matching decoder bug.
2. **Deterministic transition bursts** — interleave short, targeted 3-frame
   sequences into the random soak to force the redundancy / prefill paths at
   `C:\language\mdopus\src\opus\encoder.rs:1452-1505`.
3. **Stereo→mono targeted test** — construct the exact five-condition AND at
   `C:\language\mdopus\src\opus\encoder.rs:1476-1486` with a scripted
   sequence, rather than hoping a random walk stumbles into it.
4. **PLC during mode transitions** — drop packets surrounding a SILK/CELT
   transition so `prev_redundancy` divergence (if any) becomes visible in
   PLC output (`C:\language\mdopus\src\opus\decoder.rs:665`, line 938, 969,
   1023, 1096).
5. **Periodic state comparison** — read every available `OPUS_GET_*` value
   from both encoders (and decoders) every N frames and compare, so a drifting
   internal counter is caught within one polling window.

All of this lives inside the existing `mdopus-compare` binary. Four items
extend `cmd_torture`; one new subcommand (`stereo-transition-test`) and one
new subcommand (`plc-transitions`) are added alongside it. `cmd_torture`
itself gains a `--cross-decode` flag and a `--state-check-interval N` flag so
the review gate can switch the extra work on without branching the command
tree.

No production code under `src/` changes. The only FFI addition is three new
`OPUS_GET_*_REQUEST` constants in
`C:\language\mdopus\tests\harness\bindings.rs`, which are already exposed by
the C reference library (`reference/include/opus_defines.h:131-179`) and
merely need to be declared on the Rust side.

## Section 2: Cross-Decoding

### Problem

Today's `cmd_torture`
(`C:\language\mdopus\tests\harness\main.rs:5502-5837`) encodes each frame
with both encoders, byte-compares the packets, then feeds **only the C
packet** to both decoders (line 5693). If the encoders drift apart but we
missed it, only the byte-compare catches it. If both decoders have a matching
bug that hides the drift in PCM space — for example, both silently ignore a
reserved-bit flip that changes future context — we never see it.

Cross-decoding decouples the encoder check from the decoder check: each
packet produced by either encoder is decoded by both decoders.

### Design

In cross-decode mode, every frame produces up to two encoded packets (C and
Rust). Each packet is decoded with both decoders. That is a 2×2 matrix per
frame:

```
         decode with C   decode with Rust
encode C:     A                B
encode R:     C                D
```

We perform all four decodes using **separate decoder instances** dedicated to
each column of the matrix. A single C decoder cannot alternate between C and
Rust packets frame by frame because mode-transition state is maintained
across packets and must track the sequence of packets that decoder has
actually seen. So cross-decode mode allocates:

- `c_dec_of_c_enc`  — fed the C-encoded stream only
- `r_dec_of_c_enc`  — fed the C-encoded stream only (Rust decoder instance)
- `c_dec_of_r_enc`  — fed the Rust-encoded stream only
- `r_dec_of_r_enc`  — fed the Rust-encoded stream only

Four decoders total. Encoders unchanged (still one C and one Rust).

### Equality criterion

The **strong** pass criterion is:
1. `c_pkt == rust_pkt` (encoder byte-equality — the existing check)
2. A.pcm == B.pcm == C.pcm == D.pcm (all four decoders produce identical
   i16 samples)
3. `final_range(A) == final_range(B) == final_range(C) == final_range(D)`
4. `decoder_state_count(A/B) == decoder_state_count(C/D)` — meaning
   `get_bandwidth()`, `get_last_packet_duration()` match across all four.

If the encoders already agree byte-for-byte (which is the project's hard
requirement), A == C and B == D trivially, so cross-decoding collapses to
"A == B" and is equivalent to the existing test. The value of cross-decoding
shows up only when the encoders **diverge** — which is exactly the bug class
we are worried about.

### Weaker fallback for encoder-divergence cases

If `c_pkt != rust_pkt`, strong equality is unreachable (the two input
streams to the decoders differ). In that case we want to know whether
**each encoder** is self-consistent with both decoders, because that is the
real signal:

- **Encoder-self-consistency check**: A == B (C packet decodes the same with
  C and Rust decoders) *and* C == D (R packet decodes the same with C and
  Rust decoders). This is the criterion we actually care about on divergent
  frames: it answers "given whatever the encoder chose to produce, do the
  decoders agree on its meaning?"

So cross-decode reports two counters:
- `encode_mismatches` (same as today)
- `cross_decode_mismatches` — frames where A != B **or** C != D

If `encode_mismatches > 0` but `cross_decode_mismatches == 0`, we know the
encoder bug is "semantically visible" — both decoders agree, they just see
two different valid-looking packets. If `cross_decode_mismatches > 0`, the
decoder disagrees about at least one packet's meaning, which is a decoder
bug (possibly independent of the encoder bug).

### Wiring

Cross-decoding is an extension to `cmd_torture`, gated by a new flag:

```rust
// cmd_torture signature additions
fn cmd_torture(
    duration_secs: i32,
    seed: u64,
    change_interval: usize,
    sample_rate: i32,
    channels: i32,
    cross_decode: bool,        // NEW
    state_check_interval: usize, // NEW (see Section 6)
) { ... }
```

CLI: `mdopus-compare torture --cross-decode --duration N`. Default off so
the normal torture run stays fast. `tools/torture_test.sh` will add one
invocation with `--cross-decode` at a shorter duration (Section 7).

Implementation note: `generate_noise` is deterministic from seed, so the
four decoders all see identical input sequences; no extra PCM buffering is
needed. Roughly 2x memory (two extra decoders) and roughly 2x decode time;
encoders are unchanged.

### Pseudo-code (new inner loop, abridged)

```rust
// Encode once with each
let c_len  = unsafe { bindings::opus_encode(c_enc, ...) };
let r_len  = rust_enc.encode(...)?;

// Four decodes
let (a_pcm, a_rng) = c_decode(c_dec_of_c_enc,  &c_pkt[..c_len]);
let (b_pcm, b_rng) = r_decode(r_dec_of_c_enc,  &c_pkt[..c_len]);
let (c_pcm, c_rng) = c_decode(c_dec_of_r_enc,  &r_pkt[..r_len]);
let (d_pcm, d_rng) = r_decode(r_dec_of_r_enc,  &r_pkt[..r_len]);

if c_pkt[..c_len] == r_pkt[..r_len] {
    // Strong: all four must agree
    require(a_pcm == b_pcm && b_pcm == c_pcm && c_pcm == d_pcm);
    require(a_rng == b_rng && b_rng == c_rng && c_rng == d_rng);
} else {
    encode_mismatches += 1;
    // Weak: self-consistency per encoder
    if a_pcm != b_pcm || a_rng != b_rng { cross_decode_mismatches += 1; }
    if c_pcm != d_pcm || c_rng != d_rng { cross_decode_mismatches += 1; }
}
```

"Equal" throughout this section means bit-identical: `c_pcm[..n] ==
r_pcm[..n]` as `[i16]` slices of the same length, and `u32 == u32` for the
final range. No tolerance is applied; the whole project is bit-exact.

## Section 3: Deterministic Transition Bursts

### Problem

`random_torture_config` (`main.rs:5389-5435`) builds a fresh config every
~50 frames with uniformly-weighted choices. The redundancy path at
`src\opus\encoder.rs:1458-1473` fires when `prev_mode` and current `mode`
differ in CELT-only-ness; the prefill path at line 1500 fires on
CELT→SILK specifically. Both only happen if the encoder is actually in
SILK/Hybrid at frame N and forced to CELT at N+1, etc. Since `force_mode`
is chosen at 3/5 weight for `OPUS_AUTO`, and even when non-AUTO it is picked
independently at each config change, the probability of hitting the exact
three-frame pattern needed to light up both paths plus the stereo→mono
transition plus the redundancy-with-prefill edge is tiny.

### Design

Add an **interleaver** to `cmd_torture`. Every `burst_interval` frames
(default 200), suspend the random walk and run a scripted 3-frame burst
chosen deterministically from a small table. After the burst, resume the
random walk from where it was.

### Burst table

A burst is `(name, [(cfg, n), (cfg, n), (cfg, n)])` where each config is
applied and held for exactly `n` frames (usually 1 or 2). The list is
rotated through deterministically — burst index
`(frame_idx / burst_interval) % BURSTS.len()`.

| # | Name                            | Step 1                | Step 2               | Step 3                |
|---|---------------------------------|-----------------------|----------------------|-----------------------|
| 1 | silk_celt_silk                  | force_mode=SILK,20ms  | force_mode=CELT,20ms | force_mode=SILK,20ms  |
| 2 | celt_silk_celt                  | force_mode=CELT,20ms  | force_mode=SILK,20ms | force_mode=CELT,20ms  |
| 3 | silk_celt_silk_short            | force_mode=SILK,10ms  | force_mode=CELT,10ms | force_mode=SILK,10ms  |
| 4 | hybrid_celt_hybrid              | force_mode=HYBRID¹,20 | force_mode=CELT,20   | force_mode=HYBRID¹,20 |
| 5 | stereo_mono_stereo              | force_channels=2      | force_channels=1     | force_channels=2      |
| 6 | stereo_mono_auto                | force_channels=2      | force_channels=1     | force_channels=AUTO   |
| 7 | vbr_cbr_at_low_br               | vbr=1,br=8000         | vbr=0,br=8000        | vbr=1,br=8000         |
| 8 | framesize_60_2p5_20             | frame_ms=60.0         | frame_ms=2.5         | frame_ms=20.0         |
| 9 | framesize_20_40_20              | frame_ms=20.0         | frame_ms=40.0        | frame_ms=20.0         |
|10 | bitrate_crash_6k_510k_6k        | br=6000               | br=510000            | br=6000               |
|11 | bandwidth_NB_FB_NB              | max_bw=NB             | max_bw=FB            | max_bw=NB             |
|12 | fec_off_on_off                  | fec=0                 | fec=1,loss=20        | fec=0                 |

¹ "HYBRID" is requested via `force_mode=1001` in the config struct; the
  encoder treats it as a SILK-with-CELT mode. Note that the frame size 2.5
  ms in burst 8 must be run through CELT-capable configs only (i.e. each
  step's `frame_size` is a property of the burst, not the surrounding
  config).

All other config fields for the burst step come from a baseline "neutral"
config (bitrate 64k, complexity 10, VBR on) unless overridden above. The
random config in effect immediately before the burst is **saved** and
re-applied immediately after, so the burst is a true injection, not a
regime change.

### Frequency

- Default `--burst-interval = 200`. At 20 ms frame size that is 4 seconds
  of random walk between each 60 ms burst. In a 30-minute soak we hit
  ~450 bursts, cycling through the table 37 times, which is overkill but
  cheap.
- Random walk unchanged otherwise (still every `--change-interval`, default
  50).
- With 12 bursts in the table, the user gets ~37 pulls per burst variant
  per 30-minute run, enough to be statistically meaningful.

### Selection rule

**Deterministic round-robin** from a fixed ordered table, not pseudo-random:

```rust
let burst_idx = (frame_idx / burst_interval) % BURSTS.len();
```

Rationale: deterministic means a seed-and-bug report can be reproduced
exactly. A random selection on top of an already-random walk would make
reduction harder, not easier. The burst table is constant; adding new
bursts is an append operation that only changes the schedule on future
runs.

### Pseudo-code

```rust
// Inside cmd_torture's frame loop, alongside the existing change_interval
// block:

let in_burst = burst_interval > 0
    && frame_idx >= burst_interval
    && (frame_idx - burst_start) < BURST_LEN;  // 3 or 6 frames

if burst_interval > 0 && frame_idx % burst_interval == 0 && !in_burst {
    saved_cfg = current_cfg.clone();
    burst_start = frame_idx;
    burst = &BURSTS[(frame_idx / burst_interval) % BURSTS.len()];
    step = 0;
    current_cfg = burst.steps[0].apply_to(&NEUTRAL_BASE);
    apply_config_to_both(&current_cfg);
} else if in_burst && /* step advance needed */ {
    step += 1;
    if step < burst.steps.len() {
        current_cfg = burst.steps[step].apply_to(&NEUTRAL_BASE);
        apply_config_to_both(&current_cfg);
    } else {
        // Burst complete — restore random-walk config
        current_cfg = saved_cfg.clone();
        apply_config_to_both(&current_cfg);
    }
}
```

Bursts count toward `configs_tested` and any mismatch inside a burst gets a
specialised log line: `BURST MISMATCH burst=silk_celt_silk step=2 ...`
so the diagnostic call-out is better than today's random-config dump.

### Testing

A unit test in the harness (not in `cmd_torture`; a small `cmd_burst_check`
if we want it separable, but simpler: add a branch inside `cmd_torture` that
runs a single pass through every burst once and reports) will verify that
each burst produces a byte-identical encode on both sides when run
stand-alone. That guards against the burst table itself silently shifting
under future refactors.

## Section 4: Stereo→Mono Targeted Test

### Problem

The condition at `src\opus\encoder.rs:1476-1486` is:

```rust
if self.stream_channels == 1
    && self.prev_channels == 2
    && self.silk_mode.to_mono == 0
    && mode != MODE_CELT_ONLY
    && self.prev_mode != MODE_CELT_ONLY
{
    self.silk_mode.to_mono = 1;
    self.stream_channels = 2;   // NB: stays 2, not 1!
}
```

Five conditions must hold in the same frame:

1. The current mode decision chose **mono** (stream_channels == 1) — can be
   caused by `force_channels=1` or by auto-downmix at low bitrate.
2. The **previous** frame ran as stereo (`prev_channels == 2`) — needs at
   least one prior successful encode in stereo.
3. `silk_mode.to_mono == 0` — i.e. we have not already executed this
   transition in the immediately preceding frame.
4. The current frame is **not** CELT-only — i.e. SILK or Hybrid.
5. The **previous** frame was not CELT-only — i.e. SILK or Hybrid too.

Random walks almost never hit this because `force_channels` randomises
independently of `force_mode` and bitrate, and because the "hold SILK for
two consecutive frames with different channel counts" requirement is a
two-frame memory condition — random config changes every 50 frames make it
much more likely that the encoder is already in a stable SILK-mono steady
state before `force_channels` gets randomised.

### Design

New subcommand `stereo-transition-test` — no CLI arguments, fully scripted,
runs a handful of sub-scenarios to exercise variations of the condition.

Each sub-scenario follows a strict shape:
1. Create **stereo** (2-ch) encoders at 48 kHz.
2. Apply a base config: `bitrate = 96000`, `force_mode = 1000` (SILK_ONLY),
   `max_bandwidth = OPUS_BANDWIDTH_WIDEBAND`, `force_channels = OPUS_AUTO`.
3. Encode `N_PRE = 10` frames of 20 ms noise. After this, both encoders
   are deep in stereo SILK with stable state.
4. Apply a **mutation** that should trigger the transition. The mutation
   varies per sub-scenario — see below.
5. Encode exactly **one** frame (the "transition frame"). C and Rust must
   agree byte-for-byte.
6. Immediately query the transition flags (see below) on both encoders and
   compare.
7. Encode `N_POST = 3` more frames. Each must match byte-for-byte and
   decode to identical PCM.

### Sub-scenarios

| # | Name                      | Mutation before transition frame                              |
|---|---------------------------|----------------------------------------------------------------|
| 1 | force_channels=1          | `set_force_channels(1)`                                        |
| 2 | force_channels=1, hybrid  | base `force_mode=HYBRID (1001)`, then `set_force_channels(1)`  |
| 3 | force_channels=1, 10ms    | base `frame_ms=10.0`, then `set_force_channels(1)`             |
| 4 | downmix_by_low_bitrate    | leave channels AUTO, drop bitrate to 12000 (triggers auto mono) |
| 5 | repeat_transition         | do scenario 1, encode 5 more frames, then re-set force=1 again |

Scenario 5 is important because after the first transition, `silk_mode.to_mono
== 1` blocks re-entry for exactly one frame. Within a few frames the
encoder will clear it; forcing another stereo→mono shortly after should
produce the same behaviour again. A bug in the clear-down path (missing
one-shot reset) would only show up in scenario 5.

### Verification in the transition frame

The most informative signal is the internal state immediately after the
transition encode completes. The existing debug hook in
`bindings.rs:316-323` is:

```rust
pub fn debug_get_encoder_hp_state(
    enc: *mut OpusEncoder,
    hp_mem_out: *mut opus_int32,
    variable_hp_smth2: *mut opus_int32,
    mode_out: *mut opus_int32,
    stream_channels_out: *mut opus_int32,
    bandwidth_out: *mut opus_int32,
);
```

For this test we need slightly more: `to_mono` flag and `prev_channels`.
Rather than add new FFI symbols, the cheaper approach is to **read from
the packet itself**:

- `opus_packet_get_nb_channels(&c_pkt[0])` / `rust packet.nb_channels()` —
  Rust has an equivalent introspection function. A mono packet after a
  stereo history is a necessary (but not sufficient) signature of the
  transition.
- TOC byte comparison: `c_pkt[0] == rust_pkt[0]`. The TOC encodes the
  stream-channel count and mode; if the two encoders disagree on the
  transition-frame header, this catches it.

For **full verification** (not just necessary), we use the existing
`debug_get_encoder_hp_state` to read `stream_channels_out` immediately
after the transition frame. Per C line 1483 the expected value is **2,
not 1**, on the transition frame itself (the comment in the C source is
explicit about this non-obvious behaviour). Verifying this pinned value
exercises the exact line we care about.

`to_mono` itself is not currently exposed. Adding a single helper to
`bindings.rs` (and the matching `extern "C"` stub in
`tests/harness/build.rs`'s shim) would make the test sharper, but it is
not strictly required — byte equality + `stream_channels == 2` is already
stronger than what random walks can check.

### Pass criterion

For each sub-scenario:
1. All 14 encoded frames (10 pre + 1 transition + 3 post) are byte-identical
   between C and Rust encoders.
2. All 14 decoded PCM outputs are sample-identical.
3. On the transition frame, the C and Rust encoders both report
   `stream_channels == 2` (via `debug_get_encoder_hp_state`).
4. Final range after each frame matches.

### Wiring

New subcommand `stereo-transition-test`. Uses `run_transition_sequence`-style
code paths — in fact the simplest implementation is to build a
`Vec<(EncodeConfig, usize)>` for each sub-scenario and call
`run_transition_sequence` with it, plus extend `run_transition_sequence` to
optionally query `debug_get_encoder_hp_state` at a specific frame index
(the "transition frame"). The function already has access to the live
encoders, so this is a small addition.

CLI: `mdopus-compare stereo-transition-test`. No arguments.
`tools/torture_test.sh` calls it alongside `transitions`.

## Section 5: PLC During Mode Transitions

### Problem

`OpusDecoder::decode` at `src\opus\decoder.rs:665` selects the PLC mode
based on `prev_redundancy`:

```rust
let plc_mode = if self.prev_redundancy {
    MODE_CELT_ONLY
} else {
    self.prev_mode
};
```

`prev_redundancy` is set at line 1096:
```rust
self.prev_redundancy = redundancy && !celt_to_silk;
```

If C and Rust disagree about `redundancy` or `celt_to_silk` on the frame
**before** a packet loss, the PLC paths diverge: one picks CELT PLC, the
other picks SILK PLC, and the decoded "lost" frame is wildly different
between implementations. **Current tests never drop packets during a mode
switch**, so this path is untested.

### Design

New subcommand `plc-transitions`. Fully deterministic; no wav input. Uses
`generate_noise` for PCM input so the test is self-contained and
reproducible.

### Test shape

For each sub-scenario, the script:

1. Creates C and Rust encoders + four decoders (two for each side; see
   below).
2. Encodes a prefix sequence of N frames in **SILK-only** mode.
3. Encodes a transition into **CELT-only** (forces `redundancy=true,
   celt_to_silk=false`, which is the interesting branch).
4. Possibly drops one of the packets around the transition.
5. Decodes with C and Rust decoders, passing `None` for dropped frames.
6. Compares PCM output byte-for-byte.

We need **two** decoders per side, because once you drop a packet, you
cannot "un-drop" it — the dropped-packet decoder state diverges from the
received-packet state immediately. To keep the comparison honest, the C and
Rust decoders on each sub-scenario **see the exact same sequence of
packets** (received, dropped, received...), so A vs B is comparing "both
decoders doing the same PLC on the same inputs".

### Sub-scenarios

Let the encoder stream be:
```
[ S₀ S₁ S₂ S₃ S₄ S₅ S₆ S₇ S₈ T C₀ C₁ C₂ C₃ C₄ ]
```
where S are SILK-only frames, T is the first frame of the transition back
to CELT (or to CELT with redundancy active, depending on direction), and
C are CELT-only frames after.

| # | Name                 | Direction    | Dropped packet | What it tests                          |
|---|----------------------|--------------|----------------|-----------------------------------------|
| 1 | plc_before_silk_celt | SILK→CELT    | S₈ (last SILK) | PLC while `prev_mode=SILK`, then real CELT decode |
| 2 | plc_on_silk_celt     | SILK→CELT    | T              | PLC on the transition packet itself    |
| 3 | plc_after_silk_celt  | SILK→CELT    | C₀             | PLC in CELT after transition frame received |
| 4 | plc_before_celt_silk | CELT→SILK    | last CELT      | PLC with `prev_redundancy` possibly true |
| 5 | plc_on_celt_silk     | CELT→SILK    | transition pkt | Transition packet lost in the CELT→SILK direction |
| 6 | plc_after_celt_silk  | CELT→SILK    | first SILK     | PLC after transition received, before next is |
| 7 | burst_around_transition | SILK→CELT | T AND T+1     | Two-packet loss straddling a transition |
| 8 | drop_transition_no_recovery | SILK→CELT | T, T+1, T+2 | Three-frame loss — maximum PLC stress over a transition |

For each scenario, an EncodeConfig sequence is built with
`run_transition_sequence`-style steps (N×SILK then N×CELT, where SILK has
`force_mode=1000, max_bandwidth=WIDEBAND, bitrate=16000` and CELT has
`force_mode=1002, bitrate=128000`). The decoder loop then walks the packet
list and drops the specified indices by passing `None` in place of the
packet.

### Decoder state

Sample rate fixed at 48 kHz, mono, 20 ms frames. This keeps the test short
and focused; stereo and multi-frame-size variants can be added later if
needed but are not required to prove `prev_redundancy` correctness.

### Equality criterion

For every frame of the output stream (including PLC-filled gaps):
- `c_pcm[i] == rust_pcm[i]` as `[i16]`, strict equality.
- `c_dec.get_final_range() == rust_dec.get_final_range()` after each
  decode call. This is the most sensitive check: the range coder's final
  range is a hash of every choice the decoder made; two PLC paths that
  produce visually similar PCM but took different branches will have
  different final ranges.

### Why "dropped vs received" does not cause false positives

The concern: when the C decoder and Rust decoder have different internal
buffer layouts but identical arithmetic, feeding `None` might update
different state fields in slightly different ways yet still be "correct".

We do not care. The test fails if and only if **the decoded PCM differs**
or **the final range differs**. Both are observable, deterministic
functions of the external API. There is no "invisible" internal state to
worry about from a black-box test perspective, and indeed the
`opus_decode(..., NULL, 0, ...)` entry point is part of the public API
that promises PLC.

### Wiring

New subcommand `plc-transitions`, no CLI args (fixed parameters). The
implementation is structurally identical to `cmd_plc`
(`main.rs:2747-2811`), with the encode step swapped for
`run_transition_sequence`-generated packets and the drop list hard-coded
per sub-scenario.

`tools/torture_test.sh` gains an invocation
`mdopus-compare plc-transitions` after the existing `plc` tests.

## Section 6: Periodic State Comparison

### Problem

A diverged internal counter — VAD state, pitch prediction memory, an LPC
filter tap, a smoothed noise estimate — can take thousands of frames to
produce an audible divergence in the packets we compare. But it would
show up immediately if we read the encoder's reported internal state via
the `OPUS_GET_*` CTLs and compared. The current torture test only
compares `final_range` on the decoder side.

### Design

Extend `cmd_torture` with a periodic "state check" that runs every N
frames (default 1000). At each checkpoint, for both the encoder pair and
the decoder pair, call every available `OPUS_GET_*` CTL on both sides and
compare the returned values.

### CTLs to query on the encoder

From `reference/include/opus_defines.h:131-181`:

| CTL                                    | Rust getter                         | Type |
|----------------------------------------|-------------------------------------|------|
| `OPUS_GET_BITRATE_REQUEST` (4003)      | `get_bitrate()`                     | i32  |

### FFI additions needed

`bindings.rs` currently exposes only `OPUS_GET_FINAL_RANGE_REQUEST` and
`OPUS_GET_BANDWIDTH_REQUEST` (lines 63-64). This HLD requires adding
16 more constants (the values are already in `opus_defines.h`; nothing
changes in the C source). These are pure numeric constants, no function
signatures required — `opus_encoder_ctl` and `opus_decoder_ctl` are
already declared with `...` varargs.

### Checkpoint frequency and placement

Default `--state-check-interval 1000`. At 20 ms frames, that is every
20 seconds of encoded audio. In a 30-minute run that is ~90 checkpoints.
Cost per checkpoint: ~25 CTL calls × 4 endpoints (c_enc, r_enc, c_dec,
r_dec) = 100 calls, all of which are constant-time lookups on both sides
(~microseconds total). Negligible overhead.

A checkpoint **also fires at end of test**, unconditionally. The goal is
to catch drift that happened but never surfaced as a packet mismatch.

### What to do when state differs but packets match

This is the interesting policy question. There are three possibilities:

1. **Informational**: state mismatch that is not currently producing
   different packets. Example: an internal smoothed value differs by 1
   fixed-point ULP because of a rounding order, but the encoder's final
   decision is the same at the current input.
2. **Latent bug**: state mismatch that will eventually produce different
   packets. Indistinguishable from (1) at checkpoint time.
3. **Spec violation**: a documented getter returns different values. By
   definition a user-visible bug, regardless of packet output.

Policy: **fail by default, with an opt-out.** A state mismatch that the
packet comparison has not caught is exactly the kind of silent drift this
check was designed to find. We should stop the test and dump the full
state table (which is cheap to print).

Rationale: the project's hard requirement is bit-exact behaviour; "state
differs but packets match" is already inconsistent with the rules. If it
turns out to be noisy (i.e. genuine but harmless quirks in
"getter returns sampled internal state"), the fix is to narrow the set of
getters we check, not to downgrade failures to warnings.

Add `--state-check-strict` (default ON) so the first few runs can be
looser while we calibrate, then flip.

### Wiring

Extension to `cmd_torture`, not a separate command. New flag
`--state-check-interval N` (default 1000, 0 disables).
`tools/torture_test.sh` does not change; the default behaviour is the
same as today plus an extra check every 1000 frames. One
`--state-check-interval 0` invocation is added for regression — a
test to confirm the check itself doesn't change outcomes.

### Pseudo-code

```rust
// Once per checkpoint:
fn state_checkpoint(
    frame_idx: usize,
    c_enc: *mut bindings::OpusEncoder,
    r_enc: &mdopus::opus::encoder::OpusEncoder,
    c_dec: *mut bindings::OpusDecoder,
    r_dec: &mdopus::opus::decoder::OpusDecoder,
) -> Result<(), String> {
    let mut diffs = vec![];

    macro_rules! check_enc {
        ($req:expr, $rust_expr:expr, $name:literal) => {{
            let mut c_val: i32 = 0;
            unsafe { bindings::opus_encoder_ctl(c_enc, $req, &mut c_val); }
            let r_val = $rust_expr;
            if c_val != r_val {
                diffs.push(format!("{}: C={} R={}", $name, c_val, r_val));
            }
        }};
    }
    check_enc!(bindings::OPUS_GET_BITRATE_REQUEST, r_enc.get_bitrate(), "bitrate");
    check_enc!(bindings::OPUS_GET_BANDWIDTH_REQUEST, r_enc.get_bandwidth(), "bandwidth");
    // ... all listed CTLs ...

    // Same for decoder
    // ...

    if diffs.is_empty() {
        Ok(())
    } else {
        Err(format!("frame {}: {} state diffs: {:?}", frame_idx, diffs.len(), diffs))
    }
}
```

## Section 8: Risk Assessment

### Cross-decoding (Section 2)

- **False positive risk: low.** All four decoders see deterministic
  inputs; the existing bit-exact equality is used as the comparator.
  The only new failure mode is "encoder outputs are different but
  decoders agree with each encoder individually", which is a genuine bug
  signal (the encoders disagree).
- **Over-engineering: no.** This is the canonical Fuzz-Swiss-cheese fix
  for encoder/decoder coverage.
- **Complexity: moderate.** Four decoder instances, two encoded streams
  per frame, four-way comparison logic. The decoder instance bookkeeping
  is the fiddly part; encoders are unchanged.

### Deterministic transition bursts (Section 3)

- **False positive risk: very low.** Bursts are small, deterministic
  configs that should encode cleanly on both sides. If any burst fails
  stand-alone on both implementations, the burst is excluded or marked
  as a known incompatibility — but at current pass rates there's no
  reason to expect this.
- **Over-engineering: no.** Each burst targets a specific branch the
  DA identified.
- **Complexity: moderate.** A burst table, deterministic scheduler,
  state save/restore around bursts. Straightforward but bulky.

### Stereo→mono targeted test (Section 4)

- **False positive risk: low.** Pinning
  `stream_channels == 2` after the transition frame is a documented
  behaviour (C source comment at line 1567). If that check ever goes
  red, it's a real bug.
- **Over-engineering: possibly.** The existing
  `Stereo->ForceMono->Auto->Stereo` transition test
  (main.rs:6177-6203) already hits a version of this. The gap is that
  it does not **verify the internal state flag**, only the output bytes.
  A narrower extension to that existing test might suffice. See open
  question #1.
- **Complexity: low.** Mostly reuses existing scaffolding.

### PLC during mode transitions (Section 5)

- **False positive risk: moderate.** Packet drops introduce
  implementation-specific choices in state recovery. The PCM
  bit-equality requirement is correct because PLC output must also be
  deterministic across implementations — but if there is an
  implementation bug in either decoder's PLC specifically, this test
  will flag **decoder bugs** rather than bugs in the transition logic
  we actually wanted to test. That's fine (it's still a real bug) but
  failure attribution requires care.
- **Over-engineering: possibly.** The DA finding is that
  `prev_redundancy` could diverge in PLC. If our encoder is correct and
  the decoders agree on normal packets, the divergence is actually very
  unlikely. Lower priority than the other items.
- **Complexity: significant.** Deterministic drop schedule, dual
  decoder instances, careful about FEC-vs-PLC flag (we want pure PLC,
  so `decode_fec=false`).

### Periodic state comparison (Section 6)

- **False positive risk: low if we pick the getters right.** Static
  getters (`sample_rate`, `lookahead`) will never drift. Dynamic getters
  (`bandwidth`, `final_range`, `last_packet_duration`, `in_dtx`) are
  the ones most likely to surface a bug — and if they do surface a
  bug, it's a real one. The one getter we should watch is `get_pitch`
  on the decoder, because that can differ by one sample due to
  tie-breaking. If it shows spurious mismatches, it comes out of the
  checklist.
- **Over-engineering: no.** 100 CTL calls every 1000 frames is free.
  The main cost is the 16 new FFI constant declarations.
- **Complexity: trivial.** A single function with a fixed list of
  `check_enc!` and `check_dec!` macro calls.

### Aggregate

Risk summary: expect **0-1 flaky items** among the five, most likely
Section 5 (PLC during transitions) or one specific getter from Section 6.
Both are fixable by narrowing scope rather than weakening the criterion.

## Section 9: Open Questions

1. **Should stereo-transition-test be a new subcommand, or extend the
   existing `transitions` command?** The existing command is a flat
   list of 8 cases; adding a 9th with internal-state-verification would
   be a style break (none of the existing cases poke at internal
   state). A separate subcommand is cleaner. Team to decide if "one
   command for mode transitions, one for the stereo corner case" is
   the right factoring or if we prefer everything under `transitions`.

2. **Do we want `to_mono` and `prev_channels` exposed via a new
   `debug_get_encoder_channel_state` shim in `tests/harness/build.rs`?**
   It would make the stereo transition test strictly more sensitive,
   at the cost of a new debug-FFI helper. Current design gets most of
   the value from `stream_channels_out` already in
   `debug_get_encoder_hp_state`. Worth a small diff but can be deferred.

3. **Cross-decoding in `transitions` too?** Section 2 only wires it
   into `cmd_torture`. The transition tests
   (`cmd_transitions`, `main.rs:6049`) would also benefit, because
   they exercise exactly the cases where an encoder divergence would
   be most likely to produce a decoder-invisible bug. Low cost to
   extend — should we do it as part of this HLD or defer?

4. **Default for `--burst-interval` when the user also sets
   `--change-interval`.** If a user sets
   `--change-interval 10` to speed up coverage, the burst interval of
   200 is 20× larger — still fine, but worth documenting. If the
   user sets `--change-interval 1000`, bursts at 200 are in the
   middle of a static config — also fine, but less obviously
   interesting. Should `--burst-interval` auto-scale from
   `--change-interval`? I lean toward "no, keep it independent".

5. **State check failure policy.** Section 6 proposes
   "`--state-check-strict` on by default, with opt-out". The alternative
   is "warn on first run, flip to strict once we've characterised
   typical noise." Preference is strict-by-default because we're
   bit-exact elsewhere and drift-shaped bugs are the scary ones.
   Team to confirm.

6. **PLC `decode_fec` mode.** The PLC-during-transition test at
   Section 5 uses pure PLC (`decode_fec=false`). Should there be a
   parallel test that uses FEC recovery across transitions? That would
   require the encoder to be producing FEC side information, which
   exists but is a different code path. Scope question: one test or
   two?

7. **Cross-decode cost at 30 minutes × 4 streams.** Worth an early
   benchmark on typical hardware. If cross-decode mode makes a 30-min
   torture run take 2 hours, we reduce the default duration for the
   cross-decode test invocation. Current plan is "10 min" in
   `tools/torture_test.sh`, which should be safe, but this is a
   guess until measured.

8. **Where should per-burst diagnostics (Section 3) live?** The
   existing `format_torture_config` (`main.rs:5478`) dumps a single
   config. For bursts we want to say "burst=X step=Y within-burst-frame=Z",
   which is a different schema. Propose extending `cmd_torture`'s
   mismatch-print code with a small enum `{ Random(cfg), Burst(name,
   step, frame) }` that wraps the context. Team review welcome on
   whether that's worth the refactor vs just printing a second line.

## Section 10: Decisions

Team-lead answers to the open questions in Section 9 — these lock the
design for implementation.

1. **stereo-transition-test is a new subcommand**, not an extension of
   `transitions`, because its internal-state verification style differs
   from the flat 8-case pattern of the existing command.

2. **Do not add a `to_mono` / `prev_channels` debug FFI shim now**; the
   `stream_channels_out` field already exposed by
   `debug_get_encoder_hp_state` is sufficient, and the shim can be added
   later if scenario 5 (repeat_transition) needs sharper signal.

3. **Cross-decoding is wired into both `cmd_torture` and
   `cmd_transitions`**, because the transition tests are exactly where
   encoder divergence is most likely and the four-decoder infrastructure
   is shared.

4. **`--burst-interval` stays independent of `--change-interval`** (no
   auto-scaling); the interaction is documented in CLI help text.

5. **State check failure policy is strict by default**
   (`--state-check-strict=true`), matching the project's bit-exact
   requirement; `--state-check-strict=false` exists as an opt-out for
   calibration runs.

6. **FEC-recovery across mode transitions is deferred as a follow-up
   test**; the PLC-only coverage in Section 5 is sufficient for this
   HLD, with a note that the FEC variant is a known future extension.

7. **Cross-decode cost will be benchmarked during implementation** and
   the `tools/torture_test.sh` duration tuned from measurement; the
   initial 10-minute plan is the starting point, not a commitment.

8. **Per-burst diagnostics use the
   `{ Random(cfg), Burst(name, step, frame) }` enum refactor**, because
   the diagnostic improvement justifies the small refactor cost.

### Scope lock for this session

- **Implementing now**: Sections 2 (cross-decoding, including in
  `cmd_transitions` per decision 3), 3 (deterministic transition bursts),
  and 6 (periodic state comparison).
- **Deferred**: Section 4 (stereo→mono targeted test) and Section 5
  (PLC during mode transitions). Both remain in the HLD as the
  reference design for a future implementation session.
