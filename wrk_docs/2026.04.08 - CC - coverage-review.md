## Main Findings

1. The current suite is heavily skewed toward low-level helper tests.
2. The largest gaps are in stateful codec execution paths:
   - `silk::encoder`
   - `silk::decoder`
   - `opus::encoder`
   - `opus::decoder`
   - `opus::multistream`
3. Branch coverage is much worse than line coverage, which means the suite is touching surfaces without exercising the real decision trees.
4. The repo already contains a large harness binary (`tests/harness/main.rs`) with many executable comparison commands, but that binary currently contributes almost no coverage because it had zero actual tests.
5. The fastest honest way to move coverage is to turn those harness commands into automated tests, then fill remaining holes with direct unit tests in zero-test modules.

## Risk Assessment

- 98% overall coverage is not close. It is a major testing project.
- The repo is large and parity-sensitive; careless assertions can create brittle tests.
- The biggest gains are likely from integration-style execution, not from adding more micro-tests to already-covered math modules.
- Branch coverage will be the hardest metric to drag upward, especially in SILK and Opus control flow.

## Immediate Recommendation

Prioritize the existing harness first:

1. Convert stable `mdopus-compare` command paths into actual automated tests.
2. Use those tests to drive end-to-end encode/decode, PLC, FEC, DTX, repacketizer, packet parsing, and multistream flows.
3. Only after that, attack zero-test source files directly.

This is the only path that has a realistic chance of producing large coverage gains quickly.
