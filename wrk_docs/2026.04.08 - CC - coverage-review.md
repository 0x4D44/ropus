## What The Numbers Mean

### 1. The shortfall is dominated by a few large files

The top five files account for most of the missing lines. That is good news because broad repo-wide cleanup is unnecessary. The practical route upward is targeted work in those modules, not small cosmetic additions across already well-covered code.

### 2. Branch coverage is the weak metric

Line and region coverage are already respectable in many modules. Branch coverage is the real gap, especially in:

- `src/silk/encoder.rs`
- `src/opus/encoder.rs`
- `src/dnn/lpcnet.rs`
- `src/celt/decoder.rs`
- `src/celt/bands.rs`
- `src/opus/repacketizer.rs`

That points to untested mode switches, invalid-input handling, PLC/FEC/DTX transitions, codec decision heuristics, and branchy packet/layout logic rather than missing “basic smoke” tests.

### 3. Public API smoke coverage exists but is not deep enough

`src/coverage_tests.rs` already exercises important end-to-end API flows, which is why the public surface is not collapsed. What is missing is branch depth: invalid ctl values, transition-order bugs, mode forcing edge cases, and error/fallback behavior.

### 4. Internal DSP helpers are generally healthier

Many smaller CELT, SILK, and DNN helper modules already sit in the 94-100% band. That means the fastest path to improvement is not writing more arithmetic micro-tests; it is covering the large control-flow units that orchestrate those helpers.

## Risks And Constraints

- The missing `reference/` checkout means parity-harness coverage is out of scope for now.
- `98%` line coverage is theoretically reachable with enough targeted work, but `98%` branch coverage is far more expensive from the current `69.35%` baseline.
- `src/silk/encoder.rs` is the single biggest missed-line pool, but it is also the easiest place to burn time for weak returns if tests are not chosen carefully.
- The current tree emits warnings in `src/opus/multistream.rs` and one unused variable warning in `src/celt/bands.rs`; they did not block coverage, but they are noise during validation.

## Recommended Attack Order

1. `src/opus/encoder.rs`
2. `src/celt/decoder.rs`
3. `src/dnn/lpcnet.rs`
4. `src/opus/multistream.rs`
5. `src/silk/encoder.rs`
6. `src/celt/bands.rs`

This order balances three things:

- large missed-line pools
- branch-heavy behavior
- likelihood of adding deterministic unit tests without needing the external reference harness

## Current Conclusion

The repository is already moderately well covered, but the remaining gap is concentrated in the hardest kind of code: branch-heavy codec control paths. Reaching 98% will require multiple deliberate test waves against the large orchestrator modules, not one-off touch-up tests.
