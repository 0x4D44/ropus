## Structural Findings

1. Existing tests are heavily skewed toward helpers, invariants, and narrow unit behavior.

Evidence:

- 386 library tests exist already.
- Many low-level math/tables modules are already very well covered.
- The weakest areas are the large orchestration files where mode selection, bandwidth decisions, PLC/FEC/DTX paths, stereo transitions, resampling, and packet layout branches live.

2. The biggest ROI is end-to-end codec-path testing, not more micro-tests in already-healthy files.

Reason:

- `silk\encoder.rs`, `silk\decoder.rs`, `opus\encoder.rs`, and `opus\decoder.rs` contain the bulk of missed lines.
- Small helper tests will not close a 14k-line gap fast enough.

3. `celt\ec_ctx.rs` is a cheap isolated win.

Reason:

- It has only 93 executable lines and almost no coverage.
- Focused tests can hit trait dispatch for `RangeEncoder`/`RangeDecoder` and the default panic paths quickly.

4. The binary targets are effectively uncovered.

Reason:

- `src\main.rs` is at 0%.
- `tests\harness\main.rs` has no unit tests.
- This is not the main percentage driver, but it is dead surface today.

5. DNN coverage is mixed.

Reason:

- `dnn\lossgen.rs` is already healthy.
- `dnn\core.rs`, `dnn\lpcnet.rs`, and `dnn\fargan.rs` still have substantial untouched logic.
- These are likely better as a later-stage sweep after the codec runtime paths.

## Assessment of the 98% Goal

Brutally: the repo is too far from 98% for that to be reached by one or two surgical patches. The target is only realistic with a broad, staged test campaign that:

- exercises all major encode/decode modes
- covers stereo/mono transitions
- drives FEC, PLC, DTX, VBR/CBR, bandwidth caps, force-mode, and frame-duration branches
- adds dedicated tests for `silk::common`, `celt::ec_ctx`, and the lower-coverage DNN helpers
- repeats full branch coverage measurement after every stage

The next implementation stages should prioritize absolute uncovered-line reduction, not cosmetic percentage gains in already healthy files.

## Recommended Priority Order

1. `src\opus\encoder.rs`
2. `src\opus\decoder.rs`
3. `src\silk\encoder.rs`
4. `src\silk\decoder.rs`
5. `src\celt\ec_ctx.rs`
6. `src\silk\common.rs`
7. `src\opus\multistream.rs`
8. `src\dnn\core.rs`
9. `src\dnn\lpcnet.rs`
10. `src\dnn\fargan.rs`

Why this order:

- The first four are the highest-value runtime orchestrators.
- `ec_ctx` is an easy isolated jump.
- `silk\common` is a broad utility dependency and still thinly covered.
- `multistream` already has some end-to-end tests and is a good extension point after single-stream coverage rises.
- DNN modules are meaningful, but they are not the first place to spend effort while the core codec control-flow is still largely dark.
