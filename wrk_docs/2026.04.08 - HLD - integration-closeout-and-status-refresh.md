# Integration Closeout and Status Refresh

## Problem

The codec port itself is no longer the main gap. The repo now has a large Rust implementation, a wide comparison harness, saved green runs, and recent journals showing that parity work reached the finish line for the current scope.

The problem is that the repo's official integration story is stale:

- `src/` contains 37 Rust files and about 35.8k lines of Rust code.
- `wrk_journals/2026.04.05 - JRN - VICTORY.md` records a 280/280 pass result.
- `wrk_journals/2026.04.07 - JRN - test harness expansion.md` records a much broader harness with 11 high-level commands plus fuzz targets.
- `logs/integrate_20260406_194133.log` shows a saved run with 280 passed, 0 failed, 0 skipped.
- `tools/coordinator.py phase integrate` still fails because it runs `cargo test --test integration`, but there is no `integration` test target.

So the repo contains a green implementation, but the top-level tooling still claims integration is unfinished. That is the next problem to solve.

## Goal

Make the repo's integration story truthful and reproducible using the tools that already exist.

After this change:

1. There is one obvious green path Arthur can run.
2. The coordinator integrate phase succeeds when the code is green.
3. The README and status surfaces match the real workflow.
4. Normal green-path outputs are readable and not polluted by old debug traces.

## Non-Goals

- No new codec algorithm work.
- No new testing framework.
- No feature flags or environment-variable driven modes.
- No migration or rollback support.
- No attempt to merge every tool into one giant script.

## Design

### 1. Use the existing gates instead of inventing new ones

The repo already has two useful validation layers:

1. `python tools/integrate.py test`
   Purpose: broad corpus regression over the generated WAV set.
   Today this is the source of the saved 280/280 result.

2. `cargo run --bin mdopus-compare -- test-all`
   Purpose: feature-path regression across API parity, sweep, PLC, FEC, DTX,
   packet parsing, decode formats, repacketizer, multistream, longsoak, and
   quality sanity.

The continuation change should make these two commands the canonical integration gate.

This is intentionally simple. We already have both tools. The problem is orchestration, not missing capability.

### 2. Redefine the coordinator integrate phase around the real gates

`tools/coordinator.py` should stop calling the nonexistent `cargo test --test integration`.

Replace the current integrate phase with:

1. Run `python tools/integrate.py test`.
2. If that passes, run `cargo run --bin mdopus-compare -- test-all`.
3. Mark `integrate` complete only if both commands return success.

Coordinator behavior should stay simple:

- log which gate failed;
- keep the existing log file behavior;
- update `coordinator_state.json` only on full success;
- avoid parsing deep output beyond pass/fail.

No new state schema is needed.

### 3. Keep debug tools, but remove debug noise from the normal path

The repo has useful debug commands and ad-hoc tracing hooks from the parity hunt. That is fine. The mistake is letting those traces leak into normal success artifacts such as saved integration output.

The continuation change should:

- keep explicit debug commands such as the stereo VOIP and 2.5 ms helpers;
- remove or disable stray debug `println!`/`eprintln!` calls from normal encode, decode, roundtrip, sweep, and `test-all` execution paths;
- keep normal command output focused on pass/fail and first-difference reporting.

This is not cosmetic only. Clean output makes green runs easier to trust, compare, and archive.

### 4. Refresh the status surfaces

The repo should stop making Arthur infer the truth from scattered journals.

Refresh the lightweight status surfaces:

- `README.md`
  Add the real validation commands and a short current-state summary.

- `CLAUDE.md`
  Either refresh the status text or mark it clearly as advisory if it is no longer maintained.

- coordinator status
  Once the integrate phase is fixed, `python tools/coordinator.py status` becomes useful again.

This does not require a new documentation system. It only requires the existing entry points to agree.

## Why This Scope Is Right

This project does not need another architecture pass. The port is already large, the harness is already broad, and the journals show the heavy codec debugging has largely happened.

The highest-value next step is to make the repo admit that reality:

- green code should have a green top-level workflow;
- current commands should be the documented commands;
- integration evidence should be easy to regenerate.

Anything broader would be churn.

## Execution Order

1. Remove debug noise from normal harness and codec execution paths.
2. Update `tools/coordinator.py` integrate phase to run the two real gates.
3. Refresh `README.md` and any still-used status notes.
4. Run the canonical gate:
   - `python tools/integrate.py test`
   - `cargo run --bin mdopus-compare -- test-all`
5. Confirm `python tools/coordinator.py phase integrate` and `python tools/coordinator.py status` now report success.

## Verification Plan

The change is correct when all of the following are true:

1. `cargo build` succeeds.
2. `python tools/integrate.py test` succeeds and reports the corpus result cleanly.
3. `cargo run --bin mdopus-compare -- test-all` succeeds.
4. `python tools/coordinator.py phase integrate` succeeds.
5. `python tools/coordinator.py status` shows `integrate` in `completed_phases`.
6. Saved normal-path outputs no longer contain ad-hoc debug tags from old investigations.

## Scope

| File | Change |

| `src/opus/encoder.rs` and other touched codec files | Only remove leftover debug tracing from normal execution paths; no algorithm changes in this continuation |

| `CLAUDE.md` | Refresh or clearly downgrade to advisory status |

## Risks

### Risk 1: False green from saved artifacts

Saved logs already show green runs, but the continuation change should not trust those alone. It must rerun the gates locally.

### Risk 2: In-flight codec edits

The worktree is currently dirty in `src/opus/encoder.rs`. If codec work is still in progress, run the integration closeout after that work stabilizes, or keep the coordinator/doc edits separate from the codec branch.

### Risk 3: Overreach

There will be a temptation to fold `integrate.py` into the harness, redesign the coordinator, or invent a richer test dashboard. Resist that. The repo needs one honest path, not another layer.

## Summary

The port appears complete enough for the current development target. The continuation is not "build more codec." The continuation is:

- use the existing regression gates as the official integrate phase;
- clean the normal output path;
- refresh the repo status so the code, logs, and commands all tell the same story.

That is the smallest change that materially improves day-to-day confidence.
