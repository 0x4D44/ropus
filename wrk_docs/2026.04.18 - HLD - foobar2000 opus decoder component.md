# HLD — foobar2000 input component backed by ropus

**Status:** draft
**Author:** Arthur + Claude
**Date:** 2026-04-18

## 1. Motivation

We already have a working Rust Opus decoder (`ropus`) and a libopus-shaped C ABI
(`capi`). foobar2000 is a widely-used Windows audio player with a stable C++
component SDK. Putting our decoder behind an `input_impl` service gives us:

- a realistic long-form listening test (hours of varied content, live seeking,
  A/B against foobar2000's built-in libopus decoder),
- a publishable artifact that exercises the C boundary in a third-party host,
- confidence that the decoder's threading and resource behaviour survives
  something more stressful than our harness.

foobar2000 already decodes Opus via its bundled libopus. The new component
replaces that path for `.opus` files with our code.

## 2. Scope

**In:**
- Read `.opus` files in an Ogg container (RFC 7845), channel-mapping family 0
  (mono / stereo), 48 kHz.
- Decode to interleaved PCM and hand foobar2000 an `audio_chunk`.
- Report duration, bitrate, channel count, sample rate in file properties.
- Surface `OpusTags` (vorbis_comment) so foobar2000 can show artist / title /
  album / date.
- Seekable playback.
- Ship as a single `.fb2k-component` (zip) containing one DLL.

**Out (explicit non-goals):**
- Writing / editing tags.
- Multistream / surround (channel_mapping 1 or 255).
- Gapless chaining across logical streams.
- ReplayGain scanning (reading an existing `R128_TRACK_GAIN` tag is fine; we
  won't compute one).
- Decoder options panel (fb2k preferences page).
- Non-Ogg containers (Matroska, MP4).
- Encoding (diskwriter). Playback only.
- Any attempt to replace foobar2000's built-in libopus at link time. Ours is a
  separate input that registers a content-type handler.

## 3. Architecture

```
  .opus on disk
       │
       ▼
┌────────────────────────────────────┐
│ fb2k-mdopus (C++ DLL)              │   ← foobar2000 SDK
│  input_impl_mdopus_opus            │
│    open / decode_initialize        │
│    decode_run                      │   hands audio_chunk to fb2k
│    decode_seek / decode_get_info   │
└─────────────────┬──────────────────┘
                  │ C ABI (mdopus_fb2k.h)
                  ▼
┌────────────────────────────────────┐
│ mdopus-fb2k (Rust cdylib)          │
│  OggOpusReader                     │   Ogg demux + page index
│    └─> ropus::Decoder              │   actual decode
└────────────────────────────────────┘
```

Two artefacts. The Rust cdylib owns the file, the Ogg demux, and the decoder.
The C++ DLL is a thin adapter: open file → pass path to Rust → pump decoded
samples into fb2k's audio_chunk → forward seeks.

This division keeps the C++ layer small and the interesting logic in Rust
where we can test it.

## 4. Rust crate — `mdopus-fb2k`

New top-level workspace crate. `cdylib` only.

```
mdopus-fb2k/
├── Cargo.toml
├── include/
│   └── mdopus_fb2k.h         # hand-written, shipped to the C++ side
└── src/
    ├── lib.rs                # #[no_mangle] entry points + ffi_guard
    ├── reader.rs             # OggOpusReader: demux, header parse, seek index
    └── error.rs              # tiny thread-local last-error slot
```

### 4.1 Dependencies

- `ropus` (workspace path) — the decoder.
- `ogg = "0.9"` — same Ogg crate ropus-cli already depends on.

Nothing else. No symphonia, no async runtime, no logging framework.

### 4.2 C header

```c
// mdopus_fb2k.h  (hand-written; stable across the C++ DLL and Rust cdylib)

#include <stdint.h>
#include <stddef.h>

typedef struct MdopusFb2kReader MdopusFb2kReader;

typedef struct {
    uint32_t sample_rate;     /* always 48000 for our scope */
    uint8_t  channels;        /* 1 or 2 */
    uint16_t pre_skip;        /* 48 kHz samples, already consumed on first read */
    uint64_t total_samples;   /* per-channel, post-pre_skip; 0 if unknown */
    int32_t  nominal_bitrate; /* bits/s; -1 if unknown */
} MdopusFb2kInfo;

/* All ints returning a status: 0 = OK, negative = error code (see err table). */

MdopusFb2kReader* mdopus_fb2k_open(const char* path_utf8);
void              mdopus_fb2k_close(MdopusFb2kReader*);

int mdopus_fb2k_get_info(MdopusFb2kReader*, MdopusFb2kInfo* out);

/* Tag enumeration. Keys/values are UTF-8, null-terminated, valid only for the
 * duration of the callback. Returns 0 on success. */
typedef void (*MdopusFb2kTagCb)(void* ctx, const char* key, const char* value);
int mdopus_fb2k_read_tags(MdopusFb2kReader*, MdopusFb2kTagCb cb, void* ctx);

/* Decode next packet into caller buffer (interleaved float, 48 kHz).
 * `max_samples_per_ch` must be >= 120*48 = 5760 (longest Opus frame).
 * Returns samples-per-channel decoded, 0 on EOF, negative on error. */
int mdopus_fb2k_decode_next(MdopusFb2kReader*,
                            float*  out_interleaved,
                            size_t  max_samples_per_ch);

/* Seek to per-channel sample position (48 kHz). Clamped to [0, total_samples]. */
int mdopus_fb2k_seek(MdopusFb2kReader*, uint64_t sample_pos);

/* Thread-local last-error string for the calling thread. Never NULL. */
const char* mdopus_fb2k_last_error(void);
```

Sized and shaped for what `input_impl` actually needs — nothing else.

### 4.3 Internals

- `OggOpusReader::open(path)`:
  1. mmap-free `BufReader<File>` wrapping the file.
  2. Read first page → parse `OpusHead`. Reject channel_mapping ≠ 0 up front.
  3. Read second page → parse `OpusTags`, keep as `Vec<(String, String)>`.
  4. Build a seek index: walk the file once, record
     `(absolute_granule, byte_offset)` at every Ogg page boundary. On a
     5-minute 64 kbps track this is ≈ 15 k entries — few hundred KB — and
     keeps seek `O(log n)` without a second disk scan.
  5. Capture `nominal_bitrate` from file-size × 8 / duration.

- `decode_next`: drive `PacketReader`, hand each Opus packet to `ropus::Decoder::decode_float`,
  write interleaved samples into the caller's buffer, trim leading pre-skip
  on first call. Return per-channel count.

- `seek(sample_pos)`:
  1. Binary-search the page index for the largest `granule_pos ≤ target`.
  2. Seek the underlying reader to that page's byte offset.
  3. Reset `ropus::Decoder` state (the decoder exposes a reset path already
     used by `capi`).
  4. Discard decoded samples until we pass `target`.

  This is the Opus-recommended pattern (RFC 7845 §4.2). No pre-roll of 80 ms
  for perfect seeks — the audible artifact on a stream-boundary reset is
  below the threshold for casual listening. If it proves offensive, add the
  pre-roll later.

- All `#[no_mangle] extern "C"` entry points wrap their bodies in
  `std::panic::catch_unwind` using the same `ffi_guard!` idiom as `capi`. A
  panic becomes a negative status + a thread-local error string.

### 4.4 Threading

foobar2000 calls `decode_run` from a single worker thread per input instance.
Our reader is `!Sync` but `Send`; each handle is owned by exactly one thread.
The Rust side does not spawn threads and does not hold global mutable state
apart from the thread-local last-error slot.

### 5.1 SDK

foobar2000 SDK is not redistributable. `sdk/` is git-ignored; README.md
documents downloading it from foobar2000.org and unpacking into that folder.
No submodule — the SDK changes rarely and vendoring a zip is more robust than
mirroring a repository we don't control.

### 5.2 `input_impl` responsibilities

Single class `input_mdopus_opus` implementing `input_impl`:

| `input_impl` method       | Implementation |
|---------------------------|----------------|
| `open`                    | `mdopus_fb2k_open(path)`; throw `exception_io_data` on NULL. Call `get_info` and stash. |
| `get_info`                | Translate `MdopusFb2kInfo` into `file_info`. Fill artist/title/etc. via `read_tags`. |
| `get_file_stats`          | Forward to foobar2000's filesystem layer. |
| `decode_initialize`       | No-op (already open). |
| `decode_run`              | Call `decode_next` into a scratch `float[2 * 5760]`. Fill `audio_chunk_impl`. Return `false` on EOF. |
| `decode_seek`             | Convert seconds → 48 kHz samples, forward to `mdopus_fb2k_seek`. |
| `decode_can_seek`         | Return true. |
| `decode_get_dynamic_info` | Return false — we don't change bitrate mid-stream. |
| `decode_on_idle`          | No-op. |
| `retag / remove_tags`     | Throw `exception_io_unsupported_format`. We don't write. |

Registered via `input_singletrack_factory_t<input_mdopus_opus>` with content
type `audio/ogg` / `audio/opus` and extension `.opus`. Required statics on
the class:

| Static                                 | Value |
|----------------------------------------|-------|
| `g_get_guid()`                         | A fresh, hard-coded GUID generated once for this component. Distinct from any built-in. |
| `g_get_name()`                         | `"mdopus (Rust Opus decoder)"` — shown in the prefs list. |
| `g_is_our_path(path, ext)`             | `stricmp_utf8(ext, "opus") == 0` |
| `g_is_our_content_type(ct)`            | `audio/ogg` or `audio/opus` |
| `g_is_low_merit()`                     | `false` |

### 5.3 Winning the `.opus` association

foobar2000 ≥ 1.4 exposes a user-editable decoder priority list at
*Preferences → Playback → Decoding*. `input_manager_v2::open_v2` walks the
registered `input_entry_v2` services in that list, catching
`exception_io_no_handler_for_path` / `exception_io_unsupported_format` and
falling through to the next one. First non-throwing opener wins.

Where a newly-installed decoder lands on first sighting is controlled by
`g_is_low_merit()`:

- `false` → inserted at the **top** of the list. This is what we want.
- `true`  → inserted at the **bottom**. vgmstream uses this deliberately
  because it does *not* want to beat built-ins; we do the opposite.

After first install, the user's saved order is authoritative — fb2k never
overwrites it. So:

1. **On a fresh install of our component**, the combination of a distinct
   GUID + `is_low_merit == false` puts us above `foo_input_std`'s Opus entry
   and we handle `.opus` without any user action.
2. **On a machine where fb2k has already indexed Opus files before our
   component existed**, we land at the top of new entries but the user's
   existing priority list is untouched. In practice the built-in is still
   on top. README must document the one-time fix: drag *mdopus* above
   *Standard Input Array* on the Decoding prefs page (or Ctrl+Shift+Up).

Precedent: foo_input_monkey is described in the foobar2000 v2.26 preview
changelog as *"You can once again install Monkey's Audio decoder component
to replace built-in FFmpeg decoder"* — same pattern.

### 5.4 Error translation

- Rust returns negative code → C++ throws `exception_io_data` with the message
  from `mdopus_fb2k_last_error()`.
- Rust returns NULL handle → same.
- foobar2000 catches `exception_io_*` itself and skips the track; no
  additional logging needed.

## 6. Build and packaging

### 6.1 Rust side

Standard `cargo build --release -p mdopus-fb2k` produces, on Windows MSVC:

- `target/release/mdopus_fb2k.dll`
- `target/release/mdopus_fb2k.dll.lib` (import library)

These drop into the C++ project's `lib/` folder. No cbindgen — the header is
hand-written and reviewed. It's ~40 lines; generating it adds a build
dependency for no win.

### 6.2 C++ side

CMake project targeting Visual Studio 2022, Windows SDK 10. One `add_library`
producing a DLL renamed `foo_input_mdopus.dll`. Links:

- `lib/mdopus_fb2k.dll.lib`
- `sdk/foobar2000/shared/shared-Release.lib` and friends

Post-build step: copy the DLL + `mdopus_fb2k.dll` into a zip named
`foo_input_mdopus.fb2k-component`. That file is what users drop into
foobar2000.

### 6.3 No workspace integration of the C++ build

We don't wire CMake into `cargo build`. Two reasons: (1) it pulls Windows-only
toolchain detection into the workspace; (2) the Rust workspace CI doesn't need
the component built on every push. A standalone `fb2k-mdopus/build.bat` is
enough — it runs `cargo build -p mdopus-fb2k --release` then `cmake --build`.

## 7. Testing

Tier 1 — Rust-only (runs in normal CI):
- Unit tests in `mdopus-fb2k/tests/` that round-trip a known `.opus` through
  the public C API using `libloading` or direct linkage as an `rlib`. Decoded
  PCM must match what `ropus-cli decode` produces sample-for-sample.
- Seek test: open a ≥ 30 s file, seek to 10 s, decode the next 200 ms,
  compare against a contiguous reference decode windowed at 10.0–10.2 s
  (allowing a small tolerance for the stream-boundary reset, quantified
  experimentally on first run).

Tier 2 — manual (documented in the component README):
- Install in foobar2000, play a handful of tracks covering 32 / 64 / 128 / 256
  kbps, mono and stereo.
- Seek within each track; confirm audio resumes without a stall or click.
- A/B against foobar2000's built-in Opus decoder on the same files. Any
  audible difference is a bug.
- Confirm the component loads on a clean foobar2000 install with no extra
  Visual C++ runtime bits (we statically link the CRT or document the
  dependency).

Tier 3 — deferred:
- Automated playback-regression across a corpus. Useful, but out of scope
  for MVP and belongs in a wider conformance-over-time effort.

## 8. Open questions

1. **Foobar2000 SDK version target.** SDK 2024-03 drops support for older
   runtimes. Pick a concrete version and pin it in the README.

2. **CRT linkage.** `/MT` (static) avoids a `vcruntime140.dll` dependency but
   bloats the DLL and complicates mixing with fb2k's own allocator. `/MD`
   (dynamic) is smaller but the user needs VS 2022 runtime installed.
   foobar2000 itself ships `/MD`; match them.

*(Resolved — association priority: see §5.3. Our component wins on fresh
install via a distinct GUID + `g_is_low_merit() == false`; documented manual
reorder for users who indexed Opus files before installing us.)*

## 9. Milestones

Each is committable on its own and leaves the tree green.

1. **M1 — Rust cdylib skeleton.** `mdopus-fb2k` crate compiles, exposes
   `mdopus_fb2k_open` / `close` / `last_error`. Round-trips a header-only
   parse for a real `.opus` file in a Rust test. ~1 day.

2. **M2 — decode + info.** `decode_next` and `get_info` wired to
   `ropus::Decoder`. Rust test compares against `ropus-cli decode` byte-for-byte.
   ~1 day.

3. **M3 — tags + seek.** `read_tags` and `seek` with the page index. Rust
   test covers both. ~1 day.

4. **M4 — C++ component shell.** Empty `input_impl` that logs its methods;
   loads in foobar2000, shows up in Components list. Proves SDK wiring.
   ~half a day once SDK is in place.

5. **M5 — wire decoding.** Flesh out `input_mdopus_opus` using the Rust FFI.
   First audible playback. ~1 day.

6. **M6 — packaging.** `build.bat` + `.fb2k-component` zip. README with
   install steps. ~half a day.

Total: roughly a calendar week of focused work.

## 10. Explicitly not included

- No feature flags. No environment variables. One build configuration, one
  release artifact.
- No cross-platform story. foobar2000 is Windows; so is this component.
  `mdopus-fb2k` itself is portable Rust and will happen to build on Linux,
  but we don't promise or test that.
- No back-compat shim for a pre-existing `foo_input_mdopus`. There isn't one.
- No tag-writing path or preferences page. Adding either is a separate HLD.
