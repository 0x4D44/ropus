# VOIP Mode HP Filter and Mode Selection Fixes

## Problem

The Opus encoder's VOIP mode produces divergent output compared to the C reference. Five bugs have been identified in the HP filter path and mode selection logic in `src/opus/encoder.rs`. Bug 7 (biquad rounding) is the root cause of the largest divergence: because `hp_cutoff()` processes every sample in VOIP mode, a per-sample rounding error cascades through the entire encoding pipeline.

The remaining bugs (8-10 and the mode_music index) compound on top of this, affecting the HP smoothing coefficient, DTX behavior, voice estimation, and mode threshold calculation.

## Root Causes

| Bug | Summary | Severity | Effect |
|-----|---------|----------|--------|
| 7 | Biquad output uses round-half-up instead of ceiling | CRITICAL | Every VOIP sample can differ by +/-1; cascading divergence |
| 8 | HP smoothing coefficient 492 (Q15) instead of 983 (Q16) | HIGH | HP cutoff frequency drifts over time |
| 9 | SILK DTX ignores `is_silence` | MEDIUM | DTX active during silence, alters mode decisions |
| 10 | `voice_ratio` not reset per-frame | MEDIUM | Stale voice estimate affects mode selection |
| extra | `mode_music` uses `MODE_THRESHOLDS[0][1]` not `[1][1]` | LOW | Currently benign (both values are 10000), but structurally wrong |

## Fix Plan

### Bug 7: Wrong biquad output rounding

**Location:** `src/opus/encoder.rs:589`, function `silk_biquad_res`

**C behavior (line 398):**
```c
out[ k*stride ] = silk_SAT16( silk_RSHIFT( out32_Q14 + (1<<14) - 1, 14 ) );
```
This is a ceiling operation: add `(1 << 14) - 1` then right-shift by 14. For positive values, this rounds up. For negative values, this rounds toward zero.

**Rust behavior (line 589):**
```rust
output[k * stride] = silk_sat16(silk_rshift_round(out32_q14, 14)) as i16;
```
`silk_rshift_round` adds `1 << (shift-1)` = `1 << 13` = 8192, then right-shifts. This is round-half-up, which differs from ceiling for all values where the fractional part is less than 0.5.

**Fix:** Replace `silk_rshift_round(out32_q14, 14)` with an explicit ceiling: `(out32_q14 + (1 << 14) - 1) >> 14`. This is a single-line change.

**Risk:** LOW. Purely arithmetic, directly matching C. The state update lines (579, 584) correctly use `silk_rshift_round` matching C's `silk_RSHIFT_ROUND` -- only the output line differs.

### Bug 8: HP smoothing coefficient off by 1 LSB

**Location:** `src/opus/encoder.rs:72` (constant definition) and line 1742 (usage)

**C behavior (opus_encoder.c:1974-1975):**
```c
silk_SMLAWB(st->variable_HP_smth2_Q15,
    hp_freq_smth1 - st->variable_HP_smth2_Q15,
    SILK_FIX_CONST(VARIABLE_HP_SMTH_COEF2, 16));
```
`SILK_FIX_CONST(0.015, 16)` = `(int)(0.015 * 65536 + 0.5)` = 983.
`silk_SMLAWB(a, b, c)` = `a + ((b * (int16)c) >> 16)`.
Net operation: `a += (diff * 983) >> 16`.

**Rust behavior (line 1742):**
```rust
self.variable_hp_smth2_q15 += mult16_32_q15(VARIABLE_HP_SMTH_COEF2, diff);
```
`VARIABLE_HP_SMTH_COEF2 = qconst16(0.015, 15)` = `(0.5 + 0.015 * 32768)` = 492.
`mult16_32_q15(492, diff)` = `(492 * diff) >> 15`.
Net operation: `a += (diff * 492) >> 15` = `(diff * 984) >> 16`.

The coefficient is 984/65536 instead of 983/65536 -- a 1 LSB difference in Q16 that accumulates over frames, slowly shifting the HP cutoff frequency.

**Fix:** Two changes:
1. Change the constant to Q16: `const VARIABLE_HP_SMTH_COEF2: i32 = 983;` (or use an equivalent of `SILK_FIX_CONST(0.015, 16)`)
2. Change the smoothing line to use `silk_smlawb` (or manual `>> 16` multiply-accumulate) instead of `mult16_32_q15`.

**Risk:** LOW. Purely arithmetic. The constant value changes from 492 to 983 and the shift changes from 15 to 16, matching C exactly.

### Bug 9: SILK DTX ignores is_silence

**Location:** `src/opus/encoder.rs:1277`

**C behavior (opus_encoder.c:1463, in `#else` / FIXED_POINT path):**
```c
st->silk_mode.useDTX = st->use_dtx && !is_silence;
```
When silence is detected, SILK's internal DTX is disabled so the generalized (Opus-level) DTX logic handles silence instead.

**Rust behavior (line 1277):**
```rust
self.silk_mode.use_dtx = if self.use_dtx != 0 { 1 } else { 0 };
```
The `is_silence` check is missing. SILK DTX remains enabled during silence, which changes how the encoder handles silent frames and affects mode selection downstream.

**Fix:** Add the `!is_silence` condition:
```
self.silk_mode.use_dtx = if self.use_dtx != 0 && !is_silence { 1 } else { 0 };
```

Note: The Rust port does not use float analysis (`DISABLE_FLOAT_API` equivalent), so the `#else` branch of the C code is the correct reference. The `is_silence` variable is already computed at line 1152.

**Risk:** LOW. Simple boolean logic addition.

### Bug 10: voice_ratio not reset per-frame

**Location:** `src/opus/encoder.rs`, between lines ~1151-1154 (after `is_silence` computation, before voice estimation at line 1224)

**C behavior (opus_encoder.c:1275-1276 and 1306-1307):**
In the `#ifndef DISABLE_FLOAT_API` path, `voice_ratio` is reset to -1 when `!is_silence`.
In the `#else` path (which the Rust port corresponds to), `voice_ratio` is unconditionally reset to -1.

Either way, `voice_ratio` is reset at the start of each encode call. This ensures the voice estimation at line 1224-1239 uses the default paths (`voice_est = 115` for VOIP, `48` otherwise) unless `voice_ratio` is explicitly set by the application between frames.

**Rust behavior:** Never resets `voice_ratio`. The field initializes to -1 in the constructor (line 898) but is never reset during encoding. If the application calls `set_voice_ratio()`, the value persists across all subsequent frames.

**Fix:** Add `self.voice_ratio = -1;` after the `is_silence` computation, before the voice estimation block. This matches the C `#else` path (unconditional reset).

**Risk:** LOW. Single assignment matching C behavior.

### Bug extra: mode_music interpolation index

**Location:** `src/opus/encoder.rs:1288-1290`

**C behavior (opus_encoder.c:1498-1499):**
```c
mode_music = (MULT16_32_Q15(Q15ONE-stereo_width, mode_thresholds[1][1])
    + MULT16_32_Q15(stereo_width, mode_thresholds[1][1]));
```
Both terms use `mode_thresholds[1][1]`. This appears intentional in the C reference -- the music threshold is always the stereo value regardless of stereo_width. (This is likely a deliberate design choice since music detection is less sensitive to channel count.)

**Rust behavior (line 1288):**
```rust
let mode_music = MODE_THRESHOLDS[0][1]
    + (stereo_width as i64 * (MODE_THRESHOLDS[1][1] - MODE_THRESHOLDS[0][1]) as i64
        / Q15ONE as i64) as i32;
```
Uses `MODE_THRESHOLDS[0][1]` as the base, then interpolates toward `[1][1]`. Currently both values are 10000 so the interpolation term is zero and the result is identical. But if the table values ever change, this would diverge.

**Fix:** Change the `mode_music` calculation to use `MODE_THRESHOLDS[1][1]` for both terms, matching C. Simplest form: `let mode_music = MODE_THRESHOLDS[1][1];` since both MULT16_32_Q15 terms sum to `mode_thresholds[1][1]` when both use the same index.

**Risk:** NONE. Currently produces identical output; fix is for structural correctness only.

## Execution Order

Fix bottom-up by severity and dependency:

1. **Bug 7** (biquad ceiling) -- root cause, largest per-sample divergence
2. **Bug 8** (HP smoothing coefficient) -- second HP filter bug, logically adjacent
3. **Bug 10** (voice_ratio reset) -- independent, trivial
4. **Bug 9** (DTX is_silence) -- independent, trivial
5. **Bug extra** (mode_music index) -- cosmetic, zero risk

Bugs 7 and 8 are in the HP filter pipeline and should be fixed together. Bugs 9, 10, and extra are independent of each other and of the HP filter.

## Verification Plan

**Unit-level:** After each fix, run the existing comparison test harness (`tests/compare/`) which links the C reference via FFI and compares outputs in-process. The biquad fix (Bug 7) should produce the largest improvement in sample-level match rates for VOIP-mode encodes.

**Specific test vectors:**
- VOIP mode, mono and stereo, at 8/12/16/24/48 kHz sample rates
- Silence input (exercises DTX path -- bugs 9, 10)
- Speech input (exercises HP filter -- bugs 7, 8)
- Varying bitrates to exercise mode selection thresholds (bug extra)

**Regression check:** Non-VOIP modes (AUDIO, RESTRICTED_LOWDELAY) should be unaffected by bugs 7 and 8 since they use `dc_reject` instead of `hp_cutoff`. Bugs 9, 10, and extra apply to all modes that use SILK, so verify SILK-only and hybrid modes as well.

**Bit-exact criterion:** After all five fixes, VOIP-mode encodes should produce byte-identical compressed output compared to the C reference for the same input, configuration, and random seed.
