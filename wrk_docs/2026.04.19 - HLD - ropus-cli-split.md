# HLD: split ropus-cli into four opus-tools-style binaries

Author: Arthur + Claude
Date: 2026-04-19
Status: DELIVERED (commits `37b82a3` → `1ff869b` → `12ef7c2` → `fdbd023`)

See `wrk_journals/2026.04.19 - JRN - ropus-cli-split-implementation.md` for
stage-by-stage rationale, including the two notable supervisor overrides
(devil's-advocate caught the `Decoder::set_gain` primitive vs. re-rolled
scaling; `transcode` subcommand dropped in Stage 2 after a silent
arg-loss bug surfaced).

## Goal

Replace the single `ropus-cli` multi-subcommand binary with four flat-style
binaries that mirror the familiar `opus-tools` muscle memory:

| Tool         | Role                                         |
|--------------|----------------------------------------------|
| `ropusenc`   | encode any symphonia-supported input → Ogg Opus (MP3/FLAC/OGG/AAC handled transparently on the flat path) |
| `ropusdec`   | decode `.opus` → 16-bit PCM WAV               |
| `ropusinfo`  | print stream info for an `.opus` file         |
| `ropusplay`  | play any audio file via the default device    |

Also fix a correctness bug: `OpusHead.output_gain` is currently parsed but
never applied on decode.

## Non-goals

- Closing the opus-tools feature-parity gaps (`--framesize`, `--float`, metadata,
  `--raw`, etc.). Those are follow-up work — see the gap list in the
  conversation log. This HLD is about packaging and the one correctness bug.
- Backwards compatibility. The old `ropus-cli` binary is deleted outright;
  we are in a development sandbox and there are no published consumers to
  migrate.
- Feature flags, optional deps, or any Cargo feature gating. Explicitly out of
  scope — adds complexity for a tradeoff LTO+strip already handles.

## Current state

`ropus-cli/` is a single binary crate with five subcommands (`encode`,
`decode`, `transcode`, `info`, `play`) and well-factored internal modules
(`audio/`, `container/`, `commands/`, `cli.rs`, `ui.rs`, `util.rs`,
`consts.rs`). Round-trip smoke test lives at `ropus-cli/tests/round_trip.rs`.
Everything compiles, works, and has colour/banner UI.

## Dependency strategy (no feature flags)

`ropus-tools-core` declares the full dependency set in its `Cargo.toml`:
`ropus`, `symphonia`, `rubato`, `ogg`, `rodio`, `anyhow`, `colored`. `clap` is
deliberately NOT a core dep — clap lives only in the four binary crates.
`ropus::Application` is re-exported from core so binaries don't need to
depend on `ropus` directly.

Every binary pulls in `ropus-tools-core` and gets the full dep graph. We rely
on **linker + LTO dead-code elimination**: the lib itself compiles the union
of code paths, but the release binary only keeps the transitive closure of
functions each `main` actually calls. Cargo does not selectively compile
parts of a lib — if the lib declares `symphonia` as a dep, `symphonia` gets
compiled whenever any binary that depends on the lib is built.

This accepts that building `ropusdec` in debug mode still compiles symphonia,
rubato, and rodio into core. That's a compile-time cost, not a
runtime/binary-size cost. Given dev machines already have these crates
cached from `ropus-cli`, the marginal pain is near zero and the architectural
simplicity is large.

If binary size turns out to matter more than this design allows, we revisit.
Not before.

## CLI shape

Each binary is **flat-argument style**, matching `opusenc`/`opusdec`/`opusinfo`:

```
ropusenc INPUT [-o OUTPUT] [--bitrate N] [--complexity N]
               [--application {voip,audio,lowdelay}] [--vbr|--cbr]
               [-q|--quiet] [--no-color]
    # Any symphonia-supported input (WAV/MP3/FLAC/OGG/AAC/Vorbis) is
    # accepted directly — no separate "transcode" verb. An earlier draft
    # kept a `transcode` subcommand for muscle memory; it was dropped in
    # Stage 2 after review found the `Option<Subcmd>` + flat-flatten
    # pattern silently discarded top-level flags when the verb was
    # present.

ropusdec INPUT.opus [-o OUTPUT.wav] [-q] [--no-color]

ropusinfo INPUT.opus [-q] [--no-color]

ropusplay INPUT [--volume V] [-q] [--no-color]
```

Global `-q/--quiet` and `--no-color` are handled by the shared
`prelude::run_prelude()` which sniffs raw argv before clap runs (so they work
with `--help`/`--version` too).

## Clap ↔ core boundary

Core exposes plain option structs with no clap derive:

```rust
// ropus-tools-core/src/options.rs
pub struct EncodeOptions {
    pub input: PathBuf,
    pub output: Option<PathBuf>,
    pub bitrate: Option<u32>,
    pub complexity: Option<u8>,
    pub application: ropus::Application,
    pub vbr: bool,
    pub vendor: String,        // written into OpusTags; each binary passes its own name
}
// ... DecodeOptions, InfoOptions, PlayOptions similar
```

The `vendor` field replaces the hardcoded `"ropus-cli"` string in the current
`build_opus_tags` call. Each binary fills it from `env!("CARGO_PKG_NAME")`
so `opusinfo` / `ropusinfo` output shows `ropusenc` (or whichever tool
produced the file).

Each binary's `main.rs` defines its own clap struct (with `#[derive(Parser)]`),
maps it to the options struct, and calls the core function via the shared
`run()` helper:

```rust
// ropusenc/src/main.rs (sketch)
fn main() -> ExitCode {
    let (_quiet, _no_color) = ropus_tools_core::prelude::run_prelude();
    let args = Args::parse();
    let opts = EncodeOptions {
        input: args.input,
        output: args.output,
        bitrate: args.bitrate,
        complexity: args.complexity,
        application: map_application(args.application),
        vbr: !args.cbr,
        vendor: env!("CARGO_PKG_NAME").to_string(),
    };
    ropus_tools_core::prelude::run(ropus_tools_core::commands::encode(opts))
}
```

`prelude::run(Result<()>) -> ExitCode` handles the `anyhow` chain printing
that each binary would otherwise duplicate. This keeps clap out of the
library surface — another entry point (tests, GUI, fb2k plugin) that wants
to call `encode` programmatically never has to construct clap types.

## Banner / build info

Each binary has an identical five-line `build.rs` emitting `BUILD_TIMESTAMP`
and `BUILD_GIT_SHA` as rustc-envs, and a three-line `main.rs` preamble:

```rust
ropus_tools_core::ui::print_banner(
    env!("CARGO_PKG_NAME"),
    env!("CARGO_PKG_VERSION"),
    env!("BUILD_TIMESTAMP"),
    env!("BUILD_GIT_SHA"),
);
```

Accepting a 5-line copy across four crates in exchange for not introducing a
dedicated `ropus-build-info` helper crate. No env vars are used beyond the
two rustc-env values that `build.rs` already emits today.

## `output_gain` bug fix

### Current behaviour

`ropus-cli/src/container/ogg.rs:48` parses `output_gain: i16` as Q7.8 dB per
RFC 7845 §5.1, but the field is marked `#[allow(dead_code)]` and
`commands::decode` never reads it. Files with a non-zero header gain decode
at the wrong level.

### Fix

`ropus::Decoder` already exposes `pub fn set_gain(gain: i32) -> Result<(), i32>`
(`ropus/src/opus/decoder.rs:1493`) that applies the Q8-dB gain inside the
decoder — in fixed-point, **before** the i16 saturating clamp — and is
differential-tested against the C reference. This is exactly the
`opus_decoder_ctl(OPUS_SET_GAIN, ...)` path the canonical `opusdec` uses.

Don't re-implement the math. Just call it. In `commands::decode`, right
after `RopusDecoder::new(...)`:

```rust
if head.output_gain != 0 {
    decoder.set_gain(head.output_gain as i32)
        .map_err(|e| anyhow!("set_gain from OpusHead failed: {e}"))?;
}
```

Four lines. No new helper, no new formula, no new tests duplicating
coverage the `ropus` crate already has. `set_gain` takes an `i32` and
internally validates the range `[-32768, 32767]`, which a Q7.8 `i16` always
satisfies.

Drop the `#[allow(dead_code)]` attribute on the `OpusHead::output_gain`
field.

### Test

A single integration-style test in `commands::decode`'s test module:

1. Synthesize a minimal OpusHead byte slice with `output_gain = -2560`
   (Q7.8 for -10 dB), parse it via the existing `parse_opus_head`, confirm
   the field round-trips correctly.
2. (Optional, time permitting) Decode a short pre-encoded Opus packet
   twice — once with the header gain applied, once with it zero — and
   assert the RMS ratio is ~0.316. If the pre-encoded packet fixture is
   not readily available, skip this and rely on the `ropus::Decoder`
   crate's own `set_gain` tests at `ropus/src/opus/decoder.rs:2012` which
   already verify the scaling math end-to-end.

The reason to avoid re-testing the gain scale factor at the `ropus-cli` layer:
`ropus::Decoder` is the unit under test for the *math*; `commands::decode`
is the unit under test for *"does it call set_gain when the header says
to"*. Don't conflate them.

### Why the obvious "scale the i16 output" approach is wrong

An earlier draft of this HLD proposed scaling the decoded i16 buffer
after the packet loop using `10f64.powf(gain / (256.0 * 20.0))`. Devil's-
advocate review caught three problems:

1. `ropus::Decoder` already exposes the primitive — re-implementing the
   math is code we didn't need to write, and a drift risk against the C
   reference.
2. Post-decode i16 scaling loses precision at low volumes (a -40 dB track
   quantises twice). The decoder's internal path scales the pre-clamp
   samples, preserving dynamic range.
3. Tests built from the same formula as the implementation are
   tautological; they prove the formula is self-consistent, not that it
   matches the RFC. The `ropus::Decoder` unit tests already verify the
   math against libopus.

Use the existing primitive.

## Integration test

Round-trip test lives at `ropus-tools-core/tests/round_trip.rs` and calls
`commands::encode` + `commands::decode` **directly as library functions** —
no subprocess, no binary path lookup. Encode a sine fixture from
`tests/vectors/48k_sine1k_loud.wav` to a temp `.opus`, then decode to a
temp `.wav`, then RMS-SNR check the result against the input.

This is cleaner than the original plan (which wanted a separate
`ropus-tools-tests` workspace member to drive the built binaries):
`CARGO_BIN_EXE_<name>` is only populated for bins within the same package
as the test, not for path dev-deps — so the subprocess approach was
broken. Testing the library directly covers the interesting logic (the
four binaries are each ~30 lines of clap + one call into core) and
needs no extra workspace member.

The existing `ropus-cli/tests/round_trip.rs` is deleted with the rest of
`ropus-cli/`.

### Profile handling

Cargo applies `[profile.*]` sections **only from the workspace root
Cargo.toml** — profile sections in member crates are silently ignored. The
existing `ropus-cli/Cargo.toml` has `lto = true, codegen-units = 1,
strip = true` in `[profile.release]`; those have been dead code since they
were written. When we delete `ropus-cli/` we lose nothing.

New binary crates MUST NOT carry `[profile.*]` sections — we put any desired
tuning in the workspace root.

Workspace currently has:

```toml
[profile.release]
lto = "thin"
codegen-units = 1
panic = "unwind"
```

**Decision landed:** add `strip = "symbols"` to `[profile.release]` only.
`[profile.dev]` is left alone — its default is `strip = "none"`, which
preserves full DWARF and symbol tables for gdb/lldb/panic backtraces during
development. That split is the standard Rust pattern: debug builds
prioritise debugability, release builds prioritise size.

The resulting workspace release profile:

```toml
[profile.release]
lto = "thin"
codegen-units = 1
panic = "unwind"
strip = "symbols"
```

This applies to every workspace binary — the four new `ropus*` tools, the
`harness` (ropus-compare), and the `capi` cdylib. `strip = "symbols"`
preserves the dynamic symbol table required for cdylib exports, so the
C ABI surface of `capi` is unaffected; only internal/static symbols go.

## Implementation order

0. Fix `output_gain` on the current `ropus-cli/` tree with its unit test.
   Commit as its own behaviour-change patch so the fix is easy to find in
   `git log` and trivial to revert independently of the refactor.
1. Scaffold `ropus-tools-core/` with the module tree; move code verbatim from
   `ropus-cli/src/` and convert private `pub(crate)` to `pub` at the module
   boundaries core consumers need. Introduce `options.rs` (including the
   `vendor` field on `EncodeOptions`) and rewrite each `commands::*` function
   to take the options struct. Add `prelude::run()` for `Result<()> -> ExitCode`.
   Port the round-trip test to `tests/round_trip.rs` using in-process library
   calls.
2. Create `ropusenc/`, `ropusdec/`, `ropusinfo/`, `ropusplay/` crates, each
   with clap definitions, `build.rs`, and a thin `main.rs` that dispatches
   to core via `prelude::run()`.
3. Update workspace `Cargo.toml` (members list + `strip = "symbols"` in
   release profile). Delete `ropus-cli/`.
4. `cargo build --workspace` + `cargo test --workspace` clean. Spot-check
   each binary with `--help` and one real invocation (including a round-trip
   through actual CLI invocations, not just the library test).
5. Single commit for the refactor:
   `refactor(cli): split ropus-cli into ropus{enc,dec,info,play}`.

Step 0 lands first, on its own commit. Steps 1-5 land as a second commit.

## Risks and open questions

- **Symphonia/rodio compile time on every `cargo build -p ropusdec`.**
  Mitigation: none at this design's simplicity level; if it hurts in
  practice, revisit with feature flags or crate splitting later. Not before.
- **`ropusplay` with no audio device** (CI, headless). The current code
  already errors cleanly; no change.
- **`cpal` platform deps become workspace-wide.** Today only `ropus-cli`
  pulls `rodio` → `cpal` → (`alsa-sys` on Linux, CoreAudio on macOS). After
  the split, `ropus-tools-core` pulls them, so every workspace member
  transitively needs the platform audio headers to build. On Windows this is
  invisible. On a headless Linux CI without `libasound2-dev`,
  `cargo check --workspace` will fail where `cargo check -p ropus` (or
  `-p harness`) used to succeed. Accepted: keep in mind if/when we stand up
  Linux CI.
- **Duplicated `build.rs` across four binaries.** Accepted as cheaper than a
  helper crate for five lines.

## Out of scope (explicit)

- Opus-tools feature-parity flags (`--framesize`, `--float`, metadata, raw
  PCM I/O, etc.). Separate effort.
- Stdin/stdout piping for any binary.
- Channel mapping family 1/2 (surround, ambisonics).
- Opus-in-WebM / Opus-in-MP4 containers.
- Publishing any of the new crates to crates.io.
