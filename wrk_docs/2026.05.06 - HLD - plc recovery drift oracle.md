# PLC Recovery Drift Oracle HLD

Date: 2026.05.06

Scope: Phase 3.2 of `/home/md/language/ropus/wrk_docs/2026.05.05 - PLN - testing improvement plan.md`.

## Problem Statement

`/home/md/language/ropus/harness/tests/c_ref_differential.rs` currently has two PLC/recovery signals for the deterministic decode24 lost-frame case:

- `test_silk_plc_first_subframes_match_c_reference` is active and protects the first 288 16-bit PLC samples against the C reference exactly.
- `test_24bit_decode_plc_matches_c_reference` is ignored because full-frame decode24 PLC still drifts after roughly sample 288.

That is an honest local comment, but it is a weak release contract. A normal test run hides the post-288-sample region completely. The goal for this slice is not to claim full PLC parity. The goal is to stop hiding the known residual drift window by replacing or supplementing the ignored all-or-nothing full-frame test with an active tiered oracle that makes the exact, bounded-drift, and debt regions explicit.

This slice must not change codec implementation, SILK math, PLC math, or decode24 conversion behavior.

## Stakeholder Value

The main stakeholders are maintainers using the harness to decide whether recovery paths are regressing.

Value delivered:

- The existing exact first-288-sample regression remains a hard gate.
- The post-288-sample full-frame PLC region becomes visible in normal validation.
- Drift is bounded by measured metrics instead of being hidden behind `#[ignore]`.
- The oracle matrix can say "exact prefix plus bounded residual drift" rather than generic PLC coverage.
- Future PLC math work gets a clear target: reduce or eliminate the explicit full-frame exactness debt.

The practical outcome: a green run means "this deterministic PLC lane still has exact prefix parity and no worse residual drift than the documented bound", not "PLC is fully bit-exact".

## Current Evidence

Relevant current comments in `/home/md/language/ropus/harness/tests/c_ref_differential.rs` document:

- deterministic fixture: 48 kHz mono, 20 ms VOIP packet, seed `55`, one successful prime decode, then one lost frame via `data=None` / C `data=NULL, len=0`;
- first exact region: 288 samples;
- previous improvement: mismatching samples reduced from roughly 925/960 to roughly 615/960;
- current residual maximum absolute drift: roughly 17 16-bit LSBs after the known PLC state fixes;
- full decode24 exact parity remains ignored because residual LSB-level rounding drift still exists.

Those numbers are enough to choose initial bounds, but the implementation should still measure the current deterministic output before landing the active test. If measured values differ, do not widen the bound blindly; update this HLD or the implementation journal with the observed reason.

## Considered Options

### Option A: Unignore the full-frame exact decode24 test

Pros:

- Strongest possible oracle.
- No new policy language.

Cons:

- Known to fail today.
- Turns Phase 3.2 into a SILK/PLC math fix, which is explicitly out of scope.
- Encourages risky parity-sensitive changes before the residual drift is measured and tracked.

Decision: reject for this slice.

### Option B: Leave the ignored exact test and add an active bounded-drift test

Pros:

- Keeps full-frame exact parity visible as unpaid debt.
- Adds a green, automation-visible check for the known residual window.
- Requires only harness test/helper work and documentation.
- Small and easy to roll back.

Cons:

- Two tests describe the same fixture, so comments must stay clear.
- A bounded oracle is weaker than exact parity and can be misread if not named carefully.

Decision: accept.

### Option C: Convert the ignored exact test into the bounded-drift test

Pros:

- Avoids duplicate fixture setup.
- Fewer test names.

Cons:

- Loses the executable marker for the intended final full-frame exact oracle.
- Makes it easier to forget that full exact parity is still debt.

Decision: reject unless the implementation finds severe duplication. Prefer leaving the exact test ignored with a refreshed reason and adding a separate active test.

### Option D: Only update documentation and the oracle matrix

Pros:

- Lowest risk.

Cons:

- Normal validation still hides the drift window.
- The plan's Phase 3.2 objective is not met.

Decision: reject.

## Recommended Design

Add one active tiered oracle around the existing deterministic decode24 PLC fixture in `/home/md/language/ropus/harness/tests/c_ref_differential.rs`.

Recommended test name:

```rust
test_24bit_decode_plc_has_bounded_recovery_drift()
```

Keep `test_silk_plc_first_subframes_match_c_reference` unchanged as the exact first-window regression. It is already a clear regression guard for the previously fixed SILK PLC state bugs.

Keep `test_24bit_decode_plc_matches_c_reference` ignored, but refresh its comment so it points to the new active bounded-drift test and states that full-frame exact decode24 parity is still debt.

### Shared Fixture Helper

To avoid subtle fixture drift, extract the common setup used by the ignored decode24 PLC test into a small local helper in the same test file. Keep it private to the test module/file.

Suggested shape:

```rust
struct Decode24PlcDriftReport {
    rust_n: i32,
    c_n: i32,
    n_samples: usize,
    exact_prefix_len: usize,
    first_diff: Option<usize>,
    max_abs_diff_raw_24: i64,
    max_abs_diff_16_lsb_equiv: i64,
    drift_count_after_prefix: usize,
}
```

The helper should return both PCM buffers or the report plus enough context for assertion messages. Do not add a broad abstraction for all decode parity tests. This is a single known-debt fixture.

### Tiered Oracle

Use these tiers for the active test:

| Tier | Sample range | Active assertion | Meaning |
|---|---:|---|---|
| Exact region | `0..288` | Raw decode24 samples match exactly. | Protects the already-fixed PLC state window. |
| Bounded drift region | `288..960` for the current 20 ms mono fixture | Sample count matches; drift metrics stay within documented bounds. | Makes residual post-288 rounding drift visible and bounded. |
| Explicit debt region | Full-frame exact parity over `0..960` | Remains represented by ignored exact test. | Full decode24 PLC exactness is not claimed. |

The active test should assert:

- Rust and C both return `960` samples for the lost frame.
- `n_samples == 960`.
- `exact_prefix_len >= 288`.
- `first_diff.is_none() || first_diff >= 288`.
- `max_abs_diff_16_lsb_equiv <= 17`.
- `drift_count_after_prefix <= 615`.

For the fixed-point `!ENABLE_RES24` build, decode24 samples are 16-bit PCM left-shifted by 8. Compute and report raw i32 absolute drift too, but express the policy threshold in 16-bit LSB-equivalent units because the existing observed data and comments use "17 out of 32767" scale. The equivalent raw decode24 bound is `17 << 8 == 4352`.

If a future build enables native 24-bit residual output, the implementation should revisit this oracle instead of treating the `17 << 8` conversion as universal.

### Reporting

On failure, the assertion message should include:

- fixture parameters: `sr=48000`, `ch=1`, `frame_size=960`, `seed=55`;
- Rust and C sample counts;
- exact prefix length;
- first divergent sample index;
- max raw decode24 drift;
- max 16-bit LSB-equivalent drift;
- drift count after prefix;
- first few mismatching sample pairs after the exact prefix.

This should be normal `panic!`/assertion context, not an operator-only ignored report. The point is to make regressions actionable from a failed `cargo test`.

## Risks And Tradeoffs

- Bounded drift can be misread as parity. Mitigation: use explicit test names, failure text, and oracle matrix language.
- The `615` drift-count bound comes from existing comments, not a fresh measurement in this HLD pass. Mitigation: implementation must measure first and record the actual numbers.
- The deterministic packet is produced by the Rust encoder helper. If encoder behavior changes, the PLC drift fixture can move. Mitigation: failure output must include packet/fixture context; if packet bytes change intentionally, update the measured thresholds deliberately.
- The raw decode24 drift threshold assumes fixed-point decode24 equals 16-bit decode left-shifted by 8. That matches current tests but is not a future native-res24 policy.
- Leaving the ignored exact test means there is still ignored debt. That is intentional for this slice; the new active bounded test prevents the debt from being invisible.

## Rollback Plan

Rollback is simple because the implementation should be test-only plus documentation:

1. Remove the new bounded-drift helper/test from `/home/md/language/ropus/harness/tests/c_ref_differential.rs`.
2. Restore the old ignored-test comment if necessary.
3. Revert the oracle matrix row/class update.

No codec state, fixture assets, or public API changes are involved.

## Key Decisions

- Do not implement or refactor SILK/PLC math in Phase 3.2.
- Keep the active first-288-sample exact regression.
- Keep the full-frame exact decode24 PLC test ignored as explicit debt.
- Add a separate active bounded-drift oracle for the full deterministic lost frame.
- Use initial thresholds from existing observed data: exact prefix at least 288 samples, max residual drift at most 17 16-bit LSB-equivalent units, and drift count after prefix at most 615 samples.
- Measure the deterministic fixture during implementation before treating those thresholds as final.
