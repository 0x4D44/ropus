# HLD: CELT Encoder Rate Control Fixes

## Problem

CELT encode tests diverge from the C reference on any configuration that exercises VBR, constrained VBR, CBR clamping, hybrid mode, or silence frames. The root cause is nine missing or incomplete rate control blocks in `src/celt/encoder.rs`, all in the `celt_encode_with_ec` function. The Rust code computes bitrate targets, entropy coder buffer sizes, and VBR reservoir updates differently from the C reference (`reference/celt/celt_encoder.c`), causing cascading divergence in bit allocation and final encoded output.

## Root Causes

| # | Bug | Impact |
|---|-----|--------|
| 12 | CBR rate clamping missing | CBR mode never shrinks the encoder buffer to match the target bitrate |
| 1 | Constrained VBR early clamping missing | CVBR can exceed its budget — reservoir overflows, later frames diverge |
| 4 | equiv_rate clamping missing | Short frames inflate equiv_rate, changing alloc_trim and band decisions |
| 11 | overlap_max never updated | Silence detection uses only current-frame samples, misses overlap region |
| 5 | packet_size_cap missing in VBR | VBR can attempt >510 kb/s, which the CELT allocator cannot use |
| 6 | Hybrid VBR target path missing | Hybrid always calls compute_vbr() instead of using simplified silk_info path |
| 2 | VBR silence override missing | Silent VBR frames don't freeze drift, causing rate spikes after silence |
| 3 | ec_enc_shrink after VBR missing | Entropy coder retains original buffer size after VBR adjustment |
| extra | Dead effective_bytes update | Cosmetic — discards value with `let _ =` instead of updating variable |

## Fix Plan

### Bug 12: CBR rate clamping (C lines 1910-1921)

**Location:** `encoder.rs` ~line 2224, inside the `else` branch (non-VBR path).

**What C does:** When `st->bitrate != OPUS_BITRATE_MAX` (i.e., not "max bitrate" mode), C computes `tmp = st->bitrate*frame_size`, adds `tell*mode->Fs` if tell>1, then clamps `nbCompressedBytes = IMAX(2, IMIN(nbCompressedBytes, (tmp+4*mode->Fs)/(8*mode->Fs) - !!st->signalling))` and calls `ec_enc_shrink(enc, nbCompressedBytes)`.

**What Rust should do:** In the `else` branch at line 2224, before computing `effective_bytes`, add:
- Compute `tmp = st.bitrate * frame_size` (i32 — watch for overflow on high bitrates).
- If `tell > 1`, add `tell * mode.fs` to tmp.
- If `st.bitrate != OPUS_BITRATE_MAX`, clamp `nb_compressed_bytes` to `max(2, min(nb_compressed_bytes, (tmp + 4*mode.fs) / (8*mode.fs) - (st.signalling != 0) as i32))`.
- Call `enc_ref.shrink(nb_compressed_bytes as u32)`.
- `nb_compressed_bytes` must be `let mut` (it already is at line 2195).

**Risk:** Low. Straightforward arithmetic. The `!!st->signalling` maps to `(st.signalling != 0) as i32`.

### Bug 1: Constrained VBR early clamping (C lines 1935-1960)

**Location:** `encoder.rs` ~line 2227, after `nb_available_bytes` is computed.

**What C does:** When `vbr_rate > 0 && st->constrained_vbr`, computes:
- `vbr_bound = vbr_rate`
- `max_allowed = min(max(tell==1 ? 2 : 0, (vbr_rate + vbr_bound - st->vbr_reservoir) >> (BITRES+3)), nbAvailableBytes)`
- If `max_allowed < nbAvailableBytes`, shrinks the encoder: `nbCompressedBytes = nbFilledBytes + max_allowed`, updates `nbAvailableBytes`, calls `ec_enc_shrink`.

**What Rust should do:** After line 2227, add a block guarded by `vbr_rate > 0 && st.constrained_vbr != 0`. Compute `max_allowed` using the same formula. If it's less than `nb_available_bytes`, update both `nb_compressed_bytes` and `nb_available_bytes` and call `enc_ref.shrink(...)`.

**Prerequisite:** `nb_available_bytes` must become `let mut` (currently `let` at line 2227).

**Risk:** Low. The vbr_bound == vbr_rate simplification means `vbr_rate + vbr_bound == 2 * vbr_rate`.

**Ordering note:** This block must execute BEFORE `total_bits` is set (C line 1961: `total_bits = nbCompressedBytes*8`). Currently the Rust sets `total_bits` at line 2228 BEFORE this block would go. The `total_bits` assignment must move to after this block to match C ordering.

### Bug 4: equiv_rate clamping (C lines 1926-1927)

**Location:** `encoder.rs` ~line 2232, after equiv_rate computation.

**What C does:** `if (st->bitrate != OPUS_BITRATE_MAX) equiv_rate = IMIN(equiv_rate, st->bitrate - (40*C+20)*((400>>LM) - 50))`.

**What Rust should do:** After line 2232, add:
```
if st.bitrate != OPUS_BITRATE_MAX {
    equiv_rate = equiv_rate.min(st.bitrate - (40 * c + 20) * ((400 >> lm) - 50));
}
```
Requires `equiv_rate` to become `let mut`.

**Risk:** Low. Direct arithmetic translation. The subtracted term is already computed identically in the existing equiv_rate formula.

---

### Bug 11: overlap_max never updated (C lines 1969-1973)

**Location:** `encoder.rs` ~line 2236, silence detection block.

**What C does:**
1. `sample_max = MAX32(st->overlap_max, celt_maxabs_res(pcm, CC*(N-overlap)/st->upsample))` — incorporates previous frame's overlap tail.
2. `st->overlap_max = celt_maxabs_res(pcm + CC*(N-overlap)/st->upsample, CC*overlap/st->upsample)` — stores current frame's overlap tail for next frame.
3. `sample_max = MAX32(sample_max, st->overlap_max)` — incorporates current overlap tail too.

**What Rust does:** Only computes `celt_maxabs16` over `pcm[..(n-overlap)/upsample*cc]`. Never reads or writes `st.overlap_max`.

**What Rust should do:** Replace the current `sample_max` computation with:
1. `let mut sample_max = st.overlap_max.max(celt_maxabs16(&pcm[..(cc*(n-overlap)/st.upsample) as usize]))` — note: the C scans over interleaved samples including both channels, the slice range must be `cc * (n - overlap) / upsample`.
2. `st.overlap_max = celt_maxabs16(&pcm[(cc*(n-overlap)/st.upsample) as usize .. (cc*n/st.upsample) as usize])` — the overlap tail region.
3. `sample_max = sample_max.max(st.overlap_max)`

**Risk:** Medium. The PCM slice indices must match C exactly. The current code casts via `.iter().map(|&x| x as i32).collect::<Vec<_>>()` which is wasteful — the fix should also remove that allocation if `celt_maxabs16` can work on the PCM type directly. Verify `celt_maxabs16` vs `celt_maxabs_res` — these may differ for fixed-point vs floating-point. For fixed-point, `celt_maxabs_res` operates on `opus_val32` (i32). The Rust PCM input is `&[i16]`, so the cast to i32 may still be needed.

### Bug 5: packet_size_cap in VBR (C line 2445)

**Location:** `encoder.rs` ~line 2766, at the start of the `if vbr_rate > 0` block.

**What C does:** `nbCompressedBytes = IMIN(nbCompressedBytes, packet_size_cap>>(3-LM))`. This caps the buffer to ~510 kb/s equivalent, because the CELT allocator cannot use more than that.

**What Rust should do:** Add this cap as the first line inside the `if vbr_rate > 0` block. The `packet_size_cap` variable does not exist in Rust yet — it must be introduced. In C, it's set much earlier at line ~1836: `packet_size_cap = nbCompressedBytes + bitrate_to_bits(64000, mode->Fs, frame_size) / 8`. The Rust needs this variable computed at the equivalent location (before the VBR block) and used here.

**Risk:** Medium. Requires introducing a new variable earlier in the function and threading it through. The 64000 constant is the per-channel overhead cap.

---

### Bug 6: Hybrid VBR target path (C lines 2456-2475)

**Location:** `encoder.rs` ~line 2780, where `compute_vbr()` is called unconditionally.

**What C does:** Branches on `!hybrid`:
- Non-hybrid: calls `compute_vbr()` (current Rust behavior).
- Hybrid: sets `target = base_target` directly, then applies three adjustments based on `st->silk_info`:
  1. If `silk_info.offset < 100`: `target += 12 << BITRES >> (3-LM)` (tonal boost).
  2. If `silk_info.offset > 100`: `target -= 18 << BITRES >> (3-LM)` (noisy reduction).
  3. `target += MULT16_16_Q14(tf_estimate - QCONST16(.25, 14), 50 << BITRES)` (transient boost).
  4. If `tf_estimate > QCONST16(.7, 14)`: `target = max(target, 50 << BITRES)` (transient floor).

**What Rust should do:** Wrap the existing `compute_vbr()` call in `if !hybrid { ... } else { ... }` and implement the hybrid branch with the four adjustments above.

**Risk:** Low. The `SILKInfo` struct already has the `offset` and `signal_type` fields (used elsewhere for `allow_weak_transients`). The `MULT16_16_Q14` and `QCONST16` macros have Rust equivalents.

---

### Bug 2: VBR silence override (C lines 2494-2499)

**Location:** `encoder.rs` ~line 2806, after `delta` is computed and before VBR state updates.

**What C does:** After computing `delta = target - vbr_rate`, C does:
1. `target = nbAvailableBytes << (BITRES+3)` — reassign target from the rounded nb_avail.
2. If `silence`: `nbAvailableBytes = 2; target = 2*8<<BITRES; delta = 0;` — freeze all drift adjustments.
3. Then uses the (possibly overridden) `target` and `delta` for reservoir and drift updates.

**What Rust does:** Computes `delta` at line 2806, then immediately uses `(nb_avail << (BITRES + 3))` in the reservoir update (line 2810) — equivalent to `target` reassignment. But it never overrides for silence, and `delta` is never zeroed.

**What Rust should do:** After line 2806:
1. Reassign target: `target = nb_avail << (BITRES + 3);` (cosmetic but aligns with C).
2. Add silence override: if `silence { nb_avail = 2; target = 2 * 8 << BITRES; delta = 0; }`.
3. Use `target` (not `nb_avail << ...`) in the reservoir update at line 2810.
4. `delta` must become `let mut`.

**Risk:** Low. Direct translation. This is critical for correctness on silence frames — without it, the reservoir and drift accumulate errors that persist into non-silent frames.

---

### Bug 3: ec_enc_shrink after VBR (C line 2532)

**Location:** `encoder.rs` ~line 2839, after `nb_compressed_bytes = nb_compressed_bytes.min(nb_avail)`.

**What C does:** `ec_enc_shrink(enc, nbCompressedBytes)` — tells the entropy coder the buffer is now smaller.

**What Rust does:** Nothing. The entropy coder still believes it has the full original buffer.

**What Rust should do:** Add `enc_ref.shrink(nb_compressed_bytes as u32);` after line 2838.

**Risk:** Low. The `shrink` method exists and is already used in the silence path (line 2258). Without this call, the entropy coder may write beyond the intended buffer boundary, producing incorrect output or a corrupted bitstream.

---

### Bug extra: Dead effective_bytes update

**Location:** `encoder.rs` line 2839.

**What it does:** `let _ = nb_compressed_bytes - nb_filled_bytes;` discards the computed value.

**What Rust should do:** The variable `effective_bytes` (defined at line 2220) is never read after the VBR block, so this line is genuinely dead code. In C, `effectiveBytes` is also not read after the VBR block (its last use is in `offsets[0]` at C line 2352, which precedes the VBR block). Remove the dead line entirely.

**Risk:** None. Removing dead code.

## Execution Order

The fixes have dependencies — some change variables that later fixes read, and some restructure code blocks that later fixes insert into. Implement in this order:

### Phase 1: Variable preparation
1. **Bug 12 (CBR clamping)** — modifies `nb_compressed_bytes` in the else branch. Independent.
2. **Bug 4 (equiv_rate clamping)** — adds one line after equiv_rate. Independent.
3. **Bug 11 (overlap_max)** — rewrites silence detection. Independent.

These three are fully independent and can be done in any order or in parallel.

### Phase 2: VBR early path
4. **Bug 1 (CVBR early clamping)** — inserts a block after `nb_available_bytes`, requires `nb_available_bytes` to become `let mut`, and requires `total_bits` assignment to move after the new block.

This depends on Phase 1 being complete because it touches adjacent lines.

### Phase 3: VBR late path
5. **Bug 5 (packet_size_cap)** — first line inside VBR block. Requires introducing `packet_size_cap` variable earlier.
6. **Bug 6 (hybrid VBR target)** — wraps `compute_vbr()` call in conditional. Independent of Bug 5.
7. **Bug 2 (silence override)** — inserts block after `delta` computation. Must come before Bug 3.
8. **Bug 3 (ec_enc_shrink)** — one line after VBR clamping.
9. **Bug extra (dead code)** — remove the `let _ = ...` line. Do last since it's at the same location as Bug 3.

Within Phase 3, Bugs 5 and 6 are independent of each other. Bug 2 must precede Bug 3 (the shrink call goes after the silence-adjusted nb_avail). Bug extra is trivially last.

### Suggested commit sequence
1. Commit: Phase 1 (Bugs 12, 4, 11) — "Fix CBR clamping, equiv_rate cap, and overlap_max tracking"
2. Commit: Phase 2 (Bug 1) — "Add constrained VBR early clamping"
3. Commit: Phase 3 (Bugs 5, 6, 2, 3, extra) — "Fix VBR target path, silence override, and encoder shrink"

## Verification

### Expected test changes

After all fixes, the following sweep categories should improve:

| Test category | Affected bugs | Expected change |
|---|---|---|
| CBR encode (any bitrate) | 12 | Byte-exact match where CBR frames were oversized |
| CVBR encode | 1, 2, 3 | Reservoir tracking matches C; drift convergence is correct |
| VBR encode with silence | 2, 11 | Silent frames produce 2-byte packets; no rate spike after silence |
| VBR short frames (2.5ms, 5ms) | 4, 5 | equiv_rate and packet size are bounded; allocator decisions match |
| Hybrid encode (any rate) | 6 | Target uses silk_info offsets instead of full compute_vbr |
| All encode tests | 3 | Entropy coder buffer size is consistent; no corrupted tails |

### Specific validation steps

1. Run the existing encode sweep. Record which configs FAIL before the fix.
2. Apply Phase 1 and re-run. CBR configs should flip. Short-frame VBR configs may flip.
3. Apply Phase 2 and re-run. CVBR configs that were diverging after several frames should improve.
4. Apply Phase 3 and re-run. Remaining VBR, hybrid, and silence configs should flip.
5. Verify no PASS->FAIL regressions at any phase.
6. Specifically test silence frames: encode a zero-PCM buffer in VBR mode, verify packet size is 2 bytes and `vbr_reservoir` matches C after 10+ frames.
