# ropus — HLD: IETF vectors, projection test, and regression_test()

**Date:** 2026-04-19
**Status:** Proposed
**Author:** Claude (with Arthur)

## Goal

Close the three conformance gaps identified after the official-test-suite work landed:

1. **`run_vectors.sh`** — run the IETF RFC 6716 / RFC 8251 bitstream vectors against our decoder via `opus_demo` + `opus_compare`. Canonical spec-level oracle.
2. **`test_opus_projection.c`** — compile the ambisonics test against our codec. The matrix data + projection encode/decode are already in-tree; only the capi surface and internal matrix API are missing.
3. **`regression_test()`** un-stub — replace `regression_test_stub.c` with the real `opus_encode_regressions.c` so `test_opus_encode` exercises historical crash repros (surround, MS CBR, SILK gains, range-coder edges, projection overflow).

Explicit outcome: **`cargo test -p conformance` builds and passes every reference test we have decided is in scope, end-to-end, on Linux and Windows, with zero modifications to the upstream `.c` sources.**

## Oracles

Same framing as the previous conformance HLD (`2026.04.18 - HLD - official-test-suite.md`). The new surface adds:

| Oracle | What it encodes | Why it matters |
|---|---|---|
| IETF vectors (`run_vectors.sh`) | RFC 6716 / RFC 8251 bitstreams, pre-decoded PCM, quality thresholds | Spec-level. Any deviation here is a bitstream-compat bug. |
| `test_opus_projection.c` | Ambisonics matrix math (against reference C impl, compiled in) + end-to-end projection round-trip through our codec with tolerance | End-to-end covers our Rust codec; the unit-test half is a sanity check on the reference impl. |
| `regression_test()` | 10+ named crash reproducers for historically-fixed bugs | Surfaces "did we port the fix?" regressions the differential harness won't find (most are API-behaviour assertions, not byte-exactness). |

## Non-goals

- `test_opus_dred.c` — requires DRED decoder + RDOVAE inference (~1700 LOC C, model weights). Separate HLD. DRED is Opus 1.5+, not RFC 6716.
- `test_opus_custom.c` — custom-mode streams (`#ifdef CUSTOM_MODES`) are non-interoperable, effectively deprecated, and outside the public Opus surface. Explicit drop. Document in README.
- `ENABLE_QEXT` / `ENABLE_DRED` branches inside `opus_encode_regressions.c`. We compile with both macros undefined, as we already do for the rest of the suite. 3 of 13 repros compile out cleanly.
- Porting either `opus_demo.c` or `opus_compare.c` to Rust. We compile the upstream `.c` files verbatim and call their `main`s via the same `-Dmain=<stem>_main` trick the rest of the conformance suite uses.
- Re-validating end-to-end projection roundtrip. The `projection_roundtrip` harness binary (commit `1f43890`) already does this. We add the *internal-API* test on top.
- Shipping IETF vectors in-tree. ~71 MB of binary fixtures belong out-of-repo.

## Piece A — `run_vectors.sh` via `opus_demo` + `opus_compare`

### Shape

`run_vectors.sh` calls two binaries per vector:
- `opus_demo -d <rate> <channels> -ignore_extensions <in.bit> <out.pcm>` — decode only.
- `opus_compare [-s] -r <rate> <ref.pcm> <out.pcm>` — quality metric; exit 0 on pass.

So we need:
- **`opus_demo`** — the whole file (1294 lines, encode arm dead on `-d`). Compiles verbatim.
- **`opus_compare`** — pure PCM comparator, no libopus dependency (382 lines).

### Build

Follow the existing conformance pattern exactly. In `tests/conformance/build.rs`:

```rust
// Extra "binary-like" C sources: compiled as static libs with main renamed,
// same as the reference tests. Called from Rust via their renamed entry points.
let bins: &[(&str, &str, &[&str])] = &[
    ("src/opus_demo.c",    "opus_demo",    &["src/lossgen_stub.c"]),
    ("src/opus_compare.c", "opus_compare", &[]),
];
// source paths relative to reference/, not reference/tests/
// Both lib compiles must add: .define("FIXED_POINT", "1")
```

### Dependencies that need explicit handling

Four points the HLD v1 hand-waved that actually matter:

- **`opus_multistream_decode24`.** `opus_demo.c:1145/1151/1154` call it *unconditionally* on every decoded packet (via the `#define opus_decode24 opus_multistream_decode24` alias at `opus_demo.c:310`). It takes `opus_res*` output (i32 in fixed-point). This is **not** currently in our capi — we expose `opus_multistream_decode` / `_float` only. Add `opus_multistream_decode24` as a capi wrapper: ~80 LOC, same pattern as the existing `_float` variant. Scope-incrementally: also add plain `opus_decode24` since the test vector file paths will naturally exercise it.

- **Lossgen link stub.** `LOSSGEN_SOURCES = dnn/lossgen.c dnn/lossgen_data.c`. `lossgen.c` transitively pulls in `nnet.h`, `parse_lpcnet_weights.c`, and the full LPCNet dependency tree — too much. But `opus_demo` only calls `lossgen_init` and `sample_loss` at runtime if the user passes `-sim_loss`. `run_vectors.sh` doesn't. So at link time we provide a 15-line `tests/conformance/src/lossgen_stub.c` with no-op implementations of both functions and the `LossGenState` struct typedef. Documented as "run_vectors.sh does not exercise lossgen; compile out the DNN dep".

- **`FIXED_POINT` define.** Our capi is a fixed-point port, so `opus_res` is `opus_int32`. The reference headers typedef `opus_res` to `float` unless `FIXED_POINT` is defined. If we compile `opus_demo.c` / `opus_compare.c` / `test_opus_projection.c` without `-DFIXED_POINT`, the `opus_res*` signatures between the C side and our capi side silently disagree and we get memory corruption (not a link error — C lets the mismatch through). Every `cc::Build` for conformance C sources must `.define("FIXED_POINT", "1")`. Applies equally to the existing encode/decode/api/extensions/padding test compiles — they work today because none of them call `opus_res*` functions, but the hygienic fix is to pass it uniformly.

- **`opus_alloc` / `opus_free` / `os_support.h`.** `reference/src/mapping_matrix.c` (Piece B) and a handful of opus_demo internals reference these. Real `os_support.h` is in `reference/celt/`. Two options: (a) add `reference/celt/` to the include path; (b) vendor a 10-line `os_support.h` stub that maps `opus_alloc(n)` → `malloc(n)`, `opus_free(p)` → `free(p)`. Prefer (a) — cheaper, avoids stub drift — unless it drags in transitive headers we don't want, in which case fall back to (b).

### Fetching vectors

IETF test vectors are ~71 MB, hosted at `https://opus-codec.org/static/testvectors/opus_testvectors-rfc8251.tar.gz` (verified 2026-04-19; old `/docs/` path 301-redirects to this). Not checked in.

- `tools/fetch_ietf_vectors.{sh,ps1}` — downloads, verifies SHA-256, extracts to `tests/vectors/ietf/` (gitignored).
- The conformance test checks for the directory. If missing: `panic!("IETF vectors not present. Run tools/fetch_ietf_vectors.sh first.");` — same pattern Windows devs already expect for `reference/`.
- No environment variable; path is hard-coded relative to `tests/conformance/`.

### Rust test driver

`tests/conformance/tests/ietf_vectors.rs`:

```rust
// Pseudo — one #[test] per vector × {mono, stereo}.
#[test] fn testvector01_mono()  { run_one("testvector01", 1, 48000); }
// ... 12 vectors × 2 = 24 tests ...

fn run_one(stem: &str, channels: i32, rate: i32) {
    let bit = vectors_dir().join(format!("{stem}.bit"));
    let ref_pcm = vectors_dir().join(format!("{stem}.dec"));
    let out = tmp_dir().join(format!("{stem}_{channels}.out"));

    // opus_demo -d <rate> <ch> -ignore_extensions <bit> <out>
    assert_eq!(run_opus_demo(&["-d", &rate.to_string(), &channels.to_string(),
                                "-ignore_extensions", bit.as_str(), out.as_str()]), 0);

    // opus_compare [-s] -r <rate> <ref> <out>  — accept either .dec or .decm (like the shell)
    let r1 = run_opus_compare(channels, rate, &ref_pcm, &out);
    let r2 = run_opus_compare(channels, rate, &ref_with_suffix("m"), &out);
    assert!(r1 == 0 || r2 == 0, "{stem} ch{channels}: neither .dec nor .decm matched");
}
```

`run_opus_demo` / `run_opus_compare` build argv arrays and call the renamed `extern "C" fn opus_demo_main(c_int, *const *const c_char) -> c_int` (and its compare sibling). Same plumbing as every other conformance test — no subprocess layer.

Temp files live under the system temp dir (one per `stem`/channels combination, scoped by PID + atomic counter). Cleanup is best-effort via an inline `std::fs::remove_dir_all` at the end of a passing test; on a panicking test the directory is intentionally left behind for debugging.

### Test-serialisation (load-bearing)

`opus_demo.c` has file-scope globals (`decoder_ctl`, `ignore_extensions`, `iseed`/`Rw`/`Rz`, FILE* handles mid-function), and calling its `main()` 24 times in one process requires strict serialisation. Add `# --test-threads=1` to the conformance test invocation in CI and note it in README. Same requirement already applies to the existing 5 tests; this just extends it.

## Piece B — `test_opus_projection.c`

### What's missing

Commit `1f43890` ported the 10 ambisonics mixing matrices and proved end-to-end projection roundtrip is byte-exact vs C (via `harness/src/bin/projection_roundtrip.rs`). The gaps for `test_opus_projection.c` are:

1. The test pokes the **internal** `mapping_matrix_*` API directly (unit tests of matrix math against hand-specified expected output).
2. The test calls the **`opus_projection_*`** public API for end-to-end encode/decode: `opus_projection_ambisonics_encoder_create`, `opus_projection_encode{,_float}`, `opus_projection_encoder_ctl` (for `OPUS_PROJECTION_GET_DEMIXING_MATRIX{,_SIZE,_GAIN}`), mirror decoder surface. Not currently in our capi.

### Key ABI incompatibility

`reference/src/mapping_matrix.c` uses a **flexible-array layout**: `mapping_matrix_get_size(rows, cols)` returns `align(sizeof(MappingMatrix)) + rows*cols*sizeof(opus_int16)`; callers allocate one buffer, cast it to `MappingMatrix*`, and `mapping_matrix_init` stores the data immediately after the struct header. `mapping_matrix_get_data(m)` returns a pointer past the aligned header.

Our Rust `MappingMatrix { rows, cols, gain, data: Vec<i16> }` has a 24-byte `Vec` header inline (ptr/len/cap). Different layout by construction; Rust doesn't offer `#[repr(C)]` flex arrays, so there's no clean way to expose our Rust matrix functions under the same C ABI the test expects.

### Design (simplified)

Rather than fighting the ABI, **compile `reference/src/mapping_matrix.c` verbatim** into the projection test's static lib. The C file is self-contained — it only depends on `opus_types.h`, `opus_projection.h`, `float_cast.h`, `mathops.h`, and `os_support.h`'s `opus_alloc`/`opus_free`. No libopus dependency. This gives the test a spec-compliant matrix implementation for its unit tests.

Our Rust matrix math (in `ropus/src/opus/multistream.rs`) stays Rust-native and is already validated end-to-end by `projection_roundtrip`. The conformance test exercises the reference C matrix implementation for its unit tests; the end-to-end projection encode/decode path still flows through our Rust codec via `opus_projection_ambisonics_*`.

Steps:

1. **Expose `opus_projection_*` public API in capi.** `capi/src/projection.rs` — thin wrappers over our existing `OpusMSEncoder`/`OpusMSDecoder` with `channel_mapping=3`. ~150 LOC, same handle-indirection pattern as multistream. Plus the three `OPUS_PROJECTION_GET_DEMIXING_MATRIX*` CTLs in `ctl_shim.c`.
2. **Add `reference/src/mapping_matrix.c` as an extra** to the `test_opus_projection` row in `build.rs`. Add `reference/celt/` to the include path (or vendor a 10-line `os_support.h` stub if the former drags in too much).
3. **Vendor the projection headers** needed by the test: `opus_projection.h` (verbatim from `reference/include/`), `mapping_matrix.h` (verbatim from `reference/src/`), and a `mathops.h` minimal subset if needed. The `#include "../src/mapping_matrix.h"` line in the test resolves via the `reference/src/` include path, same pattern as `../src/opus_private.h` uses today (force-include our stub if the real one drags in too much).
4. **Add `tests/conformance/tests/projection.rs`** with the one-`#[test]`-wrapping-`main` pattern. No Rust codec port needed.

Net scope: ~150 LOC capi surface + build.rs/include plumbing. Zero new Rust codec code. Down from ~320 LOC in the previous draft.

### Handle indirection

Projection encoder/decoder state is `OpusMSEncoder` / `OpusMSDecoder` under the hood. The capi already exposes those with handle-indirection shims; projection reuses them unchanged. No new aliasing work.

## Piece C — `regression_test()` un-stub

### What's in scope

With `ENABLE_QEXT` and `ENABLE_DRED` undefined (matching the rest of the suite), `regression_test()` runs 10 of 13 repros:

| Function | Feature exercised | Prerequisites |
|---|---|---|
| `celt_ec_internal_error` | Surround mono 16kHz VOIP edge | Piece B (surround encoder) |
| `mscbr_encode_fail10` | MS CBR encode | (existing) |
| `mscbr_encode_fail` | MS CBR encode | (existing) |
| `surround_analysis_uninit` | Surround + float analysis | Piece B |
| `ec_enc_shrink_assert` | Range coder shrink | (existing) |
| `ec_enc_shrink_assert2` | Range coder shrink #2 | (existing) |
| `silk_gain_assert` | SILK gain quantisation | (existing) |
| `analysis_overflow` | Float-API analysis | (existing, float API already wired) |
| `projection_overflow` | Projection encoder | Piece B |
| `projection_overflow2`/`3` | Projection encoder (float paths) | Piece B |

All call surfaces we already expose except the projection ones, which Piece B adds. So Piece C is a dependency-ordered child of Piece B.

### Design

1. Remove `tests/conformance/src/regression_test_stub.c`.
2. Change the `test_opus_encode` row in `build.rs` to extras: `&["../../../reference/tests/opus_encode_regressions.c"]`. This compiles the upstream regressions file into the encode test's static lib, supplying the real `regression_test()` definition that `test_opus_encode.c`'s `main()` calls.
3. Force-include our stubbed `opus_private.h` for the regressions file too, same `/FI` / `-include` mechanism as the other `#include "../src/opus_private.h"` clients.
4. If any unconditional repro exercises a path not yet in capi (e.g. a CTL we haven't wired), Piece C grows by that much plumbing. I expect zero new plumbing based on the function bodies I read — all use APIs `test_opus_api.c` and `test_opus_encode.c` already drive — but we'll discover the truth by compiling.

No new tests; this reuses the existing `encode` test binary.

## Phasing

Dependency graph: **A ⫫ B → C**. A and B are independent and can go in either order; C blocks on B.

1. **Piece A — IETF vectors.** Add `opus_multistream_decode24`/`opus_decode24` to capi. Wire `opus_demo` + `opus_compare` into `tests/conformance/build.rs` with `FIXED_POINT` define and lossgen stub. Write fetch scripts. Write `ietf_vectors.rs`. Run against local vectors. **~3 days** (was 2 before `opus_multistream_decode24` was scoped in).
2. **Piece B — projection test.** Expose `opus_projection_*` capi surface. Compile `reference/src/mapping_matrix.c` verbatim into the test lib. Vendor projection headers. Wire `test_opus_projection.c`. **~2 days** (was 3 before the simplification).
3. **Piece C — regression_test un-stub.** Swap the stub for `opus_encode_regressions.c` in `build.rs`. **~1 day if clean; +1-2 days per latent ropus bug the repros surface.** The first conformance pass surfaced 6 bugs; expect at least 1-2 here on the surround/projection-overflow paths. Realistic range: **1-5 days**.
4. **README update.** Already shipped alongside this HLD. Refresh once implementation is green. **~0.5 day.**

Total: **~7-11 working days** (optimistic / pessimistic).

## Success criteria

- `cargo test -p conformance` runs 8 test binaries (5 existing + 3 new: `ietf_vectors`, `projection`, and the re-enabled `regression_test()` within `encode`) to exit-code-0 on both platforms, with zero modifications to any upstream `.c` source.
- `tools/fetch_ietf_vectors.{sh,ps1}` brings the repo from a fresh clone to "IETF-ready" with one command.
- README's testing-scope section is accurate: what we run, what we don't, why.
- No new cargo feature flags. No new environment variables. No new dependencies outside the `cc` crate we already use.

## Known risks

- **`opus_demo.c` internal header expectations.** It `#include`s from `reference/celt/` and `reference/silk/` directly. Most of those headers we already vendor; a few may need new stubs. If `celt.h` or `silk/API.h` is pulled in and drags internal layout assumptions, we vendor a minimal stub the same way we did `opus_private.h`. Mitigation: first compile attempt tells us which.
- **`opus_compare` floating-point determinism across platforms.** Its metric uses `float` math; tiny FP drift between MSVC and gcc could trip the quality threshold. Mitigation: the metric has built-in tolerance; if not enough, we're comparing *our* decode to *the reference's* pre-decoded PCM (which is fixed), so the only FP drift is within `opus_compare.c` itself — small.
- **`regression_test()` hits a latent ropus bug.** Probable. That's the point — it's the exact class of bug the official suite exists to surface. Each becomes a journal entry + focused fix + regression test in `harness/tests/c_ref_differential.rs`, mirroring the pattern from the first round (`encoder-final-range-bug.md`). Estimate reflected in Piece C's 1-5 day range.
- **IETF vectors licensing.** Vectors are IETF Contribution (RFC 8251 supplementary materials). No known redistribution restrictions; we're linking to the canonical `opus-codec.org` archive rather than redistributing, so there is no licensing action item beyond citing the source.

## Open questions

1. **Bundling vs. fetch-on-demand.** Default here is fetch-on-demand: keeps clone size small (71 MB stays out of repo), sidesteps the license-provenance question (we link to the canonical archive rather than shadowing it), and costs one setup step for fresh checkouts. Revisit if CI cold-start latency becomes noticeable.
2. **`os_support.h` strategy** for Piece B — add `reference/celt/` to the include path vs. vendor a 10-line stub. Decide during Piece B's first compile.
