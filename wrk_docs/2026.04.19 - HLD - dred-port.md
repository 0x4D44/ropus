# HLD — DRED port (DRED-only, OSCE deferred)

**Date.** 2026-04-19
**Author.** Claude (supervisor)
**Parent.** `wrk_docs/2026.04.19 - HLD - stages-6-8-relaxed-oracles.md`
(this doc narrows Stage 8 to DRED; OSCE is explicitly out of scope)
**Status.** Draft — awaiting Arthur approval before Stage 8.0 starts.

## Mission

Port DRED (Deep REDundancy, Opus 1.5's in-band neural packet-loss
resilience feature) from the xiph/opus C reference to Rust, wire it
into `OpusEncoder` for payload emission and into a new
`OpusDREDDecoder` for payload parsing and feature reconstruction.
Leave the audio-reconstruction last mile (features → PCM via FARGAN)
as a seam for the Stage 7 team.

## What DRED actually is (for context)

On the encoder side, DRED embeds a heavily compressed snapshot of the
previous ~1 s of audio alongside the current frame, using an RDOVAE
(rate-distortion-optimised variational autoencoder). When packets
drop, a surviving later packet contains DRED data from which the
decoder can reconstruct the lost audio. Extension ID 126 in the Opus
extension stream. Bitrate overhead ~10–25 %. Both encoder and decoder
are bitstream-visible.

## What we need to add

1. **Laplace-p0 range coder primitives** (~50 lines).
   `ec_laplace_encode_p0` and `ec_laplace_decode_p0` exist in
   `reference/celt/laplace.c` but are not used by CELT's existing
   `quant_bands` port, so they were skipped. Add them to
   `ropus/src/celt/range_coder.rs` with bit-exact round-trip tests.

2. **`dred_stats`** — static `u8` quantization tables (~2 KB), direct
   translation of `dred_rdovae_stats_data.c`. Byte-for-byte diff test
   against the C arrays (same pattern as Stage 6.2's `mlp_data.rs`).

3. **`dred.rs`** — single Rust module holding:
   - `DREDEnc` state struct (RDOVAE encoder state + LPCNet feature
     state + input ring buffer + resample filter memory).
   - `OpusDRED` payload struct (decoded state + latents + metadata).
   - `compute_quantizer` helper (6-line function from `dred_coding.c`).
   - `dred_rdovae_encode_dframe` — RDOVAE encoder forward pass
     (calls existing `dnn::core` primitives).
   - `dred_rdovae_decode_qframe` — RDOVAE decoder forward pass (ditto).
   - `dred_compute_latents` — resamples to 16 kHz, runs LPCNet
     feature extraction, pushes through RDOVAE.
   - `dred_encode_silk_frame` — range-codes the latents + state into
     the DRED extension payload.
   - `dred_ec_decode` — reverse of the above for the decoder side.

4. **`opus/dred.rs`** — `OpusDREDDecoder` public API:
   `create`/`init`/`destroy`, `parse`, `process`. Mirrors the
   corresponding C API in `opus_decoder.c:1369-1600`. The final
   `decode` entry point is a stub until Stage 7 wires FARGAN.

5. **Encoder integration** in `opus/encoder.rs`:
   - Add `dred_encoder: DREDEnc` field.
   - Call `dred_compute_latents` per frame in the encode path.
   - Emit DRED extension bytes on the first frame of each packet when
     `dred_duration > 0` and weights are loaded.

## Weights strategy

Follow Stage 7 HLD's precedent: **embed the weight blob in the crate
via `include_bytes!`, parse at `OpusEncoder::new()` time with
`dnn::core::parse_weights`.** Single consistent pattern across all
DNN modules; no Cargo feature flag, no runtime file loading, no build
script generating giant static arrays.

- **Artefact.** One binary file, `ropus/src/dnn/dred_weights.bin`,
  checked into the repo. Produced by the xiph `opus_get_wts.sh`
  pipeline from the pinned upstream checkpoint (see Format pinning).
- **Size target.** The two `.c` weight files together are ~11 MB of
  source but encode ~3–4 MB of binary (f32/int8 interleaved). Acceptable.
- **Failure mode.** If the blob's format version doesn't match what
  `parse_weights` expects, `OpusEncoder::new()` returns an error —
  normal error path, no silent fallbacks.
- **Crate size impact.** The published crate grows by the blob size.
  Acceptable for a codec library; if someone later wants a no-DRED
  build, they can prove the case and we add a feature flag then.

Note: `dred_rdovae_stats_data.c` is handled differently — it's a
couple of KB of `u8` quantisation tables referenced by name
(`dred_state_quant_scales_q8` etc.), not by `parse_weights`. It
becomes a compiled-in `const` array in `dred_stats.rs`.

## Upstream format pinning

The DRED bitstream format is flagged experimental
(`DRED_EXPERIMENTAL_VERSION 12`, `DRED_EXTENSION_ID 126`, both in
`dred_config.h`). Upstream xiph may still revise it.

**Mitigation.** At Stage 8.0 we record the upstream commit sha we
port against, in this HLD:

```
Pinned upstream:  reference/ @ <sha>           ← filled in at 8.0
Date observed:    2026-04-19
DRED version:     12 (experimental)
Extension ID:     126
```

No tooling to enforce — just a documented reference point. If xiph
revises DRED before we ship, we re-port the deltas; the RDOVAE
forward pass and entropy coding rarely change even when the payload
framing does.

## Tier split (per HLD v2 relaxed-oracle framework)

Attempted tier 1 everywhere; fall back to tier 2 per component.

| Component                          | Tier 1 target                        | Tier 2 fallback                      |
|------------------------------------|--------------------------------------|--------------------------------------|
| `ec_laplace_*_p0` + `dred_stats`   | byte-exact round-trip                | (no fallback — integer code)         |
| `dred_rdovae_encode_dframe`        | bit-exact against C on synthetic in  | SNR ≥ 60 dB on latents & state       |
| `dred_rdovae_decode_qframe`        | bit-exact against C on synthetic in  | SNR ≥ 60 dB on feature output        |
| `dred_compute_latents` (full path) | bit-exact against C on WAV input     | SNR ≥ 60 dB on latents               |
| `dred_encode_silk_frame` payload   | byte-exact against C on WAV input    | (integer — no realistic fallback)    |
| `dred_ec_decode` → latents, state  | byte-exact input ⇒ bit-exact floats  | (integer — no realistic fallback)    |

**Rationale for mixed tiers.** The entropy coder and quantisation
tables are integer maths; tier 1 is straightforward. The RDOVAE
forward pass is float, so like analysis.c in Stage 6 it may or may
not achieve tier 1. The encoded payload (`dred_encode_silk_frame`)
can still be byte-exact if the RDOVAE output fed into quantisation
is bit-exact — if RDOVAE drifts, the payload will too, and we fall
back to tier 2 on the RDOVAE layer and accept the downstream byte
drift (same logic Stage 6 ended under).

**PCM-level check.** Once Stage 7 FARGAN is available, run the
Stage 6.5 `pcm_drift` pattern with an engineered packet-loss drop:
C-reconstructed PCM vs Rust-reconstructed PCM vs original input.
Accept if Rust's SNR-vs-input is within 0.1 dB of C's, matching the
Stage 6.5 bar.

## Stage 7 coordination

Two concrete touchpoints; everything else is independent.

1. **`SET_DNN_BLOB` dispatch.** Stage 7 owns CTL routing. For DRED we
   expose `DREDEnc::load_model(&[u8]) -> Result<(), i32>` from
   `dnn/dred.rs` (mirrors C `dred_encoder_load_model`). Stage 7's CTL
   handler calls that. We publish the signature early and they wire.

2. **FARGAN reconstruction seam.** Stage 7's FARGAN wiring fills in
   the `opus_decoder_dred_decode*` body. We ship the public API with
   a stub that returns `OPUS_UNIMPLEMENTED`. When FARGAN is ready,
   one joint PR removes the stub.

**Harness `build.rs` conflict.** Stage 7 is already adding DNN C
sources (`nnet.c`, `nnet_default.c`, `fargan.c`). We add DRED C
sources on top. Coordinate via a quick agreement on who edits the
file last; worst case, small merge.

## What we're explicitly NOT building

- Fallback encoder paths when the blob is missing. If the blob isn't
  loaded, encoder DRED functions return `OPUS_UNIMPLEMENTED`.
  (Matches C.)
- Telemetry or debug dumps for RDOVAE internals.
- A DRED-specific harness CLI binary (we reuse `ropus-compare` with
  a `--dred-duration=N` flag, added inside Stage 8.8's wiring work).
- Documentation of the DRED format — Opus 1.5 spec is the reference;
  we don't duplicate it.

## Risks

1. **RDOVAE float drift.** Same tier-1 risk as analysis.c in Stage 6.
   Likely outcome: module-level bit-exact, integrated drift. Mitigated
   by the tiered acceptance above.

2. **Upstream format churn.** Experimental version 12. If xiph
   re-numbers before we ship, we re-port the small deltas and regen
   the blob.

3. **Stage 7 slip blocks end-to-end decode validation.** We can still
   validate encode (tier 1 byte-exact) and decode-parse (tier 1
   round-trip through `dred_ec_decode`). End-to-end sample-exact
   decode test is deferred until FARGAN lands.

4. **Blob generation pipeline.** Upstream's weight blob comes from a
   Python pipeline (`opus_get_wts.sh`). We need to run it once, not
   replicate it. If it's broken or undocumented, fall back to
   extracting the weights directly from the C `const` arrays via a
   one-off Rust/Python tool.

5. **Crate size.** ~3–4 MB of binary weights pushed into the crate.
   Acceptable but worth naming — ropus is currently ~1 MB source; the
   DRED blob will dominate.
