# HLD — `compute_burg_cepstrum` f32/f64 parity fix

Date: 2026-05-07
Author: Claude (HLD agent)
Status: Draft, awaiting supervisor review

## 1. Problem statement

`compute_burg_cepstrum` in `ropus/src/dnn/lpcnet.rs` runs two arithmetic
operations in `f32` where the C reference (`reference/dnn/freq.c`) runs them
in `double`. Each costs ~1 ULP per intermediate and propagates through
`forward_transform` -> `compute_band_energy_inverse` -> `dct` into the 36-band
classical-PLC feature vector. Severity: cosmetic-to-low for runtime PLC
output (sub-percept), but it breaks bit-parity claims and is the same shape as
the six precision deviations fixed earlier today. Blast radius: PLC analysis
path only — `burg_cepstral_analysis` is not on the DRED encode path, so
tier-1/tier-2 DRED gates are unaffected.

## 2. Investigation findings

### 2.1 Rust source (current)

`/home/md/language/ropus/ropus/src/dnn/lpcnet.rs:986-1018`:

```
998:        xbuf[i + 1] = -burg_lpc[i] * 0.995f32.powi(i as i32 + 1);
...
1011:        ly[i] = (1e-2 + eburg[i]).log10();
```

Both intermediates are pure `f32`.

### 2.2 C reference

`/home/md/language/ropus/reference/dnn/freq.c:151-181`:

```
169:  for (i=0;i<order;i++) x[i+1] = -burg_lpc[i]*pow(.995, i+1);
174:    Ly[i] = log10(1e-2+Eburg[i]);
```

`pow` and `log10` resolve to the `double` overloads from `<math.h>` (no `f`
suffix). `.995` and `1e-2` are double literals. The `-burg_lpc[i] * <double>`
multiply happens in `double`; narrowing occurs only on the store to
`x[i+1]`/`Ly[i]` (both `float`). Identical pattern to the
`lpc_from_cepstrum` fix in commit b1bfa6f.

### 2.3 Caller graph

`grep -rn "compute_burg_cepstrum\|burg_cepstral_analysis" ropus/ reference/`
shows:

- Rust callers of `burg_cepstral_analysis`: only `lpcnet.rs:2434` inside the
  PLC analysis loop (`self.plc_net` / `plc_features` / `compute_plc_pred`
  region). Not invoked by the DRED encoder.
- C callers of `burg_cepstral_analysis`: `lpcnet_plc.c:168` (runtime PLC) and
  `dump_data.c:465` (training/data-dump tool, not in the runtime build).
- `compute_burg_cepstrum` itself is `static` in C; only reachable via

Confirms the diagnostic agent's "PLC-only" claim. No DRED tier-1 surface.

### 2.4 Other f32/f64 patterns in the same function

Audited line-by-line against `freq.c:151-181`:

| Op | C semantics | Rust semantics | Match? |
|----|-------------|----------------|--------|
| `burg_in[i] = pcm[i+1] - PREEMPHASIS*pcm[i]` | f32 (PREEMPHASIS is `0.85f`) | f32 (`PREEMPHASIS_COEF: f32`) | yes |
| `g /= len - 2*(order-1)` | int->float, f32 divide | f32 cast + f32 divide | yes |
| `x[i+1] = -burg_lpc[i]*pow(.995, i+1)` | **double** pow + multiply | f32 powi | **NO (headline)** |
| `Eburg[i] *= .45*g*(1.f/(WS*WS*WS))` | f32 (explicit `1.f`, `(float)WINDOW_SIZE`) | f32 | yes |
| `Ly[i] = log10(1e-2+Eburg[i])` | **double** add + double log10, narrow on store | f32 add + f32 log10 | **NO (second deviation)** |
| `MAX16(...)` chain | f32 | f32 | yes |
| `dct(...)`, `cepstrum[0] -= 4.0` | f32 / int-literal | f32 / `4.0` literal | yes |

So there are **two** deviations to bundle, not one. Both follow the same
"libm double overload promoted via the call site" pattern as the prior six
fixes.

## 3. Proposed fix

Single bundle, two sites in `compute_burg_cepstrum`. Surgical, no signature
or storage changes.

### Fix A — `pow(.995, i+1)` (headline)

Before (`lpcnet.rs:998`):

```rust
xbuf[i + 1] = -burg_lpc[i] * 0.995f32.powi(i as i32 + 1);
```

After:

```rust
// Match reference/dnn/freq.c:169 — pow() is the double overload, multiply
// stays in double until the store narrows to f32.
xbuf[i + 1] = (-(burg_lpc[i] as f64) * 0.995_f64.powi(i as i32 + 1)) as f32;
```

Rationale: `0.995_f64.powi(...)` is exact-equivalent to libm `pow(.995, n)`
for small integer exponents and is what commit b1bfa6f used for
`lpc_from_cepstrum`. Keeping the multiply in f64 mirrors C's implicit
promotion at the call site.

### Fix B — `log10(1e-2 + Eburg[i])`

Before (`lpcnet.rs:1011`):

```rust
ly[i] = (1e-2 + eburg[i]).log10();
```

After:

```rust
// Match reference/dnn/freq.c:174 — 1e-2 is a double literal and log10()
// is the double overload; the f32 result narrows only on store to Ly[i].
ly[i] = (1e-2_f64 + eburg[i] as f64).log10() as f32;
```

Rationale: identical to Fix A's argument — preserves the C semantics that the
argument of `log10` is computed in double via promotion of `1e-2` (double
literal) before being passed to `log10` (double overload).

No other lines change. No callers change. No public API change.

## 4. Verification strategy

The function sits behind a public C symbol (`burg_cepstral_analysis` is in
`reference/dnn/freq.h:56` — non-static), so a differential test is
straightforward and is the right oracle here. Inspection-only is rejected:
the prior six fixes show this pattern is easy to reintroduce, and we already
have the FFI infrastructure.

### Recommended: new differential unit test

Add a new test file
`harness-deep-plc/tests/burg_cepstral_analysis_diff.rs` (or
`harness-classical-plc/tests/...` if a classical-PLC harness exists; check
during implementation, otherwise extend `harness-deep-plc` since it already
links the C reference and exposes FFI thunks).

Shape, mirroring `dred_lpcnet_feature_drift.rs`:

1. Generate a deterministic `FRAME_SIZE` (=160) `f32` PCM buffer — reuse the
   existing 440 Hz fixture or a synthetic prbs/sine. Same buffer for both
   sides.
2. Call C `burg_cepstral_analysis(ceps_c, x)` via a new
   `ropus_test_burg_cepstral_analysis(x: *const f32, ceps: *mut f32)` thunk
   added to `harness-deep-plc/c` (one-line wrapper, links freq.c).
3. Call Rust `ropus::dnn::lpcnet::burg_cepstral_analysis(&mut ceps_r, &x)`.
4. Assert `ceps_c == ceps_r` element-wise as `f32` bit-pattern equality
   across all `2 * NB_BANDS` (= 36) outputs.

Locked-bound style is overkill here: post-fix this should be **bit-exact**
(both deviations are the only f32/f64 mismatches in the function per §2.4),
matching the cepstral-features-0..17-bit-exact result observed in commit
b1bfa6f. If a residual arises, surface it as an open question rather than
locking a bound.

The test serves as a regression gate: any future change to
`compute_burg_cepstrum` that reintroduces f32 arithmetic on these two sites
will fail the test on the lock host.

### Existing tests

- `test_burg_cepstral_analysis_*` in `lpcnet.rs:2931+` are sanity checks
  (finite output, identical-halves -> small delta), not parity gates. Keep
  unchanged; expected to continue passing.
- `dred_lpcnet_feature_drift.rs` does **not** exercise this path; not
  expected to change. Run as a no-regression check.
- Workspace `cargo test` and `cargo clippy --workspace --all-targets`.

## 5. Risk analysis

- **Tier-1 surface**: none. `burg_cepstral_analysis` is PLC-only. DRED
  encode tests do not touch it.
- **Behavioural change for non-PLC callers**: there are none. C has one
  runtime caller (`lpcnet_plc.c`) and one tooling caller (`dump_data.c`);
  Rust has one (`lpcnet.rs:2434`).
- **Perf**: two extra f64 ops per band per half-frame (= 2 \* 18 = 36 f64
  ops per analysis call) plus an f64 powi loop bounded by `LPC_ORDER` = 16.
  PLC analysis runs at most once per 10-20 ms frame on the loss path.
  Negligible.
- **Numerical**: post-fix outputs are expected bit-exact-against-C for these
  two sites. Any residual would indicate a third deviation we missed and
  would be surfaced by the new differential test.
- **Unsafe / FFI**: the new thunk is a one-line C wrapper, same shape as the
  existing `ropus_test_*` exports.

## 6. Out of scope

- Other f32/f64 deviations elsewhere in the LPCNet feature pipeline. The
  audit in §2.4 is scoped to `compute_burg_cepstrum`. The remaining
  features-18..21 drift tracked by `dred_lpcnet_feature_drift.rs` is a
  separate, already-tracked surface.
- Conversion of the Rust function to take/return `f64` end-to-end. C uses
  f32 storage; we match it.
- Refactoring `burg_cepstral_analysis` or its caller in `lpcnet.rs:2434`.
- Adding `compute_burg_cepstrum` itself as a public symbol — `static` in C,
  module-private in Rust; we test via the public wrapper.
