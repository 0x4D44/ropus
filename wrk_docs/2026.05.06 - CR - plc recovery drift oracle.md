## Threshold Assessment

The active threshold is intentionally tight to the measured current lane, not a broad tolerance band. The validation run measured:

- `exact_prefix_len=289`
- `max_abs_diff_raw_24=4352`
- `max_abs_diff_16_lsb_equiv=17`
- `drift_count_after_prefix=615`

This makes the oracle actionable: any additional post-prefix drift count or larger max drift fails immediately. That is stricter than a comfort margin, but it matches the signed HLD and the stated goal of preventing hidden drift regression while leaving full exact parity as debt.

## Residual Risks

- The bounded oracle is for one deterministic 48 kHz mono, 20 ms VOIP prime-plus-lost-frame fixture. It does not generalize to broader PLC/FEC parameters, which the matrix states at `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md:32`.
- The raw-to-16-bit-equivalent conversion assumes the current fixed-point `!ENABLE_RES24` decode24 behavior. If native residual 24-bit output is enabled later, this oracle should be revisited rather than mechanically widened.
- The active bounds are exactly at the current measured maximum/count. That is good for regression sensitivity, but it means intentional fixture or encoder-packet changes must update the thresholds deliberately with new evidence.
