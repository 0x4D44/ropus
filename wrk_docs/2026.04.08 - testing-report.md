# Testing Report: mdopus Codec

Date: 2026-04-08

## Scope

This report audits the current testing posture of the `mdopus` codec port in `C:\language\mdopus`.

Questions answered:

- How strong is the current testing overall?
- What is covered well?
- Where are the important gaps?
- What is code coverage like?
- What exists for mutation, fuzz, stress, and performance testing?

This was investigated with parallel agents plus direct local validation.

## Method

Evidence came from:

- Source inspection across `C:\language\mdopus\src`, `C:\language\mdopus\tests`, `C:\language\mdopus\tools`, and `C:\language\mdopus\README.md`
- `cargo test`
- `cargo llvm-cov --summary-only`
- Parallel-agent coverage run using `cargo +nightly llvm-cov --branch --summary-only`
- `python tools/integrate.py status`
- Local harness spot checks:
  - `cargo run --bin mdopus-compare -- bench tests/vectors/48k_sine1k_loud.wav --iters 5`
  - `cargo run --bin mdopus-compare -- longsoak --duration 3 --sample-rate 48000`
  - `cargo run --bin mdopus-compare -- quality tests/vectors/48k_sine1k_loud.wav`
  - `cargo run --bin mdopus-compare -- rngtest`

## Executive Assessment

The codec is tested better than a typical in-progress port, but the testing program is uneven.

The strong part is regression checking for deterministic codec behavior:

- `cargo test` is broad and fast.
- The project has a real C-vs-Rust comparison harness in `C:\language\mdopus\tests\harness\main.rs`.
- The integration corpus in `C:\language\mdopus\tools\integrate.py` is meaningful and the current saved run reports 280/280 passing cases.
- Coverage is high enough to be credible, not cosmetic.

The weak part is the non-functional lane:

- Fuzzing exists as scaffolding, not as a clearly operationalized discipline.
- Mutation testing does not exist.
- Stress and performance checks exist as commands, but not as baseline-driven regression gates.
- The hardest modules still have the weakest coverage, especially by branch.

Bottom line: the repo has a strong regression backbone, but it does not yet have a mature verification program around the trickiest state-machine and robustness risks.

## Coverage caveats

- The stable `cargo llvm-cov --branch --summary-only` run hit a Windows `llvm-profdata` merge failure after test execution. This looks like a tooling/file-lock issue, not a codec failure.
- The branch numbers above came from a successful parallel-agent run using nightly.
- Fuzz targets are not part of the normal coverage totals.
- Manual harness commands like `bench` and `longsoak` are also not part of the normal coverage gate unless explicitly wrapped in tests.

## What Is Strong

### 1. The project tests behavior, not just helpers

The presence of `C:\language\mdopus\src\coverage_tests.rs`, the harness smoke tests, and the comparison CLI means the suite is not limited to tiny white-box unit tests.

It exercises:

- actual encode/decode paths
- public API behavior
- PLC and FEC behavior
- multistream handling
- packet parsing and repacketization
- format conversion paths

That is the right shape for a codec.

### 2. The C reference is used as an oracle

This is the biggest strategic strength.

`C:\language\mdopus\tests\harness\main.rs` compares Rust against the upstream C behavior for bytes, PCM, packet shapes, final ranges, and related outputs. For a parity-sensitive port, that is much stronger than ordinary assertions alone.

### 3. The regression matrix is deterministic and broad

`C:\language\mdopus\tools\integrate.py` gives the project a repeatable matrix across:

- multiple sample rates
- mono and stereo
- silence, sine, and noise
- four bitrates
- encode and roundtrip modes

The current saved result of 280/280 passing gives real confidence for the covered matrix.

### 4. Test runtime is fast enough to be practical

`cargo test` completes quickly. That matters because slow test suites stop being used.

### 5. There is at least some robustness and non-functional tooling

The repo is not empty-handed here:

- fuzz crate exists
- longsoak exists
- bench exists
- quality exists
- rngtest exists

That is a strong starting point even if the discipline is incomplete.

## Where The Gaps Are

### 1. Branch-heavy core logic is still under-tested

The weak branch numbers in `opus/encoder.rs`, `opus/repacketizer.rs`, `silk/encoder.rs`, `silk/decoder.rs`, and `opus/decoder.rs` show the biggest remaining risk.

This is where codec bugs usually live:

- mode transitions
- packet edge cases
- FEC/PLC interactions
- multistream layout handling
- bitrate/frame-size/application combinations

### 2. Fuzzing is present but not operationalized

`C:\language\mdopus\tests\fuzz\Cargo.toml` declares five fuzz targets:

- `fuzz_decode`
- `fuzz_encode`
- `fuzz_roundtrip`
- `fuzz_repacketizer`
- `fuzz_packet_parse`

That is good.

What is missing:

- no checked-in seed corpus
- no checked-in crash regression corpus
- no visible CI or nightly automation
- no evidence that fuzz findings are fed back into ordinary regression tests

The targets are also fairly narrow today:

- they lean heavily toward "must not panic" robustness
- they are centered mostly on 48 kHz mono paths
- they do not appear to do broad differential checking against the C reference

That is useful, but weaker than invariant-rich, differential fuzzing across more configurations.

### 3. Mutation testing does not exist

I found no sign of:

- `cargo-mutants`
- mutation scripts
- mutation CI
- mutation thresholds or reports

That is a real gap. With this much unit coverage, mutation testing would be worthwhile now.

### 4. Stress testing exists, but only as a tool

`cmd_longsoak` in `C:\language\mdopus\tests\harness\main.rs` is a good idea, and a 3-second local run passed:

- PCM matched
- final-range values matched

But this is not yet a serious stress program:

- no long-duration soak evidence
- no memory-growth checks
- no repeated multi-hour or overnight runs
- no visible automation
- the bundled `test-all` path only runs a 5-second soak, which is appropriate for smoke coverage but not for endurance testing

### 5. Performance testing exists, but not as a regression gate

The benchmark harness works and produced useful local signal:

- C encode: 3.870 ms/iter
- Rust encode: 16.955 ms/iter
- Rust encode is 4.38x slower
- C decode: 1.551 ms/iter
- Rust decode: 5.679 ms/iter
- Rust decode is 3.66x slower

That is valuable.

But this is still not a mature performance-testing setup:

- single ad hoc sample
- no baseline storage
- no thresholds
- no trend tracking
- no CI regression gate

### 6. Quality checking is informational, not a hard gate

`cmd_quality` reports SNR across bitrates, and on `48k_sine1k_loud.wav` the local run produced:

- 16 kbps: 4.6 dB
- 32 kbps: 36.1 dB
- 64 kbps: 41.0 dB
- 128 kbps: 47.3 dB
- 256 kbps: 47.7 dB

That is useful for sanity checking.

But `test-all` explicitly treats quality as informational, not failing, which means it is not a true regression gate today.

### 7. No visible CI workflow

I found no `.github` workflow directory in the repo. If automation exists elsewhere, it is not visible here.

That means the repo currently does not show an enforced pipeline for:

- coverage
- fuzzing
- soak
- performance
- warning-free builds

### 8. Warning noise remains

Current builds still emit warnings in untouched code, especially around:

- `C:\language\mdopus\src\opus\multistream.rs`
- `C:\language\mdopus\tests\harness\main.rs`

This is not the biggest quality issue, but it lowers signal quality and works against the repo's stated preference for warning-free builds.

## Mutation Testing

Current status: absent.

Assessment:

- This is a gap, not a crisis.
- The repo is finally tested enough that mutation testing would be useful.
- It should not be the first thing fixed, but it is now worth piloting once branch and non-functional gaps are tightened.

## Fuzz Testing

Current status: present but immature.

Assessment:

- Good foundation: five libFuzzer targets exist.
- Weak operational maturity: no visible seed corpus, no crash corpus, no automation, no reporting trail.

Net judgment: better than nothing by a large margin, but not yet strong enough to count as a robust fuzz program.

## Stress Testing

Current status: basic tooling exists.

Assessment:

- Positive: `longsoak` is real and works.
- Gap: there is no evidence of sustained, automated, resource-aware stress testing.

Net judgment: promising tool, weak process.

## Performance Testing

Current status: measurement exists, regression control does not.

Assessment:

- Positive: the harness can compare C vs Rust performance directly.
- Negative: the results are not yet institutionalized into targets or alerts.

Net judgment: useful benchmarking capability, but not yet a performance test program.

## Overall Rating

If I reduce this to a blunt judgment:

- Unit and deterministic regression testing: strong
- Differential testing against C: strong
- Coverage depth: good overall, uneven in the hardest files
- Fuzzing: moderate foundation, weak process
- Stress testing: basic
- Performance regression testing: basic
- Mutation testing: absent

In plain English: the repo is already well beyond "barely tested", but it is not yet at the level where I would trust the hardest codec behavior without continued strengthening of branch-heavy paths and non-functional automation.

## Most Important Takeaway

The project's biggest asset is not its test count. It is the combination of:

- a C-reference oracle
- deterministic corpus comparisons
- broad module tests
- decent coverage

The project's biggest weakness is not missing unit tests. It is that the most failure-prone lanes are still the least institutionalized:

- branch-heavy top-level control flow
- fuzzing discipline
- stress discipline
- performance regression discipline
- mutation testing
