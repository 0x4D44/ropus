# HLD - Fix OpusEncoder::get_final_range divergence on CELT‚ÜíSILK redundancy

**Date:** 2026-04-18
**Module:** `ropus/src/opus/encoder.rs`
**Surfaced by:** xiph/opus `test_opus_encode.c:501` (via capi FFI shim); reproduced by phase-4 debug agent's standalone differential test.

## 1. Symptom

Encoder differential test: `set_force_mode(MODE_SILK_ONLY)`, bitrate 6809 bps, 1920-sample frames at 48 kHz, bandwidths cycling NB‚ÜíMB‚ÜíWB‚ÜíNB‚ÜíMB‚ÜíWB‚Üí‚Ä¶

Iteration results (three values compared per frame: ropus encoder `get_final_range`, ropus decoder `get_final_range`, C-ref decoder `OPUS_GET_FINAL_RANGE`):

| Iter | BW | Enc         | Dec (ropus) | Dec (C-ref) | Match |
|------|----|-------------|-------------|-------------|-------|
| 0    | NB | X‚ÇÄ          | X‚ÇÄ          | X‚ÇÄ          | yes   |
| 1    | MB | X‚ÇÅ          | X‚ÇÅ          | X‚ÇÅ          | yes   |
| 2    | WB | `0x0c4c9290`| `0x43f05790`| `0x43f05790`| **no**|
| 3-14 | ‚Ä¶  | ‚Ä¶           | ‚Ä¶           | ‚Ä¶           | yes   |

Only iter 2 diverges. XOR delta between encoder and decoder reports = `0x4fbcc500`.

## 2. Reproducer

(As above ‚Äî phase-4 agent wrote a standalone differential test in `harness/tests/` demonstrating this.) The bitstream ITSELF is valid: two independent decoders agree with each other. Only the encoder's `final_range` bookkeeping is wrong.

## 3. Root cause

**File:** `c:\language\mdopus\ropus\src\opus\encoder.rs:2392-2409` (the "CELT‚ÜíSILK redundancy frame" block).

Compared to C reference `reference/src/opus_encoder.c:2427-2442`, ropus is missing the call that captures the CELT encoder's final range into the `redundant_rng` variable after the redundancy frame is encoded but before `OPUS_RESET_STATE` wipes it.

### C reference (correct):

```c
/* 5 ms redundant frame for CELT->SILK */
if (redundancy && celt_to_silk)
{
    int err;
    celt_encoder_ctl(celt_enc, CELT_SET_START_BAND(0));
    celt_encoder_ctl(celt_enc, OPUS_SET_VBR(0));
    celt_encoder_ctl(celt_enc, OPUS_SET_BITRATE(OPUS_BITRATE_MAX));
    err = celt_encode_with_ec(celt_enc, pcm_buf, st->Fs/200,
                              data+nb_compr_bytes, redundancy_bytes, NULL);
    if (err < 0) { ... return OPUS_INTERNAL_ERROR; }
    celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng));   // <-- LINE 2440
    celt_encoder_ctl(celt_enc, OPUS_RESET_STATE);
}
```

### ropus (buggy):

```rust
// --- CELT‚ÜíSILK redundancy frame ---
if redundancy && celt_to_silk {
    if let Some(ref mut celt) = self.celt_enc {
        celt.ctl(CeltEncoderCtl::SetStartBand(0));
        celt.ctl(CeltEncoderCtl::SetVbr(0));
        celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX));
        redundancy_frame = vec![0u8; redundancy_bytes as usize];
        celt_encode_with_ec(
            celt, &pcm_buf, self.fs / 200, &mut redundancy_frame,
            redundancy_bytes, None,
        );
        // *** MISSING: redundant_rng = celt.rng; ***
        celt.ctl(CeltEncoderCtl::ResetState);       // <-- wipes celt.rng to 0
    }
}
```

The Rust code calls `ResetState` without first capturing `celt.rng`. Since `CeltEncoder::reset()` sets `self.rng = 0` (see `ropus/src/celt/encoder.rs:3198`), the redundancy-frame rng is lost.

Later, at line 2564:

```rust
range_final ^= redundant_rng;   // XORs with 0 instead of the real rng
```

The encoder's `range_final` therefore ends up equal to the SILK-finalized range (unchanged), but the decoder ‚Äî which DOES capture the corresponding rng from its CELT decoder during redundancy decode (`ropus/src/opus/decoder.rs:851`; mirrors `reference/src/opus_decoder.c:589`) ‚Äî XORs in the real value. Hence the divergence `0x4fbcc500` observed in the reproducer IS the CELT encoder's rng after encoding the 5ms redundancy frame.

### Why only first WB activation?

`redundancy && celt_to_silk` is true only when `silk_bw_switch` is set at frame entry (ropus encoder line 1901). `silk_bw_switch` is set (line 2241) in the PREVIOUS frame only when SILK signals `opus_can_switch` ‚Äî which happens on an UPWARD SILK-internal bandwidth increase (e.g., 12 kHz ‚Üí 16 kHz). In the NB‚ÜíMB‚ÜíWB cycle, SILK's switch-readiness policy only fires on the 12 kHz‚Üí16 kHz step (i.e., the MB‚ÜíWB boundary), placing redundancy on iteration 2 and no others. The bug is present every time the branch fires; it is not "first WB" specific per se ‚Äî it just happens to fire there in this particular test pattern.

## 4. Proposed fix

Add one line after `celt_encode_with_ec` and before `ResetState`, mirroring C line 2440:

```rust
if redundancy && celt_to_silk {
    if let Some(ref mut celt) = self.celt_enc {
        celt.ctl(CeltEncoderCtl::SetStartBand(0));
        celt.ctl(CeltEncoderCtl::SetVbr(0));
        celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX));
        redundancy_frame = vec![0u8; redundancy_bytes as usize];
        celt_encode_with_ec(
            celt, &pcm_buf, self.fs / 200, &mut redundancy_frame,
            redundancy_bytes, None,
        );
        redundant_rng = celt.rng;                   // *** NEW LINE ***
        celt.ctl(CeltEncoderCtl::ResetState);
    }
}
```

That is the entire fix. No other paths or helpers need to change.

## 5. Correctness argument

1. **Mirrors reference semantics exactly.** C line 2440 (`celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng))`) reads the CELT encoder's `rng` field between the redundancy encode and the state reset. Our fix does precisely the same Rust-idiomatically (`redundant_rng = celt.rng`), which is how the SILK‚ÜíCELT branch already does it (line 2556 pattern).

2. **Decoder already mirrors it.** `ropus/src/opus/decoder.rs:851` captures the decoder-side `redundant_rng` in exactly the same place (between CELT decode of the redundancy frame and setting start band). Encoder ‚Üî decoder symmetry is restored by the fix.

3. **Same-frame initialization preserved.** `redundant_rng` is declared `let mut redundant_rng: u32 = 0;` at line 1993 ‚Äî fresh each call. Writing to it only in the path that needs it means non-redundancy frames still XOR with 0, preserving their behavior.

4. **Does not perturb the bitstream.** The fix only writes to internal state used for the `get_final_range` query. The bytes emitted on the wire are unchanged. All existing tests that check encoded bytes will continue to pass. Tests that check decoded audio will continue to pass.

5. **Does not affect any other codepath.** The only branch touched is `redundancy && celt_to_silk`. In all of:
   - CELT-only frames (`redundancy` = false, by C line 1944 / ropus line 1907)
   - SILK-only with no bw-switch (`redundancy` = false)
   - SILK‚ÜíCELT switch (`celt_to_silk = false`)
   the new line is outside the active conditional and cannot execute.

6. **Parity with the SILK‚ÜíCELT branch.** That branch (encoder.rs:2556: `redundant_rng = celt.rng;`) already has this pattern correctly. The bug is purely an omission in the CELT‚ÜíSILK branch.

## 6. Alternative considered & rejected

**Alternative:** Move `enc.done()` / `range_final = enc.get_rng()` after the redundancy encode. ‚Äî Does not address the missing capture; still XORs with 0.

**Alternative:** Skip the XOR entirely when `celt_to_silk`. ‚Äî Would diverge from C reference and still mismatch both decoders (both C-ref and ropus decoder).

**Alternative:** Capture rng into `range_final` directly (skip the `redundant_rng` temporary). ‚Äî Works functionally but muddles the intent; the XOR pattern maps 1:1 to C ref and is clearer.

## 7b. Bug #2: `encode_multiframe` overwrites `range_final`

After fixing Bug #1, the conformance `encode` test progresses much further (completing all 3 LP and all 4 Hybrid sub-tests at several SEEDs) but fails at the first `j=7` test (CELT_ONLY, 60ms frames at ~512k-1M bps, which triggers the multi-frame encode path).

### Root cause

`ropus/src/opus/encoder.rs:1840-1846`, at the end of `encode_multiframe`:

```rust
self.range_final = if let Some(ref celt) = self.celt_enc {
    celt.rng
} else {
    0
};
```

This UNCONDITIONALLY overwrites `self.range_final` with `celt.rng` after the multi-frame loop completes.

The inner loop (lines 1769-1814) calls `encode_frame_native` once per sub-frame. Each such call correctly sets `self.range_final` to the sub-frame's final entropy-coder range ‚Äî including any XOR with `redundant_rng` when redundancy fires on that sub-frame (as it does on the LAST sub-frame when `to_celt=true`, per ropus line 1774-1776 mirroring C line 1782-1783).

After the last sub-frame's `encode_frame_native` returns, `self.range_final` holds the correct "last frame's rng, XOR'd with any redundancy rng" ‚Äî which is exactly what the decoder will compute when it decodes the packet's final sub-frame. The C reference does NOT touch `rangeFinal` again after the multi-frame loop (`reference/src/opus_encoder.c:1831-1838` ‚Äî just repacketize and return). The ropus code diverges by overwriting with the raw `celt.rng`, clobbering any redundancy XOR.

### Concrete failing scenario (test_opus_encode.c `j=7`)

- Outer state: `prev_mode == MODE_HYBRID` (from the previous `j=6` Hybrid test), caller forces `MODE_CELT_ONLY`, `frame_size = 2880` (60 ms).
- Mode-transition logic (`encode_float` line 1482-1494) sets `redundancy=true, celt_to_silk=false, to_celt=true, mode = prev_mode = MODE_HYBRID`. Enters `encode_multiframe` with 3 sub-frames of 20 ms each.
- Sub-frame i=2 (last): `frame_redundancy = true`, `frame_to_celt = true` ‚Üí inside `encode_frame_native`, the SILK‚ÜíCELT redundancy branch fires and the final `range_final ^= redundant_rng` (lines 2556-2573) produces the correct value. `self.range_final` is set correctly.
- Back in `encode_multiframe`: line 1842 clobbers `self.range_final` with `celt.rng`, which at this point is the POST-redundancy CELT rng (i.e., just `redundant_rng`), not the XOR.
- Result: encoder reports `redundant_rng` but the decoder (mirroring the bitstream) reports `main-rng XOR redundant_rng`. Mismatch. `test_opus_encode.c:501` asserts `enc_final_range == dec_final_range` and aborts.

### Fix for Bug #2

Delete lines 1840-1846. The correct `self.range_final` is already set by the last sub-frame's `encode_frame_native`. Matches C reference (no analogous overwrite).

```rust
// DELETE THIS BLOCK:
self.range_final = if let Some(ref celt) = self.celt_enc {
    celt.rng
} else {
    0
};
```

### Correctness argument for Bug #2 fix

1. **C reference doesn't overwrite.** The C reference's multi-frame loop (`reference/src/opus_encoder.c:1769-1838`) simply returns after repacketizing. `st->rangeFinal` holds whatever the last `opus_encode_frame_native` set.
2. **Every sub-frame correctly sets `self.range_final`.** The inner `encode_frame_native` (line 2573) assigns `self.range_final = range_final;` where `range_final` has already been XOR-mixed with `redundant_rng` if any redundancy path fired. Sub-frames that don't trigger redundancy still set `self.range_final = celt.rng` via the same path (line 2564 XORs with 0). So removing the overwrite is safe for every sub-frame configuration.
3. **Deterministic per sub-frame.** The loop is sequential; after iteration i=nb_frames-1, `self.range_final` is the last sub-frame's final value. That's the value the decoder will compute when it decodes the repacketized packet's last sub-frame.
4. **DTX handled.** If the last sub-frame is DTX (`range_final = 0` via line 2582 or 2607), the correct behavior is to keep that zero. The overwrite currently replaces zero with `celt.rng`, which may be garbage if CELT wasn't used ‚Äî yet another reason the overwrite is wrong.

No other paths or helpers need to change.

## 9. Risk assessment

- **Risk of regression:** Very low. The fix adds one line writing to a local variable inside an already-guarded branch. No control flow changes, no API changes.
- **Risk the fix is wrong:** Very low. It mirrors the C reference line-for-line and is symmetric with an already-correct sibling branch (SILK‚ÜíCELT).
- **Confidence root cause is correct:** 5/5. The XOR delta observed by the phase-4 agent IS the CELT encoder's rng after redundancy encoding ‚Äî this is a direct prediction of the hypothesis.
