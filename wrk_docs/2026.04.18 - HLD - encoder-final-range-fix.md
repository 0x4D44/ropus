# HLD - Fix OpusEncoder::get_final_range divergence on CELT→SILK redundancy

**Date:** 2026-04-18
**Module:** `ropus/src/opus/encoder.rs`
**Surfaced by:** xiph/opus `test_opus_encode.c:501` (via capi FFI shim); reproduced by phase-4 debug agent's standalone differential test.

## 1. Symptom

Encoder differential test: `set_force_mode(MODE_SILK_ONLY)`, bitrate 6809 bps, 1920-sample frames at 48 kHz, bandwidths cycling NB→MB→WB→NB→MB→WB→…

Iteration results (three values compared per frame: ropus encoder `get_final_range`, ropus decoder `get_final_range`, C-ref decoder `OPUS_GET_FINAL_RANGE`):

| Iter | BW | Enc         | Dec (ropus) | Dec (C-ref) | Match |
|------|----|-------------|-------------|-------------|-------|
| 0    | NB | X₀          | X₀          | X₀          | yes   |
| 1    | MB | X₁          | X₁          | X₁          | yes   |
| 2    | WB | `0x0c4c9290`| `0x43f05790`| `0x43f05790`| **no**|
| 3-14 | …  | …           | …           | …           | yes   |

Only iter 2 diverges. XOR delta between encoder and decoder reports = `0x4fbcc500`.

## 2. Reproducer

(As above — phase-4 agent wrote a standalone differential test in `harness/tests/` demonstrating this.) The bitstream ITSELF is valid: two independent decoders agree with each other. Only the encoder's `final_range` bookkeeping is wrong.

## 3. Root cause

**File:** `c:\language\mdopus\ropus\src\opus\encoder.rs:2392-2409` (the "CELT→SILK redundancy frame" block).

Compared to C reference `reference/src/opus_encoder.c:2427-2442`, ropus is missing the call that captures the CELT encoder's final range into the `redundant_rng` variable after the redundancy frame is encoded but before `OPUS_RESET_STATE` wipes it.

### C reference (correct):

```c
/* 5 ms redundant frame for CELT->SILK */
if (redundancy && celt_to_silk)
{
    int err;
    celt_encoder_ctl(celt_enc, CELT_SET_START_BAND(0));
    celt_encoder_ctl(celt_enc, OPUS_SET_VBR(0));
    celt_encoder_ctl(celt_enc, OPUS_SET_BITRATE(OPUS_BITRATE_MAX));
    err = celt_encode_with_ec(celt_enc, pcm_buf, st->Fs/200,
                              data+nb_compr_bytes, redundancy_bytes, NULL);
    if (err < 0) { ... return OPUS_INTERNAL_ERROR; }
    celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng));   // <-- LINE 2440
    celt_encoder_ctl(celt_enc, OPUS_RESET_STATE);
}
```

### ropus (buggy):

```rust
// --- CELT→SILK redundancy frame ---
if redundancy && celt_to_silk {
    if let Some(ref mut celt) = self.celt_enc {
        celt.ctl(CeltEncoderCtl::SetStartBand(0));
        celt.ctl(CeltEncoderCtl::SetVbr(0));
        celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX));
        redundancy_frame = vec![0u8; redundancy_bytes as usize];
        celt_encode_with_ec(
            celt, &pcm_buf, self.fs / 200, &mut redundancy_frame,
            redundancy_bytes, None,
        );
        // *** MISSING: redundant_rng = celt.rng; ***
        celt.ctl(CeltEncoderCtl::ResetState);       // <-- wipes celt.rng to 0
    }
}
```

The Rust code calls `ResetState` without first capturing `celt.rng`. Since `CeltEncoder::reset()` sets `self.rng = 0` (see `ropus/src/celt/encoder.rs:3198`), the redundancy-frame rng is lost.

Later, at line 2564:

```rust
range_final ^= redundant_rng;   // XORs with 0 instead of the real rng
```

The encoder's `range_final` therefore ends up equal to the SILK-finalized range (unchanged), but the decoder — which DOES capture the corresponding rng from its CELT decoder during redundancy decode (`ropus/src/opus/decoder.rs:851`; mirrors `reference/src/opus_decoder.c:589`) — XORs in the real value. Hence the divergence `0x4fbcc500` observed in the reproducer IS the CELT encoder's rng after encoding the 5ms redundancy frame.

### Why only first WB activation?

`redundancy && celt_to_silk` is true only when `silk_bw_switch` is set at frame entry (ropus encoder line 1901). `silk_bw_switch` is set (line 2241) in the PREVIOUS frame only when SILK signals `opus_can_switch` — which happens on an UPWARD SILK-internal bandwidth increase (e.g., 12 kHz → 16 kHz). In the NB→MB→WB cycle, SILK's switch-readiness policy only fires on the 12 kHz→16 kHz step (i.e., the MB→WB boundary), placing redundancy on iteration 2 and no others. The bug is present every time the branch fires; it is not "first WB" specific per se — it just happens to fire there in this particular test pattern.

## 4. Proposed fix

Add one line after `celt_encode_with_ec` and before `ResetState`, mirroring C line 2440:

```rust
if redundancy && celt_to_silk {
    if let Some(ref mut celt) = self.celt_enc {
        celt.ctl(CeltEncoderCtl::SetStartBand(0));
        celt.ctl(CeltEncoderCtl::SetVbr(0));
        celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX));
        redundancy_frame = vec![0u8; redundancy_bytes as usize];
        celt_encode_with_ec(
            celt, &pcm_buf, self.fs / 200, &mut redundancy_frame,
            redundancy_bytes, None,
        );
        redundant_rng = celt.rng;                   // *** NEW LINE ***
        celt.ctl(CeltEncoderCtl::ResetState);
    }
}
```

That is the entire fix. No other paths or helpers need to change.

## 5. Correctness argument

1. **Mirrors reference semantics exactly.** C line 2440 (`celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng))`) reads the CELT encoder's `rng` field between the redundancy encode and the state reset. Our fix does precisely the same Rust-idiomatically (`redundant_rng = celt.rng`), which is how the SILK→CELT branch already does it (line 2556 pattern).

2. **Decoder already mirrors it.** `ropus/src/opus/decoder.rs:851` captures the decoder-side `redundant_rng` in exactly the same place (between CELT decode of the redundancy frame and setting start band). Encoder ↔ decoder symmetry is restored by the fix.

3. **Same-frame initialization preserved.** `redundant_rng` is declared `let mut redundant_rng: u32 = 0;` at line 1993 — fresh each call. Writing to it only in the path that needs it means non-redundancy frames still XOR with 0, preserving their behavior.

4. **Does not perturb the bitstream.** The fix only writes to internal state used for the `get_final_range` query. The bytes emitted on the wire are unchanged. All existing tests that check encoded bytes will continue to pass. Tests that check decoded audio will continue to pass.

5. **Does not affect any other codepath.** The only branch touched is `redundancy && celt_to_silk`. In all of:
   - CELT-only frames (`redundancy` = false, by C line 1944 / ropus line 1907)
   - SILK-only with no bw-switch (`redundancy` = false)
   - SILK→CELT switch (`celt_to_silk = false`)
   the new line is outside the active conditional and cannot execute.

6. **Parity with the SILK→CELT branch.** That branch (encoder.rs:2556: `redundant_rng = celt.rng;`) already has this pattern correctly. The bug is purely an omission in the CELT→SILK branch.

## 6. Alternative considered & rejected

**Alternative:** Move `enc.done()` / `range_final = enc.get_rng()` after the redundancy encode. — Does not address the missing capture; still XORs with 0.

**Alternative:** Skip the XOR entirely when `celt_to_silk`. — Would diverge from C reference and still mismatch both decoders (both C-ref and ropus decoder).

**Alternative:** Capture rng into `range_final` directly (skip the `redundant_rng` temporary). — Works functionally but muddles the intent; the XOR pattern maps 1:1 to C ref and is clearer.

## 7. Other codepaths worth auditing (same bug pattern)

Searched `ropus/src/opus/encoder.rs` for all `OPUS_GET_FINAL_RANGE`/`celt.rng` sites:

- Line 2487: `range_final = celt.rng;` after main CELT encode — present, correct. Matches C line 2509.
- Line 2556: `redundant_rng = celt.rng;` after SILK→CELT redundancy — present, correct. Matches C line 2544.
- **Line 2409 area: missing `redundant_rng = celt.rng;` after CELT→SILK redundancy — BUG #1. Matches missing C line 2440.**

## 7b. Bug #2: `encode_multiframe` overwrites `range_final`

After fixing Bug #1, the conformance `encode` test progresses much further (completing all 3 LP and all 4 Hybrid sub-tests at several SEEDs) but fails at the first `j=7` test (CELT_ONLY, 60ms frames at ~512k-1M bps, which triggers the multi-frame encode path).

### Root cause

`ropus/src/opus/encoder.rs:1840-1846`, at the end of `encode_multiframe`:

```rust
self.range_final = if let Some(ref celt) = self.celt_enc {
    celt.rng
} else {
    0
};
```

This UNCONDITIONALLY overwrites `self.range_final` with `celt.rng` after the multi-frame loop completes.

The inner loop (lines 1769-1814) calls `encode_frame_native` once per sub-frame. Each such call correctly sets `self.range_final` to the sub-frame's final entropy-coder range — including any XOR with `redundant_rng` when redundancy fires on that sub-frame (as it does on the LAST sub-frame when `to_celt=true`, per ropus line 1774-1776 mirroring C line 1782-1783).

After the last sub-frame's `encode_frame_native` returns, `self.range_final` holds the correct "last frame's rng, XOR'd with any redundancy rng" — which is exactly what the decoder will compute when it decodes the packet's final sub-frame. The C reference does NOT touch `rangeFinal` again after the multi-frame loop (`reference/src/opus_encoder.c:1831-1838` — just repacketize and return). The ropus code diverges by overwriting with the raw `celt.rng`, clobbering any redundancy XOR.

### Concrete failing scenario (test_opus_encode.c `j=7`)

- Outer state: `prev_mode == MODE_HYBRID` (from the previous `j=6` Hybrid test), caller forces `MODE_CELT_ONLY`, `frame_size = 2880` (60 ms).
- Mode-transition logic (`encode_float` line 1482-1494) sets `redundancy=true, celt_to_silk=false, to_celt=true, mode = prev_mode = MODE_HYBRID`. Enters `encode_multiframe` with 3 sub-frames of 20 ms each.
- Sub-frame i=2 (last): `frame_redundancy = true`, `frame_to_celt = true` → inside `encode_frame_native`, the SILK→CELT redundancy branch fires and the final `range_final ^= redundant_rng` (lines 2556-2573) produces the correct value. `self.range_final` is set correctly.
- Back in `encode_multiframe`: line 1842 clobbers `self.range_final` with `celt.rng`, which at this point is the POST-redundancy CELT rng (i.e., just `redundant_rng`), not the XOR.
- Result: encoder reports `redundant_rng` but the decoder (mirroring the bitstream) reports `main-rng XOR redundant_rng`. Mismatch. `test_opus_encode.c:501` asserts `enc_final_range == dec_final_range` and aborts.

### Fix for Bug #2

Delete lines 1840-1846. The correct `self.range_final` is already set by the last sub-frame's `encode_frame_native`. Matches C reference (no analogous overwrite).

```rust
// DELETE THIS BLOCK:
self.range_final = if let Some(ref celt) = self.celt_enc {
    celt.rng
} else {
    0
};
```

### Correctness argument for Bug #2 fix

1. **C reference doesn't overwrite.** The C reference's multi-frame loop (`reference/src/opus_encoder.c:1769-1838`) simply returns after repacketizing. `st->rangeFinal` holds whatever the last `opus_encode_frame_native` set.
2. **Every sub-frame correctly sets `self.range_final`.** The inner `encode_frame_native` (line 2573) assigns `self.range_final = range_final;` where `range_final` has already been XOR-mixed with `redundant_rng` if any redundancy path fired. Sub-frames that don't trigger redundancy still set `self.range_final = celt.rng` via the same path (line 2564 XORs with 0). So removing the overwrite is safe for every sub-frame configuration.
3. **Deterministic per sub-frame.** The loop is sequential; after iteration i=nb_frames-1, `self.range_final` is the last sub-frame's final value. That's the value the decoder will compute when it decodes the repacketized packet's last sub-frame.
4. **DTX handled.** If the last sub-frame is DTX (`range_final = 0` via line 2582 or 2607), the correct behavior is to keep that zero. The overwrite currently replaces zero with `celt.rng`, which may be garbage if CELT wasn't used — yet another reason the overwrite is wrong.

No other paths or helpers need to change.

## 10. Bug N — SILK→CELT redundancy writes at stale offset in HYBRID mode

**Date:** 2026-04-18
**File:** `ropus/src/opus/encoder.rs:2532-2568` (the SILK→CELT redundancy branch, `if redundancy && !celt_to_silk`).

### Symptom

After Bugs K + M were fixed (see `wrk_journals/2026.04.18 - JRN - encoder-final-range-bug.md`), `test_opus_encode.c:501` still fails sporadically — the specific failing frame has:
- `mode = MODE_HYBRID` (1001)
- `redundancy = true`, `celt_to_silk = false`, `to_celt = true`
- `ret < nb_compr_bytes` (CELT internally shrank storage well below the allocated max)

Instrumentation shows the encoder's `get_final_range` and the decoder's differ, and the `redundant_rng` in particular diverges because the decoder is decoding stale garbage from the wrong byte offset rather than the bytes the encoder wrote.

### Root cause (confidence 5/5)

In HYBRID VBR, after the main CELT encode runs, `enc.storage` is shrunk by CELT internally to `ret` (the actual byte count used). The opus-level variable `nb_compr_bytes` is, however, still the PRE-encode max (`(max_data_bytes-1) - redundancy_bytes`). The ropus SILK→CELT redundancy block writes the redundancy frame at `enc_data[nb_compr_bytes..]`, which in HYBRID VBR is way past the actual CELT end, leaving a gap of stale bytes between the main CELT data and the redundancy.

The returned packet length is `ret + 1 + redundancy_bytes`, so the LAST `redundancy_bytes` of the packet — what the decoder strips as redundancy — are the stale bytes sitting in `enc_data[ret..ret+redundancy_bytes]`, NOT the bytes the encoder just wrote. Decoder decodes garbage for the redundancy frame, produces some `redundant_rng`, and XORs it into its final range. Encoder captured the "correct" redundant_rng from its CELT encode but XORed it onto its main `range_final` too — so the two final-ranges disagree.

C reference at `opus_encoder.c:2529-2534` avoids this by:
```c
if (st->mode == MODE_HYBRID)
{
    /* Shrink packet to what the encoder actually used. */
    nb_compr_bytes = ret;
    ec_enc_shrink(&enc, nb_compr_bytes);
}
```

The `ec_enc_shrink` is a no-op here because CELT already shrunk `enc` to `ret`. The semantically important line is `nb_compr_bytes = ret;`, which repositions the redundancy write offset.

### Fix

Inside the SILK→CELT redundancy block, before the CELT redundancy encode, add:
```rust
if self.mode == MODE_HYBRID {
    nb_compr_bytes = ret;
}
```
Also change `let nb_compr_bytes: i32;` (line 1992) to `let mut nb_compr_bytes: i32;` to permit the assignment.

### Correctness argument

1. **Mirrors reference semantics.** One-line analogue of C `opus_encoder.c:2532`. The C `ec_enc_shrink` is elided because ropus's CELT encoder has already performed that shrink during its internal VBR encode and the main `enc` is out of scope by this point.
2. **Only affects the intended path.** `self.mode == MODE_HYBRID` gates exactly the HYBRID subset of `redundancy && !celt_to_silk`. SILK_ONLY with SILK→CELT redundancy (the j=7 60ms case covered by the earlier fix) is unchanged — its `nb_compr_bytes` was already set to `ret` at line 2384.
3. **No bitstream format change.** The ON-WIRE format — [TOC][CELT data (ret bytes)][redundancy (redundancy_bytes bytes)] — is exactly what the C encoder produces and what every Opus decoder expects. The fix just writes redundancy at the right offset.
4. **Decoder unchanged.** Decoder side is already correct: `len - redundancy_bytes` locates the CELT portion, the last `redundancy_bytes` are the redundancy. The fix aligns the encoder output to match.

### Reproducer

`harness/tests/c_ref_differential.rs::test_encoder_final_range_matches_decoder_on_hybrid_silk_to_celt_redundancy`:
- 3 warmup Hybrid FB 20ms stereo frames at 24 kbps → `prev_mode = HYBRID`.
- Then force `MODE_CELT_ONLY` + `OPUS_BANDWIDTH_FULLBAND` + 20 ms stereo. The mode-transition-undo reclassifies to `mode=HYBRID, redundancy=true, celt_to_silk=false, to_celt=true`.
- At 24 kbps, CELT's internal VBR encoder shrinks `ret` to ~54 bytes while `nb_compr_bytes` is ~1239 — exactly the condition that triggers the bug.
- Asserts `enc.get_final_range() == ropus_dec.get_final_range() == c_ref_dec.get_final_range()`.
- Fails before the fix, passes after.

## 11. Bug O — `encode_multiframe` clobbers final range captured by last sub-frame

**Date:** 2026-04-18
**File:** `ropus/src/opus/encoder.rs:1840-1846` (the tail of `encode_multiframe`).

### Symptom

`test_opus_encode.c:501` with `SEED=977015203` advances through 39 encode
iterations (LP/Hybrid/MDCT × VBR/CVBR/CBR) and then diverges on the first
j=7 (MDCT FB VBR, rates[7]=512000) frame — the first frame that takes the
multi-frame encode path with a mode transition from the previous j=6
(Hybrid FB VBR) loop.

Instrumentation prints:
```
MULTIFRAME: mode=1001 (HYBRID) redundancy=true celt_to_silk=false
            to_celt=true nb_frames=3
            pre_rng=0x04cc8c00  celt_rng=0x01ba4f00
```
`pre_rng` is the value set by the last sub-frame's `encode_frame_native`
(correctly `main_rng XOR redundant_rng`). `celt_rng` is what ropus THEN
overwrites the field with. They disagree. The decoder's final-range
matches `pre_rng`; encoder reports `celt_rng`; test_opus_encode.c fails.

### Root cause (confidence 5/5)

C reference `opus_encoder.c:1770-1838` runs the multi-frame loop with
each sub-frame calling `opus_encode_frame_native`, which itself sets
`st->rangeFinal` at either line 2511 (plain CELT/CELT-only rng) or line
2553 (XOR with `redundant_rng`). After the sub-frame loop C does NOT
touch `st->rangeFinal` again — the last sub-frame's value IS the returned
final range.

Pre-fix ropus, at the end of `encode_multiframe`, does:
```rust
self.range_final = if let Some(ref celt) = self.celt_enc {
    celt.rng
} else {
    0
};
```
This has two separate defects:

1. **Clobbers the redundancy XOR.** Even when the last sub-frame's
   `range_final` is correctly `main_rng XOR redundant_rng`, the overwrite
   replaces it with a raw `celt.rng` (no XOR). Observed `pre_rng` and
   `celt_rng` differ by exactly a non-zero XOR delta whenever
   `frame_redundancy` fired on the last sub-frame.
2. **Captures the wrong `celt.rng`.** When SILK→CELT redundancy runs
   on the last sub-frame, the CELT encoder is invoked TWICE inside
   `encode_frame_native`: first for the main stream (capturing
   `main_rng`), then reset and invoked again for the 5 ms redundancy
   tail (ending with a completely different `celt.rng` =
   `redundant_rng`). Reading `celt.rng` after the multi-frame loop
   therefore returns `redundant_rng`, not `main_rng`.

Both defects collapse into "trust the sub-frame's bookkeeping, don't
override."

### Trigger conditions

1. `frame_size > self.fs/50` (needs multi-frame path): 40/60/80/100/120 ms
   in CELT-only; 80/120 ms in SILK-only.
2. Last sub-frame has `frame_redundancy = true`. Per C/ropus logic:
   `frame_redundancy = redundancy && (frame_to_celt || (!to_celt && i==0))`.
   On the last sub-frame `frame_to_celt = to_celt`, so `frame_redundancy =
   redundancy && to_celt`. Hence: `redundancy=true, to_celt=true` — i.e.
   transitioning from HYBRID/SILK to CELT at a long frame.
3. `compute_redundancy_bytes` returns > 0 at the per-sub-frame bitrate.

In `test_opus_encode.c`, the j=6→j=7 transition (Hybrid FB → MDCT FB)
at 60 ms produces exactly this: prev_mode=HYBRID, user forces CELT_ONLY,
`compute_redundancy` undoes the transition (`mode=HYBRID, to_celt=true`),
and the multi-frame split runs 3× 20 ms HYBRID sub-frames with the final
one carrying the SILK→CELT redundancy tail.

### Fix

Delete the 5-line overwrite. `encode_frame_native` has already stored
the correct value at every exit path (including DTX and budget-bust). No
replacement is needed.

### Correctness argument

1. **Mirrors reference.** `opus_encoder.c:1770-1838` has no analogue of
   the deleted code.
2. **Previous `self.range_final` writes remain correct.** `encode_frame_native`
   writes `self.range_final` on every exit path (main set at line 2586,
   DTX-reset at 2595, budget-bust reset at 2620). After the loop the
   field already holds the last sub-frame's value.
3. **No DTX edge case.** If the last sub-frame DTXed, `encode_frame_native`
   set `self.range_final = 0`, which IS the correct value to report
   (matches C line 2569).

### Reproducer

`harness/tests/c_ref_differential.rs::test_encoder_final_range_matches_decoder_on_multiframe_silk_to_celt_redundancy`:
- 3 warmup Hybrid FB 20 ms stereo frames at 700 kbps → `prev_mode = HYBRID`.
- Then force `MODE_CELT_ONLY` + FULLBAND + 60 ms stereo. Mode-transition-undo
  yields `mode=HYBRID, redundancy=true, celt_to_silk=false, to_celt=true`.
  60 ms > 20 ms → `encode_multiframe`, 3 sub-frames.
- Asserts encoder's `get_final_range()` equals both the ropus decoder's
  and the C-ref decoder's.
- Fails before the fix (encoder reports `celt_rng` after the redundancy
  tail), passes after.

### Downstream effects

Fix advances `test_opus_encode.c` past the entire rc=0/1/2 (VBR/CVBR/CBR)
× mode (LP/Hybrid/MDCT) encode matrix (39 iterations) on SEED=977015203.
The next failure is in the subsequent multi-stream section (line 591),
which is a different bug class and handled separately.

## 12. Bug P — CELT encoder missing hybrid 37-bit min_allowed floor

**Date:** 2026-04-18
**File:** `ropus/src/celt/encoder.rs:2810` (the `min_allowed` computation).

### Symptom

`test_opus_encode.c:501` with `SEED=460330478` passes all VBR iterations
(j=0..12) then fails on the first CVBR Hybrid FB iteration (j=3,
`rates[3]=16000`, rate≈17109 bps, 20 ms stereo at 48 kHz).

Instrumentation after the encode and at the decoder's end of CELT decode
showed the rng diverge even though the written packet bytes were bit-exact:

```
[ENC_AFTER_SILK] rng=0x0311bd44 tell=279        (encoder)
[DEC_AFTER_SILK] rng=0x0311bd44 tell=279        (decoder, matches)
[CELT_END]     rng=0x0fa80d00 storage=39 tell=312 nbits=340  (encoder)
[CELT_DEC_END] rng=0x07608500 storage=39 tell=312 nbits=339  (decoder)
bytes (encoder)  = 8d06c9e8dad3…e65b7f8   (39 bytes)
bytes (decoder)  = 8d06c9e8dad3…e65b7f8   (39 bytes, identical)
```

Encoder and decoder agree on the post-SILK state. They agree on the final
byte stream. Yet `nbits_total` diverges by 1, i.e. the encoder consumed
exactly one more range-coder symbol than the decoder.

### Root cause (confidence 5/5)

The asymmetry is in the hybrid "redundancy signal" bit.

Encoder (`ropus/src/opus/encoder.rs:2363-2369`, mirrors
`reference/src/opus_encoder.c:2351-2355`):
```rust
if self.mode != MODE_CELT_ONLY
    && enc.tell() + 17 + 20 * (if self.mode == MODE_HYBRID { 1 } else { 0 })
        <= 8 * (max_data_bytes - 1)
{
    if self.mode == MODE_HYBRID {
        enc.encode_bit_logp(redundancy, 12);   // always writes 1 symbol
    }
    ...
}
```
`max_data_bytes = 1500` (the caller's `MAX_PACKET` cap), so
`8*(max_data_bytes-1) = 11992`. With `ec_tell(&enc) = 279` after SILK,
`279+37 = 316 <= 11992` → TRUE → the encoder always writes the
`redundancy=false` symbol in hybrid. `ec_enc_bit_logp(0, 12)` consumes
~0.0003 bits but advances the range-coder `rng` field by a non-zero
multiplier.

Decoder (`ropus/src/opus/decoder.rs:822`, mirrors
`reference/src/opus_decoder.c:501`):
```rust
if !decode_fec
    && mode != MODE_CELT_ONLY
    && data.is_some()
    && dec.tell() + 17 + 20 * (mode == MODE_HYBRID) as i32 <= 8 * len
{
    ...
}
```
Here `len` is the FINAL packet size minus TOC (= `ret`). If CELT's VBR
shrunk `ret` to 39 bytes, `8*len = 312`. `279+37 = 316 > 312` → FALSE →
the decoder SKIPS the redundancy bit entirely.

**Result: the encoder has advanced its range-coder state by one symbol
that the decoder never decodes.** All subsequent CELT symbols are written
to the buffer under one range coder and read back under another, which
produces byte-identical packets but different final `rng` values.

C reference handles this in `celt_encoder.c:2432-2433`:
```c
if (hybrid)
   min_allowed = IMAX(min_allowed,
        (tell0_frac+(37<<BITRES)+total_boost+(1<<(BITRES+3))-1)
        >> (BITRES+3));
```
This floor forces `nb_available_bytes ≥ ceil((tell0_frac + 37*8 +
total_boost + 63)/64)` when encoding hybrid. In our failing scenario,
`tell0_frac ≈ 2232` → `min_allowed ≥ 40 bytes`, so CELT's VBR clamp at
line 2529 (`nbCompressedBytes = IMIN(nbCompressedBytes, nbAvailableBytes)`)
cannot shrink the packet below the 37-bit redundancy-signal budget. The
decoder's check then succeeds and both sides read/write the bit
symmetrically.

Ropus was missing this `if (hybrid)` adjustment entirely — only the
generic `min_allowed` with `tell_frac + total_boost + (1<<(BITRES+3)) - 1`
was applied. For low-bitrate hybrid where CELT's VBR target bytes <
`ceil((tell0+37*8)/64)`, the shrink fell below the floor.

### Fix

`ropus/src/celt/encoder.rs:2810-2828`:
```rust
let mut min_allowed = ((enc_ref.tell_frac() as i32 + total_boost_enc
    + (1 << (BITRES + 3))
    - 1)
    >> (BITRES + 3))
    + 2;
// Take into account the 37 bits we need to have left in the packet to
// signal a redundant frame in hybrid mode. Creating a shorter packet
// would create an entropy coder desync between the encoder (which wrote
// the redundancy bit under its `8*(max_data_bytes-1)` budget) and the
// decoder (whose same check uses `8*len` against the final packet size).
// Matches C `celt_encoder.c:2432-2433`.
if hybrid {
    min_allowed = min_allowed.max(
        (tell0_frac + (37 << BITRES) + total_boost_enc + (1 << (BITRES + 3)) - 1)
            >> (BITRES + 3),
    );
}
```
Also promoted the previously-unused `_tell0_frac` (captured at line 2230)
to `tell0_frac: i32` so it can participate in the formula.

### Correctness argument

1. **Mirrors reference semantics exactly.** Line-for-line port of C
   `celt_encoder.c:2432-2433`. `tell0_frac`, `37<<BITRES`, `total_boost`,
   and the `(1<<(BITRES+3))-1` round-up term are identical to C.
2. **Only affects hybrid frames.** `hybrid = start != 0` (start band
   nonzero). For CELT-only (start=0) this branch is dead.
3. **No bitstream format change.** The fix changes only how small
   `nb_available_bytes` can become during CELT's internal VBR decision.
   The on-wire format is unchanged; the packet simply carries one extra
   byte (or up to a few bytes) when the encoder had been shrinking below
   the 37-bit floor.
4. **Decoder unchanged.** This is purely an encoder-side floor; the
   decoder's budget check was always correct.
5. **Confidence root cause is correct:** 5/5. Encoder and decoder rng
   divergence observed in trace with identical byte sequence exactly
   matches the one-symbol-ahead prediction of the hypothesis (encoder
   `nbits_total=340` vs decoder `nbits_total=339`, off by 1).
