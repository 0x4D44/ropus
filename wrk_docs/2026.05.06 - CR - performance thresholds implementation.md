# Code Review: Performance Thresholds Implementation

Date: 2026.05.06

Scope: independent review of Ampere's uncommitted performance release threshold
implementation.

## Findings

### Issue: unresolved 8.74x live decode smoke contradicts the initial release threshold

References:

- `/home/md/language/ropus/full-test/src/bench.rs:268` defines the
  `CELT 48k mono sine 1k loud` threshold row.
- `/home/md/language/ropus/full-test/src/bench.rs:273` sets that row's decode
  release-fail threshold to `1.26x`.
- `/home/md/language/ropus/wrk_docs/2026.05.06 - MEA - performance threshold baseline.md:21`
  says the implementation did not replace the README-derived table with a fresh
  five-sweep baseline.
- `/home/md/language/ropus/wrk_journals/2026.05.06 - JRN - performance thresholds implementation.md:65`
  records the live smoke measurement where C decode was `0.647ms`, Rust decode
  was `5.650ms`, and the ratio was `8.74x`.

The gate semantics correctly classify a ratio above the per-row release
threshold as release-blocking. The problem is that the only live benchmark
evidence collected during this slice would breach the proposed threshold by a
large margin. That may be a real severe regression, a short `--iters 3`
measurement artifact, or stale README-derived threshold data, but it is not
resolved in the baseline note.

Do not silence this by widening the threshold to `8.74x`. Before treating the
slice as complete, either run a focused calibrated measurement for this row
(preferably the signed `--iters 30` path, repeated if practical) or record an
explicit supervisor decision that the new non-quick release preflight is
expected to fail on the current code until the decode regression is investigated.
Right now the implementation risks committing an immediately red release gate
without making that decision visible in the measurement artifact.

### Issue: two required release-blocking classes are implemented but not unit-tested

References:

- `/home/md/language/ropus/wrk_docs/2026.05.06 - CR - performance thresholds hld signoff.md:23`
  requires unit-tested threshold classification using synthetic rows.
- `/home/md/language/ropus/wrk_docs/2026.05.06 - CR - performance thresholds hld signoff.md:40`
  and `/home/md/language/ropus/wrk_docs/2026.05.06 - CR - performance thresholds hld signoff.md:44`
  require crashed rows and missing thresholds to be release-blocking.
- `/home/md/language/ropus/full-test/src/bench.rs:414` implements missing-threshold
  blocking.
- `/home/md/language/ropus/full-test/src/bench.rs:426` implements crashed-row
  blocking.
- `/home/md/language/ropus/full-test/src/bench.rs:1163` through
  `/home/md/language/ropus/full-test/src/bench.rs:1300` cover release skip,
  missing fixture, missing timing, ratio breach, and pass cases, but not the
  missing-threshold or release-thresholded crashed-row cases.

The production code appears to handle both classes, but these were explicitly
called out in the signed contract. Add small pure tests that construct:

- an unknown-label release-thresholded row and assert `banner_fail()`,
  `missing threshold spec`, and `BenchThresholdStatus::MissingThreshold`;
- a release-thresholded crashed row and assert `banner_fail()`,
  `benchmark row crashed`, and `BenchThresholdStatus::Crashed`.

## Assessment

The implementation mostly matches the signed HLD/signoff scope. It stays within
the approved files, avoids codec changes, does not add dependencies or versioned
file churn, and keeps default/local benchmark behavior WARN-only.

Profile behavior is correct:

- default with benchmarks selects `observed-warn-only`;
- default with `--skip-benchmarks` selects `not-claimed`;
- `--quick --release-preflight` selects `not-claimed`;
- non-quick `--release-preflight` selects `release-thresholded`;
- non-quick `--release-preflight --skip-benchmarks` stays
  `release-thresholded` and fails instead of silently downgrading.

Release-thresholded failure semantics are mostly correct in code: build failure,
missing binary, skipped/missing required fixtures, crashed rows, incomplete
timings, zero gateable rows, missing thresholds, and threshold breaches all feed
`release_blocking_issues()`.

Thresholds are per-vector/per-operation and are applied to same-run Rust/C
ratios parsed from each benchmark row. The old global `1.15x` ratio remains a
WARN-only visual/reporting signal, not the release-fail threshold.

Report, help, HTML, JSON, and oracle-matrix wording are honest about the claim
being thresholded smoke over the existing vector set, not broad performance
coverage.
