## High-Level Read

The repo is healthy on many helper-heavy modules and materially weaker in large control-flow modules. The current suite is broad, and the cheap helper wins have mostly been harvested already. The remaining debt is concentrated in large encoder state machines plus a second tier of packet/decode orchestration files.

This is not a “find two missing tests” situation. The remaining gap to `98%` is still over a thousand uncovered branches, which means the target is now dominated by expensive control-flow work inside large stateful modules.

## Most Important Coverage Interpretation

There are four different coverage problems in the repo:

1. `src\silk\encoder.rs` and `src\opus\encoder.rs` are still the dominant debt buckets. They are large, stateful, and branch-heavy.
2. `src\silk\decoder.rs`, `src\opus\repacketizer.rs`, and `src\opus\decoder.rs` remain worthwhile because they still expose deterministic branch debt without the full encoder-state cost.
3. `src\celt\bands.rs` and `src\celt\encoder.rs` are now firmly in the critical set because many easier Opus/DNN gains have already been captured.
4. Functions are no longer the issue. Branches and regions are the honest constraint now.

## Best Near-Term Return

The fastest realistic next wave is:

## Risks To The 98% Target

The target is still technically measurable but expensive.

Main risks:

- branch coverage is still only `80.93%`, so line-only helper tests are no longer enough
- the largest remaining debt lives in orchestrator-style encoders with wide state surfaces
- later gains will get more expensive per point because most low-risk helpers are already covered
- `98%` branch coverage from the current position would require covering `975` more branches, which is still a large engineering project rather than a tidy cleanup pass
