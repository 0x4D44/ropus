## High-Level Read

The repo is healthy on many helper-heavy modules and materially weaker in large control-flow modules. The current suite is broad, but the easy gains have mostly been harvested already. The remaining debt is concentrated in a handful of encoder, decoder, and packet-orchestration files.

This is not a “find two missing tests” situation. The measured gap to `98%` is still large enough that the work has to be staged and hotspot-driven.

## Most Important Coverage Interpretation

There are four different coverage problems in the repo:

1. `src\silk\encoder.rs` and `src\opus\encoder.rs` are the dominant debt buckets. They are large, stateful, and branch-heavy.
2. `src\dnn\lpcnet.rs` is still missing a lot of lines, but it is deterministic and comparatively test-friendly.
3. `src\opus\multistream.rs`, `src\opus\decoder.rs`, and `src\opus\repacketizer.rs` are good branch-recovery targets because they expose a lot of validation and packet-shape logic without deep codec scaffolding.
4. Most CELT helper modules are no longer the best use of time for repo-wide movement.

## Best Near-Term Return

The fastest realistic next wave is:

1. `src\dnn\lpcnet.rs`
2. `src\opus\multistream.rs`
3. `src\opus\decoder.rs`
4. `src\opus\repacketizer.rs`

Why this order:

- they still hold meaningful uncovered code
- most misses are deterministic branch or validation paths
- new tests can live in existing inline test modules
- they are less likely to destabilize parity-sensitive hot paths than the two big encoders

## Clean-Run Blockers

There are currently no test failures blocking branch coverage.

The blockers found during this pass were:

- nightly is required for `cargo llvm-cov --branch`
- the output directory for JSON artifacts must exist before writing to a custom path

Neither blocker required source fixes.

## Risks To The 98% Target

The target is still technically measurable but expensive.

Main risks:

- branch coverage is only `76.57%`, so line-only helper tests will not be enough
- the largest remaining debt lives in orchestrator-style encoders with wide state surfaces
- later gains will get more expensive per point because many low-risk helpers are already covered

## Recommended Next Actions

1. Add deterministic tests in `src\dnn\lpcnet.rs` for model-load, feature-buffer, and PLC/FEC helper paths.
2. Add validation and branch-matrix tests in `src\opus\multistream.rs`.
3. Add packet-shape and wrapper error-path tests in `src\opus\decoder.rs` and `src\opus\repacketizer.rs`.
4. Re-run branch-aware coverage after that wave and re-rank hotspots from fresh JSON.
5. Only then decide how much of the remaining gap must be paid down inside `src\silk\encoder.rs` and `src\opus\encoder.rs`.
