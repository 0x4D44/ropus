# HLD: Feature-Flagged SIMD Acceleration for Hot Paths

## 1. Summary

This design adds optional SIMD acceleration to the four most performance-critical
functions in the mdopus codec, gated behind a `cargo` feature flag so that the
default build remains unchanged: scalar, safe Rust, zero dependencies.

The codec is a fixed-point implementation. All arithmetic operates on `i32`
values (with `i16` intermediate truncations and `i64` widening multiplies).
There are no floating-point operations in the hot paths, which eliminates
associativity/ordering concerns but introduces its own SIMD challenge: the
dominant operation is `mac16_16` (16x16 -> 32 multiply-accumulate with `i16`
truncation of inputs), which does not map trivially to a single x86 intrinsic.

**Key design decision:** We recommend a **hybrid approach** -- the `wide` crate
(Option B) for the primary SIMD implementation, with auto-vectorization hints
(Option D) as a secondary, zero-cost optimization layer. This combination
respects the safe-Rust constraint while delivering predictable speedups on
x86_64. See Section 3 for the full trade-off analysis.


## 2. Hot Path Analysis

### 2.1 xcorr_kernel -- Cross-Correlation (Priority 1)

**File:** `c:\language\mdopus\src\celt\pitch.rs:33-119`

**Called from:** `celt_pitch_xcorr` (same file, line 157), which is called on
every frame during pitch analysis. For a typical 20ms frame at 48kHz (960
samples) searching 18.75ms of pitch range (900 lags), `xcorr_kernel` is called
~225 times, each processing ~960 multiply-accumulates. That is ~860k MAC
operations per frame.

**Current structure:** Computes 4 cross-correlations simultaneously by sliding
a window of `y` values across `x`. The inner loop is manually unrolled 4x,
processing 4 x-values per iteration. Each iteration performs 16 `mac16_16`
calls (4 correlations x 4 x-values). A tail loop handles the final 0-3
elements.

**Data types:**
- `x: &[i32]`, `y: &[i32]` -- input signals, values are Q12 or Q15 but stored
  as i32. Only the low 16 bits are used by `mac16_16` (which casts to i16).
- `sum: &mut [i32; 4]` -- accumulators, wrapping add via `.wrapping_add()`.
- `mac16_16(c, a, b)` = `c.wrapping_add((a as i16 as i32) * (b as i16 as i32))`

**Vectorization opportunity:** This is textbook SIMD territory. The 4 sums are
independent accumulators updated with the same `x[j]` value but different
`y[j+0..j+3]` offsets. With SSE2 (4x i32 lanes):
- Load 4 y-values into one `i32x4` register
- Broadcast x-value to all 4 lanes
- Truncate both to i16, multiply, accumulate into sum register
- The sliding window means y-values shift by one each outer iteration

With AVX2 (8x i32 lanes), we could process 8 correlations at a time in the
outer `celt_pitch_xcorr` loop rather than 4, though this requires restructuring
the caller.

**Critical detail -- i16 truncation:** `mac16_16` truncates both operands to
i16 before multiplying. The SIMD path must replicate this exactly: pack i32
values to i16 (discarding high bits, not saturating), then use `i16` multiply
widening to i32. On SSE2 this is `_mm_packs_epi32` + `_mm_madd_epi16`, but the
pack operation saturates rather than truncates. The `wide` crate's `i16x8`
and `i32x4` types provide the right semantics via explicit bitwise masking.

**Expected speedup:** 3-4x with SSE2 (4-wide i32), potentially 6-8x with AVX2
(8-wide). The function is purely compute-bound with excellent data locality.

### 2.2 FFT Butterflies -- Radix-4 Twiddle Multiplies (Priority 2)

**File:** `c:\language\mdopus\src\celt\fft.rs:185-243` (kf_bfly4, general case)

**Called from:** `opus_fft_impl` (same file, line 375), which runs the mixed-radix
FFT at the core of every MDCT. For a 480-point FFT (typical 20ms frame), the
factorization is 4x4x4x4x... yielding multiple radix-4 stages operating on the
full buffer.

**Current structure:** The general radix-4 butterfly (lines 211-243) iterates
over `m` elements per block, `n` blocks. For each element:
1. Three complex twiddle multiplies: `c_mul(data, twiddle)` -- each twiddle
   multiply is 4 `s_mul` calls (which are `mult16_32_q15` = i16 x i32 >> 15
   via i64 intermediate).
2. Butterfly additions/subtractions: 8 `add32_ovflw`/`sub32_ovflw` operations.
3. Final store of 4 complex outputs.

**Data types:**
- `fout: &mut [KissFftCpx]` where `KissFftCpx { r: i32, i: i32 }`.
- Twiddle factors: `KissTwiddleCpx { r: i16, i: i16 }`.
- `s_mul(a, b)` = `mult16_32_q15(b, a)` = `((b as i16 as i64) * (a as i64) >> 15) as i32`

**Vectorization opportunity:** Each butterfly iteration is independent across
the `m` index (different frequency bins, same twiddle at the same stride
position). We can process 4 butterflies in parallel by loading 4 consecutive
complex values from each of the 4 butterfly arms, performing the twiddle
multiplies in SIMD, and executing the butterfly additions.

The complication is `mult16_32_q15`: this is a 16x32->64>>15 operation. SSE2
has no direct i16xi32 multiply. The implementation would need to:
1. Load twiddle as i16, sign-extend to i32.
2. Multiply i32 x i32 -> i64 (via `_mm_mul_epi32` on SSE4.1, or emulated on SSE2).
3. Shift right by 15.

This is feasible but less clean than the `xcorr_kernel` case.

**Also target:** `fft_downshift` (lines 120-134) -- a simple element-wise
right-shift over the full FFT buffer. Trivially vectorizable.

**Also target:** The degenerate radix-4 case (lines 193-210) processes 4
complex values per iteration with no twiddle multiplies -- pure add/sub.
Easily vectorized.

**Expected speedup:** 2-4x. The butterfly has good parallelism but the
twiddle multiply is more complex than a simple MAC. Memory access patterns
(strided reads) may reduce effective throughput.

### 2.3 Band Energy Computation (Priority 3)

**File:** `c:\language\mdopus\src\celt\bands.rs:129-163`

**Called from:** `compute_band_energies`, invoked once per frame per channel for
both encoder and decoder. Iterates over 21 bands (typical 48kHz configuration),
summing squared coefficients within each band.

**Current structure:** For each band:
1. Find `maxval` via `celt_maxabs32` (max absolute value scan, line 144).
2. Compute dynamic shift to prevent overflow (line 146-150).
3. Inner loop: `sum += mult32_32_q31(shl32(x[j], shift), shl32(x[j], shift))`
   -- shift input left, then square via 32x32->64>>31 (lines 152-155).
4. Final `celt_sqrt32` on the sum.

**Data types:**
- `x: &[i32]` -- frequency coefficients.
- `sum: i32` -- accumulator, non-wrapping addition (via `add32`).
- `mult32_32_q31(a, b)` = `((a as i64) * (b as i64) >> 31) as i32`

**Vectorization opportunity:** Two sub-functions are good SIMD targets:

(a) `celt_maxabs32` (`c:\language\mdopus\src\celt\math_ops.rs:564-572`): A
simple max/min scan over i32 values. Vectorizes to parallel `max_epi32` /
`min_epi32` (SSE4.1) with a final horizontal reduction. Band sizes range from
4 to ~96 coefficients.

(b) The squared-sum inner loop (bands.rs lines 152-155): Each iteration is
`shl32(x, shift)` then `mult32_32_q31(xv, xv)` then `add32(sum, ...)`. The
shift is uniform across the band, so this vectorizes to: load 4 x i32, shift
left, widen to i64, square, shift right 31, accumulate. The widening to i64
is necessary -- `i32 * i32` overflows 32 bits -- which limits us to 2
multiplies per 128-bit register (or 4 per 256-bit AVX2 register).

**Expected speedup:** 2-3x. The bands are relatively short (4-96 elements), so
setup overhead matters. The i64 widening halves effective SIMD width.

### 2.4 Windowed Overlap-Add (Priority 4)

**File:** `c:\language\mdopus\src\celt\mdct.rs:220-238`

**Called from:** `clt_mdct_backward` (same file, line 71), the inverse MDCT.
Runs once per frame (or once per sub-block for short-block modes). Overlap is
typically 120 samples.

**Current structure:** Iterates `overlap/2` times (typically 60 iterations).
Each iteration:
1. Read two output values (`x1`, `x2`) from opposite ends of the overlap region.
2. Read two window values from opposite ends of the window table.
3. Compute two results, each a sum/difference of two `s_mul` calls.
4. `s_mul(a, b)` = `mult16_32_q15(b, a)` = i16 x i32 -> i64 >> 15.

**Data types:**
- `output: &mut [i32]` -- signal buffer (read and written in-place).
- `window: &[i16]` -- synthesis window coefficients in Q15.
- Each iteration does 4 `mult16_32_q15` calls, 1 `add32_ovflw`, 1 `sub32_ovflw`.

**Vectorization opportunity:** The loop body is 4 multiplies + 2 add/sub per
iteration. With 60 iterations, that is only 240 multiplies total -- modest
absolute work. The access pattern (two pointers walking inward from opposite
ends) complicates SIMD loading but is manageable with reverse-load operations.

The `mult16_32_q15` operations are the same type as in the FFT butterflies,
so the SIMD primitive developed for Section 2.2 can be reused here.

**Expected speedup:** 2-4x in isolation, but the absolute time spent here is
small compared to xcorr_kernel and FFT. This is a "while we're at it" target.


## 3. SIMD Strategy Recommendation

### Options Evaluated

| Option | Approach | Safe Rust? | Stable? | Dependency | Speedup predictability |
|--------|----------|-----------|---------|------------|----------------------|
| A | `std::arch` intrinsics | No (unsafe) | Yes | None | High |
| B | `wide` crate | Yes (safe) | Yes | `wide` | High |
| C | `std::simd` portable SIMD | Yes (safe) | No (nightly) | None | High |
| D | Auto-vectorization hints | Yes (safe) | Yes | None | Low |

### Recommendation: Option B (wide) as primary, Option D as secondary

**Primary: `wide` crate (v0.7.x)**

The `wide` crate wraps `std::arch` intrinsics in safe Rust abstractions,
providing types like `i32x4`, `i16x8`, `i32x8` that map directly to SSE2/AVX2
registers. It is:

- **Safe:** No `unsafe` in user code. The crate's own `unsafe` is audited and
  encapsulated -- the same trust model as using `Vec` or `HashMap`.
- **Stable:** Works on stable Rust, no nightly features required.
- **Mature:** Actively maintained, used in production by several projects
  (e.g., `softbuffer`, font rasterizers).
- **Zero-cost:** Compiles down to the same instructions as hand-written
  intrinsics. The safe wrappers are `#[inline(always)]`.
- **Multi-platform:** Provides SSE2, AVX2, and NEON backends behind the
  same API, with scalar fallback on unsupported platforms.

The `wide` crate's type system naturally enforces correct lane widths and
prevents the accidental type confusion that plagues raw intrinsic code.

**Secondary: Auto-vectorization optimization (Option D)**

Even without explicit SIMD types, we can improve auto-vectorization of
existing scalar code through structural changes:

- Replace index-chasing loops with iterator patterns that LLVM recognizes.
- Use `.chunks_exact()` to hint at loop trip counts.
- Ensure inner loop bodies have no data-dependent branches.
- Add `#[target_feature(enable = "...")]` annotations (these are safe on
  function declarations, only unsafe when calling).

These changes benefit all platforms including those without `wide` support,
and they cost nothing -- no dependency, no feature flag, just better code.

**Why not Option A (std::arch)?**

The project rule states: "Safe Rust only; `unsafe` permitted only for FFI to C
reference in test harness." Using raw `std::arch` intrinsics would require
`unsafe` blocks throughout the codec's core computation paths. This conflicts
directly with the project's safety posture. While the operations themselves are
safe (no memory unsafety possible with properly typed SIMD arithmetic), the
`unsafe` annotations would be pervasive and would undermine the project's
ability to use `unsafe`-counting as a code quality metric.

**Why not Option C (std::simd)?**

`std::simd` is the ideal long-term solution -- safe, zero-dependency, in the
standard library. However, it requires `#![feature(portable_simd)]` which pins
the project to nightly Rust. Since mdopus targets stable Rust for production
use, this is not viable today. When `std::simd` stabilizes, we should migrate
from `wide` to `std::simd` and drop the dependency. The migration will be
straightforward since both use similar type-safe vector abstractions.

### Dependency risk assessment for `wide`

- **Compile-time cost:** Minimal. `wide` is a thin wrapper with no transitive
  dependencies beyond `safe_arch` and `bytemuck`.
- **Maintenance burden:** The `wide` API is stable and rarely breaks. Our usage
  will be confined to 4 functions behind a feature flag.
- **Audit surface:** `wide` + `safe_arch` + `bytemuck` are small crates with
  well-understood `unsafe` internals. All three are widely used and audited.
- **Exit strategy:** If `wide` becomes unmaintained, migration to `std::simd`
  (when stabilized) or to raw `std::arch` (relaxing the safe-Rust rule) is
  straightforward since our SIMD code will be isolated behind the feature gate.


## 4. Feature Flag Design

### Cargo.toml

```toml
[features]
default = []
simd = ["wide"]

[dependencies]
wide = { version = "0.7", optional = true }
```

The `simd` feature is opt-in (not in `default`). The codec compiles and works
identically without it. Users opt in with `cargo build --features simd`.

### Code Organization

New module: `src/celt/simd.rs` -- contains all SIMD implementations.

```
src/celt/
  simd.rs          # SIMD implementations of hot path functions (new)
  pitch.rs         # xcorr_kernel, celt_pitch_xcorr (existing, unchanged)
  fft.rs           # butterflies, fft_downshift (existing, unchanged)
  bands.rs         # compute_band_energies (existing, unchanged)
  mdct.rs          # clt_mdct_backward (existing, unchanged)
  math_ops.rs      # celt_maxabs32 (existing, unchanged)
```

### Dispatch Pattern: Module-Level Conditional Import

Each hot path function gets a SIMD variant in `simd.rs`. The call sites use
`#[cfg]` to select between scalar and SIMD at compile time:

```rust
// In src/celt/pitch.rs:

#[cfg(not(feature = "simd"))]
pub fn xcorr_kernel(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    // ... existing scalar implementation (unchanged) ...
}

#[cfg(feature = "simd")]
pub fn xcorr_kernel(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    super::simd::xcorr_kernel_simd(x, y, sum, len);
}
```

**Rationale for compile-time dispatch over runtime dispatch:**

- **Zero overhead:** No function pointer indirection, no branch on every call.
- **Simpler correctness reasoning:** Only one code path is active per build.
- **Matches the project pattern:** The C reference uses `#ifdef` for the same
  purpose (e.g., `OPUS_ARM_MAY_HAVE_NEON`).
- **Runtime detection is unnecessary:** Users who enable `simd` know their
  target platform. Cross-compiled binaries for specific platforms are the norm
  for codec deployment.

If runtime CPU feature detection is needed later (e.g., for a universal binary),
it can be added as a `simd-runtime` feature that uses `std::arch::is_x86_feature_detected!`
with function pointer dispatch. But this is not needed for the initial design.

### Preserving the Scalar Path

The existing scalar implementations are NOT modified or deleted. They remain as
the default (non-`simd`) code path and serve as the correctness oracle for
testing. The `#[cfg]` attributes gate which version is compiled, but both
versions exist in the source at all times.


## 5. Implementation Plan

### Phase 1: Infrastructure (1 day)

1. Add `simd` feature and `wide` dependency to `Cargo.toml`.
2. Create `src/celt/simd.rs` with module declaration gated on `#[cfg(feature = "simd")]`.
3. Add `#[cfg(feature = "simd")]` module declaration in `src/celt/mod.rs`.
4. Implement SIMD helper: `mac16_16_x4` -- the vectorized equivalent of 4
   parallel `mac16_16` calls. This is the core primitive reused everywhere.

```
// Pseudocode for the fundamental SIMD primitive:
fn mac16_16_x4(acc: i32x4, a: i32x4, b: i32x4) -> i32x4 {
    // Truncate a and b to i16 (mask off high bits, sign-extend)
    let a16 = (a << 16) >> 16;  // sign-extend low 16 bits
    let b16 = (b << 16) >> 16;
    // Multiply i32 x i32 -> i32 (low 32 bits, which is correct since
    // both inputs fit in i16 range after truncation)
    let product = a16 * b16;
    // Wrapping accumulate
    acc + product
}
```

The key insight: since both inputs are truncated to i16 range before multiply,
the product fits in 31 bits (max: 32767 * 32767 = 1,073,676,289). We can use
i32 x i32 -> i32 multiply (which `wide`'s `i32x4` supports directly) without
needing i64 widening. This is a significant simplification.

### Phase 2: xcorr_kernel (2 days)

Implement `xcorr_kernel_simd` in `src/celt/simd.rs`.

**Strategy:** The existing scalar code processes 4 correlations simultaneously.
With SSE2/`i32x4`, we can compute all 4 sums in a single register.

```
// Pseudocode for the SIMD inner loop (one x-value per iteration):
fn xcorr_kernel_simd(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    let mut acc = i32x4::from(*sum);
    for j in 0..len {
        let xv = i32x4::splat(x[j]);              // broadcast x[j]
        let yv = i32x4::new([y[j], y[j+1], y[j+2], y[j+3]]);  // 4 consecutive y's
        acc = mac16_16_x4(acc, xv, yv);
    }
    *sum = acc.to_array();
}
```

This is simpler than the scalar version because SIMD eliminates the need for
manual 4x unrolling -- the 4-way parallelism is in the SIMD lanes instead.
The scalar version's register-renaming dance (y_0/y_1/y_2/y_3) becomes a
simple sequential load.

We could further unroll the SIMD loop 4x (processing 4 x-values per iteration,
matching the scalar version's stride) to improve instruction throughput.

**Wire-up:** Add `#[cfg]` dispatch in `pitch.rs` as described in Section 4.

### Phase 3: FFT Butterflies (3 days)

Implement SIMD variants of `fft_downshift`, `kf_bfly4` (degenerate case), and
`kf_bfly4` (general case).

**fft_downshift** (`fft.rs:120-134`): Trivial. Load 4 elements, shift right,
store. Process the buffer in i32x4 chunks.

```
fn fft_downshift_simd(x: &mut [KissFftCpx], n: usize, total: &mut i32, step: i32) {
    let shift = imin(step, *total);
    *total -= shift;
    if shift > 0 {
        // Process as flat i32 slice (r,i,r,i,...)
        // Load 4 i32s at a time, add rounding bias, shift right
    }
}
```

**kf_bfly4 degenerate** (`fft.rs:193-210`): Each iteration processes 4 complex
values using only add/sub. We can load 2 complex values (4 i32s) per register
and perform butterfly operations with `i32x4` add/sub.

**kf_bfly4 general** (`fft.rs:211-243`): This requires SIMD `c_mul`, which
involves `mult16_32_q15`. For this operation:

```
// mult16_32_q15(a, b) = ((a as i16 as i64) * (b as i64) >> 15) as i32
// For SIMD: a is i16 (twiddle), b is i32 (data)
// Strategy: use i32 multiply + shift, exploiting that a fits in i16
fn mult16_32_q15_x4(a: i32x4, b: i32x4) -> i32x4 {
    // a values are known to fit in i16; sign-extend for safety
    let a16 = (a << 16) >> 16;
    // We need (a16 * b) >> 15, but a16*b can overflow i32.
    // Split: b = b_hi*32768 + b_lo where b_lo = b & 0x7FFF, b_hi = b >> 15
    // result = a16 * b_hi + (a16 * b_lo) >> 15
    // This matches the C decomposed MAC16_32_Q15 form.
}
```

This decomposed form avoids the i64 intermediate entirely, staying in i32x4
throughout. It matches the non-OPUS_FAST_INT64 path of the C reference, which
uses the same decomposition. We must verify bit-exactness against the i64 path
(they should be equivalent for i16-range `a` values).

**Wire-up:** The FFT functions are private to `fft.rs`. We will either:
(a) Make SIMD variants in `simd.rs` and call them from `fft.rs` via `cfg`, or
(b) Inline the `cfg` dispatch within `fft.rs` for each butterfly function.

Option (b) is preferred since the butterflies are tightly coupled to the FFT
control flow and factorization logic.

### Phase 4: Band Energy + Overlap-Add (2 days)

**Band energy** (`bands.rs:129-163`):

Two sub-targets:

(a) Vectorize `celt_maxabs32` in `math_ops.rs:564-572`. Replace the scalar
max/min scan with `i32x4` parallel max/min, finishing with a horizontal
reduction. Use `wide`'s `i32x4::max` and `i32x4::min`.

(b) Vectorize the squared-sum loop in `compute_band_energies` (bands.rs:152-155).
The `mult32_32_q31(xv, xv)` operation requires i64 widening: `(xv as i64)^2 >> 31`.
With SSE2/`i32x4`, process 2 elements at a time (each producing an i64 product).
With AVX2, process 4 at a time. Use `wide`'s `i64x2` or `i64x4` types.

**Overlap-add** (`mdct.rs:220-238`):

This loop accesses data from both ends walking inward. For SIMD:
1. Load 4 consecutive `output[yp1..yp1+4]` and 4 consecutive
   `output[xp1-3..xp1+1]` (reversed).
2. Load 4 window values from each end.
3. Perform 4 `mult16_32_q15` calls in SIMD (reusing the Phase 3 primitive).
4. Store results.

The reverse-access pattern for `xp1` requires a shuffle/reverse operation after
loading. `wide` provides lane-shuffle operations for this.

With only 60 iterations (overlap=120), this gives ~15 SIMD iterations. The
speedup will be real but modest in absolute terms.

### Phase 5: Optimization Pass (2 days)

After all four functions are implemented and verified:

1. **AVX2 variants:** Add `#[cfg(target_feature = "avx2")]` paths that use
   `i32x8` (8-wide) for xcorr_kernel and FFT butterflies. The `wide` crate
   supports this transparently via `i32x8`.
2. **Alignment:** Ensure hot buffers are 32-byte aligned for AVX2 loads. This
   may require changing allocation patterns for FFT scratch buffers.
3. **Auto-vectorization audit:** Check compiler output for remaining scalar
   loops (e.g., `dual_inner_prod`, `celt_inner_prod`, `celt_fir5`) to see if
   LLVM auto-vectorizes them. Add `.chunks_exact()` patterns where it helps.


## 6. Testing Strategy

### 6.1 Bit-Exactness Requirement

**The SIMD path must produce identical output to the scalar path, bit for bit.**

This is not optional. The codec's fixed-point arithmetic is designed so that
every intermediate value is deterministic. Changing even one bit in the
xcorr_kernel output could shift pitch detection by a sample, changing the
entire encoded bitstream. The existing integration test suite (280 test vectors)
is the primary verification tool.

### 6.2 Test Plan

**Level 1: Unit tests (per-function)**

For each SIMD function, add tests in `src/celt/simd.rs` that:
1. Generate random input data (seeded RNG for reproducibility).
2. Call both scalar and SIMD versions.
3. Assert exact equality of all output values.
4. Cover edge cases: len=3 (minimum for xcorr_kernel), len=1, empty bands,
   overlap=2 (minimum), large values near i16/i32 boundaries.

```rust
#[cfg(test)]
mod tests {
    #[test]
    fn xcorr_kernel_simd_matches_scalar() {
        let x = [/* test data */];
        let y = [/* test data */];
        let mut sum_scalar = [0i32; 4];
        let mut sum_simd = [0i32; 4];
        super::super::pitch::xcorr_kernel_scalar(&x, &y, &mut sum_scalar, x.len());
        super::xcorr_kernel_simd(&x, &y, &mut sum_simd, x.len());
        assert_eq!(sum_scalar, sum_simd);
    }
}
```

This requires keeping the scalar versions accessible for testing even when the
`simd` feature is enabled. We achieve this by giving the scalar functions a
`_scalar` suffix alias:

```rust
// Always compiled (both with and without simd feature):
pub(crate) fn xcorr_kernel_scalar(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    // ... existing implementation ...
}

// Public API dispatches based on feature:
#[cfg(not(feature = "simd"))]
pub fn xcorr_kernel(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    xcorr_kernel_scalar(x, y, sum, len);
}

#[cfg(feature = "simd")]
pub fn xcorr_kernel(x: &[i32], y: &[i32], sum: &mut [i32; 4], len: usize) {
    super::simd::xcorr_kernel_simd(x, y, sum, len);
}
```

**Level 2: Integration tests (full codec)**

Run the existing 280-case test suite with `--features simd`:

```bash
cargo test --features simd
cargo run --features simd --bin mdopus-compare -- roundtrip assets/test.wav
```

If any test diverges, the SIMD implementation has a bug. The existing test
infrastructure already compares Rust output against C reference byte-for-byte,
so this is a complete correctness check.

**Level 3: CI matrix**

Add a CI job that runs the full test suite under both configurations:
- `cargo test` (scalar, default)
- `cargo test --features simd` (SIMD)

Both must pass. This prevents SIMD regressions from being merged.

### 6.3 The mult16_32_q15 Decomposition Correctness Proof

The SIMD FFT butterfly uses a decomposed form of `mult16_32_q15` to avoid i64:

```
result = a * (b >> 15) + (a * (b & 0x7FFF)) >> 15
```

where `a` fits in i16. This must produce the same result as:

```
result = ((a as i64) * (b as i64) >> 15) as i32
```

The equivalence holds when `a` fits in i16 range [-32768, 32767]. We should add
an exhaustive test that verifies this for all i16 values of `a` against a
representative sample of `b` values (including edge cases: 0, 1, -1,
i32::MAX, i32::MIN, powers of 2).


## 7. Benchmark Plan

### Micro-benchmarks

Use `criterion` (or `divan`) to benchmark each hot path function individually.

Add to `Cargo.toml`:
```toml
[dev-dependencies]
criterion = "0.5"

[[bench]]
name = "simd_benchmarks"
harness = false
```

Benchmark structure:
```
benches/simd_benchmarks.rs
  - xcorr_kernel: len=960, random data, scalar vs SIMD
  - kf_bfly4: m=120, n=4, scalar vs SIMD
  - compute_band_energies: full 21-band config, scalar vs SIMD
  - overlap_add: overlap=120, scalar vs SIMD
  - fft_downshift: n=480, scalar vs SIMD
```

Each benchmark runs both scalar and SIMD variants (the scalar version via
the `_scalar` suffix function) so they can be directly compared in the
same criterion report.

### Macro-benchmarks

Time full encode and decode operations:

```bash
# Scalar baseline
cargo build --release
time target/release/mdopus-compare roundtrip assets/test.wav

# SIMD
cargo build --release --features simd
time target/release/mdopus-compare roundtrip assets/test.wav
```

Report wall-clock time and derive the overall codec speedup percentage.
The expected overall speedup is modest (20-40%) since the hot paths are only
a fraction of total codec work. The micro-benchmarks demonstrate the per-function
improvement.

## 8. Risks and Open Questions

### Risk 1: safe-Rust tension (medium)

The `wide` crate internally uses `unsafe` for its intrinsic wrappers. While
this `unsafe` is encapsulated and audited, some stakeholders may view adding
a dependency that contains `unsafe` as violating the spirit of the safe-Rust
rule. **Mitigation:** The rule specifically says `unsafe` is not permitted
in our code. Using a safe API provided by a dependency is the same pattern
as using `Vec::push` (which is safe, backed by `unsafe` allocation). The
`wide` crate is the allocation abstraction equivalent for SIMD.

**Decision needed:** Arthur should confirm that "safe Rust only" means "no
`unsafe` in our crate's source" rather than "no `unsafe` anywhere in the
dependency tree." If the latter, only Option D (auto-vectorization) is viable.

### Risk 2: mult16_32_q15 decomposition (medium)

The i32-only decomposed form of `mult16_32_q15` avoids i64 widening but must
produce bit-exact results. The mathematical equivalence is provable for
i16-range `a`, but implementation details (rounding direction of the intermediate
shift, signed vs unsigned masking of `b & 0x7FFF`) could introduce off-by-one
errors. **Mitigation:** Exhaustive testing of the decomposition (Section 6.3)
before using it in any SIMD path.

### Risk 3: Platform portability (low)

The `wide` crate falls back to scalar operations on platforms without SIMD
support (e.g., WASM, some embedded targets). The `simd` feature will still
compile on these platforms but will not provide speedup. This is acceptable --
the feature is opt-in and documented as x86_64-focused. **Mitigation:** Document
that `--features simd` is beneficial on x86_64 and aarch64 only.

### Risk 4: wide crate maintenance (low)

If the `wide` crate becomes unmaintained, we have two exit paths: (1) vendor
the relevant code into our crate (it is small), or (2) migrate to `std::simd`
when it stabilizes. **Mitigation:** Keep SIMD code isolated in `simd.rs` with
a clean interface boundary.

### Risk 5: AVX2 alignment requirements (low)

AVX2 loads from unaligned addresses incur a performance penalty on some
microarchitectures. Our FFT buffers are `Vec<KissFftCpx>` which are 8-byte
aligned (two i32 fields), not 32-byte aligned. **Mitigation:** Use unaligned
loads initially (still faster than scalar). Add aligned allocation in Phase 5
if benchmarks show it matters.

### Risk 6: Compile-time increase (low)

Adding `wide` as an optional dependency adds ~2-3 seconds to compile time when
the `simd` feature is enabled. This is negligible. The default build (without
`simd`) is completely unaffected.

### Open Question 1: ARM NEON

The `wide` crate supports NEON on aarch64. Should we test and validate on ARM
as part of this work, or defer to a follow-up? **Recommendation:** Defer. Get
x86_64 working and validated first. NEON support should "just work" via `wide`
but needs its own benchmark and test pass.

### Open Question 2: celt_pitch_xcorr restructuring for 8-wide

The current `celt_pitch_xcorr` processes 4 lags at a time via `xcorr_kernel`.
AVX2 could process 8 lags at a time, but this requires changing `xcorr_kernel`
to accept `sum: &mut [i32; 8]` and restructuring the caller's loop.
**Recommendation:** Start with 4-wide (matching current structure) and add
8-wide as a Phase 5 optimization if benchmarks justify it.

### Open Question 3: Auto-vectorization of existing scalar code

Before implementing explicit SIMD, we should measure what LLVM already
auto-vectorizes. Compile with `-C opt-level=3 -C target-cpu=native` and inspect
the generated assembly for `xcorr_kernel` and `compute_band_energies`. If LLVM
already vectorizes them well, the explicit SIMD work may have lower ROI than
expected. **Recommendation:** Run this measurement as the first task before
writing any SIMD code. It takes 30 minutes and could significantly change
the priority ordering.
