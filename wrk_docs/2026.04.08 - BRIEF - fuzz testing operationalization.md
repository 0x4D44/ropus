# Brief: Fuzz Testing Operationalization

Date: 2026-04-08

## Context

Five libFuzzer targets exist in `tests/fuzz/Cargo.toml`:
- `fuzz_decode`
- `fuzz_encode`
- `fuzz_roundtrip`
- `fuzz_repacketizer`
- `fuzz_packet_parse`

These are structurally present but not operationalized. No seed corpus, no crash regression corpus, no automation, no evidence of sustained runs or findings being fed back into the test suite.

The project recently added `unsafe` unchecked indexing in the FFT and MDCT hot paths (`src/celt/fft.rs`, `src/celt/mdct.rs`) gated behind the `unchecked-indexing` feature (on by default). This raises the stakes — a bounds violation in these paths is now UB, not a panic. Fuzz testing is the primary tool for catching these.

## Objective

Turn the existing fuzz targets into a functioning fuzz program that runs regularly, accumulates findings, and feeds results back into the regression suite.

## Scope

### 1. Seed corpus

Create `tests/fuzz/corpus/<target>/` directories with meaningful seed inputs:

- **`fuzz_decode`**: Valid Opus packets at various bitrates, bandwidths (NB/MB/WB/SWB/FB), frame sizes (2.5/5/10/20/40/60ms), mono and stereo. Extract from the integration test vectors using the harness encode command. Include SILK-only, CELT-only, and hybrid mode packets.
- **`fuzz_encode`**: Short PCM buffers at 8/12/16/24/48 kHz, mono/stereo, silence/sine/noise. Can reuse or derive from `tests/vectors/*.wav`.
- **`fuzz_roundtrip`**: Same PCM seeds as encode.
- **`fuzz_repacketizer`**: Valid multi-frame packets (code 1, 2, 3) plus single-frame packets. Generate from the harness.
- **`fuzz_packet_parse`**: Mix of valid packets and known edge cases (zero-length, max-length, all TOC byte values).

Target: ~50-100 seeds per target covering the configuration space.

### 2. Differential fuzzing

The most valuable fuzz mode for a parity port is differential: feed the same input to both C and Rust, compare outputs. The current targets appear to be crash-only ("must not panic").

Extend at minimum `fuzz_decode` and `fuzz_roundtrip` to:
1. Decode/encode with both C reference (via FFI) and Rust
2. Compare outputs byte-for-byte
3. Report any divergence as a finding (not just crashes)

This catches correctness bugs, not just safety bugs. It's the single highest-value fuzzing improvement.

### 3. Configuration coverage

Current targets appear to be 48 kHz mono. Extend to fuzz across:
- Sample rates: 8000, 12000, 16000, 24000, 48000
- Channels: 1, 2
- Applications: VOIP, AUDIO, RESTRICTED_LOWDELAY
- Bitrates: 6000-510000 range
- Complexity: 0-10

Use the fuzzer input bytes to select configuration (structured fuzzing), or create separate targets per major configuration axis.

### 4. Crash regression corpus

Create `tests/fuzz/crashes/<target>/` directories. Any crash or divergence found during fuzzing gets saved here permanently and added to the CI test suite as a regression test. This prevents regressions from re-introducing fixed bugs.

### 5. Automation

Create a script (e.g., `tools/fuzz_run.sh`) that:
- Runs each target for a configurable duration (default: 10 minutes per target)
- Uses the seed corpus
- Reports new findings
- Can be run manually or scheduled (nightly)

The goal is not CI-gated fuzzing (that's a larger infrastructure decision) but a one-command way to run a meaningful fuzz session.

## Acceptance criteria

- Seed corpus checked in with 50+ seeds per target
- At least `fuzz_decode` does differential comparison against C reference
- Configuration space coverage beyond 48 kHz mono
- `tools/fuzz_run.sh` runs all targets and reports findings
- Any crashes found are fixed and added to regression corpus

## Out of scope

- CI/CD integration (infrastructure decision for later)
- Coverage-guided corpus minimization (nice-to-have, not blocking)
- Custom mutators (standard libFuzzer mutations are sufficient initially)
