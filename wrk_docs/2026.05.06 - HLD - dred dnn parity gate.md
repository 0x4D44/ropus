# HLD - DRED/DNN Parity Gate

Date: 2026.05.06

Scope: Phase 3.1 testing-improvement slice for DRED/DNN parity and release-preflight policy. This is a design document only; no production or test code is changed by this HLD.

## Problem Statement

The current test suite can report green while DNN/DRED coverage is weaker than the result implies.

There are three separate issues:

- The `ropus` build keeps bare checkouts buildable by emitting an empty embedded DNN weight blob when `reference/dnn/*_data.c` assets are absent.
- `full-test --release-preflight` currently treats `dnn_base_weights`, `dred_rdovae_weights`, and `float_deep_plc_assets` as optional/report-only assets.
- The strongest DRED payload-byte parity test exists but is ignored because known LPCNet feature-extractor drift blocks byte-exact payload parity.

The goal for this slice is not to fix DRED itself. The goal is to make the validation contract honest and automatable: a full release-preflight run should not imply neural/DRED confidence when the assets were absent or the strongest DRED parity lane is still ignored.

## Users And Stakeholders

- Arthur, as release and testing steward, needs "green" to carry a precise meaning.
- Codec implementers need a small gate that separates real DNN/DRED regressions from known DRED payload parity debt.
- Agent teams need a clear next implementation slice that does not accidentally start broad neural codec work.
- Downstream users care that DRED/DNN claims are not inferred from tests that silently skipped.

## Current Evidence And Gaps

`/home/md/language/ropus/ropus/build.rs` intentionally writes an empty `weights_blob.bin` when DNN reference assets are absent. That is good fresh-clone ergonomics, but it means default `cargo build` cannot prove neural coverage.

`/home/md/language/ropus/full-test/src/preflight.rs` probes:

- `dnn_base_weights`
- `dred_rdovae_weights`
- `float_deep_plc_assets`

but currently marks them optional and report-only. The unit test `optional_dnn_and_dred_assets_never_banner_block` locks that behavior in.

`/home/md/language/ropus/full-test/src/tests.rs` has a `QuickReleasePreflight` Stage 2 profile that excludes `ropus-harness-deep-plc`. That is acceptable for quick smoke, but it means quick preflight must explicitly not claim neural/DRED gate coverage.

`/home/md/language/ropus/harness-deep-plc/tests/tier2_snr.rs` contains active DNN PLC SNR gates:

- `dnn_plc_tier2_snr_above_50db`
- `dnn_plc_tier2_lossless_regression`

Those are useful neural PLC gates when the assets are present.

`/home/md/language/ropus/harness-deep-plc/tests/dred_integrated_encode.rs` has active DRED format-level cross-parse tests, but they return early when `WEIGHTS_BLOB` is empty:

- Rust encoder packets parse on the C reference.
- C encoder packets parse with the Rust decoder.
- Rust encoder and Rust DRED decoder round-trip through parse/process.

`/home/md/language/ropus/harness-deep-plc/tests/dred_encode_payload_diff.rs` is the intended byte-exact DRED payload gate, but it is ignored with a named blocker: Stage 7 LPCNet `Ex[]`/FFT-path bit-exactness drift. This should stay visible as exact-byte-parity debt until that blocker is actually resolved.

`/home/md/language/ropus/ropus/src/opus/dred.rs` still returns `OPUS_UNIMPLEMENTED` for DRED PCM reconstruction. Existing tests pin that stub behavior, so this slice must not claim DRED recovered PCM coverage.

`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_dnn_blob.rs` is a safety fuzz target for `OpusDecoder::set_dnn_blob`. It is useful parser/safety coverage, not DNN parity coverage.

`/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md` currently marks DNN PLC as optional-report and DRED payload-byte differential tests as ignored/debt. That is directionally honest, but the release-preflight policy does not yet enforce the distinction.

Current code, help text, and reports describe the release-preflight asset profile as C1/core. That terminology is too narrow once neural/DRED assets become banner-blocking. The implementation must expose profile/claim wording that matches the actual policy.

## Proposed Testing Slice

### 1. Define Release-Preflight Neural/DRED Policy

Keep default builds and default `full-test` tolerant of missing DNN/DRED assets.

Expose three distinct preflight claim profiles in CLI help, stdout/HTML/JSON report wording, and any preflight profile field:

| CLI mode | Claim wording | DNN/DRED asset policy |
|---|---|---|
| Default, no `--release-preflight` | `default-report-only` | DNN/DRED assets are reported but do not affect the banner. |
| `--quick --release-preflight` | `release-core-smoke-no-neural-claim` | Core smoke profile. DNN/DRED assets are reported but do not affect the banner because quick Stage 2 excludes `ropus-harness-deep-plc`. |
| Non-quick `--release-preflight` | `release-core-plus-neural-dred-gate` | Core plus neural/DRED gate. Missing DNN/DRED assets are banner-blocking. |

Do not leave the UI saying only `core` or `C1/core profile` when neural/DRED assets are banner-blocking. The report must make clear whether the run claims neural/DRED gate coverage.

For `--quick --release-preflight`, keep DNN/DRED assets report-only because the quick Stage 2 profile excludes `ropus-harness-deep-plc`. The report should make this explicit: quick preflight is a core smoke profile and does not claim neural/DRED gate coverage.

For non-quick `--release-preflight`, promote these assets to banner-blocking:

- `dnn_base_weights`
- `dred_rdovae_weights`
- `float_deep_plc_assets`

Rationale: non-quick release-preflight includes `ropus-harness-deep-plc`, so it is the first local profile that can plausibly claim neural PLC and DRED format coverage. If the weights are missing, several DRED tests can early-return and the claim is false.

Implementation contract: `/home/md/language/ropus/full-test/src/preflight.rs` must not receive only a `release_preflight: bool`. It should receive either the full `/home/md/language/ropus/full-test/src/cli.rs` `Options` or a small preflight policy enum derived from `Options`, for example:

```rust
enum PreflightPolicy {
    DefaultReportOnly,
    ReleaseCoreSmokeNoNeuralClaim,
    ReleaseCorePlusNeuralDredGate,
}
```

Prefer the enum if it keeps `preflight.rs` decoupled from CLI parsing. Derive it in one place from the same `Options` values used for Stage 2 profile selection so quick/non-quick preflight policy cannot drift from `/home/md/language/ropus/full-test/src/tests.rs` `Stage2Profile`.

### 2. Keep Existing Active Neural/DRED Tests As The First Gate

Do not unignore `/home/md/language/ropus/harness-deep-plc/tests/dred_encode_payload_diff.rs` in this slice.

The active gate should be:

- DNN PLC SNR tier-2 tests from `tier2_snr.rs`.
- DRED integrated encode/parse/process tests from `dred_integrated_encode.rs`.
- DRED RDOVAE encode/decode differential tests where their current tiered oracles already pass with weights present.
- Existing synthetic exact payload decode test in `dred_decode_payload_diff.rs`.

This gives a real full-preflight neural/DRED signal without pretending full DRED payload byte parity or DRED PCM reconstruction is done.

### 3. Add A Small Asset-Policy Test Surface

In `full-test`, add focused unit tests for the new policy:

- Default/non-release preflight reports missing DNN/DRED assets but does not banner-block.
- Quick release-preflight reports missing DNN/DRED assets but does not banner-block and records that neural/DRED gates are not claimed.
- Non-quick release-preflight banner-blocks missing DNN/DRED assets.
- Present DNN and DRED reference files clear the non-quick release-preflight blocker.
- The profile/claim string changes with default, quick release-preflight, and non-quick release-preflight modes.

Avoid tests that require actual model execution in `full-test`; those belong in `ropus-harness-deep-plc`.

All asset-policy tests must simulate files in tempdirs. They must not depend on actual local `/home/md/language/ropus/reference/dnn` content, because Arthur's checkout may have weights while CI or another agent's checkout may not.

### 4. Update Oracle Matrix Wording

Update `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md` so it distinguishes:

- DNN PLC SNR tests: `snr-gated-parity`, required under non-quick release-preflight neural profile.
- DRED integrated parse/process tests: format-level cross-parse/process gate, not byte-exact payload parity.
- DRED payload-byte differential test: still `ignored` exact-byte-parity debt due known LPCNet feature drift.
- DRED PCM reconstruction: explicitly unimplemented and not asserted.

## Validation Commands

Fast policy/unit validation:

```bash
timeout 120 cargo test -p full-test preflight
timeout 120 cargo test -p full-test cli::tests
timeout 120 cargo test -p full-test banner
timeout 120 cargo test -p full-test report
timeout 120 cargo test -p full-test html
timeout 120 cargo fmt --check
```

Neural/DRED scoped validation with assets present:

```bash
timeout 240 cargo test -p ropus-harness-deep-plc dnn_plc_tier2
timeout 240 cargo test -p ropus-harness-deep-plc dred_integrated_encode
timeout 240 cargo test -p ropus-harness-deep-plc dred_rdovae
timeout 240 cargo test -p ropus-harness-deep-plc dred_ec_decode_matches_c_on_c_emitted_payload
```

Release-preflight smoke validation:

```bash
timeout 300 cargo run -p full-test -- --release-preflight --quick --skip-coverage --skip-benchmarks --skip-ambisonics
timeout 600 cargo run -p full-test -- --release-preflight --skip-coverage --skip-benchmarks --skip-ambisonics
```

Expected behavior:

- The quick command should not banner-block missing neural/DRED assets, but should report that neural/DRED gates are not claimed in the quick profile.
- The non-quick command should fail preflight when DNN/DRED assets are missing.
- With assets present, the non-quick command should proceed to the active `ropus-harness-deep-plc` tests.

## Risks

- Making DNN/DRED assets banner-blocking for non-quick release-preflight may surprise anyone who used that command as a core-only gate. The quick profile remains available for core smoke.
- Preflight probes source files under `reference/dnn`, while model execution uses the compiled embedded blob in Cargo `OUT_DIR`. Cargo rerun triggers should rebuild when those files change, but stale local build artifacts are still worth watching during implementation validation.
- The DRED integrated tests are format and parse/process gates, not byte-exact payload parity. The oracle matrix must not overstate them.
- The ignored byte-exact DRED payload test may stay ignored for a while. This slice improves honesty and asset enforcement; it does not close the underlying LPCNet feature drift.
- If full non-quick release-preflight is too expensive for routine local use, the implementation should document that cost rather than weakening the policy silently.

## Non-Goals

- Do not implement DRED PCM reconstruction.
- Do not fix LPCNet feature-extractor or FFT-path drift.
- Do not unignore `dred_encode_silk_frame_bytes_match_c_reference` unless a separate implementation pass proves it now passes.
- Do not make default `cargo build` or default `cargo test` fail on missing DNN/DRED weights.
- Do not start fuzz campaigns or add new corpora.
- Do not add a new CI framework or GitHub workflow.
- Do not change codec behavior.
- Do not touch `harness-deep-plc` DRED test code in this slice.

## Implementation Signoff Conditions

Before implementation starts, supervisor signoff should confirm:

- The three profile/claim labels above are acceptable, or provide exact replacement wording.
- `preflight::capture` will receive full `Options` or a derived `PreflightPolicy`, not a boolean.
- Unit tests will use tempdirs for all asset-presence combinations and will not read the real `reference/dnn` tree.
- The implementation scope is limited to `full-test` policy/reporting plus `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`.
- No codec changes, no DRED PCM reconstruction, and no unignoring exact DRED payload parity are included in this slice.

## Design Decisions Needing Supervisor Signoff

1. Treat non-quick `--release-preflight` as the local profile that claims neural/DRED gate coverage, and therefore make DNN/DRED assets banner-blocking there.
2. Keep `--quick --release-preflight` as core smoke only, with explicit report wording that neural/DRED gates are not claimed.
3. Keep the exact DRED payload-byte test ignored in this slice and represent it as named exact-byte-parity debt rather than attempting to force a brittle gate.
4. Use existing `ropus-harness-deep-plc` active tests as the first neural/DRED gate instead of inventing a new harness command.
