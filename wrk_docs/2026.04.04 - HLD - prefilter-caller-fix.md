# HLD: Fix Prefilter Caller-Side Code Differences (C vs Rust)

**Document**: HLD for prefilter-caller discrepancies  
**Date**: 2026-04-04  
**Status**: Ready for implementation  
**Regression**: 48kHz CELT sine test failure

---

## 1. Problem Statement

The un_prefilter function itself is verified correct (C and Rust match), but the caller code around un_prefilter in the Rust encoder has **four critical differences** from the C reference:

1. **Missing analysis parameter**: C passes &st->analysis to un_prefilter, Rust does not
2. **Missing pitch_change condition**: The C condition checks (!st->analysis.valid || st->analysis.tonality > .3) but Rust omits this
3. **Wrong ICDF precision bits**: When encoding prefilter tapset, C uses 2 bits (tb=2) but Rust uses 4 bits (tb=4)
4. **Unconditional tapset encoding**: C always encodes tapset in prefilter, but Rust has a complexity gate (if st.complexity >= 2)

These differences cause:
- Incorrect pitch_change flag computation → wrong decisions about pitch changes
- Misaligned bitstream encoding of prefilter parameters → decoder/encoder mismatch
- Missing or gated analysis influence on prefilter behavior

### Bug #2: Missing pitch_change Condition - Analysis Tonality Check

**C Code (celt_encoder.c, lines 2055-2057)**
`c
if ((gain1 > QCONST16(.4f,15) || st->prefilter_gain > QCONST16(.4f,15)) 
    && (!st->analysis.valid || st->analysis.tonality > .3)    // <-- REQUIRED
    && (pitch_index > 1.26*st->prefilter_period || pitch_index < .79*st->prefilter_period))
   pitch_change = 1;
`

**Rust Code (encoder.rs, lines 2018-2025)**
`ust
pitch_change = if (gain1 > qconst16(0.4, 15) || st.prefilter_gain > qconst16(0.4, 15))
    && (pitch_index as f64 > 1.26 * st.prefilter_period as f64
        || (pitch_index as f64) < 0.79 * st.prefilter_period as f64)
    // MISSING: (!st->analysis.valid || st->analysis.tonality > .3)
{
    1
} else {
    0
};
`

**Impact**: The pitch_change flag is used to determine if significant pitch changes occurred. This affects timing decisions and filter updates. Rust incorrectly triggers pitch_change without checking if the signal is tonal enough.

### Bug #3: Wrong ICDF Precision for Prefilter Tapset Encoding

**C Code (celt_encoder.c, line 2073)**
`c
ec_enc_icdf(enc, prefilter_tapset, tapset_icdf, 2);  // ftb=2 (4 levels)
`

**Rust Code (encoder.rs, lines 2043-2045)**
`ust
if st.complexity >= 2 {
    enc_ref.encode_icdf(prefilter_tapset as u32, &TAPSET_ICDF, 4);  // ftb=4 (16 levels)
}
`

**Analysis**: 
- TAPSET_ICDF = [2, 1, 0] (3 symbols)
- C uses tb=2 → range is 2^2 = 4 (correct for 3 symbols)
- Rust uses tb=4 → range is 2^4 = 16 (incorrect, wastes precision)

**Impact**: The encoder writes the tapset symbol with wrong precision, causing bitstream mismatch. The decoder will misinterpret the encoded value.

---

### Bug #4: Unconditional tapset Encoding vs Complexity Gated

**C Code (celt_encoder.c, line 2073)**
`c
// No complexity check - ALWAYS encode tapset
ec_enc_icdf(enc, prefilter_tapset, tapset_icdf, 2);
`

**Rust Code (encoder.rs, lines 2043-2045)**
`ust
// Complexity gate - only encode if complexity >= 2
if st.complexity >= 2 {
    enc_ref.encode_icdf(prefilter_tapset as u32, &TAPSET_ICDF, 4);
}
`

**Impact**: For low complexity modes (0, 1), Rust does not encode the tapset but C does. This causes bitstream divergence for all non-test scenarios (test typically uses default complexity=5).
