# HLD: Fuzz coverage expansion — closing the 10 gaps

**Date:** 2026-04-30
**Author:** Claude (with Arthur)
**Scope target:** `tests/fuzz/fuzz_targets/*.rs`, `tests/fuzz/Cargo.toml`,
`tests/fuzz/fuzz_targets/c_reference.rs`, `tests/fuzz/corpus/<target>/`,
plus one small helper module under `tests/fuzz/fuzz_targets/`.

## Context

The 8-hour overnight fuzz campaign on 2026-04-29 ran ~24.4 B executions
across 9 targets and found 0 crashes / 0 panics
(`wrk_journals/2026.04.29 - JRN - overnight-fuzz-campaign.md`).
A follow-up audit catalogued 10 substantive coverage gaps. The two largest:
**every encode target hardcodes CBR**, and **multistream / DNN / FEC / DTX
have zero fuzz coverage**. This HLD designs the smallest set of changes
that closes all 10 in a sandbox environment with no migration constraints.

## Goal

Bring the fuzz suite into honest alignment with the codec's production
configuration space:

1. VBR — currently every target forces `set_vbr(0)`.
2. Multistream encode/decode/roundtrip — zero coverage today.
3. FEC, DTX, packet-loss-perc — never enabled in any fuzz target.
4. Encoder runtime setters (new in 0.12.0) — validation logic unfuzzed.
5. Repacketizer beyond a single `cat` + `out`.
6. SILK / Hybrid decode oracle — currently panic-only (PCM compare skipped).
7. DNN weight-blob ingestion — `set_dnn_blob(&[u8])` is an unfuzzed parser.
8. Saturated targets — `fuzz_packet_parse` (cov 61) and
   `fuzz_encode_multiframe` (cov 36) are CPU sinks.
9. Mutator strategy — every target uses raw-byte modulo extraction;
   no seed corpus of pathological PCM patterns.
10. Hybrid-mode SILK↔CELT switchover — emerges naturally once gap 1
    (VBR) is closed; no separate target needed.

## Non-goals

- No CI integration. Campaigns remain manual via
  `tools/overnight_fuzz_launch.sh` (already shipped).
- No coverage-guided corpus-minimisation tooling.
- No custom libFuzzer mutators. We use `Arbitrary` derive where structured
  inputs help, raw bytes elsewhere.
- No fuzz target for analysis, soft-clip, or projection encoders. They are
  callable from outside the fuzz harness; if a future bug points at them
  we add a target then.
- No cross-target deduplication of the panic-capture boilerplate (lifted
  from an existing helper) — see "Deliberately excluded" below.
- No new env vars. The existing `FUZZ_PANIC_CAPTURE_DIR` (debug-helper for
  Windows, set by `overnight_fuzz_launch.sh`) is the only one.
- No feature flags. New behaviour is unconditional; saturated targets are
  deleted, not hidden behind a cfg.

## Design

The work decomposes into four streams. Each stream is independently
mergeable and independently runnable in a fuzz campaign.

### Stream A — Extend existing encode targets in place (gaps 1, 3, 4, 10)

`fuzz_encode.rs`, `fuzz_encode_safety.rs`, `fuzz_roundtrip.rs`,
`fuzz_roundtrip_safety.rs` all share the same structured-input prologue:

```rust
sample_rate    = SAMPLE_RATES[data[0] % 5]
channels       = (data[1] & 1) ? 2 : 1
application    = APPLICATIONS[data[2] % 3]
bitrate        = 6000 + u16::from_le_bytes([d[3],d[4]]) % 504_001
complexity     = data[5] % 11
pcm_bytes      = &data[6..]
```

Extend the prologue to consume two more config bytes:

```rust
// new — total config = 8 bytes
vbr            = (data[6] & 0b0001) != 0
vbr_constraint = (data[6] & 0b0010) != 0
inband_fec     = data[6] & 0b1100 >> 2  // 0=Off, 1=On, 2=Forced, 3=On
dtx            = (data[7] & 0b0001) != 0
loss_perc      = (data[7] >> 1) % 101    // 0..=100
pcm_bytes      = &data[8..]
```

Then after constructing the encoder, drop the unconditional `set_vbr(0)`
and replace with:

```rust
enc.set_vbr(if vbr { 1 } else { 0 });
enc.set_vbr_constraint(if vbr_constraint { 1 } else { 0 });
enc.set_inband_fec(inband_fec);
enc.set_dtx(if dtx { 1 } else { 0 });
enc.set_packet_loss_perc(loss_perc);
```

This closes gaps 1 (VBR), 3 (FEC/DTX/loss-perc), and 10 (the hybrid-mode
switchover that VBR + bitrate randomization naturally exercises).

#### CBR-only assertions become VBR-aware

The existing CBR-determinism block in `fuzz_encode.rs:163` and
`fuzz_roundtrip.rs` only holds when `vbr == false`. Wrap it:

```rust
if !vbr {
    // existing determinism check: re-encode with same config, expect
    // identical bytes
}
```

VBR is by spec non-deterministic across encoder instances (analysis state
diverges); we drop the determinism oracle on the VBR path and rely on the
differential-vs-C oracle, which still holds byte-for-byte.

#### Random runtime-setter sequence (gap 4)

After construction, before the first `encode()` call, replay a short
deterministic sequence of setter calls driven by the input bytes:

```rust
fn apply_setter_sequence(enc: &mut OpusEncoder, bytes: &[u8]) {
    for chunk in bytes.chunks_exact(2) {
        match chunk[0] % 11 {
            0  => { enc.set_complexity((chunk[1] % 11) as i32); }
            1  => { enc.set_signal(SIGNAL_VALUES[chunk[1] as usize % 3]); }
            2  => { enc.set_force_channels(FORCE_CH[chunk[1] as usize % 3]); }
            3  => { enc.set_max_bandwidth(BW[chunk[1] as usize % 5]); }
            4  => { enc.set_packet_loss_perc((chunk[1] % 101) as i32); }
            5  => { enc.set_inband_fec((chunk[1] % 3) as i32); }
            6  => { enc.set_dtx(((chunk[1] & 1) != 0) as i32); }
            7  => { enc.set_lsb_depth(8 + (chunk[1] % 17) as i32); }
            8  => { enc.set_phase_inversion_disabled((chunk[1] & 1) as i32); }
            9  => { enc.set_prediction_disabled((chunk[1] & 1) as i32); }
            10 => { enc.set_bitrate(byte_to_bitrate(chunk[1], chunk[1])); }
            _  => unreachable!(),
        }
    }
}
```

Capped at 16 setter calls (32 bytes) so PCM payload still has room in
the typical libFuzzer input length.

This is **safety-target-only**. The differential targets (`fuzz_encode`,
`fuzz_roundtrip`) keep a fixed configuration so the C-vs-Rust comparison
remains apples-to-apples. We're testing two different things:

- Differential targets: "given identical config, do Rust and C produce
  identical output?"
- Safety targets: "given any config sequence, does Rust panic?"

Mixing the runtime-setter shuffle into the differential targets would
require teaching `c_reference::c_encode` the same shuffle, which doubles
the surface for marginal gain. Safety-target-only keeps the change
minimal.

### Stream B — Retire saturated, rebuild slim (gap 8)

#### Retire `fuzz_packet_parse`

The 5 packet-query functions (`get_bandwidth`, `get_nb_channels`,
`get_nb_frames`, `get_samples_per_frame`, `get_nb_samples`) are already
exercised in `fuzz_decode.rs:147` (parse-validation block) on every
encoded packet. Standalone target adds no coverage; deleting it
reclaims 11% of campaign CPU.

Action: delete `tests/fuzz/fuzz_targets/fuzz_packet_parse.rs`, remove
its `[[bin]]` entry from `tests/fuzz/Cargo.toml`, drop its corpus dir
from `.gitignore` follow-up tracking.

#### Rebuild `fuzz_encode_multiframe` around `Arbitrary`

Current target's structured-input scheme ran 5.7 B execs and added
2 new corpus entries — its mutator can't break out of its own input
shape. Replace with a derived `Arbitrary` impl:

```rust
#[derive(Debug, arbitrary::Arbitrary)]
struct MultiframeInput {
    sample_rate_idx: u8,    // index into [8000,12000,16000,24000,48000]
    channels_minus_one: u8, // 0..=1 -> 1..=2 channels
    application_idx: u8,    // 0..=2 -> {VOIP, AUDIO, LOWDELAY}
    frames: Vec<FrameConfig>, // libfuzzer caps at ~16 KB; bound to 16 frames
}

#[derive(Debug, arbitrary::Arbitrary)]
struct FrameConfig {
    bitrate_raw: u16,
    complexity: u8,
    vbr: bool,
    fec: u8,
    dtx: bool,
    loss_perc: u8,
    pcm: Vec<i16>,  // 20ms worth, capped via take_rest pattern
}
```

Each `FrameConfig` is applied via setter calls before its `encode()`
invocation, so the campaign exercises *intra-stream* config drift
(adaptive bitrate, FEC toggle on push-to-talk) — the realistic shape
that current modulo-byte structuring can't reach.

Bound `frames.len()` to `[2, 16]` in a manual `Arbitrary` impl
(libfuzzer-sys's derive lets a `Vec` grow to fill the input) to keep
each iteration within ~200 ms.

### Stream C — Three new targets (gaps 2, 5, 7)

#### `fuzz_multistream` (gap 2)

Combines encode + decode + roundtrip in one target, switched on a
config byte. Restricted to `OpusMSEncoder::new_surround(channels,
mapping_family)` rather than arbitrary stream/coupled/mapping
combinations — arbitrary mappings hit validation, not codec, and
saturate the validator instead of the codec.

```rust
#[derive(arbitrary::Arbitrary)]
struct MSInput {
    op: u8,                    // 0=encode, 1=decode, 2=roundtrip
    sample_rate_idx: u8,       // [8000,12000,16000,24000,48000]
    channels: u8,              // 1..=8 (clamped)
    mapping_family: u8,        // 0, 1, 2, 255 (clamped)
    application_idx: u8,
    bitrate_raw: u16,
    complexity: u8,
    vbr: bool,
    payload: Vec<u8>,          // PCM (encode/roundtrip) or packet (decode)
}
```

Differential against C via three new FFI symbols added to
`tests/fuzz/fuzz_targets/c_reference.rs`:

```rust
extern "C" {
    fn opus_multistream_surround_encoder_create(...) -> *mut OpusMSEncoder;
    fn opus_multistream_encode(...) -> i32;
    fn opus_multistream_decoder_create(...) -> *mut OpusMSDecoder;
    fn opus_multistream_decode(...) -> i32;
    fn opus_multistream_encoder_destroy(...);
    fn opus_multistream_decoder_destroy(...);
}
```

The C symbols are already linked (`tests/fuzz/build.rs:175-177`
compiles `opus_multistream*.c`); only the Rust extern declarations
need adding. Mirror the pattern of the existing mono symbols at
`c_reference.rs:68-86`.

Oracle: byte-exact compare of compressed output for encode/roundtrip;
PCM compare for decode (with the same SILK-skip carve-out as
`fuzz_decode`, replaced in Stream D).

#### `fuzz_repacketizer_seq` (gap 5)

Replaces and extends `fuzz_repacketizer.rs`. Reads N (1..=8) packets
from the input bytes, calls `cat()` for each, then exercises both
`out()` and `out_range(begin, end)` on the accumulated state. Compares
against C reference repacketizer step-by-step.

```rust
#[derive(arbitrary::Arbitrary)]
struct RepacketizerSeq {
    packets: Vec<Vec<u8>>,    // bounded to 1..=8 packets
    out_range_begin: u8,      // 0..=nb_frames
    out_range_end: u8,        // begin+1..=nb_frames
}
```

The existing `fuzz_repacketizer.rs` (current cat+out only) is replaced,
not augmented — its test surface is a strict subset of the new target
and the corpus is regenerated naturally.

#### `fuzz_dnn_blob` (gap 7)

Pure safety target. `OpusDecoder::set_dnn_blob(&[u8])` is a parser
surface that takes arbitrary bytes and is currently unfuzzed.

```rust
fuzz_target!(|data: &[u8]| {
    let mut dec = OpusDecoder::new(48000, 2).expect("valid sr/ch");
    let _ = dec.set_dnn_blob(data); // must not panic, OOB-read, or hang
    // Optional: try decoding a single small valid packet afterwards to
    // exercise the post-blob state.
});
```

No differential — the C reference does not expose `set_dnn_blob` with
the same semantics; it's a Rust-side blob format. Panic-free is the
only oracle.

DRED encoder coverage (`set_dred_duration`) is **not** added in this
HLD: the encoder lazily allocates DRED state and consumes the same
embedded weights, so once the decoder blob target is solid, an encoder
counterpart is a 30-line follow-up.

### Stream D — Better oracles and seeds (gaps 6, 9)

#### SILK / Hybrid decode SNR oracle (gap 6)

`fuzz_decode.rs` and `fuzz_roundtrip.rs` currently skip PCM compare
when the encoded packet is SILK or Hybrid. Replace the skip with a
bounded-SNR check, matching the project's tier-2 envelope (≥50 dB
documented in `CLAUDE.md`).

```rust
const SILK_DECODE_MIN_SNR_DB: f64 = 50.0;

fn snr_db(reference: &[i16], test: &[i16]) -> f64 {
    if reference.len() != test.len() { return f64::NEG_INFINITY; }
    let (mut sig, mut noise) = (0.0f64, 0.0f64);
    for (&r, &t) in reference.iter().zip(test) {
        sig   += (r as f64).powi(2);
        noise += ((r as f64) - (t as f64)).powi(2);
    }
    if noise == 0.0 { return f64::INFINITY; }
    if sig   == 0.0 { return f64::NEG_INFINITY; } // pure silence with non-zero error = bug
    10.0 * (sig / noise).log10()
}
```

For SILK / Hybrid packets, replace the skip with:

```rust
let snr = snr_db(&c_pcm, &rust_pcm);
assert!(
    snr >= SILK_DECODE_MIN_SNR_DB,
    "SILK decode SNR {snr:.2} dB < {SILK_DECODE_MIN_SNR_DB:.0} dB \
     (sr={sample_rate}, ch={channels}, packet_len={pkt_len})"
);
```

This converts a panic-only oracle into a correctness oracle for the
~30% of decode inputs that route through SILK. CELT-only packets
remain byte-exact compared (no regression on the existing oracle).

The helper lives in a new `tests/fuzz/fuzz_targets/oracle.rs`
included via `#[path = "oracle.rs"] mod oracle;` in each target that
needs it — same pattern as `c_reference.rs` already uses.

#### Pathological seed corpus (gap 9)

A one-shot script generates ~10 PCM seeds per encode target covering
known-pathological signals:

- DC offset (constant max +i16, constant min, constant 0)
- Max-amplitude sine at 100 Hz, 1 kHz, 8 kHz, 22 kHz (Nyquist)
- White noise at full scale
- Step click (silence → max → silence)
- Periodic impulse train (pitch-detector stress)
- Near-silence with single-LSB toggling (quantizer stress)

Implemented as `tools/gen_fuzz_seeds.sh` (bash, ~50 lines) that emits
binary files into `tests/fuzz/corpus/<target>/` matching the existing
per-target structured-input layout (8 config bytes + PCM). Run once,
checked in as part of the Stream D PR; corpus dir is gitignored
*directories* but the seed files we add are explicitly added.

Two lines of `.gitignore` adjustment: switch `/tests/fuzz/corpus/`
to `/tests/fuzz/corpus/*/!.seeds/` (or simpler: track a flat
`tests/fuzz/seeds/<target>/` and have the launcher copy seeds into
`corpus/` at start). The latter is simpler and avoids gitignore
gymnastics — implement that.

## Acceptance criteria

A subsequent overnight campaign run after all four streams land must:

1. Build cleanly (`cargo +nightly fuzz build --fuzz-dir tests/fuzz`).
2. Show all 10 targets running and producing libFuzzer pulse output.
3. Observe non-CBR encode paths in coverage (verify by checking
   `cov:` line in `fuzz_encode/run.log` exceeds the 8020 ceiling
   from the 2026-04-29 run; expected: ≥9000 with VBR enabled).
4. Observe SILK-decode SNR assertions firing as oracles (verify by
   running one packet through with a deliberately-broken Rust SILK
   decoder; the assert must trigger, then revert).
5. Run for 8 h with 0 crashes / 0 panics, OR if crashes are found,
   minimise them and produce regression entries — that is the
   intended outcome of this work.

## Deliberately excluded

The following extensions are **not** in this HLD; adding them later
requires a separate HLD-level review. YAGNI per CLAUDE.md.

- **Custom libFuzzer mutators.** Standard mutations + structured
  `Arbitrary` cover the surface. A custom Opus-aware mutator (TOC byte
  preservation, valid-frame-count maintenance) is a research project.
- **Coverage-guided corpus minimisation.** `cargo fuzz cmin` exists;
  we don't run it. Corpus growth on a 9-target run is ~1.5 GB; that's
  on disk, not committed, and libFuzzer happily reuses it.
- **Soft-clip / projection encoder targets.** No reported bugs; no
  evidence the surface is at risk; YAGNI.
- **DRED encoder fuzz target.** Follow-up after `fuzz_dnn_blob` proves
  the embedded-weights plumbing.
- **Tonality analysis target.** Encoder-internal; reachable through
  encode targets transitively. No standalone target unless a bug
  points there.
- **Cross-target panic-capture deduplication.** The 35-line panic-hook
  block at the head of every target is duplicated. Refactoring it
  into a shared module is tempting; it is also a churn change with
  no fuzz value, so we leave it. If a future change touches all
  targets for a different reason, fold the dedup in as a drive-by.
- **CI-gated fuzzing.** Out of scope per the original BRIEF
  (`2026.04.08 - BRIEF - fuzz testing operationalization.md`). The
  manual launcher remains the entrypoint.

## Implementation order

Each stream is a separate PR. Streams are independently mergeable
and provide standalone value.

1. **Stream A** (smallest delta, biggest coverage win): VBR + FEC +
   DTX in encode-path targets, runtime-setter shuffle in safety
   targets. ~150 lines diff across 4 targets. **First**, because
   gap 1 (VBR) is the single largest uncovered surface and the
   change is mechanical.
2. **Stream B**: retire `fuzz_packet_parse`, rebuild
   `fuzz_encode_multiframe` with `Arbitrary`. ~250-line net change
   (-124 + ~370). **Second**, because it unlocks CPU for Stream C
   targets.
3. **Stream C**: three new targets (multistream, repacketizer_seq,
   dnn_blob). Each ~150-250 lines. Can ship as one PR or three.
   **Third**, because the multistream FFI extension needs care and
   shouldn't be rushed alongside other changes.
4. **Stream D**: SILK SNR oracle + pathological seeds. ~80 lines
   (oracle helper) + ~50 lines (seed generator script) + seed
   binaries. **Fourth**, because the SNR oracle is most useful once
   gap 1 is closed (more SILK packets exercised = more SNR samples).

Recommend overnight campaigns *between* streams to validate each
landing. ~32 h of fuzz time across 4 PRs is cheap insurance against
a late-stage regression that bisecting becomes painful for.

## Open questions for Arthur

1. **Is byte-exact compare the right oracle for VBR?** libopus' VBR
   path is bit-deterministic *for a fixed encoder instance*. Two
   instances seeded identically and fed identical input *should*
   produce identical output, since analysis state is initialised
   from constants. If we observe non-determinism at the
   differential-vs-C boundary in VBR mode, we relax to a
   length-only or SNR oracle on VBR packets. Worth tracking on the
   first overnight run after Stream A.

2. **Multistream channel count cap.** `new_surround` accepts up to
   255 channels. Anything above 8 produces validator-level
   rejection paths but minimal codec exercise. Cap at 8 for fuzz
   purposes? (Recommend yes.)

3. **Seed checking-in policy.** Pathological seeds are ~10 KB each ×
   ~10 patterns × 4 encode targets = ~400 KB. Small enough to commit
   but conflicts with the existing "no multi-megabyte corpora" hint
   in the README. Treat them as fixtures, not corpus, and put them
   under `tests/fuzz/seeds/` (committed). The launcher copies them
   into `corpus/<target>/` at run start.
