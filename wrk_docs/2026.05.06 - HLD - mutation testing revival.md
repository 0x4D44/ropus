# Mutation Testing Revival HLD

Date: 2026.05.06

Status: draft for supervisor signoff

Scope: final remaining testing-infrastructure slice from the 2026.05.05 testing
improvement plan, limited to reviving mutation testing as a periodic audit lane.

## Problem Statement

Mutation testing is currently not a reliable signal in this repository. The
existing runner still assumes top-level `src/...` paths, but the codec crate now
lives under `/home/md/language/ropus/ropus/src/...`. The cargo-mutants config
also carries hidden baseline skips, which makes any mutation score hard to
interpret without reading local config first.

The goal is not to create another release gate. The goal is to make an
occasional mutation audit reproducible and honest enough to answer one narrow
question:

> Which already-tested code paths have tests that are too weak to kill simple
> behavioral mutations?

That is a test-quality audit, not a codec-correctness oracle.

## Stakeholders

- Arthur and maintainers: need the original testing-improvement plan closed
  without adding misleading green status.
- Codec contributors: need actionable survivor reports that point to weak or
  missing tests, not noise from stale paths or hidden skips.
- Reviewers: need to see whether a mutation result was strict, skip-degraded,
  unit-only, or unsupported.
- Future release users: benefit indirectly from better tests, but should not
  wait on mutation audits as a release prerequisite.

## Proposed Small Slice

Revive mutation testing as a scriptable, periodic, unit-test sensitivity audit
for the `ropus` crate only.

### 1. Correct target discovery and path resolution

Update `/home/md/language/ropus/tools/mutation_test.sh` so every default module
uses current paths:

- `/home/md/language/ropus/ropus/src/silk/common.rs`
- `/home/md/language/ropus/ropus/src/opus/decoder.rs`
- `/home/md/language/ropus/ropus/src/opus/encoder.rs`
- `/home/md/language/ropus/ropus/src/silk/decoder.rs`
- `/home/md/language/ropus/ropus/src/celt/decoder.rs`
- `/home/md/language/ropus/ropus/src/celt/encoder.rs`

Short user inputs should resolve through the `ropus/src` prefix. For example:

- `--file silk/common` -> `ropus/src/silk/common.rs`
- `--file ropus/src/silk/common.rs` -> unchanged

The script should call cargo-mutants with `-p ropus`. A resolved file that
exists but produces zero mutants should be reported as non-green for the audit,
because that is how the stale-path problem stayed hidden.

### 2. Make baseline policy visible

Keep `.cargo/mutants.toml` limited to mechanical mutation settings such as
timeouts and stable mutation exclusions. Do not hide baseline test skips there.

The runner should have two explicit modes:

- `strict` default: run the unmutated baseline with no hidden skips. If the
  baseline fails, stop and record `baseline_failed`; do not publish a mutation
  score.
- `skip-degraded` optional: if Arthur explicitly accepts a known failing test
  list, pass those skips on the command line and stamp the report as
  `baseline_skipped`, including each skipped test name.

This follows the same testing principle used in the release gate work: a weaker
oracle can exist, but it must be named at the point where someone reads the
result.

### 3. Keep the first revived run unit-test-only

Do not try to bring the C-reference differential binary into cargo-mutants in
this slice. Local `cargo mutants --help` shows `--test-tool` supports `cargo`
and `nextest`, not an arbitrary comparison harness. Wrapping the C harness into
the mutation loop would require new harness integration or a custom wrapper and
would turn this from an infrastructure revival into a correctness-oracle design
project.

The report must state:

```text
Oracle scope: ropus library cargo tests only. C-reference differential harness
is not in this mutation loop.
```

### 4. Produce an audit report, not just stdout

For each module, save under `/home/md/language/ropus/logs/mutation-testing/<timestamp>/`:

- exact command line and cargo-mutants version,
- baseline status and skipped tests if any,
- package and file path,
- mutant count from `--list`,
- caught, missed, timeout, and unviable counts,
- top missed mutants for triage,
- whether the run was strict or skip-degraded,
- statement that the result is periodic audit signal, not a release gate.

Use `cargo mutants --output` where practical so the native `mutants.out`
artifacts are preserved. A compact `summary.txt` or `summary.md` should remain
for quick reading.

### 5. Run one scoped audit to validate the lane

Recommended first target:

```text
/home/md/language/ropus/ropus/src/celt/math_ops.rs
```

Rationale: it has prior mutation pilot data, currently lists 230 mutants, and
is small enough to validate the revived runner without turning the first slice
into an overnight campaign.

Follow-up target after the lane is proven:

```text
/home/md/language/ropus/ropus/src/silk/common.rs
```

Rationale: it is high-value fixed-point codec code and previous notes said its
exact-value pinning tests still needed a full rerun, but it currently lists
1199 mutants and is less suitable as the first infrastructure smoke.

## Policy

Mutation testing should be:

- periodic or on-demand,
- supervisor-triggered,
- non-release-blocking,
- scoped to one module or explicit shard at a time,
- interpreted through survivor triage rather than a raw kill-rate target.

Mutation testing should not be:

- part of default `cargo test`,
- part of quick or non-quick `--release-preflight`,
- a required green check before releases,
- used to claim C-reference parity,
- expanded to the whole workspace in this slice.

Recommended cadence after revival: monthly, before large codec refactors, or
after a major batch of tests lands in a high-risk module.

## Risks

- Runtime and CPU contention: cargo-mutants can dominate a machine. Keep jobs
  low by default and avoid parallel agent mutation runs.
- Baseline failure: removing hidden skips may expose current failing tests. That
  is acceptable; the correct audit result is `baseline_failed`, not a synthetic
  score.
- Equivalent mutants: fixed-point boundary logic can produce survivors that are
  not test gaps. Require triage categories before acting.
- Static-data noise: table-heavy modules can show poor raw kill rates while
  tested functions are fine. Avoid broad raw-score comparisons.
- Unit-only scope: the audit will miss mutations that only the C-reference
  harness would catch. The report must state that limitation every time.
- Tool drift: cargo-mutants output format can change; prefer native `--output`
  artifacts and keep shell parsing conservative.

## Non-Goals

- No codec implementation changes.
- No new tests or mutation-killing pinning tests in this HLD slice.
- No full-workspace mutation campaign.
- No CI-provider workflow.
- No release-preflight or `full-test` gate integration.
- No custom C-reference mutation wrapper.
- No performance, platform, corpus, DRED, PLC, or sanitizer expansion.
- No mutation score threshold in the first revived lane.

## Decisions Needing Supervisor Signoff

1. Mutation testing remains a periodic, non-release-blocking audit lane.
2. The first revived lane is unit-test-only and explicitly does not include the
   C-reference differential harness.
3. Strict baseline is the default; hidden `.cargo/mutants.toml` test skips are
   removed or moved behind an explicit skip-degraded mode.
4. The script uses `-p ropus` and `ropus/src/...` as the authoritative source
   path convention, and zero-mutant resolved files are non-green.
5. The first scoped audit target is `ropus/src/celt/math_ops.rs`; `silk/common`
   becomes the next scheduled target after the lane is proven.
6. No kill-rate threshold is introduced yet; survivor triage quality matters
   more than a headline percentage.
7. Add or update the oracle matrix only after a reproducible strict or
   skip-degraded audit report exists.
