## Problem Statement

The 2026-05-07 overnight 8 h campaign found two real C-vs-Rust parity bugs in `fuzz_encode_multiframe` within ~2 minutes of each launch (`9b8c552` PLC `delay_buffer`, `55e224c` SILK LBRR `ec_prev_*`). Both are the same shape: C mutates encoder state in-place inside a helper, Rust takes `&State` and compensates at call sites, and one call site on a less-trodden code path got missed. See `/home/md/language/ropus/wrk_journals/2026.05.07 - JRN - overnight 8h fuzz campaign.md`, "Findings summary".

That fast TTF on a brand-new target argues the bug class is dense in the input space — the campaign found the easy ones, not the last ones. Saturation analysis on the other 9 targets (957 M execs, 0 new findings) backs this up: they are at or near coverage saturation; multiframe is not.

The current `fuzz_encode_multiframe` grammar (lines 125-176 of the target) varies a small set of per-frame knobs:

| `frame_duration` | 6 values via `multiframe_shape_from_byte` | full |

`fuzz_ctl_sequence` (`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_ctl_sequence.rs`) does already exercise *random* shuffles of every setter against the C ref including out-of-range and getter-parity assertions (line 156-307), but with `n_frames` capped at 1..=8, hard-coded 20 ms PCM, and no byte-exact differential. It hits the API-shape surface; it does *not* hit the *encode-byte-divergence-after-state-accumulation* surface that the multiframe target hits, and it does not pair non-default valid CTL configurations with the long-frame, multi-rate, multi-duration combinations multiframe already varies. Stream A's broadening is complementary: keep multiframe's deep-state byte-exact differential, give it a larger valid configuration vocabulary.

This HLD is fuzz-infrastructure-only. No codec source changes.

## Stakeholder Value

Maintainers — primary user. Need fuzz coverage on encoder state-accumulation bugs that are *reachable* but currently outside the input grammar.

This broadening adds value by:

- Reaching the still-untouched valid-config surface (`max_bandwidth`, `signal`, `force_channels`, `prediction_disabled`, `vbr_constraint=1`) under multi-frame state accumulation, where bugs of the campaign-found shape live.
- Keeping the existing 2-bug-finding signal: byte-exact diff, 2..=16 frames, no relaxation of the analysis.c-divergence dodge.
- Not duplicating `fuzz_ctl_sequence` or the planned Streams C/D (stereo bias, SILK↔CELT transitions): general-purpose grammar widening, not specialised-target work.
- One implementation pass; no codec source changes; one validation pass; reversible target-locally.

## Considered Options

### Option 1: Add a per-frame "extended setters" tuple drawn from a fixed small enum

Add a per-`FrameConfig` `ext: ExtendedConfig` field. `ExtendedConfig` carries five new u8 selectors and is consumed via `Arbitrary`. Each selector indexes a small validated list. Setters fire on both the Rust side and a new `apply_c_extended_config` helper in `c_reference.rs`. `vbr_constraint` is promoted from a hardcoded 0 to a u8 fed equally to both sides.

Pros:

- Concrete, predictable input shape — selectors are bounded, so corpus mutations stay small.
- All five new dimensions are *valid-config* values: the byte-exact differential remains a hard equality.
- Symmetric C and Rust application paths preserve parity by construction.
- Bounded byte cost: 6 extra bytes per frame max (5 selectors + 1 promoted `vbr_constraint`), so a 16-frame packet adds 96 bytes — ~0.15% of `max_len=65536`.
- Compatible with libFuzzer's existing corpus: `Arbitrary` consumption is append-only at the end of `FrameConfig::arbitrary_for_shape`, so old corpora still parse for the prefix; the trailing PCM consumption simply consumes a smaller-than-before tail.

Cons:

- Per-iteration cost rises slightly (extra setter calls per frame, both sides). Quantified below — negligible vs the existing per-frame `opus_encode` cost.
- Adds five new dimensions at once, so a regression triage on a brand-new failure has more axes to bisect. Mitigated by including the new selector values in the panic message and the 16-byte fingerprint (only one byte spare; we'll pack tightly).

Decision: **accept**. Concrete, complete, decision-ready in one pass.

### Option 2: Two-phase split — `vbr_constraint` + 1 setter first, the rest later

Promote `vbr_constraint` from hardcoded 0 to per-frame, and add `set_signal` only. Defer the others to a follow-up.

Pros:

- Smallest possible diff. Easier triage attribution.
- Lower risk of stalling on a non-codec issue (e.g. a getter-parity gap exposed by `set_force_channels`).

Cons:

- Doesn't justify its process overhead. The five setters Option 1 adds are all bounded validated CTL ranges; the C side already has setter wrappers. Splitting just doubles validation passes for no real risk reduction.
- Leaves the highest-signal dimensions (`force_channels`, `force_mode`, `prediction_disabled`) under-covered for another cycle.

Decision: **reject**. Process-only optimisation; no real risk reduction.

### Option 3: Vary frame duration *within* a packet via `set_expert_frame_duration`

Allow each frame in a sequence to choose its own legal `OPUS_FRAMESIZE_*` enum, applied via `set_expert_frame_duration` mid-stream.

Pros:

- Would exercise mode-transition logic that depends on per-frame requested duration.
- The setter is on the encoder API.

Cons:

- The current target passes `frame_size` as the literal sample count to `opus_encode` (line 255, 268). Mixing per-frame `set_expert_frame_duration` with a fixed `frame_size` argument has subtle, undocumented interactions with the auto-switch logic; getting C and Rust to agree on what "per-frame variable duration in a single-frame `opus_encode` call" means at the byte level is a separate problem.
- Slice B of the 2026-05-06 frame-duration HLD already locked the rule "one selected legal frame duration per generated sequence" (`/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - legal frame duration stateful fuzz.md`, Option D, decision: reject for Slice B). Reversing that here without a separate design would break the contract.

Decision: **reject**. Out of scope for this stream. Recorded as a candidate for a future dedicated stream if signal demands it. See "Out of Scope".

### Option 4: Random CTL-shuffle (any valid setter on any frame)

Pull in the `fuzz_ctl_sequence` shuffle approach: each frame draws a random subset of N valid CTLs.

Pros:

- Maximally broad surface coverage in one design.

Cons:

- Duplicates `fuzz_ctl_sequence`'s scope. That target is already running clean (it was not part of the overnight 8 h workers, but it was added in an earlier cycle; running it again is on Streams C/D's domain, not Stream A's).
- Erodes the multiframe target's clearest property: a deterministic config-per-frame that survives byte-exact comparison. The two campaign-found bugs were both found because the byte-mismatch *pinpointed* a specific config tuple. Random shuffle pollutes that signal.

Decision: **reject**. Conflicts with multiframe's signal-quality property.

## Recommended Design (Option 1 detail)

### Setter selection — 5 of 8 added

| Setter | Added? | Rationale |
|---|---|---|
| `set_vbr_constraint` | **yes** | promote hardcoded 0 → u8 from input. Free dimension; both sides already wired. |
| `set_max_bandwidth` | **yes** | gates SILK→Hybrid bandwidth ladder; pairs with the existing `frame_duration` dimension. |
| `set_signal` | **yes** | mode-decision bias independent of `application`. |
| `set_force_channels` | **yes** | mid-stream channel-count surgery — high-state-mutation surface, exactly the campaign-bug shape. |
| `set_prediction_disabled` | **yes** | SILK LTP toggle; the campaign's 2nd bug was in the SILK LBRR / `ec_prev_*` neighbourhood. |
| `set_force_mode` | **dropped** | Hits the SILK↔CELT mode-transition surface. Stream D explicitly owns that scope (kickoff journal, line 29). Don't duplicate. |
| `set_lsb_depth` | **dropped** | Rate-allocation knob, mostly absorbed downstream. Lower-signal-per-byte than the 5 selected. Cap on dimension count to keep triage tractable; this is the easiest cut. |
| `set_phase_inversion_disabled` | **dropped** | CELT-only side effect; Stream C (stereo) is a better home if it warrants its own dimension. |

### Concrete grammar additions

In `MultiframeInput::arbitrary` (no change to top-level fields).

In `FrameConfig::arbitrary_for_shape`, append after the existing fields:


### Fingerprint extension

Current `input_fingerprint` (line 80-97) packs first-frame summary into 16 bytes. `fp[15]` is currently spare. Pack the new dimensions into `fp[15]` for the first frame:

```rust
fp[15] = ((first.vbr_constraint as u8) & 0x1)
       | ((first.max_bw_idx & 0x7) << 1)
       | ((first.signal_idx & 0x3) << 4)
       | ((first.force_ch_idx & 0x3) << 6);
// prediction_disabled doesn't fit; that's acceptable —
// fingerprint is a hint, not a oracle.
```

Bit budget: 1 + 3 + 2 + 2 = 8 bits. `prediction_disabled` is omitted from the fingerprint by design — the fingerprint is a triage hint, not an oracle, and the libFuzzer artifact preserves the raw bytes.

### Cost analysis

| Cost | Before | After | Delta |
|---|---|---|---|
| Bytes consumed per `Arbitrary` per frame (config) | ~7 (1 u16 bitrate + 5 u8 + 1 u8 loss_perc clamp) | ~12 | +5 |
| Bytes consumed per packet (16 frames worst case) | ~112 (config) + PCM | ~192 (config) + PCM | +80 (~0.12% of max_len=65536) |
| Setters called per frame, Rust | 7 | 11 | +4 |
| Setters called per frame, C | 7 (via `apply_c_encoder_config`) | 11 | +4 |
| `opus_encode` call per frame | 1 (Rust) + 1 (C) | 1 (Rust) + 1 (C) | 0 |

The dominant cost in this target is `opus_encode` itself (and downstream `OpusDecoder::decode` for the round-trip sanity check at line 367-387). Setter calls are O(constant) state writes; quantified at `< 1 µs` each on a hot encoder. Going from 7 to 11 setters per frame in two encoders adds < 100 ns to a per-frame cost already in the 5-50 ms range. The exec/sec hit is bounded above at ~0.001%, lost in noise.

### Out-of-bounds rejection (free invariant check)

`set_max_bandwidth` rejects `OPUS_BANDWIDTH_NARROWBAND - 1` and `OPUS_BANDWIDTH_FULLBAND + 1`. Our table only contains the 5 valid values, so we never exercise the rejection path here. That's deliberate: this stream is byte-exact differential, so we stay in the *valid-config* zone. Out-of-range CTL behaviour is `fuzz_ctl_sequence`'s scope.

Re-anchored thresholds (both must hold):

1. **Absolute floor: `NEW_UNITS_POST >= 300`.**
   Rationale: 5 new dimensions × ~60 conservative new edges per dimension. If the broadening reaches each new dimension's local code paths at all, ≥ 60 edges per dimension is the floor. Anything below 300 absolute means at least one dimension is inert (or libFuzzer hasn't surfaced its mutator schedule for it yet — investigate before signing off).

2. **Rate ratio: `NEW_UNITS_POST / NEW_UNITS_PRE >= 2.0`.**
   Rationale: with the pre-run anchoring to the same machine and corpus, a 2× ratio washes out machine variance and proves broadening *actually opened new code*, on top of whatever residual baseline-rate growth multiframe still has.

If only one criterion holds, treat as ambiguous and triage:
- Floor passed, ratio missed → broadening reached new code but baseline was already healthy; acceptable, document the case and proceed.
- Floor missed, ratio passed → broadening *relatively* helped but absolute coverage is still tiny; investigate whether one or more dimensions is inert (e.g. `set_force_channels` always rejected because `force_ch_idx` mutators clustered on `2` against mono).
- Both missed → broadening is inert. Stop, re-evaluate which dimensions are reaching new code (use `cargo +nightly fuzz coverage` against both runs to compare).

## Risks and Trade-offs

- **Setter-ordering coupling**: the Rust and C side must apply the new setters in identical order. Setter side-effects in `set_force_channels` (writes through to mode logic) are reorder-sensitive. Mitigation: a single shared comment block at both call sites pinning the canonical order, and the validation plan's smoke run catches regressions.
- **Existing `CEncodeConfig` literals in other fuzz targets break the build until updated** (transient compile error during the implementation pass). Bounded — ~3 call sites; `cargo +nightly fuzz build` catches them all in one pass.
- **`set_force_channels(2)` on a mono encoder returns `OPUS_BAD_ARG` and is a no-op on both sides — verified at hash `bc446cb`.**
  - Rust handler `/home/md/language/ropus/ropus/src/opus/encoder.rs:3510-3516`:
