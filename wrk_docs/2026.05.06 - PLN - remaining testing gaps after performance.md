## Testing Principle

The testing campaign has converted most hidden false-green risks into explicit
release claims, warnings, or debt. The next slice should therefore add a missing
release-confidence dimension that is cheap enough to keep honest.

Use the test pyramid idea pragmatically here: do not add a broad, flaky matrix.
Add one or two narrow, automatable platform signals that catch classes of bugs
the current optimized local path cannot see.

## Recommended Next Slice

Recommended HLD-first slice:

```text
Platform and sanitizer release-breadth gate
```

This should add a narrow, explicit platform/sanitizer signal to `full-test`
without expanding into a full CI matrix.

### Why This Next

- It is the strongest remaining testing-infrastructure gap after performance:
  current local builds intentionally assume `x86-64-v3`, while users can rebuild
  generic `x86-64`.
- It attacks a different failure mode from all prior work. Exact codec oracles
  can pass on the optimized host while target-feature assumptions, unchecked
  indexing, C FFI boundaries, or sanitizer-only memory issues remain unseen.
- It is more bounded than mutation testing and more infrastructure-shaped than
  DRED/PLC/performance optimization work.
- It can be designed with honest claim states: unavailable, report-only,
  required-release-breadth, fail, warn, and skipped.

### Proposed Scope

1. Add a `platform` or `breadth` stage/profile to `full-test`.
2. In default local runs, report platform/sanitizer checks as not claimed or
   optional/report-only.
3. In non-quick `--release-preflight`, require at least:
   - one generic `x86_64` build/test smoke that deliberately avoids the
     repository's `x86-64-v3` assumption, and
   - one practical sanitizer smoke when the local nightly/toolchain supports it,
     or an explicit unavailable status that is banner-blocking only if the HLD
     signs off that sanitizer availability is a release requirement.
4. Keep quick release preflight as core smoke only; it should make no platform
   breadth or sanitizer claim.
5. Update JSON/HTML/banner/help text and the oracle matrix so the release report
   states exactly which breadth claims were made.

Recommended first gate shape:

- `generic-x86_64-smoke`: release-blocking under non-quick release preflight.
- `asan-smoke`: start as report-only unless the implementing HLD proves the
  Linux nightly setup is deterministic enough locally. Promote later after one
  clean implementation/review cycle.

## What Not To Mix In

Do not mix these into the next testing-infrastructure slice:

- Performance optimization or threshold recalibration. The `8.74x` decode result
  needs a separate calibrated sweep or performance investigation.
- DRED payload-byte parity, DRED PCM reconstruction, FARGAN implementation, or
  LPCNet feature-drift fixes.
- PLC full-frame exact-parity repair.
- Corpus asset curation, WebM support, seek/output-gain semantics, or external
  media commits.
- Mutation-testing revival.
- A full GitHub Actions/platform matrix or ops release choreography.
- Changing the project's default `x86-64-v3` policy. The next slice should
  test outside it, not debate or undo it.

## Risks

- Cargo rustflags precedence must be verified. The generic smoke must prove it
  really overrides the `x86-64-v3` default rather than accidentally compiling
  with both flags in the wrong order.
- Sanitizer support may vary across stable/nightly, host linker, and C-reference
  build settings. The first slice should make unavailability explicit.
- Sanitizing the C reference/harness path may expose reference-side or leak
  noise. Start with the `ropus` crate smoke, then decide whether a harness smoke
  is stable enough.
- A new stage can bloat `full-test` if it tries to become a full CI matrix. Keep
  it to one generic CPU profile and one sanitizer probe.
- Existing codec debts remain real. This slice improves release breadth; it does
  not close DRED, PLC, corpus, or performance debts.
