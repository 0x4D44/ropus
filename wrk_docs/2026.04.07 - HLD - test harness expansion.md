# HLD: Maximal Test Suite for mdopus

## Problem

The test harness validates bit-exact C/Rust matching but only exercises one
configuration: 64 kbps CBR, complexity 10, 20 ms frames, `OPUS_APPLICATION_AUDIO`.
The Rust implementation is feature-complete (SILK, CELT, hybrid, multistream,
repacketizer, FEC, DTX, PLC, all decode formats) but most of these features
have zero integration-level test coverage against the C reference.

## Goal

A test suite that, when it passes, gives high confidence that the Rust
implementation is a correct drop-in replacement for the C reference across
the full Opus feature set.

## Architecture

Everything extends the existing `mdopus-compare` binary. New commands are
added alongside existing ones (`encode`, `decode`, `roundtrip`, `bench`, etc.).

A separate `tests/fuzz/` directory holds cargo-fuzz targets.

## Part 1: EncodeConfig (foundation for everything else)

The current helpers hardcode application mode, VBR, and frame duration.
Refactor them to accept a config struct:

```rust
struct EncodeConfig {
    sample_rate: i32,
    channels: i32,
    bitrate: i32,
    complexity: i32,
    application: i32,     // VOIP, AUDIO, or LOWDELAY
    vbr: i32,             // 0=CBR, 1=VBR
    vbr_constraint: i32,  // 0=unconstrained, 1=constrained
    frame_ms: f64,        // 2.5, 5, 10, 20, 40, 60
    fec: i32,             // 0=off, 1=on
    packet_loss_pct: i32, // 0-100, influences FEC decisions
    dtx: i32,             // 0=off, 1=on
    signal: i32,          // OPUS_AUTO, OPUS_SIGNAL_VOICE, OPUS_SIGNAL_MUSIC
    bandwidth: i32,       // OPUS_AUTO or forced bandwidth
    force_channels: i32,  // OPUS_AUTO, 1, or 2
}
```

All new and existing commands use this. Existing commands fill it from CLI args
with current defaults, so no behavior change.

General encode/decode helpers become:

```rust
fn c_encode_cfg(pcm: &[i16], cfg: &EncodeConfig) -> Vec<u8>;
fn rust_encode_cfg(pcm: &[i16], cfg: &EncodeConfig) -> Vec<u8>;
fn c_decode_cfg(encoded: &[u8], sample_rate: i32, channels: i32, frame_ms: f64) -> Vec<i16>;
fn rust_decode_cfg(encoded: &[u8], sample_rate: i32, channels: i32, frame_ms: f64) -> Vec<i16>;
```

## Part 2: Parametric Sweep (`sweep`)

Runs C-vs-Rust roundtrip comparison across a curated set of configurations.

### Parameter space

| Parameter | Values | Notes |
|-----------|--------|-------|
| Sample rate | 8000, 12000, 16000, 24000, 48000 | All 5 valid rates |
| Channels | 1, 2 | |
| Bitrate | 6000, 12000, 24000, 32000, 64000, 128000, 256000, 510000 | Full range |
| Frame duration | 2.5ms, 5ms, 10ms, 20ms, 40ms, 60ms | All valid sizes |
| Application | AUDIO, VOIP, LOWDELAY | All 3 modes |
| VBR | CBR, VBR, constrained VBR | 3 modes |
| Complexity | 0, 5, 10 | Low, mid, high |
| Signal hint | AUTO, VOICE, MUSIC | |

### Case generation

Not a full cross-product (would be ~50,000 configs). Instead, a curated flat
list of ~200–300 cases organized into sections:

| Section | Focus | ~Cases |
|---------|-------|--------|
| CELT core | 48k, bitrates × frames × vbr × app × complexity | 50 |
| SILK core | 8k, bitrates × frames × vbr × complexity | 30 |
| Hybrid | 16k/24k, bitrate range crossing mode boundaries | 25 |
| Application modes | VOIP vs AUDIO vs LOWDELAY behavior differences | 15 |
| VBR modes | CBR, VBR, constrained VBR across modes | 20 |
| Frame durations | All 6 durations × 3 sample rates | 18 |
| Signal hints | VOICE vs MUSIC at 16k and 48k | 10 |
| Stereo | Stereo-specific paths across all modes | 25 |
| Bandwidth forcing | NB, MB, WB, SWB, FB forced at 48k | 10 |
| Extreme params | min/max bitrate, complexity 0, 2.5ms+LOWDELAY | 15 |
| Force channels | Force mono from stereo, stereo from mono | 6 |

Each case runs against a 1-second WAV matched by sample rate and channels.
Uses noise signals (most codepath coverage). Runtime: ~1-3 minutes.

### Pass/fail

Both CBR and VBR must match byte-for-byte (VBR is deterministic).
Decoded PCM compared sample-for-sample. Any mismatch = FAIL.

### CLI

```
mdopus-compare sweep [--filter <pattern>] [--stop-on-fail]
```

WAV directory is hardcoded to `tests/vectors/`. The `--filter` flag matches
against case labels (substring, case-insensitive).

## Part 3: Packet Loss Concealment (`plc`)

Simulates dropped frames and verifies PLC output matches C.

### Design

1. Encode WAV with C encoder.
2. Decode with both C and Rust decoders, dropping specific frames
   (pass `None` to decode at drop positions).
3. Compare PCM output sample-for-sample, including the concealment frames.

### Drop patterns

- **Isolated**: Drop frames 5, 15, 25, 35, 45 (10% random-ish loss)
- **Burst**: Drop frames 20-22 (3 consecutive)
- **Heavy**: Drop every 3rd frame (33% loss)

Run at multiple sample rates (8k, 16k, 48k) to exercise SILK PLC, CELT PLC,
and hybrid PLC.

### CLI

```
mdopus-compare plc <input.wav> [--bitrate N] [--pattern isolated|burst|heavy]
```

## Part 4: FEC (Forward Error Correction) (`fec`)

Tests FEC encoding and recovery.

### Design

1. Encode with `fec=1, packet_loss_pct=20`.
2. Verify C and Rust produce identical encoded packets (FEC is deterministic).
3. Decode with `decode_fec=true` on specific frames (simulating recovering a
   previous frame from current frame's FEC data).
4. Compare recovered PCM.

This exercises the SILK LBRR path and the redundancy encoding in the Opus
encoder.

### CLI

```
mdopus-compare fec <input.wav> [--bitrate N] [--loss-pct N]
```

## Part 5: DTX (Discontinuous Transmission) (`dtx`)

Tests silence detection and DTX behavior.

### Design

1. Use a signal with silence gaps (e.g., alternating 200ms noise + 200ms silence).
2. Encode with `dtx=1` with both C and Rust.
3. Verify identical encoded output (some frames will be empty/DTX frames).
4. Decode with both and compare PCM.

This exercises the VAD, DTX mode selection, and CNG (comfort noise generation)
in the decoder.

### Test vector

Generate `48k_dtx_test.wav`: 5 seconds of alternating 200ms noise / 200ms
silence. Also generate at 8k and 16k for SILK DTX testing.

### CLI

```
mdopus-compare dtx <input.wav> [--bitrate N]
```

## Part 6: Multistream (`multistream`)

Tests multi-channel encoding/decoding.

### Design

The multistream API is implemented but not exported publicly. For testing,
use `pub(crate)` visibility or add the test inside the crate (as an
integration test that accesses internal modules).

Test cases:
1. **Surround 5.1**: 6-channel encoding/decoding with Vorbis channel mapping.
2. **Stereo via multistream**: 2-channel multistream (should produce identical
   output to single-stream stereo).
3. **Mono via multistream**: 1-channel multistream.

For each: create C and Rust multistream encoders, encode identical PCM, compare
encoded bytes and decoded PCM.

### Test vectors

Generate multi-channel WAVs programmatically in the harness (sine waves at
different frequencies per channel). No need to store large WAV files.

### CLI

```
mdopus-compare multistream [--channels 6] [--mapping-family 1]
```

## Part 7: Repacketizer (`repacketizer`)

Tests packet manipulation.

### Design

1. Encode a signal to get real Opus packets.
2. Feed packets through both C and Rust repacketizers.
3. Operations to test:
   - `cat`: Concatenate multiple frames into one packet.
   - `out`: Extract frames from a multi-frame packet.
   - `out_range`: Extract specific frame range.
   - `pad` / `unpad`: Add/remove padding bytes.
4. Compare output packets byte-for-byte.

### CLI

```
mdopus-compare repacketizer <input.wav>
```

## Part 8: Packet Introspection (`packets`)

Tests all packet parsing functions.

### Design

Encode at various configurations, then for each packet verify that C and Rust
agree on:
- `opus_packet_get_bandwidth()`
- `opus_packet_get_nb_channels()`
- `opus_packet_get_nb_frames()`
- `opus_packet_get_samples_per_frame()`
- `opus_packet_get_nb_samples()`
- `opus_packet_has_lbrr()`

Run across sweep configurations to cover all TOC byte variants.

### CLI

```
mdopus-compare packets <input.wav> [--bitrate N]
```

## Part 9: Decode Formats (`decode-formats`)

Verify all three decode output formats match C.

### Design

For a given encoded stream, decode with both C and Rust using:
1. `decode()` → i16 PCM (already tested)
2. `decode24()` → i32 (24-bit in 32-bit container)
3. `decode_float()` → f32

Compare output for each format. The i16 path is already covered by roundtrip;
this adds the other two.

### CLI

```
mdopus-compare decode-formats <input.wav>
```

## Part 10: Long-Duration Stability (`longsoak`)

Runs encode/decode on long signals to catch state-accumulation bugs.

### Design

Generate long test signals programmatically in the harness:
- 30 seconds of noise at 48kHz stereo
- 30 seconds of noise at 8kHz mono
- 30 seconds of alternating noise/silence for DTX stress

Encode and decode with both C and Rust, compare every sample. Also verify
`final_range` matches after every frame to catch state drift early.

Runtime: ~30-60 seconds per signal.

### CLI

```
mdopus-compare longsoak [--duration 30] [--sample-rate 48000]
```

## Part 11: Quality Sanity Check (`quality`)

Verify that encode-decode roundtrip quality is in expected range.

### Design

This is NOT a C-vs-Rust comparison. It's a sanity check that the codec
produces reasonable output. Even if bit-exact with C, if both produce garbage
we'd want to know.

For each roundtrip:
1. Compute SNR (signal-to-noise ratio) of decoded vs original.
2. Verify SNR is above a minimum threshold for the given bitrate.

Expected SNR ranges (approximate, from Opus spec):
- 64 kbps stereo 48kHz: SNR > 25 dB
- 128 kbps stereo 48kHz: SNR > 30 dB
- 24 kbps mono 16kHz: SNR > 15 dB
- 8 kbps mono 8kHz: SNR > 8 dB

Implementation: compute in the harness, no external tools.

### CLI

```
mdopus-compare quality <input.wav> [--bitrate N] [--min-snr N]
```

## Part 12: Fuzz Testing (`tests/fuzz/`)

Separate from the harness. Uses `cargo-fuzz` with libFuzzer.

### Targets

1. **`fuzz_decode`**: Feed random bytes to `OpusDecoder::decode()`. Must not
   panic. May return errors (expected). Must not infinite-loop. Run with a
   timeout per input.

2. **`fuzz_encode`**: Feed random i16 PCM to `OpusEncoder::encode()`. Must
   not panic. Output must be a valid Opus packet (decodable without error).

3. **`fuzz_roundtrip`**: Encode random PCM, decode the result. Must not panic.
   Decoded length must match expected frame size.

4. **`fuzz_repacketizer`**: Feed random bytes to `OpusRepacketizer::cat()`.
   Must not panic.

5. **`fuzz_packet_parse`**: Feed random bytes to all `opus_packet_get_*`
   functions. Must not panic.

### Setup

```
tests/fuzz/
  Cargo.toml          # cargo-fuzz project
  fuzz_targets/
    fuzz_decode.rs
    fuzz_encode.rs
    fuzz_roundtrip.rs
    fuzz_repacketizer.rs
    fuzz_packet_parse.rs
```

### Corpus seeding

Seed the decode fuzzer with real Opus packets extracted during sweep test.
Seed the encode fuzzer with real PCM from test vectors.

## Part 13: API Validation (`api`)

Tests error handling: invalid parameters return correct error codes.

### Design

For both C and Rust, verify identical behavior for:
- Invalid sample rates (e.g., 44100, 0, -1)
- Invalid channel counts (0, 3, 256)
- Invalid application modes
- Bitrate out of range
- Complexity out of range
- Frame size mismatches
- Null/empty packet decode
- Decode with wrong sample rate vs encoded rate
- Encoder CTL with invalid request codes

For each case: call both C and Rust, verify same error code returned.

### CLI

```
mdopus-compare api
```

## Test Vectors

### Existing (keep)

35 WAV files at 8k/12k/16k/24k/48k, mono/stereo, silence/sine/noise/etc.

### New (generate in harness, not stored)

Long-duration signals and DTX test signals are generated programmatically
at runtime using deterministic RNG. This avoids bloating the repo with large
WAV files.

```rust
fn generate_noise(sample_rate: i32, channels: i32, duration_secs: f64, seed: u64) -> Vec<i16>;
fn generate_dtx_signal(sample_rate: i32, channels: i32) -> Vec<i16>;
fn generate_multitone(sample_rate: i32, channels: i32, freqs: &[f64]) -> Vec<i16>;
```

| Phase | Work | Unlocks |
