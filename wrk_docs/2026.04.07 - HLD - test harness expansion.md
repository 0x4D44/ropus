# HLD: Test Harness Expansion

## Problem

The test harness validates bit-exact C/Rust matching, but only exercises one
configuration: 64 kbps CBR, complexity 10, 20 ms frames, `OPUS_APPLICATION_AUDIO`.
The Opus codec has hundreds of valid parameter combinations that select different
internal codepaths (SILK, CELT, hybrid, different transform sizes, VBR rate
control, mode-switching logic). A bug in any untested path would ship silently.

## Approach

Extend the existing `mdopus-compare` binary with two new commands:

1. **`sweep`** — Parametric configuration sweep. Runs C-vs-Rust roundtrip
   comparison across a curated set of parameter combinations.
2. **`plc`** — Packet loss concealment test. Simulates dropped frames and
   verifies the decoder PLC path matches C.

No new binaries, no new crates, no feature flags. Everything stays in
`tests/harness/main.rs`.

## Design

### 1. Generalize encode/decode helpers

The current `c_encode`, `rust_encode`, `c_decode`, `rust_decode` functions
hardcode application mode, VBR, and frame duration. Refactor them to accept a
config struct:

```rust
struct EncodeConfig {
    sample_rate: i32,
    channels: i32,
    bitrate: i32,
    complexity: i32,
    application: i32,   // VOIP, AUDIO, or LOWDELAY
    vbr: i32,           // 0=CBR, 1=VBR
    frame_ms: f64,      // 2.5, 5, 10, 20, 40, 60
}

impl EncodeConfig {
    fn frame_size(&self) -> usize {
        (self.sample_rate as f64 * self.frame_ms / 1000.0) as usize
    }
}
```

Existing commands (`encode`, `roundtrip`, `bench`, etc.) construct an
`EncodeConfig` from CLI args with the current defaults, so nothing changes for
existing callers.

### 2. The `sweep` command

`sweep` takes a WAV directory and iterates over a curated parameter matrix.
For each combination, it encodes the appropriate WAV file with both C and Rust,
then decodes with both, comparing byte-for-byte (encode) and sample-for-sample
(decode).

**Parameter matrix** — designed to cover every major codepath without
combinatorial explosion:

```
Sample rates:   8000, 16000, 48000
                (covers SILK-only, hybrid, CELT-only)

Channels:       1, 2

Bitrates:       8000, 16000, 32000, 64000, 128000, 256000
                (low→high; triggers mode switching at boundaries)

Frame durations: 5ms, 10ms, 20ms, 60ms
                (2.5ms is CELT-only edge case; 20ms is default; 60ms
                tests multi-frame; 5ms and 10ms test short transforms)

Application:    AUDIO, VOIP, LOWDELAY

VBR:            0, 1

Complexity:     0, 5, 10
```

**Not a full cross-product.** Many combinations are invalid or redundant:
- LOWDELAY restricts frame sizes (no 60ms)
- Very low bitrates at 48 kHz are valid but don't add coverage
- SILK-only rates at LOWDELAY force CELT mode anyway

The sweep uses a flat list of curated test cases, not nested loops. This makes
it easy to add/remove cases and understand what's covered. The list targets
roughly **100–150 configurations**, each running against 1 second of audio.
Runtime estimate: ~30–60 seconds total.

**Curated list structure:**

```rust
struct SweepCase {
    cfg: EncodeConfig,
    label: &'static str,   // e.g. "SILK 8k mono CBR low-bitrate"
}

fn sweep_cases() -> Vec<SweepCase> { ... }
```

Organized into sections:

| Section | Sample rate | Purpose | ~Cases |
|---------|------------|---------|--------|
| CELT core | 48000 | bitrate × frame × vbr × app × complexity | ~40 |
| SILK core | 8000 | bitrate × frame × vbr × complexity | ~20 |
| Hybrid | 16000 | bitrate range that crosses SILK/hybrid boundary | ~15 |
| Wideband | 24000 | hybrid at higher sample rate | ~10 |
| Stereo | 48000, 16000, 8000 | stereo-specific paths (mid-side, etc.) | ~15 |
| Edge | mixed | min/max bitrate, min/max frame, complexity 0 | ~10 |

**Pass/fail logic:**

For CBR: encoded bytes must match exactly (byte-for-byte).
For VBR: encoded bytes must still match exactly — VBR is deterministic given
identical state. If VBR diverges, that's a real bug.
Decoded PCM always compared sample-for-sample.

**Output format:**

```
sweep: 147 configurations
  [  1/147] PASS  CELT 48k mono CBR 64kbps 20ms cx10 AUDIO
  [  2/147] PASS  CELT 48k mono CBR 64kbps 20ms cx10 VOIP
  ...
  [ 58/147] FAIL  SILK 8k mono VBR 8kbps 60ms cx5 AUDIO
  ...
sweep: 146 PASS, 1 FAIL
```

On failure, print the first divergence details (same as current `roundtrip`
output) and continue to the next case.

**CLI:**

```
mdopus-compare sweep <wav_directory>
mdopus-compare sweep <wav_directory> --filter silk    # run only SILK cases
mdopus-compare sweep <wav_directory> --stop-on-fail   # stop at first failure
```

The WAV directory is `tests/vectors/`. The sweep picks the right WAV for each
case by matching sample rate and channel count (e.g., `8000hz_mono_noise.wav`
for an 8 kHz mono case). Uses noise signals by default since they exercise the
most codepaths.

### 3. The `plc` command

Tests packet loss concealment by simulating dropped frames:

1. Encode a WAV with C encoder (deterministic baseline).
2. Decode all frames normally with both C and Rust decoders.
3. Re-decode, but drop frames at positions 5, 10, 15 (pass `None` to decode).
4. Compare C and Rust decoded PCM sample-for-sample.

This exercises the `decode_lost()` codepath in the CELT decoder and the PLC
path in the SILK decoder.

```
mdopus-compare plc <input.wav> [--bitrate N] [--drop-pattern 5,10,15]
```

Default drop pattern: every 5th frame for the first 20 frames, then a burst
of 3 consecutive drops. This covers both isolated and burst loss.

### 4. Test vector gap: long-duration signal

Add one longer test vector (~10 seconds) to catch state-accumulation bugs.
Generate it from existing vectors by concatenating noise samples. Add to
`tests/vectors/` as `48k_long_noise_10s.wav`.

No speech signals needed — SILK's pitch detection and VAD run on any signal,
and bit-exact matching catches codepath bugs regardless of signal content.
Speech would only matter for perceptual quality testing, which isn't our goal.

## What this does NOT cover (and why)

- **Multistream/repacketizer**: Separate concern, low priority for core codec stability.
- **FEC decode**: Uses the same decode pipeline; PLC testing covers the
  critical divergence point. Can add later if needed.
- **Corrupt packet fuzzing**: Would test error handling, not codec correctness.
  Add as a separate fuzzing effort if/when we want robustness guarantees.
- **Quality metrics (PESQ/POLQA)**: We test bit-exactness, not perceptual
  quality. If output matches C, quality is identical by definition.

## Execution order

1. Introduce `EncodeConfig` and refactor existing helpers (internal, no behavior change).
2. Build the `sweep_cases()` list and `cmd_sweep`.
3. Run sweep; fix any failures found (these are real bugs).
4. Add `plc` command.
5. Generate long-duration vector, add to sweep.
