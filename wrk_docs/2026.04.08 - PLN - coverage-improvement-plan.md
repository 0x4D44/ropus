# Coverage Improvement Plan

Date: 2026-04-08
Target command: `mdtimeout 300 cargo +nightly llvm-cov --branch --json --summary-only --output-path target\coverage-summary.json`
Starting point:
- Regions: 50.36%
- Functions: 64.27%
- Lines: 48.94%
- Branches: 35.10%

## Goal

Improve repository coverage with minimal-risk test additions, prioritizing measured gains over cosmetic test count increases.

## Stage 1: Cheap Structural Wins

Objective:
- Cover small untested modules and obvious public entry points.

Files:
- `src\celt\ec_ctx.rs`
- `src\celt\rate.rs`
- `src\silk\common.rs`
- `src\main.rs`

Test themes:
- trait forwarding and wrong-side panic behavior in `EcCoder`
- pulse/bit conversion and allocation edge cases in `rate.rs`
- boundary / rounding / reciprocal / expander helpers in `silk::common`
- a trivial binary `main()` execution test

Expected outcome:
- modest total gain
- strong branch gain per line of test code
- low risk because these are narrow, local seams

## Stage 2: Expand Top-Level Opus API Scenarios

Objective:
- Execute real codec flows rather than helper-only branches.

Files:
- `src\opus\encoder.rs`
- `src\opus\decoder.rs`

Test themes:
- mono/stereo encode and decode round-trips at multiple sample rates
- `encode_float`, `decode_float`, and `decode24`
- PLC after a real packet
- tiny-packet / invalid frame-size error branches
- CTL permutations: bitrate, complexity, VBR, bandwidth, FEC, DTX, signal type, force-channels, frame duration
- state reset and getter consistency after real work

Expected outcome:
- meaningful line gain across `opus::*`
- indirect gains in `celt::*` and `silk::*`

## Stage 3: Multistream and Mapping Paths

Objective:
- Pull more orchestration code into coverage with realistic multi-stream cases.

Files:
- `src\opus\multistream.rs`

Test themes:
- encode/decode with non-trivial layouts
- surround and ambisonics constructors beyond the current happy path
- invalid layouts and packet/control edge cases
- final-range and state-reset behavior after multi-stream work

Expected outcome:
- moderate gain in `opus::multistream`
- indirect coverage for shared encode/decode machinery

## Stage 4: SILK Scenario Coverage

Objective:
- attack the largest cold region with scenario tests instead of helper micro-tests.

Files:
- `src\silk\encoder.rs`
- `src\silk\decoder.rs`
- `src\silk\common.rs`

Test themes:
- real SILK-only encode/decode flows via Opus settings that force SILK paths
- packet-loss / FEC / PLC combinations
- bandwidth changes and frame-duration variations
- state transitions between SILK-only, hybrid, and CELT-only paths

Expected outcome:
- the only realistic way to move the global percentage sharply

## Stage 5: Residual Branch Sweep

Objective:
- use the post-Stage-4 report to patch isolated cold branches.

Files:
- whichever files remain worst after re-measurement

Test themes:
- branch-directed tests only
- no broad code churn

Expected outcome:
- incremental cleanup of stubborn branches after the big scenario tests land

## Execution Rules

1. After each stage:
   - run scoped tests first
   - rerun `cargo +nightly llvm-cov --branch`
   - record the delta

2. Prefer tests that execute shared top-level paths over tests that only hit one helper.

3. Avoid refactors unless a test seam is otherwise impossible.

4. Commit after each meaningful coverage increment, keeping the commits atomic.

## Reality Check On 98%

98% is not a short hop from the current baseline. The current repo is at:
- 48.94% lines
- 35.10% branches

That means:
- helper-only additions are insufficient
- full encode/decode scenario coverage is mandatory
- SILK has to move dramatically, not cosmetically

The plan is therefore staged to front-load low-risk wins, then pivot hard into high-leverage scenario execution where the actual missing code lives.
