# Coverage Improvement Plan

Date: 2026-04-08
Target: Improve `cargo +nightly llvm-cov --branch` coverage and push toward 98%.

## Baseline

- Lines: 48.95%
- Functions: 64.31%
- Regions: 50.36%
- Branches: 35.10%

## Strategy

The plan has to maximize covered execution per test added. Small helper tests will not move this repo enough. The first stages focus on end-to-end codec paths, then on direct unit tests for remaining uncovered utility modules, then on branch closure.

## Stage 0: Keep Coverage Running Cleanly

Goal: preserve a reliable branch-coverage loop.

- Keep `cargo +nightly llvm-cov --branch` passing on Windows.
- Re-run coverage after every significant change.
- Commit each significant coverage increment separately.

Exit criteria:

- Clean `llvm-cov --branch` run with no test failures.

## Stage 1: Activate The Existing Harness

Goal: turn `tests/harness/main.rs` into an actual coverage engine.

Actions:

- Add automated tests around stable harness flows instead of leaving them as CLI-only commands.
- Start with the highest leverage commands:
  - `cmd_api`
  - `cmd_sweep` or carefully selected sweep subsets
  - `cmd_plc`
  - `cmd_fec`
  - `cmd_dtx`
  - `cmd_packets`
  - `cmd_decode_formats`
  - `cmd_repacketizer`
  - `cmd_multistream`
- Prefer smoke-style assertions that verify execution and parity, not brittle byte-perfect expectations unless those are already established.

Expected effect:

- Large gains in `silk::encoder`, `silk::decoder`, `opus::encoder`, `opus::decoder`, `opus::multistream`, and packet-handling code.

Stage target:

- Push line coverage above 65%.
- Push branch coverage above 50%.

## Stage 2: Fill The Zero-Test Source Modules

Goal: eliminate the obvious test deserts.

Primary targets:

- `src/celt/ec_ctx.rs`
- `src/celt/rate.rs`
- `src/celt/lpc.rs`
- `src/silk/common.rs`

Actions:

- Add direct unit tests in-module so private helpers can be exercised.
- Focus on edge cases and branch-driving inputs, not only happy paths.
- Use narrow deterministic vectors to avoid fragile tests.

Expected effect:

- Good function coverage gains.
- Moderate line/branch gains from modules that currently have almost nothing.

Stage target:

- Push line coverage above 75%.
- Push branch coverage above 60%.

## Stage 3: Expand Stateful Encoder/Decoder Matrices

Goal: cover the large decision trees in Opus/SILK.

Actions:

- Add parameter-matrix tests for:
  - sample rates: 8k, 16k, 24k, 48k
  - mono and stereo
  - frame sizes: 2.5ms, 5ms, 10ms, 20ms, 40ms, 60ms where valid
  - applications: audio, voip, restricted lowdelay
  - controls: VBR, FEC, DTX, packet loss, bandwidth, force channels, force mode
- Exercise both Rust-only roundtrips and C-vs-Rust comparison helpers where stable.
- Add decode-side tests that explicitly hit PLC and FEC branches.

Expected effect:

- Biggest branch-coverage improvements.
- Largest remaining line gains in `silk::encoder`, `silk::decoder`, `opus::encoder`, and `opus::decoder`.

Stage target:

- Push line coverage above 90%.
- Push branch coverage above 80%.

## Stage 4: Branch Closure Sweep

Goal: close out stubborn uncovered decisions.

Actions:

- Use missing-line and per-file branch reports to target specific holes.
- Add compact tests for:
  - invalid argument paths
  - boundary values
  - optional CTL toggles
  - alternate mode-selection branches
  - low-bitrate/high-bitrate transitions
- Focus on files with the most uncovered branches remaining after Stage 3.

Expected effect:

- Smaller but necessary gains.
- Converts "touched but not explored" code into genuinely covered decision trees.

Stage target:

- Push all four metrics as high as practical.
- Continue only while each test batch produces meaningful gains.

## Decision Rule For 98%

98% remains the explicit goal, but the baseline gap is so large that each stage needs a checkpoint. After every stage:

1. Run `cargo +nightly llvm-cov --branch`.
2. Record new totals and highest-risk gaps.
3. Decide whether the next stage is still producing worthwhile movement.

If the curve flattens far below 98%, the blocker needs to be called out explicitly rather than pretending the target was realistic.
