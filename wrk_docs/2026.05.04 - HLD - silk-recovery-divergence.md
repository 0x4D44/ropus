## 1. Classification

This is one bug class with two discovery surfaces:

| Surface | Classification | Current status |
|---|---|---|
| `fuzz_decode` standalone decode | SILK/Hybrid recovery PCM divergence on malformed-but-parseable packet | Known failure marker; SILK/Hybrid PCM SNR oracle is currently disabled on attacker-controlled decode |
| `fuzz_multistream` decode path | Duplicate of the standalone class through family-0 mono single-stream multistream wrapper | Known failure marker; SILK/Hybrid PCM SNR oracle is currently disabled on multistream attacker-controlled decode |

Severity split:

- **Codec/user severity:** low. Both decoders return success and sample-count parity on malformed/recovery-shaped inputs; no safety issue or well-formed roundtrip regression was observed.
- **Fuzz oracle severity:** medium. Disabling the SILK/Hybrid SNR oracle removes coverage that would catch genuine decode drift on attacker-controlled SILK/Hybrid packets. The work should restore that oracle only after it can distinguish comparable coded audio from recovery/DTX concealment.

The multistream fixtures are not an independent multistream decoder bug. `cargo fuzz fmt` shows all four use `mapping_family=0`, `channels=1`, and `op=1` decode mode, so `OpusMSDecoder` reduces to a single `OpusDecoder` path.

## 4. Likely root cause

The immediate cause is oracle overreach, not a proven normal-decode codec regression:

1. Each fixture is packet-parseable enough for both decoders to return `Ok`.
2. Packet-level duration/count checks pass.
3. The second code-2 VBR sub-frame has length 0, which drives `opus_decode_frame` into PLC/DTX recovery.
4. Recovery PCM is not bit-exact or SNR-comparable between implementations for these SILK/Hybrid states.
5. The original energy precheck only filtered quiet recovery output. The standalone loud fixture has enough reference energy to pass that gate, so it trips the 50 dB SNR assertion despite being recovery PCM.

There may still be an underlying parity bug inside the SILK recovery path, but fixing that should be treated as optional hardening unless Arthur wants bit-exact behavior for malformed recovery packets. The current value target is better oracle classification: compare coded audio strongly; avoid pretending concealment audio has a stable SNR oracle.

Suspect code paths for a parity investigation, if we choose to chase the underlying recovery difference:

- `/home/md/language/ropus/ropus/src/opus/decoder.rs`: `decode_native` packet loop and `decode_frame` PLC/DTX branch for `sizes[i] <= 1`.
- SILK decoder PLC/lost-frame handling reached from `decode_frame` for Hybrid/SILK modes.
- C reference comparison points in `/home/md/language/ropus/reference/src/opus_decoder.c`, especially `opus_decode_native` passing `(data, size[i])` and `opus_decode_frame` setting `data = NULL` for `len <= 1`.
- Final range and last packet duration CTLs are useful sanity checks, but they are not sufficient oracles for PCM comparability.

## 5. Investigation plan

Use hypothesis-driven steps, one at a time:

1. **Confirm structural shape for all fixtures.** Use `opus_packet_parse_impl` to record TOC, frame count, payload offset, and per-frame sizes. Expected: all current fixtures are code-2 VBR with a zero-length final frame.
2. **Confirm wrapper collapse.** For multistream fixtures, assert family 0, mono, streams=1, coupled=0, and compare multistream decode output with standalone decode output on the extracted payload.
3. **Trace recovery entry.** Add temporary diagnostics only, not permanent production behavior: log whether Rust and C enter PLC/DTX on the zero-length sub-frame, with mode, bandwidth, previous mode, frame size, `last_packet_duration`, and final range before/after each sub-frame.
4. **Split coded vs recovery output.** Decode first non-empty sub-frame alone and compare PCM/final range. Then run explicit PLC after the first frame. If the coded first frame is comparable and divergence starts in the recovery frame, the oracle gate can be justified without needing a codec fix.
5. **If parity is required, descend into SILK PLC.** Compare SILK decoder state before the zero-length frame, then inside lost-frame concealment. Stop at the first divergent scalar/state block instead of broad refactoring.

This investigation should not start by changing numeric code. The first deliverable should be a small diagnostic report proving where coded decode ends and recovery decode begins.

## 6. Proposed oracle strategy

Introduce an explicit comparability classifier for fuzz decode outputs:

```text
DecodeOracleClass::CeltCodedComparable
DecodeOracleClass::SilkHybridCodedComparable
DecodeOracleClass::RecoveryOrDtxOnly
DecodeOracleClass::ErrorAgreementOnly
```

Proposed behavior:

| Class | Oracle |
|---|---|
| CELT coded comparable | sample-count parity + byte-exact PCM |
| SILK/Hybrid coded comparable | sample-count parity + `SNR >= 50 dB` when reference energy is high enough |
| Recovery/DTX-only | error symmetry + sample-count parity only |
| Both error | accept, with optional error-code parity if a future task wants it |

Classification rule for this known class:

- If packet parsing yields any frame size `<= 1` on a SILK/Hybrid packet, treat the whole decoded output as `RecoveryOrDtxOnly` for attacker-controlled decode targets.
- Preserve the stronger SNR oracle for roundtrip paths where the packet was produced by the encoder in the same iteration; those packets are well-formed by construction and already use `snr_oracle_applicable_for_packet(..., true)` in relevant roundtrip code.

For multistream:

- Parse each sub-packet using the existing multistream sub-packet parsing path.
- Classify each stream independently.
- Overall multistream decode is SNR-comparable only if every active SILK/Hybrid sub-packet is coded-comparable; one zero/one-byte recovery sub-frame downgrades the PCM oracle to sample-count parity.

Naming matters: do not call this a generic "well formed" gate. Zero-length DTX-style frames can be structurally legal. The property we need is narrower: **SNR-comparable coded audio**.

## 7. Fix strategy options

Recommended path:

1. Add a fuzz-oracle helper that exposes packet frame sizes and returns the comparability class.
2. Re-enable SILK/Hybrid SNR in `fuzz_decode` only when the helper returns `SilkHybridCodedComparable`.
3. Apply the same classifier to `fuzz_multistream` decode by classifying each sub-packet.
4. Keep the five known-failure fixtures as regression markers that must route to `RecoveryOrDtxOnly`, not panic.
5. Add focused tests covering:
   - loud standalone fixture routes to `RecoveryOrDtxOnly`;
   - four multistream fixtures route to `RecoveryOrDtxOnly`;
   - normal encoder-produced SILK/Hybrid packet still routes to `SilkHybridCodedComparable`;
   - CELT-only packet remains byte-exact.

Optional later hardening:

- If exact recovery parity becomes a product goal, add a dedicated SILK recovery trace harness and fix the first divergent recovery-state field. This should be a separate task because it risks touching parity-sensitive decoder code for low product payoff.

Rejected path:

- Raising `SNR_PRECHECK_MIN_REF_ENERGY` again. The loud fixture already proves energy is the wrong axis. More energy tuning will either keep false positives or hide real loud coded drift.

## 9. Risks

- **Under-classification risk:** A real SILK/Hybrid coded drift could be incorrectly routed to sample-count-only if the classifier is too broad. Keep the downgrade rule narrow: frame size `<= 1` or proven recovery/DTX entry, not "all malformed-looking packets."
- **Over-classification risk:** Re-enabling SNR on packets that still enter recovery will recreate the current false positives.
- **Multistream parsing risk:** Family 1/2/255 sub-packet framing is more complex than family 0. Use existing parser behavior instead of duplicating ad hoc byte walking.
- **Terminology risk:** "Malformed" is imprecise here. Some zero-size DTX/recovery shapes may be structurally accepted. The oracle should classify comparability, not moral validity.
- **Parity-fix risk:** Chasing exact SILK recovery output can spill into fragile decoder state. Do it only with a trace oracle and a small repro, not as cleanup.
