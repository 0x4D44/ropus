# HLD - DRED LPCNet Feature Drift Diagnostic

Date: 2026.05.06

Scope: Phase 3.1 of `/home/md/language/ropus/wrk_docs/2026.05.05 - PLN - testing improvement plan.md`. This is a diagnostic-only slice for the existing ignored DRED byte-parity test in `/home/md/language/ropus/harness-deep-plc/tests/dred_encode_payload_diff.rs`. It does not change codec, DNN, or DRED behavior. It does not unignore the existing exact-byte-parity test.

This HLD lives next to and depends on the named blocker in `/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - dred dnn parity gate.md`: "Stage 7 LPCNet `Ex[]`/FFT-path bit-exactness drift". That blocker is currently a hand-wave. This slice replaces the hand-wave with a measured, locked-bound active oracle.

## Goal

Make drift at the LPCNet feature-extraction boundary between ropus and the xiph C reference a measured, locked, automation-visible number rather than an `#[ignore]` comment. The diagnostic bounds the *output* of LPCNet feature extraction; it does not localise drift to a specific sub-step (preemphasis, FFT, `Ex[]`, bands, cepstrum, pitchdnn) inside LPCNet. Give the next remediation slice concrete starting numbers — max abs drift, RMS drift, drifting-frame count, first-divergent frame index, AND a per-feature-index max-drift breakdown so the next agent can see *which* feature dimension drifts worst without re-instrumenting.

## Supervisor revisions (post-devil's-advocate)

Five refinements baked in here. The rest of the HLD remains as written; only the items below are amended.

### Revision A — soften "where drift originates"

The diagnostic only inspects the post-`compute_latents` 36-wide feature vector, which is the *output* of the entire LPCNet feature pipeline. Calling that "where drift originates" overstates the result. The new wording (above and throughout) is "drift at the LPCNet feature-extraction boundary".

### Revision B — per-feature-index max-drift array

Add a fifth metric to `LpcnetFeatureDriftReport`: `per_feature_max_abs_drift: [f32; NB_TOTAL_FEATURES]` — for each of the 36 feature indices, the max abs drift over all compared frames. This is purely additive: no new assertion (the aggregate `max_abs_drift` already covers correctness); the per-feature array ships in the printed report and the failure context. Cost: ~10 LOC. Benefit: the next remediation slice sees exactly which feature dimension drifts worst (e.g. cepstrum band 7) without re-instrumenting.

### Revision C — reproducibility before locking

Add an acceptance criterion: the implementation agent must run the diagnostic three consecutive times with `--nocapture` on the lock host and confirm the four primary numerics (`MAX_ABS_FEATURE_DRIFT`, `MAX_RMS_PER_FRAME_DRIFT`, `MAX_DRIFTING_FRAME_COUNT`, `FIRST_DIVERGENT_FRAME_AT_LEAST`) are identical across all three runs before locking. The per-feature-index array (Revision B) need not be checked across runs — it's auxiliary report data, not gated. Mirror discipline from the corpus-slice empirical-reproducibility check.

If the three runs disagree, the implementation agent stops, journals the divergence, and surfaces to Arthur. The supervisor does NOT have re-lock authority for a reproducibility miss.

### Revision D — `n_compared_frames` minimum

Promote Risk #5's escalation threshold to an acceptance criterion: `report.n_compared_frames >= 4`. `> 0` is too weak (a single-frame trivial pass would satisfy it). With the existing 1-second 440 Hz fixture and ~50 Opus frames, far more than 4 frames produce a comparison; this is a backstop, not a tightening.

### Revision E — drifting-count definition explicit in panic context

In the failure-context helper, the line for `drifting_frame_count` must annotate the bit-exact definition explicitly, e.g.:

```text
drifting_frame_count = <N> / <M>  (frames where any of 36 features differs by > 0.0 absolute; bit-exact under -ffp-contract=off)
```

So a future reader doesn't try to "relax" the count with an epsilon and inadvertently weaken the oracle.

## Stakeholder Value

The strongest current DRED parity oracle, `dred_encode_silk_frame_bytes_match_c_reference`, is `#[ignore]`'d behind a bare comment that names "Stage 7 LPCNet drift" without measuring it. A normal `cargo test` therefore cannot regress on the feature drift in either direction:

- A drift increase from a Rust DNN refactor is invisible until somebody manually unignores the test.
- A drift improvement (e.g. someone fixes the FFT path) is invisible until somebody manually re-runs the ignored test.

This slice replaces that comment with an active, locked-bound diagnostic test. After this slice:

- A green run means "ropus LPCNet features are no farther from the C reference than the locked baseline".
- A red run means "the upstream blocker for byte-exact DRED payload parity changed; investigate".
- The next agent who chooses to remediate has four concrete numbers to compare against, not a prose phrase.

The active oracle joins the non-quick `--release-preflight` neural/DRED gate per the policy in `/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - dred dnn parity gate.md` lines 70-87.

### Measured-And-Locked Bounds

The HLD intentionally does not pin numerics. The implementation agent records the four values on the first `--nocapture` run, then locks them as the constants above:

| Metric | What it measures | Lock-source |
|---|---|---|
| `MAX_ABS_FEATURE_DRIFT` | Max over all `(frame, feat_idx)` of `|c[i][j] - r[i][j]|`. f32. | First measurement. |
| `MAX_RMS_PER_FRAME_DRIFT` | Max over frames of `sqrt(mean_j ((c[i][j]-r[i][j])^2))`. f32. | First measurement. |
| `MAX_DRIFTING_FRAME_COUNT` | Count of frames where any of 36 features is bit-different (any non-zero abs diff). usize. | First measurement. |
| `FIRST_DIVERGENT_FRAME_AT_LEAST` | Lower bound on the index of the first feature-divergent frame. usize. If features diverge from frame 0, this is 0. | First measurement. If observed first divergence is N, lock as N (no margin). |

Margin policy: lock the bounds at the observed values with no slack. Re-locking requires explicit Arthur signoff, mirroring the PLC drift policy (`/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - plc recovery drift oracle.md` lines 240-243). A drift improvement (e.g. the FFT path is fixed and observed drift drops) is a normal codec change handled by re-running this test under a follow-up slice that updates the bounds downward and references the responsible commit.

CPU FP determinism: `ffp-contract=off` is already in effect for the harness build (`/home/md/language/ropus/harness-deep-plc/build.rs:287`), so scalar float paths agree bit-for-bit on amd64 between gcc/clang and rustc. The Rust side does not introduce SIMD in the LPCNet feature extractor today. The agent must record the measurement on the same hardware Arthur uses for `--release-preflight`. If a future cross-platform agent observes a strictly larger drift on a different CPU, that is a new finding worth a follow-up slice rather than a silent re-lock — record the divergence in `wrk_journals/` and ask before widening.

## Risks And Mitigations

1. **CPU-FP determinism vs locked bounds**. Risk: the locked numerics are deterministic on Arthur's amd64 dev box but might shift on a different host (different gcc, glibc math, or future Rust libm). Mitigation: `-ffp-contract=off` is already enforced in `harness-deep-plc/build.rs:287`, the LPCNet path on the Rust side has no SIMD today, and both sides read the same compile-time RDOVAE/pitchdnn tables via the embedded blob. Document the lock-host in the implementation journal. Re-locking on a different host requires explicit Arthur signoff (no automatic widening).

2. **FFI shim might need extension**. Risk: a future per-LPCNet-call (vs per-Opus-frame) granularity might require new C-side getters. Mitigation: this slice's granularity is per-Opus-frame snapshots of the most-recent feature vector, which both sides already expose. Confirmed by inspection at `harness-deep-plc/dred_encode_shim.c:136` and `ropus/src/dnn/lpcnet.rs:1545`. No FFI extension is needed in this slice. If the next remediation slice needs the intermediate `f0` vector (the first of two per `process_frame`), that is a separate FFI-extension slice with its own HLD.

3. **Heavy harness-deep-plc rebuild loop during iteration**. Risk: edits to test code in this crate trigger full C re-link of the DRED + LPCNet sources (`harness-deep-plc/build.rs:243-307`). Mitigation: keep all edits in the new test file; do not touch `build.rs`, the shim C, or the lib.rs FFI surface. Run focused per-test invocations (`cargo test -p ropus-harness-deep-plc --test dred_lpcnet_feature_drift`) to scope re-linking. Helper unit tests run inside the same crate so compile cost is amortised.

4. **Over-tightening risk: locking against a non-drift regression**. Risk: locked bounds might trigger on an unrelated change (e.g. a future toolchain bump, a `-ffp-contract=off` removal, a `cc` upgrade). Mitigation: the test's failure context must include observed-vs-locked for all four metrics, the first eight mismatch tuples, and the fixture parameters, so a reviewer can quickly tell whether observed drift increased meaningfully. Re-locking is allowed on explicit Arthur signoff with a journal entry naming the responsible commit and the new bounds. Do not relax bounds to make the test green without that signoff.

5. **Risk that `c_fill > 0` is rare on this fixture**. Risk: if the encoder buffers up many frames before any chunk completes, only a handful of frames produce a comparison, undermining the diagnostic. Mitigation: the existing byte-parity test at `dred_encode_payload_diff.rs:227-398` confirms `c_fill > 0` for several frames within the 1-second 440 Hz fixture, and the encoder is initialised identically on both sides. The acceptance criterion requires `n_compared_frames > 0`. If the implementation agent finds fewer than 4 compared frames, escalate to Arthur; the fixture or the granularity may need revisiting (out of scope to change here).

6. **Risk of misreading the bounded oracle as parity**. Risk: a future reader sees "feature-drift-bounded" green and assumes byte parity. Mitigation: oracle matrix wording explicitly cross-references the ignored byte-parity row; the new test name contains "feature_drift" not "parity"; the test panic context labels each metric; the ignored byte-parity test's `#[ignore]` reason now explicitly names the active diagnostic and quantifies the blocker.
