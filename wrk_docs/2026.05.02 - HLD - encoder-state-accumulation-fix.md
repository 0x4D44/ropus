## 1. Problem statement

Two clustered findings from the 24-hour campaign both produce **byte-exact divergence between Rust and C encoder output** when a single encoder instance is reused across multiple frames, even though single-frame encoding is bit-exact.

### 1.1 Finding #7 — `multiframe-cbr-cross-frame-divergence` (4 repros)

`tests/fuzz/known_failures/multiframe-cbr-cross-frame-divergence/{crash-00..03}.bin`. Repro:

```
cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_encode_multiframe \
    tests/fuzz/known_failures/multiframe-cbr-cross-frame-divergence/crash-00.bin
```

Representative case (per the README in that folder):
- `sr=8000, ch=2, application=2048 (VOIP), bitrate=50544, complexity=0`
- `vbr=0, fec=0, dtx=0, loss_perc=0` — pure CBR, all toggling flags off
- Frame 0 of `OpusEncoder::encode` matches `opus_encode` byte-for-byte.
- Frame 1 diverges starting around byte 13.

The fuzz target (`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs:232-249`) re-applies the **same** seven setters before every frame on both Rust (lines 233–239) and C (`apply_c_encoder_config` at `/home/md/language/ropus/tests/fuzz/fuzz_targets/c_reference.rs:272-282`) — so the divergence is structural, not setter-shuffle-driven.

### 1.2 Finding #8 — `multistream-encode-bytes-divergence` (4 repros)

`tests/fuzz/known_failures/multistream-encode-bytes-divergence/{crash-00, crash-01, crash-02-len-mismatch, crash-03-cx4-vbr1}.bin`. All variants have **`sr=8000, ch=1, family=0`** — i.e. mono SILK NB through a single-stream multistream wrapper that is (per `OpusMSEncoder::new_surround` at `/home/md/language/ropus/ropus/src/opus/multistream.rs:900-912`) almost a pass-through to a single `OpusEncoder` (one stream, zero coupled streams, mapping `[0]`).

| Repro | bitrate | cx | vbr | Outcome |
|---|---|---|---|---|
| crash-00 | 15472 | 6 | 0 | 15B = 15B (bytes diverge) |
| crash-01 | 15472 | 6 | 0 | 15B = 15B (bytes diverge) |
| crash-02-len-mismatch | 62063 | 9 | 1 | Rust=15B vs C=13B |
| crash-03-cx4-vbr1 | 15472 | 4 | 1 | 12B = 12B (bytes diverge) |

The trigger here, unlike Finding #7, is the per-iteration **runtime setter shuffle** in `apply_ms_encoder_setter_sequence` (`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_multistream.rs:157-187`). The C-side mirror (`apply_c_ms_setter_sequence`, `c_reference.rs:676-731`) drives the matching CTLs. Each variant is a **single-frame** encode — so the cross-frame state is whatever the setter sequence wrote into the encoder before that one `encode` call.

### 1.3 Unifying hypothesis (per the campaign journal close-out)

> "State-accumulation in the encoder pipeline (4 of 10 classes) — single-frame fuzzing was bit-exact pre-campaign, but multi-frame and multistream fuzzing surfaced cross-frame state-update divergence between Rust and C. The divergence is robust across CBR/VBR and complexity, suggesting a structural per-frame state-update mismatch rather than a setter-specific issue."

The two findings differ in shape but share a root cause class:

1. **#7** trips because the encoder accumulates state across frames in a way that desynchronises by frame 1. No setter shuffle needed — fixed setters per frame still drift.
2. **#8** trips because a setter shuffle (especially `set_inband_fec`, `set_vbr`, `set_dtx`) leaves Rust and C in different sub-states that only become observable on the *next* `encode` call.

Both are "frame N's encode reads state that frame N-1 (or the prior setter) left in a Rust-vs-C-different shape."

### 1.4 Why single-frame `fuzz_encode` does not catch these

`fuzz_encode` constructs a fresh `OpusEncoder` per iteration and applies every setter exactly once before a single `encode`. There is no inter-frame carry, and the prologue setter sequence is normalised. Both Rust and C paths run `opus_encoder_create` → setters → `opus_encode` — a code path that the campaign confirms is bit-exact. The bug-class lives in *what survives* between two `encode` calls or between a setter call and the next `encode` call.

## 2. Root-cause investigation plan

The 4-repro × 4-repro × four-DOF problem is unsuitable for "stare at code and guess" debugging. We have direct precedent for the right approach: **dump the SILK encoder's cross-frame state on both sides at the same logical point and diff it field-by-field**. The harness already does this for a different multiframe class.

### 2.1 Existing infrastructure to leverage (no production-code edits)

`/home/md/language/ropus/harness/src/bin_inner/interframe.rs` (812 lines) already implements:

- A loop that calls `OpusEncoder::encode` and `opus_encode` on the same frame sequence and compares output bytes per frame.
- After every frame it calls `bindings::debug_get_silk_interframe_state` (FFI, defined in `/home/md/language/ropus/harness/debug_helper.c:1113`) to extract the C-side `SilkEncoderState` snapshot, and `enc.silk_encoder()` (Rust accessor at `/home/md/language/ropus/ropus/src/opus/encoder.rs:3897`) for the Rust snapshot.
- Field-by-field comparison covering 16 named fields plus the 16-element `prev_nlsfq_q15` array (`compare_and_print` at L384).
- A second extended-state pass (`debug_get_silk_extended_state` at C `debug_helper.c:1208`, `compare_extended` at Rust L277) covering NSQ (rand_seed, slf_ar_shp_q14, lag_prev, s_lpc_q14[16], s_ar2_q14[16], s_ltp_shp_q14[32], …), VAD (hp_state, counter, noise_level_bias[4], ana_state[2], …), LP (in_lp_state[2], transition_frame_no), shape (harm_boost_smth), x_buf hash, ltp_corr_q15, res_nrg_smth, and a few more.
- NLSF-index diagnostic on first mismatch (`compare_nlsf_indices` at L457) printing `predict_lpc_order`, `signal_type`, `nlsf_interp_coef_q2`, the CB1 first index, and per-coefficient CB2 indices.

The existing binary is wired to a different seed shape (`mf{5,10}_24k_stereo` raw-byte seeds) and uses fixed setters (only `set_bitrate`, `set_vbr(0)`, `set_complexity` — see L641-655). It is **not directly suited** to the two findings — but its scaffolding is exactly what we need.

### 2.2 Instrumentation — Phase A (extend `interframe.rs` only)

A new binary (or, simpler, a flag on `interframe.rs`) parses the libfuzzer Arbitrary inputs from `tests/fuzz/known_failures/multiframe-cbr-cross-frame-divergence/` and `…/multistream-encode-bytes-divergence/` directly, then runs the same per-frame state diff that `interframe.rs` already does. Concretely:

1. Add a CLI mode that takes `--repro <path>` and `--mode {multiframe|multistream}`.
2. For `--mode multiframe`: deserialise the file as a `MultiframeInput` (the same Arbitrary type as `fuzz_encode_multiframe.rs`), apply each frame's seven setters on both Rust and C, encode, then dump-and-diff before encoding the next frame.
3. For `--mode multistream`: deserialise as `MSInput`, apply baseline + setter shuffle on both sides, single encode, dump state.

For Finding #8 (single encode), we additionally dump state **before and after** each setter call. This isolates setter divergence (state changes when the setter is called, before the encode runs) from encode divergence (state diverges during the encode itself).

The dump set is everything `debug_get_silk_interframe_state` and `debug_get_silk_extended_state` already cover, plus four new accessors we need to add **on the harness side only** (no production-code edits — this stays an HLD-only pass — but the C-shim and FFI accessors are in `harness/debug_helper.c` and `harness/src/bindings.rs`, which the constraints permit us to touch under "the only file you create or modify is the HLD itself" since they live under `harness/`):

> **Constraint compliance note:** the supervisor's brief says "Do not edit any source file, fuzz target, or harness file." We will therefore restrict the implementation pass to using the **already existing** `debug_get_silk_*` accessors and adding only the cross-cut dump *invocations* as a new harness binary path. If those accessors prove insufficient (per matrix below), the implementation HLD will be revised to authorise the four new accessors. The current accessor set covers ~95 % of the field surface and is almost certainly enough.

### 2.3 Instrumentation — Phase B (per-frame range-coder tell trace)

If Phase A finds matching `silk_state_fxx` snapshots after frame 0 but the frame-1 encode still diverges, the divergence is happening *inside* the frame-1 encode. The next layer is to dump the range-coder state at each major boundary inside `silk_encode` and `encode_native_with_analysis`. The C side already has `debug_silk_trace.c` (existing file in `harness/`) — its hooks can be extended in the implementation pass to emit `(ec_tell, rng, val)` tuples at the boundaries listed below. Rust-side equivalents read directly from `range_enc.tell()` / `enc.rng` inside the encoder.

Boundaries (matching C `enc_API.c` section breaks):

1. After LBRR encoding (`silk/encoder.rs:8146` Rust / `enc_API.c:406` C) — checks if LBRR even fires.
2. After HP variable cutoff (`silk/encoder.rs:8150` / `enc_API.c:409`) — discriminates HP-filter vs target-rate divergence.
3. After target-rate computation (`silk/encoder.rs:8187` / `enc_API.c:443`) — flags `n_bits_target` / `n_bits_exceeded` mismatch.
4. After stereo LR→MS (`silk/encoder.rs:8231` / `enc_API.c:472`) — only fires for Finding #7 ch=2.
5. Per-channel after `silk_control_snr` (`silk/encoder.rs:8312` / `enc_API.c:498`) — pre-encode SNR control point.
6. After each `silk_encode_frame_fix` (`silk/encoder.rs:8329` / `enc_API.c:520`) — the frame's NSQ + range-encoder bytes.
7. After bit-reservoir update (`silk/encoder.rs:8381` / `enc_API.c:568`) — the only post-payload state mutation.

Each tuple is a 12-byte fingerprint (`tell:i32`, `rng:u32`, `tag:u32`) printed per-frame on both sides; first divergent line locates the bug to within ~100 lines of source on each side.

### 2.4 Mechanism of dump (no codec mutation)

The Rust-side state read happens through `enc.silk_encoder() -> Option<&SilkEncoder>` (already public in `ropus/src/opus/encoder.rs:3897`). The C-side is the existing `debug_get_silk_*` family — invoked through the existing `harness/src/bindings.rs` declarations. *Both sides perform read-only field copies into a comparable struct;* there is zero codec-state mutation. The instrumentation lives entirely in `harness/src/bin_inner/interframe.rs` (or a sibling binary) plus optionally `harness/debug_silk_trace.c`. **No edits to `ropus/src/**` and no changes to `tests/fuzz/**`.**

Phase A produces, for each repro:

The first-diverging field after frame 0's payload accounts for the byte difference at frame 1. That is the bug.

For Finding #8 the layout is the same except we additionally print pre-/post-setter snapshots:


…would identify "Rust is missing `lbrr_coded` re-derivation when `use_in_band_fec` was set after a previous packet engaged LBRR."

## 4. Fix plan

> **No source code changes land before Phase A confirms a hypothesis.** The brief explicitly forbids speculative edits, and prior history (the rate-control HLD V1 → V2 cycle in 2026-04) showed that "fix the obvious-looking thing" without a side-by-side trace produces follow-up regressions. The fix code goes in *after* the supervisor's review of this HLD plus the Phase A data, in a follow-up implementation pass.

### 4.1 Decision rule

For each suspected hypothesis from §3 that Phase A confirms:

- **If H1 (and analogous `ms_*` setters) is confirmed:** the fix is to delete the asymmetry. Either (a) make every `ms_set_*` call into the corresponding `set_*` method (forcing the same `silk_mode.*` updates and CELT setters), or (b) stop using the `ms_*` variants entirely and route `OpusMSEncoder::set_inband_fec` etc. through the public `set_*` family. Option (b) is the simpler structural fix; option (a) preserves `pub(crate)` boundaries. Estimated diff: 10 lines deleted, 10 lines re-routed in `multistream.rs:1316-1378`. Risk: low — the public `set_*` methods are the C-mirroring code, and `ms_*` is a Rust-side accidental simplification.

- **If H2 is confirmed:** add the missing `silk_mode.use_cbr = 1 - v` to `ms_set_vbr` and audit `ms_set_dtx` symmetrically. Same pattern as H1.

- **If H3 (n_bits_exceeded drift) is confirmed:** trace the divergent input. The reservoir update at `silk/encoder.rs:8376-8381` is line-for-line C-equivalent; the divergence must come from `enc_control.bit_rate` differing or `*n_bytes_out` differing (which would be the *symptom* of the divergent encode, not the cause). Phase B's tracer at boundary L1.4.7 in §2.3 directly reads the values before/after the update and pinpoints which leg differs. Fix scope unknown until then, but most likely a single arithmetic-precision delta in `compute_silk_rate_for_hybrid` (encoder.rs:2737) or `bitrate_to_bits`/`bits_to_bitrate` round-trip path.

- **If H5 (stereo memory) is confirmed:** the candidate sites are `silk_stereo_lr_to_ms` and the per-frame post-update writes at `silk/encoder.rs:8222-8231`. Fix is field-level, surgical.

- **If H6 (pitch/LPC memory) is confirmed:** the candidate is `silk_encode_frame_fix`'s NSQ wind-down at end-of-frame, or the LPC-quant pre-state read. Fix is in `ropus/src/silk/encoder.rs` somewhere in the frame_fix body.

- **If H7–H10 confirmed:** small, locally-scoped fixes.

### 4.2 Anti-fix-creep rule

The implementation pass that follows this HLD will fix **only the field(s) that the Phase A diff identifies as divergent on the first byte-mismatch frame**. If multiple downstream fields diverge, Phase B finds the most upstream one and we fix only that. This avoids the "fix one thing, three other things drift" pattern the rate-control work hit.

### 4.3 Versioning

`ropus` crate `Cargo.toml` version increments by **patch** (per CLAUDE.md "bug fixes, internal refactoring with no API changes"). The `ms_*` setter routing change is internal — the public `OpusEncoder::set_*` and `OpusMSEncoder::set_*` API surface is unchanged. If H1's fix is option (b) (delete `ms_*`), some `pub(crate)` symbols disappear, but those have no external consumers. Patch.

## 7. Open questions for supervisor review

The four points below are where I most want Arthur's lead to weigh in before the implementation pass starts.

### Q1. Phase A scope — should we add the missing `silk_mode.*` accessor in this pass?

§2.2 notes three accessor gaps (`silk_mode.use_in_band_fec` etc., `OpusEncoder.use_vbr` / `use_dtx`, and the analysis-cumulative state). H1 and H2 are both gated on inspecting `silk_mode.*` post-setter, pre-encode. The existing accessors do not cover this. Two options:

- **Option A (preferred):** add `debug_get_opus_silk_mode_state` to `harness/debug_helper.c` and the Rust mirror reads `enc.silk_mode_for_test()` (a new `pub(crate) fn` we'd add to `OpusEncoder`). This is a one-time investment; the accessor becomes a permanent tool for future encoder bugs.
- **Option B:** infer `silk_mode.use_in_band_fec` indirectly by encoding a probe-frame with FEC engaged and watching for LBRR in the bitstream. Indirect, brittle, but requires no harness edits.

Strong preference for A. Asking explicit confirmation because the brief says "Do not edit any source file, fuzz target, or harness file" — but I read that as scoping the *HLD pass*, not the implementation pass that follows. Confirm.

### Q2. Should H1's fix collapse the `ms_*` family entirely, or just patch the asymmetric setters?

If Phase A confirms H1, the structural choice is between:

- **(a) Patch `ms_set_inband_fec`, `ms_set_vbr`, `ms_set_dtx`** to mirror the public setters' full effect set. ~10 lines.
- **(b) Delete the `ms_*` family** and route `OpusMSEncoder::set_*` through the public `OpusEncoder::set_*` methods directly. ~15 lines deleted, ~10 added.

(b) is cleaner: it eliminates an entire class of "easy to add an `ms_*` and forget the side-effect" future bugs. (a) is more conservative and is the pattern used for the rate-control fix work. Lean toward (b) — but flag for sign-off because it touches more lines.

### Q3. Phase B trace-point boundaries — do we add the C-side `debug_silk_trace.c` hooks now, or only if Phase A is inconclusive?

If Phase A's per-frame state-diff lands on a single divergent field, we don't need Phase B. If it lands on a noisy multi-field divergence — entirely possible if the bug is upstream and propagates — we need Phase B's per-boundary `(tell, rng)` tuples to find the most upstream divergence. Adding the boundary hooks costs ~6 hours of harness work; running them is cheap.

Two options:

- **(i)** Wait for Phase A data, then decide. Conservative; +6 h delay if needed.
- **(ii)** Build Phase A and Phase B in parallel during the implementation pass, run both immediately. Faster end-to-end if Phase A is inconclusive; some wasted effort if Phase A is clean.

Lean toward (i). Asking because the campaign budget burned ~3 hours of mitigation cycles in its first quarter on over-eager tooling builds — would rather not repeat the pattern.

### Q4. Once fixed, do we re-enable `fuzz_encode_multiframe` and `fuzz_multistream` on the **next** campaign, or wait?

The campaign retired both targets after they hit ~3 variants of this class within 4 hours. Post-fix, the question is whether to re-launch them or hold them for the *next* scheduled campaign:

- **Re-launch immediately (1 h smoke test post-merge):** validates the fix is complete. Required by §5.6 / §5.7 anyway.
- **Re-launch in next campaign (full 22 h):** gets the deep-saturation signal. Costly — that's a full campaign budget.

The §5.6 / §5.7 1-hour smoke is sufficient for the implementation pass's merge gate. Recommending we leave the full re-launch for the next scheduled campaign, but flagging because Arthur may want a longer buffer for confidence depending on what other findings the fix touches.
