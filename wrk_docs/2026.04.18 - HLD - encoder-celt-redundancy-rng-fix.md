# HLD - Fix OpusEncoder::get_final_range divergence on SILK→CELT redundancy

**Date:** 2026-04-18
**Module:** `ropus/src/celt/encoder.rs` (one-constant change)
**Surfaced by:** xiph/opus `test_opus_encode.c:501` (via capi FFI shim).
**Builds on:** `190d4c6a` (CELT→SILK redundancy primary fix).

## 1. Symptom

After fixing the primary `CELT→SILK` redundancy rng bug (commit `190d4c6a`), the
`encode` conformance test still fails at the first `j=7` iteration (pure-CELT
forced mode, 60 ms stereo, ≥512k bps). The prior agent's instrumented trace
narrowed the failing frame down to:

```
[ENC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x015658e0
[DEC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x099b6b00
```

— `mode=SILK_ONLY, redundancy=true, celt_to_silk=false, to_celt=true` (the
mode-transition-undo + bandwidth-downgrade branch at
`ropus/src/opus/encoder.rs:1486-1487, 1642-1644`). Base SILK range final
matches. Only the `SILK→CELT redundancy` rng diverges.

## 2. Reproducer

Standalone: 2× Hybrid-FB 20 ms stereo frames followed by
`set_force_mode(MODE_CELT_ONLY); set_bandwidth(WIDEBAND);` +
`encode(..., frame_size=2880)`. Asserts
`enc.get_final_range() == ropus_dec.get_final_range() ==
c_ref_dec.get_final_range()`.

Failing trace with byte-level instrumentation:

```
[CELT_ENC_TRACE] entry: nb_compressed_bytes=92 st.vbr=0 st.bitrate=29200 ...
[CELT_ENC_TRACE] after silence detect: nb_compressed_bytes=18 silence=false vbr_rate=0
[CELT_ENC_END]  n=240 start=0 end=17 cc=2 c=2 rng=0x01b99956 nb_compr_bytes=18
[ENC_RED_DONE]  celt.rng=0x01b99956 redundant_rng=0x01b99956 bytes=6a 5e 98 65 e5 13 12 f3 f3 0b 4c cf 78 84 b4 9f 96 c0 00 00 00 00 ... (92 bytes, last 74 all 00)
[DEC_RED_BYTES] 6a 5e 98 65 e5 13 12 f3 f3 0b 4c cf 78 84 b4 9f 96 c0 00 00 00 00 ... (same 92 bytes)
[CELT_DEC_END]  n=240 start=0 end=17 cc=2 c=2 rng=0x03088300 len=92
```

— Encoder writes exactly 18 bytes of real CELT data + 74 bytes of zeros and
reports `redundant_rng = 0x01b99956`; decoder reads the full 92 bytes and
reports `redundant_rng = 0x03088300`. Same bytes, different rng.

## 3. Root cause

**Two sibling modules define `OPUS_BITRATE_MAX` with conflicting values:**

| Module | Constant | Value |
|---|---|---|
| `ropus/src/opus/encoder.rs:39` | `OPUS_BITRATE_MAX` | `-1` ✓ matches C |
| `ropus/src/celt/encoder.rs:58` | `OPUS_BITRATE_MAX` | **`-1000` ✗** |

The C reference (`reference/include/opus_defines.h:212`) defines
`OPUS_BITRATE_MAX = -1`. The opus-level constant was correct; the celt-level
constant was a copy-paste of `OPUS_AUTO = -1000` (defined two lines above in
C) into the wrong symbol name.

### Failure mechanism

1. `OpusEncoder::encode_frame_native` (our fresh-fix at
   `ropus/src/opus/encoder.rs:2539`) calls
   `celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX))` where
   `OPUS_BITRATE_MAX` resolves via `use super::...` to the opus-level `-1`.
2. `CeltEncoder::ctl` handler at
   `ropus/src/celt/encoder.rs:3280-3283` checks
   `if v <= 500 && v != OPUS_BITRATE_MAX` where
   `OPUS_BITRATE_MAX` is the celt-level `-1000`. With `v = -1`:
   `-1 <= 500 && -1 != -1000` is `true` → returns `OPUS_BAD_ARG`. The ctl
   silently fails; `celt.bitrate` is unchanged.
3. `celt.bitrate` retains its previous value — whatever the last `SetBitrate`
   set it to, e.g. `29200` from an earlier Hybrid CELT encode.
4. The subsequent `celt_encode_with_ec(...)` for the 5 ms redundancy enters
   `celt_encode_core` at `ropus/src/celt/encoder.rs:2178`.
5. At line 2237 (`if st.bitrate != OPUS_BITRATE_MAX`), the comparison again
   uses the celt-level `-1000`. With `st.bitrate = 29200`,
   `29200 != -1000` → true → the CBR rate-clamping branch fires:
   ```rust
   nb_compressed_bytes = nb_compressed_bytes
       .min((tmp + 4 * mode.fs) / (8 * mode.fs) - (signalling as i32))
       .max(2);
   enc_ref.shrink(nb_compressed_bytes as u32);
   ```
   For 29200 bps × 240 samples at 48 kHz: the clamp reduces 92 → 18.
   `enc_ref.shrink(18)` narrows the range-encoder storage to 18.
6. Encoder writes 18 bytes of range-coded data into the first 18 bytes of the
   92-byte redundancy slot; `done()` zeros `[offs, 18)`. Bytes `[18, 92)` are
   whatever was there before (zeros from SILK's `done()`).
7. `st.rng = enc_ref.get_rng()` at line 3098 captures the range encoder's rng
   computed over an 18-byte storage ↔ the range coder interprets `end_offs`
   (raw-bit tail pointer) relative to storage=18.
8. The decoder's `celt_decode_with_ec` at
   `ropus/src/opus/decoder.rs:968-975` sees the full 92-byte slice and
   initializes its `RangeDecoder` with `storage=92`. The decoder walks the
   range-coded symbols correctly (they cap out around byte 18), but `rng`
   at the end is a **different function** of `storage`: when the decoder
   ultimately calls `get_rng()`, the rng value reflects `storage=92`, not
   `storage=18`.
9. `redundant_rng` mismatches. `range_final ^= redundant_rng` then
   `self.range_final` mismatches → `test_opus_encode.c:501` aborts.

### Why `j=7` triggers it

The failing frame uses the mode-transition-undo path at
`ropus/src/opus/encoder.rs:1487`: requested `MODE_CELT_ONLY` with bandwidth
downgraded to `SILK_ONLY` + `to_celt=true`. This is the ONLY path in the
encoder that enters the SILK→CELT redundancy branch through
`encode_frame_native` (not `encode_multiframe`) and triggers the bad
`SetBitrate(OPUS_BITRATE_MAX)` call.

### Why earlier tests pass

The CELT→SILK redundancy branch at
`ropus/src/opus/encoder.rs:2395-2417` also calls
`celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX))` with the same bug.
But there, the preceding `ResetState` sets `celt.bitrate = OPUS_BITRATE_MAX`
(celt-level default = `-1000`, per `CeltEncoder::new` line 3153); then the
failed `SetBitrate(-1)` ctl leaves it at `-1000`. At line 2237 the comparison
`-1000 != -1000` is false, so the CBR shrink branch is NOT taken,
`nb_compressed_bytes` stays at `redundancy_bytes`, and the rng matches.
That's why fixing the missing `redundant_rng = celt.rng;` in commit
`190d4c6a` "made the test progress further" — the bug was masked in that
path by the reset-to-default.

The SILK→CELT branch at `ropus/src/opus/encoder.rs:2532-2568` also calls
`ResetState`. So why doesn't the same masking save us?
Looking at the code: SILK→CELT calls `ResetState` FIRST, then
`SetStartBand(0)`, `SetPrediction(0)`, `SetVbr(0)`,
`SetBitrate(OPUS_BITRATE_MAX)`. After `ResetState`, `celt.bitrate = -1000`
(default). Then `SetBitrate(-1)` fails and leaves it at `-1000`. So
`st.bitrate = -1000` at entry to `celt_encode_core`.

Wait — that's the same as the CELT→SILK case. So why does it fail?

Careful re-read: `CeltEncoder::reset()` at
`ropus/src/celt/encoder.rs:3194-3230` does NOT touch `self.bitrate`. It only
clears fields `rng` onward (matching the C `ENCODER_RESET_START` marker at
`reference/celt/celt_encoder.c:90`). So `bitrate` persists across
`ResetState` — including both redundancy paths.

For CELT→SILK: there's only ONE use of `celt.ctl(SetBitrate)` before
redundancy in the encode_frame_native flow: at line 2258
(`celt.ctl(SetBitrate(OPUS_BITRATE_MAX))`) — which is broken the same way.
So `celt.bitrate` was last set to something else (perhaps during a prior
frame's main CELT encode, e.g. 29200).

But we observe CELT→SILK works. So... let me re-check the flow.

Actually looking at the trace prefix:
```
[CELT_ENC_TRACE] entry: nb_compressed_bytes=1275 st.vbr=1 st.bitrate=29200 ...
```
During the warmup Hybrid FB encode (the run of tests that precede the
failing j=7), `st.bitrate=29200` (set by the opus encoder at line 2453).
Because `st.vbr=1` (user_vbr), the VBR path at line 2227 is taken and the
"CBR shrink" branch at 2237 is NOT executed. The VBR branch (line 2806)
computes `nb_avail` from `vbr_rate` independently. So the stale `st.bitrate`
value doesn't harm the VBR path.

For the redundancy encodes, opus sets `SetVbr(0)` → `st.vbr=0`. Now the
CBR branch is taken. With `st.bitrate` leftover from the last Hybrid encode
(`29200`, or from earlier main-CELT) and NOT reset to MAX by the failing
`SetBitrate(-1)`, the CBR clamp fires and shrinks the buffer.

## 4. Proposed fix

Single-line: change
`ropus/src/celt/encoder.rs:58` from
```rust
pub const OPUS_BITRATE_MAX: i32 = -1000;
```
to
```rust
pub const OPUS_BITRATE_MAX: i32 = -1;
```

Add a comment explaining the coincidence with `OPUS_BAD_ARG` (both are `-1`
numerically, but live in disjoint value spaces in the C ABI —
`OPUS_BITRATE_MAX` is a parameter value, `OPUS_BAD_ARG` is a return code).

No other code changes are required: the existing comparisons
`st.bitrate != OPUS_BITRATE_MAX` at lines 2228, 2237, 2251 and the ctl
validation at line 3280 all continue to work correctly against `-1`.

## 5. Correctness argument

1. **Mirrors C reference exactly.** The C header
   `reference/include/opus_defines.h:212` defines `OPUS_BITRATE_MAX = -1`.
2. **Cross-module consistency.** After this fix, both `opus::encoder`
   and `celt::encoder` define `OPUS_BITRATE_MAX = -1`, matching the
   OpusEncoder → CeltEncoder contract: the opus layer passes its value to
   the celt layer, which uses it as both sentinel input (`SetBitrate`
   validation) and as an internal test (`st.bitrate != OPUS_BITRATE_MAX`
   enabling CBR clamping).
3. **ctl no longer silently fails.** With the fix, `SetBitrate(-1)` from
   `OpusEncoder::encode_frame_native` is accepted
   (`-1 <= 500 && -1 != -1` → `-1 != -1` is `false` → no `OPUS_BAD_ARG`).
   The ctl sets `st.bitrate = -1`.
4. **CBR clamp correctly disabled.** In `celt_encode_core`, line 2237
   `if st.bitrate != OPUS_BITRATE_MAX` becomes `-1 != -1` → `false`. The
   CBR clamp branch is skipped. `nb_compressed_bytes` retains the caller-
   supplied `redundancy_bytes` value. `enc_ref.shrink(...)` is not called.
5. **Encoder and decoder agree on storage.** The range encoder operates
   with storage=`redundancy_bytes`; the range decoder (on the other side)
   also initialises with storage=`redundancy_bytes`. `end_offs` semantics,
   `nbits_total`, `offs`, and rng updates all remain symmetric →
   `enc_ref.get_rng() == dec.get_rng()` after identical symbol processing.
6. **No bitstream perturbation in other paths.**
   - Opus-level `OPUS_BITRATE_MAX` is already `-1`; no change there.
   - `CeltEncoder::new` (line 3153) initialises `bitrate = OPUS_BITRATE_MAX`.
     Before: `bitrate = -1000`. After: `bitrate = -1`. Both are "max" in
     their respective (now unified) comparisons; the CBR branch is
     disabled on the fresh encoder in both cases. No behavior change for
     encoders that never call `SetBitrate`.
   - The ctl validation `v <= 500 && v != OPUS_BITRATE_MAX` now permits
     `-1` (previously rejected as out-of-range). This matches C.
   - No other code calls `SetBitrate(OPUS_BITRATE_MAX)` through a path
     where the previous `-1000` semantics mattered (unit tests use the
     local constant, which now evaluates to `-1`; they continue to pass).

## 6. Alternatives considered & rejected

- **Fix the opus encoder to pass `-1000`.** Would make `OpusEncoder →
  CeltEncoder` internally consistent but DIVERGE from C. The opus-level
  public API must expose `OPUS_BITRATE_MAX = -1` (C ABI). So the celt
  layer must either accept `-1` or the opus encoder must translate.
  Translation adds no value and obscures intent.
- **Have opus encoder check ctl return codes.** Good hygiene but doesn't
  address root cause (the ctl rejection itself was spurious).
- **Reset `st.bitrate` via `ResetState`.** Would paper over the bug
  in this specific site but not fix `SetBitrate` ctl behaviour. Any
  other caller that calls `SetBitrate(OPUS_BITRATE_MAX)` would remain
  broken. Also diverges from C reference (C's `OPUS_RESET_STATE` does
  not touch `st->bitrate` — it is before `ENCODER_RESET_START`).
- **Update `SetBitrate` ctl to accept both -1 and -1000.** Hacky,
  ambiguous, and still diverges from C.

## 7. Risk assessment

- **Risk of regression:** Very low. The change is one constant value. All
  existing CELT encoder unit tests reference the local constant
  `OPUS_BITRATE_MAX`, so they automatically pick up the new value; the
  tests' behaviour (validating `SetBitrate(OPUS_BITRATE_MAX)` is accepted)
  remains correct.
- **Risk the fix is wrong:** Very low. The C reference is authoritative
  and defines `OPUS_BITRATE_MAX = -1`. The prior value was incorrect.
- **Confidence root cause is correct:** 5/5. Byte-level tracing shows
  the CBR clamp firing with the stale bitrate, leading to a
  `storage=18` encoder vs `storage=92` decoder — the exact asymmetry
  that explains the final-rng divergence despite identical on-wire
  bytes.
