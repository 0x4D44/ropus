## Recommended Next Single Slice

Implement Phase 2.2 narrowly: add fuzz oracle class debt reporting for the existing shared decode oracle.

Definition of done:

- Each weakened `DecodeOracleClass` has a stable display name and a concise statement of the correctness property that is not asserted.
- The fuzz oracle helper exposes a cheap, testable classification/reporting helper that can summarize class counts from a bounded packet set without invoking libFuzzer campaigns.
- The oracle matrix is updated so `sample-count-only`, `recovery-or-dtx-only`, and `error-agreement-only` are not collapsed into one vague "weakened" bucket.
- Unit tests cover the summary/report boundaries using existing fixtures and synthetic packets.

Likely files:

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs`
- `/home/md/language/ropus/tests/fuzz/tests/oracle_tests.rs`
- `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`
- optional, only if needed: `/home/md/language/ropus/tools/fuzz_run.sh` for a non-campaign inventory/report flag

Suggested HLD focus:

- Keep this as reporting, not codec correctness. The HLD should define the oracle taxonomy, where counts come from, and how the output changes interpretation of fuzz results.
- Avoid making libFuzzer hot paths allocate or print per input. Prefer pure helpers and bounded offline/unit-test reporting.
- Explicitly distinguish three weakened cases:
  - `sample-count-only`: decode shape agrees, PCM value oracle deliberately disabled for attacker-controlled high-band Hybrid.
  - `recovery-or-dtx-only`: packet routes through PLC/DTX-style recovery, so sample count and error symmetry are meaningful but PCM is not a stable SNR oracle.
  - `error-agreement-only`: packet structure is unusable for PCM comparison; only decode result symmetry/shape can be trusted.

Likely blockers and decisions:

- Whether class distribution must be emitted by fuzz campaigns themselves. My recommendation is no for this slice; campaign-time counters add noise and libFuzzer integration complexity. Start with pure classification helpers and an optional bounded report over committed seed/crash inputs.
- Existing fuzz input formats differ by target. Do not try to summarize every target in one pass. Start with packet-oriented decode/multistream classes already covered by `oracle.rs`.
- If the optional shell report is added, it must remain non-campaign and bounded, similar in spirit to `--list` and `--sanity`.

Scoped validation for that implementation slice:

- `timeout 120 cargo test --manifest-path tests/fuzz/Cargo.toml --test oracle_tests`
- `timeout 10 bash tools/fuzz_run.sh --list` if shell output changes
- `timeout 120 cargo fmt --check`
