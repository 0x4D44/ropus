## Likely Root Cause

The test compares raw bytes for a Rust `#[repr(C)]` struct that contains padding:

- `/home/md/language/ropus/ropus/src/opus/analysis.rs:127` defines `AnalysisInfo` as `#[repr(C)]`.
- `AnalysisInfo` has 63 bytes of explicit fields:
  - 11 four-byte scalar fields = 44 bytes.
  - `leak_boost: [u8; 19]` = 19 bytes.
- Because `AnalysisInfo` has 4-byte alignment, `repr(C)` rounds the struct size to 64 bytes, leaving 1 trailing padding byte.
- `/home/md/language/ropus/ropus/src/opus/analysis.rs:635` stores `info: [AnalysisInfo; DETECT_SIZE]`, and `DETECT_SIZE` is 100, so the array includes 100 padding bytes.

`TonalityAnalysisState::new()` at `/home/md/language/ropus/ropus/src/opus/analysis.rs:642` initializes semantic fields, but Rust does not guarantee that padding bytes in the stack object are zeroed. `TonalityAnalysisState::new_boxed()` at `/home/md/language/ropus/ropus/src/opus/analysis.rs:690` explicitly zeroes the heap allocation with `write_bytes`, including padding.

The test currently forms raw byte slices over both objects at `/home/md/language/ropus/ropus/src/opus/analysis.rs:1976` and `/home/md/language/ropus/ropus/src/opus/analysis.rs:1982`, then asserts byte equality. That assertion is stricter than Rust's field-initialization contract and is sensitive to whatever bytes happen to occupy stack padding after earlier aggregate tests.

## Proposed Minimal Fix

Make the test compare semantic fields, not full object bytes.

Minimal test-only approach:

1. Replace the `stack_bytes == heap_bytes` assertion with a typed comparison between `TonalityAnalysisState::new()` and `TonalityAnalysisState::new_boxed()`.
2. Use an exhaustive destructuring helper in the test module with no `..` pattern, so adding future fields forces the helper to be updated at compile time.
3. Keep or add explicit assertions that all public fields are logical zero for both constructors.
4. If the heap-zeroing implementation contract still needs coverage, keep a separate assertion that the `new_boxed()` allocation bytes are zeroed, clearly scoped to the heap constructor and not compared against `new()`.

This preserves the production constructor and removes only the invalid test oracle.

## Risks

- A field-by-field helper can miss future fields if written with `..`; avoid that by destructuring every field explicitly.
- Keeping a raw-byte assertion for `new_boxed()` still relies on unsafe object-byte inspection. It should be limited to the heap-zeroing implementation contract, not used as a semantic equivalence oracle.
- Removing the stack-vs-heap byte comparison means the test no longer proves padding equality. That is intentional because padding equality is not a valid requirement for `new()`.
- If another test depends on raw object representation of `TonalityAnalysisState`, it may have the same padding problem and should be reviewed separately rather than folded into this fix.
