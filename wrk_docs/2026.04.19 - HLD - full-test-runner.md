# HLD: `full-test` runner

**Date**: 2026-04-19
**Scope**: add a `full-test` binary that runs every validation lane and
emits an HTML report, modelled on `C:\language\mddosem\src\bin\full_test.rs`.

## Motivation

Today the validation lanes are scattered:

| Lane                  | Entry point                                                       |
|-----------------------|-------------------------------------------------------------------|
| Formatting            | `cargo fmt --check`                                               |
| Lints                 | `cargo clippy --workspace -- -D warnings`                         |
| Unit + integration    | `cargo test --workspace -- --test-threads=1`                      |
| Conformance           | (subset of above via `tests/conformance/`)                        |
| IETF vectors          | (subset of above; needs fetched vectors in `tests/vectors/ietf/`) |
| Ambisonics roundtrip  | `cargo run --release --bin projection_roundtrip`                  |
| Coverage              | `cargo llvm-cov --workspace`                                      |
| Bench                 | `bash tools/bench_sweep.sh --iters=30`                            |

No single command exercises all of them, and results aren't archived. An
orphan HTML report at `tests/results/full_test_20260410_224113.html` shows
a prior agent produced one ad-hoc, but the generator was never committed.

## Non-goals

- **Fuzz campaigns** — hours-long, scheduled separately (`tools/campaign*_launcher.sh`).
- **Replacement for `cargo test`** — full-test shells out to cargo, doesn't
  replace it. Developers keep using `cargo test` for fast iteration.
- **CI integration** — local dev tool first; wiring to GitHub Actions is a
  later question.
- **Cross-platform parity** — must work on Windows (Arthur's primary box);
  best-effort on Linux; macOS untested.

## Stages

Numbered stages run in order. Any stage's failure is captured and reported
but does not short-circuit the rest (mirrors mddosem's `--no-fail-fast`
ethos). **Exception**: if stage 2's cargo invocation fails to *build* (not
fails a test — fails to compile), stages 3 and 4 are marked "skipped
(upstream build failure)" rather than being attempted, since they depend
on the same compiled artefacts.

**Stage 0 (setup, not in report):**
- Capture git commit / branch / version.
- Pre-flight check: does `tests/vectors/ietf/testvector01.bit` exist? Record the boolean; used by stage 2 to decide whether the `ietf_vectors` test binary gets a clear-skip note vs. a real invocation. No auto-fetch — network in a test runner is surprising.
- Set below-normal process priority (propagates to child cargo invocations).

**Stage 1 — Quality.** `cargo fmt --check`, then `cargo clippy --workspace --all-targets -- -D warnings`.

**Stage 2 — Tests + coverage.** Single cargo invocation:
```
cargo llvm-cov --json --output-path <tmp>.json \
    test --workspace --lib --tests --no-fail-fast \
    -- --test-threads=1
```
Coverage flags go *before* the `test` subcommand; cargo-test flags after; binary-level args after the final `--`. Produces `<tmp>.json` (coverage) and cargo's normal stderr (test results). Coverage denominator excludes `reference/` and `tests/vectors/`.

**Thread policy**: `--test-threads=1` applies uniformly across the workspace. This serializes ropus unit tests unnecessarily (they would parallelize safely) and costs ~10-20 min on a multi-core box, but keeps the runner to a single cargo invocation — deliberately chosen for simplicity.

**Coverage scoping**: `cargo llvm-cov` is invoked with `--workspace` (measures all crates) but the report surfaces **two numbers**: ropus/ alone (the headline, since that's the actual codec) and workspace total (footnote). Computed by filtering the JSON by `filename` prefix.

If stage 0 flagged IETF vectors as missing, the 24 `testvectorNN` tests are filtered via libtest `--skip ::testvector` (deviation from the original `--exclude conformance`, which drops the entire conformance crate — 30 tests including 6 unrelated category probes). The leading `::` pins the substring match to a module-path component boundary. The report carries a note pointing at `tools/fetch_ietf_vectors.sh`. All other conformance tests still run.

**Stage 3 — Ambisonics roundtrip.** `cargo run --release -p ropus-harness --bin projection_roundtrip`. Scraped for the `PASS: all fixtures ...` line. Not a `#[test]` so stage 2 misses it.

**Stage 4 — Benchmark sweep.** Implemented **natively in Rust**, not via `bash tools/bench_sweep.sh`. The full-test binary invokes `ropus-compare bench <wav> --iters 30 --bitrate N` in a loop across the same vector set that the shell script uses, parses the `C encode|Rust encode|C decode|Rust decode` lines, computes the ratios. Eliminates the Git-Bash-on-PATH dependency that breaks bare Windows installs. The shell script stays in the tree (canonical tool for the README numbers at `--iters=50`).

**Stage 5 — Report generation.** Writes `tests/results/full_test_<YYYYMMDD_HHMMSS>.html`.

## Ignored-test policy

`#[ignore]` is used in this repo for **two distinct categories**:
- **Known-residual tests** (e.g. `test_24bit_decode_plc_matches_c_reference` — sub-LSB PLC drift, documented in journals).
- **Diagnostic probes** (field-offset dumps, SNR measurement, Stage 6.4 bit-exactness prints — not conformance tests).

Full-test does **not** pass `--include-ignored`. Including diagnostics in a
pass/fail report is noise; including known-residual tests guarantees a red
report. Ignored tests stay ignored; the HTML notes the count in a
footnote.

## Flags

- `--quick` — skip stages 1 and 4 (stage 2 still runs; if combined with `--skip-coverage`, it downgrades to plain `cargo test`). Useful for pre-commit iteration.
- `--skip-quality` — skip stage 1.
- `--skip-coverage` — downgrade stage 2 to plain `cargo test` (no instrumentation).
- `--skip-benchmarks` — skip stage 4.
- `--skip-ambisonics` — skip stage 3.

No env-var knobs. No feature flags. Minimal surface area.

## PASS / FAIL / WARN semantics

The banner badge at the top of the report distinguishes three states:

- **PASS** — stages 1, 2, 3 all green. Stage 2 "green" means: zero test failures (ignored tests don't count), zero compile errors, clippy/fmt passed.
- **FAIL** — any test failure, any clippy/fmt error, ambisonics roundtrip mismatch, or a stage 2 build failure. The report is still written; the banner says FAIL.
- **WARN** — stages 1-3 green, but stage 4 produced anomalous bench ratios (configurable threshold, default: any vector regressing >15% versus the previous report in `tests/results/`) or coverage dropped (default: no gate; purely informational). Never blocks a PASS on quality-of-code grounds; a WARN is "look at this, but the codec is fine."

Exit code: 0 on PASS, 1 on FAIL, 0 on WARN (so `--quick` pre-commit runs don't spuriously block developers on bench noise).

## Output format

HTML report matches the existing orphan report's structure:

1. **Header** — PASS/FAIL badge, timestamp, version, branch, commit hash (via `git rev-parse --short HEAD`).
2. **Category summary** — CELT / SILK / Opus / DNN / Types / Coverage / Conformance roll-ups.
3. **Phase summary** — per-stage total / passed / failed / skipped / duration.
4. **Module breakdown** — every `<crate>::<module>` with ≥1 test, pass/fail counts.
5. **Failed tests** — bulleted list of fully-qualified test names.
6. **Coverage metrics** — lines, functions, branches, regions (covered / total / percent).
7. **Benchmark table** — per-vector encode and decode C-ref-ratio, matching the README performance table.

Dark theme, inline CSS (matches the 2026.04.10 orphan report). Self-contained HTML — no external assets, viewable in any browser / served from repo without a web server.

## Location & layout

New crate: `full-test/` (workspace member, `name = "full-test"`, binary `full-test`).

Rationale: `ropus-tools-core` is a library for the end-user CLIs; dumping a 1k-LOC test harness there would muddle the role. A dedicated crate keeps dependencies (`chrono`, `colored`, `base64`) out of the shipping CLI surface.

Invocation:

```bash
cargo run --release --bin full-test -- [--quick] [--skip-*]
```

Report lands at `tests/results/full_test_<YYYYMMDD_HHMMSS>.html`. Add `tests/results/` to `.gitignore` (except an optional `.keep` file); history retention is not the goal — reports are disposable artefacts.

## Implementation sketch

~800–1000 LOC total:

- `main.rs` — CLI parsing, priority-class setup, stage dispatch, HTML emission.
- `cargo_parse.rs` — regex parsers for `cargo test` output (`test <name> ... ok|FAILED`, `test result: ...`), plus detection of "could not compile" / "error[E" lines to signal build failure.
- `llvm_cov_parse.rs` — parses the JSON report from `cargo llvm-cov --json`.
- `bench.rs` — runs `ropus-compare bench` per vector, parses timings, computes ratios. Vector set mirrors `tools/bench_sweep.sh`.
- `html.rs` — builds the report. Hand-written HTML; no templating engine.

Dependencies: `chrono` (timestamps), `colored` (console output), `regex` (parsers), `serde_json` (llvm-cov JSON). All already in the workspace's lockfile via other crates.

## Runtime expectations

Ballpark on the current box (Windows, `master`, cold cache):

- Stage 0 (setup): <1 sec.
- Stage 1 (quality): ~1 min.
- Stage 2 (test+coverage): ~45 min (test_opus_encode alone is 41 min; llvm-cov adds ~10-20% instrumentation overhead).
- Stage 3 (ambisonics): ~15 sec.
- Stage 4 (bench, `--iters=30`): ~25 min.
- Stage 5 (report): <1 sec.

Total: ~75 min for a full run. `--quick` collapses to ~45 min (still dominated by test_opus_encode). Expected cadence: daily or pre-release, not pre-commit.

Below-normal process priority via `SetPriorityClass(BELOW_NORMAL_PRIORITY_CLASS)` on Windows, `nice(10)` on Unix. Copies mddosem verbatim.

## Decisions

1. **Bench iters = 30.** Full-test's bench stage is a regression check, not a publication number; the README table stays on `--iters=50` via the standalone `tools/bench_sweep.sh`.
2. **Coverage = report only.** No gating threshold. Mirrors mddosem. A baseline can be introduced later once the first N runs establish steady-state coverage.
3. **Report retention = accumulate.** Reports are ~50 KB; a year of daily runs is under 20 MB. `tests/results/` is gitignored so accumulation has no repo impact.

## Risks & mitigations

- **cargo output format changes** — the `test <name> ... ok` format has been stable since ~2015; low risk. Fallback: read `cargo test --format=json` (requires nightly) as a hardening step if it ever breaks.
- **llvm-cov Windows quirks** — known to be finicky; `cargo llvm-cov` has first-class Windows support but some flags don't carry over. Validate on day 1 of implementation by running the full invocation in an isolated shell before wiring it in.
- **IETF vectors missing** — handled by stage 0 pre-flight; no silent skip.
- **Build failure mid-run** — stage 2 parser detects `error[E` / "could not compile" and signals build-failed rather than test-failed; stages 3-4 skip with a clear upstream-failure note instead of each crashing independently.
- **Concurrent agents modify code mid-run** — if another process touches `ropus/src/` during a 75-min run, cargo test may pick up an inconsistent state and fail. No mitigation in the runner; user re-runs.
- **Toolchain drift is a feature, not a risk** — no `rust-toolchain.toml` is added. If a new stable Rust introduces a lint that breaks `clippy -- -D warnings`, full-test surfacing that is the desired signal; we want to adopt the new lint, not pin around it.

## Success criteria

- `cargo run --release --bin full-test` emits a PASS report on a clean master.
- Report covers all lanes from the "Motivation" table.
- `--quick`, `--skip-coverage`, `--skip-benchmarks` each do the obvious thing.
- Report is self-contained HTML, viewable offline.
- Total binary size (source LOC): ≤1200.
