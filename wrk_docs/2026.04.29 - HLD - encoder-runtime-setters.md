# HLD: Runtime setters on `ropus::api::Encoder` — V3 (final)

**Date:** 2026-04-29
**Author:** Claude (with Arthur)

**Supersedes:** V1 and V2 of this document.

## Changelog from V2

Editorial-only refinements; no design changes. Three review rounds converged
"ready to ship" on V2 with these surgical edits queued:

- **`get_user_bitrate_bps` accessor visibility**: `pub(crate)`, not `pub`.
  It is internal plumbing for the high-level facade, not a public-API method.
- **Mid-stream safety section accuracy**:
  - Renamed the citation from `compute_frame_size` (no such function) to the
    actual read site `encode_native_with_analysis` at `encoder.rs:1727`.
  - Added a caveat that `self.bitrate_bps` is a per-frame derived cache, not
    a source of truth — `Encoder::bitrate()` deliberately reads
    `user_bitrate_bps`, not the cache.
  - Added a sentence noting that `set_inband_fec` and `set_vbr` mutate
    paired SILK fields; `&mut self` exclusivity precludes torn reads.
- **`Bitrate::from_c_int` semantics for `0` and stray negatives**: explicit

  drive-by additions: any later inclusion requires the same HLD-level
  review as this work.
- **Rename clarification**: `get_vbr_constraint` (returns `i32`) →
  `vbr_constraint` (returns `bool`); `get_packet_loss_perc` (returns
  `i32`) → `packet_loss_perc` (returns `u8`). Rename **and** retype.
- **Acceptance wording**: "11 *new* getters" rather than "11 getters",
  to avoid confusion with existing `sample_rate()` / `channels()` /
  `lookahead()`.

## Problem

The high-level idiomatic API at `ropus/src/api.rs:390` exposes only `encode` /
`encode_float` and six getters. Every encoder knob is configured exclusively
via `EncoderBuilder` and frozen at `build()`.

This is wrong for Opus specifically. The codec was designed assuming runtime
mutation: adaptive bitrate streaming (WebRTC, RTCP feedback, TMMBR/TMMBN),
packet-loss-percent tracking link quality, FEC/DTX toggling on push-to-talk
transitions, and complexity adjusting under CPU pressure all require
`&mut self` setters with no encoder rebuild.

Tearing down and rebuilding the encoder mid-stream is not a workaround:
it discards SILK predictor history, CELT pitch state, range-coder context,
NSQ memory, and the LP↔CELT crossover, producing audible artefacts at every
rate change.

The underlying `ropus::opus::OpusEncoder` already implements every CTL
needed (encoder.rs:3466–3622). The high-level facade simply does not
delegate to them. Today the only escape hatch is to drop down to the
low-level type, which is what consumers will do — making the high-level
API dead weight.

## Goal

Add `&mut self` runtime setters and matching getters on `ropus::api::Encoder`
mirroring every `EncoderBuilder` field that is safe to mutate mid-stream.
Each is a thin delegation to the existing `OpusEncoder` method.

## Non-goals

- No **behavioural** changes to `OpusEncoder`, `OpusMSEncoder`, or codec
  internals. One read-only `pub(crate)` accessor (`get_user_bitrate_bps`)
  is added to `OpusEncoder` to expose already-stored state to the facade.
- No changes to `capi/` (already wired via CTLs).
- No new top-level error types; reuse `EncodeError`.
- No mid-stream mutation of fields that would desync framing or restart
  semantic mode (see "Deliberately excluded" below).
- No multistream high-level facade work.

## Design

### Methods to add on `impl Encoder`

All setters return `Result<(), EncodeError>`. Some setters with typed
arguments cannot fail in practice; uniform `Result` is kept for API
symmetry across the surface (the underlying `OpusEncoder` methods are
fallible at the int32 level, which is what the wrapping `from_code`
mechanism exists to handle).

| Setter | Delegates to | Builder counterpart |
|---|---|---|
| `set_bitrate(b: Bitrate)` | `inner.set_bitrate(b.as_c_int())` | `bitrate` |
| `set_complexity(c: u8)` | `inner.set_complexity(c as i32)` | `complexity` |
| `set_signal(s: Signal)` | `inner.set_signal(s.as_c_int())` | `signal` |
| `set_vbr(on: bool)` | `inner.set_vbr(i32::from(on))` | `vbr` |
| `set_vbr_constraint(on: bool)` | `inner.set_vbr_constraint(i32::from(on))` | `vbr_constraint` |
| `set_force_channels(c: ForceChannels)` | `inner.set_force_channels(c.as_c_int())` | `force_channels` |
| `set_max_bandwidth(b: Bandwidth)` | `inner.set_max_bandwidth(b.as_c_int())` | `max_bandwidth` |
| `set_packet_loss_perc(pct: u8)` | `inner.set_packet_loss_perc(i32::from(pct))` | `packet_loss_perc` |
| `set_inband_fec(mode: InbandFec)` | `inner.set_inband_fec(mode.as_c_int())` | *(new)* |
| `set_dtx(on: bool)` | `inner.set_dtx(i32::from(on))` | *(new)* |

Getters (all `&self -> T`):

| Method | Delegates to | Notes |
|---|---|---|
| `bitrate() -> Bitrate` | `Bitrate::from_c_int(inner.get_user_bitrate_bps())` | reads user-set value, round-trips |
| `effective_bitrate_bps() -> i32` | `inner.get_bitrate()` | rustdoc: equivalent of `OPUS_GET_BITRATE` |
| `complexity() -> u8` | `inner.get_complexity()` | |
| `signal() -> Signal` | `Signal::from_c_int(inner.get_signal())` | |
| `vbr() -> bool` | `inner.get_vbr() != 0` | |
| `vbr_constraint() -> bool` | `inner.get_vbr_constraint() != 0` | rename + retype of existing |
| `force_channels() -> ForceChannels` | `ForceChannels::from_c_int(inner.get_force_channels())` | |
| `max_bandwidth() -> Bandwidth` | `Bandwidth::from_c_int(inner.get_max_bandwidth())` | |
| `packet_loss_perc() -> u8` | `inner.get_packet_loss_perc() as u8` | rename + retype of existing |
| `inband_fec() -> InbandFec` | `InbandFec::from_c_int(inner.get_inband_fec())` | |
| `dtx() -> bool` | `inner.get_dtx() != 0` | |

The renames of `get_vbr_constraint` (`i32`) → `vbr_constraint` (`bool`)
and `get_packet_loss_perc` (`i32`) → `packet_loss_perc` (`u8`) are both
**rename and retype** — the previous `i32` returns leaked the C ABI; the
new types are honest. They align with the existing api.rs convention
where `sample_rate()`, `channels()`, `lookahead()` are already
unprefixed and typed. Dev sandbox; no migration support needed.

**Scope of the rename: `ropus::api::Encoder` only.** The low-level
`OpusEncoder::get_vbr_constraint` and `OpusEncoder::get_packet_loss_perc`
methods are **not** renamed — they intentionally mirror the libopus C ABI
naming and have call sites in `capi/src/ctl.rs:198, 200, 568, 580` and
`harness/src/cli.rs:5821, 5827` that must keep working. Do not rename
the low-level methods; that would break capi conformance and the
differential harness.

Existing tests at api.rs:1149 (`assert_eq!(encoder.get_vbr_constraint(),
i32::from(on))`) update mechanically to the new shape.

### Default state

`OpusEncoder::new` initialises `user_bitrate_bps = OPUS_AUTO`
(encoder.rs:1284), so `Encoder::bitrate()` returns `Bitrate::Auto`
before any `set_bitrate` call. Other defaults: VBR on, FEC disabled,
DTX off, complexity 9, `Signal::Auto`, `ForceChannels::Auto`,
`Bandwidth::Fullband`. The matching getters reflect these without
needing any special-case initialisation in the facade.

### `InbandFec` enum (new)

```rust
/// In-band forward-error-correction mode.
///
/// Wraps libopus' `OPUS_SET_INBAND_FEC` request, which accepts three values.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum InbandFec {
    /// FEC disabled (CTL value 0). Default.
    Disabled,
    /// FEC enabled (CTL value 1). Encoder may add FEC redundancy when
    /// packet-loss-percentage and bitrate budget permit.
    Enabled,
    /// FEC always enabled (CTL value 2). Forces FEC redundancy regardless
    /// of bandwidth-adaptation heuristics. Reachable in the encoder's
    /// `fec_config != 2` branch at `encoder.rs:1950`.
    Forced,
}
```

Mirrors the existing `as_c_int` / `from_c_int` pattern.

### One accessor added to `OpusEncoder`

In `ropus/src/opus/encoder.rs`, alongside the existing `get_bitrate`:

```rust
/// Return the user-set bitrate request as stored, before any
/// frame-size-dependent clamping. Counterpart to `set_bitrate` for
/// faithful round-trip queries; use `get_bitrate` when you want the
/// computed effective value the encoder will use for the next frame.
///
/// Crate-internal: this is plumbing for `ropus::api::Encoder::bitrate()`.
/// External callers should use `get_bitrate` (the libopus `OPUS_GET_BITRATE`
/// equivalent).
pub(crate) fn get_user_bitrate_bps(&self) -> i32 {
    self.user_bitrate_bps
}
```

Read-only, single-line, scoped to the crate. The `user_bitrate_bps` field
is at `encoder.rs:3479`. Confirmed writers: constructor (`encoder.rs:1284`),
`set_bitrate` (`:3479`/`:3481`), and `ms_set_user_bitrate` (`:1355`) — no
surprise mutators that would corrupt the round-trip claim.

### Why two bitrate getters

`OpusEncoder::set_bitrate(b)` stores `b` into `user_bitrate_bps` after
clamping the input to `[500, 750_000 * channels]`. Consumers asking
`encoder.bitrate()` after `set_bitrate(Bitrate::Bits(64_000))` reasonably
expect to read back `Bits(64_000)`.

`OpusEncoder::get_bitrate()` calls `user_bitrate_to_bitrate`, which further
reduces the value to `min(user_bitrate, max_bitrate_for_current_framesize)`.
At `fs=48000`, `prev_framesize=960`, that ceiling is ~510_400 bps —
so a user who asked for `Bits(750_000)` would read back `Bits(510_400)`,
and a user who asked for `Auto` (= -1000) would read back something like
`Bits(64_000)`. The value also mutates between encodes as `prev_framesize`
shifts.

`bitrate()` uses `get_user_bitrate_bps()` so that round-trip is faithful
and `Auto`/`Max` reconstruct correctly. `effective_bitrate_bps()` exposes
the legitimate "what is the encoder actually targeting right now" use
case as a plain `i32`, with rustdoc tying it to libopus'
`OPUS_GET_BITRATE` for readers familiar with the C API. Both surfaces
are useful; users now have an honest answer to either question.

### Builder field additions

Add `inband_fec: Option<InbandFec>` and `dtx: Option<bool>` to
`EncoderBuilder`, plus `inband_fec(self, mode: InbandFec) -> Self`
and `dtx(self, on: bool) -> Self` methods, plus the corresponding
`if let Some(...)` block in `build()`.

A consumer configuring a high-loss VoIP encoder may want
`InbandFec::Forced` declared at construction; a push-to-talk consumer may
flip both knobs at runtime. Both shapes are real, and the builder remains
1:1 with the runtime-setter surface.

### Deliberately excluded

The following CTLs are accepted by `OpusEncoder` and exposed via capi but
deliberately not wired through the high-level facade:

- **`set_application(VOIP|Audio|RestrictedLowDelay)`** — switches the
  rate-control profile, mode-selection priors, and SILK/CELT
  pre-processing. libopus permits it but consumer code that flips it
  mid-stream universally regrets it. If a consumer needs it, drop to
  `OpusEncoder` and accept the responsibility.
- **`set_lsb_depth(8..=24)`** — niche; controls the dynamic-range hint
  for the encoder's noise shaping. No real-world adaptive use case.
- **`set_phase_inversion_disabled(bool)`** — niche; controls a stereo
  out-of-phase compatibility tweak. Set-once at most.
- **`set_prediction_disabled(bool)`** — niche; intended for
  ultra-low-latency intra-only encoding. Real consumers pin this at
  construction.
- **`set_expert_frame_duration` / construction-only fields** — flipping
  mid-stream desyncs RTP/Ogg packetisation.

**Adding any of these later requires the same HLD-level review as this
PR; do not slip them in as drive-by additions.** YAGNI for now per
CLAUDE.md.

### Mid-stream safety verification

For each of the 10 included setters, mid-stream mutation cannot corrupt
state because the underlying value is consulted **per-frame** at encode
time, not snapshotted into derived state at construction. Verified by
locating the read sites in `encode_native_with_analysis`
(`encoder.rs:1727` onwards):

- `user_bitrate_bps` — read at `encoder.rs:1727`; result published into
  the per-frame derived cache `self.bitrate_bps` at `:1733`.
- `silk_mode.complexity` — read at `:1631`, `:1838`. `celt.complexity`
  reset by `set_complexity` directly into the CELT encoder.
- `use_vbr`, `vbr_constraint`, `silk_mode.use_cbr` — read at `:1739`,
  `:1822`, `:1844`, `:2288`, `:2382`, `:3127`, `:3314`, `:3324`, `:3362`.
- `force_channels` — read at `:1719`, `:1869`, `:2033`, `:2281`.
- `max_bandwidth` — read each frame as a cap (`:2090`).
- `silk_mode.packet_loss_percentage` — read at `:1839`, `:2138`.
- `fec_config`, `silk_mode.use_in_band_fec` — read at `:1950`.
- `use_dtx`, `silk_mode.use_dtx` — read at `:1909`, `:3267`, `:3771`.
- `signal_type` — read at `:1673`, `:1852-54`.

None of these flip a SILK↔CELT mode-transition flag, NSQ predictor reset,
or range-coder reset.

**Caveats worth noting:**

- `self.bitrate_bps` is a **per-frame derived snapshot**, not a source of
  truth. `Encoder::bitrate()` deliberately reads `user_bitrate_bps` (via
  `get_user_bitrate_bps`), not this cache, to avoid serving a value that
  reflects the previous frame's `frame_size` rather than the user's
  most-recent intent.
- `set_inband_fec` mutates **two paired fields** (`fec_config` and
  `silk_mode.use_in_band_fec`, `encoder.rs:3590-3591`); `set_vbr`
  mutates `use_vbr` and `silk_mode.use_cbr` (`:3515-3516`). These pairs
  are written sequentially within a single setter call. `&mut self`
  exclusivity precludes any concurrent reader from observing a torn
  state — single-threaded callers see them as atomic, and the borrow
  checker rules out multi-threaded access during mutation.

### Error handling

All setters return `Result<(), EncodeError>`.

`EncodeError` is what callers already import for `Encoder::encode`. Its
`BadArg` and `Internal` variants are the only ones reachable from setters;
`BufferTooSmall` and `InvalidPacket` simply never arise. Adding a new
`EncoderConfigError` would be type-bloat for negligible benefit.

The builder continues to use `EncoderBuildError` (the existing `check_ok`
chain in `build()` is unchanged). The minor asymmetry — build-time setter
failures map to `EncoderBuildError`, runtime setter failures map to
`EncodeError` — is acceptable: each path's error type matches what the
caller already handles for that path.

### Type translation helpers

`Bitrate::as_c_int`, `Signal::as_c_int`, `ForceChannels::as_c_int`, and
`Bandwidth::as_c_int` already exist. The reverse direction needs:

- `Bitrate::from_c_int(i32) -> Bitrate`:
  - `-1000` (`OPUS_AUTO`) → `Auto`
  - `-1` (`OPUS_BITRATE_MAX`) → `Max`
  - `n > 0` → `Bits(n as u32)`
  - anything else: `debug_assert!(false, ...)`, fall back to `Auto`.
- `Signal::from_c_int(i32) -> Signal` — three-value mapping.
- `ForceChannels::from_c_int(i32) -> ForceChannels` — three-value mapping.
- `Bandwidth::from_c_int(i32) -> Bandwidth` — five-value mapping.
- `InbandFec::from_c_int(i32) -> InbandFec` — three-value mapping.

For unrecognised values, each helper uses `debug_assert!` followed by a
sensible default (`Signal::Auto` for unknown signals, etc.). This catches
regressions in dev builds while keeping release builds robust. The
"unknown value" arm is genuinely unreachable for normal flows because
the encoder only stores values it itself accepted via the matching
setter — the assertion documents that contract. The existing
`channels_from_c_int` (api.rs:41) is the precedent.

### `check_ok` reuse

The existing `check_ok` returns `Result<(), EncoderBuildError>`. Add a
sibling `check_ok_runtime` returning `Result<(), EncodeError>`:

```rust
fn check_ok_runtime(code: i32) -> Result<(), EncodeError> {
    if code == 0 { Ok(()) } else { Err(EncodeError::from_code(code)) }
}
```

Three lines. Generic-over-error-type would force `From<i32>` impls on
both error types and add a trait bound for no readability gain.

## Implementation sketch

Total expected size: ~180 LOC of delegations + helpers + ~140 LOC of
tests + rustdoc. One PR. One file modified outside api.rs (the new
`pub(crate) fn get_user_bitrate_bps` accessor on `OpusEncoder`).

## Testing

Add to `api.rs`'s existing `#[cfg(test)] mod tests`:

1. **Round-trip per setter (mono fixture).** For each setter applicable
   to a mono encoder, build a default encoder, call the setter, call
   the matching getter, assert the value matches.
2. **Round-trip per setter (stereo fixture).** `set_force_channels(Stereo)`
   only validates on a `Channels::Stereo` encoder (`encoder.rs:3537`
   returns `BadArg` for force-channels > self.channels).
3. **Bitrate round-trip across all three forms.** `set_bitrate(Auto)` /
   `Max` / `Bits(64_000)` each round-trip through `bitrate()` faithfully.
   This is the test that V1 would have failed.
4. **`effective_bitrate_bps` reflects framing.** After
   `set_bitrate(Bits(750_000))`, `effective_bitrate_bps()` returns a
   value `≤ 750_000` (clamped by frame size). Documents the difference
   between the two getters.
5. **BadArg propagation.** `set_complexity(11)` and
   `set_packet_loss_perc(101)` return `Err(EncodeError::BadArg)`.
6. **Mid-stream mutation produces real effect on output.** Drive the
   encoder with a 1 kHz sine tone (not silence — DTX would mask the
   signal). Encode 100 frames at `Bits(16_000)`, then
   `set_bitrate(Bits(96_000))`, encode 100 more frames. Assert
   `mean(batch2.packet_size) ≥ 3 × mean(batch1.packet_size)` (expected
   ≈ 6×; threshold gives a 2× safety margin against rate-control noise).
   Catches "setter is wired but to the wrong field" bugs.
7. **Mid-stream bitrate flip in CBR mode.** Same as test 6 but with
   `set_vbr(false)` first. The CBR codepath mutates `silk_mode.use_cbr`
   in addition to `user_bitrate_bps` (`encoder.rs:3515-3517`) — different
   plumbing, needs its own coverage.
8. **`set_max_bandwidth` round-trips.** `set_max_bandwidth(Narrowband)`
   then `assert_eq!(encoder.max_bandwidth(), Bandwidth::Narrowband)`.
   Simpler than TOC parsing; same delegation coverage.
9. **Mid-stream mutation does not panic across many flips.** Loop 100
   times: at iteration `i`, choose op = `i % 5` and apply one of
   {bitrate-flip, vbr-toggle, fec-cycle, packet_loss-step, dtx-toggle}
   between encodes. A fixed deterministic cycle is used rather than a
   seeded RNG so the test adds no new dev-dependency (`rand` is not
   currently in `ropus/Cargo.toml`'s dev-deps; only `proptest` and
   `ogg` are). Assert no panic, *and* assert `decoder.decode(packet,
   ...)` succeeds for each — catches malformed-packet emission, not
   just no-panic.

No new differential tests against the C reference are needed: the
underlying `OpusEncoder::set_*` methods are exercised by the conformance
suite *via the C ABI path* (capi → ctl_shim.c → ropus). The new high-level
facade tests cover the new delegation layer specifically; they intentionally
do not duplicate the capi-level CTL coverage.

## Out of scope (explicit non-actions)

- Multistream / projection facade additions.
- Decoder-side runtime setters beyond the existing `set_gain`.
- Renaming `EncoderBuildError` to a unified `EncoderConfigError`.
- Adding the deliberately-excluded CTLs above.
- Splitting `effective_bitrate_bps` into a dedicated newtype — it's a
  plain `i32` for the same reason `lookahead()` is `u32`.

## Risks

- **`Bitrate::from_c_int` clamp-to-default arm hides bugs in unanticipated
  encoder states.** Mitigation: `debug_assert!` makes dev builds loud.
  The arm is genuinely unreachable in release flows; release-build
  silent-default is preferable to panicking on a dependency upgrade.
- **`InbandFec::Forced` is rarely used.** Adding it to the enum costs
  ~5 lines and one test arm. Worth it: bool would have been a permanent
  API mistake.
- **`set_bitrate` silently clamps requested values above the libopus
  contractual range** (`[500, 750_000 * channels]`). After
  `set_bitrate(Bits(2_000_000_000))`, `bitrate()` reads back the clamped
  `Bits(1_500_000)` on stereo. The facade faithfully reflects the
  encoder's stored value, not the requested one. This is libopus
  behaviour, not a facade bug; document on `set_bitrate`'s rustdoc that
  out-of-range values are clamped. Consumers needing pre-validation
  should use the existing `Bitrate::try_bits` helper before calling
  `set_bitrate`.
