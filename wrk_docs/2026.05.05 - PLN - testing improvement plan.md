# Testing Improvement Plan

Date: 2026.05.05

Scope: concrete follow-up plan for improving test confidence in
`/home/md/language/ropus`, based on
`/home/md/language/ropus/wrk_docs/2026.05.05 - testing gaps review.md`.

## Goal

Make "green" mean something precise and automatable.

The repo already has strong test volume. The next work should not be another
generic coverage push. It should harden the release-confidence contract by
making comparison failures fail, skipped asset-backed oracles visible, fuzz
target coverage current, and weak oracle classes explicitly named.

## Non-Goals

- Do not rewrite codec implementations while fixing gate plumbing.
- Do not turn every local diagnostic into a release blocker.
- Do not pretend DRED/PLC/neural parity is solved by smoke tests.
- Do not make fresh-clone ergonomics hostile; release gates can be stricter
  than default `cargo build`.

## Success Criteria

1. Headline comparison commands fail with a non-zero exit code on real
   mismatches.
2. Length mismatches are never counted as exact byte/sample equality.
3. Release preflight distinguishes present, missing, and intentionally skipped
   oracle assets.
4. Fuzz runners exercise exactly the active targets in
   `/home/md/language/ropus/tests/fuzz/Cargo.toml`.
5. The repo has a maintained oracle matrix that labels exact byte parity, exact
   PCM parity, SNR-gated parity, sample-count-only, smoke-only, ignored, and
   asset-skipped coverage.
6. DRED, DNN, PLC/recovery, real-world corpus, frame-duration fuzzing, and
   performance coverage have tracked follow-up gates instead of vague
   "covered/not covered" claims.

## Phase 0: Gate Integrity First

Target duration: 1 focused implementation slice.

### 0.1 Make comparison stats length-aware

Files:

- `/home/md/language/ropus/harness/src/cli.rs`

Changes:

- Extend `CompareStats` with both input lengths, or an explicit
  `length_mismatch` field.
- Compare only overlapping elements for value equality.
- Treat extra trailing bytes/samples as mismatches even if the absent side would
  have been zero.
- Update output text to show both lengths when they differ.

Validation:

- Add focused unit tests for:
  - equal slices,
  - same-length value mismatch,
  - left has trailing zero,
  - right has trailing zero,
  - left/right empty asymmetry.
- Run:
  - `timeout 120 cargo test -p ropus-harness compare`

Definition of done:

- A missing zero tail cannot produce `matching == total`.

### 0.2 Make headline compare commands fail automation

Files:

- `/home/md/language/ropus/harness/src/cli.rs`

Changes:

- Make non-diagnostic `encode`, `decode`, and `roundtrip` exit non-zero when
  their primary comparison mismatches.
- Preserve explicit skip behavior for genuinely unimplemented paths, but make
  skips machine-readable and documented.
- Leave exploratory diagnostics able to print rich output without becoming
  impossible to use.

Validation:

- Add a small negative test if the harness has a command-test pattern; otherwise
  add a helper-level test around the decision function so the CLI behavior is
  not just manual.
- Run:
  - `timeout 120 cargo test -p ropus-harness`
  - `timeout 120 cargo run -p ropus-harness --bin ropus-compare -- unit range_coder`

Definition of done:

- A script can trust process exit status for `encode`, `decode`, and
  `roundtrip`.

### 0.3 Add an explicit release preflight

Candidate location:

- `/home/md/language/ropus/full-test/src`
- or a small dedicated binary/crate if that matches the existing runner better.

## Suggested Implementation Order

1. Implement Phase 0.1 and 0.2 together. They touch the same harness file and
   directly prevent false green automation.
2. Implement Phase 1.1 next. It is small, mechanical, and protects all future
   fuzz work.
3. Add the release preflight from Phase 0.3. This makes asset-backed claims
   explicit before touching DRED/DNN tests.
4. Publish the oracle matrix from Phase 2.1. This gives us a shared language for
   deciding which weak areas deserve engineering time.
5. Pick one weak codec area: DRED payload parity if neural coverage matters
   most, or PLC/recovery drift if decode robustness matters most.

## First Slice Proposal

Start with harness gate integrity:

- `/home/md/language/ropus/harness/src/cli.rs`
  - length-aware `CompareStats`,
  - unit tests for compare helpers,
  - non-zero exit for `encode|decode|roundtrip` mismatches.

Why this first:

- It is the highest leverage fix.
- It is scoped to one module.
- It improves every future automation layer.
- It does not require solving codec parity or asset provisioning first.

Expected validation:

- `timeout 120 cargo test -p ropus-harness compare`
- `timeout 120 cargo test -p ropus-harness`
- one known-good `ropus-compare` command using an existing deterministic vector,
  if local assets are present.

Open question for Arthur:

- Should `SKIP` from a headline compare command exit `0`, `77`, or non-zero?
  My recommendation is:
  - local diagnostic skip: `0` with clear `SKIP`,
  - release/preflight mode skip: non-zero,
  - future CI convention: use one explicit skip code only if a runner consumes
    it deliberately.
