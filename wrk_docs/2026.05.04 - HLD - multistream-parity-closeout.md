## 1. Scope

This document covers the current `ropus-harness` multistream parity failure in
`ropus-compare`. The concrete symptoms are:

- Multistream encode: first C/Rust packet diff is frame 0, byte 13.
- Multistream decode of C-encoded packets: passes sample-for-sample.
- Multistream roundtrip: first PCM diff is sample 482, `max_diff=23392`.

The target is to close out what evidence is needed, the smallest likely fix,
and the validation matrix for the implementation phase. The implementation is
explicitly out of scope for this HLD.

## 3. Current Evidence

The harness case is hard-coded in
`/home/md/language/ropus/harness/src/cli.rs:8533`:

- `sr=48000`
- `channels=2`
- `streams=1`
- `coupled=1`
- `mapping=[0, 1]`
- `application=OPUS_APPLICATION_AUDIO`
- `frame_size=960`
- `bitrate=64000`
- `complexity=10`
- `vbr=0`
- deterministic noise generated with seed `42`

Decode of C-encoded packets passes in the same command, so the first-order
failure is encode-side. The roundtrip PCM mismatch is downstream of the byte
divergence unless later evidence proves the Rust decoder also mishandles the
Rust-produced packet.

The critical C/Rust control-flow difference is in the substream encode call.
C `opus_multistream_encode_native` passes the original multistream PCM and
downmix selectors into `opus_encode_native`:

- `/home/md/language/ropus/reference/src/opus_multistream_encoder.c:1027`
- `/home/md/language/ropus/reference/src/opus_multistream_encoder.c:1028`

```c
len = opus_encode_native(enc, buf, frame_size, tmp_data, curr_max, lsb_depth,
      pcm, analysis_frame_size, c1, c2, st->layout.nb_channels, downmix, float_api);
```

In that C path, analysis is live for this repro because the fixed-point gate is
`complexity >= 10`, `16000 <= Fs <= 48000`, and not restricted-SILK:

- `/home/md/language/ropus/reference/src/opus_encoder.c:1250`
- `/home/md/language/ropus/reference/src/opus_encoder.c:1257`

Rust multistream extracts the substream PCM correctly, but then calls the
no-analysis wrapper:

- `/home/md/language/ropus/ropus/src/opus/multistream.rs:1148`
- `/home/md/language/ropus/ropus/src/opus/multistream.rs:1149`

```rust
let enc = &mut self.encoders[s];
let len = enc.encode_native(&buf, frame_size, &mut tmp_data, curr_max, lsb_depth)?;
```

That wrapper explicitly passes `None` for analysis:

- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1503`
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1511`

Public single-stream encode does pass analysis input:

- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1430`
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1452`

and Rust's internal analysis gate mirrors the C gate:

- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1541`
- `/home/md/language/ropus/ropus/src/opus/encoder.rs:1580`

So the leading hypothesis is: **Rust multistream sub-encoders skip tonality
analysis, while the C reference runs it for this 48 kHz, complexity-10, audio
configuration.** The first diff at byte 13 is consistent with analysis-driven
mode, bandwidth, voice-ratio, or CELT analysis decisions changing the packet
early in frame 0.

This is separate from the older `sr=8000, ch=1, family=0` low-rate SILK
`fuzz_multistream` class documented in
`/home/md/language/ropus/wrk_docs/2026.05.03 - HLD - multistream-encode-bytes-divergence-V2.md`.
The old class used complexity below 10 and was localized inside
`silk_encode_frame_FIX`; the current harness failure is at 48 kHz and
complexity 10, so analysis plumbing is a better first hypothesis.

## 4. Evidence Needed To Localize Root Cause

Before changing production behavior, collect the smallest evidence set that can
confirm or reject the analysis-plumbing hypothesis.

1. Add temporary diagnostic output or a throwaway harness branch that logs, for
   frame 0 only, whether `run_analysis` is entered on the Rust sub-encoder and
   whether the resulting `AnalysisInfo.valid`, `voice_ratio`, and
   `detected_bandwidth` match C.

2. Run a complexity control. If the same multistream fixture passes at
   `complexity=9` and fails at `complexity=10`, that strongly confirms the
   analysis gate as the trigger. The current CLI hardcodes `10`, so this
   likely needs either a temporary harness flag or a focused regression test.

3. Compare the single substream packet before repacketization. For this
   `streams=1, coupled=1` case, the multistream packet should be the subpacket.
   Capturing the raw `tmp_data` before `OpusRepacketizer` on both sides will
   rule out repacketizer framing as the byte-13 source.

4. Dump the C and Rust sub-encoder state immediately before the frame-0
   substream encode: configured bitrate, VBR/CBR, complexity, application,
   LSB depth, force channels, forced mode, current bandwidth, and analysis
   read positions. These should already match before analysis.

5. If analysis does not explain the diff, fall back to the prior SILK/CELT
   boundary tracing approach:
   `/home/md/language/ropus/harness/src/bin_inner/fuzz_repro_diff.rs` and the
   existing C trace shims are the right model. Do not start by instrumenting
   `silk_encode_frame_FIX` for this harness case; the current configuration
   points higher in the stack.

## 5. Proposed Minimal Fix Approach

Thread the C-equivalent analysis input through the Rust multistream substream
encode call.

Concretely:

1. Extend the internal encoder analysis path so callers can pass:
   original PCM byte slice, original `analysis_frame_size`, downmix function,
   `c1`, `c2`, and `analysis_channels`.

2. Preserve the existing public single-stream behavior by passing
   `c1=0`, `c2=-2`, and `analysis_channels=self.channels` from
   `OpusEncoder::encode` / `encode_float`.

3. In `OpusMSEncoder::encode_native`, keep the extracted `buf` as the actual
   encoded substream input, but pass the original multistream PCM bytes and
   the stream-specific `c1` / `c2` selectors into `encode_native_with_analysis`.
   This mirrors C:
   - coupled stream: `c1=left`, `c2=right`
   - mono stream: `c1=chan`, `c2=-1`
   - `analysis_channels=layout.nb_channels`

4. Leave the no-analysis `encode_native` wrapper available for internal paths
   that genuinely need it. Do not turn multistream into a call to public
   `OpusEncoder::encode`; that would work for this stereo family-0 repro but
   would be wrong for arbitrary channel mappings, muted channels, and surround
   layouts because C analyzes the original layout with explicit selectors.

## 7. Risks

- Changing the analysis tuple/signature touches `OpusEncoder::encode`,
  `encode_float`, and the internal substream encode path. A small wrapper or
  struct is preferable to positional tuple sprawl.

- Passing the extracted substream PCM into analysis would be an attractive
  shortcut, but it would not match C for arbitrary multistream layouts. The fix
  must analyze the original PCM with C-equivalent `c1`, `c2`, and channel count.

- The older low-rate SILK multistream class is a separate root-cause area.
  If a validation seed from that class fails, do not fold it into this fix
  without fresh evidence.

- Analysis state is cumulative. The implementation must preserve C's analysis
  read-position behavior across frames; one-frame parity is necessary but not
  sufficient.

- Surround mode already computes and applies an energy mask. The analysis fix
  should not disturb `OPUS_SET_ENERGY_MASK` behavior or rate allocation.
