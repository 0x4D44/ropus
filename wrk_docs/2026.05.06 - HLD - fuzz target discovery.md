# Fuzz Target Discovery HLD

## Problem Statement

The fuzz runners currently duplicate fuzz target names in shell arrays. That creates two failure modes:

- Newly added fuzz binaries in `/home/md/language/ropus/tests/fuzz/Cargo.toml` are silently skipped by runner scripts.
- Removed or renamed fuzz binaries remain in runner scripts and fail only when a human starts a fuzz run.

The smallest robust fix is to make `/home/md/language/ropus/tests/fuzz/Cargo.toml` the single source of truth and have the shell runners discover `[[bin]] name = ...` entries at runtime.

## Proposed Approach

Add one small shell helper, for example `/home/md/language/ropus/tools/fuzz_targets.sh`, and source it from both runners. The helper should carry both discovery and consistent error reporting, so malformed manifests fail the same way in both scripts.

The helper should expose a tiny `die` function and one discovery function:

```bash
die() {
    echo "ERROR: $*" >&2
    exit 1
}

discover_fuzz_targets() {
    local manifest="$1"
    [[ -r "$manifest" ]] || {
        echo "cannot read fuzz manifest: $manifest" >&2
        return 2
    }

    awk '
        /^\[\[bin\]\][[:space:]]*$/ { in_bin = 1; next }
        /^\[/ { in_bin = 0 }
        in_bin && /^[[:space:]]*name[[:space:]]*=/ {
            line = $0
            sub(/^[^=]*=[[:space:]]*/, "", line)
            sub(/[[:space:]]*(#.*)?$/, "", line)
            if (line ~ /^"[^"]+"$/) {
                sub(/^"/, "", line)
                sub(/"$/, "", line)
                print line
            } else {
                exit 2
            }
        }
    ' "$manifest"
}
```

Implementation details after sign-off:

- Both scripts should source `/home/md/language/ropus/tools/fuzz_targets.sh` after computing `SCRIPT_DIR`, `ROOT`, and `FUZZ_DIR`.
- Both scripts should use a failure-propagating load pattern. Do not use `mapfile -t TARGETS < <(discover_fuzz_targets "$manifest")`, because `mapfile` can succeed while the process substitution command failed.
- The preferred pattern is:

```bash
load_fuzz_targets() {
    local manifest="$1"
    local target_text

    target_text=$(discover_fuzz_targets "$manifest") || die "failed to discover fuzz targets from $manifest"
    [[ -n "$target_text" ]] || die "no fuzz targets declared in $manifest"

    mapfile -t TARGETS <<<"$target_text"
}
```

- `tools/fuzz_cycle.sh` should call the loader before timestamp/log directory creation. A malformed manifest or no-target failure should exit without creating a new `/home/md/language/ropus/logs/fuzz_cycle_<timestamp>/` directory.
- `tools/fuzz_run.sh` should replace `ALL_TARGETS=(...)` with the same capture-then-`mapfile` pattern into `ALL_TARGETS`, before argument parsing needs `--list` output or default target expansion.
- Both scripts should fail fast with a clear error if discovery returns no targets or if the helper reports malformed input.
- `fuzz_run.sh --target <name>` should validate requested targets against `ALL_TARGETS` before build time. That turns typos and stale documentation into a fast, readable error instead of a later cargo-fuzz failure.

This is deliberately not a full TOML parser. It only recognizes the syntax this manifest already uses: one `[[bin]]` table per target, with a quoted `name = "..."` entry. That keeps the implementation small while removing the duplicated knowledge that is causing the bug.

```bash

```

Expected result: both commands list the same twelve targets from `/home/md/language/ropus/tests/fuzz/Cargo.toml`, including `fuzz_repacketizer_seq`, `fuzz_ctl_sequence`, `fuzz_dnn_blob`, `fuzz_multistream`, and `fuzz_decode_plc_seq`, and excluding `fuzz_repacketizer` and `fuzz_packet_parse`.

## Edge Cases

- Empty or unreadable manifest: fail with a direct error naming `/home/md/language/ropus/tests/fuzz/Cargo.toml`.
- No `[[bin]]` entries: fail before creating logs or invoking cargo-fuzz. In `fuzz_cycle.sh`, this specifically means before timestamped log directory creation.
- A `[[bin]]` table without a simple quoted `name`: fail instead of guessing.
- Comments after a `name = "..."` line should be tolerated.
- Target names with spaces are not expected and should not be supported; cargo-fuzz target names here are Rust binary names.
- `mapfile` is Bash-specific, but both scripts already require Bash via `#!/usr/bin/env bash`.
- If a future manifest uses multiline TOML strings or generated target metadata, this helper should be revisited rather than expanded ad hoc.

## Risks And Tradeoffs

- A real TOML parser would be more correct, but would add a dependency on Python `tomllib`, `cargo metadata` plus JSON parsing, or a third-party tool such as `jq`. That is unnecessary for the current manifest shape.
- Duplicating the `awk` function in each script would avoid a new helper file, but it would preserve duplicated logic. A sourced helper is the smallest shared abstraction that removes the duplication cleanly.
- Discovery happens at runtime, so a malformed manifest now breaks `--list` and dry-run paths immediately. That is a useful failure mode because these scripts cannot run valid fuzz targets from a malformed manifest anyway. The implementation must capture discovery output before feeding `mapfile` so this failure is not masked by Bash process substitution semantics.
- The helper parses every `[[bin]]` in the fuzz manifest, not only files under `fuzz_targets/`. That matches cargo-fuzz target selection and avoids inventing a second filtering rule.
