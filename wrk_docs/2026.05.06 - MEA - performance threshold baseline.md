# Measurement Note: Performance Threshold Baseline

Date: 2026.05.06

Scope: initial calibration for the non-quick `full-test --release-preflight`
performance gate.

## Source

The first threshold table is based on the published README performance ratios
for the canonical 10-vector benchmark sweep:

```text
tools/bench_sweep.sh --iters=30
```

Those ratios are documented as measured on an Opus 1.5.2 C reference baseline
for current `master`, with detail in
`/home/md/language/ropus/wrk_journals/2026.04.19 - JRN - avx2-baseline.md`.

I did not replace that with a fresh five-sweep baseline in this implementation
slice. The HLD allows an initial calibration when a full multi-sweep baseline
is not available, provided the release contract says so explicitly.

## Threshold Formula

For each vector and operation:

```text
release_fail_ratio = max(published_ratio + 0.05, published_ratio * 1.20, 1.05)
```

Values are rounded upward to two decimal places before being encoded in
`/home/md/language/ropus/full-test/src/bench.rs`.

The `1.05` floor is not the old global `1.15x` warning threshold. It only stops
rows currently faster than C from being forced to remain faster than C in the
first release gate. The tracked threshold is still per-vector and
per-operation.

## Intended Use

- Default/local benchmark runs stay observed/WARN-only.
- Non-quick `--release-preflight` uses these thresholds as severe-regression
  blockers over the existing 10-vector smoke set.
- Recalibrate with at least five clean same-machine sweeps before claiming
  broader or tighter performance coverage.

## Calibration Decision: Live 48k Sine Decode Smoke

A bounded implementation smoke ran one short benchmark for
`tests/vectors/48k_sine1k_loud.wav`:

```text
timeout 300 cargo run --release -p ropus-harness --bin ropus-compare -- bench tests/vectors/48k_sine1k_loud.wav --iters 3 --repeats 1 --bitrate 64000 --color=never

C encode    1.511ms
Rust encode 1.623ms
C decode    0.647ms
Rust decode 5.650ms
encode ratio 1.07x slower than C
decode ratio 8.74x slower than C
```

The new release-thresholded gate would correctly fail that decode measurement:
the initial threshold for `CELT 48k mono sine 1k loud` decode is `1.26x`, and
the smoke measured `8.74x`.

Decision: do not weaken the threshold to match this one short smoke. Treat the
measurement as unresolved release-blocking evidence: either a current decode
performance regression or an environment/measurement artifact that needs a
calibrated follow-up sweep. This slice makes the gate honest; it does not claim
a clean release-performance pass on the current local measurement.
