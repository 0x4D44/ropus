# HLD: IETF Vector Provisioning Before Conformance Gate

**Date:** 2026-05-04
**Status:** Implemented 2026-05-04
**Option:** #1 - keep IETF vectors unvendored, but provision them before the
validation path is allowed to claim conformance success.

## Goal

Make the canonical validation path prove the IETF RFC 6716 / RFC 8251
bitstream vectors are present before treating conformance as green.

Today `full-test` records `ietf_vectors_present`, filters the 24
`testvectorNN_{mono,stereo}` tests with `--skip ::testvector` when the
vectors are missing, and can still return PASS if the rest of the workspace
passes. That is useful for local iteration, but it is too weak for a
conformance gate: a fresh checkout can skip the spec-level oracle and still
look clean.

The new behavior should be:

1. Leave `tests/vectors/ietf/` gitignored and unvendored.
2. Reuse the existing pinned fetch scripts as the provisioning mechanism.
3. Have `full-test` attempt provisioning before Stage 2 builds its cargo
   command.
4. Run the IETF tests as normal when vectors are already present or
   provisioning succeeds.
5. If vectors are still unavailable after provisioning, mark the validation
   run as not gateable instead of silently skipping those tests.

## Current Behavior

- `tools/fetch_ietf_vectors.sh` downloads
  `https://opus-codec.org/static/testvectors/opus_testvectors-rfc8251.tar.gz`,
  verifies size and SHA-256, and extracts into `tests/vectors/ietf/`.
- `tools/fetch_ietf_vectors.ps1` mirrors the same flow for Windows.
- `tests/conformance/tests/ietf_vectors.rs` panics if
  `tests/vectors/ietf/` is absent, then checks each vector's `.bit` and
  compares decoded output against `.dec` or `m.dec`.
- `full-test/src/setup.rs` only checks that all 12 `.bit` files exist.
- `full-test/src/tests.rs` injects `--skip ::testvector` when Stage 0 says
  the vectors are absent, and surfaces that as a skip reason.
- `full-test/src/html.rs` renders a note telling the user to run the fetch
  script manually.
- `README.md` and `tests/conformance/README.md` document a manual fetch-once
  step.

## Proposed Scope

Keep the direct conformance tests deterministic and local-only. The
provisioning behavior belongs in `full-test`, because `full-test` is the
workspace validation gate and already owns setup, reporting, and PASS/FAIL
classification. Do not make `cargo test -p conformance --test ietf_vectors`
perform network I/O inside libtest.

Add a small IETF vector provisioning layer to `full-test`:

- Detect complete vector availability before any download attempt.
- If incomplete, invoke the platform fetch script from the workspace root.
- Re-check completeness after the script exits.
- Record a structured status: `present`, `provisioned`, or `unavailable`.
- Feed that status to Stage 2, the JSON envelope, and HTML.
- Remove the default green-path skip of `::testvector` from gate runs. A
  missing vector set after provisioning must be a FAIL-level result.

Completeness should mean the actual conformance test can run:

- `testvector01.bit` through `testvector12.bit` all exist.
- For each `testvectorNN`, at least one reference PCM exists:
  `testvectorNN.dec` or `testvectorNNm.dec`.

This is stricter than the current `full-test` bitstream-only preflight and
closer to what `ietf_vectors.rs` really consumes.

## Design Details

### Provisioning Model

`full-test` Stage 0 should call a function shaped like:

```text
provision_ietf_vectors(workspace_root) -> IetfVectorProvision
```

The result should contain:

- `available: bool`
- `status: Present | Provisioned | Unavailable`
- `attempted_fetch: bool`
- `script: Option<PathBuf>`
- `exit_code: Option<i32>`
- `reason: Option<String>`

The function should be pure around detection and narrowly impure around
script execution:

1. Run the completeness check.
2. If complete, return `Present` without touching the network.
3. If incomplete, select `tools/fetch_ietf_vectors.ps1` on Windows and
   `tools/fetch_ietf_vectors.sh` elsewhere.
4. Invoke from the workspace root so relative paths and script diagnostics are
   stable.
5. Capture stdout/stderr with the same 1 MiB cap pattern already used by
   Stage 2.
6. Re-run completeness check.
7. Return `Provisioned` only if the post-check passes.
8. Return `Unavailable` otherwise, preserving script failure details.

### Stage 2 Behavior

The important rule is simple: a gate run must not PASS when the IETF vectors
were absent and could not be provisioned.

Preferred implementation:

- If status is `Present` or `Provisioned`, run Stage 2 without
  `--skip ::testvector`.
- If status is `Unavailable`, run Stage 2 with `--skip ::testvector` only to
  keep the rest of the workspace signal, then add a synthetic failed test:
  `conformance::ietf_vectors::provisioning`.

That keeps local diagnostics useful while preserving the hard gate. It also
avoids misusing `build_failed`, so Stage 3 and Stage 4 can still run unless
the real cargo invocation failed to compile.

Alternative acceptable implementation:

- If status is `Unavailable`, do not invoke Stage 2 and return a skipped
  Stage 2 outcome plus a dedicated FAIL-level provisioning field.

This is simpler, but worse for developers because it withholds unrelated test
results. Prefer the synthetic failure approach unless it tangles the parser
too much.

### Fetch Script Hardening

The shell and PowerShell scripts currently short-circuit on a single sentinel
(`testvector12m.dec`). That is not enough for gate provisioning: a partial or
manual directory can be incorrectly accepted.

Both scripts should:

1. Use a common expected-file check for all 12 `.bit` files and at least one
   PCM reference per vector.
2. Only skip download when that check passes.
3. Download and verify in a temp directory.
4. Extract into a temp staging directory.
5. Validate the staging directory.
6. Publish into `tests/vectors/ietf/` only after validation passes.

Publishing can overwrite the generated `tests/vectors/ietf/` directory because
it is gitignored external data, but the script should never remove unrelated
workspace files and should keep the "abort without touching destination on
hash mismatch" guarantee.

## No-Network Behavior

- If vectors are already complete, no network is used and validation proceeds.
- If vectors are missing and the machine has no network, no downloader, DNS
  failure, TLS failure, HTTP failure, size mismatch, SHA mismatch, or invalid
  tar layout, provisioning returns `Unavailable`.
- `Unavailable` is a FAIL-level gate result. It should say that conformance
  was not proven, not that codec behavior failed.
- The report should include the fetch script path and a short captured error
  so the fix is obvious.
- Direct `cargo test -p conformance --test ietf_vectors -- --test-threads=1`
  remains no-network and continues to fail locally if vectors are absent.
- CI or offline machines that cannot reach `opus-codec.org` should pre-seed
  `tests/vectors/ietf/` from a cache/artifact before running `full-test`.

This is deliberately strict. Treating no-network as WARN would recreate the
current hole: a machine without vectors could still produce a green
conformance gate.

## Failure Modes

| Failure | Expected behavior |
|---|---|
| Complete vectors already exist | No fetch; Stage 2 runs all conformance tests. |
| Empty `tests/vectors/ietf/` | Fetch attempted; success provisions; failure makes gate FAIL. |
| Partial directory with sentinel present | Fetch attempted because complete-file check fails. |
| Download tool missing (`curl`/`wget`, or PowerShell web request failure) | Provisioning unavailable; report names missing tool/failure. |
| Network unavailable or HTTP error | Provisioning unavailable; gate FAIL with fetch diagnostics. |
| Download size mismatch | Script exits before publishing; gate FAIL. |
| SHA-256 mismatch | Script exits before publishing; gate FAIL. |
| Unexpected tar layout | Script exits before publishing; gate FAIL. |
| Staging extraction incomplete | Script exits before publishing; gate FAIL. |
| Destination unwritable | Script exits non-zero; gate FAIL. |
| Cargo test later fails one vector | Normal test failure; gate FAIL as today. |
| `reference/` missing | Existing conformance `no_reference` behavior remains separate; do not fold it into vector provisioning. |

## Validation Commands

Docs-only validation for this HLD:

```bash
timeout 10s test -f "wrk_docs/2026.05.04 - HLD - ietf-vector-provisioning-gate.md"
```

Implementation validation should use fast unit checks first:

```bash
timeout 120s cargo test -p full-test ietf
timeout 120s cargo test -p full-test setup
timeout 120s cargo test -p full-test tests
```

Then validate the direct conformance oracle after vectors are provisioned:

```bash
timeout 1800s bash tools/fetch_ietf_vectors.sh
timeout 1800s cargo test -p conformance --test ietf_vectors -- --test-threads=1
```

Validate the gate path with benchmarks disabled for faster signal:

```bash
timeout 7200s cargo run --release -p full-test -- --skip-benchmarks
```

Validate no-network behavior in a disposable checkout or temporary copy, not
by damaging another agent's working directory:

```bash
timeout 120s cargo test -p full-test no_network
```

The no-network command assumes unit tests inject a mock failing fetch script
or mock provisioner. Do not rely on live network sabotage for deterministic
tests.

## Non-Goals

- Vendoring the 71 MB IETF vector archive or extracted files.
- Adding the vector archive to Git LFS.
- Moving `tests/vectors/ietf/` out of its current gitignored location.
- Adding network behavior to `tests/conformance/tests/ietf_vectors.rs`.
- Changing codec, capi, or harness behavior.
- Changing the IETF vector URL, expected size, or SHA-256 unless upstream
  changes the canonical archive and Arthur explicitly accepts that update.
- Adding public environment-variable knobs or a broad `--offline` mode.
- Making no-network missing-vector runs green or WARN-only.
- Reworking `full-test` stage structure beyond the vector provisioning gate.

## Open Questions

1. Should `full-test --quick` also auto-provision? Recommendation: yes.
   `--quick` still runs Stage 2, so it should preserve Stage 2's gate
   semantics.
2. Should there be a developer-only `--skip-ietf-vectors` flag? Recommendation:
   no for this phase. The existing direct cargo commands already provide
   lower-level iteration paths, and another skip flag weakens the gate story.
3. Should script stdout/stderr appear in HTML? Recommendation: include only a
   capped short reason in the normal report and leave full logs to terminal
   output. This keeps reports readable and avoids embedding a long curl trace.

## Success Criteria

- A fresh checkout with network access can run `full-test` and have IETF
  vectors fetched automatically before Stage 2.
- A checkout without network and without vectors cannot produce PASS.
- A checkout without network but with complete vectors can produce PASS.
- HTML and JSON distinguish `present`, `provisioned`, and `unavailable`.
- Direct conformance documentation still explains manual fetching.
- No production codec code is touched.
