## Per-artifact

### Artifact `8728aaa0` (27 B)

**Structured input decode** (matches `fuzz_decode_safety.rs:108-138`):

- `data[0] = 0x00` → `sr_idx = 0` → **sample_rate = 8000**
- `data[1] = 0xc6` → bit 0 = 0 (mono), bit 1 = 1 → **channels = 1, float PCM**
- `payload = data[2..]` = 25 bytes; `len > 24` so split: **packet = 9 bytes**, **setter_bytes = last 16 bytes**
- packet hex: `5d 85 3d f1 00 00 00 00 00`
  - TOC `0x5d` → `cfg = 11`, `s = 1`, `c = 1` → **SILK-only, WB, 60 ms, code 1 (two equal-size frames)**
  - 8-byte payload split 4+4 (CBR), each frame 4 bytes (mostly zeros)
  - The TOC `s=1` bit asks for stereo, but the decoder was opened mono. Decoded as a mono-render of a stereo packet — this is legal under the spec.
- Setter shuffle (8 calls × 2 bytes): `set_gain(0)`, `set_gain(5)`, `set_gain(1)`, `set_gain(0)`, `set_gain(0)`, `set_gain(0)`, `set_complexity(10)`, `set_complexity(5)`. **No `reset()` is fired.**

**Wall-time** (3 runs, post-warm-cache): **`Executed in 6505 ms`** (libFuzzer-reported), `wall = 6.9 s`. The original campaign saw 15-24 s under load (22 fuzzers contending on a 28-core box, cold caches).

**Localisation** (eprintln-bisection, per-function timing rolled up):

The trace shape per run is:
1. `decode_native` for the 9-byte packet, frame_size=960: ~1 ms (two 60 ms SILK frames decode normally)
2. `decode_native` PLC, frame_size=160: 1 SILK PLC frame. **~120 ms** for a single 20 ms PLC chunk — already 6× slower than realtime.
3. `decode_native` FEC, frame_size=5760, fec=true: enters the FEC branch at `ropus/src/opus/decoder.rs:1252-1286`. With `frame_size - packet_frame_size = 5760 - 480 = 5280`, the recursive call `decode_native(None, pcm, 5280, false)` runs PLC for 660 ms of phantom audio. That recurses through `decode_frame` (frame_size > f20) which loops in 20 ms chunks via lines 754-763, producing **33 SILK PLC chunks**. Each chunk: ~50 ms. Total ≈ **1.7 s**.
4. The fuzz target then calls `decode_float(Some(packet), …, MAX_FRAME, true)` (line 165) — same shape, another ~1.7 s.
5. libFuzzer reruns the target body twice for stability ⇒ ~×2 again.

**Where the 50 ms per PLC chunk goes** — confirmed by per-step instrumentation inside `silk_decode_frame`'s lost-path branch (`ropus/src/silk/decoder.rs:2371-2469`):

| step | budget |
|---|---|
| `silk_plc_conceal` (classical PLC) | ~50 µs |
| `lpcnet_block` (deep neural PLC) | **~50 ms** |
| `silk_cng` | <50 µs |
| `silk_plc_glue_frames` | <10 µs |
| `silk_resampler` | <100 µs |

**Hot path: `LPCNetPLCState::conceal` in `ropus/src/dnn/lpcnet.rs:2429-2524`** — called twice per silk_decode_frame (loop at decoder.rs:2400-2405, `pair_len = 2 * subfr_length = 160`, `frame_length = 320`, so 2 iterations). Each iteration runs the full FARGAN synthesizer (`self.fargan.synthesize_int`) + Burg cepstral analysis on the very first one + `compute_plc_pred` MLP forward pass.

**Why the deep PLC fires at all** — `OpusDecoder::new` auto-loads the embedded weight blob at `ropus/src/opus/decoder.rs:592-601` (gated on `dnn::embedded_weights::has_embedded_weights()`), and this build has the blob populated. The setter shuffle's `set_complexity(10)` keeps `enable_deep_plc = true` (gate at `ropus/src/opus/decoder.rs:834`). No `reset()` was hit, but reset wouldn't help — `OpusDecoder::reset` does not unload the model.

**Root cause hypothesis (HIGH confidence)** — The fuzz target asks `decode_float(Some(packet), …, MAX_FRAME=5760, fec=true)` against a 60 ms / 8 kHz packet. The decoder honours the request: for FEC, `decode_native` runs PLC for the leading `frame_size - packet_frame_size` samples (`opus/decoder.rs:1263-1270`) **before** decoding the FEC frame from this packet. At 8 kHz, this is 660 ms of phantom output. With the embedded weights loaded, every 20 ms PLC chunk runs the FARGAN synthesizer, which is a heavy neural pipeline. The PLC count scales with the API/packet sample-rate ratio: at 8 kHz, MAX_FRAME=5760 produces 33 PLC chunks per FEC call; at 48 kHz it would be 0. The shape "tiny SILK packet at low sample rate + max-buffer FEC" is precisely what the fuzzer minimised toward.

**Classification**: **Setter-shuffle/fuzz-harness interaction.** The packet content barely matters; the decoder is doing exactly what it was asked. The slowness is intrinsic to running deep PLC for many lost frames at low API rates — not a decoder bug.

**C-reference parity check** — The C reference in `harness/build.rs:349-355` does *not* compile in `_dnn_sources`, so the harness's libopus_ref runs classical PLC only and would not exhibit the same slowdown. **Production xiph builds do compile DEEP_PLC by default**, so an apples-to-apples comparison against a real libopus build with the same blob would show similar slowness. (I did not exec a side-by-side because the harness build deliberately omits DNN sources — but the recursion shape in `opus_decoder.c:716-768` is identical to ours.)

**Recommendation**: **Fuzz-target fix (tighten the harness).** Cap the FEC `frame_size` to a reasonable multiple of `packet_frame_size` so the harness doesn't synthesise hundreds of milliseconds of phantom PLC every iteration. This is what real callers do.

## Cross-cutting summary

**One root cause, three artifacts.** All three are the same fuzz-harness interaction:

1. The fuzz target opens an `OpusDecoder` at a low sample rate (≤ 16 kHz).
2. `OpusDecoder::new` auto-loads the embedded DEEP_PLC weights (`ropus/src/opus/decoder.rs:592-601`).
3. The setter shuffle leaves `complexity ≥ 5` so `enable_deep_plc = true`.
4. The fuzz target's last decode call (`fuzz_decode_safety.rs:165`) requests `decode_float(Some(packet), &mut pcm, MAX_FRAME=5760, decode_fec=true)`.
5. With `frame_size = 5760` API-rate samples and a small packet (`packet_frame_size` = 80-480 API samples), `decode_native` runs PLC for `frame_size - packet_frame_size` samples (`ropus/src/opus/decoder.rs:1263-1270`) — **240-720 ms of phantom audio at the API rate** — before finally decoding the FEC frame.
6. The recursive PLC chunk loop (`ropus/src/opus/decoder.rs:754-763`) splits this into 24-36 SILK PLC frames of 20 ms each.
7. Each PLC frame fires `LPCNetPLCState::conceal` twice (loop at `ropus/src/silk/decoder.rs:2400-2405`), each call running the full FARGAN synthesizer.

The dominant cost is `dnn/lpcnet.rs:2429` (`LPCNetPLCState::conceal`) doing FARGAN synthesis for ~50 frames of audio that the caller is asking for but cannot reasonably consume.

The artifacts differ only in *how many* PLC frames they trigger:
- `8728aaa0`: sr=8000, lots of SILK PLC (~33 chunks per FEC call) — slowest at 6.5 s.
- `75d8b2d8`: sr=8000, similar shape (~36 chunks) — 4.6 s.
- `46a6f8fa`: sr=12000, fewer chunks but Hybrid mode adds a CELT pass — 3.4 s.

**The packet content is barely material** — what matters is `(sample_rate, MAX_FRAME, complexity≥5, embedded blob loaded)`. The fuzzer minimised toward the configuration that makes the decoder synthesise the most phantom audio per byte of input.

**Single fix** addresses all three. Suppression-only is *not* my recommendation — the fuzz target is still useful, it's just asking the decoder to do unrealistic work. The cleanest action is to clamp the FEC `frame_size` in the fuzz target so it represents what real callers actually do.

**No production-decoder fix is needed.** The decoder is honouring the API contract as documented; libopus does the same. A "DoS guard" that caps PLC duration would be a production hardening enhancement, but it is out of scope for this triage.

## Proposed Fix HLD

### Scope

`tests/fuzz/fuzz_targets/fuzz_decode_safety.rs` only. No changes to library code. No changes to the C reference. No new dependencies.

The fix replaces the unconditional `MAX_FRAME = 5760` argument on the FEC and the post-shuffle decode_float calls with a `frame_size` derived from the packet's actual duration, mirroring how real callers (and the existing normal-decode path at `decode_native`) clip `frame_size` via `opus_packet_get_nb_samples`.

### Change sketch

In `tests/fuzz/fuzz_targets/fuzz_decode_safety.rs`:

1. After computing `packet`, compute `expected_total = opus_packet_get_samples_per_frame(packet, sample_rate) * opus_packet_get_nb_frames(packet)?` for the API rate.
2. For the FEC line (line 165 / 190), pass `frame_size = expected_total` (and a correspondingly sized `pcm` buffer slice) instead of `MAX_FRAME`.
3. Leave the normal-decode and PLC calls untouched — `decode_float` already clips `frame_size` to the packet length on the normal path (`ropus/src/opus/decoder.rs:1437-1445`), and the explicit PLC call at line 162-164 uses `plc_frame = sample_rate / 50` which is already 20 ms (sane).

### Validation plan

1. **All three slow-unit artifacts replay in < 1 s.** Concretely: `Executed in <1000 ms` for each, no libFuzzer slow-unit warning.
2. **Re-fuzz `fuzz_decode_safety` for 30 min.** Expect zero new slow-unit artifacts. If new slow-units appear, capture and triage separately.
3. **No reduction in coverage of the FEC code path.** The packet-sized FEC call still exercises `decode_native(…, decode_fec=true)` end-to-end including the FEC-PLC composition — just at a sane scale.
4. The existing `assert_eq!(rust_samples, expected_total, …)` invariant remains; clipping `frame_size` to `expected_total` doesn't conflict with it (the normal-decode assertion stays exactly as today; the FEC call doesn't currently assert sample count).

### Risks

- **Coverage loss on the "FEC at giant frame_size" path** — *acceptable*. We've now established that this path's slowness is intrinsic and not a bug. There is no remaining oracle that would benefit from exploring it further at high cost. If we later want to keep that surface covered, we can spawn a tiny (capped-budget) probe in a separate target.
- **Coverage delta — prefix-PLC-into-FEC composition** — *explicit*. After fix: the prefix-PLC-into-FEC composition path is no longer exercised by `fuzz_decode_safety`. Concretely, the recursive PLC chunker at `ropus/src/opus/decoder.rs:754-763` reached via the FEC branch (`opus/decoder.rs:1252-1286`, when `frame_size > packet_frame_size`) stops being entered: with `frame_size = expected_total = packet_frame_size`, the prelude `decode_native(None, …, frame_size - packet_frame_size, false)` has `len=0` and the prelude PLC never runs. The standalone PLC chunker is still covered by the line-162 `decode_float(None, …, plc_frame, false)` call (and by the same shape in `fuzz_decode.rs`). The FEC-with-prefix-PLC composition is an unusual pattern real callers don't typically hit; coverage loss is judged acceptable.
- **Edge case: `opus_packet_get_nb_samples` rejects malformed packets** — handled by short-circuiting the FEC call when the helper returns Err. This matches what real callers should do.

### Audit: same pattern in other fuzz targets?

Greps run on `tests/fuzz/fuzz_targets/`:

```
grep -nP "decode.*MAX_FRAME.*true" tests/fuzz/fuzz_targets/*.rs
grep -nP "decode_float.*MAX_FRAME.*true" tests/fuzz/fuzz_targets/*.rs
```

Hits: **only** `fuzz_decode_safety.rs:165` (`decode_float(..., MAX_FRAME, true)`) and `fuzz_decode_safety.rs:190` (`decode(..., MAX_FRAME, true)`). Both are in the file we're already fixing.

Cross-checked the other plausible candidates by hand:

| target | shape | exposed? |
|---|---|---|
| `fuzz_decode.rs` | `MAX_FRAME, false` (FEC=false) | No — prelude PLC only fires on FEC. |
| `fuzz_decode_plc_seq.rs` | `frame_size, false` | No — uses caller-provided frame_size, not MAX_FRAME. |
| `fuzz_roundtrip.rs` | `frame_size, false` | No — uses encoder's per-call frame_size, FEC=false. |
| `fuzz_roundtrip_safety.rs` | `frame_size, false` | No — same as roundtrip. |
| `fuzz_ctl_sequence.rs`, `fuzz_decode_plc_seq.rs`, others | various, all FEC=false or not invoking decode | No. |

**Conclusion**: `fuzz_decode_safety.rs` is the only target with the slow-PLC-FEC shape. Single-file fix is sufficient; no fan-out required.

### Out-of-scope (explicit)

- A production "DoS cap" on `decode_native`'s FEC PLC duration. This would be a separate library-level decision (and would need to mirror what the C reference does, i.e. nothing). I'm not recommending one.
- Disabling the embedded DEEP_PLC blob in fuzz builds. That would lose coverage of the neural PLC path, which is real production code. The harness fix above keeps coverage while making the work bounded.
