# Legal Frame Duration Stateful Fuzz HLD

Date: 2026.05.06

Scope: Phase 3.3 Slice B for `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs` and `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode_plc_seq.rs`.

## Problem Statement

Slice A added the shared legal frame-duration helper in `/home/md/language/ropus/tests/fuzz/fuzz_targets/frame_duration.rs` and applied it to `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_roundtrip.rs`.

The two remaining stateful fuzz targets still encode and decode only 20 ms frames through `sample_rate / 50`:

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs` varies encoder state across 2-16 sequential frames, but every frame has the same 20 ms PCM length.
- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode_plc_seq.rs` encodes a sequence with 20 ms frames, drops selected packets, then asks both decoders for PLC using a hard-coded requested decode frame size of `5760`.

That leaves legal Opus frame durations of 2.5, 5, 10, 40, and 60 ms underexplored in the stateful paths. The PLC target has an additional semantic mismatch: `5760` is a maximum 120 ms capacity at 48 kHz, but when passed as the decode frame-size argument for a dropped packet it is also the requested PLC duration. For this slice, dropped-frame PLC should request the selected legal frame duration, not a fixed 120 ms duration.

This design covers test-only fuzz infrastructure. It does not change codec behavior, run fuzz campaigns, or alter oracle taxonomy/reporting.

## Stakeholder Value

Maintainers need fuzz coverage that reflects legal Opus API usage in stateful flows, not only the common 20 ms case.

This slice adds value by:

- Exercising encoder state accumulation across legal short and long frame durations.
- Exercising decoder and PLC state across drops where requested PLC duration matches the stream's encoded frame duration.
- Keeping structured fuzz inputs parseable without adding new top-level schema fields.
- Preserving existing oracle meaning, so any new failures remain attributable to the expanded duration dimension rather than changed reporting rules.
- Keeping the work small enough for one implementation slice with scoped unit and build validation.

## Considered Options

### Option A: Implement both remaining targets in one Slice B

Apply the shared helper to both stateful targets in one implementation pass.

Pros:

- Both targets are structured `Arbitrary` inputs with stateful sequence behavior.
- Both changes are small call-site rewires around existing `frame_size` and PCM length calculations.
- One shared validation pass can cover helper tests plus a fuzz build.
- Avoids coordination overhead for two tiny follow-up slices.

Cons:

- A build failure could initially involve either target.
- PLC requested-duration behavior is semantically more delicate than multiframe encode.

Decision: accept. The change remains small and testable if the implementation limits itself to selector derivation, `frame_size` propagation, fingerprint/debug context, and tests. Splitting further would add process overhead without reducing much implementation risk.

### Option B: Split into Slice B1 and Slice B2

Handle `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs` first, then `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode_plc_seq.rs`.

Pros:

- Simplest attribution if one target fails.
- Lets PLC requested-duration behavior be reviewed in isolation.

Cons:

- Two very small implementation rounds.
- Duplicates validation and review cost.

Decision: reject for now. Use this fallback only if implementation discovers hidden coupling in PLC decode or the fuzz build fails in a way that makes attribution noisy.

### Option C: Add explicit duration fields to both structured inputs

Add `frame_duration_selector: u8` as a separately consumed field in each `Arbitrary` implementation.

Pros:

- Easy to read.
- Duration is independent from existing knobs.

Cons:

- Reinterprets every structured corpus seed by shifting all following fields.
- Consumes more input bytes without real need.
- Violates the broad HLD preference to reuse existing selector material where clean.

Decision: reject. Preserve parseability and bounded input shapes by deriving duration from already consumed bytes.

### Option D: Vary per-frame duration inside a sequence

Allow each frame in a stateful sequence to choose its own legal duration.

Pros:

- Broader state-transition coverage.

Cons:

- Much larger triage surface.
- More complex PCM sizing in `Arbitrary`.
- Harder to reason about expected decoder sample-count parity and PLC history.

Decision: reject for Slice B. Every frame in one generated sequence should use one selected legal duration. Per-frame duration drift can be a later, explicitly designed dimension.

## Recommended Design

### Shared Principles

- Use actual encode/decode `frame_size` arguments from `legal_frame_size_samples_per_channel`, not `set_expert_frame_duration`.
- Keep all existing oracle classes and reporting semantics unchanged.
- Add duration labels or selected frame size only to assertion/debug messages where they aid crash triage.
- Keep every sequence internally consistent: one sample rate, one channel count, one selected legal frame duration, and every PCM frame sized for that tuple.
- Keep output packet buffers unchanged at `4000` bytes.
- Keep decode output buffers large enough for the maximum legal decoded output, while passing the selected frame size as the decode request.

### `fuzz_encode_multiframe.rs`

Affected current behavior:

- `MultiframeInput::arbitrary` draws `n_frames` directly with `u.int_in_range(2..=16)?`.
- It computes `frame_size = (sample_rate / 50) as usize`.
- Runtime recomputes `let frame_size = sample_rate / 50;`.
- Crash fingerprint does not include frame duration.
- Assertions do not include frame duration or selected frame size.

Implementation design:

1. Import the shared helper with `#[path = "frame_duration.rs"] mod frame_duration;`.
2. Add `frame_duration_selector: u8` to `MultiframeInput`.
3. Replace the direct `n_frames` draw with one bounded shape byte:

```rust
let sequence_shape: u8 = u.arbitrary()?;
let frame_duration_selector = sequence_shape % 6;
let n_frames = 2 + ((sequence_shape / 6) as usize % 15); // 2..=16
```

This keeps the structured input compact, makes all six durations reachable, and avoids shifting subsequent per-frame fields by an extra byte compared with drawing selector and frame count separately.

4. In `Arbitrary`, compute:

```rust
let frame_size = frame_duration::legal_frame_size_samples_per_channel(
    sample_rate,
    frame_duration_selector,
) as usize;
let pcm_bytes_needed = frame_size * channels as usize * 2;
```

5. Store `frame_duration_selector` in `MultiframeInput`.
6. In `input_fingerprint`, include the selector. Recommended placement is `fp[14] = input.frame_duration_selector;`, leaving `fp[15]` available. Do not expand the fingerprint beyond 16 bytes.
7. In the fuzz body, derive:

```rust
let frame_duration_selector = input.frame_duration_selector;
let frame_duration_label = frame_duration::legal_frame_duration_label(frame_duration_selector);
let frame_size = frame_duration::legal_frame_size_samples_per_channel(
    sample_rate,
    frame_duration_selector,
);
let samples_per_frame = frame_size as usize * channels as usize;
```

8. Use that `frame_size` for Rust encode, C multiframe encode, and Rust decode sample-count sanity.
9. Keep `decoded = vec![0i16; samples_per_frame]`; it remains exactly sized because this target decodes packets it just encoded with the same frame duration.
10. Add `frame_duration={frame_duration_label}` and/or `frame_size={frame_size}` to existing mismatch, dump, parseability, and decode-sample-count messages. This is reporting context, not oracle taxonomy.

Input-size bounds:

- Maximum PCM per frame becomes 60 ms, 48 kHz, stereo, i16: `2880 * 2 * 2 = 11520` bytes.
- Maximum theoretical full sequence is `16 * 11520 = 184320` PCM bytes plus config bytes.
- This target already uses structured `u.bytes(pcm_bytes_needed)?`; if the libFuzzer `max_len` budget is smaller, `Arbitrary` returns `NotEnoughData` rather than allocating partial frames. That preserves parseability and bounded per-iteration execution.
- No new heap shape beyond larger legal PCM frames is introduced.

### `fuzz_decode_plc_seq.rs`

Affected current behavior:

- `PlcSeqInput::arbitrary` computes `pcm_bytes_per_frame = (sample_rate / 50) as usize * channels as usize * 2`.
- Runtime computes `let frame_size = sample_rate / 50;`.
- Rust decode allocates `max_pcm = 5760usize * channels as usize` and calls `rust_dec.decode(arg, &mut pcm, 5760, false)`.
- C decode receives `frame_size = 5760` through `c_decode_seq_with_drops`.
- `drop_mask` uses low bits for the drop schedule, while bits above the maximum frame count are available.

Implementation design:

1. Import the shared helper with `#[path = "frame_duration.rs"] mod frame_duration;`.
2. Do not add a new parsed field. Derive the selector from unused high bits of the already parsed `drop_mask`:

```rust
let frame_duration_selector = (drop_mask >> 29) as u8;
```

Only bits 0 through 11 are needed for the current 2-12 frame drop schedule, so bits 29 through 31 are available without changing the parse shape.

3. In `Arbitrary`, after reading `drop_mask`, compute:

```rust
let frame_duration_selector = (drop_mask >> 29) as u8;
let frame_size = frame_duration::legal_frame_size_samples_per_channel(
    sample_rate,
    frame_duration_selector,
) as usize;
let pcm_bytes_per_frame = frame_size * channels as usize * 2;
```

4. Keep the existing availability gate:

```rust
let avail = u.len();
let max_fit = (avail / pcm_bytes_per_frame.max(1)).min(12);
if max_fit < 2 {
    return Err(arbitrary::Error::NotEnoughData);
}
let n_frames = u.int_in_range(2..=max_fit)?;
```

The comment should be updated to account for 60 ms. At 48 kHz stereo i16, each 60 ms frame needs `11520` bytes; with a 32 KB `max_len`, the body can still run with two frames plus config bytes.

5. In the fuzz body, derive the same selector from `input.drop_mask` and compute `frame_duration_label`, `frame_size`, and `samples_per_frame`.
6. Keep Rust encode calls using selected `frame_size`.
7. Keep the decode output buffer capacity large enough:

```rust
let max_pcm = 5760usize * channels as usize;
let mut pcm = vec![0i16; max_pcm];
```

This buffer is still the capacity. It should not be reduced to `samples_per_frame`, because C/Rust decode paths may need room for the maximum legal Opus API output and future small decode differences should not become output-buffer-limit noise.

8. Replace the old requested decode frame size of `5760` with the selected `frame_size` in Rust decode:

```rust
rust_dec.decode(arg, &mut pcm, frame_size, false)
```

For kept packets, this is the expected decoded duration of the encoded packet. For dropped packets, this is the requested PLC duration and should match the stream's frame duration.

9. Replace the old `5760` argument to `c_reference::c_decode_seq_with_drops` with the same selected `frame_size`.
10. Keep the existing oracle stance:

- Frame-count parity remains required.
- PCM length parity remains required.
- CELT-only kept packets still require bit equality.
- SILK/Hybrid and dropped frames remain sample-count/result-symmetry only.
- Error-code equality is still not required.

11. Add `frame_duration={frame_duration_label}` and/or `frame_size={frame_size}` to panic messages and the fingerprint. The existing fingerprint can append one byte after `drop_mask` and before `frames.len()` without changing input parsing. Example order:

```rust
buf.extend_from_slice(&input.drop_mask.to_le_bytes());
buf.push(frame_duration_selector);
buf.push(input.frames.len() as u8);
```

This is crash-reporting context only.

PLC capacity/requested-duration decision:

- The selected `frame_size` must replace `5760` as the requested decode frame size for both Rust and C decode calls.
- The output buffers should remain `5760 * channels` samples.
- Do not introduce a second independent PLC duration selector in this slice. That would conflate legal encoded frame-duration coverage with PLC request-duration fuzzing.

## Risks And Tradeoffs

- Structured corpus semantics shift. Existing inputs remain parseable when enough data is available, but `fuzz_encode_multiframe` will interpret the old frame-count byte as a combined shape byte and PLC will interpret high `drop_mask` bits as duration. That is acceptable for fuzz seeds.
- Long frames raise input-size pressure. At 60 ms, 48 kHz stereo i16 frames need 11520 PCM bytes each. The PLC target's `max_fit` gate keeps this bounded; the multiframe target may reject more small inputs through `NotEnoughData`.
- Duration dimensions are correlated. Multiframe duration correlates with frame count, and PLC duration correlates with high `drop_mask` bits. This is a deliberate tradeoff to avoid schema expansion.
- Replacing `5760` changes PLC behavior materially. It narrows this target from "sometimes request 120 ms PLC" to "request PLC matching selected stream duration." That is correct for legal frame-duration coverage, but it removes accidental 120 ms PLC exploration from this target.
- Keeping decode output buffers at `5760 * channels` uses more memory than the selected frame would require. The cost is small and avoids turning this slice into an output-capacity fuzz dimension.
- New short or long frame failures may expose real codec parity issues. The implementation should not weaken the existing oracles to quiet them.
