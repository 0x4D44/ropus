# Legal Frame Duration Fuzz Dimensions HLD

Date: 2026.05.06

Scope: Phase 3.3 of `/home/md/language/ropus/wrk_docs/2026.05.05 - PLN - testing improvement plan.md`.

## Problem Statement

The main encode/roundtrip/stateful PLC fuzz targets still exercise one hard-coded frame duration: 20 ms via `sample_rate / 50`.

That leaves a real codec surface underexplored. Opus legally accepts 2.5 ms, 5 ms, 10 ms, 20 ms, 40 ms, and 60 ms frames at the supported sample rates. Those durations alter frame sizing, buffering, packet duration, mode decisions, PLC duration, and multiframe state transitions. A green fuzz run centered on 20 ms is therefore narrower than the release-confidence story implies.

This HLD is deliberately not a fuzz campaign plan and not a codec fix. It defines the smallest implementation slice to make selected fuzz targets generate legal frame sizes beyond 20 ms while preserving the Phase 2.2 oracle taxonomy and reporting.

## Stakeholder Value

Maintainers care that fuzz coverage reflects real Opus API usage, not only the most common 20 ms path.

This slice adds value by:

- Exercising legal short frames that stress smaller buffers and low-delay behavior.
- Exercising legal long frames that stress packet duration, PLC duration, and state accumulation.
- Keeping inputs well-shaped, so libFuzzer spends time in codec logic rather than invalid-size rejection.
- Keeping oracle interpretation honest: strong and weakened oracle classes remain named by `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs`; frame-duration expansion must not weaken those classes or hide debt.

## Current Behavior

Relevant current targets:

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_roundtrip.rs`
  - Byte-slice input format.
  - Computes `frame_size = sample_rate / 50`.
  - Uses that size for Rust/C encode, Rust decode, packet sample-count invariants, and PCM/SNR oracle checks.

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs`
  - Structured `Arbitrary` input.
  - Computes one 20 ms PCM size for every frame in the sequence.
  - Compares Rust and C encoded packets frame-by-frame and decodes Rust packets for sample-count sanity.

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode_plc_seq.rs`
  - Structured `Arbitrary` input.
  - Encodes a sequence with 20 ms frames, applies packet drops, and decodes Rust/C statefully.
  - Passes `5760` as decode capacity to the PLC sequence helper. For dropped frames, that means the requested PLC duration is currently 120 ms at 48 kHz rather than the encoded 20 ms frame duration.

The shared oracle file `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs` already has Phase 2.2 metadata and reporting for `celt-coded-comparable`, `silk-hybrid-snr-comparable`, `sample-count-only`, `recovery-or-dtx-only`, and `error-agreement-only`.

## Considered Options

### Option A: Add an explicit duration field to every target

Add `frame_duration_idx` to each target input schema and use it to select one of six legal durations.

Pros:

- Easy to read.
- Independent duration mutation.

Cons:

- Byte-slice targets need a new header byte or shifted PCM offset.
- Structured `Arbitrary` targets consume an additional byte and reinterpret existing corpora more sharply.
- The request explicitly asks not to explode the input schema.

Decision: reject as the default. Use only if an implementation agent proves an existing selector cannot be reused cleanly.

### Option B: Derive duration from existing configuration bytes

Map existing, already-consumed input bits to one of six legal durations.

Pros:

- No new target binary and no new top-level schema field in byte-slice targets.
- Existing corpus files remain valid inputs.
- Target code stays small.

Cons:

- Some dimensions become mildly correlated with existing knobs.
- Old corpus entries no longer mean exactly the same frame duration.

Decision: accept. Fuzz corpus compatibility here should mean "old inputs still parse and execute when long enough", not "old inputs preserve identical semantics forever".

### Option C: Split into six fixed-duration targets

Create separate targets such as `fuzz_roundtrip_2_5ms`, `fuzz_roundtrip_60ms`, and so on.

Pros:

- Very explicit campaign control.
- No input schema change.

Cons:

- Multiplies manifest entries, runner inventory, crash triage, and build time.
- Duplicates target logic.
- Fights the Phase 1 goal of keeping fuzz target inventory honest.

Decision: reject for Phase 3.3.

### Option D: Use `set_expert_frame_duration`

Select `OPUS_FRAMESIZE_*` constants through encoder CTL and keep passing 20 ms frame buffers.

Pros:

- Directly exercises the CTL.

Cons:

- This is not the same as varying the encode/decode frame size argument.
- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_ctl_sequence.rs` already owns CTL sequencing.
- It can create misleading failures where CTL-requested duration and supplied PCM duration disagree.

Decision: reject for this slice. Phase 3.3 should vary actual frame sizes passed to encode/decode. Expanding `fuzz_ctl_sequence` to include 40 ms and 60 ms expert-frame-duration CTLs is a separate, smaller follow-up.

## Recommended Design

### 1. Add a tiny shared frame-duration helper

Create `/home/md/language/ropus/tests/fuzz/fuzz_targets/frame_duration.rs`.

The helper should be pure and cheap:

```rust
pub const LEGAL_FRAME_DURATION_MS_X2: [u16; 6] = [5, 10, 20, 40, 80, 120];

pub fn legal_frame_duration_index(selector: u8) -> usize {
    selector as usize % LEGAL_FRAME_DURATION_MS_X2.len()
}

pub fn legal_frame_size_samples_per_channel(sample_rate: i32, selector: u8) -> i32 {
    match legal_frame_duration_index(selector) {
        0 => sample_rate / 400,       // 2.5 ms
        1 => sample_rate / 200,       // 5 ms
        2 => sample_rate / 100,       // 10 ms
        3 => sample_rate / 50,        // 20 ms
        4 => sample_rate / 25,        // 40 ms
        5 => 3 * sample_rate / 50,    // 60 ms
        _ => unreachable!(),
    }
}

pub fn legal_frame_duration_label(selector: u8) -> &'static str {
    match legal_frame_duration_index(selector) {
        0 => "2.5ms",
        1 => "5ms",
        2 => "10ms",
        3 => "20ms",
        4 => "40ms",
        5 => "60ms",
        _ => unreachable!(),
    }
}
```

Keep this helper in `tests/fuzz/fuzz_targets/` so fuzz binaries can include it with `#[path = "frame_duration.rs"] mod frame_duration;`. Add an integration test that includes the same file by path. Do not move fuzz helpers into the main `ropus` crate.

Expected sample counts per channel:

| Sample rate | 2.5 ms | 5 ms | 10 ms | 20 ms | 40 ms | 60 ms |
|---:|---:|---:|---:|---:|---:|---:|
| 8000 | 20 | 40 | 80 | 160 | 320 | 480 |
| 12000 | 30 | 60 | 120 | 240 | 480 | 720 |
| 16000 | 40 | 80 | 160 | 320 | 640 | 960 |
| 24000 | 60 | 120 | 240 | 480 | 960 | 1440 |
| 48000 | 120 | 240 | 480 | 960 | 1920 | 2880 |

### 2. Apply target order conservatively

Implement in two small source slices, even though this HLD covers all three listed targets.

Slice A:

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/frame_duration.rs`
- `/home/md/language/ropus/tests/fuzz/tests/frame_duration_tests.rs`
- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_roundtrip.rs`

Reason: `fuzz_roundtrip` is the smallest byte-slice target with both encode and decode invariants. It proves the mapping and oracle interaction before touching stateful sequence targets.

Slice B:

- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs`
- `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode_plc_seq.rs`

Reason: both are structured, stateful sequence targets. They should share the same helper after Slice A is green.

Do not touch all three behavior targets in one commit unless Arthur explicitly wants a single implementation pass. The blast radius is small, but failures will be easier to attribute if roundtrip lands first.

### 3. Selection approach per target

#### `fuzz_roundtrip.rs`

Do not add a new byte.

Use existing high bits from `data[7]`:

```rust
let frame_duration_selector = data[7] >> 5;
let frame_size =
    frame_duration::legal_frame_size_samples_per_channel(sample_rate, frame_duration_selector);
```

Leave the existing lower-bit interpretations intact:

- `data[7] & 0b0001` for DTX.
- `((data[7] & 0b1111_1110) >> 1) % 101` for loss percent.

This intentionally correlates duration with high loss-percent bits, but it avoids a schema shift and keeps every old corpus input parseable.

Lower the initial minimum-length guard from `8 + 320` to the minimum legal i16 frame at 8 kHz mono, `8 + 40`. After parsing sample rate, channels, float/i16 mode, and selected duration, keep the existing `bytes_needed` guard. This allows 2.5 ms frames to execute instead of being rejected by a stale 20 ms guard.

All packet duration and decoded sample-count assertions should compare against the selected `frame_size`.

Error messages should add `frame_duration={label}` or `frame_size={frame_size}` where failures already print sample rate and channels.

#### `fuzz_encode_multiframe.rs`

Avoid adding an extra consumed field to the `Arbitrary` stream.

Replace the current `n_frames` draw with one shape byte:

```rust
let sequence_shape: u8 = u.arbitrary()?;
let frame_duration_selector = sequence_shape % 6;
let n_frames = 2 + ((sequence_shape / 6) as usize % 15); // 2..=16
```

Store `frame_duration_selector` on `MultiframeInput`.

Use the shared helper to compute `frame_size` before calculating `pcm_bytes_needed`. Every frame in a generated sequence should use the same selected duration. Per-frame duration variation is a future feature; mixing it into this slice would make stateful divergence triage harder.

Update `input_fingerprint` to include the duration selector. The fingerprint is a repro hint, so this is worth one byte.

Keep byte-exact packet parity and decode sample-count sanity unchanged except for using selected `frame_size`.

#### `fuzz_decode_plc_seq.rs`

Do not add a field.

Use the currently unused high bits of `drop_mask`:

```rust
let frame_duration_selector = (drop_mask >> 29) as u8;
```

Frames are limited to 12, and the drop schedule only consumes bits 0 through 11. Bits 29 through 31 are available for duration selection without changing the parse shape.

Store or recompute the selector so `pcm_bytes_per_frame`, encode calls, and decode calls use the same selected `frame_size`.

Important PLC change: pass selected `frame_size` to Rust and C sequence decode instead of `5760`. For non-dropped packets this is a sufficient output capacity because the target encoded packets with the same duration. For dropped packets it is the requested PLC duration and should match the encoded sequence's frame duration. Keeping `5760` would fuzz "legal encode duration plus 120 ms PLC request", which is a different dimension and a poor default for this slice.

Keep the existing oracle stance:

- CELT-only kept frames assert PCM equality.
- SILK/Hybrid and dropped frames assert decode-result symmetry and sample-count parity.
- Do not introduce SNR assertions into this target.

### 4. Preserve Phase 2.2 oracle taxonomy

Do not edit `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs` unless a compile error forces an import adjustment.

Frame-duration selection must be treated as an input dimension, not an oracle class. The existing Phase 2.2 names and reporting remain the interpretation layer:

- `celt-coded-comparable`
- `silk-hybrid-snr-comparable`
- `sample-count-only`
- `recovery-or-dtx-only`
- `error-agreement-only`

For `fuzz_roundtrip`, keep the existing CELT exact and SILK/Hybrid SNR behavior. For `fuzz_decode_plc_seq`, keep the target's weaker PLC-specific oracle and document in comments that duration variation does not change the oracle strength.

## Risks And Tradeoffs

- Existing corpus semantics shift. Old byte inputs remain parseable where long enough, but some will select non-20 ms durations. This is acceptable because fuzz corpora are mutation seeds, not stable behavioral fixtures.
- Duration correlation remains. `fuzz_roundtrip` ties duration to `data[7]` high bits, and PLC ties it to high `drop_mask` bits. That is the price of not expanding schemas. Coverage-guided mutation should still reach all six selectors cheaply.
- Long frames need more input bytes. 60 ms stereo float at 48 kHz requires much more PCM than the old 20 ms i16 minimum. The target should return early when data is too short, not allocate partial frames.
- Stateful PLC behavior changes. Passing selected `frame_size` instead of `5760` makes dropped-frame PLC duration match the encoded stream. That is the right default for this slice, but it may stop exploring 120 ms PLC requests in this target. A separate PLC-duration fuzz dimension can be added later if needed.
- Short frames can expose existing codec parity bugs. That is expected. The implementation should not weaken byte/SNR/sample-count oracles to make the new dimension quiet.
- The CTL target still only lists expert-frame-duration values through 20 ms. This HLD intentionally leaves that alone; mixing CTL expansion into Phase 3.3 would blur the result.

## Rollback Plan

Rollback is simple because the helper and call sites are test-only fuzz infrastructure:

1. Revert the target call sites to `frame_size = sample_rate / 50`.
2. Remove `/home/md/language/ropus/tests/fuzz/fuzz_targets/frame_duration.rs`.
3. Remove `/home/md/language/ropus/tests/fuzz/tests/frame_duration_tests.rs`.
4. Leave Phase 2.2 oracle metadata untouched.

If only one target regresses, roll back that target while keeping the shared helper and any other green target. The target order above is designed to make that partial rollback clean.

## Key Design Decisions

- Add one shared pure helper for legal frame-duration mapping.
- Cover exactly six Opus frame durations: 2.5 ms, 5 ms, 10 ms, 20 ms, 40 ms, and 60 ms.
- Use actual encode/decode `frame_size` arguments, not `set_expert_frame_duration`.
- Implement in two source slices: `fuzz_roundtrip` first, then `fuzz_encode_multiframe` and `fuzz_decode_plc_seq`.
- Preserve corpus validity by reusing existing selector bits where practical.
- Keep Phase 2.2 oracle taxonomy/reporting unchanged.
- Unit-test mapping and selector policies through normal `cargo test`, without running libFuzzer.
