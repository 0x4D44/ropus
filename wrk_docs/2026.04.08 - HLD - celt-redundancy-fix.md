# HLD: CELT SILK->CELT Redundancy Divergence Fix

Author: celt-analyst
Date: 2026-04-08
Status: Proposed (not yet approved)

## Problem Statement

When the Opus encoder switches mode from SILK to CELT mid-stream (triggered by
`force_mode=CELT` while `prev_mode=SILK`, or by a bitrate/application change
that yields the same transition), the resulting 20 ms packet diverges from the
xiph/opus C reference on the first packet after the switch.

Failing harness output (`mdopus-compare transitions`):

```
SILK->CELT->SILK->CELT (mono): FAIL (68 mismatches in 400 frames, first at step 1 frame 0)
    DIAG step=1 frame=0: C=320 bytes Rust=320 bytes
    C_toc=0x4b R_toc=0x4b
    C_pkt: 4b 41 63 98 48 80 03 6d f0 6a 47 24 d7 09 44 fe
    R_pkt: 4b 41 63 98 48 80 03 6d f0 6a 47 24 d7 09 44 fe
    First diff at byte 153: C=0x66 R=0x7f
```

The TOC byte and the SILK-encoded portion (bytes 0..152) match. The divergence
is at byte 153, which is inside the 5 ms CELT redundancy frame that is
appended after the SILK data when the transition is from SILK to CELT. The
same divergence pattern (`C=0x66 R=0x7f`) appears across every transition test
that exercises the SILK->CELT redundancy path (SILK->CELT, VOIP->AUDIO,
Bitrate extremes 6k->510k, FEC off->loss->off).

Non-transition sweep tests (CELT-only, SILK-only, BW sweep, VBR cycling,
stereo mixing) all pass - the bug is specific to the CELT encoder being
invoked to produce a 5 ms redundancy frame with `LM=1` (240 samples) from a
fresh reset state, which only happens on this exact transition path.

## Root Cause Analysis

I instrumented both the Rust and C encoders with hash dumps of every CELT
state field and intermediate buffer along the prefill->redundancy-encode path
(scratch instrumentation, already reverted). With that telemetry I was able
to confirm the divergence point with no hand-waving.

### 1. The PCM input, reset, and CTL sequence match

At the entry to the SILK->CELT redundancy block
(`src/opus/encoder.rs:2452-2488`), every aspect of the state going into
`celt_encode_with_ec` is bit-identical between C and Rust after my
instrumentation was in place:

- PCM sample data (`pcm_buf + channels*(frame_size-N2-N4)` for the prefill,
  `pcm_buf + channels*(frame_size-N2)` for the redundancy) matches sample for
  sample. The `.min()` clamp in the Rust slicing does not fire for the
  20 ms @ 48 kHz mono scenario because `frame_size - N2 - N4 = 600` and
  `pcm_buf.len() - N4 = 1032`, so the offset stays at 600 (same as C).
- `CELT_SET_START_BAND(0)`, `CELT_SET_PREDICTION(0)`, `OPUS_SET_VBR(0)`,
  `OPUS_SET_BITRATE(OPUS_BITRATE_MAX)` are applied in the same order after
  `OPUS_RESET_STATE` in both codebases.
- After `OPUS_RESET_STATE`, every field I could hash matches: `rng`, `start`,
  `end`, `spread_decision`, `delayed_intra`, `tonal_average`,
  `last_coded_bands`, `hf_average`, `tapset_decision`, `prefilter_period`,
  `prefilter_gain`, `prefilter_tapset`, `consec_transient`, `vbr_*`,
  `overlap_max`, `stereo_saving`, `intensity`, `spec_avg`, `in_mem[]`,
  `prefilter_mem[]`, `old_band_e[]`, `old_log_e[]`, `old_log_e2[]`,
  `energy_error[]`, `preemph_mem_d[]`, and `preemph_mem_e[1]`.

### 2. The 2.5 ms prefill produces the same state as C (bit for bit)

After the prefill call (`celt_encode_with_ec(celt, &pcm[600..], Fs/400, dummy,
2, None)`), the CELT encoder's in-memory buffers all match the C reference.
Specifically the overlap buffer `in_mem[0..120]`, the prefilter history, the
energy envelope `old_band_e[0..21]`, `old_log_e[0..21]`, `old_log_e2[0..21]`,
and `energy_error[0..21]` all hash identically between C and Rust.

The only state that differs after the prefill is `preemph_mem_e[0]`:
- C stores `-7401935`
- Rust stores `-8708096`

This is a cosmetic difference rooted in a subtle semantic mismatch in
`celt_preemphasis`, but **it does not cause the output divergence** (more on
this in the side note below).

### 3. The divergence is inside `patch_transient_decision`

The 5 ms CELT redundancy frame is then encoded with `LM=1` (240 samples,
2 short MDCTs). Inside `celt_encode_core`, I confirmed by hashing that:

- Preemphasis output `input[0..360]` matches C bit-for-bit.
- MDCT output `freq[0..240]` matches C bit-for-bit.
- `compute_band_energies` output `band_e[0..17]` matches C bit-for-bit.
- `amp2log2` output `band_log_e[0..17]` matches C bit-for-bit.
- `old_band_e[0..17]` (populated by the prefill) matches C bit-for-bit.

But the `patch_transient_decision` call on line 2548 then returns different
answers in the two implementations **from the same input data**:

- C: `mean_diff_pre=88064`, divisor=14, `mean_diff=6290`, threshold=16777216, returns 0
- Rust: `mean_diff_pre=994596352`, divisor=14, `mean_diff=71042596`, threshold=16777216, returns 1

Because Rust returns "transient detected", it re-runs the MDCT with
`short_blocks = M = 2`, overwrites `band_log_e`, and passes completely
different quantisation parameters to `quant_all_bands`. The first byte of the
range coder output then diverges (first `7f` vs `66`), followed by the rest
of the redundancy frame.

### 4. Why `patch_transient_decision` diverges

The C implementation (`reference/celt/celt_encoder.c:547-581`):

```c
static int patch_transient_decision(celt_glog *newE, celt_glog *oldE,
                                    int nbEBands, int start, int end, int C)
{
   ...
   c=0; do {
      for (i=IMAX(2,start);i<end-1;i++)
      {
         opus_val16 x1, x2;                          /* <-- int16 ! */
         x1 = MAXG(0, newE[i + c*nbEBands]);         /* int32 -> int16 truncation */
         x2 = MAXG(0, spread_old[i]);                /* int32 -> int16 truncation */
         mean_diff = ADD32(mean_diff, MAXG(0, SUB32(x1, x2)));
      }
   } while (++c<C);
```

In the CELT fixed-point build (which is what `mdopus` targets), `celt_glog ==
opus_val32 == int32`, and `opus_val16 == int16`. The expressions `MAXG(0,
newE[i])` and `MAXG(0, spread_old[i])` both evaluate to `int32`, but the
results are then **assigned to `opus_val16` locals**, which implicitly
truncates the high 16 bits. With `DB_SHIFT=24`, the inputs to this function
are log-energies in Q24 - values like `52363264` (=3.12 in Q24) - which do
*not* fit in 16 bits and wrap to spurious negatives or small positives when
truncated.

Concretely, at byte 153 of the failing packet the loop sees
`bandLogE[2] = 52363264 = 0x031E9A00`. Truncating to `int16`:

```
0x9A00 (unsigned) = 39424
0x9A00 (signed)   = -26112
```

`MAXG(0, -26112) = 0`. C therefore accumulates tiny values, yields
`mean_diff_pre=88064`, divides by 14 to get 6290, compares against
`GCONST(1.0) = 16777216`, and returns 0 (not a transient).

The Rust translation at `src/celt/encoder.rs:773-830`:

```rust
let mut ch = 0;
loop {
    for i in (2.max(s))..(e - 1) {
        let x1 = 0i32.max(band_log_e[i + ch * nb]);   // stays i32
        let x2 = 0i32.max(spread_old[i]);             // stays i32
        mean_diff = mean_diff.wrapping_add(0i32.max(x1.wrapping_sub(x2)));
    }
    ch += 1;
    if ch >= c as usize { break; }
}
mean_diff /= c * (end - 1 - start.max(2));

mean_diff > gconst(1.0)
```

The Rust variables `x1`, `x2`, and the sum `mean_diff` are all `i32`. There is
no 16-bit truncation, so the Rust path accumulates the **real** log-energy
differences: `mean_diff_pre = 994596352`, divides to `71042596`, which is
4.23x the threshold, and returns `true`.

**Mathematically, the Rust code is "correct" and the C code is buggy** -
truncating a Q24 log-energy to int16 produces garbage. But since xiph/opus is
our bit-exact reference, we MUST match the C behaviour. The port of
`patch_transient_decision` already has four commented bug-fix notes in it
("Bug 1 fix" through "Bug 5 fix"), but this particular C quirk was missed
because it is invisible without reading the type declarations carefully.

### 5. Why this only shows up on the SILK->CELT redundancy path

The CELT-only sweep and the BW-sweep tests pass because they all encode 20 ms
frames (`LM=3`, 960 samples) from encoder state that has been running for
many frames. In that steady state, the `bandLogE` values are typically large
positive log-energies near the threshold (values like `106692591`,
`131710976`, etc.), and the `old_band_e` values are similarly large. Both C
and Rust compute small differences, and even though the Rust values are
mathematically 16x bigger than the C ones, both stay comfortably below
`GCONST(1.0)*C*(end-2) = 16777216*18` of the ACCUMULATED pre-divide total,
which is what actually matters for the boolean decision.

But the SILK->CELT redundancy path hits a pathological case:

1. The CELT encoder is freshly reset, so `old_band_e[0..17]` starts at values
   that the 2.5 ms prefill has just barely filled in - a signal that is very
   different from the much-longer 5 ms redundancy frame input.
2. For our failing vector, this produces `bandLogE` values around 50-130
   million, `spread_old` values around -100 to +50 million, and a per-index
   difference large enough to trip the threshold in Rust but not in C.

In other words, the transient detector is oversensitive on the very first
CELT frame after a SILK->CELT transition, precisely because Rust is doing the
"correct" math while the reference is silently 16-bit truncating. The bug
manifests as a false positive in Rust.

### Side note - the cosmetic `preemph_mem_e[0]` difference

The Rust `celt_preemphasis` fast path in `src/celt/encoder.rs:597-604`:

```rust
if upsample == 1 {
    let mut m = *mem;
    for i in 0..nu {
        let x = shl32(pcm[(pcm_offset + i * ccu) as usize] as i32, SIG_SHIFT);
        inp[inp_offset + i] = x - mult16_32_q15(coef0, m);
        m = x;                                    // stores raw x
    }
    *mem = m;
}
```

The C reference at `reference/celt/celt_encoder.c:612-625`:

```c
if (coef[1] == 0 && upsample == 1 && !clip)
{
   for (i=0;i<N;i++)
   {
      celt_sig x;
      x = RES2SIG(pcmp[CC*i]);
      inp[i] = x - m;
      m = MULT16_32_Q15(coef0, x);                /* stores coef0*x */
   }
   *mem = m;
   return;
}
```

The Rust stores `x` as `mem`, then on the next call multiplies it by `coef0`
inside the subtraction. The C stores `coef0*x` directly, then subtracts it
with no multiplication. These are **mathematically equivalent** for every
`inp[i]` value (I verified: with the fix applied to match C's semantics, the
Rust `first16_red` bytes are still divergent, so this is not the cause of
the bug). The two formulations produce the same filtered output, but the
stored `mem` has different units. Matching the C semantics makes the dumped
state fields match, which is useful for debugging and slightly easier to
reason about, but it is not required for bit-exactness. **I am not proposing
to change this in the main fix** - I will leave it as a separate optional
cleanup if we want the dumped state to be trivially comparable with C.

## Fix Proposal

Change `patch_transient_decision` in `src/celt/encoder.rs` to match C's
16-bit truncation of the per-band log-energy values.

### Before (src/celt/encoder.rs:812-827)

```rust
    // Compute mean increase (channel loop)
    // Bug 5 fix: use per-channel newE[i + c*nbEBands], not max of both channels.
    // Bug 4 fix: upper bound is end-1, not end.
    let mut ch = 0;
    loop {
        for i in (2.max(s))..(e - 1) {
            let x1 = 0i32.max(band_log_e[i + ch * nb]);
            let x2 = 0i32.max(spread_old[i]);
            mean_diff = mean_diff.wrapping_add(0i32.max(x1.wrapping_sub(x2)));
        }
        ch += 1;
        if ch >= c as usize {
            break;
        }
    }
    mean_diff /= c * (end - 1 - start.max(2));

    mean_diff > gconst(1.0)
}
```

### After (proposed)

```rust
    // Compute mean increase (channel loop)
    // Bug 5 fix: use per-channel newE[i + c*nbEBands], not max of both channels.
    // Bug 4 fix: upper bound is end-1, not end.
    // Bug 6 fix: C declares x1, x2 as opus_val16 (int16), so assigning the
    // int32 `celt_glog` values from MAXG(0, ...) truncates the high bits.
    // This silent truncation is what keeps the transient detector from
    // firing on fresh-reset frames, and we must replicate it for bit-exact
    // parity with the reference. See wrk_docs/2026.04.08 - HLD -
    // celt-redundancy-fix.md.
    let mut ch = 0;
    loop {
        for i in (2.max(s))..(e - 1) {
            let x1 = (0i32.max(band_log_e[i + ch * nb]) as i16) as i32;
            let x2 = (0i32.max(spread_old[i]) as i16) as i32;
            mean_diff = mean_diff.wrapping_add(0i32.max(x1.wrapping_sub(x2)));
        }
        ch += 1;
        if ch >= c as usize {
            break;
        }
    }
    mean_diff /= c * (end - 1 - start.max(2));

    mean_diff > gconst(1.0)
}
```

The cast pattern is `(i32_value as i16) as i32`. `as i16` truncates to the
low 16 bits with sign extension; the subsequent `as i32` widens back to a
signed 32-bit int preserving the sign. This matches exactly what MSVC and
GCC do for an implicit `int32 -> int16` assignment followed by a promotion
in the `SUB32(x1, x2)` arithmetic.

No other code needs to change. This is a one-hunk, two-line change inside a
single function.

## Risk Assessment

### What could break

1. **Other tests that happen to rely on Rust's "correct" math.** I searched
   the test suite and the production code for any place where the result of
   `patch_transient_decision` affects output in a way that is checked for
   bit-exactness. The only caller is `celt_encode_core` in
   `src/celt/encoder.rs:2548`, and its sole consumer is the Opus-layer
   comparison harness (`tests/harness/main.rs`). The Opus-layer tests check
   for bit-exact parity with C, so matching C is the correct behaviour by
   definition. There are no unit tests for `patch_transient_decision` that
   would contradict the fix.

2. **Perceptual quality regression in downstream decoding.** Because the new
   code makes the transient detector *less* sensitive (matching C), it may
   occasionally miss a real transient that the current Rust code would
   catch. However: (a) the C reference has shipped with this behaviour for
   years in libopus 1.x, so perceptual quality is known to be acceptable;
   (b) we have no stated requirement to exceed reference quality.

3. **Endianness.** The `as i16` truncation relies on two's-complement
   arithmetic, which Rust guarantees on all supported platforms. No risk.

4. **SIMD paths.** `patch_transient_decision` has no SIMD variants in either
   C or Rust, so there is nothing to special-case.

5. **Future DNN features.** None of the DNN modules call
   `patch_transient_decision`, so they are unaffected.

log-energy values that fall outside the low 16 bits, because they all use
steady-state encoding. The fix should be a no-op for every passing test.
