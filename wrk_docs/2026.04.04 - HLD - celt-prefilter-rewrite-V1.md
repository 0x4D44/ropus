# CELT Prefilter (run_prefilter) Rewrite

## Problem

The Rust `run_prefilter` in `src/celt/encoder.rs:1424` does not match the C reference in `reference/celt/celt_encoder.c:1411`. This causes ALL non-silence CELT frames to produce different output. Silence is unaffected because zero input produces zero output regardless of the filter.

The prefilter is an analysis comb filter that removes pitch periodicity from the signal before CELT encoding. The decoder applies the inverse (synthesis) filter. For bit-exact output, the encoder's analysis filter must match exactly.

## What the C code does

The C `run_prefilter` has three phases:

### Phase 1: Pitch estimation
Build a work buffer `pre[]` from `prefilter_mem` (history) + current frame. Run pitch search (`pitch_downsample` → `pitch_search` → `remove_doubling`) to find `pitch_index` and `gain1`. Apply adaptive threshold to decide if prefilter is on or off. Quantize gain to 3-bit `qg`.

### Phase 2: Comb filter application
This is the part that modifies the `in[]` buffer (the encoder's input signal). It happens in three steps:

1. **Copy overlap from `in_mem`** — NOT from `prefilter_mem`. `in_mem` stores the tail of the *filtered* signal from the previous frame, giving continuity across frame boundaries.

2. **Apply comb filter** with NEGATIVE gains (analysis = inverse of synthesis):
   - First `offset = shortMdctSize - overlap` samples: constant filter using old period/gain
   - Remaining `N - offset` samples: crossfade from old period/gain to new period/gain

   The filter is always applied, even when gain1=0, because the crossfade from old_gain to 0 must be smooth.

3. **Cancel check** (mono: `after > before`; stereo: per-channel threshold). If the filter made the signal worse, revert by copying unfiltered `pre[]` back and crossfading to gain=0.

### Phase 3: Memory update
- `in_mem` ← tail of filtered `in[]` (last `overlap` samples)
- `prefilter_mem` ← tail of UNFILTERED `pre[]` (last `max_period` samples)

The two memories serve different purposes: `in_mem` gives filtered overlap for the next frame's MDCT window, while `prefilter_mem` gives unfiltered history for the next frame's pitch search and comb filter reference.

## What the Rust code gets wrong

Five issues, in order of impact:

| # | Bug | Effect |
|---|-----|--------|
| 1 | Comb filter skipped when `gain1 == 0` | No crossfade from old_gain → 0; discontinuity at frame boundary |
| 2 | Overlap copied from `prefilter_mem` instead of `in_mem` | Wrong overlap signal (unfiltered vs filtered) |
| 3 | `prefilter_mem` updated from filtered input instead of unfiltered `pre[]` | Corrupts pitch search history for next frame |
| 4 | Missing cancel_pitch check | Filter can worsen signal quality without being reverted |
| 5 | Several pitch estimation details differ from C | gain1, gain scaling (×0.7), threshold adjustment, gain hysteresis |

## Design

Replace `run_prefilter` with a faithful port of the C function. The Rust struct already has both `in_mem` and `prefilter_mem` fields; we just need to use them correctly.

### Data flow

```
pre[] = prefilter_mem ++ input[overlap..]    (unfiltered work buffer)
                                              │
                              pitch_search ◄──┘
                                   │
                            pitch_index, gain1
                                   │
in[0..overlap] = in_mem            │  (filtered overlap from prev frame)
                    │              │
                    ▼              ▼
in[overlap..] = comb_filter(pre[max_period..], -old_gain → -gain1)
                    │
              cancel check: if worse, revert from pre[]
                    │
in_mem       ◄── in[N..N+overlap]      (save filtered tail)
prefilter_mem ◄── pre[N..N+max_period]  (save unfiltered tail)
```

### Function signature

Keep the existing signature but add `tf_estimate` and `nb_available_bytes` back as active parameters (they're currently prefixed with `_` and ignored):

```rust
fn run_prefilter(
    st: &mut CeltEncoder,
    input: &mut [i32],      // in-place: modified by comb filter
    cc: i32,
    n: i32,

```

### Implementation plan

The function has three sections. Rewrite each to match C:

**Section A: Pitch estimation (lines 1444-1546 in current Rust)**

Changes needed:
- Build `pre[]` buffer matching C layout: `pre[0..max_period]` = prefilter_mem, `pre[max_period..]` = input frame
- Pitch search should use `pre[]`, same as C's `pitch_downsample(pre, ...)`  
- C uses `pitch_index = max_period - pitch_index` after `pitch_search` — verify Rust matches
- Add tone detection override path (C lines 1447-1467): when `toneishness > 0.99`, compute pitch from tone frequency directly
- Scale gain by 0.7: `gain1 = MULT16_16_Q15(0.7_Q15, gain1)` — C line 1487
- Add loss_rate attenuation (C lines 1489-1494)
- Add adaptive threshold (C lines 1506-1526): adjust `pf_threshold` based on pitch continuity, available bytes, and previous gain
- Add gain hysteresis (C line 1535-1536): if gain is close to previous, snap to previous
- Dequantize: `gain1 = 0.09375_Q15 * (qg + 1)` — C line 1544

**Section B: Comb filter application (lines 1549-1590 in C)**

Rewrite completely:
1. Copy overlap from `st.in_mem` (not `prefilter_mem`)
2. Compute `offset = mode.short_mdct_size - overlap`
3. Compute `before[]` energy (sum of abs values, shifted right by 12)
4. Apply comb filter in two segments:
   - `[overlap..overlap+offset]`: constant filter with old period/gain (if offset > 0)
   - `[overlap+offset..overlap+N]`: crossfade filter from old to new period/gain
5. Compute `after[]` energy
6. Cancel check: if `after[0] > before[0]` (mono), revert and set gain=0

Note: C passes NEGATIVE gains to `comb_filter` (analysis filter). The current Rust passes positive gains — this is another bug.

**Section C: Memory update (lines 1592-1602 in C)**

Rewrite:
- `in_mem[c*overlap .. (c+1)*overlap] = input[c*(N+overlap)+N .. c*(N+overlap)+N+overlap]`
- `prefilter_mem[c*max_period .. (c+1)*max_period] = pre[c][N .. N+max_period]` (if N > max_period, direct copy; otherwise shift+copy)

### comb_filter changes

The existing `comb_filter` function (encoder.rs:235) has one bug:

Line 264: `let g10 = mult16_16_p15(g1, COMB_GAINS[tapset1 as usize][1]);`

This shadows `g10` from line 263. It should be `g11`:
```rust
let g11 = mult16_16_p15(g1, COMB_GAINS[tapset1 as usize][1]);
```

Otherwise the function structure is correct. The caller needs to pass negative gains for the analysis filter, and the `x_offset` parameter already handles the history buffer indexing.
