## Problem Statement

The 2026-05-07 overnight 8 h fuzz campaign found two real C-vs-Rust parity bugs (`9b8c552`, `55e224c`) in the `fuzz_encode_multiframe` target. Both have the same shape:

> C mutates a state field in place; Rust takes `&State` and compensates at the call site; one call site got missed.

Both bugs were found on **mono** inputs (`channels=1`). The existing target at `/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_encode_multiframe.rs` does draw `channels_minus_one ∈ 0..=1`, so it can in principle reach stereo, but:

1. The `Arbitrary`-derived path picks each option uniformly. With ~50% mono, ~50% stereo at the channel-count gate alone, libFuzzer's coverage feedback then biases corpus selection by what reaches new edges. Mono paths reach novel SILK PLC / LBRR paths that draw the corpus that way, so over time the corpus skews mono-heavy.
2. The target never calls `set_force_channels`, so the channel-coercion paths in `OpusEncoder::encode_native` (`/home/md/language/ropus/ropus/src/opus/encoder.rs:1816-1838`, the `force_channels != OPUS_AUTO && channels == 2` branch and the `stream_channels` hysteresis branch) are reached only by the default `OPUS_AUTO` policy. Mid-stream transitions like `set_force_channels(1) → set_force_channels(2) → set_force_channels(OPUS_AUTO)` are never exercised.
3. Stereo introduces real state not present in mono: `width_mem` for `compute_stereo_width` (`encoder.rs:1133`), `hybrid_stereo_width_q14`, `stream_channels` hysteresis history, ms_pred state inside SILK, and (under `application = OPUS_AUTO/AUDIO` at 16-32 kbps) the theta-RDO stereo encoding decision (`/home/md/language/ropus/wrk_docs/2026.04.04 - HLD - theta-rdo-stereo-encoding.md`).

These are exactly the kind of code paths where the C-mutates-in-place / Rust-takes-`&State` pattern that produced the prior bugs is most likely to repeat. There's a lot of stereo-only state, and most of it was effectively unfuzzed in the 957 M execs of the prior campaign (the multiframe target's mono share dominated).

This design covers test-only fuzz infrastructure. It does not change codec behavior, run a fuzz campaign, or alter oracle taxonomy. It introduces a new fuzz target that pins `channels=2`, biases input parameters toward configurations that engage stereo-specific code paths, and exercises `set_force_channels` mid-stream.

## Stakeholder Value

Maintainers need a fuzz target whose corpus pressure is directly aligned with stereo state-accumulation paths. The existing multiframe target proved its value by finding two bugs in two minutes; building a sibling target focused on the unfuzzed half of the input space (stereo) is the highest-EV next step in the campaign.

This stream adds value by:

- Forcing `channels=2` so 100% of executions exercise stereo state, instead of an emergent ~30-40% after corpus bias.
- Driving `set_force_channels` between `OPUS_AUTO`, `1`, and `2` per frame, so the channel-coercion branches and the `stream_channels` hysteresis history get systematic coverage.
- Biasing bitrate, bandwidth, and application toward configurations known to engage ms_pred (the 16-32 kbps stereo CELT/Hybrid range, MUSIC application).
- Reusing the existing differential oracle (byte-exact frame-by-frame compare against the C reference) — no new oracle, no new comparison logic to validate.
- Sharing `c_reference.rs`, `frame_duration.rs`, `oracle.rs`, and the panic-capture infrastructure unchanged via `#[path = "..."]` includes — no duplication.
- Giving libFuzzer a separate corpus directory so stereo seeds don't have to compete with mono seeds for selection pressure.
- Keeping per-iteration cost in the same order of magnitude as `fuzz_encode_multiframe` (within ~2×), so a campaign run can reasonably allocate workers across both targets.

Out of scope for this HLD: ms_pred-specific instrumentation, coverage-diff measurement infrastructure beyond `cargo +nightly fuzz coverage`, joining or replacing the existing target.

## Considered Options

### Option 1: Bias the existing `fuzz_encode_multiframe` target instead of adding a new one

Change `channels_minus_one = u.int_in_range(1..=1)?` (or weight the draw 1:8 mono:stereo), add `set_force_channels` to the per-frame setter shuffle, and bias the application/bitrate distribution.

Pros:
- One target to maintain; one corpus to seed; less Cargo.toml churn.
- Existing libFuzzer corpus partially seeds the change.

Cons:
- **Corpus cross-pollination.** libFuzzer's coverage-guided corpus will keep mono-discovered seeds even after the channel gate flips because they still reach novel edges. The corpus that took 957 M execs to build under mono-favored selection becomes a low-quality starting point for stereo: many seeds spend their budget on paths the new target's grammar can no longer reach (force_channels=1 forcibly downmixes), and seeds that *would* find new stereo edges have to be invented from scratch.
- **Reproducibility regression.** Existing crash artifacts in `tests/fuzz/known_failures/multiframe-*` decode under the existing grammar. Changing the grammar shifts every byte-position interpretation, so those archived inputs no longer trip their original bugs without manual conversion. They become dead crash records.
- **Loss of mono coverage.** The existing target's primary value is exhibited: it found 2 bugs on mono. Reducing mono share dilutes future mono-bug discovery.
- **Triage attribution.** A single target with two competing biases makes "is this a stereo bug or a mono bug?" a triage step on every finding rather than an obvious property of the target name.

Decision: **reject.** The corpus and reproducibility costs are real, and the campaign workstream split (Stream A widens the existing target's mono+stereo grammar generally; Stream C is the *focused* stereo target) explicitly assumes two separate corpora.

### Option 2 (Recommended): New target `fuzz_encode_multiframe_stereo`, separate corpus, shared infrastructure

Add a new fuzz binary alongside the existing one. Pin `channels=2`, drive `set_force_channels` per frame, bias bitrate/bandwidth/application toward ms_pred-engaging configurations. Reuse `c_reference.rs`, `frame_duration.rs`, `oracle.rs`, and the panic-capture pattern via `#[path = "..."]` includes. Same byte-exact differential oracle as the parent target — not a new oracle, just a stereo-biased input distribution.

Pros:
- Independent libFuzzer corpus directory `tests/fuzz/corpus/fuzz_encode_multiframe_stereo/`. Every saved seed earns its slot by reaching a new edge under the stereo-pinned grammar; no mono dilution.
- Independent crash artifact directory. Stereo bugs land in `tests/fuzz/artifacts/fuzz_encode_multiframe_stereo/`, attributable on sight.
- Existing target keeps its corpus, its archived crashes still reproduce, its mono coverage is undiluted.
- Triage is unambiguous: any artifact in this target is by construction a stereo input.
- Coverage diff against the parent is mechanically straightforward: run both, diff the LLVM coverage profiles.

Cons:
- ~Doubles per-iteration FFI overhead vs Option 3 (one Rust encoder + one C encoder per iteration). Acceptable: the parent target lives at the same cost.
- Requires one new entry in `tests/fuzz/Cargo.toml`.
- New file is mostly a copy of the parent target with a different Arbitrary impl; some duplication (encode loop, assertion blocks).

Decision: **accept.** Stream C's brief is "focused stereo target," and the corpus-isolation argument is exactly the reason the kickoff journal split A and C into separate workstreams.

### Option 3: New target that creates one stereo encoder once and runs many short sequences against it

Persist the encoder across iterations (pseudo-stateful fuzz, similar to libFuzzer's `LLVMFuzzerInitialize`).

Pros:
- Amortizes encoder construction.
- May reach deeper state-accumulation paths (encoder ages over many frames).

Cons:
- libFuzzer iterations are not contiguous in input space; the encoder ages on whichever inputs the corpus serves, not on a logical voice/music stream. The state path is undefined.
- C-side encoder also has to persist; `c_encode_multiframe` builds a fresh `OpusEncoder*` per call. Mirroring across iteration boundaries is non-trivial and easy to get wrong (one side resets, the other doesn't → false positives forever).
- Hard to reproduce a crash because the encoder state at the time of crash depends on every prior iteration's input.

Decision: **reject.** Per-iteration construction is a feature, not a bug — it makes every crash artifact self-contained.

### Option 4: Drive ms_pred / theta-RDO via deterministic test vectors instead of fuzz

Hand-craft 100-frame test sequences that walk the stereo decision tree.

Pros:
- Targeted; cheap; reproducible.

Cons:
- This is the work the C-vs-Rust differential param grid (`tests/c_ref_differential_param_grid`) already does. It found nothing for stereo because stereo is the path our directed tests under-cover, but adding more directed tests doesn't address the "we don't know what we don't know" gap that fuzzing is designed to close.
- Doesn't compose with the existing 8h overnight infrastructure.

Decision: **reject.** Directed tests are a complement to fuzz, not a substitute. Out of scope for this stream.

## Recommended Design

### Shared Principles

- **Pin** `channels=2`. Mono is out of scope.
- **Reuse** the byte-exact differential oracle from `fuzz_encode_multiframe`. No new oracle, no new comparison logic.
- **Reuse** `c_reference.rs`, `frame_duration.rs`, `oracle.rs`, and the panic-capture infrastructure via `#[path = "..."]` includes. No code duplication beyond the encode-loop scaffolding (which is small and target-specific).
- **Bias** the input grammar toward configurations that engage ms_pred and theta-RDO:
  - Bitrate range that includes the 16-32 kbps zone where stereo↔mono and ms_pred↔LR decisions are sharpest.
  - Bandwidth range that engages CELT-only and Hybrid (where ms_pred is most non-trivial).
  - Application weighted toward `OPUS_APPLICATION_AUDIO` (which engages MUSIC-leaning thresholds).
  - Sample rate restricted to 16/24/48 kHz (drop 8 kHz; see below).
- **Drive** `set_force_channels` per frame across {1, 2, OPUS_AUTO} so the channel-coercion paths and `stream_channels` hysteresis history get systematic coverage.
- Keep per-iteration cost within ~2× of the parent target. The parent ran ~16 K execs total (it spent most of its time triaging crashes); throughput-budget should not regress materially.

### Sample rate selection

Pin `sample_rate ∈ {16000, 24000, 48000}`. Drop 8 kHz and 12 kHz.

Reasoning:
- **8 kHz stereo** is degenerate for ms_pred coverage. At 8 kHz the encoder is locked into NB; no Hybrid; no CELT-only above 24 kbps. Stereo-width estimation runs but the decisions it drives are gross — it boils down to "downmix or don't." The interesting MS prediction happens in CELT bands above NB. Including 8 kHz would burn ~20% of fuzz cycles on a path the parent target already covers adequately for mono.
- **12 kHz stereo** is similarly low-yield for ms_pred (MB only). The parent target covers it; this target shouldn't compete.
- **16 kHz** engages WB and reaches Hybrid in MUSIC application — first interesting band for stereo MS coverage.
- **24 kHz** engages SWB; SILK MS prediction kicks in at this rate.
- **48 kHz** is the canonical full-band stereo configuration; the most ms_pred state.

Drop the 5-element `SAMPLE_RATES` constant; use a 3-element `STEREO_SAMPLE_RATES`.

### Application selection

Weight `application_idx` toward `OPUS_APPLICATION_AUDIO` (the MUSIC-leaning application). Rough split:

- 4/8 chance: `OPUS_APPLICATION_AUDIO`
- 3/8 chance: `OPUS_APPLICATION_VOIP`
- 1/8 chance: `OPUS_APPLICATION_RESTRICTED_LOWDELAY`

Reasoning:
- `OPUS_APPLICATION_AUDIO` flows into the MUSIC-leaning `STEREO_MUSIC_THRESHOLD` (17000) and the theta-RDO path. This is the highest-yield application for stereo bugs.
- `VOIP` is still useful — it engages SILK's stereo prediction (`pred_state_Q13` etc.), which has its own state-mutation pattern.
- `RESTRICTED_LOWDELAY` forces CELT-only; useful but a smaller surface than AUDIO.

Implementation: pick from a 8-entry weighted lookup table, indexed by a single u8.

### Bitrate selection

Bias the bitrate distribution to over-sample the 16-32 kbps zone (ms_pred decision boundary):

- 1/2 chance: bitrate ∈ [16000, 64000] (the ms_pred-engaging zone, with a long tail into Hybrid)
- 1/4 chance: bitrate ∈ [6000, 16000) (forced-mono / heavily-coupled zone, exercises downmix decision)
- 1/4 chance: bitrate ∈ [64000, 510000] (high-rate, theta-RDO unrestricted)

`raw_to_bitrate_stereo(raw: u32) -> i32` does this with a single u32 decoded as `(zone_selector_2_bits, value_30_bits)`. Alternative: keep the parent's `raw_to_bitrate(u16)` and shift the seam. Either is fine; the zone-selector form is more legible.

### Bandwidth selection

Drive `set_max_bandwidth` per frame across {WIDEBAND, SUPERWIDEBAND, FULLBAND, OPUS_AUTO} (skip NB/MB — they don't engage interesting stereo paths). Distribution roughly uniform, biased slightly toward FULLBAND (which engages the most CELT bands).

This is the field where ms_pred is most relevant — CELT MS prediction operates per-band, and more bands means more opportunities for divergence.

### force_channels CTL distribution per frame

Per frame, draw `force_channels_idx: u8 % 4` and dispatch:

- 0 → `set_force_channels(OPUS_AUTO)` (-1000)
- 1 → `set_force_channels(1)` (force mono downmix)
- 2 → `set_force_channels(2)` (force stereo)
- 3 → no change (inherit prior frame's setting)

The "no change" fourth option matters: it lets a force_channels=1 setting carry across multiple frames, exercising the case where stream_channels stays mono-coerced for several frames in a row, which is the regime where SILK's stereo prediction state can desync from CELT's expectation.

### Signal type

Drive `set_signal` per frame across {OPUS_AUTO, OPUS_SIGNAL_VOICE, OPUS_SIGNAL_MUSIC} with weight 2:1:2 (auto and music heavier than voice — voice forces SILK-only at 8/16/24 kHz which we want to *probe* but not dominate).

### Prediction disabled

Drive `set_prediction_disabled` per frame as a uniform bool. **No INHERIT slot for this CTL** — Stream A applies `prediction_disabled` always, and stereo + `prediction_disabled=1` is the highest-yield combo for the campaign (the 2nd campaign-found bug at `55e224c` was in the SILK LBRR / `ec_prev_*` neighborhood, which is exactly what this CTL toggles). Stereo SILK has additional state in `pred_state_Q13` that prediction-disabled stresses differently from the enabled path.

This is added on top of the parent's grammar, not in place of any other stereo dimension. Field count goes up by one.

### Per-frame fields (Arbitrary impl)

```rust
struct StereoMultiframeInput {
    sample_rate_idx: u8,           // 0..=2 → {16000, 24000, 48000}
    application_idx: u8,           // weighted to AUDIO
    frame_duration_selector: u8,   // shared with parent (frame_duration.rs)
    n_frames: u8,                  // 2..=12 (smaller ceiling than parent's 16; see cost)
    frames: Vec<StereoFrameConfig>,
}

struct StereoFrameConfig {
    bitrate_raw: u32,              // mapped via raw_to_bitrate_stereo
    complexity: u8,                // 0..=9 (same analysis.c dodge as parent)
    vbr: bool,
    vbr_constraint: bool,          // tightens stereo CBR path; same dimension Stream A adds
    fec: u8,                       // 0..=2
    dtx: bool,
    loss_perc: u8,                 // 0..=100
    force_channels_idx: u8,        // 0..=3 (auto / 1 / 2 / inherit)
    bandwidth_idx: u8,             // 0..=4 (WB/SWB/FB/auto/inherit)
    signal_idx: u8,                // 0..=3 (weighted auto/voice/music + inherit)
    prediction_disabled: bool,     // SILK LTP toggle — highest-yield combo with stereo
    pcm_bytes: Vec<u8>,            // sized for stereo at the selected fs+duration
}
```

Field-byte count per frame: ~17 bytes of config + PCM. PCM at 60 ms / 48 kHz / stereo i16 = 11520 bytes. Worst-case input shape: 12 × (17 + 11520) = 138 KB. Within libFuzzer's default `max_len` budget (1 MB).

### Encode loop sketch

```rust
fuzz_target!(|input: StereoMultiframeInput| {
    init_panic_capture();
    record_fingerprint(&input);

    let sample_rate = STEREO_SAMPLE_RATES[input.sample_rate_idx as usize];
    let channels = 2;
    let application = APPLICATIONS_WEIGHTED[input.application_idx as usize];
    let frame_size = frame_duration::legal_frame_size_samples_per_channel(
        sample_rate,
        input.frame_duration_selector,
    );
    let samples_per_frame = (frame_size as usize) * (channels as usize);

    // Decode PCM frames (chunks_exact mirrors parent).
    let pcm_frames: Vec<Vec<i16>> = input.frames.iter()
        .map(|fc| fc.pcm_bytes.chunks_exact(2)
            .map(|c| i16::from_le_bytes([c[0], c[1]]))
            .collect())
        .collect();

    // Build per-frame configs using Stream A's widened CEncodeConfig.
    // INHERIT sentinels mark CTLs that should be skipped this frame
    // (carries prior frame's setting forward).
    let frame_cfgs: Vec<CEncodeConfig> = input.frames.iter()
        .map(|fc| CEncodeConfig {
            bitrate: raw_to_bitrate_stereo(fc.bitrate_raw),
            complexity: fc.complexity as i32,
            application,
            vbr: fc.vbr as i32,
            vbr_constraint: fc.vbr_constraint as i32,
            inband_fec: fc.fec as i32,
            dtx: fc.dtx as i32,
            loss_perc: fc.loss_perc as i32,
            // Stream A's four new fields. force_channels / max_bandwidth /
            // signal carry INHERIT sentinels here; prediction_disabled is
            // always-applied per Stream A semantics.
            max_bandwidth: bandwidth_value(fc.bandwidth_idx), // may be BANDWIDTH_INHERIT
            signal: signal_value(fc.signal_idx),              // may be SIGNAL_INHERIT
            force_channels: force_channels_value(fc.force_channels_idx), // may be FORCE_CHANNELS_INHERIT
            prediction_disabled: fc.prediction_disabled as i32,
        })
        .collect();

    // --- Rust encode ---
    let mut rust_enc = OpusEncoder::new(sample_rate, channels, application).ok()?;
    let mut rust_packets = Vec::with_capacity(input.frames.len());
    for (pcm, cfg) in pcm_frames.iter().zip(frame_cfgs.iter()) {
        // Always-apply CTLs (parent's eight + Stream A's prediction_disabled).
        rust_enc.set_bitrate(cfg.bitrate);
        rust_enc.set_vbr(cfg.vbr);
        rust_enc.set_vbr_constraint(cfg.vbr_constraint);
        rust_enc.set_inband_fec(cfg.inband_fec);
        rust_enc.set_dtx(cfg.dtx);
        rust_enc.set_packet_loss_perc(cfg.loss_perc);
        rust_enc.set_complexity(cfg.complexity);
        rust_enc.set_prediction_disabled(cfg.prediction_disabled);
        // Inherit-aware CTLs (Stream A's other three).
        if cfg.force_channels != FORCE_CHANNELS_INHERIT {
            rust_enc.set_force_channels(cfg.force_channels);
        }
        if cfg.max_bandwidth != BANDWIDTH_INHERIT {
            rust_enc.set_max_bandwidth(cfg.max_bandwidth);
        }
        if cfg.signal != SIGNAL_INHERIT {
            rust_enc.set_signal(cfg.signal);
        }

        let mut out = vec![0u8; 4000];
        let len = rust_enc.encode(pcm, frame_size, &mut out, 4000).ok()?;
        out.truncate(len as usize);
        rust_packets.push(out);
    }

    // --- C reference encode (uses inherit-aware sibling helper) ---
    let frame_refs: Vec<&[i16]> = pcm_frames.iter().map(|f| f.as_slice()).collect();
    let c_packets = c_reference::c_encode_multiframe_inherit(
        &frame_refs, &frame_cfgs, frame_size, sample_rate, channels,
    ).ok()?;

    // --- Differential comparison: same byte-exact frame-by-frame as parent ---
    assert_eq!(rust_packets.len(), c_packets.len(), /* ... */);
    for (i, (rust_pkt, c_pkt)) in rust_packets.iter().zip(c_packets.iter()).enumerate() {
        assert_eq!(rust_pkt.len(), c_pkt.len(), /* with stereo-specific context */);
        assert_eq!(rust_pkt, c_pkt, /* with stereo-specific context */);
    }

    // Same parseability + decode-sample-count invariants as parent.
});
```

### `c_reference.rs` — INHERIT sentinels and sibling helper

Stream A widens `CEncodeConfig` to 12 fields total (8 existing + `max_bandwidth`, `signal`, `force_channels`, `prediction_disabled`) and widens `apply_c_encoder_config` with always-apply semantics for those four CTLs. **Stream C reuses Stream A's widened type — no parallel `StereoCEncodeConfig`.** The only extension Stream C contributes is INHERIT semantics on top of Stream A's struct: three sentinel constants and one sibling helper that conditionally skips three of the new CTLs.

Concretely, add to `c_reference.rs` (additions only — `CEncodeConfig` itself is owned by Stream A):

```rust
/// Sentinel value: "do not apply this CTL on this frame".
///
/// Safe choice of `i32::MIN`: the legal values for the three INHERIT-capable
/// CTLs are bounded:
///
/// - `OPUS_AUTO` = -1000 (smallest legal value across all three CTLs)
/// - `OPUS_BANDWIDTH_*` = small positive ints (1101..=1105)
/// - `OPUS_SIGNAL_VOICE` / `OPUS_SIGNAL_MUSIC` = 3001 / 3002
/// - `force_channels` ∈ {OPUS_AUTO, 1, 2}
///
/// `i32::MIN` (-2_147_483_648) is unambiguously outside every legal range,
/// so it cannot collide with any value the encoder would accept. The sibling
/// helper checks for this exact value and skips the CTL when matched.
pub const FORCE_CHANNELS_INHERIT: i32 = i32::MIN;
pub const BANDWIDTH_INHERIT: i32 = i32::MIN;
pub const SIGNAL_INHERIT: i32 = i32::MIN;

/// Inherit-aware sibling of `apply_c_encoder_config`.
///
/// Identical to Stream A's `apply_c_encoder_config` for every always-apply
/// CTL (bitrate, vbr, vbr_constraint, inband_fec, dtx, loss_perc, complexity,
/// prediction_disabled). The only difference: `force_channels`, `max_bandwidth`,
/// and `signal` are skipped when their value equals their respective INHERIT
/// sentinel. Stream C uses this from `c_encode_multiframe_inherit`; the parent
/// target keeps using `apply_c_encoder_config` directly.
unsafe fn apply_c_encoder_config_with_inherit(
    enc: *mut OpusEncoder,
    cfg: &CEncodeConfig,
) {
    // Always-apply CTLs (parent's eight + Stream A's prediction_disabled).
    opus_encoder_ctl(enc, OPUS_SET_BITRATE_REQUEST, cfg.bitrate);
    opus_encoder_ctl(enc, OPUS_SET_VBR_REQUEST, cfg.vbr);
    opus_encoder_ctl(enc, OPUS_SET_VBR_CONSTRAINT_REQUEST, cfg.vbr_constraint);
    opus_encoder_ctl(enc, OPUS_SET_INBAND_FEC_REQUEST, cfg.inband_fec);
    opus_encoder_ctl(enc, OPUS_SET_DTX_REQUEST, cfg.dtx);
    opus_encoder_ctl(enc, OPUS_SET_PACKET_LOSS_PERC_REQUEST, cfg.loss_perc);
    opus_encoder_ctl(enc, OPUS_SET_COMPLEXITY_REQUEST, cfg.complexity);
    opus_encoder_ctl(enc, OPUS_SET_PREDICTION_DISABLED_REQUEST, cfg.prediction_disabled);
    // Inherit-aware CTLs.
    if cfg.force_channels != FORCE_CHANNELS_INHERIT {
        opus_encoder_ctl(enc, OPUS_SET_FORCE_CHANNELS_REQUEST, cfg.force_channels);
    }
    if cfg.max_bandwidth != BANDWIDTH_INHERIT {
        opus_encoder_ctl(enc, OPUS_SET_MAX_BANDWIDTH_REQUEST, cfg.max_bandwidth);
    }
    if cfg.signal != SIGNAL_INHERIT {
        opus_encoder_ctl(enc, OPUS_SET_SIGNAL_REQUEST, cfg.signal);
    }
}

    sample_rate: i32,
    channels: i32,

    /* same loop body as c_encode_multiframe, swapping in
       apply_c_encoder_config_with_inherit at the per-frame setter point */
}
```

The `INHERIT` sentinels match the per-frame `force_channels_idx == 3` / `bandwidth_idx == 4` / `signal_idx == 3` ("no change") behavior — the helper omits the CTL call rather than re-applying a value, mirroring the Rust setter loop.

**Naming note**: chose `c_encode_multiframe_inherit` over `c_encode_multiframe_stereo` because the inherit semantics are the actual property the helper provides; the stereo bias lives in the new target's grammar, not here. This keeps the helper reusable for any future fuzz target that wants the same skip semantics without pinning channels.

**No struct duplication, no fork of always-apply semantics** — Stream C's surface in `c_reference.rs` is three `pub const`s plus one `unsafe fn` plus one `pub fn`. ~50 LoC.

### CTLs exercised mid-stream vs fixed at construction

Fixed at construction: `sample_rate`, `channels` (=2), `application`.
Per-frame (mid-stream): `bitrate`, `vbr`, `vbr_constraint`, `inband_fec`, `dtx`, `packet_loss_perc`, `complexity`, `prediction_disabled`, **`force_channels`**, **`max_bandwidth`**, **`signal`**.

Stream A adds `vbr_constraint`, `prediction_disabled`, `force_channels`, `max_bandwidth`, and `signal` to the parent target as always-apply per-frame setters. Stream C inherits all of those (via the widened `CEncodeConfig`) and adds INHERIT-skip semantics for the bolded three. `prediction_disabled` is always-apply on both sides, matching Stream A.

### Per-iteration cost vs the parent target

Parent target per-iteration cost (rough):
- 1 Rust encoder construction
- 1 C encoder construction
- N frames × (Rust encode + C encode + 2 byte-equal compare + parseability check + Rust decode)
- N ∈ 2..=16 typically.

Stereo target per-iteration cost:
- Same shape as parent post-Stream-A, with the added cost of three branch checks per frame (the INHERIT skip checks). Setter call counts on each side are the same as the parent's post-Stream-A grammar (8 always-apply + up to 3 conditional inherit-aware = 8..=11 calls per frame). At parent worst-case the two grammars differ only in branch checks, not CTL count.
- N ∈ 2..=12 (slightly tighter ceiling than parent's 16 to keep PCM byte budget within max_len at 60 ms × 48 kHz × stereo = 11520 B/frame).
- Each frame is 2× the PCM bytes of the parent's mono case at the same sample rate, but the per-byte cost in encode/decode scales sub-linearly (the bit allocator and range coder dominate, not PCM iteration).

Expected throughput: ~0.6-0.8× of parent (per-frame cost goes up ~10-20%, per-frame PCM doubles, frame count ceiling drops from 16 to 12). Order of magnitude matches the parent. Acceptable.

## Validation Plan

Validation is split into two phases:

1. **Initial-commit gates (mandatory, blocking)** — health checks: build clean, target loads, smoke run clean. These prove the target *works*, not that it's *valuable*.
2. **Campaign-end gates (mandatory, blocking, but not at commit time)** — coverage-diff against the parent target. This is the actual *broadening signal* for a new (cold-corpus) target. Stream C cannot be declared successful until this gate passes — but the coverage-diff toolchain has its own footguns, so we don't block the implementation commit on it.

The smoke gate (exec/sec ≥ 30, 0 crashes, 0 timeouts) is a *health* gate, not a *broadening* gate. For a cold-corpus target, "broadening" means "reaches stereo-specific paths the parent doesn't" — measured by coverage-diff, not by raw new-units rate.

## Risks and Tradeoffs

- **Bias parameters might be wrong.** If the bitrate / bandwidth / application weighting doesn't actually engage ms_pred more than the parent, we've spent a target slot without gaining stereo coverage density. Mitigated by the coverage-diff validation step. If wrong, the bias parameters are tunable without restructuring the target — it's a small follow-up.
- **`STEREO_SAMPLE_RATES` excludes 8 / 12 kHz, narrowing input space.** Justified above (8 kHz stereo NB has a much smaller MS surface than WB+ stereo). If post-fact we discover an 8-kHz-stereo bug class, the parent target still covers it (uniform sample-rate draw). Not a coverage regression.
- **Stale-corpus risk.** A new target starts cold, so the first hours of fuzzing are pure exploration. Smoke validation only checks 60 seconds; real bug-finding density emerges over longer runs. Mitigation: campaign-scale runs (4-8 h per worker), not the smoke validation itself.
- **`force_channels=1` mid-stream might trigger a known-unfixed C-vs-Rust state-reset divergence.** If the C encoder mutates more state on `force_channels=1` than Rust does (or vice versa), every iteration that hits force_channels_idx=1 will diverge. **This is the bug class we're trying to find.** If it's a known issue, archive the artifact under `tests/fuzz/known_failures/stereo-force-channels-state-divergence/` and adjust the bias to skip the offending CTL combo until the bug is fixed, mirroring the analysis.c complexity≤9 dodge in the parent.
- **Per-iteration cost is at the upper end.** 12 frames × 60 ms × 48 kHz × stereo PCM is non-trivial. If exec/sec drops below 30 in smoke, drop the n_frames ceiling from 12 to 8.
- **`vbr_constraint` is a new dimension.** Parent always passes 0. Adding it to this target tightens the CBR path, which has its own state (target_bps tracking). If this finds a CBR bug, it's still a stereo-relevant finding because CBR + stereo is the configuration where bit-allocation between channels is tightest.
- **No new oracle means no new false-positive class.** This is a feature: every divergence this target reports is a real C-vs-Rust byte mismatch under valid input.

## Out of Scope

- **No coverage-instrumented production runs.** Coverage diff is a one-off validation step, not part of the fuzzing infrastructure.
- **No mutation of `CEncodeConfig` or the existing `apply_c_encoder_config`.** Stream A owns those. Stream C only adds INHERIT sentinels and a sibling helper next to them.
- **No changes to `c_encode_multiframe` itself** — Stream C adds a sibling `c_encode_multiframe_inherit`, not a replacement.
- **No changes to `frame_duration.rs` / `oracle.rs` / panic capture.** Reuse only.
- **No changes to the parent target.** Stream A owns parent-target grammar broadening.
- **No hybrid mode transition focus.** Stream D owns SILK↔CELT mode transitions. If a stereo configuration naturally hits hybrid (it will, frequently — Hybrid is the SWB stereo zone), great; we don't avoid it, but we don't engineer for it either.
- **No multistream surround coupling.** `fuzz_multistream` covers multistream; this target is non-multistream stereo only.
- **No DRED / DNN paths.** Out of scope for encoder differential.
- **No initial seed corpus.** Cold start. If we want seeded coverage later (e.g., copy a few stereo seeds from `fuzz_encode`), it's a follow-up.
- **No fuzz campaign run.** Implementation produces the target; the team-lead's campaign phase runs it.

## Key Decisions

- **Sequenced after Stream A.** Reuses Stream A's widened `CEncodeConfig` (12 fields) and `apply_c_encoder_config`. No struct duplication.
- New target `fuzz_encode_multiframe_stereo`, separate corpus, separate artifacts.
- Pin `channels=2`. Drop 8 kHz and 12 kHz; keep 16/24/48 kHz.
- Application weighting biases toward `OPUS_APPLICATION_AUDIO`.
- Per-frame setters drive `force_channels`, `max_bandwidth`, `signal` with INHERIT semantics; `vbr_constraint`, `prediction_disabled`, and parent's eight setters are always-apply.
- `prediction_disabled` is a stereo grammar dimension — highest-yield combo with stereo (campaign bug `55e224c` was in this neighborhood; stereo SILK has additional `pred_state_Q13` state).
- INHERIT sentinels are `i32::MIN` for all three inherit-aware CTLs — unambiguously outside every legal CTL value range (legal values span -1000..=3002).
- `c_reference.rs` additions: three `pub const`s, one `unsafe fn apply_c_encoder_config_with_inherit`, one `pub fn c_encode_multiframe_inherit`. ~50 LoC total. No fork of Stream A's struct or always-apply helper.
- `force_channels` distribution over {AUTO, 1, 2, INHERIT} is uniform; the INHERIT slot is what makes mid-stream coercion testable across multi-frame runs.
- Bitrate distribution biases toward 16-32 kbps to engage ms_pred decisions; CBR via `vbr_constraint` is on the table per-frame.
- Reuse byte-exact differential oracle from parent. Do not introduce a new oracle.
- Frame count ceiling 12 (vs parent's 16) to keep per-iteration PCM budget under control at 60 ms / 48 kHz / stereo.
- **Two-phase validation**:
  - *Phase 1 (commit gate)*: build clean, `-runs=0` clean, 60-second smoke clean (exec/sec ≥ 30, 0 crashes, 0 timeouts).
  - *Phase 2 (Stream-C-done gate, blocking but not at commit)*: coverage-diff against parent shows strictly greater coverage on `compute_stereo_width` (`encoder.rs:1133`) and the `force_channels != OPUS_AUTO` branch (`encoder.rs:1816`), measured after the same wall-clock fuzz time.
