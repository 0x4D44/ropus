## Root cause evidence

The current sanity wrapper replays each crash with this command shape:

```bash
cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_decode -- -runs=0 <crash_file>
```

Manual replay using that same ordering does not reach `fuzz_decode`. It fails in libFuzzer argument handling:

```text
Running `.../fuzz_decode -artifact_prefix=... -runs=0 tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin tests/fuzz/corpus/fuzz_decode`
Not a directory: tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin; exiting
```

Because the fixed input is placed after the libFuzzer `--` separator, `cargo-fuzz 0.13.1` forwards it in a way that libFuzzer treats as a corpus directory. The wrapper suppresses stdout/stderr, so this infrastructure failure is reported as "still crashes".

The correct fixed-input command puts the input before the `--` separator:

```bash
timeout 120 cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_decode \
  tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin -- -runs=0
```

With that ordering, cargo-fuzz runs the file as a fixed input:

```text
fuzz_decode: Running 1 inputs 1 time(s) each.
Running: tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin
Executed tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin in 111 ms
```

I then replayed all four using the corrected fixed-input form:

```bash
timeout 180 bash -c 'set -euo pipefail; for f in tests/fuzz/crashes/fuzz_decode/*.bin; do
  cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_decode "$f" -- -runs=0
done'
```

All four passed. No panic, assertion failure, sanitizer crash, timeout, or non-zero exit was observed.

Tool versions:

```text
cargo-fuzz 0.13.1
cargo 1.97.0-nightly (4f9b52075 2026-05-01)
```

## Classification

Classification:

1. Known unresolved decoder bugs that should remain as failing regression corpus: **No current evidence.**
2. Stale corpus files that no longer match the current target/replay semantics: **Yes.** The files were historic crash corpus entries from `a9803d0`, but they no longer crash the current `fuzz_decode` target when replayed correctly.
3. Duplicate/minimizable variants of the same issue: **Not relevant to the current gate failure.** They may be old variants of previously fixed decode/oracle issues, but the current target accepts all four. Minimization is not needed for this failure.
4. Something caused by the new `--sanity` wrapper: **Yes.** The gate failure is caused by the wrapper's fixed-input replay command ordering and stderr suppression.

## Recommended next action

Do not suppress real crashes. Fix the sanity gate replay command first.

Smallest implementation slice:

1. Change `tools/fuzz_run.sh` crash replay from:

```bash
"${CARGO_FUZZ[@]}" run "${FUZZ_DIR_ARG[@]}" "$target" -- -runs=0 "$crash_file"
```

to:

```bash
"${CARGO_FUZZ[@]}" run "${FUZZ_DIR_ARG[@]}" "$target" "$crash_file" -- -runs=0
```

2. Preserve failure output, at least in sanity mode, so infrastructure errors like `Not a directory` are not misreported as target crashes.
3. Re-run:

```bash
timeout 300 bash tools/fuzz_run.sh --sanity --target fuzz_decode
```

Expected outcome after the command-order fix: `fuzz_decode build=pass crashes=4 replay=pass`.

Follow-up, separate from the gate fix:

The committed `tests/fuzz/crashes/fuzz_decode/*.bin` files are no longer failing regression inputs. Keeping resolved crash files in `crashes/` makes the directory name misleading. After the wrapper is fixed, decide whether this project wants:

- `crashes/` to mean "must fail until fixed", with resolved files moved/promoted into ordinary regression seed corpus after they pass, or
- `crashes/` to mean "historic crash regressions that must now pass", with clear metadata documenting fixed vs expected-failing status.

The current evidence supports a gate policy of "committed crash corpus must replay cleanly" for `--sanity`, not quarantining these four as expected failures and not changing the decoder oracle.
