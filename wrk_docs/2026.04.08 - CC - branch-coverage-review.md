## Scope and Method

This review used `cargo llvm-cov` on nightly because `--branch` still requires nightly on this toolchain. The stable toolchain failed immediately with `-Z coverage-options=branch`, so the usable baseline was collected with `cargo +nightly llvm-cov`.

The coverage command is repeatable locally. Tests passed under coverage: 386 passed, 0 failed.

Operational note:
- `cargo llvm-cov --branch --summary-only` produced one Windows cleanup warning on the first run because `cargo-llvm-cov` could not remove a just-executed test binary from `target\llvm-cov-target`.
- `cargo +nightly llvm-cov clean --workspace` succeeded separately.
- A follow-up run that wrote `target\coverage-summary.json` completed cleanly enough for reporting.

Cross-agent note:
- I attempted to use headless agent CLIs as requested.
- `codex exec` was blocked by sandboxed outbound network restrictions (`wss://api.openai.com` / `https://api.openai.com`).
- `claude -p` was blocked inside the sandbox by an EPERM failure while trying to spawn `C:\WINDOWS\System32\reg.exe`.
- The repo work itself is not blocked, but the multi-agent part is currently constrained by the environment rather than the codebase.

## Headline Findings

1. The gap is structural, not a flaky run.
   The suite is large and stable, but it is concentrated in helper-level tests. The largest cold regions are top-level codec orchestrators and SILK internals.

2. SILK dominates the deficit.
   `src\silk` contributes 8,525 executable lines and 1,550 branches, but only 13.13% line coverage and 7.48% branch coverage.

3. A few files account for most of the misses.
   The worst offenders are large entry-point files, not small utilities. Chipping only at helper functions will not move the total enough.

4. There are still easy local wins.
   Several files with no `#[cfg(test)]` block are small and branchy enough to justify focused unit tests: `src\celt\ec_ctx.rs`, `src\celt\rate.rs`, `src\silk\common.rs`, and `src\main.rs`.

5. 98% repo-wide coverage is far from the current reality.
   The codebase is at 48.94% line and 35.10% branch coverage. Reaching 98% requires heavy execution of full encode/decode paths across SILK, CELT, and Opus orchestrators. Small helper tests alone cannot plausibly close that gap.

## Untested or Thinly Tested Files

Files under `src/` with no local `#[cfg(test)]` block:
- `src\celt\ec_ctx.rs`
- `src\celt\lpc.rs`
- `src\celt\rate.rs`
- `src\main.rs`
- `src\silk\common.rs`

Some files technically have tests but still leave the dominant logic cold:
- `src\opus\encoder.rs`
- `src\opus\decoder.rs`
- `src\opus\multistream.rs`
- `src\silk\encoder.rs`
- `src\silk\decoder.rs`
- `src\celt\bands.rs`
- `src\dnn\core.rs`
- `src\dnn\lpcnet.rs`

Pattern:
- Existing tests skew toward constructors, constants, math helpers, and small pure functions.
- The real misses live in encode/decode orchestration, packet handling variations, state transitions, and bitrate / mode / bandwidth control flow.

## What The Current Tests Are Good At

- Static tables and pure helper math in `celt`, `dnn`, and `types`.
- Packet utility functions in `opus::decoder`.
- Repacketizer utilities in `opus::repacketizer`.
- Narrow constructor / setter behavior in `opus::encoder`, `opus::decoder`, and `opus::multistream`.

## What Is Barely Exercised

- SILK encode path selection and frame processing.
- SILK decoder PLC / transition / redundancy combinations.
- Opus top-level encode/decode orchestrators beyond basic construction and helper functions.
- CELT rate allocation / band skipping logic through real encoder activity.
- DNN larger model/state execution paths.

## Fastest Credible Leverage

Highest leverage without refactoring:

1. Add tests to the currently untested small files.
   - `src\celt\ec_ctx.rs`
   - `src\celt\rate.rs`
   - `src\silk\common.rs`
   - `src\main.rs`

2. Expand top-level Opus encode/decode tests.
   - Real `OpusEncoder::encode`, `encode_float`, `reset`, and CTL state tests.
   - Round-trip `OpusDecoder::decode`, `decode_float`, `decode24`, PLC, and state reset tests.

3. Drive multistream and SILK through real packets rather than helper-only tests.
   - Multi-rate, mono/stereo, VBR/CBR, application, bandwidth, and packet-loss configurations.

4. Only then attack deep SILK branches directly.
   The SILK files are too large to brute-force with micro-tests; they need scenario tests that execute significant shared logic.

## Constraints and Risks

- The crate is a parity-sensitive codec port. Coverage gains that come from broad refactors would be risky and not justified here.
- Some modules are large enough that line coverage can move meaningfully only when end-to-end flows are exercised.
- Branch coverage is much lower than line coverage, especially in SILK and Opus orchestration. Reaching 98% branch coverage is a much harder target than reaching 98% line coverage.
- The repo currently compiles with warnings in untouched code. That is noise, but it is not the present blocker.

## Conclusion

The current suite is healthy but badly imbalanced. The repo has a respectable quantity of tests, yet they do not execute the code that matters most for overall coverage. The shortest path upward is:

1. harvest the obvious untested helper modules,
2. deepen top-level encode/decode tests,
3. use those tests to pull SILK and Opus orchestrators into the report,
4. then iterate on the remaining cold branches with measured follow-up runs.

The baseline is real, the biggest gaps are clear, and the current environment blocks headless sub-agents rather than the code changes themselves.
