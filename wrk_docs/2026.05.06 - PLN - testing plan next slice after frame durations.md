## Blockers And Risks

- The technical blocker is policy, not reproduction: decide what bounded LSB drift is acceptable for the post-288-sample region. Existing comments mention max absolute drift around 17, but the implementation should compute and assert the actual bound from the deterministic fixture.
- A bad slice would try to "fix PLC parity" immediately. That risks parity-sensitive SILK decoder churn without first making the residual drift measurable and release-visible.
- `decode24` is a 24-bit API surface while the existing active first-window guard is 16-bit decode. The slice should either keep both views deliberately or explain why the bounded drift oracle belongs to one path.
- If the deterministic generated packet changes due to encoder behavior, the test could become less stable. The helper should print enough context to make that obvious.

## HLD Focus

The HLD should be short and concrete:

- Define the tiered oracle: exact prefix, bounded residual LSB drift, explicit debt for full-frame byte/sample exactness.
- Explain why recovery/PLC PCM is not treated the same as ordinary coded-frame parity.
- Specify the deterministic fixture shape: 48 kHz mono, 20 ms hybrid/fullband VOIP packet, one priming decode, then one lost frame.
- Name the acceptance numbers only after measuring the current deterministic output in the implementation branch; avoid inventing thresholds in advance.
- State the non-goal: no SILK PLC numeric refactor in the first slice.
