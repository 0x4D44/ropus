# HLD - Real World Corpus Gate

Date: 2026.05.06

Scope: Phase 3.4 testing-improvement slice for real-world corpus gate and corpus provisioning policy. This is a design document only; no production or test code is changed by this HLD.

## Problem Statement

The repository has a `corpus_diff` mechanism for comparing ropus decode output against the C reference on real Opus container files, but it does not yet provide a trustworthy release gate.

The current gap is not "write a decoder harness"; that already exists. The gap is that real-world corpus coverage is still an operator-populated placeholder:

- The checked-in real-world directory contains only documentation.
- `tools/fetch_corpus.sh` is explicit that its public fetch list is a reference-encoded baseline, not non-reference encoder diversity.
- All current fetch-script sources are pinned as `TBD`, so the script refuses them rather than provisioning a usable corpus.
- `corpus_diff` exits non-zero for an empty directory, but a directory with only unsupported/skipped candidates can still finish green because only mismatches and panics affect the final exit.
- `full-test` does not currently run or report a real-world corpus lane, so a release-preflight result cannot claim end-user corpus breadth.

The goal for this slice is to make any real-world corpus claim falsifiable and provenance-clean. The goal is not to accumulate media files.

## Users And Stakeholders

- Arthur needs release language where "real-world corpus covered" means a concrete set of files decoded and matched the C reference.
- Codec maintainers need a small integration oracle that catches decoder regressions on non-reference encoded Ogg Opus without turning validation into asset curation.
- Future corpus contributors need a clear policy for what may be committed, what may be generated locally, and what stays local-only because rights are unclear.
- Downstream users care that the project tests more than synthetic vectors, but they also care that redistributed assets are legal and auditable.

## Proposed Small Testing Slice

### 1. Add A Tracked Corpus Manifest

Add a small manifest under `/home/md/language/ropus/tests/vectors/real_world/`, for example:

```text
tests/vectors/real_world/corpus_manifest.toml
```

The manifest should be the source of truth for gateable corpus entries. Each entry should declare:

- stable id,
- path under `tests/vectors/real_world/`,
- corpus class: `generated-non-reference`, `public-reference-baseline`, or `local-only-manual`,
- encoder/provenance,
- source fixture or URL,
- generation command or fetch command,
- license/redistribution note,
- expected container support: `supported-ogg-family0` or `unsupported-documented`,
- whether it is required for release corpus claims,
- maximum expected size,
- oracle class: initially `exact-pcm-parity-vs-c-reference`.

Do not make `corpus_diff` discover arbitrary files as the release gate. Arbitrary files can remain useful for local exploration, but the release gate needs a manifest so skipped WebM or family-3 files cannot silently satisfy a "corpus present" claim.

### 2. Prefer Generated Non-Reference Ogg Fixtures

First gateable corpus entries should be generated locally from existing checked-in WAV fixtures using FFmpeg's native Opus encoder:

```bash
ffmpeg -y -hide_banner -loglevel error \
  -i tests/vectors/48k_sine1k_loud.wav \
  -c:a opus -strict -2 -b:a 32k \
  tests/vectors/real_world/generated/ffmpeg-native-sine-32k.opus
```

Candidate tiny fixture set:

| Id | Source fixture | Encoder | Purpose |
|---|---|---|---|
| `ffmpeg-native-sine-32k` | `tests/vectors/48k_sine1k_loud.wav` | FFmpeg native `opus` | Small deterministic-ish smoke of non-reference encoder packets. |
| `ffmpeg-native-speech-24k` | `tests/vectors/speech_48k_mono.wav` | FFmpeg native `opus` | Speech-like mono content, closer to voice-note risk than sine/noise. |
| `ffmpeg-native-music-64k-stereo` | `tests/vectors/music_48k_stereo.wav` | FFmpeg native `opus` | Stereo/music path and larger packet variety. |

These should be provisioned into the ignored corpus directory, not automatically committed, unless implementation measurement shows the generated `.opus` files are tiny, stable enough, and Arthur explicitly approves adding binary exceptions.

The release gate should compare the generated files, not compare generated file bytes to a pinned hash. FFmpeg version drift can change the encoded stream; that is acceptable for a generated-corpus lane because the oracle is ropus-vs-C-reference decode parity on the same stream. The report should still record the FFmpeg version and generated file SHA-256 for auditability.

### 3. Keep Public Reference Baseline Separate

Do not rely on `tools/fetch_corpus.sh` as the first non-reference gate. Its current sources are useful as a baseline once checksums are pinned, but they are reference-encoded and should not satisfy a non-reference corpus claim.

For this slice, either:

- leave `tools/fetch_corpus.sh` as optional baseline documentation, or
- rename/report its lane as `public-reference-baseline` if the implementer pins one or two small checksums.

No `TBD` source may be admitted into a gate. `TBD` refusal should remain hard.

### 4. Add Minimum Coverage Semantics To `corpus_diff`

The harness should distinguish:

- candidate files found,
- files decoded and compared,
- files skipped because unsupported,
- files mismatched,
- files panicked.

For a release corpus claim, success should require at least one decoded-and-compared required manifest entry, and preferably all required supported entries. A directory containing only `.webm`, family-3, malformed, or otherwise skipped files must be non-green for the corpus lane.

Minimal implementation shape:

- keep the existing exploratory `corpus_diff <dir>` behavior available for local use,
- add manifest-aware mode or a wrapper stage that evaluates required entries,
- fail the gate when `required_supported_entries > 0` and `matched_required_entries == 0`,
- fail the gate when any required supported entry is missing, skipped, mismatched, or panicked.

The exact exit code can remain implementation-defined inside `full-test`; the externally documented `corpus_diff` exit contract should be updated if the binary behavior changes.

### 5. Add A `full-test` Corpus Lane

Add a bounded real-world corpus lane to `full-test`, not to default `cargo test`.

Recommended claim policy:

| CLI mode | Corpus behavior |
|---|---|
| Default, no `--release-preflight` | Report-only: show manifest/provisioning status if available; do not banner-block missing corpus. |
| `--quick --release-preflight` | No real-world corpus claim; quick stays core smoke. |
| Non-quick `--release-preflight` | Claim real-world corpus gate; missing/unprovisioned required corpus or `corpus_diff` failure banner-blocks. |

This mirrors the DNN/DRED distinction: quick preflight should stay cheap and honest, while non-quick release-preflight is allowed to claim broader release coverage.

Implementation can be a new `full-test/src/corpus.rs` stage or a smaller preflight-plus-command hook, but it should produce structured JSON and HTML report fields rather than only printing `corpus_diff` stdout.

### 6. Update Documentation And Oracle Matrix

Update:

- `/home/md/language/ropus/tests/vectors/real_world/README.md`
- `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`
- any `full-test --help` wording touched by the implementation

The matrix row should label the first lane as:

- strongest oracle: `exact-pcm-parity`,
- required assets: `fixed_reference`, `real_world_corpus`,
- asserts: manifest-listed Ogg family-0 files decode sample-for-sample against the C reference,
- does not assert: WebM, seek, pre-skip trimming, output gain application, FEC, PLC, multichannel/projection, true end-user player behavior.

## Fixture And Provenance Policy

### Allowed Without Further Signoff

- Tracked manifest/provenance metadata.
- Generated `.opus`/`.ogg` files in ignored `tests/vectors/real_world/` directories.
- Recipes using existing checked-in WAV fixtures as source.
- Public URLs with clear redistribution terms, real pinned SHA-256 digests, and small size.

### Requires Supervisor Signoff

- Committing any `.opus`, `.ogg`, or `.webm` binary under `tests/vectors/real_world/`.
- Adding any single corpus binary larger than 100 KiB or aggregate committed corpus binaries larger than 250 KiB.
- Adding files captured from messaging apps, streaming services, browsers, SFUs, or mobile devices.
- Treating WebM, seek, output gain, pre-skip trimming, or granule-position behavior as covered by the current harness.

### Disallowed For The Release Gate

- `TBD` checksums.
- Trust-on-first-use downloads.
- Files with unclear redistribution rights.
- Local personal captures that are not reproducible by the repository.
- Large music or speech files when a sub-second to few-second fixture gives the same decoder signal.

## Risks

- FFmpeg native Opus output can vary by FFmpeg version. That makes byte-level fixture pinning brittle, but it does not undermine the decode parity oracle because both decoders consume the same generated bitstream.
- A generated corpus is not the same thing as arbitrary files captured from end-user apps. The first slice should call this `generated non-reference encoder corpus`, not overclaim "all real-world behavior".
- Adding full container semantics is tempting, but current `corpus_diff` ignores pre-skip trimming, output gain, granule positions, seek, FEC, and PLC. This slice should not pretend those are tested.
- If the implementation commits generated `.opus` binaries for stability, it must override `.gitignore` deliberately and document why the legal/size tradeoff is acceptable.
- `full-test` may become slower if it builds and runs `ropus-harness` unconditionally. Keep corpus tiny and only claim it in non-quick release-preflight.
- A local corpus directory can contain extra files. The gate must evaluate required manifest entries deterministically and avoid being affected by unrelated local captures.

## Non-Goals

- Do not implement WebM/Matroska parsing.
- Do not implement seek, output gain, pre-skip trimming, granule-position, FEC, or PLC container/player semantics.
- Do not add large binary fixtures.
- Do not commit messaging-app, streaming-service, browser, SFU, or mobile captures without explicit signoff.
- Do not treat the reference-encoded public baseline as non-reference encoder coverage.
- Do not change codec behavior.
- Do not add CI provider configuration.
- Do not make default `cargo test` or default `full-test` fail because the real-world corpus is absent.

## Design Decisions Needing Supervisor Signoff

1. Non-quick `--release-preflight` should become the first profile that claims the real-world corpus gate; quick release-preflight should explicitly make no corpus claim.
2. The first required corpus should be generated from existing repo WAV fixtures with FFmpeg native `-c:a opus`, rather than fetched from external services or committed as third-party media.
3. Generated `.opus` files should stay ignored and be provisioned locally by default; committing tiny generated binaries requires separate signoff after size/determinism measurement.
4. A manifest should define the gateable corpus. Arbitrary local files can be explored by `corpus_diff`, but they should not satisfy release corpus claims.
5. Directories with zero decoded required entries, including directories containing only skipped WebM/family-3/malformed files, must be non-green for any corpus claim.
6. The current corpus gate should claim exact PCM parity against the C reference for Ogg family-0 packet decode only, not full end-user container playback semantics.
