# Brief: Mutation Testing Pilot

Date: 2026-04-08

## Context

Mutation testing does not exist in the project. No `cargo-mutants`, no mutation scripts, no reports. The testing report assessed this as "a gap, not a crisis" and recommended piloting it once branch coverage gaps are tightened.

Current coverage: 88.4% line, 72.9% branch. Another agent is working on closing branch coverage gaps. Mutation testing should follow that work — it answers a different question than coverage: "would the tests catch it if this code were wrong?"

With 572+ unit tests and 280 integration comparisons, the project has enough test mass to make mutation testing meaningful. The C reference oracle is an exceptionally strong mutation detector — any mutation that changes numerical output will be caught by differential comparison.

## Objective

Pilot `cargo-mutants` on a targeted subset of modules to assess test suite sensitivity, identify tests that pass despite mutations (weak tests), and establish a baseline mutation score.

## Scope

### 1. Tool setup

Install and configure `cargo-mutants`:

```bash
cargo install cargo-mutants
```

Configure via `.cargo/mutants.toml` or command-line flags:
- Timeout multiplier: 3x (codec tests are fast, but mutations might hit slow paths)
- Parallel jobs: match CPU count
- Output format: JSON for machine parsing, human-readable summary

### 2. Phased rollout

Do NOT run mutation testing across all 36k lines at once. That would take hours and produce overwhelming output. Instead, target modules in priority order:

**Phase 1 — Core math and helpers (highest signal, fast)**
- `src/types.rs` — fixed-point arithmetic. Mutations here break everything. If tests don't catch them, the test suite is fundamentally weak.
- `src/celt/math_ops.rs` — math utilities.
- `src/celt/range_coder.rs` / `src/celt/ec_ctx.rs` — entropy coding. Bit-level precision required.

Expected: very high mutation kill rate (>95%) due to bit-exact comparison tests. If not, something is seriously wrong.

**Phase 2 — Transform and band processing (medium signal)**
- `src/celt/fft.rs` — FFT butterflies. Now contains `unsafe` unchecked indexing.
- `src/celt/mdct.rs` — MDCT. Same unsafe concern.
- `src/celt/bands.rs` — band processing.
- `src/celt/quant_bands.rs` — energy quantization.

Expected: high kill rate from integration tests. Surviving mutants indicate dead code or weak test input variety.

**Phase 3 — Top-level control flow (lower signal, longer runtime)**
- `src/opus/encoder.rs`
- `src/opus/decoder.rs`
- `src/silk/encoder.rs`
- `src/silk/decoder.rs`

These have the weakest branch coverage (67-78%) and are where surviving mutants are most likely. Run this phase only after branch coverage improvements are in.

### 3. Analysis and triage

For each phase, produce a report:
- Total mutants generated
- Killed (test failed) — good
- Survived (test passed despite mutation) — investigate
- Timed out — usually acceptable, treat as killed
- Skipped — note why

For surviving mutants, triage into:
1. **Weak test** — the test should catch this but doesn't. Write a targeted test.
2. **Dead code** — the mutated code is never exercised. Consider removing or adding coverage.
3. **Equivalent mutant** — the mutation produces identical behavior (e.g., changing `x + 0` to `x - 0`). Ignore.
4. **Don't-care path** — error handling or edge case that doesn't affect correctness for valid inputs. Document and skip.

### 4. Baseline and targets

After Phase 1 and 2, establish a baseline mutation score. Suggested targets:
- Phase 1 (math/helpers): >95% kill rate
- Phase 2 (transforms): >90% kill rate
- Phase 3 (control flow): >80% kill rate (aspirational, depends on branch coverage work)

These targets are not gates — they're diagnostic. A low score points to where tests need strengthening, not where the code is broken.

### 5. Integration with workflow

After the pilot, decide whether mutation testing becomes regular:
- **If useful**: Add `tools/mutant_check.sh` that runs Phase 1+2 as a quick check (~10 minutes). Run before major releases or after significant refactors.
- **If noisy**: Restrict to Phase 1 only (math/helpers) as a smoke test.
- **If too slow**: Use `cargo-mutants --in-diff` to only mutate changed code.

## Acceptance criteria

- `cargo-mutants` installed and running on Phase 1 modules
- Phase 1 mutation report produced with kill rate
- Surviving mutants triaged (weak test / dead code / equivalent / don't-care)
- At least 3 new tests written in response to surviving non-equivalent mutants
- Baseline mutation score documented
- Decision recorded on whether to continue to Phase 2

## Prerequisites

- Branch coverage improvements should be substantially complete before Phase 3
- Phase 1 can start immediately — it targets well-covered helper modules

## Out of scope

- Full-codebase mutation runs (too expensive for a pilot)
- Custom mutant operators (standard operators are sufficient)
- CI integration (premature — assess value first)
- Mutation testing of the test harness itself (`tests/harness/main.rs`)
