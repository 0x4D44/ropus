# Fuzz Oracle Debt Reporting HLD

Date: 2026.05.06

Scope: Phase 2.2 of `/home/md/language/ropus/wrk_docs/2026.05.05 - PLN - testing improvement plan.md`.

## Problem Statement

The fuzz decode oracle has intentionally weakened classes for attacker-controlled packets where PCM parity or SNR parity is not a trustworthy correctness signal. The current code in `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs` already classifies packets as:

- `CeltCodedComparable`
- `SilkHybridCodedComparable`
- `SampleCountOnly`
- `RecoveryOrDtxOnly`
- `ErrorAgreementOnly`

That is a good implementation base, but it is not yet a good reporting contract. A green fuzz run can still include many inputs whose PCM values were deliberately not asserted, and today that distribution is not visible. The debt is interpretability: "no fuzz findings" does not say how much of the explored corpus used a strong oracle versus a weakened oracle.

This HLD does not redesign fuzz input schemas, does not add legal frame-duration dimensions, and does not try to fix underlying PLC/DTX/Hybrid high-band parity drift.

## Stakeholder Value

The main stakeholders are maintainers deciding whether fuzz results support release confidence.

Value delivered:

- Stable human-facing names for every decode oracle class.
- A clear statement of the exact correctness property each weakened class no longer asserts.
- Cheap, unit-testable classification/reporting behavior that does not require a fuzz campaign.
- Oracle matrix entries that distinguish different kinds of weakened coverage instead of collapsing them into one vague bucket.

The practical outcome: a future report can say "fuzz_decode found no crashes; committed decode fixtures classify as 40% SNR-gated, 30% sample-count-only, 20% recovery-or-dtx-only, 10% error-agreement-only" rather than implying uniform PCM parity.

## Current Behavior

`/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs` owns the shared decode classification and SNR helpers.

`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_decode.rs` uses `classify_decode_packet(packet, sample_rate)` to decide whether CELT exact PCM or SILK/Hybrid SNR checks are meaningful. For `SampleCountOnly`, `RecoveryOrDtxOnly`, and `ErrorAgreementOnly`, it still checks decode result symmetry and sample count where both decoders succeed, but skips PCM value assertions.

`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_multistream.rs` uses `classify_multistream_decode_packet(&input.payload, streams, sample_rate)` similarly.

`/home/md/language/ropus/tests/fuzz/fuzz_targets/fuzz_roundtrip.rs` uses `snr_oracle_applicable_for_packet(...)` for encoder-produced packets. That path is not the primary debt target for this slice because roundtrip packets are well-formed by construction and the current weakening mostly comes from low-energy/recovery applicability rather than attacker-controlled decode classification.

`/home/md/language/ropus/tests/fuzz/tests/oracle_tests.rs` already proves several classification boundaries, including recovery fixtures and high-rate Hybrid sample-count-only behavior.

`/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md` has a fuzz row for weakened classes, but it still reports the strongest oracle as `sample-count-only` and does not separately describe `recovery-or-dtx-only` or `error-agreement-only`.

## Oracle Taxonomy

Recommended stable public names:

| Rust variant | Stable name | Debt? | Runtime assertion level |
|---|---:|---:|---|
| `CeltCodedComparable` | `celt-coded-comparable` | No | Exact PCM/float-bit parity where target path supports it. |
| `SilkHybridCodedComparable` | `silk-hybrid-snr-comparable` | No | SNR-gated parity when the reference energy floor is met. |
| `SampleCountOnly` | `sample-count-only` | Yes | Decode result symmetry and sample-count/shape parity only. |
| `RecoveryOrDtxOnly` | `recovery-or-dtx-only` | Yes | Decode result symmetry and sample-count/shape parity only. |
| `ErrorAgreementOnly` | `error-agreement-only` | Yes | Decode success/error symmetry only; sample count only when both sides decode. |

Exact properties no longer asserted for weakened classes:

| Stable name | Correctness property not asserted |
|---|---|
| `sample-count-only` | PCM sample values, exact PCM parity, and SNR threshold are not asserted for attacker-controlled high-band Hybrid decode inputs. This specifically avoids treating unstable arbitrary Hybrid high-band synthesis as a standalone PCM oracle. |
| `recovery-or-dtx-only` | PCM sample values, exact PCM parity, and SNR threshold are not asserted when at least one SILK/Hybrid sub-frame routes through PLC/DTX-style recovery. Recovery output can diverge while decode shape still agrees. |
| `error-agreement-only` | PCM sample values, exact PCM parity, SNR threshold, and in some cases sample-count expectations are not asserted because packet structure is not usable for PCM comparison. The meaningful property is that Rust and C agree on success versus failure, and agree on shape if both succeed. |

## Considered Options

### Option A: Always-on libFuzzer counters

Add per-class counters to fuzz target hot paths and print a distribution at campaign exit.

Pros:

- Distribution comes from actual campaign traffic.
- No separate operator command.

Cons:

- libFuzzer campaign output is already noisy.
- Exit-time reporting is awkward and target-specific.
- Atomic or synchronized counters add cost and complexity to the hot path.
- Per-input classification counts can mislead because coverage-guided fuzzers mutate and re-run inputs in ways that are not a stable corpus sample.

Decision: reject for Phase 2.2.

### Option B: Test-only metadata and summary helpers

Keep classification metadata and count aggregation in `oracle.rs`, then unit-test it through `oracle_tests.rs`.

Pros:

- Cheap and deterministic.
- Keeps the source of truth near the classifier.
- Does not affect fuzz campaign speed or output.
- Easy to test formatting and class boundaries.

Cons:

- A normal `cargo test` does not produce an operator-facing report unless a specific test is run with `--nocapture`.
- It summarizes bounded fixtures, not a live fuzz campaign.

Decision: accept as the core design.

### Option C: Dedicated offline report command

Add a small operator command that summarizes committed seed/crash/known-failure inputs by oracle class.

Pros:

- Clear user-facing output.
- Avoids abusing test output as a report channel.

Cons:

- A new Rust binary under `tests/fuzz` risks confusion with cargo-fuzz target discovery.
- A shell-only parser would duplicate fuzz input decoding and become fragile.
- More moving parts than Phase 2.2 needs.

Decision: defer as optional. If implementation wants a command, prefer a tiny shell wrapper in `/home/md/language/ropus/tools/fuzz_run.sh` or `/home/md/language/ropus/tools/fuzz_oracle_report.sh` that invokes the ignored/reporting test, not a new parser.

### Option D: Documentation-only matrix update

Only update `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`.

Pros:

- Very small.
- No risk to fuzz runtime behavior.

Cons:

- Drifts immediately from the classifier.
- Does not provide class distribution or testable class metadata.

Decision: reject as insufficient.

## Recommended Design

Implement a narrow combination of source metadata, testable summaries, and documentation.

### 1. Make `DecodeOracleClass` self-describing

In `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs`, add methods or a small metadata table:

- `stable_name(self) -> &'static str`
- `is_weakened(self) -> bool`
- `asserted_property(self) -> &'static str`
- `unasserted_property(self) -> Option<&'static str>`
- optional `matrix_class(self) -> &'static str` if the matrix wants to map classes to `exact-pcm-parity`, `snr-gated-parity`, or the weakened names.

Keep strings short, stable, and documentation-oriented. They become report identifiers, so changing them later should be treated like changing a test contract.

Do not rename the Rust enum variants unless the implementation agent finds strong local value. The stable external names can be provided by methods without touching match sites.

### 2. Add a small summary type

In `oracle.rs`, add a pure summary helper:

```rust
pub struct DecodeOracleClassCounts {
    pub celt_coded_comparable: usize,
    pub silk_hybrid_snr_comparable: usize,
    pub sample_count_only: usize,
    pub recovery_or_dtx_only: usize,
    pub error_agreement_only: usize,
}
```

Recommended methods:

- `new() -> Self`
- `record(&mut self, class: DecodeOracleClass)`
- `total(&self) -> usize`
- `weakened_total(&self) -> usize`
- `strong_total(&self) -> usize`
- `rows(&self) -> impl Iterator<Item = DecodeOracleClassCountRow>` or a simple fixed array helper.
- `format_report(&self, label: &str) -> String`

Prefer fixed fields or a fixed array over `HashMap` so output ordering is stable and unit tests are simple.

Add convenience constructors only where they do not duplicate target parsing:

- `summarize_decode_packets<'a>(packets: impl IntoIterator<Item = (&'a [u8], i32)>) -> DecodeOracleClassCounts`
- optional `summarize_multistream_decode_packets<'a>(items: impl IntoIterator<Item = (&'a [u8], i32, i32)>) -> DecodeOracleClassCounts`

Keep target input schema parsing outside `oracle.rs` unless a parser is already shared. The oracle module should classify packets, not understand every fuzz target's arbitrary input encoding.

### 3. Add tests for metadata and reporting

In `/home/md/language/ropus/tests/fuzz/tests/oracle_tests.rs`, add focused tests:

- Every `DecodeOracleClass` has a non-empty unique `stable_name`.
- The three weakened classes return `is_weakened() == true`.
- Strong classes have no unasserted debt property, or explicitly document why they are not debt.
- Weakened classes have non-empty `unasserted_property()`.
- `DecodeOracleClassCounts::record` and totals are correct.
- `format_report` has deterministic ordering and includes zero-count classes, so absence is visible.

Also keep/extend existing boundary tests:

- High-rate Hybrid packet at 48 kHz classifies as `sample-count-only`.
- The same Hybrid packet at 16 kHz classifies as `silk-hybrid-snr-comparable`.
- The known recovery fixture classifies as `recovery-or-dtx-only`.
- Empty or malformed packet classifies as `error-agreement-only`.
- CELT TOC packet classifies as `celt-coded-comparable`.

### 4. Add an opt-in bounded distribution report

Add a report path in `oracle_tests.rs`, preferably an ignored test named clearly, for example:

```rust
#[test]
#[ignore = "operator report; run with --ignored --nocapture"]
fn decode_oracle_class_distribution_report() { ... }
```

It should summarize only bounded, committed inputs, not run a fuzz campaign. Suggested initial input sets:

- Synthetic packets used by the classification boundary tests.
- Known failure fixtures under `/home/md/language/ropus/tests/fuzz/known_failures/` that already document oracle downgrades.
- Optionally committed `fuzz_decode` seed/crash `.bin` files where the first two bytes are existing `fuzz_decode` config and the remaining bytes are a single Opus packet.

Do not parse all target formats in this slice. `fuzz_multistream` arbitrary inputs are more structured; include only a few explicit synthetic multistream samples or known fixtures if they can be decoded through existing test helper code without copying target internals.

If Arthur wants a command-like surface, add a script or `tools/fuzz_run.sh --oracle-report` that runs:

```bash
timeout 120 cargo test --manifest-path tests/fuzz/Cargo.toml --test oracle_tests -- --ignored --nocapture decode_oracle_class_distribution_report
```

That wrapper is optional. The implementation should not block on it if the test report is clear.

### 5. Update the oracle matrix

Update `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md`:

- Split the current "Shared fuzz oracle weakened classes" row into at least:
  - `sample-count-only`
  - `recovery-or-dtx-only`
  - `error-agreement-only`
- Link the owner/follow-up for those rows to this HLD.
- In "Does not assert", use the same property language as `unasserted_property()`.
- Preserve the existing exact/SNR rows for strong classes.

Documentation must not be the only source of truth; it should mirror the metadata in `oracle.rs`.

## Runtime Versus Offline Behavior

Fuzz target runtime behavior:

- Keep current assertion behavior.
- Keep per-input classification as needed for oracle choice.
- Do not print per input.
- Do not add always-on global counters.
- Do not make campaign success depend on class distribution thresholds.

Offline/test behavior:

- Unit tests validate classification metadata and count formatting.
- An ignored/reporting test can print bounded distributions for committed examples.
- Optional shell integration can expose that report without requiring maintainers to remember the cargo-test incantation.

This split keeps fuzzing fast and quiet while making oracle debt visible when interpreting results.

## Risks And Tradeoffs

- Reported distribution is not a coverage metric. It summarizes bounded examples or committed corpora, not all inputs explored during a campaign.
- Stable names become contracts. Changing them later will churn docs and any scripts consuming reports.
- Keeping reporting in tests is slightly awkward for operators, but avoids adding a new binary that could confuse cargo-fuzz target discovery.
- Counting committed crash files may overrepresent old bug classes. That is acceptable if the report labels the input source clearly.
- A per-target all-corpus report is tempting but likely to duplicate target-specific arbitrary parsing. That should wait until there is a real consumer.

## Rollback Plan

Rollback is straightforward:

- Remove metadata/count helper additions from `/home/md/language/ropus/tests/fuzz/fuzz_targets/oracle.rs`.
- Remove added tests/reporting from `/home/md/language/ropus/tests/fuzz/tests/oracle_tests.rs`.
- Revert the oracle matrix rows to the previous fuzz weakened-class entry.
- Remove any optional shell wrapper flag if added.

Because the design does not change codec behavior or fuzz input schemas, rollback should not require corpus migration or crash artifact changes.

## Key Design Decisions

- Source of truth belongs in `oracle.rs` because that is where classification already lives.
- Reporting should be offline/test behavior, not always-on fuzz runtime behavior.
- Class distribution should be bounded and cheap, used to interpret fuzz results rather than gate them.
- The implementation should split the three weakened classes explicitly: `sample-count-only`, `recovery-or-dtx-only`, and `error-agreement-only`.
- The oracle matrix should mirror source metadata and link back to this HLD.
- Phase 3.3 frame-duration expansion remains out of scope.
