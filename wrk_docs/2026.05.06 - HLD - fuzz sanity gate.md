# HLD: Fuzz Sanity Gate

Date: 2026.05.06

Status: Design only. No source implementation in this slice.

## Problem Statement

Phase 1.1 made `/home/md/language/ropus/tests/fuzz/Cargo.toml` the source of
truth for fuzz target discovery, but the standard validation gate still does
not prove that the discovered fuzz targets build. Broken fuzz target code can
therefore survive ordinary validation and fail only when Arthur or another
agent starts a long fuzz campaign.

The next slice should add a bounded fuzz sanity gate: build the
manifest-discovered fuzz targets and replay committed crash-corpus files when
they exist. It must not start fuzz campaigns, mutate corpora, or make fresh
default validation hostile.

## Stakeholder Value

Arthur needs release-preflight output that catches stale or broken fuzz
infrastructure before overnight fuzz work is scheduled. Other agents need a
small, deterministic command they can run after touching fuzz harness code,
without accidentally launching a campaign.

The value is infrastructure confidence, not new bug discovery. A green fuzz
sanity gate should mean:

- every active fuzz target declared in
  `/home/md/language/ropus/tests/fuzz/Cargo.toml` compiles under cargo-fuzz;
- committed crash regressions, such as the current files under
  `/home/md/language/ropus/tests/fuzz/crashes/fuzz_decode/`, no longer crash;
- empty crash directories are treated as a clean no-op;
- missing fuzz tooling is reported with policy-dependent severity.

## Current Behavior

- `/home/md/language/ropus/tools/fuzz_targets.sh` parses `[[bin]]` target names
  from `/home/md/language/ropus/tests/fuzz/Cargo.toml`.
- `/home/md/language/ropus/tools/fuzz_run.sh --list` uses discovered targets
  and reports corpus/crash counts.
- `/home/md/language/ropus/tools/fuzz_run.sh --check-crashes` already builds
  selected targets before replaying `tests/fuzz/crashes/<target>/*.bin`.
- `/home/md/language/ropus/tools/fuzz_run.sh` has no build-only mode; without
  `--check-crashes` it proceeds into fuzzing after the build step.
- `/home/md/language/ropus/tools/fuzz_cycle.sh` is campaign-oriented and should
  remain outside release preflight.
- `/home/md/language/ropus/full-test/src/main.rs` owns the validation stages and
  report generation, but has no fuzz stage.
- `/home/md/language/ropus/full-test/src/preflight.rs` reports required release
  assets, but does not enumerate or build fuzz targets.

## Considered Options

1. Put the whole gate directly in `full-test` Rust.
   - Rejected as the only implementation path. It would duplicate the
     cargo-fuzz command rules, Windows ASan setup, target selection, and crash
     replay conventions already present in shell tooling.

2. Extend `/home/md/language/ropus/tools/fuzz_run.sh` only.
   - Useful but incomplete. It gives humans a good command, but release
     preflight still would not show the result in the HTML/JSON report or
     banner classification.

3. Add a dedicated shell sanity mode and invoke it from `full-test`.
   - Recommended. Shell owns the exact bounded cargo-fuzz work; `full-test`
     owns release policy, report shape, and banner effect.

4. Reuse `/home/md/language/ropus/tools/fuzz_cycle.sh`.
   - Rejected. That script is explicitly campaign/cadence-oriented and carries
     duration, logs, timeout wrapping, and "new crash" semantics that are wrong
     for a bounded release sanity gate.

## Recommended Design

Implement the gate in both places with a narrow boundary.

Shell tooling should expose a bounded, no-campaign command:

```text
timeout 300s bash tools/fuzz_run.sh --sanity
```

`--sanity` should:

1. Discover targets from `/home/md/language/ropus/tests/fuzz/Cargo.toml` through
   `/home/md/language/ropus/tools/fuzz_targets.sh`.
2. Validate explicit `--target` names against the discovered target list.
3. Require cargo-fuzz and nightly only when the mode actually needs to invoke
   cargo-fuzz.
4. Build selected targets with:

   ```text
   cargo +nightly fuzz build --fuzz-dir tests/fuzz <target>
   ```

5. Replay committed crash files for selected targets:

   ```text
   cargo +nightly fuzz run --fuzz-dir tests/fuzz <target> -- -runs=0 <crash-file>
   ```

6. Exit 0 if all selected targets build and all committed crash files replay
   without crashing.
7. Exit nonzero on target build failure, replay crash, malformed manifest,
   undeclared target, missing cargo-fuzz, or missing nightly.
8. Print a compact machine-readable summary line per target, for example:

   ```text
   fuzz_decode build=pass crashes=4 replay=pass
   fuzz_encode build=pass crashes=0 replay=skip
   ```

The existing `--check-crashes` behavior can be folded into `--sanity` or kept
as an alias. The important change is that the bounded gate has a name that
cannot be mistaken for a fuzz campaign.

`full-test` should add a small fuzz stage after Stage 2 and before stages that
depend on successful workspace builds. It should call the shell command rather
than reimplement cargo-fuzz. A new module such as
`/home/md/language/ropus/full-test/src/fuzz.rs` should own:

- command construction;
- timeout selection;
- stdout/stderr capture;
- status classification;
- summary structs for report/JSON rendering.

### Full-Test Policy

Default `full-test` without `--release-preflight`:

- Do not invoke cargo-fuzz.
- Render fuzz sanity as `not_requested` or omit the row from the phase summary.
- Missing nightly or cargo-fuzz is not a warning and not a failure.

`full-test --release-preflight`:

- Run the full bounded fuzz sanity command for all manifest-discovered targets.
- Missing `cargo-fuzz` is a hard FAIL.
- Missing `+nightly` is a hard FAIL.
- Fuzz target build failure is a hard FAIL.
- Committed crash replay failure is a hard FAIL.
- No committed crash files for a target is PASS with `replay=skip`, not WARN.
- Stale crash directories for targets not declared in the manifest should be
  WARN/report-only unless they contain `*.bin`; undeclared `*.bin` crash files
  should be FAIL because they cannot be replayed by the active target set.

`full-test --quick --release-preflight`:

- Run a very cheap fuzz sanity check only.
- Do not invoke `cargo +nightly fuzz build`.
- Do not replay crash corpora.
- Require only manifest discovery and crash-corpus inventory:
  - malformed or empty manifest is FAIL;
  - committed `*.bin` crash files under undeclared target directories is FAIL;
  - declared targets with no crash files are PASS;
  - missing cargo-fuzz or nightly is `not_checked`/WARN in the report, not a
    hard failure.

This quick policy keeps the smoke profile useful for release-preflight plumbing
without importing cargo-fuzz toolchain cost into the already-bounded quick path.
The non-quick release-preflight remains the place where fuzz target builds and
crash replay are enforced.

## TDD Plan

Write unit tests around pure policy and command construction before shelling out
to cargo-fuzz.

1. Shell syntax and discovery behavior:
   - `bash -n tools/fuzz_run.sh`
   - `bash tools/fuzz_run.sh --list`
   - a helper-level test or documented probe that `--target definitely_missing`
     fails before build.

2. Shell sanity mode shape:
   - Add a mode parser test if shell tests exist; otherwise validate through
     `bash -n` plus a dry command path that can run without cargo-fuzz, such as
     `--sanity --inventory-only` if the implementation chooses to expose that
     internal mode.
   - Assert `--sanity` never reaches the main fuzzing loop.

3. Rust policy tests in `full-test/src/fuzz.rs`:
   - default options produce `not_requested`;
   - `--release-preflight` selects full bounded build/replay;
   - `--quick --release-preflight` selects inventory-only;
   - missing cargo-fuzz is FAIL for full release-preflight and WARN/not-checked
     for quick release-preflight;
   - missing nightly follows the same split;
   - empty crash directories are PASS/skip;
   - undeclared target directories containing only `.gitkeep` are ignored or
     WARN;
   - undeclared target directories containing `*.bin` are FAIL.

4. Banner tests:
   - fuzz build failure causes FAIL under full release-preflight;
   - fuzz replay failure causes FAIL under full release-preflight;
   - quick release-preflight tool absence does not cause FAIL;
   - default mode remains unaffected.

5. Report tests:
   - JSON includes fuzz sanity status and selected policy mode.
   - HTML escapes command output and renders target rows without affecting
     existing sections.

## Validation Plan

Fast validation after implementation:

```text
timeout 10s bash -n tools/fuzz_run.sh
timeout 10s bash -n tools/fuzz_targets.sh
timeout 10s bash tools/fuzz_run.sh --list
timeout 120s cargo test -p full-test fuzz
timeout 120s cargo test -p full-test banner
```

Bounded fuzz sanity validation when nightly and cargo-fuzz are installed:

```text
timeout 300s bash tools/fuzz_run.sh --sanity --target fuzz_decode
timeout 300s bash tools/fuzz_run.sh --sanity
```

User-facing runner validation:

```text
timeout 300s cargo run --release -p full-test -- --quick --release-preflight --skip-coverage
timeout 600s cargo run --release -p full-test -- --release-preflight --skip-coverage
```

No validation command should use `cargo fuzz run` with a time budget or corpus
directory in a way that starts exploration. Crash replay must use `-runs=0` and
explicit committed crash files only.

## Risks And Tradeoffs

- Building all fuzz targets under cargo-fuzz may still take noticeable time.
  That is why the full build/replay gate belongs to non-quick
  `--release-preflight`, while quick release-preflight does inventory only.

- Replaying crash corpora proves only that known committed regressions are
  fixed. It does not search for new issues, and the report should not imply
  that fuzzing has been performed.

- Calling shell from Rust is less pure than a Rust-only implementation, but it
  avoids duplicating cargo-fuzz and Windows ASan behavior. The shell command is
  the right operational boundary for this repository today.

- Treating missing cargo-fuzz/nightly as FAIL under full release-preflight can
  annoy fresh clones. That is intentional for release claims and avoided in
  default and quick smoke modes.

- Undeclared crash files are policy-sensitive. Failing on undeclared `*.bin`
  files is stricter than today's scripts, but it prevents a stale crash corpus
  from silently falling out of replay coverage after a target rename.

## Rollback Plan

The implementation should be easy to back out because it is additive:

1. Remove the `full-test` fuzz stage from `/home/md/language/ropus/full-test/src/main.rs`.
2. Remove fuzz status from banner, report, and HTML rendering.
3. Keep or remove `tools/fuzz_run.sh --sanity` independently. If the report
   integration causes trouble, the shell sanity command can still remain useful
   as a manual bounded check.
4. If the shell change itself causes trouble, leave the existing `--check-crashes`
   path intact and revert only the `--sanity` alias/factoring.

No corpus files, source codec behavior, or long-running campaign scripts need to
change for this rollback.

## Key Policy Decisions

- The bounded fuzz sanity gate belongs in both shell tooling and `full-test`:
  shell executes cargo-fuzz; `full-test` reports and enforces release policy.
- Default `full-test` does not run fuzz sanity and does not care whether
  cargo-fuzz/nightly are installed.
- Non-quick `--release-preflight` runs all-target fuzz build plus committed
  crash replay and fails on missing cargo-fuzz, missing nightly, build failure,
  replay crash, or undeclared committed crash `*.bin` files.
- `--quick --release-preflight` runs inventory-only fuzz sanity: manifest and
  crash-corpus shape checks, no cargo-fuzz invocation, no fuzz target build.
- Empty crash corpora are a clean pass with replay skipped.
- Long fuzz campaigns stay outside `full-test` and outside this phase.
