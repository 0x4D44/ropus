# Campaign 8 — Bug K (pitch_out clamp uses wrong upper bound)

## Config
| Field | Value |
|-------|-------|
| sample_rate | 12000 Hz |
| channels | 2 (stereo) |
| application | 2048 (OPUS_APPLICATION_VOIP) |
| bitrate | 71535 bps |
| complexity | 2 |
| num_frames | 5 |
| TOC | 0x2F (config=5 SILK-WB 20 ms, stereo=1, code=3) |

## Symptoms
- Repro: `fuzz_crashes/crash_b7c3ad5aeeffb545.bin` (5910 bytes) —
  preserved as `fuzz_crashes/bugK_repro_b7c3ad5aeeffb545.bin`.
- `repro_bug_k`: frame 2/5 bytes diverge between Rust and C, both 179 bytes.
- First byte diff at offset 1 (frame-count byte): Rust `0x01` vs C `0x41` —
  different padding flag, because SILK payloads differ in length by 1 byte.
- `mdopus-interframe` extended-state dump on frame 2 ch0:
  - `prev_lag`: Rust=218, C=216.
  - `nsq_lag_prev`: Rust=218, C=216.
  - 31 downstream NSQ state fields diverge (all consequences of the
    frame-2 pitch_l[3] differing).
  - ec_prev_lag_index (191), NLSF Q15 history, VAD, stereo pred —
    all match. So the bitstream-level cascade starts at `pitch_l[3]`
    and propagates into NSQ → different pulse choices.

## Root cause

`src/silk/encoder.rs::silk_pitch_analysis_core`, line 6592 (pre-fix):

```rust
pitch_out[k] = (lag_new + cb_val).max(min_lag).min(PE_MAX_LAG as i32);
```

`PE_MAX_LAG = PE_MAX_LAG_MS * PE_MAX_FS_KHZ = 18 * 16 = 288`.

C reference `silk/fixed/pitch_analysis_core_FIX.c:540`:
```c
pitch_out[ k ] = silk_LIMIT( pitch_out[ k ], min_lag, PE_MAX_LAG_MS * Fs_kHz );
```

C clamps per sample-rate: `18 * Fs_kHz`. At 12 kHz that's `216`, at
16 kHz `288`.

The Rust port used the global `PE_MAX_LAG` (288, the max over all
supported sample rates) as the upper bound, so at 12 kHz / 16 kHz the
Rust clamp is looser than C's. When the stage-3 search picks
`lag_new=215, CBimax=15` and `silk_CB_lags_stage3[3][15]=3`, C clamps
to 216 while Rust leaves 218. That 2-sample drift enters NSQ via the
`pitch_l[]` array → different subframe LTP residual → different quant
decisions → different range-coded bytes on the same frame.

The bug needs all of:
- 12 kHz or 16 kHz encode (8 kHz uses a different branch; 24 / 48 kHz
  get down-resampled to 16 kHz internally but pitch analysis uses
  `fs_khz` = 12 or 16 only if the mode is pure SILK; pure CELT
  bypasses this function).
- Loud / saturated input where the stage-3 search picks a lag very
  close to `PE_MAX_LAG_MS * Fs_kHz` (frames 0-1 matched because lags
  were ~139, well inside the clamp).

## Verification

- Captured paired `STAGE3` / `PITCH_OUT` traces from both Rust and C:
  both encoders produce identical `lag_new=215, CBimax=15,
  CCmax=792, energy_target=144383356, contour_bias_Q15=7`. Only the
  final clamp diverges.
- Diff shows Rust `pitch_l=[212, 214, 216, 218]`, C `pitch_out=[212,
  214, 216, 216]`.
- After fix: `target/release/repro_bug_k.exe
  fuzz_crashes/crash_b7c3ad5aeeffb545.bin` → "All 5 frames match --
  NOT REPRODUCED!".
- All existing bug-D/E/F/G/H/I/J reproducers still pass (9/10/10/7/7/10/5/7
  frames match respectively).
- `cargo test --release --lib` → 1778 passed / 0 failed / 1 ignored.
- Rebuilt fuzz binary with `cargo +nightly fuzz build --fuzz-dir
  tests/fuzz fuzz_encode_multiframe` and ran against
  `fuzz_crashes/crash_b7c3ad5aeeffb545.bin` → exit 0 (no panic).

## Fix location

`src/silk/encoder.rs:6585-6594` — replace `PE_MAX_LAG as i32` upper
bound with `(PE_MAX_LAG_MS as i32) * fs_khz`. One-liner change plus a
comment explaining the sample-rate dependence.
