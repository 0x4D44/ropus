# Campaign 8 — Bug E (mid-frame SILK numerics divergence)

## Status: recorded from log evidence only. No reproducer captured yet.

## Discovery
Target: `fuzz_encode_multiframe`
Time-to-crash: exec 42078 of campaign-8's first `fuzz_encode_multiframe` run
(post-Bug-D fix). Crash fired after the Bug D fix was in place — so this is a
distinct bug, not a regression.

The previous fuzzer process panicked without persisting the input because the
`FUZZ_PANIC_CAPTURE_DIR` hook was not installed in `fuzz_encode_multiframe.rs`
at the time of the crash. The hook has now been re-applied, so the next
occurrence will save a byte-exact reproducer under `fuzz_crashes/`.

## Crash signature (verbatim from fuzz.log, exec 42078)

```
Frame 8/10 byte mismatch: sr=8000, ch=1, app=2048, br=67040, cx=5, len=168
  left (Rust):  [..., 215, 135, 176, 192, 0, 0, ...]
  right (C):    [..., 215, 167,  26, 192, 0, 0, ...]
```

- First 10 bytes of frame 8 match between Rust and C.
- Bytes 10 and 11 diverge.
- Byte 12 matches again.
- Remainder (156 bytes) is trailing zeros on both sides — identical padding.
- Both packets are 168 bytes (CBR-padded).
- Both TOCs are 0x0B = code=3 multi-frame with a frame-count byte.

## Why this is distinct from Bug D

| Aspect | Bug D | Bug E |
|--------|-------|-------|
| Bitrate | 6001 bps (budget-bust regime) | 67040 bps (comfortable headroom) |
| First divergent byte | TOC (byte 0) | Payload byte 10 (SILK range-coded stream) |
| Mode inferred | Same TOC for both (after C's rewrite), but Rust emitted 0x08 instead of C's 0x0B | Same TOC (0x0B) — both encoders chose the same framing |
| Frame at first divergence | 1 | 8 |
| Root cause | Opus-layer budget-bust handling missing in Rust encoder | Unknown — deep SILK numerics divergence after 7 matching frames |
| Fix status | Fixed in master by the reinstated 3-hunk edit to `src/opus/encoder.rs` | Waiting for reproducer |

Bug D was an Opus-layer framing bug (missing `if (ec_tell(&enc) > ...)` handler).
Bug E is a **real SILK numerics divergence**: both encoders pick the same frame
structure, same packet length, and the same leading 10 bytes of range-coded
payload — then diverge on the next two bytes only. The range coder at byte 10
emits `(135, 176)` in Rust vs `(167, 26)` in C. The downstream bytes re-sync
because the same number of bits were consumed before hitting padding.

## Hypothesis sketch

Seven frames of identical output before divergence means the encoder's
inter-frame state is being updated consistently for the first 7 frames, then
one narrow corner on frame 8 causes a symbol choice to flip. Bitrate = 67040 bps
at 8 kHz mono means roughly 1676 bits per 20 ms frame — plenty of rate for SILK
to use all its predictors without saturating gain/LTP indices the way Bug D
stressed them.

Candidate state vectors (descending likelihood):

1. **LTP predictor history** (`prev_lag_index`, `ec_prev_lag_index`). With a
   voiced tonal frame on input #8 following 7 voiced frames at medium rate, the
   LTP quantizer's delta-coded lag index may hit a boundary where the low-bit
   decision reorders versus C (e.g. integer rounding vs floor on the search
   residual). This is classically the culprit when the first diverging bytes
   are deep in the SILK payload and state up to that point agrees.

2. **Gain predictor history** (`last_gain_index`, `prev_gain_q16`). Less likely
   at 67040 bps because the gain delta code rarely saturates here, but a drift
   in `prev_gain_q16` that only shows up after 7 accumulated updates cannot be
   ruled out without a repro.

3. **NLSF delta coding memory** (`prev_NLSFq_Q15`). The NLSF codebook index
   delta coding depends on the previous-frame quantized NLSFs; a Q15 rounding
   difference accumulated over 7 frames could flip the codebook entry on frame
   8. This would affect more than 2 bytes though — likely not the signature.

4. **Pitch analysis accumulator or tonality estimator state** (e.g. `prev_HPLP`,
   `tonality_slope`, `decision_delay` scratch state). These feed the
   voiced/unvoiced gating at low complexity and could bite after exactly the
   right number of transitions.

5. **Random / unlikely**: CELT-only path. Ruled out — TOC=0x0B implies
   `MODE_SILK_ONLY` selected by both encoders.

## Strategy

**Do nothing source-wise until a reproducer lands.** The `FUZZ_PANIC_CAPTURE_DIR`
hook is now installed on `fuzz_encode_multiframe` (and 6 other targets
optionally, see campaign journal). The fuzzer was relaunched with a 12-h
budget — given that Bug D took ~36 minutes and Bug E fired at exec 42078 on
the same target, the re-occurrence budget is generous.

Once a crash file appears in `c:/language/mdopus/fuzz_crashes/`:

1. Repurpose `tools/repro_bug_d2.rs` into `tools/repro_bug_e.rs` (byte-exact
   replay from the corpus file).
2. Run `target/release/mdopus-interframe.exe <crash_file>` to walk through
   frames 0–7 comparing the 16 tracked inter-frame state fields. If any field
   diverges before frame 8 the bug is state-accumulation; if none diverge
   through frame 7 then the bug is inside `SilkEncoder::analyze_frame` or
   `encode_frame` for frame 8 itself (not state carry-over).
3. Add a `#[test]` to `src/silk/encoder.rs` that feeds the exact PCM for
   frames 0–8 and asserts byte-exact output. Fix until it passes.

## Constraints for the investigating agent

- Do **not** fix without a reproducer. The hypothesis list above is plausible
  but unconstrained — acting on it without byte-exact replay risks a wrong fix.
- Do **not** revert any part of the Bug D fix when investigating; the two are
  known-distinct.
- Keep the fuzzer alive until a repro appears — don't rebuild the multiframe
  fuzzer again during this campaign.
