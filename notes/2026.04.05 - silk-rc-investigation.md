SILK Rate Control Investigation - 2026-04-05
=============================================

## Bug 1: silk_control_snr called with wrong rate (FIXED)

In src/silk/encoder.rs, the per-channel encode loop was calling:
  silk_control_snr(&mut enc.state_fxx[n].s_cmn, enc_control.bit_rate)

But C (enc_API.c:513) uses:
  silk_control_SNR(&psEnc->state_Fxx[n].sCmn, channelRate_bps)  // = TargetRate_bps

where TargetRate_bps is adjusted for LBRR bits, bit reservoir decay, and intra-packet
balance. The raw bit_rate ignores all these corrections.

Fix: Changed to use target_rate (the adjusted rate computed earlier in the function).
Also removed a redundant silk_control_snr call before the per-channel loop that doesn't
exist in the C code.

## Bug 2: silk_nsq_del_dec produces different pulses (NOT FIXED)

Proven via tracing that at fc=1 iter=0:
- All NSQ function inputs are identical (gains, pred_coef, ltp_coef, ar, harm_shape,
  tilt, lf_shp, pitch_l, lambda, ltp_scale, x_frame) -- verified by comprehensive hash
  input_hash=5987894383036581650 in both C and Rust.
- All NSQ state buffers are identical (xq, s_ltp_shp_q14, s_lpc_q14, s_ar2_q14,
  plus scalar fields) -- verified by comprehensive hash state_hash=1846521789164128144.
- Yet the pulses differ: C produces pulse_sum=40, Rust produces pulse_sum=55.

The bug is INSIDE silk_nsq_del_dec itself. Possible causes:
- Delayed-decision state management (copy/replace logic differs from C's partial memcpy)
- Winner selection tie-breaking (if RD values are equal, order matters)
- Seed/random state initialization or propagation
- Integer overflow behavior differences (wrapping vs saturation)
- The partial memcpy emulation at line ~4170 (C uses offset-based memcpy, Rust clones
  and restores first i values)

This needs a dedicated comparison of silk_nsq_del_dec_scale_states and the inner
quantizer loop, sample by sample, for the first voiced frame (fc=1).
