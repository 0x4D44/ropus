SILK Encoder 16kbps Divergence Investigation (2026-04-05)
========================================================

Symptoms:
- Only affects 16kbps CBR encoding with noise signals (not silence or sine)
- 24kbps, 32kbps, 64kbps all pass bit-exact
- Frame 0 (first Opus packet) matches perfectly
- Frame 1 (second Opus packet) diverges starting at byte 3

Investigation method:
- Added C_TRACE (in reference C) and R_TRACE (in Rust) to compare:
  - Autocorrelation, scale, nrg, q_nrg per subframe
  - Gains (pre-tweak and post-tweak), gain_mult, gain_add
  - Gains indices, NLSF indices, quant_offset, lambda
  - NSQ pulse sums per subframe

Key findings:

1. SILK frame 1 (frame_counter=1): ALL values match exactly between C and R:
   - autocorr, scale, nrg, q_nrg, gains, nlsf_indices, gains_indices, lambda
   - NSQ pulse sums: [0,29,33,31] match

2. SILK frame 2 (frame_counter=2): DIVERGENCE begins
   - autocorr, scale match
   - gains_q16 match: [480247808,480247808,480247808,480247808]
   - gains_indices match: [55,4,4,4]
   - BUT NLSF indices differ:
     C: [10,-2,0,-1,0,0,-1,0,0,0,0]
     R: [4,-2,0,-1,0,0,0,0,1,0,0]
   - AND pulse sums (iter=0) differ:
     C: [29,31,38,28]
     R: [33,31,30,31]

3. Despite SILK frame 2 differences, Opus packet 0 (containing SILK frames 1+2)
   still matches in the encoded output. This is because the CBR rate control loop
   iterates and converges to a matching solution.

4. SILK frame 3 (frame_counter=3, first frame of Opus packet 1):
   - gains_q16 match, gains_indices match, NLSF indices match
   - BUT NSQ diverges from subfr=1 onwards:
     C: [20,28,41,26] vs R: [20,29,32,31]
   - This cascades into different NLSF indices for frame 4 and beyond

Root cause hypothesis:
- The NLSF index divergence for SILK frame 2 suggests that the LPC analysis
  (silk_burg_modified) is producing different results for frame 2.
- Since the autocorrelation from noise_shape_analysis matches, but NLSF from
  find_pred_coefs differs, the difference is in the Burg LPC estimation itself.
- The Burg method uses the raw signal x_buf, which should be identical.
- The carried-over NSQ state from frame 1 subtly differs despite identical
  pulse sums (the individual pulse VALUES and NSQ filter states may differ
  in ways that don't affect the sum but affect subsequent frames).
- At 16kbps, the quantizer is more constrained, making these subtle
  differences manifest as different quantization decisions.

Resolution (2026-04-05):
- silk_burg_modified outputs (LPC coefficients) were bit-exact between C and Rust
  for ALL frames, both full-frame (nb=4) and half-frame (nb=2) calls.
- Root cause: off-by-one in silk_find_lpc_fix NLSF interpolation search.
  In the second silk_sum_sqr_shift call (for the second subframe's residual energy):
    WRONG: silk_sum_sqr_shift(&lpc_res[d + subfr_length..d + 2 * (subfr_length - d)])
           -> length = subfr_length - 2*d (missing d samples)
    RIGHT: silk_sum_sqr_shift(&lpc_res[d + subfr_length..2 * subfr_length])
           -> length = subfr_length - d (matches C reference)
  This caused the interpolation residual energy comparison to use a shorter
  slice for the second subframe, making interpolation appear better than it was.
  The C reference consistently chose interp=4 (no interpolation), while Rust
  was incorrectly choosing various interpolation values (0-3), leading to
  different NLSF coefficients and cascading divergence.
- Fix: single line change in silk_find_lpc_fix at ~line 5484.
- All bitrates (16k, 24k, 32k, 64k) and all sample rates now bit-exact.
