assertion `left == right` failed: Frame 4/10 byte mismatch:
  sr=8000, ch=1, app=2048, br=12000, cx=5, len=30

## Hypotheses
1. Encoder prediction/gain state not updated identically across frames in one
   of the CELT-encode paths (LOWDELAY forces CELT-only; no SILK in play).
2. Some per-frame variable in the Rust encoder reads a stale value that C
   recomputes, or vice versa — latent since fix for Bug A/B/C may have
   altered interim values that only bite in LOWDELAY mode.
3. Pitch / tonality analysis state diverging across frames.

NOT same root cause as Bug A/B/C (celt_fir saturation). That fix is in master
(commit `628b288`) and the current fuzz binary was rebuilt from it.

mechanism also did not fire — its implementation likely relies on a panic hook
in the fuzz harness, which may not be installed for all targets. Reproduction
must come from the config above, not a saved input.

### What the packet bytes tell us

```
rust: [11, 65, 18, 151, 111, 175, 65, 30, 240,  79, 221,  64, 0, 0, ...]
c   : [11, 65, 18, 151, 111, 175, 65, 30, 240,  94, 213, 160, 0, 0, ...]
```

- TOC=0x0B → config=1 (SILK NB-only, 20 ms), stereo=0 (mono), code=3
  (multi-frame with frame-count byte). Byte 1 = 0x41 = 0b01000001 → v=0 (CBR),
  p=1 (padding), M=1 frame. So this packet contains one SILK NB-only 20 ms
  frame with trailing padding — pure SILK, no CELT involvement.
- Bytes 1–8 match → range-coder stream and opening SILK header (stereo flag,
  mid-only flag, VAD flags, LBRR flags, first few quantizer choices) are
  identical. Both encoders made the same gating decisions up to byte 8.
- Bytes 9–11 diverge → the range coder emits `79,221,64` vs `94,213,160`. Both
  sides pad with zeros afterward (same packet length, 30 bytes). So both sides
  consumed the _same number of bits_ but chose _different symbol values_.
- Divergent state therefore lives in a SILK predictor, NLSF codebook selection,
  LTP/pitch predictor, or gain-index prediction that was seeded differently
  across frames 0–3 — not in the range coder itself.

1. `prev_gain_q16` or `last_gain_index` path in the gain quantizer — the fuzz

## Root-cause investigation (2026-04-17, agent 2)

### Reproduction

**All 16 tracked SILK inter-frame state fields match between Rust and C after

### Root cause: missing SILK budget-bust handling in Rust opus encoder

C reference, `reference/src/opus_encoder.c:2578-2599`:

```c
/* In the unlikely case that the SILK encoder busted its target, tell
   the decoder to call the PLC */
if (ec_tell(&enc) > (max_data_bytes-1)*8)
{
   if (max_data_bytes < 2) { RESTORE_STACK; return OPUS_BUFFER_TOO_SMALL; }
   data[1] = 0;
   ret = 1;
   st->rangeFinal = 0;
} else if (st->mode==MODE_SILK_ONLY && !redundancy) {
   /* strip trailing zeros */
   while (ret>2 && data[ret]==0) ret--;
}
```

Rust, `src/opus/encoder.rs:2573-2593`:

```rust
// --- Compute total output bytes ---
// C: opus_encoder.c lines 2578-2601
// … Budget-bust check (C: lines 2578-2589)
if self.mode != MODE_SILK_ONLY {
    // For CELT/hybrid modes, check if encoder busted the budget
    // (shouldn't happen normally, but matches C safety check)
}                                  // <-- EMPTY branch, wrong condition
// Strip trailing zeros (SILK-only, no redundancy) (C: lines 2590-2598)
if self.mode == MODE_SILK_ONLY && !redundancy { … }
```

Two bugs in one block:

1. **Condition is inverted.** In C the bust check is unconditional; in Rust it
   is gated on `self.mode != MODE_SILK_ONLY`, which is exactly the case the
   bug fires in (SILK-only + tight CBR budget).
2. **Body is empty.** Even if the gate were right, the corrective action
   (`data[1]=0; ret=1; rangeFinal=0`) is missing.

### Why the bug only shows up at very low bitrate

At `br=6001, 8kHz, 20ms`:

- `bitrate_to_bits = 120 bits/frame`
- CBR path: `cbr_bytes = (120+4)/8 = 15`, `max_data_bytes = 15`
- `bits_target = min(8*(15-0)-8, 120) - 8 = 104`; range encoder buffer = 14 bytes = 112 bits.

On frame 1 the PCM contains a harsh full-scale noise burst (see `abs_avg`
around 10 000 at amplitude ±15 000). With `complexity=3` the SILK rate
distortion loop fails to stay under 104 bits; `ec_tell(&enc)` ends up above
112. C detects the bust, discards the stream, writes 1 byte of SILK data, and
lets `opus_packet_pad` rebuild the packet as code-3 + padding (which is where
the 0x0B TOC comes from). Rust skips the check and ships the over-the-limit
range-coded stream verbatim.

Because the SILK encoder itself runs identically in both paths (state dumps
match bit-for-bit), this is a pure Opus-layer framing bug, not a SILK
numerics bug. That's also why the 2026-04-16 corpus/PRNG sweep found nothing:
it targeted the default fuzz-reachable bitrate (12 000 bps), which never
triggers the budget bust.

### Proposed fix direction (NOT YET APPLIED)

In `src/opus/encoder.rs::encode_frame_native`, replace the empty mode-gated
branch around line 2578-2582 with the unconditional C-equivalent check. The
range encoder variable is not in scope at this point (it was dropped at the
end of the `{ let mut enc = RangeEncoder::new… }` block around line 2000), so
either (a) capture `enc.tell()` into a local `bits_used` before the range
encoder goes out of scope, and compare against `(max_data_bytes-1)*8` here,
or (b) hoist the check inside that inner scope. Option (a) is smaller.

Sketch (pseudocode, placement inside the inner scope right before `range_final
= enc.get_rng();`):

```rust
let bits_used = enc.tell();        // keep for outer code
// … existing code sets ret, range_final, etc.
```

And then outside the inner scope, replacing the empty branch:

```rust
if bits_used > (max_data_bytes - 1) * 8 {
    if max_data_bytes < 2 { return Err(OPUS_BUFFER_TOO_SMALL); }
    data[1] = 0;
    ret = 1;
    self.range_final = 0;
} else if self.mode == MODE_SILK_ONLY && !redundancy {
    // existing trailing-zero strip
    while ret > 2 && data[ret as usize] == 0 { ret -= 1; }
}
// fall through to `ret += 1 + redundancy_bytes` and CBR padding
```

The `else if` replaces the current separate trailing-zero loop so the two C
branches stay mutually exclusive exactly as in the reference.

Care points before applying:

- `bits_used` must come from the SILK-only branch of the inner scope (line
  ~2378 where `ret = (enc.tell()+7)>>3`). For CELT/Hybrid the same check still
  applies in C — `ec_tell` is the range coder bit count at the point of this
  test, after CELT has written everything. For Rust the CELT branch uses a
  different `ret` definition; need to confirm the bit count is still available
  (likely via `range_enc.tell()` on the outer range encoder or by capturing it
  before finalizing).
- `data[1] = 0` requires writable access to index 1 of the output buffer.
  Rust uses `data.split_at_mut(1)` earlier to split TOC from payload; the
  write needs to land in the payload slice's first byte, so the code will be
  `enc_data[0] = 0;` in Rust terms (or the equivalent after re-joining the
  split). Check the split layout before writing.

### Secondary observations

- Frames 2+ on the reproducer also mismatch on byte 0 (Rust 0x08, C 0x0B)
  with identical downstream state, which is consistent with the same bust
  firing every frame and being handled differently. Fix at the single
  location should repair all subsequent frames.
- The "strip trailing zeros" branch in Rust currently runs in parallel with a
  (missing) bust branch rather than as its `else` arm — this is a separate
  but related deviation from C. Folding both branches into the
  if/else-if structure above fixes it.
- The trailing-zero strip is functionally harmless when bits_used ≤ budget,
  so it can remain a pure `else if` without regressing any passing fuzz
  corpus entries.
- The commit log shows 30+ encoder test cases (`bc_encode_*`) that pass with
  the broken code. None exercise the "busted budget + CBR" corner. Worth
  adding a regression test that fuzzes with br=6001 on the current
  reproducer bytes once the fix lands.

### Regression-check result

  (the br=6001/cx=3 bust path is still correctly handled by the live fix).
- `repro_bug_d2 crash_1c402d7ce86c41d2.bin` → **frame 5 diverges** (new bug).

This is **NOT a regression** of the Bug D fix: the fix is live, verified
(`grep -c bits_used src/opus/encoder.rs` → 3), and passes the original
repro. The new crash exercises a different code path.

### Does the bust branch fire on frame 5?

No. Instrumenting the encoder with per-frame `eprintln!` around the bust
check gave this for frame 5:

```
BUGD_TRACE: bits_used=230 max_data_bytes=30 budget=232 mode=1000
            redundancy=false ret_before=29 orig_max=30
BUGD_TRACE: trailing-zero-strip       # data[29]=132 so no strip
BUGD_TRACE: ret_after_strip=29
BUGD_TRACE: pre_pad use_vbr=0 ret=30 orig_max=30 range_final=94634034
BUGD_TRACE: CBR_PAD skipped            # ret >= orig_max
BUGD_TRACE: final ret=30 toc=0x08
```

- `bits_used = 230`, budget `(30-1)*8 = 232`. 230 ≤ 232 → bust branch does
  NOT fire. Rust emits a full code-0 packet.
- The trailing-zero-strip can't reduce `ret` because the last SILK byte is
  `0x84`, not 0.
- `ret = 30 = orig_max_data_bytes`, so the CBR-padding guard
  `ret < orig_max_data_bytes` is false. Pad is skipped. TOC stays 0x08.

### What C does on the same frame

C emits `[0x0B, 0x41, 0x00, 27 SILK bytes]` = 30 bytes. That decodes as
code-3, M=1 frame, padding flag set, pad_amount=1. Interpretation: C's
SILK produced **27** payload bytes (~216 bits), leaving 3 bytes of room,
so C's `opus_packet_pad` re-framed as code-3 + padding. C did NOT bust.

### Why SILK bytes differ between Rust and C

`mdopus-interframe` (extended to 11 frames) shows that all 16 tracked
SILK inter-frame fields match between Rust and C **after** frame 5
(`last_gain_index`, `prev_gain_q16`, `variable_hp_smth1/2_q15`,
`harm_shape_gain_smth`, `tilt_smth`, `prev_signal_type`, `prev_lag`,
`ec_prev_lag_index`, `ec_prev_signal_type`, `prev_nlsfq_q15[0..16]`,
stereo fields, `n_bits_exceeded`). Frames 0–4 matched byte-for-byte.

Yet on frame 5 Rust's SILK produced 29 bytes (230 bits) and C's SILK
produced 27 bytes (~216 bits). The bytes also disagree from offset 2 of
the SILK payload onwards (byte 0=152, byte 1=122 match; byte 2 is
168 vs 167).

Since SILK is deterministic given input PCM + complete state, and the
PCM is the same for both sides, something in the **untracked** part of
the SILK encoder state diverged across frames 0–4 and only manifested
on frame 5's RD loop decisions. Candidates (none observed by the
interframe harness):

- `sNSQ` — noise-shape-quantizer state (biggest internal buffer, most
  likely culprit). Includes delayed-decision quantization state, LPC
  shaping filter memory, LTP buffers.
- `sLP.In_LP_State`, `In_HP_State` — pre-filter IIR memory.
- `sVAD.*` — speech-activity detector internals (energy ratios,
  NoiseLevelBias, AnaState arrays).
- `frames_since_onset`, `sum_log_gain_Q7`.
- `input_quality_bands_Q15`, `input_tilt_Q15` (feeds noise shaping).
- `pitch_LPC_win_length`, `prevLag` vs `lagPrev` in NSQ.

### Does CBR padding run correctly in the Rust bust path?

Yes. Frame 0 of this reproducer exercises a shorter-SILK case
(`bits_used=225, ret=28 after strip`), hits the `CBR_PAD running`
branch, `opus_packet_pad` returns OK, and Rust emits TOC=0x0B matching
C byte-for-byte. So the Opus-framing pad path works. The bug is SILK
producing a different number of bytes on frame 5, not padding failing
to run.

### Root-cause hypothesis (single, falsifiable)

Some non-tracked SILK encoder state variable (most likely in `sNSQ` or
pre-filter state) accumulates a bit differently between Rust and C over
frames 0–4. That drift doesn't alter the sample-by-sample output enough
to show up in the 16 tracked fields but changes SILK's rate-distortion
decisions on frame 5, producing a longer range-coded payload in Rust
(29 bytes vs 27). Rust's longer payload happens to exactly fill the 30-
byte CBR target, so CBR padding is skipped and the packet stays
code-0. C's shorter payload leaves room, so C pads and reshapes to
code-3.

This is **not** a gap in the Bug D fix and not a regression. It is a
