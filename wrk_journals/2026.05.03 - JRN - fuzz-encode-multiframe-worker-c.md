## 23:20 BST - Root cause and first validation

Incorporated devils-advocate guidance: did not classify this as the old VBR/FEC/DTX class. The config is CBR/no-FEC/no-DTX mono at `sr=12000`, and the packet prefix matters. Rust produced a code-0 SILK packet starting `08 85 f6 ...`; C produced a same-length CBR-padded code-3 packet starting `0b 01 85 f6 ...`. That means C's compressed SILK payload was one byte shorter and then padded to the CBR target, while Rust spent extra SILK bits and did not need padding.

The byte-budget symptom matches the Phase B/Phase C trace: on frame 1 C ended `ec_tell=100`, Rust ended `ec_tell=109`; frame-1 first divergence was immediately after `silk_LP_variable_cutoff` on the resampled input buffer. The failing shape downshifts SILK internal rate from 12 kHz on frame 0 to 8 kHz on frame 1.

Root cause found in `silk_setup_resamplers`: C uses `silk_resampler_init(temp_resampler_state, old_internal_fs, API_fs, 0)` for the temporary internal-to-API correction resampler. Rust called the encoder wrapper, which always used `for_enc=true`. For the observed `12 kHz -> API 12 kHz` temporary correction this picks encoder delay `7` instead of decoder delay `9`, corrupting the corrected history and causing a low-CBR frame-2 SILK bit-budget drift.

Fix applied: initialize only the temporary correction resampler with `for_enc=false`; keep the actual API-to-new-internal encoder resampler on `for_enc=true`. Added a focused unit-test assertion around the 12 kHz to 8 kHz downshift path.
