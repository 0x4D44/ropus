## Review Checklists

### CELT Fix HLD — Review Checklist
- [ ] Does it identify the EXACT state/value that differs (not just "something in CELT")?
- [ ] Does it have file:line references to both C and Rust code?
- [ ] Is the root cause supported by evidence (diagnostic output, code comparison)?
- [ ] Is the proposed fix MINIMAL — doesn't refactor surrounding code?
- [ ] Does it explain WHY this only triggers for live-encoder transitions, not fresh-encoder sweeps?
- [ ] Verification plan includes: transitions test, sweep (no regression), longsoak (no regression)
- [ ] Risk assessment names specific tests that might be affected
- [ ] Does NOT contain any implementation code (pseudo-code OK)

Red flags to watch for:
- Vague root cause ("state gets out of sync")
- Broad refactoring proposal (should be surgical)
- No explanation for why sweep passes but transitions fail

### DA Findings HLD — Review Checklist
For each of 5 findings:
- [ ] Concrete implementation approach with pseudo-code or type signatures
- [ ] Clear decision on where the code lives (new subcommand vs. extend existing)
- [ ] Pass/fail criteria defined precisely (byte-level, sample-level, or range-level)
- [ ] Integration with `tools/torture_test.sh` called out

Finding 1 (cross-decode) specific:
- [ ] 4-way matrix clearly defined
- [ ] Handles case where encoders diverge (expected for some tests)

Finding 2 (transition bursts) specific:
- [ ] Specific burst sequences listed
- [ ] Interleaving schedule defined
- [ ] Deterministic (not random) so bugs are reproducible

Finding 3 (stereo→mono) specific:
- [ ] All 5 conditions addressed explicitly
- [ ] References the C reference line 1562-1567 behavior
- [ ] Verification that `to_mono=1` fires correctly

Finding 4 (PLC during transitions) specific:
- [ ] Tests all 3 drop positions (before/during/after)
- [ ] Accounts for decoder state divergence from drops

Finding 5 (state comparison) specific:
- [ ] Lists specific CTL queries
- [ ] Defines cadence (every N frames)
- [ ] Decision: fail vs warn on state-only divergence

Red flags to watch for:
- "TODO: figure out later" or "we should consider..."
- Missing pass/fail criteria
- Over-engineering (building framework when a targeted test suffices)
- Under-engineering (hand-wave that doesn't describe the actual mechanism)
