## Findings

The next Phase 3.1 slice is not blocked if framed as gate honesty and preflight policy. It is blocked if framed as "make DRED payload byte parity pass now".

Current state:

- DNN PLC has real active SNR gates in `harness-deep-plc/tests/tier2_snr.rs`.
- DRED has active format-level encode/parse/process tests in `harness-deep-plc/tests/dred_integrated_encode.rs`.
- The strongest DRED byte-exact payload test is ignored with a clear LPCNet drift blocker.
- DRED PCM reconstruction is deliberately still `OPUS_UNIMPLEMENTED`.
- `full-test` currently reports DNN/DRED assets as optional even under release-preflight.
- Quick release-preflight excludes `ropus-harness-deep-plc`, so it should not claim neural/DRED coverage.

## Design Chosen

I wrote the HLD around a small implementation slice:

- Promote DNN/DRED assets to banner-blocking only for non-quick `--release-preflight`.
- Keep quick release-preflight as a core smoke profile and make the non-claim explicit.
- Reuse existing active `ropus-harness-deep-plc` DNN/DRED tests as the first gate.
- Keep exact DRED payload byte parity as ignored debt until the known LPCNet feature drift is resolved.
- Update oracle matrix wording so nobody reads "DRED covered" as "DRED payload bytes and DRED PCM reconstruction are done".

## Revision After Supervisor Feedback

Arthur reviewed the HLD direction and asked for sharper implementation constraints before signoff. I revised `/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - dred dnn parity gate.md` accordingly.

Changes made:

- Fixed release-preflight terminology. The HLD now requires distinct claim wording for default report-only, quick release-preflight core-smoke/no-neural-claim, and non-quick release-preflight core+neural-DRED gate.
- Made the implementation contract explicit: `preflight::capture` must receive full CLI `Options` or a small derived `PreflightPolicy` enum, not just a boolean.
- Added the requirement that unit tests simulate all asset files in tempdirs and do not depend on the actual local `reference/dnn` tree.
- Tightened scope to `full-test` policy/report/oracle-matrix docs only unless a later reviewer expands it.
- Added signoff conditions and repeated the non-goals: no codec changes, no DRED PCM reconstruction, no unignoring exact DRED payload parity.

This resolves my earlier ambiguity around "core profile". If neural assets become banner-blocking, the UI cannot keep saying only "core"; that would be exactly the kind of green-but-misleading result this testing-improvement effort is trying to remove.

## Emotional Temperature

The tempting move is to "just unignore the DRED parity test", but the code comments and older HLDs are clear that this would turn a known upstream drift into a noisy failure rather than a better gate. The useful move is to make release-preflight honest first, then let a future correctness slice attack LPCNet drift directly.

After revision: calmer. Arthur's feedback was right; the first HLD had the policy shape but left too much room for UI drift and accidental harness scope creep. The revised version is a better implementation brief.
