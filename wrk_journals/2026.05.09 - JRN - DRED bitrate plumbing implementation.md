# DRED bitrate plumbing — implementation campaign

## Why

The encoder setter-sequence audit (`2026.05.09 - JRN - encoder
setter-sequence audit.md`) cleared the surface area of the recurring
mode-transition defect class but flagged three known-gaps, all
DRED-related and all stemming from the same root cause: ropus does
not port C's `compute_dred_bitrate`. Findings:

- **F33** — DRED CBR override block (C `opus_encoder.c:2470-2477`):
  under `!use_vbr && dred_duration > 0`, C calls `OPUS_SET_VBR(1)`,
  `OPUS_SET_VBR_CONSTRAINT(0)`, `OPUS_SET_BITRATE(celt_bitrate)`. Rust
  skips the block — CELT runs CBR while C runs unconstrained VBR.
- **F53** — `st->bitrate_bps -= dred_bitrate_bps` (C 1338): Rust never
  computes `dred_bitrate_bps`, so the SILK/CELT budget over-allocates
  vs. C when DRED is active.
- **F48** — multi-frame `first_frame = (i==0) || (i==dtx_count)`
  (C 1777): Rust uses `i == 0` only. DRED attaches to the doomed
  frame when sub-frame 0 is DTX-dropped.
- **(quieter)** — `dred_q0/d_q/qmax/target_chunks` are statically
  initialised in `set_dred_duration` (`encoder.rs:1277-1280`) and
  never updated per-frame. C's `compute_dred_bitrate` writes them
  per-frame based on bitrate, packet-loss, and FEC config. Today's
  DRED extension emission therefore uses stale (wrong) quantizer
  levels relative to C.

All four divergences land on DRED-active inputs only and are
documented `TODO(stage-8.x)` in the source. Bundling because the
common dependency is `compute_dred_bitrate`.

## Quality bar

- **Bit-exact with C reference** under DRED-active inputs (the
  codec's standard Tier-1 bar). If f32 drift turns up, drop to Tier-2
  SNR-bounded **only with explicit notice in the journal** — not
  silently.
- **No regressions**: 1867/1867 lib tests, 17/17 differential param
  grid, all existing fuzz known-failures replay clean.
- **No new warnings**: `cargo build`, `cargo clippy` clean.
- **No backwards-compat shims**: this is greenfield code; the existing
  static init at `set_dred_duration` should be replaced, not left as
  a fallback.

## Plan (stages)

### Stage 1 — Detailed implementation plan
Plan agent walks the C source for `compute_dred_bitrate` /
`estimate_dred_bitrate` / `dred_bits_table` and the three call sites.
Produces an HLD-ish doc under `wrk_docs/` listing exact insertion
points, signatures, threading, test surface. No code yet.

### Stage 2 — TDD test scaffolding
Implementation agent writes failing unit tests against the to-be-ported
functions (using known C reference values for golden inputs/outputs)
and a differential test that compares Rust vs. C reference output for
DRED-enabled encode. Tests must compile and fail with clear messages.
**Commit at end of stage.**

### Stage 3 — Implementation
Implementation agent ports `dred_bits_table`, `estimate_dred_bitrate`,
`compute_dred_bitrate`, threads `dred_bitrate_bps` through
`encode_frame_native` / `encode_multiframe`, adds F33 setter block,
adds F53 subtraction, adds F48 first-frame fix, replaces the static
init at `set_dred_duration`. Tests from Stage 2 should now pass.
**Commit at end of stage.**

### Stage 4 — Code review (parallel: code-reviewer + devil's advocate)
- code-reviewer agent: standard review using `code-reviewer` skill.
- devil's advocate agent: actively challenges the implementation —
  argues for ways it could be wrong, looks for hidden assumptions,
  questions the f32 bit-exactness claim, probes edge cases.
Both produce written feedback. **No commit.**

### Stage 5 — Apply review feedback
Implementation agent applies feedback from both reviewers using the
`code-review-apply-feedback` skill. Re-runs all gates.
**Commit at end of stage.**

### Stage 6 — Re-review if material changes (skip if Stage 5 is light)
Decide based on Stage 5 diff size. If non-trivial, run another
code-reviewer pass. **Commit at end of stage if applied.**

### Stage 7 — Plan-completeness verification
Verification agent compares final state vs. Stage 1 plan, reports
any unmet items. If gaps: spawn another implementation agent to
close them, then re-verify. **Commit at end of stage.**

### Stage 8 — Closeout
Supervisor writes closeout journal section, archives crash artifacts
if any, confirms working tree clean. **Final commit if needed.**
