## Summary

New 24-bit FFI wrappers (`opus_decode24` / `opus_multistream_decode24` / `opus_encode24`), a new `OpusMSDecoder::decode24` path, and wiring of `opus_demo.c` + `opus_compare.c` as compiled-verbatim static libs driving 24 IETF vector subtests. FFI-safety patterns mirror the existing `opus_decode` shims cleanly. Implementation is careful and defensive; findings below are mostly hygiene and scope, not correctness.

## Strengths

- FFI entry points are uniformly wrapped in `ffi_guard!` + `#[unsafe(no_mangle)]`; null / negative / overflow checks are present on every new path and match the existing `opus_decode` template.
- `(s as i64 + 128) >> 8` in `opus_encode24` (`capi/src/encoder.rs:266`) is actually *safer* than C's `PSHR32(a, 8)` — C invokes signed overflow UB when `a` is near `i32::MAX`; Rust's i64 widening avoids it and still matches `SAT16(PSHR32(a, 8))` semantics for all in-range inputs.
- `FIXED_POINT=1` applied uniformly across every conformance C-source compile, not just the new ones — correct hygienic fix.
- Fetch scripts abort cleanly on size or SHA-256 mismatch before touching the destination directory.
- Test driver uses an atomic counter + PID for temp-dir uniqueness, which is defensive even though `--test-threads=1` is required.
- `force_link()` (`tests/conformance/tests/ietf_vectors.rs:40-87`) correctly references the new `opus_decode24`, `opus_multistream_decode24`, `opus_encode24` symbols alongside the existing set.
- `OPUS_STUB_UNIMPLEMENTED = -5` in `dred_stub.c:29` is correct per `opus_defines.h`.

## IMPORTANT

### I1. DRED stub scope creep — 6 of 13 symbols unnecessary

**File:** `C:\language\mdopus\tests\conformance\src\dred_stub.c:31-122`

The stub defines 13 symbols but `opus_demo.c` only references 7: `opus_dred_decoder_create`, `opus_dred_alloc`, `opus_dred_decoder_ctl`, `opus_dred_parse`, `opus_decoder_dred_decode24`, `opus_dred_free`, `opus_dred_decoder_destroy` (verified by grep against `reference/src/opus_demo.c`). The other 6 are not referenced anywhere in `opus_demo.c` or `opus_compare.c`:

- `opus_dred_decoder_get_size` (line 31)
- `opus_dred_decoder_init` (line 40)
- `opus_dred_get_size` (line 55)
- `opus_dred_process` (line 83)
- `opus_decoder_dred_decode` (line 91)
- `opus_decoder_dred_decode_float` (line 113)

Per the HLD's "minimum viable" bar and CLAUDE.md's "write as little code as possible", drop the 6 unused stubs. Saves ~40 lines and narrows the future conflict surface when DRED is ported for real.

**Fix:** Delete the 6 unused functions and their signatures. Keep only the 7 actually referenced.

### I2. `decode24_native` duplicates `decode_native` (~100 LOC)

**File:** `C:\language\mdopus\ropus\src\opus\multistream.rs:1623-1762`

`decode24_native` is a near-line-for-line duplicate of `decode_native` (lines 1764-1912), diverging only in the output type (`&mut [i32]` vs `&mut [i16]`) and the channel-routing callback (`copy_channel_out_int24` vs `copy_channel_out_short`). The C reference avoids this via a `void*` callback parameter; safe Rust can't express it quite as cleanly, but a generic parameter over the copy-out function would collapse the duplication.

**Risk:** Any future fix to packet validation, stream iteration, muted-channel handling, or the coupled/mono dispatch must be applied in both places. The supervisor's journal already notes 2 genuine decoder bugs surfaced by these tests (stereo vectors 02 and 10); if the fix lands in `decode_native` but not `decode24_native`, the latter keeps the bug.

**Fix (deferrable):** Refactor to a generic `decode_native_inner<T>(..., copy_out: fn(&mut [T], ..., Option<&[i16]>, ..., usize))`. Not a blocker for Piece A commit, but file a follow-up now so it isn't forgotten.

### I3. Journal claims "Drop guard" cleanup, code uses inline `remove_dir_all`

**File:** `C:\language\mdopus\tests\conformance\tests\ietf_vectors.rs:200`

Journal says: "Temp files live under the Cargo test target dir; cleaned up in a `Drop` guard." Actual code at line 200 is `let _ = std::fs::remove_dir_all(&tmp);` inline after the assertions. If any `assert!` / `assert_eq!` earlier panics, temp files leak. The inline comment ("OK to leave files on failure for debugging") suggests this is deliberate, but either the comment or the journal is misleading.

**Fix:** Either (a) add a real `Drop` guard as the journal claims, or (b) update the journal to match the code. Option (b) is fine — leaving temp files on panic *is* actually useful for debugging, so the current behavior is reasonable; just don't call it a "Drop guard".

### I4. `lossgen_stub.c` is dead weight

**File:** `C:\language\mdopus\tests\conformance\src\lossgen_stub.c:27-39`

The stub comment correctly notes: "We DO NOT define `ENABLE_LOSSGEN` for the conformance build of opus_demo, so these stubs are inert." Verified: all 3 references in `opus_demo.c` (lines 425, 669, 1090) are gated by `#ifdef ENABLE_LOSSGEN`. With the macro undefined, the symbols are never referenced at the object-code level, so the stub provides unused definitions that the linker drops as dead code.

This matches the HLD's design ("15-line `lossgen_stub.c`") but the HLD's rationale ("at link time we provide a 15-line stub") is slightly wrong — *no* link-time symbols are needed.

**Fix:** Two options:
- (a) Delete the stub entirely. Document the rationale in `build.rs` ("ENABLE_LOSSGEN undefined → no lossgen symbols referenced → no stub needed").
- (b) Keep the stub as belt-and-braces insurance and update the comment to match reality.

Implementer's current comment *does* acknowledge this ("they exist as belt-and-braces insurance if anyone ever flips that define"). Consistent with CLAUDE.md's "don't predict the future" — (a) is the cleaner choice. Not a blocker.

## NIT

### N1. Inaccurate docstring on `copy_channel_out_int24`

**File:** `C:\language\mdopus\ropus\src\opus\multistream.rs:809-811`

"Each sample is left-shifted by 8 so that i16 full-scale lands in the **top 24 bits** of the i32 word" — with `(i16 as i32) << 8`, an `i16` value of 32767 produces `i32` value 8388352, which lives in the low 24 signed bits (bits 0-23 with sign extension into 24-31), not the "top 24 bits". Reword to "scaled to 24-bit signed range within the i32 word" or similar.

### N2. `opus_encode24` rounding comment off-by-one

**File:** `C:\language\mdopus\capi\src\encoder.rs:263`

Comment says "SAT16(PSHR32(s, 8)): add 128, >>8, clamp to i16 range." The +128 is `1 << (shift-1) = 1 << 7`, the round-to-nearest bias. Worth calling out in the comment that +128 is the rounding bias, not an arbitrary constant (future maintainer could misread it as a magic number).

### N3. Windows path handling in test driver

**File:** `C:\language\mdopus\tests\conformance\tests\ietf_vectors.rs:111-113`

`CString::new(p.to_string_lossy().into_owned())` converts OsString → UTF-8. On Windows, `opus_demo`'s `fopen(argv[i], "rb")` uses ACP (ANSI code page), not UTF-8. If `std::env::temp_dir()` ever resolves under a path with non-ACP characters (rare on CI; possible on a developer box with a non-ASCII username), the file open will fail with a confusing error.

Low priority — in practice this won't hit, but a comment noting the limitation would save future debugging time.

### N4. Fetch script resume semantics

**File:** `C:\language\mdopus\tools\fetch_ietf_vectors.sh:48`

`curl -fL -o "${archive}" "${URL}"` doesn't pass `--continue-at -`, so an interrupted 71 MB download has to restart from scratch. Not a blocker — 71 MB on a reasonable connection is 10-60 seconds — but one extra flag would make the script more resilient on flaky networks.

### N5. Sentinel file choice for "already present" check

**File:** `C:\language\mdopus\tools\fetch_ietf_vectors.sh:39` and `.ps1:34`

Using `testvector12m.dec` as the "vectors already extracted" sentinel is fine, but if an extraction is interrupted partway through, that specific file may or may not exist depending on tarball order. The sanity-check loop at the end of the script will re-detect missing files, but only on fresh runs — a partial extraction followed by a re-run skips the download because the sentinel exists. Low probability; the `trap cleanup EXIT` handles the common interruption cases.

### N6. `#if 0` aliasing block in opus_demo is confusingly documented

**File:** `C:\language\mdopus\tests\conformance\tests\ietf_vectors.rs:69-70`

Comment says "the #if 0 block at opus_demo.c:306 is disabled so these aren't called". Journal also notes the HLD was wrong about this. Worth inlining a 1-line comment in `build.rs` or README.md near the opus_demo row explaining that `opus_multistream_decode24` is only link-resolved (not runtime-called) for future maintainer clarity.

### N7. Conformance README doesn't document temp-file behaviour

**File:** `C:\language\mdopus\tests\conformance\README.md`

The README explains the test suite well but doesn't mention that failed `ietf_vectors` tests leave temp files behind (useful for debugging). Worth one line.

## Verdict

**approve-with-nits**

Implementation is sound. The 24-bit path is correctly derived from `SHL32(EXTEND32(a), 8)` / `SAT16(PSHR32(a, 8))` per `reference/celt/arch.h`. FFI safety is consistent with existing handle-indirection patterns. Test driver and fetch scripts handle the common failure modes.

Items I1 (DRED scope trim) and I2 (duplication tracking) are worth addressing before commit or immediately after — neither is a correctness blocker. I3 and I4 are documentation/hygiene. All NITs are optional polish.

The 2 `testvector_stereo` failures are correctly flagged as out-of-scope decoder bugs for a separate workstream (per brief).

No rework needed on the reviewed surface. Ship it after I1 (DRED trim) and optionally I2 (refactor ticket filed).
