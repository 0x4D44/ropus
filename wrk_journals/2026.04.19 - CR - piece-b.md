### NIT

#### 1. Commit message overstates build.rs panic-message change
**`C:\language\mdopus\tests\conformance\build.rs:132-136`**
The commit message claims "Panic message now points at `cargo run -p
fetch-assets -- reference` like harness/build.rs does", but the actual code
still reads `"Clone https://github.com/xiph/opus into reference/."`. The
sibling `harness/build.rs` uses the `fetch-assets` phrasing.

*Fix:* Either update the panic message to match the claim:
```rust
panic!(
    "Reference source not found: {}.\n\
     Install it with:\n    cargo run -p fetch-assets -- reference",
    src_path.display()
);
```
or amend the commit message. Non-blocking ‚Äî cosmetic drift.

#### 2. `_get_size` oversizing is arbitrary and undocumented at the call site
**`C:\language\mdopus\capi\src\projection.rs:42-48`**
`proj_enc_size_for` returns `16 * 1024 * channels.max(1)` and
`proj_dec_size_for` returns `16 * 1024 * streams.max(1)`. These are sized
well above the 24-byte handle footprint so `malloc(get_size) + _init` works
without overflow. `test_opus_projection.c` never calls `_get_size`, so the
choice has no observable consequence today, but the "16 KB √ó channels"
formula is a magic number. A short comment stating the rationale (matches
reference's ambient practice of oversizing) would prevent future readers
from assuming it's load-bearing. Non-blocking.

#### 3. `opus_projection_decode_float` writes no safety assurance that a future test wouldn't silently fail
**`C:\language\mdopus\capi\src\projection.rs:544-564`**
`opus_projection_decode_float` returns `OPUS_UNIMPLEMENTED` if the handle
resolves. If a future test accidentally calls it (e.g. because
`DISABLE_FLOAT_API` is dropped from a future config), the test will see
a negative return and continue. The current test body has
`#if !defined(DISABLE_FLOAT_API) && !defined(FIXED_POINT)` gates around
float paths, so this is cleanly protected *today*.

Acceptable given the journal's rationale ("mirrors the multistream float
pattern"). Flagging as NIT for future awareness: if a consumer ever drops
`DISABLE_FLOAT_API`, they'll hit a compile-time error in `mapping_matrix.c`
(missing `FLOAT2RES`) before reaching the runtime `UNIMPLEMENTED`, so
failure is loud. No action needed. Worth checking one more time before
Piece C lands that no reference test under regression_test() exercises
projection float.

#### 4. `#[allow(dead_code)]` on `handle_to_proj_decoder_ref`
**`C:\language\mdopus\capi\src\projection.rs:215-220`**
Pattern-symmetric with `handle_to_proj_encoder_ref` (which is used), but
the decoder-ref version has no caller. Either delete it now (preferred ‚Äî
CLAUDE.md says "make the smallest change" and "don't predict the future")
or leave a TODO citing the expected Piece C consumer.

#### 5. `force_link()` in projection.rs lists `opus_projection_encoder_ctl` / `_decoder_ctl` implicitly via unresolved references
**`C:\language\mdopus\tests\conformance\tests\projection.rs:30-68`**
The comment on line 55-56 states `opus_projection_{encoder,decoder}_ctl`
are "reached through the varargs shim in ctl_shim.c" and therefore aren't
in the Rust force-link list. This works because
`test_opus_projection.c`'s reference to the varargs symbol is undefined
at its own object-file level, forcing the linker to pull in
`ctl_shim.c`'s object from the capi staticlib. It's correct, but one-line
documentation of that linker-resolution reasoning would prevent future
confusion. Non-blocking.

## Non-findings (explicit verification)

1. **Projection CTLs on encoder vs decoder.** Journal flagged as HLD
   deviation. Verified correct: `reference/include/opus_projection.h:48-50,
   72-90` define `OPUS_PROJECTION_GET_DEMIXING_MATRIX*` as encoder CTLs,
   and `reference/tests/test_opus_projection.c:205, 212, 308, 315, 323`
   all call them on `st_enc`. HLD was wrong; implementation is right.

2. **Codec changes minimal and getter-only.** Verified: the six added
   accessors on `OpusProjection{Encoder,Decoder}` (`multistream.rs:2302-2315,
   2433-2446`) each delegate one line to already-public getters on
   `ms_{encoder,decoder}` (`multistream.rs:1526-1538, 1978-1990`). No new
   logic, no behavioural change. Acceptable reading of the "minimal
   accessor" provision the HLD explicitly anticipated.

3. **Vendored header hygiene.**
   - `tests/conformance/include/opus_projection.h` ‚Äî verbatim against
     `reference/include/opus_projection.h` (diff empty).
   - `tests/conformance/include/mapping_matrix.h` ‚Äî verbatim against
     `reference/src/mapping_matrix.h` (diff empty).
   - `tests/conformance/include/os_support.h` ‚Äî verbatim against
     `reference/celt/os_support.h` (diff empty). Journal's "genuinely
     minimal ‚Äî just `opus_alloc` ‚Üí `malloc`" claim is conservative: the
     file additionally defines `opus_realloc`, `opus_alloc_scratch`,
     `OPUS_COPY`, `OPUS_MOVE`, `OPUS_CLEAR`, all trivial wrappers. No
     meaningful logic duplication.
   - `float_cast.h` and `mathops.h` ‚Äî newly authored stubs (~20 LOC each).
     Both just claim the canonical header guard and include a small
     harmless dependency. Strategy justified in comments and consistent
     with the `opus_private.h` stub pattern.
   - `arch.h` ‚Äî subset stub claiming the real `ARCH_H` guard. Macros match
     the `FIXED_POINT=1, ENABLE_RES24=0` path of the real header. The
     expanded macro set (SAT16, PSHR32, align, INT24TORES, etc.) is the
     minimum needed by `mapping_matrix.c`. Upstream-drift risk is low
     given fixed-point Opus primitives are stable.

4. **Handle layout parity.** `OpusProjectionEncoderHandle` /
   `OpusProjectionDecoderHandle` use the same magic + inner-pointer +
   generation-counter pattern as `OpusMSEncoderHandle` (see
   `ms_encoder.rs:22-27`). Magic values distinct: `MDOPUSPE` (0x4D44...5045)
   and `MDOPUSPD` (0x4D44...5044) vs `MDOPUSME` (0x4D44...4D45). No
   collision with `MDOPUSMD`, `MDOPUSEC`, `MDOPUSDC`, `MDOPUSRC`.

5. **FFI safety.** All 12 new `#[unsafe(no_mangle)]` entry points in
   `projection.rs` and all 9 in `ctl.rs` wrap their body in `ffi_guard!`.
   Every handle deref runs through `resolve_{enc,dec}{,_ref}`, which
   checks for null, checks magic, and checks inner-ptr null before
   creating a `&mut` or `&`. Out-pointer nullability is checked in every
   setter before writing. Int-ranges on `OPUS_SET_GAIN` match the MS
   decoder's (`-32768..=32767`). Consistent with `ms_encoder.rs` /
   `ms_decoder.rs`.

6. **Varargs dispatch.** `capi/src/ctl_shim.c:82-86` defines three
   request constants (6001, 6003, 6005) matching
   `reference/include/opus_projection.h:48-50` verbatim. Two-argument
   dispatch for `OPUS_PROJECTION_GET_DEMIXING_MATRIX` handled correctly:
   `va_arg(ap, unsigned char *)` then `va_arg(ap, int)`, forwarded to
   `mdopus_proj_encoder_ctl_get_demixing_matrix`. Matches the macro form
   in `opus_projection.h:90`.

7. **`_decode_float` not exercised.** Verified: no call to
   `opus_projection_decode_float` or `opus_projection_encode_float` in
   `reference/tests/test_opus_projection.c`. The test's float block is
   gated by `#if !defined(DISABLE_FLOAT_API) && !defined(FIXED_POINT)`
   and both macros are set in `build.rs`. `OPUS_UNIMPLEMENTED` return is
   safe.

8. **`DISABLE_FLOAT_API` define.** Verified: `mapping_matrix.c:84-146`
   contains a `#ifndef DISABLE_FLOAT_API` block that calls `FLOAT2RES` /
   `RES2FLOAT` ‚Äî neither of which is defined in our stub `arch.h`.
   Defining `DISABLE_FLOAT_API=1` for the projection test is a real
   build requirement, not scope creep. Interaction with `FIXED_POINT=1`
   (set uniformly for all conformance C compiles) is clean: both defines
   together select the i16 fixed-point branch.

9. **Rust-side `len` range check in `opus_projection_decode`.** Mirrors
   `capi/src/ms_decoder.rs:290-291` exactly (null data or zero len =
   PLC; otherwise negative len = OPUS_BAD_ARG). Consistent.

10. **TDD harness deviation.** The journal's FFI-symbol-collision claim
    is real: both our capi staticlib and `opus_ref` export
    `opus_projection_encode`, so a `harness/tests/*.rs` that depends on
    both would refuse to link without `#[link_name]` overrides or
    `cfg`-gated modules. The `projection_roundtrip` binary at
    `harness/src/bin/projection_roundtrip.rs` validates the Rust-codec
    side directly (not the capi wrappers), and the conformance test
    covers the capi layer end-to-end. Combined, the two give adequate
    coverage. The skipped TDD step is noted in the journal.

## Verdict

**approve-with-nits.**

All five NITs above are cosmetic or future-leaning. None blocks merge.
The implementation is consistent with the prior capi handle pattern,
FFI-safe, and demonstrates the expected test passes in the journal. The
one documented codec change (`nb_*` accessors) is genuinely minimal and
was foreseen by the HLD's "minimal accessor" language. HLD deviations
are each justified by reference to spec or tooling constraints, and
verifiable against `reference/`.

Recommend merging and proceeding to Piece C.
