It failed as handed off: frame 1/2 length mismatch, Rust `169` bytes vs C
`173` bytes for `sr=24000 ch=1 app=2049 br=40181 cx=3 vbr=1 fec=1 dtx=1
loss=32`.

`fuzz_repro_diff --mode multiframe` first showed Worker C's Phase B count
mismatch (`Rust=13`, `C=7`) and an apparent `tell=1` vs `tell=3` divergence
immediately after LBRR. The C trace macro in `harness/silk_enc_api_traced.c`
explicitly suppresses prefill calls (`psRangeEnc != NULL && !prefillFlag`),
but Rust was recording trace tuples during SILK prefill. I guarded Rust
Phase B trace pushes with `prefill_flag == 0`. After that correction, the
first divergent frame has Phase B counts `7/7` and all LBRR/SILK boundaries
match exactly through the final SILK payload tuple (`tell=832`, rng
`0x010e5e68`, `n_bits_exceeded=160`). Phase C still had an unused
`AR_Q13` tail mismatch, but all SILK range-coder tell/rng values matched.

The real packet mismatch was downstream in HYBRID CELT, not in LBRR. C calls
`CELT_SET_SILK_INFO` before the CELT->SILK redundancy frame and before the
mode-transition `OPUS_RESET_STATE`/prefill sequence. That reset clears
`silk_info`, so the main transition CELT encode uses default CELT-side SILK
info. Rust was setting `SetSilkInfo` after the reset/prefill path, so the main
HYBRID CELT VBR encode saw non-default `signal_type`/`offset` and emitted a
shorter packet (`169` bytes).

Moved the Rust `SetSilkInfo` call to the C-reference position before
redundancy/reset/prefill, and removed the late main-encode setter. Replayed
the artifact after the fix:

`timeout 180 cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_encode_multiframe tests/fuzz/artifacts/fuzz_encode_multiframe/crash-8f106fa45bb06fa21404056b62a41a1551fa0c55 -- -runs=1`

Outcome: passed fixed-input replay. `fuzz_repro_diff` also reports frame 0
`BYTES-OK` at `64B` and frame 1 `BYTES-OK` at `173B`; no divergence reproduced.

Emotional temperature: relieved but wary. The devils-advocate warning was
right: the LBRR signature was the clue, but the first LBRR divergence itself
