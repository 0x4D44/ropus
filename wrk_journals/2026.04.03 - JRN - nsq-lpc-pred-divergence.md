# 2026-04-03 - NSQ LPC Prediction Divergence

## Status at start
- All 26 Opus modules ported to Rust
- Silence @ 8kHz/16kbps: PASS
- Sine @ 8kHz/16kbps: FAIL at offset 16 (175/2100 matching)
- Sine @ 8kHz/64kbps: FAIL (CELT-only mode, offset 8)

## Key finding
The noise_shape_analysis function is bit-exact! The divergence is in the NSQ delayed-decision inner loop, specifically the **LPC prediction** (`silk_noise_shape_quantizer_short_prediction`).

At sample i=3, path k=1, first subframe of first SILK frame:
- C:    LPC_pred_Q14 = 11456
- Rust: LPC_pred_Q14 = 11472
- Difference: 16 (exactly 1 unit in Q10 after >>4)

All other inputs at this point match: n_AR, n_LF, n_LTP, diff_Q14, lf_ar_Q14, seed.
Path k=0 matches perfectly. The divergence is path-specific.

## Root cause hypothesis
The LPC prediction loop:
```rust
for j in 0..predict_lpc_order {
    lpc_pred_q14 = silk_smlawb(lpc_pred_q14, dd.s_lpc_q14[ps_lpc_idx - j], a_q12[j]);
}
```

The `dd.s_lpc_q14` history buffer for path k=1 at sample i=3 must differ from C by enough that the LPC sum differs by 16. Since a_q12 coefficients are shared, the issue is in the LPC history buffer contents.

The LPC history is updated after each quantization decision:
```
xq_Q14 = LPC_exc_Q14 + LPC_pred_Q14
dd.s_lpc_q14[idx] = xq_Q14
```

A difference in the quantized excitation or prediction at a previous sample would propagate into the history. But samples i=0,1,2 all match for all paths at the NSQ_INNER level. So the state update must differ despite having the same NSQ_INNER values.

### ROOT CAUSE: silk_quant_ltp_gains missing cross-correlation vector
- The Rust `silk_quant_ltp_gains` only receives the auto-correlation matrix (XX_Q17)
  but NOT the cross-correlation vector (xX_Q17)
- The VQ distortion should be `1 - 2*xX*cb + cb'*XX*cb` but Rust only computes `cb'*XX*cb`
- This makes the LTP completely wrong for voiced audio
- C trace: ltp=[-1152,2432,12032,3712,-1152] (correct voiced LTP)
- Rust trace: ltp=[0,0,256,0,0] (essentially zero, like unvoiced)
- Fix requires: rewrite silk_quant_ltp_gains + implement silk_VQ_WMat_EC

1. The NSQ output from the first frame is correct

3. Or there are more subtle rounding differences in the NSQ inner loop
