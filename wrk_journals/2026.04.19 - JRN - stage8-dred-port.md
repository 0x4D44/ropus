## Strategy adjustment (discovered at 8.0 pre-work)

The HLD's 8.0 step — "add `ENABLE_DRED` + DRED C sources to
`harness/build.rs`" — is **blocked**. Stage 7's team discovered that
upstream xiph forbids `FIXED_POINT + DEEP_PLC` (see
`reference/configure.ac:973`, `AC_MSG_ERROR`), and the restriction
enforces a real incompatibility: DNN code's `freq.c`/`kiss_fft` uses
different `kiss_fft_scalar` type + struct layout than fixed-point
`celt/kiss_fft.c`. Cross-TU silent memory corruption otherwise.

Every DNN-consuming DRED source (dred_encoder.c via lpcnet_enc.c, both
rdovae sources via nnet.c) trips this. Stage 7 deferred the float-mode
comparison harness to an unscheduled "Stage 7b".

**Option D adopted** (Arthur, 2026-04-19): ship the two non-DNN
sub-stages now (8.1 `ec_laplace_p0`; 8.2 `dred_stats` static tables),
pause at 8.4 RDOVAE forward pass until Stage 7b (or our own float-mode
fixture pipeline) unblocks. Clean stopping points, no rework risk.

## Sub-stage 8.1 — ec_laplace_p0 range coder primitives

**Target.** Port `reference/celt/laplace.c:136-192` into
`ropus/src/celt/range_coder.rs` as methods on `RangeEncoder` /
`RangeDecoder`. Used by DRED to code quantised latent + state values
with a symmetric Laplace distribution anchored at zero.

**No DNN surface area.** Pure integer range coding; no kiss_fft,
no fixed/float divergence. Safe to land ahead of Stage 7b.

**Port.** `ec_laplace_encode_p0` and `ec_laplace_decode_p0` added as
methods on `RangeEncoder` / `RangeDecoder` in
`ropus/src/celt/range_coder.rs`. Direct translation from C laplace.c:

-   Sign icdf: `[32768-p0, (32768-p0)/2, 0]`, coded via existing
    `encode_icdf16` / `decode_icdf16` (already in range_coder.rs).
-   Magnitude icdf: decay chain starting at `max(7, decay)`, each next
    entry `max(7-i, (prev*decay)>>15)`. Values > 7 are coded by
    repeated 7-symbol escapes.
-   Guardrail checks (C: `abs`, `IMAX`, `IMIN`) map to Rust's
    `unsigned_abs`, `.max()`, `.min()`.

**Tests.**

1.  **Rust-only round-trip** (`ropus/src/celt/range_coder.rs` tests
    module, `encode_decode_laplace_p0_roundtrip`). Encodes 13 values
    across the DRED quantiser range (0, ±1, ±2, ±7, ±8, ±15, ±30)
    under 4 representative `(p0, decay)` parameter pairs, then
    decodes and asserts byte-exact recovery. **Passes.**

2.  **Bit-exact vs C reference** (`harness/tests/dred_differential.rs`,
    `laplace_p0_encode_matches_c` and `laplace_p0_decode_matches_c`).
    Adds FFI bindings for `ec_laplace_encode_p0` /
    `ec_laplace_decode_p0` in `harness/src/bindings.rs`. Encodes with
    both C and Rust, asserts byte equality; decodes Rust output with
    both sides, asserts value equality.

    -   `laplace_p0_decode_matches_c` — **passed**.
    -   `laplace_p0_encode_matches_c` — **passed** after restricting
        input params to DRED-realistic values (p0, decay ∈
        `[128, 25600]`, the `u8<<7` range with neither saturated;
        degenerate `(p0=0, decay=1)` was causing range-coder hang).

**Files.**

-   `ropus/src/celt/range_coder.rs` — 2 new methods (~25 lines each)
    on the encoder / decoder + 1 round-trip unit test.
-   `harness/src/bindings.rs` — 2 FFI declarations.
-   `harness/tests/dred_differential.rs` (new) — harness test crate
    for DRED, seeded with the 2 laplace_p0 bit-exact tests.

**Status.** Complete. Rust-only round-trip + both C-differential
tests green post-Stage-7a/7b.2.

## Sub-stage 8.2 — dred_stats

**Target.** Port `reference/dnn/dred_rdovae_stats_data.c` (656 lines of
static `opus_uint8` quantisation tables) to
`ropus/src/dnn/dred_stats.rs` as public `const` arrays. Eight arrays
referenced by `dred_encoder.c` / `dred_decoder.c` (header lists
exactly these in `dred_rdovae_stats_data.h`):

-   `dred_latent_quant_scales_q8` / `dred_latent_dead_zone_q8` /
    `dred_latent_r_q8` / `dred_latent_p0_q8` — 400 `u8` each.
-   `dred_state_quant_scales_q8` / `dred_state_dead_zone_q8` /
    `dred_state_r_q8` / `dred_state_p0_q8` — 800 `u8` each.

Pure data port, no logic. Names retained in lowercase to match the C
symbols verbatim (gated by a file-level
`#![allow(non_upper_case_globals)]`), so future DRED encoder/decoder
porting can cross-reference by symbol name. `#![allow(dead_code)]`
bridges the gap until the consuming call-sites land in 8.3+.

**Port.** Generated mechanically from the C source via a regex +
8-per-line emit pass; exhaustive Python cross-check verified every
one of the 4800 `u8` entries matches the C initialiser byte-for-byte
before the file was written. `pub mod dred_stats` wired into
`ropus/src/dnn/mod.rs` alongside the existing modules.

**Harness integration.** Added
`reference/dnn/dred_rdovae_stats_data.c` to the harness
`opus_sources` list in `harness/build.rs`. The TU only pulls
`opus_types.h` + a few `#define`s via `nnet.h`; no `kiss_fft.h`, no
float kernels, so the FIXED_POINT / DEEP_PLC incompatibility that
blocks the rest of the DNN sources does not apply here.

**Tests.**

1.  **Byte-exact vs C reference**
    (`harness/tests/dred_differential.rs`, `dred_stats_arrays_match_c`).
    Declares the eight C arrays as `extern "C" static [u8; N]` and
    asserts each matches the Rust `const` slice-for-slice; first
    divergent index surfaced on failure for debuggability. **Passes.**

2.  Pre-existing `laplace_p0_encode_matches_c` /
    `laplace_p0_decode_matches_c` — still green after the C source
    addition (no layout disturbance).

```
cargo test -p ropus-harness --test dred_differential
  -> 3 passed; 0 failed
```

**Files.**

-   `ropus/src/dnn/dred_stats.rs` (new) — 638 lines, 8 `pub const`
    arrays.
-   `ropus/src/dnn/mod.rs` — 1 new `pub mod dred_stats;` line.
-   `harness/build.rs` — 1 new source file in `opus_sources`.
-   `harness/tests/dred_differential.rs` — 1 new `extern "C"` block
    (8 arrays) + 1 new test.

**Status.** Complete. `cargo build -p ropus` clean (only the
