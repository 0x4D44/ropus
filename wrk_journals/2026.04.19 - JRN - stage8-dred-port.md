## Strategy adjustment (discovered at 8.0 pre-work)

The HLD's 8.0 step — "add `ENABLE_DRED` + DRED C sources to
`harness/build.rs`" — is **blocked**. Stage 7's team discovered that
upstream xiph forbids `FIXED_POINT + DEEP_PLC` (see
`reference/configure.ac:973`, `AC_MSG_ERROR`), and the restriction
enforces a real incompatibility: DNN code's `freq.c`/`kiss_fft` uses
different `kiss_fft_scalar` type + struct layout than fixed-point
`celt/kiss_fft.c`. Cross-TU silent memory corruption otherwise.

Every DNN-consuming DRED source (dred_encoder.c via lpcnet_enc.c, both
rdovae sources via nnet.c) trips this. Stage 7 deferred the float-mode
comparison harness to an unscheduled "Stage 7b".

**Option D adopted** (Arthur, 2026-04-19): ship the two non-DNN
sub-stages now (8.1 `ec_laplace_p0`; 8.2 `dred_stats` static tables),
pause at 8.4 RDOVAE forward pass until Stage 7b (or our own float-mode
fixture pipeline) unblocks. Clean stopping points, no rework risk.

## Sub-stage 8.1 — ec_laplace_p0 range coder primitives

**Target.** Port `reference/celt/laplace.c:136-192` into
`ropus/src/celt/range_coder.rs` as methods on `RangeEncoder` /
`RangeDecoder`. Used by DRED to code quantised latent + state values
with a symmetric Laplace distribution anchored at zero.

**No DNN surface area.** Pure integer range coding; no kiss_fft,
no fixed/float divergence. Safe to land ahead of Stage 7b.

**Port.** `ec_laplace_encode_p0` and `ec_laplace_decode_p0` added as
methods on `RangeEncoder` / `RangeDecoder` in
`ropus/src/celt/range_coder.rs`. Direct translation from C laplace.c:

-   Sign icdf: `[32768-p0, (32768-p0)/2, 0]`, coded via existing
    `encode_icdf16` / `decode_icdf16` (already in range_coder.rs).
-   Magnitude icdf: decay chain starting at `max(7, decay)`, each next
    entry `max(7-i, (prev*decay)>>15)`. Values > 7 are coded by
    repeated 7-symbol escapes.
-   Guardrail checks (C: `abs`, `IMAX`, `IMIN`) map to Rust's
    `unsigned_abs`, `.max()`, `.min()`.

**Tests.**

1.  **Rust-only round-trip** (`ropus/src/celt/range_coder.rs` tests
    module, `encode_decode_laplace_p0_roundtrip`). Encodes 13 values
    across the DRED quantiser range (0, ±1, ±2, ±7, ±8, ±15, ±30)
    under 4 representative `(p0, decay)` parameter pairs, then
    decodes and asserts byte-exact recovery. **Passes.**

2.  **Bit-exact vs C reference** (`harness/tests/dred_differential.rs`,
    `laplace_p0_encode_matches_c` and `laplace_p0_decode_matches_c`).
    Adds FFI bindings for `ec_laplace_encode_p0` /
    `ec_laplace_decode_p0` in `harness/src/bindings.rs`. Encodes with
    both C and Rust, asserts byte equality; decodes Rust output with
    both sides, asserts value equality.

    -   `laplace_p0_decode_matches_c` — **passed**.
    -   `laplace_p0_encode_matches_c` — **passed** after restricting
        input params to DRED-realistic values (p0, decay ∈
        `[128, 25600]`, the `u8<<7` range with neither saturated;
        degenerate `(p0=0, decay=1)` was causing range-coder hang).

**Files.**

## Sub-stage 8.3 — dred.rs skeleton + blob

**Target.** Land the DRED data model on the Rust side: struct
definitions, running state, and the blob-driven `init_rdovaeenc` /
`init_rdovaedec` entry points. HLD staging table row 8.3 exit
criterion: `cargo build -p ropus` clean, load blob in-memory without
panic. No forward-pass logic — that's 8.4 / 8.5.

**Blob strategy.** Stage 7b.1.5's compile-time weight blob
(`OUT_DIR/weights_blob.bin`, produced by `ropus/build.rs` +
`ropus/build/gen_weights_blob.c`) explicitly skipped DRED / OSCE with

cleanest path is to extend the same generator to also emit

grew from 6,560,576 → 8,785,408 bytes (+2.2 MB of DRED weights). No
name collisions — DRED layers use `enc_*` / `dec_*` prefixes disjoint
from pitchdnn / fargan / plc namespaces.

-   `ropus/build.rs` — added DRED `*_data.{c,h}` to

-   `ropus/build/gen_weights_blob.c` — included
    `dred_rdovae_{enc,dec}_data.c` and emitted both arrays under the

## Sub-stage 8.4 — RDOVAE encoder forward pass

**Target.** Port `reference/dnn/dred_rdovae_enc.c:dred_rdovae_encode_dframe`
to Rust as a method on the Stage 8.3 `RDOVAEEncState` skeleton. Acceptance
(HLD staging row 8.4): module-level bit-exact vs C on synthetic input
(tier 1), tier-2 SNR ≥ 60 dB fallback if float drift.

**Port approach (layer-by-layer).** The C function is a linear pipeline of
11 stages concatenated into a single 544-wide `buffer` that feeds the
final latent + state projections:

1. `enc_dense1` (tanh) → buffer[0..64]                                 — 40 → 64 dense
2. `enc_gru1_input` + `enc_gru1_recurrent` → gru1_state                — 64 → 32 GRU
3. `enc_conv_dense1` (tanh) → conv_tmp, then `enc_conv1` (tanh, dilation 1) → buffer
4. `enc_gru2_input` + `enc_gru2_recurrent` → gru2_state
5. `enc_conv_dense2` (tanh) → conv_tmp, then `enc_conv2` (tanh, dilation 2) → buffer
6-7. Repeat stages 4-5 for GRU3 / conv3 (dilation 2)
8-9. Repeat for GRU4 / conv4
10-11. Repeat for GRU5 / conv5
12. `enc_zdense` (linear) → padded_latents[32], copied down to `latents[25]`
13. `gdense1` (tanh) → state_hidden[128]; `gdense2` (linear) → padded_state[56],
    copied down to `initial_state[50]`

Each call-site slices `buffer` down to the consuming layer's `nb_inputs`
(64, 96, 160, 192, 256, …, 544) so the Rust `compute_generic_*` primitives
only see the bytes the C reference actually reads. The private C helper
`conv1_cond_init` — a one-shot mem-zeroing on first-ever call, gated by
`enc_state->initialized` — is mirrored by a small inline Rust `fn` of the
same name (no re-export; same scope as the forward pass).

**Stack constants.** `DRED_PADDED_LATENT_DIM=32`, `DRED_PADDED_STATE_DIM=56`,
`DRED_MAX_CONV_INPUTS=128`, and a compile-time-asserted
`ENC_FORWARD_BUFFER_SIZE=544` all map directly from the C headers. No ad-hoc
magic numbers in the forward pass body.

**Tier reached.** **Tier 1** — bit-exact across all 8 frames in the
differential test, latents and initial_state both, for every sample:
`SNR = +∞ dB` every frame. The expected tier-2 fallback (float drift in
the accumulation order of the dense + GRU sgemv kernels) did not materialise
because both sides reuse xiph's scalar sgemv accumulation order, which is
what ropus's `dnn::core::compute_linear` already implements verbatim. Same
outcome as Stage 6's `analysis.c` module-level port.

**Differential harness.** Stage 7b.2's sibling float-mode harness
(`harness-deep-plc/`) is the comparison point — the fixed-mode `harness/`
can't link DRED DNN C sources (`FIXED_POINT + DEEP_PLC` incompatible per
`reference/configure.ac:973`). Five new C sources added to
`harness-deep-plc/build.rs` (`dred_rdovae_enc.c`, `dred_rdovae_enc_data.c`,
`dred_rdovae_dec.c`, `dred_rdovae_dec_data.c`, `dred_rdovae_stats_data.c`)
so the `rdovaeenc_arrays` symbol and `init_rdovaeenc` are available to
link. A local shim (`harness-deep-plc/dred_enc_shim.c`) exposes
`dred_rdovae_encode_dframe` + allocator pairs via opaque `void*` handles,
so the Rust test doesn't need to replicate the `LinearLayer` /
`RDOVAEEncStruct` layouts across FFI.

## Sub-stage 8.5 — RDOVAE decoder forward pass

**Target.** Port `reference/dnn/dred_rdovae_dec.c:dred_rdovae_decode_qframe`
(and its partner `dred_rdovae_dec_init_states`) to Rust as methods on the
Stage 8.3 `RDOVAEDecState` skeleton. Acceptance (HLD staging row 8.5):
module-level bit-exact vs C on synthetic input (tier 1), tier-2 SNR ≥ 60 dB
fallback if float drift. Mirror image of 8.4's encoder port.

**Port approach (layer-by-layer).** The C decoder is a linear pipeline of
11 stages concatenated into a single 576-wide `buffer` that feeds the final
`dec_output` projection. Structurally it mirrors the encoder but swaps:

-   The per-stage "GRU → copy state to buffer" pattern for "GRU → GLU(state)
    → write GLU output to buffer". `compute_glu` is the new primitive
    compared to 8.4.
-   All five convolutions run with **dilation = 1** (`compute_generic_conv1d`),
    not the encoder's 2-5 mix of dilation 1 / 2.

Pipeline:

1. `dec_dense1` (tanh) → buffer[0..96]                       — 26 → 96 dense
2. `dec_gru1_input` + `dec_gru1_recurrent` → gru1_state[64]
3. `dec_glu1(gru1_state)` → buffer[96..160]
4. `dec_conv_dense1` (tanh) → conv_tmp[32], then `dec_conv1` (tanh) → buffer[160..192]
5. `dec_gru2_input` + `dec_gru2_recurrent` → gru2_state[64]
6. `dec_glu2(gru2_state)` → buffer[192..256]
7-8. `dec_conv_dense2` + `dec_conv2` → buffer[256..288]
9-16. Repeat stages 5-8 for GRU3/4/5 + conv3/4/5
17. `dec_output` (linear) → qframe[0..80]

Also ported: `init_states` runs `dec_hidden_init` (50 → 128, tanh) then
`dec_gru_init` (128 → 320, tanh), then splits the 320-wide output into the
five 64-wide GRU banks. Matches C `dred_rdovae_dec_init_states`.

Each GRU call-site slices `buffer` down to the consuming layer's `nb_inputs`
(96, 192, 288, 384, 480) so the Rust `compute_generic_gru` primitive only
sees the bytes the C reference actually reads.

**Stack constants.** `DEC_FORWARD_BUFFER_SIZE=576` is compile-time
asserted, paralleling the encoder's `ENC_FORWARD_BUFFER_SIZE=544` check.
`DEC_GRU_INIT_OUT_SIZE=320` is cross-checked at compile time as equalling
the sum of the five GRU state sizes, so the `OPUS_COPY` chain in
`init_states` covers exactly the outputs of `dec_gru_init`. `DEC_DENSE1_IN_SIZE`
= `DRED_LATENT_DIM + 1` = 26, mirroring the C `dec_dense1.nb_inputs`.
`conv1_cond_init` (the 8.4 helper) is reused as-is.

### Collateral LPCNet bit-exactness fixes

Three diffs in `ropus/src/dnn/lpcnet.rs` caught during 8.6 debugging.
All are narrow bit-exactness corrections against the C reference,
identical logic just promoted to double precision in the one spot the
C does the same when `FLOAT_APPROX` is undefined (which
`harness-deep-plc`'s `config.h` does not define):

-   `dct()`: changed `(sum * scale_f32)` → `((sum as f64) * scale_f64) as f32`.
    The C source is `sum * sqrt(2./NB_BANDS)`, which promotes to double
    for the multiply and truncates to float on the store. Previously a
    1-ULP drift leaked into `features[*]`.
-   `compute_frame_features` cepstrum: changed `(1e-2 + ex[i]).log10()`
    (native `f32::log10` → libm `log10f`) to an emulation of the C
    `celt_log10` macro's non-`FLOAT_APPROX` expansion:
    `0.3010299957f * (float)(1.442695040888963387 * log((double)x))`.
    Previously the Rust path gave a bit-different `log10` result on
    many inputs.
-   `celt_log10_f32` helper (used for `if_features[0]` and
    `if_features[3*i]` via `compute_frame_features`): previously called
    `log2_approx` (the `FLOAT_APPROX` path) but the C reference under
    harness-deep-plc's config takes the non-FLOAT_APPROX path and
    computes `0.3010299957f * (float)(1.442695040888963387 * log((double)x))`.
    Emulated bit-for-bit. `log2_approx` import removed from lpcnet.rs
    since it is no longer used anywhere in the file.

None of the three changes broke existing tests: `cargo test -p ropus
--lib dnn::lpcnet::` → 63 tests passed (full `dnn::` suite → 202
passed). Important caveat: none of those 63 `dnn::lpcnet::` tests
(nor the 202 total `dnn::` tests) currently diff against a C oracle,
so they serve only as regression guards — an LPCNet drift that
cancels out at the unit-test level will pass them silently. The 8.6
payload oracle is the first bit-exact differential at full float
precision.
