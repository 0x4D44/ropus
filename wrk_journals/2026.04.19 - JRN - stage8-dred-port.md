## Strategy adjustment (discovered at 8.0 pre-work)

The HLD's 8.0 step — "add `ENABLE_DRED` + DRED C sources to
`harness/build.rs`" — is **blocked**. Stage 7's team discovered that
upstream xiph forbids `FIXED_POINT + DEEP_PLC` (see
`reference/configure.ac:973`, `AC_MSG_ERROR`), and the restriction
enforces a real incompatibility: DNN code's `freq.c`/`kiss_fft` uses
different `kiss_fft_scalar` type + struct layout than fixed-point
`celt/kiss_fft.c`. Cross-TU silent memory corruption otherwise.

Every DNN-consuming DRED source (dred_encoder.c via lpcnet_enc.c, both
rdovae sources via nnet.c) trips this. Stage 7 deferred the float-mode
comparison harness to an unscheduled "Stage 7b".

**Option D adopted** (Arthur, 2026-04-19): ship the two non-DNN
sub-stages now (8.1 `ec_laplace_p0`; 8.2 `dred_stats` static tables),
pause at 8.4 RDOVAE forward pass until Stage 7b (or our own float-mode
fixture pipeline) unblocks. Clean stopping points, no rework risk.

## Sub-stage 8.1 — ec_laplace_p0 range coder primitives

**Target.** Port `reference/celt/laplace.c:136-192` into
`ropus/src/celt/range_coder.rs` as methods on `RangeEncoder` /
`RangeDecoder`. Used by DRED to code quantised latent + state values
with a symmetric Laplace distribution anchored at zero.

**No DNN surface area.** Pure integer range coding; no kiss_fft,
no fixed/float divergence. Safe to land ahead of Stage 7b.

**Port.** `ec_laplace_encode_p0` and `ec_laplace_decode_p0` added as
methods on `RangeEncoder` / `RangeDecoder` in
`ropus/src/celt/range_coder.rs`. Direct translation from C laplace.c:

-   Sign icdf: `[32768-p0, (32768-p0)/2, 0]`, coded via existing
    `encode_icdf16` / `decode_icdf16` (already in range_coder.rs).
-   Magnitude icdf: decay chain starting at `max(7, decay)`, each next
    entry `max(7-i, (prev*decay)>>15)`. Values > 7 are coded by
    repeated 7-symbol escapes.
-   Guardrail checks (C: `abs`, `IMAX`, `IMIN`) map to Rust's
    `unsigned_abs`, `.max()`, `.min()`.

**Tests.**

1.  **Rust-only round-trip** (`ropus/src/celt/range_coder.rs` tests
    module, `encode_decode_laplace_p0_roundtrip`). Encodes 13 values
    across the DRED quantiser range (0, ±1, ±2, ±7, ±8, ±15, ±30)
    under 4 representative `(p0, decay)` parameter pairs, then
    decodes and asserts byte-exact recovery. **Passes.**

2.  **Bit-exact vs C reference** (`harness/tests/dred_differential.rs`,
    `laplace_p0_encode_matches_c` and `laplace_p0_decode_matches_c`).
    Adds FFI bindings for `ec_laplace_encode_p0` /
    `ec_laplace_decode_p0` in `harness/src/bindings.rs`. Encodes with
    both C and Rust, asserts byte equality; decodes Rust output with
    both sides, asserts value equality.

    -   `laplace_p0_decode_matches_c` — **passed**.
    -   `laplace_p0_encode_matches_c` — **passed** after restricting
        input params to DRED-realistic values (p0, decay ∈
        `[128, 25600]`, the `u8<<7` range with neither saturated;
        degenerate `(p0=0, decay=1)` was causing range-coder hang).

**Files.**

## Sub-stage 8.3 — dred.rs skeleton + blob

**Target.** Land the DRED data model on the Rust side: struct
definitions, running state, and the blob-driven `init_rdovaeenc` /
`init_rdovaedec` entry points. HLD staging table row 8.3 exit
criterion: `cargo build -p ropus` clean, load blob in-memory without
panic. No forward-pass logic — that's 8.4 / 8.5.

**Blob strategy.** Stage 7b.1.5's compile-time weight blob
(`OUT_DIR/weights_blob.bin`, produced by `ropus/build.rs` +
`ropus/build/gen_weights_blob.c`) explicitly skipped DRED / OSCE with

-   `ropus/build/gen_weights_blob.c` — included

## Sub-stage 8.4 — RDOVAE encoder forward pass

**Target.** Port `reference/dnn/dred_rdovae_enc.c:dred_rdovae_encode_dframe`
to Rust as a method on the Stage 8.3 `RDOVAEEncState` skeleton. Acceptance
(HLD staging row 8.4): module-level bit-exact vs C on synthetic input
(tier 1), tier-2 SNR ≥ 60 dB fallback if float drift.

**Port approach (layer-by-layer).** The C function is a linear pipeline of
11 stages concatenated into a single 544-wide `buffer` that feeds the
final latent + state projections:

1. `enc_dense1` (tanh) → buffer[0..64]                                 — 40 → 64 dense
2. `enc_gru1_input` + `enc_gru1_recurrent` → gru1_state                — 64 → 32 GRU
3. `enc_conv_dense1` (tanh) → conv_tmp, then `enc_conv1` (tanh, dilation 1) → buffer
4. `enc_gru2_input` + `enc_gru2_recurrent` → gru2_state
5. `enc_conv_dense2` (tanh) → conv_tmp, then `enc_conv2` (tanh, dilation 2) → buffer
6-7. Repeat stages 4-5 for GRU3 / conv3 (dilation 2)
8-9. Repeat for GRU4 / conv4
10-11. Repeat for GRU5 / conv5
12. `enc_zdense` (linear) → padded_latents[32], copied down to `latents[25]`
13. `gdense1` (tanh) → state_hidden[128]; `gdense2` (linear) → padded_state[56],
    copied down to `initial_state[50]`

Each call-site slices `buffer` down to the consuming layer's `nb_inputs`
(64, 96, 160, 192, 256, …, 544) so the Rust `compute_generic_*` primitives
only see the bytes the C reference actually reads. The private C helper
`conv1_cond_init` — a one-shot mem-zeroing on first-ever call, gated by
`enc_state->initialized` — is mirrored by a small inline Rust `fn` of the
same name (no re-export; same scope as the forward pass).

**Stack constants.** `DRED_PADDED_LATENT_DIM=32`, `DRED_PADDED_STATE_DIM=56`,
`DRED_MAX_CONV_INPUTS=128`, and a compile-time-asserted
`ENC_FORWARD_BUFFER_SIZE=544` all map directly from the C headers. No ad-hoc
magic numbers in the forward pass body.

**Tier reached.** **Tier 1** — bit-exact across all 8 frames in the
differential test, latents and initial_state both, for every sample:
`SNR = +∞ dB` every frame. The expected tier-2 fallback (float drift in
the accumulation order of the dense + GRU sgemv kernels) did not materialise
because both sides reuse xiph's scalar sgemv accumulation order, which is
what ropus's `dnn::core::compute_linear` already implements verbatim. Same
outcome as Stage 6's `analysis.c` module-level port.

**Differential harness.** Stage 7b.2's sibling float-mode harness
(`harness-deep-plc/`) is the comparison point — the fixed-mode `harness/`
can't link DRED DNN C sources (`FIXED_POINT + DEEP_PLC` incompatible per
`reference/configure.ac:973`). Five new C sources added to
`harness-deep-plc/build.rs` (`dred_rdovae_enc.c`, `dred_rdovae_enc_data.c`,
`dred_rdovae_dec.c`, `dred_rdovae_dec_data.c`, `dred_rdovae_stats_data.c`)
so the `rdovaeenc_arrays` symbol and `init_rdovaeenc` are available to
link. A local shim (`harness-deep-plc/dred_enc_shim.c`) exposes
`dred_rdovae_encode_dframe` + allocator pairs via opaque `void*` handles,
so the Rust test doesn't need to replicate the `LinearLayer` /
`RDOVAEEncStruct` layouts across FFI.

## Sub-stage 8.5 — RDOVAE decoder forward pass

**Target.** Port `reference/dnn/dred_rdovae_dec.c:dred_rdovae_decode_qframe`
(and its partner `dred_rdovae_dec_init_states`) to Rust as methods on the
Stage 8.3 `RDOVAEDecState` skeleton. Acceptance (HLD staging row 8.5):
module-level bit-exact vs C on synthetic input (tier 1), tier-2 SNR ≥ 60 dB
fallback if float drift. Mirror image of 8.4's encoder port.

**Port approach (layer-by-layer).** The C decoder is a linear pipeline of
11 stages concatenated into a single 576-wide `buffer` that feeds the final
`dec_output` projection. Structurally it mirrors the encoder but swaps:

-   The per-stage "GRU → copy state to buffer" pattern for "GRU → GLU(state)
    → write GLU output to buffer". `compute_glu` is the new primitive
    compared to 8.4.
-   All five convolutions run with **dilation = 1** (`compute_generic_conv1d`),
    not the encoder's 2-5 mix of dilation 1 / 2.

Pipeline:

1. `dec_dense1` (tanh) → buffer[0..96]                       — 26 → 96 dense
2. `dec_gru1_input` + `dec_gru1_recurrent` → gru1_state[64]
3. `dec_glu1(gru1_state)` → buffer[96..160]
4. `dec_conv_dense1` (tanh) → conv_tmp[32], then `dec_conv1` (tanh) → buffer[160..192]
5. `dec_gru2_input` + `dec_gru2_recurrent` → gru2_state[64]
6. `dec_glu2(gru2_state)` → buffer[192..256]
7-8. `dec_conv_dense2` + `dec_conv2` → buffer[256..288]
9-16. Repeat stages 5-8 for GRU3/4/5 + conv3/4/5
17. `dec_output` (linear) → qframe[0..80]

Also ported: `init_states` runs `dec_hidden_init` (50 → 128, tanh) then
`dec_gru_init` (128 → 320, tanh), then splits the 320-wide output into the
five 64-wide GRU banks. Matches C `dred_rdovae_dec_init_states`.

Each GRU call-site slices `buffer` down to the consuming layer's `nb_inputs`
(96, 192, 288, 384, 480) so the Rust `compute_generic_gru` primitive only
sees the bytes the C reference actually reads.

**Stack constants.** `DEC_FORWARD_BUFFER_SIZE=576` is compile-time
asserted, paralleling the encoder's `ENC_FORWARD_BUFFER_SIZE=544` check.
`DEC_GRU_INIT_OUT_SIZE=320` is cross-checked at compile time as equalling
the sum of the five GRU state sizes, so the `OPUS_COPY` chain in
`init_states` covers exactly the outputs of `dec_gru_init`. `DEC_DENSE1_IN_SIZE`
= `DRED_LATENT_DIM + 1` = 26, mirroring the C `dec_dense1.nb_inputs`.
`conv1_cond_init` (the 8.4 helper) is reused as-is.

### Collateral LPCNet bit-exactness fixes

Three diffs in `ropus/src/dnn/lpcnet.rs` caught during 8.6 debugging.
All are narrow bit-exactness corrections against the C reference,
identical logic just promoted to double precision in the one spot the
C does the same when `FLOAT_APPROX` is undefined (which
`harness-deep-plc`'s `config.h` does not define):

-   `dct()`: changed `(sum * scale_f32)` → `((sum as f64) * scale_f64) as f32`.
    The C source is `sum * sqrt(2./NB_BANDS)`, which promotes to double
    for the multiply and truncates to float on the store. Previously a
    1-ULP drift leaked into `features[*]`.
-   `compute_frame_features` cepstrum: changed `(1e-2 + ex[i]).log10()`
    (native `f32::log10` → libm `log10f`) to an emulation of the C
    `celt_log10` macro's non-`FLOAT_APPROX` expansion:
    `0.3010299957f * (float)(1.442695040888963387 * log((double)x))`.
    Previously the Rust path gave a bit-different `log10` result on
    many inputs.
-   `celt_log10_f32` helper (used for `if_features[0]` and
    `if_features[3*i]` via `compute_frame_features`): previously called
    `log2_approx` (the `FLOAT_APPROX` path) but the C reference under
    harness-deep-plc's config takes the non-FLOAT_APPROX path and
    computes `0.3010299957f * (float)(1.442695040888963387 * log((double)x))`.
    Emulated bit-for-bit. `log2_approx` import removed from lpcnet.rs
    since it is no longer used anywhere in the file.

### API-translation notes for `OpusDREDDecoder`

The C API has four allocator functions (`_get_size`, `_init`, `_create`,
`_destroy`) because C relies on the caller to choose stack vs heap
placement. Rust owns lifetime via the type system, so we collapse them:

-   `OpusDREDDecoder::new()` builds a fresh handle, auto-loads the
    crate-embedded weight blob if present (mirrors the Stage 7b.1.5
    pattern `OpusDecoder::new` already adopted), and sets `loaded` on
    success. Returns `Self` directly — no separate error return
    because the only failure path in C (`OPUS_ALLOC_FAIL`) is
    impossible in Rust when `Self` is returned by value.
-   `OpusDREDDecoder::load_model(&[u8])` is the public counterpart of
    C `dred_decoder_load_model`, exposed for call-sites that want
    explicit blob-driven init. Returns a raw `OPUS_OK` / `OPUS_BAD_ARG`
    int, matching the C signature for easy pass-through in a future
    CTL dispatcher (Stage 7 territory).
-   `OpusDREDDecoder::reset()` has no direct C counterpart — C expects
    callers to `destroy` + `create` for that. A thin reset primitive
    is cheap in Rust and matches the Stage 7 `OpusDecoder::reset`
    convention already in the codebase.
-   `OpusDred` already exists from 8.3 with `Default::default()`, so no
    separate `_alloc`/`_free`/`_get_size` are needed — callers just
    own the struct.
-   `OpusDREDDecoder::parse` drops C's `defer_processing` and
    `dred_end` out-params. The brief explicitly puts `process` out of
    the parse call chain ("feed through `decode_qframe` from 8.5"),
    and `dred_end` (C: `IMAX(0, -dred->dred_offset * Fs / 400)`) is
    trivially derivable from the returned `OpusDred::dred_offset` — no
    functional loss.

Magic-number + arch fields from C are omitted: Rust's type system
replaces the magic-number "have I been initialised?" guard, and we
have no platform dispatch in ropus (per the CLAUDE.md coding rules).

### Interaction with 8.6

8.7 inherits 8.6's "encoder bytes drift upstream of
`encode_silk_frame`" caveat **but is unaffected by it**, because the
8.7 test never routes through `dred_compute_latents` /
`dred_process_frame`. Inputs are hand-synthesised directly into
`state_buffer` + `latents_buffer` via the new shim entry points, so
the LPCNet path never runs. The test's symmetric `c_bytes ==
r_bytes` assertion confirms this: on direct-poke inputs, the C and
Rust encoders produce byte-identical output. This narrows the range
of candidates for 8.6's drift — the range coder, quantiser tables,
and payload header layout all agree bit-exactly between C and Rust.

**What 8.7 does NOT close.** The 8.4/8.5 "compensating-error blind
spot" is about correlated `linear_init` bugs on the RDOVAE **float
forward pass** (shared weights, shared kernel conventions). 8.7
exercises the range-coded **integer path** (quantise → Laplace →
dequantise) and never touches RDOVAE weights, so it cannot
validate the float forward pass. That caveat remains open as
documented in 8.4/8.5 and is addressed separately by the layered
diagnostics in 8.6's payload diff test (resample → input_buffer →
LPCNet features → latents/state → payload), once the upstream
LPCNet drift is resolved.

The moment 8.6's LPCNet drift gets fixed and
`dred_encode_silk_frame_bytes_match_c_reference` is un-ignored, the
full encoder-side pipeline will be tier-1 validated end-to-end
(LPCNet features → RDOVAE → payload bytes). 8.7's independent
validation of the post-RDOVAE stages makes that bisect trivial.

## Sub-stage 8.8 — Wire DREDEnc into OpusEncoder

**Goal.** Drive the Rust `DREDEnc` from inside `OpusEncoder::encode`
so that packets emitted by the full Rust pipeline carry a DRED
extension the C reference can parse — and vice versa — without the
harness test having to orchestrate `compute_latents` /
`encode_silk_frame` by hand. Matches the exit criterion in the
8.x HLD: "packets C can parse and vice versa" (format-level
cross-compat; 8.6's upstream LPCNet drift still blocks byte
exactness, so this is not a tier-1 goal yet).

**What the previous agent had landed (working but gated off).**

- `ropus/src/opus/encoder.rs` fields: `dred_encoder: Option<Box<DREDEnc>>`,
  `dred_duration`, `dred_q0`, `dred_d_q`, `dred_qmax`,
  `dred_target_chunks`, `activity_mem: Vec<u8>` (416 bytes),
  `first_frame_flag: bool` (mirrors C's stack-local
  `first_frame` at `opus_encoder.c:1777`). Construction initialises
  the pair to `None` / `false` / zeroed; reset zeroes the ring
  buffer and calls `DREDEnc::reset()` when present.
- CTL surface: `set_dred_duration(i32) -> i32`, `get_dred_duration()`,
  `load_dnn_blob(&[u8]) -> i32`. First non-zero `set_dred_duration`
  lazily allocates `DREDEnc::new(fs, channels)` which auto-loads
  the embedded weight blob.
- Multi-frame loop flips `self.first_frame_flag = (i == 0)` around
  each `encode_frame_native` call so the nested emit branch runs
  only on the head sub-frame.
- Two skeleton blocks in `encode_frame_native` — one before SILK
  (the `compute_latents` call site) and one after the packet body
  (the extension emission site) — were fully written out but wrapped
  in `if false && ...` so no runtime effect. Build was clean because
  Rust evaluates the short-circuit statically and drops the dead
  branch.

**What 8.8 actually needed (the final gap).**

1. `ropus/src/opus/encoder.rs:2585-2623` — enable the
   `compute_latents` block. When `dred_duration > 0 && loaded`:
     - Convert the post-HP/DC-reject `pcm_buf[total_buffer*ch..]`
       slice from `i16` to `f32 / 32768.0` (matches the convention
       the 8.6 payload-diff harness already validated).
     - Call `dred.compute_latents(&pcm_f32, frame_size, total_buffer)`.
     - Shift `activity_mem` right by `frame_size * 400 / fs`
       entries, then stamp the newly-free head with the current
       `activity` value (cast as `u8`; `VAD_NO_DECISION` becomes
       `0xFF` which `dred_voice_active` correctly rejects).
     - `else` branch: `latents_buffer_fill = 0`, `activity_mem.fill(0)`.
     Matches C `opus_encoder.c:2027-2041` 1:1.

2. `ropus/src/opus/encoder.rs:2858-2880` — after SILK has run and
   overwritten `activity = VAD_NO_DECISION` with a real flag, also
   back-patch the `activity_mem` entries we just stored. C does
   this at `opus_encoder.c:2235-2240`. Missing this step was the
   root cause of every emitted frame reporting `nb_latents=0`: for
   a pure-tone VOIP input in `MODE_SILK_ONLY`, the pre-SILK
   `activity` is `VAD_NO_DECISION` → `0xFF` in the mem → never
   voiced → `encode_silk_frame` walks `latent_offset` past
   `latents_buffer_fill` → `chunk_count_cap` is negative → no
   chunks are emitted. After SILK we know the frame was voiced,
   and `dred_voice_active` only counts `== 1`, so the patch is
   necessary for anything to come out.

3. `ropus/src/opus/encoder.rs:3296-3366` — drop the `if false &&`
   on the extension emission branch so it runs on every first
   frame with DRED enabled and loaded.

The previous agent's emission block itself was correct. The only
edits beyond removing the `false &&` guards were the `compute_latents`
call site and the SILK back-patch.

**Extension emission mechanism.** The emitted bytes land in the
packet exactly the way C emits them:
`opus_packet_pad_impl(data, ret, orig_max_data_bytes, pad_flag, &[ext])`
splices a single `OpusExtensionData { id: 126, frame: 0, data:
['D', version, payload...], len: N }` into the padding region of
the code-3 frame. The `126` is the standard DRED extension ID
(`DRED_EXTENSION_ID`); the 2-byte `'D' + DRED_EXPERIMENTAL_VERSION`
prefix is the experimental-container marker that both sides'
`find_payload` walk-and-match on. `pad_flag = !use_vbr` matches C
— for VBR the packet grows by exactly the extension's sliver; for
CBR the splice happens inside the already-padded tail.

**Gating (matches C `opus_encoder.c:2603-2642` 1:1).**

- `first_frame` (the `i == 0` head-of-packet flag).
- `dred_duration > 0` (user-requested DRED).
- `dred_encoder.loaded` (weights present; embedded blob auto-loaded
  on `DREDEnc::new`).
- `dred_chunks >= 1` after `imin((dred_duration+5)/4, DRED_NUM_REDUNDANCY_FRAMES/2)`
  and the VBR target-chunk cap.
- `dred_bytes_left >= DRED_MIN_BYTES + DRED_EXPERIMENTAL_BYTES`
  after accounting for the 3-byte code-3 header / length prefix /
  extension ID byte plus the +1-per-255 multi-byte padding-length
  overhead.
- `encode_silk_frame` returns `> 0` (inner gating: at least one
  voiced 40 ms chunk had budget to encode).

**Caveats observed but not fixed.**

- 8.6's LPCNet upstream drift is still in place; the encoder
  emits valid DRED extensions the C parser accepts, but the
  byte-exact payload test
  (`dred_encode_silk_frame_bytes_match_c_reference`) stays
  `#[ignore]`. Closing that is 8.6's job, not 8.8's.
- `compute_dred_bitrate` is not ported — we hard-code
  `q0=9, d_q=3, qmax=15, target_chunks=(duration+5)/4` as the
  experimental-fallback equivalent. Matches the `qmax=15` ceiling
  C uses and works for every tested frame rate. A future 8.x sub-
  stage can port the rate-adaptive tuning once byte-exactness is
  in reach.
- The back-patch after SILK only runs when the original
  `activity == VAD_NO_DECISION` (i.e. SILK/HYBRID modes). In
  CELT-only the energy-based activity is already final at the
  compute site, so no back-patch is needed — C does the same.
- `first_frame_flag` is a simple `i == 0` in the multi-frame loop
  rather than C's `(i == 0) || (i == dtx_count)`. The difference
  only shows up on multi-frame packets where the head frame is
  DTX'd; DRED would then ride on the first non-DTX sub-frame in C
  but on index 0 in Rust. For single-frame encodes (what 8.8 is
  validated against) the behaviour is identical. Flag deferred to
  a later sub-stage.

### Known gaps / TODOs

- **CELT-budget steal (encoder.rs near L3055)** — C shrinks CELT budget for DRED at tight bitrates; Rust doesn't. Integration tests pass at 32 kbps; would fail at <= 20 kbps. TODO flagged inline.
- **DTX multi-frame first-frame shift (encoder.rs near L2358)** — C shifts DRED emission past DTXed lead frames; Rust doesn't. Single-frame packets (20 ms) don't hit this. TODO flagged inline.
- **Hardcoded quantiser params** (`dred_q0=9, dred_d_q=3, dred_qmax=15`) — placeholders instead of C's `compute_dred_bitrate` adaptive tuning. Tests pass at 32 kbps with these; a differential test at lower rates might diverge. Worth porting if Stage 8.10 audit flags rate-dependent behavior.
- **CTL-dispatch not wired** — `set_dred_duration` / `get_dred_duration` / `load_dnn_blob` are Rust-level setters only. CTL routing (`OPUS_SET_DRED_DURATION` etc.) sits behind Stage 7's `capi/src/ctl.rs` per the HLD coordination section. Flag for Stage 7 pickup.

## Sub-stage 8.9 — FARGAN seam stub

**Goal.** Land the public decode entry points for DRED-assisted
packet reconstruction, but stubbed to return `OPUS_UNIMPLEMENTED`
until Stage 7 wires FARGAN synthesis. Compile-only per HLD row 8.9
("the real call path is exercised in a Stage 7 follow-up commit").

**C signatures → Rust equivalents** (C at
`reference/src/opus_decoder.c:1609-1691`, Rust at
`ropus/src/opus/dred.rs`):

| C                                                                                                        | Rust                                                                                                          |
|----------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------|
| `int opus_decoder_dred_decode(OpusDecoder *st, const OpusDRED *dred, opus_int32 dred_offset, opus_int16 *pcm, opus_int32 frame_size)` | `OpusDREDDecoder::decode(&self, _dred: &OpusDred, _dred_offset: i32, _pcm: &mut [i16], _frame_size: i32) -> Result<i32, i32>` |
| `int opus_decoder_dred_decode24(OpusDecoder *st, const OpusDRED *dred, opus_int32 dred_offset, opus_int32 *pcm, opus_int32 frame_size)` | `OpusDREDDecoder::decode24(&self, _dred: &OpusDred, _dred_offset: i32, _pcm: &mut [i32], _frame_size: i32) -> Result<i32, i32>` |
| `int opus_decoder_dred_decode_float(OpusDecoder *st, const OpusDRED *dred, opus_int32 dred_offset, float *pcm, opus_int32 frame_size)` | `OpusDREDDecoder::decode_float(&self, _dred: &OpusDred, _dred_offset: i32, _pcm: &mut [f32], _frame_size: i32) -> Result<i32, i32>` |

**Return code.** All three methods return
`Err(OPUS_UNIMPLEMENTED)` (`-5`, defined at
`ropus/src/opus/decoder.rs:40`). Matches C's `#else` branch when
`ENABLE_DRED` is undefined — same code C emits today for a decoder
built without DRED support. The existing `parse`/`process`
unloaded-decoder paths already return `OPUS_UNIMPLEMENTED`, so the
new stubs reuse the established convention.

**Placement rationale.** C's `opus_decoder_dred_decode*` lives on
`OpusDecoder` because the audio render needs channel count,
preemph state, and the resampler — all classical-decoder
bookkeeping. For the stub we placed the methods on
`OpusDREDDecoder` instead so the 8.9 change stays scoped to
`opus/dred.rs` and avoids touching `opus/decoder.rs` (which sits
inside Stage 7's FARGAN reconstruction footprint). When Stage 7
lands, the joint follow-up commit is free to migrate the methods
onto `OpusDecoder` alongside the real body — the public crate
surface for DRED-assisted decode is still "one method call away
from a parsed `OpusDred`" either way.

**Stage 7 seam.** When FARGAN lands, the joint follow-up commit
replaces the `Err(OPUS_UNIMPLEMENTED)` return in each of the three
methods with the actual inference call path. C's body shows what
the replacement looks like: validate `frame_size > 0`, allocate a
`float` scratch of `frame_size * channels`, route through
`opus_decode_native(..., dred, dred_offset)` (which internally
drives FARGAN off `dred.fec_features`), then quantise back to the
output sample type (`RES2INT16` / `RES2INT24` for the integer
variants; pass-through for `decode_float`). The three unit tests
added here (`decode_returns_unimplemented`,
`decode24_returns_unimplemented`, `decode_float_returns_unimplemented`)
will fail loudly the moment the stub bodies are replaced, forcing
the Stage 7 author to update the tests alongside the
implementation — a deliberate trip-wire so the seam can't be
silently removed.
