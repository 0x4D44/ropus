## Context gathered

`tools/fuzz_run.sh --check-crashes` already builds selected manifest-discovered
targets and replays committed `*.bin` crash files with `-runs=0`. That is close
to the requested shell behavior, so the right move is to add `--sanity` as the
explicit no-campaign name and preserve `--check-crashes` as an alias.

`full-test` has no fuzz stage yet. Banner, JSON, and HTML are explicit structs,
so the new fuzz result needs to be threaded through those places.

## Implementation notes

Added `tools/fuzz_run.sh --sanity` and kept `--check-crashes` as an alias. The
sanity path validates undeclared committed crash `.bin` files, requires
cargo-fuzz/nightly only when it will actually build/replay, builds selected
targets, replays explicit committed crash files with `-runs=0`, prints compact
per-target summary lines, and exits before the campaign loop. Empty crash
directories now skip cleanly.

Added `full-test/src/fuzz.rs` for policy selection, quick inventory, bounded
shell command construction, captured output, and JSON/HTML report data. Default
full-test leaves fuzz as `not_requested`; quick release-preflight does
manifest/crash inventory only; non-quick release-preflight runs
`timeout 300 bash tools/fuzz_run.sh --sanity`.

Emotional temperature: calmer after the first test pass. The only frustration
is that the newly surfaced `fuzz_decode` crash corpus still reproduces crashes,
but that is exactly the kind of fact this gate is supposed to make impossible
to miss during release preflight.
