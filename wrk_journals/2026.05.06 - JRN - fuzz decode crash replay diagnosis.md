# fuzz decode crash replay diagnosis journal

## 2026-05-06

Started with Arthur's framing: the new uncommitted fuzz sanity gate reports four committed `fuzz_decode` crash files still crashing. I treated this as a replay/oracle triage problem, not a license to hide failures. I read the repo instructions, `tools/fuzz_run.sh`, `tests/fuzz/fuzz_targets/fuzz_decode.rs`, and the shared oracle code before running any replay commands.

First emotional temperature: cautious and a little suspicious of the wrapper because the task mentioned a "new `--sanity` wrapper". The target has substantial differential logic, so real crashes were plausible, but the wrapper also suppresses replay output.

Ran:

```bash
timeout 300 bash tools/fuzz_run.sh --sanity --target fuzz_decode
```

Observed the reported failure: all four crash files marked as still crashing, with build passing.

Then replayed a file manually using the same argument ordering as the wrapper. This was the key observation: libFuzzer printed `Not a directory: tests/fuzz/crashes/fuzz_decode/crash_...bin; exiting`. That means the target was not actually proving a decoder crash in that path. Frustration level went up a notch because stderr suppression in the gate hid the most important clue.

Hypothesis: the sanity wrapper passes fixed-input files after the libFuzzer `--` separator, causing cargo-fuzz/libFuzzer to treat the crash file as a corpus directory rather than as a fixed input.

Tested the hypothesis with the fixed input before `--`:

```bash
timeout 120 cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_decode \
  tests/fuzz/crashes/fuzz_decode/crash_48502ce358a93876.bin -- -runs=0
```

That replay executed the target code on the file and exited successfully. I repeated the corrected replay across all four crash files. All passed with no panic, assertion failure, sanitizer crash, timeout, or non-zero exit.

Current conclusion: the four committed `fuzz_decode` files are stale or resolved historical crash inputs under the current target/oracle semantics. The immediate red gate is caused by the new sanity wrapper's replay command ordering, not by unresolved decoder bugs.

Gut check: this should not turn into a policy of deleting or ignoring crash files to get green. The right next slice is to fix replay correctness and make the gate show infrastructure errors. Only after that should Arthur decide how to represent resolved historic crash files: promote to corpus seeds, keep as "must now pass" crash regressions, or add explicit metadata for expected-failing cases.

No source or corpus files were modified during triage. I only wrote the requested triage document and this journal.
