# Journal: Release Preflight And Oracle Matrix

Date: 2026-05-06

Task: Workstream C design for release preflight and oracle matrix.

## Entry

Read the workspace instructions, `full-test/src`, the three requested build
scripts, and the testing plan/review docs. I kept this round design-only and did
not edit source code.

The current code is already partway there: IETF vector provisioning is now
structured and fail-level in `full-test`. The remaining gap is that `reference/`
and DNN/DRED weights can still disappear behind build-script warnings or empty
blob behavior while release language sounds stronger than what actually ran.

Design choice: put release preflight in `full-test` behind an explicit
`--release-preflight` flag. That keeps fresh-clone ergonomics intact while
giving release validation a stricter contract. I also designed the oracle matrix
as documentation first, with controlled oracle classes and preflight asset keys.

Emotional temperature: calm, but slightly wary of scope creep. The bad version
of this task would turn into another giant validation runner. The useful version
is much smaller: name the claims, name the assets, and fail only when a release
profile tries to claim something it cannot prove.

Blockers: none.

Next option after sign-off: implement `full-test/src/preflight.rs`, wire it into
HTML/JSON/banner classification, then add `wrk_docs/testing-oracle-matrix.md`.

## Follow-up Revision

Arthur reviewed the HLD and pointed out a real design gap: I had described
profile-like statuses without defining any first profile or CLI semantics. That
would leave implementation guessing, which is exactly the thing this HLD is
supposed to prevent.

Revised the HLD in place:

- C1 now has one concrete `--release-preflight` core profile.
- C1 required assets are only fixed-point C reference, upstream conformance
  sources, and IETF vectors.
- DNN base weights, DRED RDOVAE weights, and float-mode deep PLC assets are
  report-only in C1.
- Status values are now machine-actionable:
  `present_required`, `provisioned_required`, `missing_required`,
  `present_optional`, and `missing_optional`.
- IETF absence stays aligned with the existing synthetic Stage 2 failure path,
  so preflight should not double-count it.
- C2 is explicitly future work for neural/DRED profiles or matrix automation.

Emotional temperature: a little annoyed at myself for leaving the profile
contract fuzzy, but the correction is straightforward and makes the design much
more implementable.
