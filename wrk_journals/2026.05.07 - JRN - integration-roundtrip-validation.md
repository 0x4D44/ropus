### Second debug agent — pinned the bug, surgical 4-line fix
Bisect via paired Rust + C `if (lfe)` instrumentation. First-divergent
stage was *pre*-compute_vbr — `tell=422` (Rust) vs `548` (C),
`tot_boost=0` vs `384`, `offsets[0]=0` vs `384`.

Root cause: C `celt_encoder.c:2363-2365` unconditionally injects band-0
boost for LFE substreams between `init_caps` and the dynalloc encoding
loop:
```c
if (st->lfe) offsets[0] = IMIN(8, effectiveBytes/3);
```
This is the *only* mechanism that gives LFE its band-0 allocation,
because `dynalloc_analysis` early-returns on `lfe!=0`. Rust was missing
the line entirely. With `effective_bytes=24`, C runs the dynalloc loop
8 successful iterations + terminator (~126 frac-bits of `tell` plus
`tot_boost=384`); Rust skipped the loop, under-allocating by ~16 bits.

Fix at `ropus/src/celt/encoder.rs:2893-2898` (commit `f354712`):
```rust
if st.lfe != 0 {
    offsets[0] = imin(8, effective_bytes / 3);
}
```

### Real find — 5.1 multistream LFE byte divergence
- **Cell**: 6-channel mapping_family=1, 4 streams, 2 coupled (L+R, Ls+Rs)
  + 2 mono (C, LFE).

  CTL fan-out for LFE differs by ~16 bits, suggesting
  `OpusMSEncoder::set_bitrate` LFE bit-budget split.

  plumbing-fix area, validated only against 2-stream dual-mono. 5.1 was
  uncovered. We just closed that gap and immediately found the bug it was
  hiding.
- **Determinism**: frame 0, no state warm-up, signal-dependent. Easy
  bisect target.
