configs encoded byte-identical between C and Rust. No fix required.

parity-aligning style only, not a correctness fix.

### DRED LPCNet feature drift root cause
**Verdict: HIGH-confidence root cause located. Two surgical fixes.**

First-divergent stage: final radix-5 butterfly of the 320-point FFT
inside `forward_transform` → propagates into `Ex[]` band-energy sums
→ surfaces as ~1e-6 cepstrum drift, amplified to ~3.25e-5 in feats
20-21 by a separate f32-vs-f64 `pow` divergence.

**Fix 1** — `ropus/src/dnn/lpcnet.rs:432-447` (`kf_bfly5` grouping):
- C reference at `reference/celt/kiss_fft.c:289-302` groups non-`s0`
  terms first: `s0 + (a + b)`.
- Rust currently uses left-to-right: `(s0 + a) + b`.
- `-ffp-contract=off` rules out FMA. Pure addition associativity
  difference. ULP-class drift on each of 320 FFT bins is exactly the
  right magnitude.
- Affects all callers of `opus_fft_320` (DRED, FARGAN, OSCE, PLC) —
  fix improves parity universally; no regression risk.

**Fix 2** — `ropus/src/dnn/lpcnet.rs:814` (`lpc_from_cepstrum` powf):
- C reference at `reference/dnn/freq.c:313` uses double `pow(10.f, Ex[i])`.
- Rust uses `f32::powf`.
- Switch to `10.0_f64.powf(ex[i] as f64) as f32`. This explains the
  larger drift in feats 20-21 (which directly read `lpc[]`).

Locked test bounds at `harness-deep-plc/tests/dred_lpcnet_feature_drift.rs:33-37`
will need re-locking after the fix; new values to be measured.

Out-of-scope finds:
- `compute_burg_cepstrum` (`lpcnet.rs:983`) uses `0.995f32.powi(i+1)`
  vs C `pow(.995, i+1)`. Affects classical PLC feature parity only;
  not on DRED path. Track for future.

### Second debug agent — pinned the bug, surgical 4-line fix
Bisect via paired Rust + C `if (lfe)` instrumentation. First-divergent
stage was *pre*-compute_vbr — `tell=422` (Rust) vs `548` (C),
`tot_boost=0` vs `384`, `offsets[0]=0` vs `384`.

Root cause: C `celt_encoder.c:2363-2365` unconditionally injects band-0
boost for LFE substreams between `init_caps` and the dynalloc encoding
loop:
```c
if (st->lfe) offsets[0] = IMIN(8, effectiveBytes/3);
```
This is the *only* mechanism that gives LFE its band-0 allocation,
because `dynalloc_analysis` early-returns on `lfe!=0`. Rust was missing
the line entirely. With `effective_bytes=24`, C runs the dynalloc loop
8 successful iterations + terminator (~126 frac-bits of `tell` plus
`tot_boost=384`); Rust skipped the loop, under-allocating by ~16 bits.

Fix at `ropus/src/celt/encoder.rs:2893-2898` (commit `f354712`):
```rust
if st.lfe != 0 {
    offsets[0] = imin(8, effective_bytes / 3);
}
```
