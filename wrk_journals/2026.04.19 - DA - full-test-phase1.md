# DA: full-test Phase 1

Scope: `full-test/` crate (6 files, ~1100 LOC incl. tests). Deltas: workspace member added, `.gitignore` rule, `tests/results/.gitkeep`. Uncommitted.

## Top 3 challenges (actionable — supervisor should decide)

1. **`clap` is a dependency but not used.** `full-test/Cargo.toml:21` pulls in `clap 4.6 + derive`, yet the only `clap` reference in the crate is that line. The CLI is hand-rolled in `cli.rs`. Either the impl agent pre-staged it for a Phase 2/3/4 use that never materialised in Phase 1, or it was pulled in "just in case." Supervisor: strip it before commit, or explicitly document which later phase needs it. Shipping a 100+ transitive-dep crate as dead weight in the lockfile is exactly the scope creep the HLD's "Minimal surface area" line warns against.

2. **`workspace_version()` primary lookup is dead code.** `setup.rs:69` reads `json["workspace_metadata"]["package"]["version"]`. But `cargo metadata --no-deps --format-version 1` emits the key as **`metadata`**, not `workspace_metadata` (verified: top-level keys are `packages, workspace_members, workspace_default_members, resolve, target_directory, build_directory, version, workspace_root, metadata`). The first branch will always miss; the `ropus` fallback saves it but the "workspace package" path was never actually exercised. Either fix the key or delete the branch — don't ship code that claims to handle a case it can't.

3. **Clippy fails on clean main (28+ `missing_safety_doc` errors).** HLD success criterion #1: "`cargo run --release --bin full-test` emits a PASS report on a clean master." Today that's impossible: pre-existing `ropus/` unsafe-fn docs miss `# Safety` sections, clippy fires 28+ errors, Stage 1 goes red. My position: **clean the clippy debt first, then commit Phase 1.** Shipping a runner that red-flags on an untouched tree conditions Arthur to ignore the FAIL banner — exactly the "cry wolf" failure mode that kills validation tools. If cleanup is out of scope, the HLD must be amended to say "Phase 1 documents the pre-existing clippy debt; clean-up is a follow-on." Don't ship ambiguity.

## Other pokes

4. Stage 1 claims "< 1 sec" stage 0 — fine (cargo metadata cached hot ~40ms), but on a cold build-script-recompile a `cargo metadata` can hit ~500ms-1s; "<1 sec" will occasionally fail. Not worth fixing, worth noting as observed-upper-bound.
5. `nice(10)` return value is dropped (`setup.rs:123`). `nice(3)` returns the new priority on success *or* -1 on error with errno. The comment says "always succeeds for positive increments" — not strictly true (EPERM if RLIMIT_NICE clamps). Harmless in practice, but the SAFETY comment is wrong.
6. Child-process priority inheritance: Windows inherits by default (OK). On Unix, `nice(10)` changes the current process; `fork`+`exec` children inherit — OK too. The impl didn't **prove** either; it trusted documentation. Acceptable for a test runner but zero test coverage of "child cargo actually saw below-normal."
7. `cargo fmt --check` exit code: returns 1 on formatting issues *and* on IO errors *and* on "rustfmt not installed." The runner surfaces all three as undifferentiated FAIL. Fine for a local dev tool, but the "issues" list will be empty on missing-rustfmt and confusing.
8. CLI: `--skip-quality=true` form not supported — treated as unknown arg (error). Bundled flags not supported. `-h` works. Positional-after-flag is rejected. Tests cover this. Fine for internal use.
9. 1 MiB stderr cap (`issues.rs:19`): clippy on a 36k-line workspace with 28+ errors *already* emits close to this. The truncation marker works, char-boundary slicing is correct — but the cap is tight for this very workspace. Bump to 4 MiB, memory footprint is trivial.
10. `.gitignore` rule `tests/results/*.html` (diff `+/tests/results/*.html`) excludes future JSON sidecars, LCOV summaries, or the cargo llvm-cov JSON that Phase 2 will emit. `tests/results/*` with `!.gitkeep` is the future-proof form. Phase 2 will revisit this within days — fix now.
11. `git rev-parse --abbrev-ref HEAD` returns the literal string `"HEAD"` on a detached checkout. Runner stores it verbatim. Minor cosmetic concern in the report header; not fatal.
12. No integration test that actually **invokes** `cargo fmt` / `cargo clippy`. All tests stub or skip. So Stage 1's happy path is unexercised in CI. Phase 5's smoke-test plan covers this — just confirm the plan isn't dropped.
13. Report envelope is serialised with `serde_json::to_string_pretty` then `println!` — fine, but if later phases emit huge JSON, `println!` line buffering on Windows can truncate. Switch to `BufWriter` on `stdout`.

## Defensible choices (impl got these right)

- Hand-rolled CLI over clap (given clap ends up unused anyway — the call is right, the unused dep is wrong).
- Regex unification of `error[…]:`, `error:`, `warning:` with leading-whitespace tolerance — matches cargo's actual output shape.
- UTF-8 char-boundary truncation in `cap_stderr`.
- `Outcome::skipped()` treats skipped stages as green for exit-code purposes (matches HLD `--skip-*` semantics).
- Priority class setup happens **after** CLI parsing — right call; bad flag on a CI runner shouldn't mutate scheduler class.
- Dropping the JSON envelope to stdout as a Phase 1 artefact (not HTML) — pragmatic; the HLD reserves HTML for Phase 4.
