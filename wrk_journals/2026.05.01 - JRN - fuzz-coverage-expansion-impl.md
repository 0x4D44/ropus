## Decision log

### 2026-05-01 — Stream A first agent: NaN/Inf differential finding

First implementation agent completed all 6 targets but the smoke run on
`fuzz_encode` produced a real differential divergence within seconds:
`OpusEncoder::encode_float` sanitises NaN inputs differently from C
`opus_encode_float` (Rust's `float2int16` returns 0 for NaN; C returns
-32768). This is a legitimate codec-level bug, but **out of scope** for
Stream A.

**Decision:** in the four f32-PCM-input targets (`fuzz_encode`,
`fuzz_encode_safety`, `fuzz_roundtrip`, `fuzz_roundtrip_safety`),
sanitise NaN/Inf to 0.0 immediately after `f32::from_le_bytes`. The
fuzzer's job is to find codec divergences; float-conversion semantics
are a separate PR.

**Open finding** (track for later):
- Repro: `tests/fuzz/findings/nan-encode-float/crash-54acd282f7bd446d92811b719f183d6eea294f84`
- Root cause: `ropus/src/types.rs::float2int16` does not match C reference's NaN handling.
- Severity: low (production callers don't pass NaN through `encode_float`).
- Fix: separate PR against `ropus/src/types.rs`. Probably one line — clamp NaN to -32768 to match `_mm_cvttps_epi32`-style C semantics.

The crash artifact has been moved from `tests/fuzz/artifacts/fuzz_encode/`
to `tests/fuzz/findings/nan-encode-float/` so subsequent fuzz runs don't
trip over it as a regression seed.

### 2026-05-01 — Stream C: complete

Three new targets land:

- `fuzz_dnn_blob` — pure safety. Calls `OpusDecoder::set_dnn_blob(data)`
  with arbitrary bytes, then a no-input PLC step. Cov 497 / 933 features
  in 30 s — saturates fast (the parser surface is small) but covers it
  fully.
- `fuzz_repacketizer_seq` — replaces saturated `fuzz_repacketizer`.
  Reads N (1..=8) packets via `Arbitrary`, calls `cat()` for each, then
  exercises `out()` and `out_range()`. Cov 689 / 2685 features in 30 s.
  Adds `opus_repacketizer_out_range` and `opus_repacketizer_get_nb_frames`
  to `c_reference.rs`.
- `fuzz_multistream` — biggest. Combines encode / decode / roundtrip
  via an op byte. Channels capped 1..=8. Mapping families {0, 1, 2, 255}.
  Setter shuffle (gap 13) applied symmetrically to BOTH Rust and C
  encoders before differential pass. SILK-skip PCM carve-out.

**Reviewer feedback applied** (3 should-fix from std reviewer + 4
substantive from devil's advocate):

- C-side encoder construction: switched from
  `opus_multistream_encoder_create` to
  `opus_multistream_surround_encoder_create`. Mapping family 1/2
  (Vorbis surround / ambisonics) now takes the same preprocessing
  path on both sides. Mapping byte sequence asserted equal Rust vs C
  — divergence here would itself be a finding.
- Setter-shuffle bitrate-byte extraction typo fixed
  (`[chunk[1], chunk[1]]` → `[chunk[0], chunk[1]]`).
- Setter shuffle moved from "throwaway Rust encoder before differential"
  to "applied symmetrically to both encoders within the differential pass".
  Worst-of-both-worlds design eliminated.
- SILK-skip carve-out added to PCM compare in decode + roundtrip paths.
- Byte-exact compressed-output compare added to roundtrip path BEFORE
  the decode-and-compare step.
- OOB-finding README severity upgraded to P0 DoS with threat-model section.

**Findings produced**: a real safety bug in `OpusMSDecoder::decode`. A
20-byte attacker-controlled packet triggers a slice-OOB panic at
`ropus/src/opus/multistream.rs:624` (`parse_multistream_subpacket`
returns a packet_offset that can exceed sub_data.len(); the subsequent
slice panics in safe Rust where C reference returns
`OPUS_INVALID_PACKET`). Tracked at
`tests/fuzz/known_failures/multistream-decode-subpacket-oob/` (4 repros
across smoke runs).

**This is the most severe finding so far — DoS-grade.** Anyone using
`ropus` to decode untrusted Opus packets is vulnerable. Out-of-scope
for Stream C per HLD V2 acceptance criterion 6 (campaigns produce
findings; codec fixes ship in separate PRs), but supervisor flagged
to Arthur for triage.

### 2026-05-01 — Stream B: complete

Implementation rewrote `fuzz_encode_multiframe` end-to-end with
`Arbitrary`-derived structured input. Coverage jumped from 36 (saturated
baseline) to ~11157 in a 30-second smoke run — ~310× improvement,
exactly as designed.

**Findings produced**: 6 unique differential repros at sr=8000, ch=2,
VOIP, br≈59713 with VBR + FEC + DTX toggled across frames. Tracked as
`tests/fuzz/known_failures/multiframe-vbr-fec-dtx-divergence/`.

**Reviewer feedback applied**:
- `tools/overnight_fuzz_launch.sh:21` — removed retired
  `fuzz_packet_parse` from TARGETS array. Added per-target max_len
  override case block (set to 65536 for `fuzz_encode_multiframe` so
  high-sample-rate stereo frames fit; default 16384 elsewhere).
- `tools/gen_fuzz_seeds.py` — removed all 14 references to
  `fuzz_packet_parse` (single-source TARGETS list + `gen_packet_parse_seeds`
  function + call site + docstring).
- `fuzz_encode_multiframe.rs` — replaced `format!("{:?}", input)` panic
  capture (~62 KB Debug string per iteration) with a 16-byte
  fingerprint helper. The libFuzzer artifact dir already retains the
  raw input; the fingerprint is enough for human-eyeball debugging.
- Dropped dead PCM-size defensive check (`Arbitrary` guarantees the
  equality).
- Replaced `nb_frames.unwrap()` with a `match` so panic messages
  include the actual error or unexpected-Ok value.

**Devil's-advocate concerns resolved**:
- Per-frame setter shuffle proven sound (Rust setters bit-identical to
  C CTLs; no RESET_STATE on either side; `silk_mode.use_in_band_fec`
  read live each frame in `silk_Encode`).
- libFuzzer max_len starvation addressed by per-target `-max_len=65536`.
