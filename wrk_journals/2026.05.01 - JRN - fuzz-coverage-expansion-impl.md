## Decision log

### 2026-05-01 — Stream A first agent: NaN/Inf differential finding

First implementation agent completed all 6 targets but the smoke run on
`fuzz_encode` produced a real differential divergence within seconds:
`OpusEncoder::encode_float` sanitises NaN inputs differently from C
`opus_encode_float` (Rust's `float2int16` returns 0 for NaN; C returns
-32768). This is a legitimate codec-level bug, but **out of scope** for
Stream A.

**Decision:** in the four f32-PCM-input targets (`fuzz_encode`,
`fuzz_encode_safety`, `fuzz_roundtrip`, `fuzz_roundtrip_safety`),
sanitise NaN/Inf to 0.0 immediately after `f32::from_le_bytes`. The
fuzzer's job is to find codec divergences; float-conversion semantics
are a separate PR.

**Open finding** (track for later):
- Repro: `tests/fuzz/findings/nan-encode-float/crash-54acd282f7bd446d92811b719f183d6eea294f84`
- Root cause: `ropus/src/types.rs::float2int16` does not match C reference's NaN handling.
- Severity: low (production callers don't pass NaN through `encode_float`).
- Fix: separate PR against `ropus/src/types.rs`. Probably one line — clamp NaN to -32768 to match `_mm_cvttps_epi32`-style C semantics.

The crash artifact has been moved from `tests/fuzz/artifacts/fuzz_encode/`
to `tests/fuzz/findings/nan-encode-float/` so subsequent fuzz runs don't
trip over it as a regression seed.
