# 2026-04-19 — LTO × PGO bake-off

## Context

Current `Cargo.toml` sets `lto = "fat"`, and `tools/pgo_build.sh` is
opt-in. Arthur recalled that ThinLTO (without PGO) was historically the
sweet spot. Two earlier journals today (`restore-unchecked-indexing.md`
and `silk-encoder-unchecked-indexing.md`) benchmarked everything under
"PGO + fat LTO". With Stage 1–3 indexing work now committed, it was
worth re-measuring the four cells to see which combo actually wins
on the current tree.

## Methodology

Ran on a git worktree at HEAD `ddfd151` (clean, unaffected by a
concurrent agent's uncommitted `ropus-fb2k` crate-in-progress):

```
git worktree add /c/tmp/ropus-bakeoff HEAD
```

For each config:
1. Edit worktree `Cargo.toml` → set `lto = "fat"` or `"thin"`.
2. For no-PGO: `cargo build --release --bin ropus-compare`.
   For PGO: `bash tools/pgo_build.sh --iters=3` (full train+use cycle).
3. `bash tools/bench_sweep.sh --iters=50` over all 10 vectors.

Same binary, same vectors, same machine, back-to-back runs. Worktree
gets its own `target/` so no cross-contamination with the main repo.

## Analysis

### Encode: ThinLTO is a clear, consistent win over FatLTO

ThinLTO beats FatLTO on 9 of 10 vectors (the 10th is SILK NB, which
sits at parity anyway). The margin is striking:
- SILK WB: 1.521→1.423 (−10pp)
- CELT square: 1.388→1.280 (−11pp)
- MUSIC: 1.283→1.092 (−19pp)

Mean encode ratio: FatLTO ≈ 1.26×, ThinLTO ≈ 1.19×. That's a full
6–8 percentage points on the whole encoder, across the board,
from a single-line `Cargo.toml` change. Bigger than any single
indexing stage we landed today.

### Decode: roughly a wash

Mean decode ratio is within ±0.01 across all four configs. Per-vector
the ordering flips both ways — this is bench noise. No clear winner
for decode.

### PGO: marginal to negative

- **FatLTO + PGO helps MUSIC specifically** (1.283→1.139, −14pp) but
  barely moves other vectors.
- **ThinLTO + PGO is mixed-to-worse.** It hurts CELT sweep dramatically
  (1.062 → 1.372, +31pp regression), hurts CELT square (+5pp), hurts
  SPEECH (+3pp), hurts MUSIC (+3pp). Wins on SILK WB are tiny
  (1.423→1.422, noise). The sweep regression is almost certainly PGO
  mis-optimization: the training vectors don't include sweep-like
  content, so the profile guides the wrong choices.
- Mean ratios: ThinLTO no-PGO (1.194) actually beats ThinLTO + PGO
  (1.234). PGO makes ThinLTO *worse* on average.

### PGO build-time cost

PGO adds ~6–8 min of train-instrument-merge-rebuild per build cycle.
ThinLTO itself compiles faster than FatLTO too (less whole-program
work). So dropping PGO is a double win: faster builds *and* better
(or equal) runtime perf in this tree.

## Verdict

**ThinLTO without PGO is the sweet spot on the current tree.** Arthur
remembered correctly. Mean encode ratio 1.194× vs the current config's
1.256×; mean decode indistinguishable. The encode win is larger than
the Stage 3 CELT indexing work gave us.

## Recommendations

1. **Change `Cargo.toml`**: `lto = "fat"` → `lto = "thin"`.
2. **Remove `tools/pgo_build.sh` from the default bench path.** Keep
   the script — it's still useful for one-off what-if measurements —
   but stop quoting "PGO + fat LTO" as the canonical number in
   journals. The canonical build is now plain `cargo build --release`
   with ThinLTO.
3. **Re-run previous bench tables** under the new config if we want
   numbers that match what a normal `cargo install ropus` produces.
4. **Don't bother investigating the sweep PGO regression.** It's a
   PGO-profile-vs-sweep-content artifact; the fix is "don't use PGO".
