# 2026-05-11 — Performance baseline recalibration

## Why

The 2026-05-06 performance-threshold calibration journal recorded an
unresolved "decode 8.74× slower than C" smoke on `48k_sine1k_loud.wav` and
flagged it as either a real regression or a measurement artifact, deferring
the resolution to a calibrated follow-up. Today's session ran that follow-up.

Outcome: no codec regression. The smoke was a bench-harness artifact. With the
harness fixed (commit `54fef85`), Rust is at parity with the C reference —
encode 1.06× mean (4/10 vectors faster than C), decode 1.01× mean (5/10
vectors faster than C).

## Root cause of the artifact

`bench_decode_rust` (and `_c`, and both encode variants) constructed a fresh
codec **inside** the per-iter loop:

```rust
for _ in 0..iters {
    let mut dec = OpusDecoder::new(...);
    // ...decode every frame in the file...
}
```

`OpusDecoder::new()` allocates an `LPCNetPLCState` and, if the build has
embedded DNN weights, calls `load_model` on the 8.78 MB `WEIGHTS_BLOB`.
Microbench (`/tmp/decoder_ctor_bench`): **3.34 ms per call**.

For a 1-second clip at 50 frames/iter × 30 iters, that's 100 ms of bench
overhead per vector — comparable to or larger than the actual decode work.
The C side pays only the ~5 μs `opus_decoder_create` cost per iter.

The artifact is invisible:
- On long clips: 3.34 ms amortises across ~hundreds of frames per iter, so
  MUSIC/SPEECH looked roughly correct (1.05–1.11× decode).
- In the published crate: `cargo publish` ships an empty `WEIGHTS_BLOB`
  (consumers run `fetch-assets` to populate it). With an empty blob,
  `load_model` is skipped, and `OpusDecoder::new()` is fast.

So this bug only fired locally on developers who had run `fetch-assets`, and
only on short benchmark clips. The README's published baseline was measured
without local weights (or on a much faster CPU), which hid the artifact at the
"decode is faster than C" level it was claiming.

## Fix

Commit `54fef85` (`bench: amortise codec construction across iters`):

- Decoder constructed once outside the timed loop; same for encoder; same for
  both Rust and C paths (otherwise the comparison is asymmetric).
- Matches realistic caller usage: construct once per stream, decode many
  packets through it.
- Side fix: `tools/bench_sweep.sh` hardcoded a Windows `.exe` path. Now falls
  back to the Linux binary.

## Methodology

Host: i7-14700KF (Raptor Lake, AVX2/FMA/BMI2), Linux 6.6.87.2-microsoft-
standard-WSL2, x86-64-v3 target via `.cargo/config.toml`. ThinLTO release
profile (unchanged).

5 back-to-back sweeps:
```
for i in 1 2 3 4 5; do bash tools/bench_sweep.sh --iters=30 > /tmp/sweep_$i.log; done
```

Per (vector, op) ratio = `rust_ms / c_ms` taken directly from the bench's
per-iter timings (the printed `1.04x FASTER` / `SLOWER` text strips the sign;
parsing the raw ms is unambiguous).

Across-sweep aggregation: **median**, not mean. Sweep 1 was uniformly noisy
(SILK WB encode 1.66× vs 1.04–1.08× on sweeps 2–5, similar pull-ups across
other rows — probably background activity on the host during that run).
Median is robust to a single bad sweep; mean is not.

Encode faster than C: SILK NB, SILK WB, Hybrid, CELT sweep, CELT sine (5/10).
Decode faster than or at C: SILK NB, CELT FB mono noise, CELT sine,
CELT sweep, CELT square, plus Hybrid/sine roughly at parity (6/10).

Worst rows:
- Encode: CELT square / SPEECH at 1.20×, MUSIC at 1.16×, CELT FB mono noise
  at 1.13×. C's hand-tuned SSE on the CELT MDCT/quant path is doing real work
  here; closing that gap would mean SIMD-ifying the equivalent Rust paths.
- Decode: SILK WB noise at 1.10× and MUSIC/CELT FB stereo at 1.07× are the
  only meaningful regressions vs C; none look like algorithmic problems.

## Variance notes

- `stdev` column is the standard deviation across the 5 sweep samples for each
  ratio, NOT a per-sweep stdev across the 30 iters. So 0.05–0.10 is normal
  cross-sweep noise on WSL2; 0.10–0.17 indicates one or two rows in one sweep
  pulled the spread.
- The 0.317 stdev on SILK WB encode is sweep-1-driven (single outlier at 1.66×
  vs cluster around 1.04–1.08×). Median absorbs it cleanly.

## Threshold table

`full-test/src/bench.rs` THRESHOLDS regenerated. Each row uses the new median
as the baseline and the same formula as the 2026-05-06 initial calibration:

```
release_fail_ratio = max(baseline + 0.05, baseline * 1.20, 1.05)
```

rounded up to two decimal places. `THRESHOLD_SOURCE` const updated to point at
this journal.

Sanity check: every measured ratio is well inside its new threshold (the
worst row, CELT square encode 1.20×, has fail threshold 1.44×).

## Out of scope

- Closer per-row variance characterisation (per-sweep stdev across iters, or a
  Mann-Whitney test). The current 5-sweep median is enough to make the release
  gate honest; tighter calibration is only worth doing if the gate starts
  flagging false positives.
- Multi-host calibration (Windows native, native Linux, Apple Silicon). The
  threshold rows are explicitly tied to the host in `THRESHOLD_SOURCE`.
- Optimising the worst rows. The bench is now honest; whether to chase the
  1.13–1.20× encode rows is a separate decision.
