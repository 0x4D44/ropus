# CR: full-test Phase 2

## Verdict
request-changes

The design is sound and well-tested in isolation, but two issues will bite Phase 3+ integration and one contradicts its own code comment. Fix and re-review.

## Blockers

1. **`coverage` sub-object is `Some(zeros)` on spawn/build failure, not `None`** — `full-test/src/tests.rs:204-216`. `tempfile::Builder::tempfile()` *creates* an empty file on disk; if cargo never writes to it (spawn err, or build failure before llvm-cov produces output), `fs::read_to_string(cov_path)` returns `Ok("")`, matching the `Some(Ok(text))` arm. `llvm_cov_parse::parse("")` then returns `CoverageResult::default()` (all zeros). Result: the JSON envelope shows `coverage: { ropus: { lines_covered: 0, lines_total: 0, ... }, workspace: {...} }` rather than `coverage: null`. This contradicts the code comment at `tests.rs:210-213` which claims None is returned, and will confuse Phase 4's HTML template: "coverage 0/0 = 0%" is a very different signal from "coverage unavailable." Gate on `text.is_empty()` or check `cov_path` modification time against the tempfile-creation timestamp before calling `parse`.

2. **Stderr size cap bypassed in Stage 2** — `full-test/src/tests.rs:183-194`. Phase 1 documents a 1 MiB stderr cap (`issues::MAX_STDERR_BYTES`, `issues::cap_stderr`) as shared infrastructure. Stage 2's dispatcher captures `o.stderr` via `String::from_utf8_lossy(...).into_owned()` with no ceiling, then hands the full buffer to `cargo_parse::parse`. On a ~45 min run with thousands of `Compiling` lines plus occasional test backtraces, stderr can run to many MiB; the Phase 2 journal (Hints, line 60) even says "Stage 3 stderr… feel free to reuse [`cap_stderr`]" but Phase 2 itself doesn't. Apply `issues::cap_stderr` to both stdout and stderr before parsing — the `test result:` summary lines are tiny relative to failure-body spam, and per-test outcome lines are ~80 bytes × ≤2000 tests = <200 KB, well under the cap. The build-failure detector only needs the head of stderr anyway.

## Important

3. **Cross-platform path filter is correct, but the doc-comment is wrong.** `full-test/src/llvm_cov_parse.rs:129-134`. The code `normalise_path` replaces `\` → `/` before calling `is_ropus_src`, which only inspects `/ropus/src/` and `ropus/src/` (forward-slash). So the Windows case is handled after normalisation. But the module doc at line 30-31 says "path contains `ropus/src/` (Unix) or `ropus\src\` (Windows)" — that claim is stale; the Windows variant is never matched against. Fix the comment to describe the normalise-then-forward-slash-only approach.

4. **Silent misclassification of benchmark output.** `full-test/src/cargo_parse.rs:250`. The regex `^test\s+(\S+)\s+\.\.\.\s+(.+)$` matches `test foo ... bench:     1,234 ns/iter (+/- 56)` and `classify_outcome` classifies anything not starting with `failed`/`ignored` as `Pass`. Today Stage 2 passes `--lib --tests` (no `--benches`), so benches don't run; but there's no defensive check, and Phase 4+ may someday pull in bench output. Either add `bench:` → `Outcome::Pass` with an explicit comment, or return a distinct `Outcome::Bench` that the summary total-counter excludes.

5. **Tempfile dependency is pulling `fastrand`/`getrandom` for ~6 lines of use.** `full-test/Cargo.toml:21`. The `NamedTempFile` is created, its path passed to cargo, the read-back done synchronously in the same function. A 4-line `std::env::temp_dir().join(format!("ropus-llvm-cov-{}-{}.json", std::process::id(), epoch_ms))` with an explicit `fs::remove_file` at function exit would avoid the whole dep tree. Not a blocker — the deps are already in the lockfile's transitive graph — but it widens the supply-chain surface of a dev-only binary for marginal ergonomic gain. Your call.

6. **ANSI colour on the leading `test` token not tolerated.** `full-test/src/cargo_parse.rs:250`. Regex requires `^test` at column 0; a colour prefix like `\x1b[32mtest\x1b[0m foo ... ok` won't match. Cargo suppresses colour when stdout is piped (which is what `Stdio::piped()` does), so this is a latent not active risk. Belt-and-braces: pass `--color=never` or set `CARGO_TERM_COLOR=never` on the `Command` to make the contract explicit and portable across Windows terminal emulators that sometimes leak TTY semantics into pipes. One-line fix in `tests.rs` before `cmd.args(&args)`.

7. **User test output collision risk.** If any test `println!("test my::thing ... ok")` — e.g. an integration test that spawns a sub-harness and echoes output — `cargo_parse::parse` would treat it as a real outcome line, inflating totals. No such tests exist today (`grep` confirms). Document the assumption in `cargo_parse.rs:84` module docs so future work knows to escape it.

## Nits

8. **`Outcome::skipped()` `#[cfg(test)]` gate is a code smell** — `full-test/src/tests.rs:50-62`. The journal (§ Deviations #5) justifies the gate by noting "no production path calls it today." That's fine, but the pattern implements a constructor purely to satisfy envelope tests that could instead be rewritten to construct `TestsResult { skipped: true, ... }` inline — the auxiliary constructor adds API surface for test-only callers. Either expose it unconditionally (zero runtime cost for an unused public method) or drop it and adjust the ~3 test sites. Leaning: expose unconditionally — Phase 4's upstream-build-failure path will want it.

9. **HLD deviation worth calling out formally.** The `--skip testvector` choice is the right technical call (verified — only the 24 IETF tests match; `grep -rn "fn testvector" ...` returns empty outside `tests/conformance/tests/ietf_vectors.rs`). Good documentation in `tests.rs:11-21` and the journal. Add a note to the HLD itself (`wrk_docs/2026.04.19 - HLD - full-test-runner.md:64`) so the next reader doesn't chase the phantom `--exclude conformance` spec.

10. **`build_cargo_args` signature is readable.** `full-test/src/tests.rs:81-110`. Three booleans risk an argument-swap bug; consider a small config struct (`Stage2Spec { skip_coverage, ietf_present, cov_path }`) for clarity. Minor.

11. **`binary_stem` hash regex is hard-coded to 16 hex chars.** `full-test/src/cargo_parse.rs:280`. Cargo's hash length has been 16 since the ~2018 rework; if it ever changes (nightly-only), labels regress to `ropus-abcdef...`. Not a blocker; worth a `// if cargo changes hash length, labels just lose the stem-strip — parser still works` comment.
