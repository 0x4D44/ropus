## For the code reviewer

- Drift originates squarely in the cepstral-band feature cluster (feats
  18-21), with feat 20 the worst at `3.2544136e-5`. The remaining
  features 22-35 are bit-exact across all 50 frames. This pinpoints the
  region of the LPCNet feature pipeline (post-`Ex[]` cepstrum / band
  energies) the next remediation slice should focus on; the upstream
  `Ex[]` / FFT / band-energy code paths are listed but not localised by
  this diagnostic (Revision A wording).
- The Rust DRED encoder updates `lpcnet_enc_state.features` to the
  *second* feature vector (`f1`) per `process_frame` (per HLD line 90).
  The C shim copies the same final vector. One snapshot per Opus frame is
  the per-frame granularity gated by this slice; intra-`process_frame`
  granularity (recording `f0` separately) would need new FFI and is out
  of scope.
- `cargo fmt --check` passes. The new file has no compile warnings under
  `cargo test -p ropus-harness-deep-plc --test dred_lpcnet_feature_drift`.
- Bounds were locked with no margin (HLD policy). If the next agent
  reduces drift, they're expected to re-lock the bounds *downward* under
  Arthur signoff with a journal entry naming the responsible commit.
