## Findings

### 1. Missing reference + weights tarball
First `cargo build --workspace --all-targets` failed because
`reference/dnn/dred_rdovae_stats_data.c` was absent. That file ships in
the xiph weights tarball; running `cargo run -p fetch-assets -- weights`
extracted the tarball under `reference/dnn/` and the C compile then
proceeded.

This is environment setup, not a defect.

### 2. Link failures in harness tests/bins after C reference compiled
Once the C library built, three Rust targets failed at the final link
step with `undefined symbol` errors for the xiph C reference functions
they exercise:

- `harness::find_first_divergence` (binary, via `bin_inner/find_first_divergence.rs`)
- `harness::tests::dred_differential` (integration test)
- `harness::tests::c_ref_differential` (integration test)

Root cause: each of these has its own `unsafe extern "C" { ... }` block
declaring xiph symbols, but **no `#[link(...)]` attribute**. They relied
on the build script's `cargo:rustc-link-lib=static=opus_ref` directive
being passed transitively through the `ropus_harness` lib's metadata.

That transitive propagation works only when the dependent crate
**actually consumes** symbols from the lib's `bindings` module. These
three files don't `use ropus_harness::bindings::*` — they import only
from `ropus::*` — so rustc gc's the `ropus_harness` rlib out of the
final link, taking the inline `#[link]` directive with it.

The fix that rustc/Cargo expect: each `extern "C"` block whose symbols
need a specific native library should declare it with
`#[link(name = "opus_ref", kind = "static")]`. That makes the
dependency self-contained and propagates directly to the bin/test
linker invocation regardless of which other rlibs survive gc.

Verified hypothesis on `find_first_divergence.rs` first; `cargo build
-p ropus-harness --bin find_first_divergence` then linked clean.

Applied the same `#[link(...)]` attribute to the first `extern "C"`
block in each of the three files. (One per crate-link-unit is enough —
`-l static=opus_ref` becomes a global linker arg covering all extern
blocks.)

### 3. Residual cc-rs warning: `#pragma message` from opus_decoder.c

```
reference/src/opus_decoder.c:38:10: note:
  '#pragma message: You appear to be compiling without optimization, if so opus will be very slow.'
```

This fires in debug builds because the C code suppression flag
`OPUS_WILL_BE_SLOW` is not defined. cc-rs forwards `note:` lines as
`cargo:warning=` so it shows up at the workspace level. Defining
`OPUS_WILL_BE_SLOW` in both harness build.rs files silences it without
losing real warnings.

### 4. Build-script staleness on first re-run
`cargo build` after the weights fetch still surfaced the
"reference DNN data files not found" warning from `ropus/build.rs` and
the equivalent from `harness-deep-plc/build.rs`. Cargo cached the build
script outputs from the earlier failed run and didn't re-run them when
the missing files appeared. `cargo clean -p ropus -p ropus-harness
-p ropus-harness-deep-plc` shook them loose. Not a code defect — the
build scripts already declare `rerun-if-changed` for the data files,
but Cargo's rerun semantics for files that didn't exist at script time
appear to have a hole. Worth a follow-up; not blocking this sweep.
