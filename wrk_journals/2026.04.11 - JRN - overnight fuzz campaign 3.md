### 22:30 — Comprehensive Decode Divergence Scan

Wrote `test_fuzz_decode_scan_all_configs` — scans all 64 TOC configs × 5 sample rates × 2 channels × 4 payload patterns. **Found 195 divergences.**

**Critical pattern analysis:**
1. ALL SILK-mode TOC configs (0-15) diverge — SILK-only AND Hybrid
2. CELT-only configs (16-31) are 100% CLEAN
3. 10ms SILK frames PASS; 20ms+ frames FAIL (state carryover between sub-frames?)
4. ALL sample rates affected (8k, 12k, 16k, 24k, 48k)
5. Only extreme payloads trigger it: 0xFF×50 and 0x80×50 fail; 0x00×50 and ramp pass
6. max_diff ranges from 2 (20ms single-frame) to 65535 (40ms+ multi-frame)

**Root cause hypothesis:** Something in the SILK decoder core produces different numerical results between Rust and C when processing extreme values. The difference is small per sub-frame but cascades across sub-frame boundaries within multi-frame packets. Real audio passes because the intermediate values never hit the pathological code paths.

**Key suspects:**
- NLSF→LPC conversion with extreme NLSF values
- LPC synthesis filter overflow handling
- Gain/excitation decoding with saturating values
- Fixed-point rounding/saturation that differs between Rust and C

### 23:30 — Bug #14: Safety Overflow Found by fuzz_decode_safety!

**Target:** fuzz_decode_safety, iteration #131,072 (after ~25 min of fuzzing)
**File:** `silk/decoder.rs:1279` (original line; PLC LPC synthesis path)
**Error:** `attempt to add with overflow`

**Root cause:** The PLC (Packet Loss Concealment) and CNG (Comfort Noise Generation) LPC synthesis loops use non-wrapping `+=` for accumulating the LPC prediction:
```rust
lpc_pred_q10 += ((s_lpc_q14[idx] as i64 * a_q12[j] as i64) >> 16) as i32;
```

The main `decode_core` path correctly uses wrapping arithmetic (`(... as i64 + ...) as i32`), but the PLC and CNG paths were missed.

**Affected locations:**
1. PLC path: `silk_plc_conceal` LPC synthesis (~line 1330)
2. CNG path: `silk_cng` LPC synthesis (~line 1499)
3. Possibly PLC LTP prediction `pred +=` (~line 1310)

**Fix:** Change `+=` to the wrapping pattern `= (... as i64 + ...) as i32`.
Sent to silk-fixer agent for combined fix with Bug #13/#15.

### 23:30 — Bug #15: LBRR Flags Missing +1 (found by fix agent)

The fix agent discovered that `silk_decode` was missing `+ 1` on the LBRR flags decode:
- **Rust:** `rc.decode_icdf(SILK_LBRR_FLAGS_ICDF_PTR[nfpp - 2], 8)`
- **C ref:** `ec_dec_icdf(...) + 1`

The `+ 1` converts from 0-based ICDF symbol to 1-based LBRR flag bitmask. Without it, LBRR flags are wrong, causing different bit consumption from the range coder, which cascades into different parameter decoding. This resolved ~95 of the 195 divergences (multi-frame LBRR cases) but is NOT the root cause of Bug #13 (WB order-16 specific divergence remains with 100 configs).

### 01:30 — Bug #16: NLSF2A OOB Index Found!

**Target:** fuzz_decode_safety (rebuilt with Bug #14 fix), crashed after running clean past Bug #14
**File:** `silk/common.rs:1419` in `silk_nlsf2a()`
**Error:** `index out of bounds: len is 129 but index is 18446744073709551488`
**Root cause:** `nlsf[k]` can be negative with garbage input. `f_int = nlsf_k >> 8` becomes negative, `as usize` wraps to huge value → OOB on cos table.
**Fix:** Clamp f_int to [0, 127] before indexing. C has `silk_assert(f_int >= 0)` which is no-op in release (UB).

Sent to fix agent. **This is the SECOND safety bug found tonight.**
