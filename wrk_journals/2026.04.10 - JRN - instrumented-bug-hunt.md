# 2026.04.10 — Instrumented Bug Hunt

## Objective

Fix the 4 remaining encoder mismatches that code-level comparison failed
to resolve. Strategy: per-frame instrumentation to pinpoint divergence,
then root cause + HLD + fix.

## Previous attempts (failed)

Two rounds of code comparison agents examined LBRR config, bit accounting,
CELT loss_rate, prefill ordering, clone-vs-inplace silk_mode. All matched
C exactly. The bugs are too subtle for code comparison alone.

## New strategy

Each agent will:
1. Add `ec_tell()` / state comparison checkpoints at key stages inside
   the SILK/CELT encoding path
2. Run the failing transition test and capture divergence data
3. Narrow the bug to a specific ~200-line function
4. Compare that function against C reference to find exact root cause
5. Write HLD with root cause + proposed fix

### T+1 — Both agents converge on same root cause

Both agents independently found the same encoder bug:

**Root cause:** SILK prefill modified a CLONE of `delay_buffer` instead of
modifying in-place. In C (`opus_encoder.c:2198-2204`), `delay_buffer` is
modified in-place (gain_fade + zero). Later, `tmp_prefill` (used for CELT
prefill) is copied from the ALREADY-MODIFIED `delay_buffer`. Rust's clone
caused `tmp_prefill` to get the original (un-gain-faded) data, producing
different CELT output.

**Fix applied:** Changed prefill path to modify `self.delay_buffer` in-place,
matching C behavior. The `celt_window` is cloned to a `Vec` to avoid borrow
conflicts.

**Results after fix:**
- FEC: 5 → 4 mismatches
- Other 3 tests: unchanged at 1 each

**Critical discovery:** The transition test's `mismatches` counter counts
BOTH encoder packet differences (line 7123) AND decoder PCM differences
(line 7191). The agents confirmed encoder packets now match at all
divergence frames. The remaining mismatches are DECODER-side: identical
packets produce different PCM between C and Rust decoders.

**Cleanup:** Removed agent's stray additions (full_test binary, chrono/colored
deps, instrumentation getter methods). Kept debug helper FFI functions
(debug_get_celt_encoder_state, etc.) as useful infrastructure.

### T+2 — Focus shift to decoder

All remaining mismatches are decoder PCM differences at mode transitions:
- The `loss_cnt` fix (already applied) was correct but insufficient
- The PLC bugs from the earlier decoder HLD (LTP rewhitening, noise source,
  CNG behavior) are likely causes
- Need to fix the Opus decoder's mode-transition PLC/crossfade path

Dispatching focused decoder fix agent.

### T+3 — Decoder PLC rewrite attempt

Decoder-fix agent performed a major PLC rewrite:
- Rewrote `silk_plc_update`, `silk_plc_conceal`, `silk_plc_glue_frames`
- Added LTP rewhitening, energy-based noise source, unvoiced gain scaling
- Agent verified PLC state matches C at entry points

**Result: No improvement.** Transition tests unchanged (1+1+1+4 = 7 mismatches).
PLC rewrite broke 2 unit tests (index out of bounds in `silk_lpc_analysis_filter`
called with empty buffer). Sweep still 210/210 PASS.

**Root cause identified by agent:** The PLC synthesis produces matching 16-bit
output but slightly different 32-bit LPC internal state (`s_lpc_q14_buf`).
This state cascades through `outBuf` to affect subsequent frames. The
divergence is in the LPC synthesis loop — a rounding difference in
`silk_SMLAWB` multiply-accumulate that produces different 32-bit results
truncating to the same 16-bit value.

**Decision:** Reverted PLC rewrite. Re-applied only the `loss_cnt` fix.
Unit tests back to pre-existing 3 failures only (792 pass / 3 fail).
