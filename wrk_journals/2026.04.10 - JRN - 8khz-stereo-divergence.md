# 8 kHz Stereo Decode Divergence Investigation

**Date:** 2026-04-10
**Trigger:** fuzz_decode panicked within 1 second on an 8 kHz stereo seed
**Finding:** PCM mismatch at sr=8000, ch=2, pkt_len=21, samples=80
**Verdict:** Real bug in `silk_decode_signs` — **FIXED**

## Context

Differential fuzzer (fuzz_decode) feeds synthetic Opus packets to both Rust and C
decoders, compares PCM output byte-for-byte. Divergence found on first run with
synthetic seed (TOC + incrementing payload bytes).

## Investigation Log

### Phase 1 — Agent team exploration

Three agents dispatched to explore fuzz infrastructure, decode path, and harness.
Initial hypotheses focused on stereo M/S conversion (`pred0_q13 as i16 as i64` casts)
and type mismatches (i16 vs opus_int32 for predictor storage). These were ruled out —
the values are in range for well-formed codec data and the cast semantics are correct.

### Phase 2 — Reproduction and isolation

Added `decode-raw` command to harness, testing multiple configurations:

| Config          | Mono | Stereo 8k | Stereo 12k | Stereo 48k |
|-----------------|------|-----------|------------|------------|
| cfg0 (NB 8kHz)  | —    | MATCH     | —          | —          |
| cfg4 (MB 12kHz) | MATCH | **DIVERGE** | **DIVERGE** | **DIVERGE** |
| cfg8 (WB 16kHz) | —    | MATCH     | —          | —          |

Key deductions:
- Mono MB matches → per-channel SILK decode and resampler are correct
- Stereo MB diverges at ALL API sample rates → bug is NOT in the resampler
- NB/WB stereo match → bug is medium-band-specific

### Phase 3 — Final range comparison

**Critical finding:** `opus_decoder_ctl(GET_FINAL_RANGE)` shows the range coder
states diverge: Rust=0x4248d6fc vs C=0x0b579730. This means the two decoders are
consuming different numbers of bits from the compressed bitstream — a parsing-level
bug, not signal processing.

### Phase 4 — Range coder tracing

Added `rc.tell()` checkpoints throughout the SILK decode pipeline. All checkpoints
before `silk_decode_frame` are identical between NB and MB. The divergence occurs
DURING frame decoding. Further traced to `silk_decode_pulses` / `silk_decode_signs`.

### Phase 5 — Root cause identified

**File:** `src/silk/decoder.rs:648` — `silk_decode_signs`

The Rust code had:
```rust
if base + j < length && pulses[base + j] > 0 {
```

The C reference has:
```c
if( q_ptr[ j ] > 0 ) {
```

The Rust code incorrectly bounds-checks against `frame_length`, skipping sign bits
for pulse positions beyond the frame boundary. But the shell decoder fills
`iter × 16` positions (rounded up), so the LAST block may have valid non-zero pulses
beyond `frame_length`. The C code decodes sign bits for ALL positions in the block,
consuming range coder bits. Rust's extra check skips them, causing range coder desync.

**Why only medium-band?** The shell codec frame is 16 samples. NB frame_length=80
(80/16=5.0, exact), WB frame_length=160 (160/16=10.0, exact), but MB frame_length=120
(120/16=7.5, not exact). Only MB has a partial last block where the bug manifests.

### Fix

Removed the `base + j < length` check in `silk_decode_signs`, matching the C reference
behavior. The `pulses` buffer is already sized to `iter × SHELL_CODEC_FRAME_LENGTH`,
so the access is safe.

### Verification

All 6 test configurations now produce MATCH (identical PCM and final range).
All 36 existing SILK decoder unit tests pass. 789/792 total library tests pass
(3 pre-existing failures in unrelated DNN/encoder modules).
