# Journal: validation and fuzz plan

Date: 2026-05-04

## Entry 1

Arthur is considering a full test-suite run plus fuzzing after the current wave
of parity and oracle work. My read is that this is the right direction, but the
order matters: deterministic tests should run before long fuzzing so ordinary
compile/test failures do not burn fuzz time.

Plan:

1. Run fast structural checks first: formatting/check/tests where scoped and
   available.
2. Run the broader deterministic test suite with timeouts.
3. Run bounded fuzz replays/smokes for changed areas: float encode/roundtrip,
   decode oracle, multistream, and CTL sequence if the new target builds.
4. Report failures without trying to fix unrelated dirty-tree changes.

Emotional temperature: cautious and slightly tense because the working tree is
large and clearly touched by multiple threads of work. The safest posture is to
collect signal first, resist cleanup impulses, and only narrow into failures
that are actually in the touched validation surface.
