Current state: the shared decode oracle already has stable Rust enum variants for the useful classes. The debt gap is that those classes are not exposed as stable human-facing names, the skipped correctness property is spread through comments, and there is no cheap distribution view to interpret "no fuzz findings".

Emotional temperature: focused and mildly wary. The tempting overreach is to make campaign-time reporting or broaden fuzz schemas; that would violate the slice. The useful move is smaller: make the existing classifier self-describing and add a bounded, opt-in report.

## Design Direction

Recommended direction for the HLD:

- Keep `tests/fuzz/fuzz_targets/oracle.rs` as the source of truth for class names, debt status, and unasserted properties.
- Keep fuzz target runtime behavior quiet and fast. No per-input prints. No always-on campaign counters.
- Add unit-testable summary/reporting helpers around the existing classifier.
- Add an ignored or explicitly named test/report path in `tests/fuzz/tests/oracle_tests.rs` so maintainers can get a bounded distribution over committed fixtures without running a fuzz campaign.
- Update `/home/md/language/ropus/wrk_docs/testing-oracle-matrix.md` to split the weakened classes instead of hiding them behind a single sample-count-only row.

Gut check: a standalone tool may be useful eventually, but this slice does not need a new binary unless the implementation agent finds the test-only report too awkward. A shell wrapper around an ignored test is enough if an operator-facing command is wanted.
