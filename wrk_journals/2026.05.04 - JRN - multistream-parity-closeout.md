# Journal: Multistream Parity Closeout

Date: 2026-05-04
Workspace: `/home/md/language/ropus`

## Entry 1

Arthur asked for HLD-only investigation of the `ropus-harness` multistream
encode/roundtrip parity failure. I read `AGENTS.md` first and kept the work to
docs and command-line investigation. I used the debugging-strategies and
software-design skills because this is mostly root-cause scoping and a minimal
fix plan.

Commands run:

```bash
timeout 180 cargo run -p ropus-harness --bin ropus-compare -- multistream
```

Result: reproduced the exact current failure. Multistream encode fails on frame
0 at byte 13; multistream decode of C-encoded packets passes; multistream
roundtrip first differs at sample 482 with `max_diff=23392`.

```bash
timeout 300 cargo test --release -p ropus-harness --bin ropus-compare \
  smoke_resilience_and_multistream_commands -- --nocapture
```

Result: release test path reproduced the same multistream failure. This is a
good high-signal repro because it avoids the full harness sweep but still
exercises the failing command.

Main insight: this current harness failure is probably not the old low-rate
SILK multistream class. The failing fixture is 48 kHz stereo, family 0,
complexity 10, audio application. C passes original multistream PCM and
downmix selectors into `opus_encode_native`; Rust multistream calls the
no-analysis `OpusEncoder::encode_native` wrapper for the sub-encoder. Since
analysis is gated on `complexity >= 10`, `16000 <= Fs <= 48000`, and non-
restricted-SILK, the C path should run analysis while the Rust path skips it.

I wrote the HLD at:

`/home/md/language/ropus/wrk_docs/2026.05.04 - HLD - multistream-parity-closeout.md`

Emotional temperature: clearer now than at the start. I was initially wary of
getting pulled into the older multistream fuzz failure, but the harness config
points at a cleaner top-level mismatch. Mild frustration: several old docs have
line numbers from earlier revisions, so I refreshed current line evidence
instead of trusting them. I did not implement anything.

## Entry 2

Arthur conditionally approved implementation only after evidence confirmed the
analysis-plumbing hypothesis. I started with a temporary diagnostic binary that
ran the deterministic multistream fixture at complexities 9 and 10. The
diagnostic was removed before the production change.

Evidence command:

```bash
timeout 180 cargo run -p ropus-harness --bin ms_complexity_probe
```

Result:

```text
complexity 9: PASS (25 packets byte-exact)
complexity 10: FAIL frame=0 byte=13 C_len=160 R_len=160
```

This confirmed the failure is tied to the complexity-10 analysis gate. With the
same deterministic multistream case, complexity 9 stays byte-exact while
complexity 10 diverges on frame 0 at the known first byte. That was sufficient
to proceed with the minimal analysis-plumbing fix.

Implementation notes:

- Added a crate-private `EncodeAnalysisInput` struct in
  `/home/md/language/ropus/ropus/src/opus/encoder.rs` so analysis callers can
  provide original PCM, frame size, channel selectors, channel count, and the
  downmix function.
- Preserved public single-stream behavior by keeping `encode` and
  `encode_float` on their existing paths and only changing how they package
  their existing analysis inputs.
- Updated multistream encode in
  `/home/md/language/ropus/ropus/src/opus/multistream.rs` to keep extracted
  substream PCM as the encode input while passing the original multistream PCM
  plus C-equivalent `c1`/`c2` selectors into the sub-encoder analysis path.
- Marked the no-analysis `encode_native` convenience wrapper as `#[cfg(test)]`
  after validation showed normal code now routes through the analysis-aware
  path and the wrapper is only used by tests.

Insight: the HLD direction was right. The relevant semantic gap was not the
per-stream PCM fed into encoding; that must stay extracted. The missing part was
the original multistream PCM plus selector metadata used only by tonality
analysis.

Emotional temperature: satisfied but cautious. The change is narrow and the
repro flips cleanly to pass, but codec parity bugs are brittle. I kept the
validation focused on the directly affected harness and unit paths and avoided
the unrelated fuzz worktree changes.
