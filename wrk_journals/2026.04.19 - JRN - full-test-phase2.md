## Deviations from HLD / brief

1. **`--skip ::testvector` instead of `--exclude conformance`**.
   The HLD proposes `--exclude conformance` when IETF vectors are missing, but that flag removes the entire `tests/conformance` crate (30 tests — 24 testvectors + 6 category probes, verified via `cargo test -p conformance -- --list`) rather than just the 24 IETF-vector probes. Every IETF probe is named `conformance::ietf_vectors::testvectorNN`; `--skip ::testvector` (name-based, passed after `--`) filters exactly those tests and keeps the rest of the conformance crate live. The leading `::` pins the substring match to a module-path component boundary — a bare `--skip testvector` would be a substring match and would silently drop any future unrelated test whose name merely contains `testvector`. The HLD should be amended; until then, the deviation is documented in `tests.rs` module docs and here.

2. **`cargo_parse` parses `test result:` summaries as the source of truth**.
   The brief suggests per-test counts + a per-binary summary, both derived from parsing. I kept both but the `total_passed/failed/ignored` are derived from the summary line, not a count of outcome lines. Rationale: a harness panic can produce outcome lines without a summary (or vice versa); the summary matches what a human reading the log would trust. The `per_test: Vec<TestOutcome>` list is still populated from outcome lines for the HTML module breakdown.

3. **`TestOutcome` gained a `binary: String` field**.
   The brief's sketch had `{fqn, outcome}`. I added `binary` because Phase 4's HTML report needs a per-binary module breakdown and recomputing the attribution later would require re-parsing stderr. Small addition, low cost, removes a Phase 4 dependency.

4. **`TestsResult` gained `ignored_test_names: Vec<String>`**.
   Brief says "record their names in a footnote list" for ignored tests. Simplest place is the top-level struct; Phase 4 pulls the list straight into the HTML footnote.

5. **No `--skip-tests` flag and no `quick`-gates Stage 2 path**.
   Matches the HLD: `--quick` skips stages 1 and 4, not 2. The `Outcome::skipped()` constructor exists but is `#[cfg(test)]`-gated because no production path calls it today; Phase 4 (HTML report) may wire it to an upstream-build-failure skip path.

## Blockers hit / none-found

- **No llvm-cov syntax surprises**. `cargo-llvm-cov`'s `--json --output-path <path>` invocation is documented; writes llvm-cov's `export -format=text` shape to the given path. JSON layout confirmed against `tools/cov_report.py` (already parses this format for branch analysis).
- **Tempfile lifetime** worked cleanly: `tempfile::Builder.tempfile()` returns a `NamedTempFile` that lives on the stack for the full `run()` duration, auto-deletes on drop. One subtle bit: I pass `--output-path <path>` to a subprocess that will write to it, so I convert to `PathBuf` before the subprocess spawn and keep the guard alive past the `fs::read_to_string` that reads it back.
- **No clippy regressions**: workspace passed `-D warnings` on full-test after 3 iterations (initial `module_inception` + a pile of `field_reassign_with_default` + `doc_overindented_list_items`). The `tests.rs` filename collides with `tests` as a conventional submodule name — I renamed the `#[cfg(test)] mod` to `unit_tests` to resolve.

## Hints for Phase 3 agent

- **Shared pattern: `build_cargo_args`**. Stage 2's dispatcher splits command construction (`build_cargo_args`) from spawning so both can be unit-tested. Stage 3 should follow suit: a pure `build_ambisonics_args()` that the runner calls, tested against a fixture expected-argv vector.
- **Shared pattern: `skipped()` constructor is `#[cfg(test)]`**. The Outcome types have a test-only skipped constructor. Phase 3 should keep this discipline — production paths should never "skip" an ambisonics run other than via the CLI flag.
- **Envelope extension**. `report::tests_to_json` uses `serde_json::to_value(&struct)` to auto-serialise, then splices in `coverage` / `coverage_skipped`. Stage 3 should add a top-level `serde::Serialize` struct (e.g. `AmbisonicsResult`) and follow the same splice pattern; keeps the report module from growing a hand-written field mirror.
- **Stderr bytes cap**. `issues::cap_stderr` (1 MiB) is shared infrastructure; Stage 3 stderr (ambisonics tool output) is small enough not to need it but feel free to reuse if the output ever grows.
- **Parse tolerance first**. `cargo_parse` and `llvm_cov_parse` both degrade to empty data rather than panic on malformed input. Stage 3's scrape of the `PASS: all fixtures …` line should be similarly defensive; don't rely on exact wording, match on the leading `PASS:` / `FAIL:` token with a regex.
- **Process priority**. Stage 0 already lowered it once; child `cargo` invocations inherit. Stage 3's `cargo run --release -p ropus-harness --bin projection_roundtrip` will inherit too — no need to re-apply.
- **The HLD's `--exclude conformance` deviation is already documented in `tests.rs` and here**; Stage 3 has no equivalent flag decision.
