### HLD sign-off
HLD: `wrk_docs/2026.05.07 - HLD - pitchdnn-feat18-residual.md`.

**Verdict: SIGN OFF.** The investigation is one of the strongest in this
project's history. Per-layer instrumentation traced the divergence to
the very first `tanh_approx` call (input bit-exact between sides;
output diverges by 4.2e-5). Independent f32 Python reproduction
matches Rust exactly to all printed digits.

**Root cause**: C reference's SSE2 path
(`reference/dnn/vec_avx.h:486`) uses `_mm_rcp_ps` — 12-bit-precision
approximate reciprocal (max relative error 3.66e-4 per Intel SDM) —
inside `tanh4_approx` instead of an honest f32 division. Rust does
honest division and is the spec-compliant path.

**This is the *inverse* of the prior eight DRED parity fixes.** Those
were Rust f32 narrowings vs C's double promotions. This time Rust is
correct; C is using a SIMD shortcut.

**Decision: Option B — accept the residual, document in the test comment
block. No production code changes.** Reasoning:
- Rust matches the polynomial spec exactly (verified by Python
  reproduction).
- Option A (mimic `_mm_rcp_ps` in Rust via x86 unsafe intrinsic)
  violates the project's "Safe Rust only; no platform-specific
  intrinsics" rule and would intentionally degrade Rust precision.
- Option C (rebuild C reference without SIMD) measures against a
  non-default reference no user ships — wrong oracle.
- 1.013e-5 drift on `dnn_pitch` is orders of magnitude below the
  integer pitch-period bin spacing; acoustically inaudible.

**Answers to HLD's open questions**:
1. **Option B** — confirmed.
2. **Tier-2 SNR ledger** — defer. Useful but scope creep now; track
   for future.
3. **Pin unit test for `tanh_approx`** — yes. Cheap, locks Rust
   against future drift, documents the spec-compliant value
   (-0.46118161... at input -0.498856246). Add to
   `ropus/src/dnn/core.rs` test module.
