# Journal - DRED/DNN Parity Implementation

Date: 2026.05.06

## Implementation Start

Read the signed-off HLD and review signoff before editing. Scope is deliberately
small: full-test policy/reporting plus oracle-matrix wording only. No codec code,
no harness-deep-plc tests, and no exact DRED payload parity changes.

Design choice: replaced the boolean preflight parameter with a small
`PreflightPolicy` enum:

- `default-report-only`
- `release-core-smoke-no-neural-claim`
- `release-core-plus-neural-dred-gate`

The enum derives from the same `quick` and `release_preflight` flags used by
Stage 2 package-profile selection. Quick release preflight remains core smoke
only and keeps neural assets report-only. Non-quick release preflight requires
the DNN/DRED assets because it is the first profile that can honestly claim the
deep-PLC/DRED test lane.

Testing approach: added tempdir-only asset policy tests so the checks do not
depend on this checkout's `reference/dnn` contents. The tests cover default,
quick release, and non-quick release behavior separately.

Emotional temperature: steady, slightly wary. The dangerous failure mode here is
not a Rust bug; it is wording that lets a green report imply coverage it did not
run. I am keeping the labels blunt and repeated across JSON, HTML, and help text
for that reason.
