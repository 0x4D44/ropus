### Finding #1 Investigation — 23:58–00:25

**Reproduction path:**

1. **Corpus-replay agent (first attempt):** built `tools/replay_fuzz_decode.rs` + `tools/fuzz_decode_panics.json` to iterate all 166 fuzz_decode corpus seeds and diff Rust vs C. Result: **zero** corpus seeds diverge. The 4 panics were on live mutations after libFuzzer iter ~3679 in each of 4 parallel jobs, but libFuzzer on Windows MSVC does not persist Rust-panic artifacts (the MSVC `__fastfail` 0xC0000409 path bypasses libFuzzer's SEH handler).

2. **Direct-seed replay (first attempt):** extracted the 4 libFuzzer seeds from the original fuzz.log — `3000291547, 3000291763, 3000264009, 3000335554`. Ran each against the already-built `fuzz_decode.exe` with `-max_total_time=240`. First try failed: ASan DLL not on PATH (the `fuzz_run.sh` `setup_windows_asan` helper is required). Second try (with PATH set) reproduced **3 of 4** panics — same family, slightly different metadata (e.g. `pkt_len=114 samples=720` instead of original `pkt_len=75 samples=720`). libFuzzer's mutation is not bit-deterministic across runs even with same seed (likely corpus directory iteration order influences it). Still no `crash-*` artifacts produced.

3. **Panic-hook instrumentation:** modified `tests/fuzz/fuzz_targets/fuzz_decode.rs` to install a `std::panic::set_hook` at first fuzz_target invocation that saves the current input to `FUZZ_PANIC_CAPTURE_DIR` (stable filename hashed from bytes). Rebuilt fuzz_decode binary in 46s — clean. Third-attempt replay is now running with the new binary (background task bk2v9df9n).

**Parallel static investigation:** `agent ac2d2b2d5f4e9fe1a` (general-purpose) reading `src/celt/decoder.rs` vs `reference/celt/celt_decoder.c` looking for multi-frame state carry-over bug. Still running as of 00:25.

**Divergence characterisation:**
Using the PCM arrays captured from panic logs, all 4 F1 findings have:
- Large deltas (hundreds of LSBs), not precision errors → control-flow or state bug.
- Divergence always appears **within a later subframe** of a multi-subframe code-3 packet (F1a diverges at sample 259 — inside subframe 2 of 3, F1b seed 3000291763 reprod diverges around sample 480-560 — inside subframe 3 of 3).
- Strongly implicates a state-reset/carry path in the CELT per-subframe loop.

Not yet confirmed: whether the Opus-layer dispatcher iterates code-3 CELT frames as separate `celt_decode` calls, or if the CELT decoder handles the multi-frame iteration internally. This is the first thing the static agent will report on.

### Finding #1 Static Investigation — 00:25–04:30

**Three parallel investigation attempts:**

1. **Static agent (ac2d2b2d5f4e9fe1a, 1054s):** exhaustive line-by-line comparison of `opus_decode_native`, `opus_decode_frame`, and `celt_decode_with_ec_dred` between Rust and C reference. Confirmed: outer loop matches, per-sub-frame state carry matches (postfilter, energies, decode_mem shift), bit budget calculation matches, prefilter_and_fold flag matches. Found zero line-level divergences. Recommends dynamic FFI tracing. Confidence: LOW.

2. **Pinpoint agent (a00652c4a7f47435e):** hit rate limit after 110 tool uses, no output. Wasted.

3. **Supervisor manual investigation (45+ min):** verified decode_native iteration (line 1228-1240), decode_frame CELT call (line 951), range decoder creation (fresh per sub-frame), `unquant_coarse_energy` → `old_band_e` carry chain, mono consistency copy, energy history updates, decode_mem shift (matches C exactly: `DECODE_BUFFER_SIZE - N + overlap`), postfilter state update chain, `opus_packet_parse_impl` code-3 VBR path, `parse_size` arithmetic. All semantically equivalent to C.

**Strongest clue from static agent:** "The shape of the divergences — both F1c and F1d landing in the second comb_filter call's region" (for lm>0 cases). For our actual 2.5ms crashes (lm=0), the second comb_filter is never called, so the bug is likely in:
- `clt_compute_allocation` / `quant_all_bands` (bit budget / PVQ decode) — if total_bits or the allocation tree differs by 1 bit, the entire PVQ output flips.
- `unquant_coarse_energy` or `unquant_fine_energy` — if the energy delta coding drifts after sub-frame 1.

**Conclusion: DEFER. Dynamic FFI tracing required.** See Bug #13 tracing plan (previous journal, lines 145-160) for the instrumentation specification. Bug only affects fuzz-generated garbage inputs — does NOT affect any real audio workload (210 sweep configs remain bit-exact).

### Finding #2 — SILK Encoder Divergence (fuzz_roundtrip, ~00:52)

**Signature:** sr=8000 ch=2 (stereo) app=VOIP br=71535 cx=2. Both Rust and C encoders produce 179 bytes, but the bytes are DIFFERENT. Same-length-different-content implies the encoder bit allocation or quantization path diverges.

**Notably:** `fuzz_encode` ran CLEAN (178K runs, 0 crashes). But `fuzz_roundtrip` triggered the encoder divergence. The difference is likely the PCM input structure — fuzz_encode uses structured i16 samples while fuzz_roundtrip uses more random/extreme values. This suggests the divergence requires specific extreme-value PCM that only random fuzzing hits.

**Status:** Not yet investigated. Lower priority than CELT decode bug. Will triage after campaign ends.
