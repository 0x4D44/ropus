## Objective

Build a torture testing suite that exercises the codec under sustained, varied, and adversarial conditions. The current testing is broad (280 configs) but shallow in duration and narrow in mid-stream variation. Real bugs hide in transitions.

## Architecture Decisions

### Design: `cmd_torture()` — Random Parameter Variation Soak

The existing `longsoak` pre-encodes ALL PCM in one batch, then decodes frame-by-frame. For torture testing, we need a fundamentally different architecture:

1. Create C + Rust encoder/decoder pairs (kept alive for the entire run)
2. Generate a long PCM signal upfront
3. Loop frame-by-frame:
   - Every N frames, randomize encoder settings on BOTH C and Rust encoders
   - Encode frame with both → compare encoded bytes
   - Decode with both → compare PCM samples + final_range
   - On first divergence: full state dump (frame#, config, PCM delta, ranges)
4. Report summary

**Key parameters varied mid-stream:**
- Bitrate: 6000-510000
- Bandwidth: NB/MB/WB/SWB/FB (via max_bandwidth)
- Application: VOIP/AUDIO/LOWDELAY (triggers SILK↔CELT mode switches)
- Complexity: 0-10
- Signal: VOICE/MUSIC/AUTO
- Force mode: AUTO/SILK_ONLY/CELT_ONLY/HYBRID
- VBR: on/off + constraint
- FEC: on/off + packet loss %
- DTX: on/off
- Frame size: 2.5/5/10/20/40/60ms

**Fixed per run:** sample_rate, channels (set at encoder creation)

### Design: `cmd_transitions()` — Targeted Mode Transition Tests

Deterministic sequences exercising the riskiest transitions:
1. SILK → CELT → SILK (force_mode cycling)
2. NB → FB → NB (bandwidth sweep)
3. VBR stress (silence → loud → silence signal with VBR on)
4. Application mode switching (VOIP → AUDIO → LOWDELAY cycling)

These are NOT random — they're designed to hit specific risky paths.

## Devil's Advocate Review

### Blind Spots

1. **We only compare C vs Rust — not C vs correct.** If both implementations have the same
   bug (e.g., a spec violation in the reference), our tests give false confidence. We have
   NO oracle for "what the output SHOULD be" — only "does Rust match C." This is fine for
   a port but doesn't catch bugs in the C reference that we faithfully replicated.

2. **Decoder transition testing is weak.** We decode C-encoded packets with both decoders,
   but we DON'T decode Rust-encoded packets. If the Rust encoder produces valid-but-different
   packets (which it does — the transition tests show this), we should also verify that BOTH
   decoders produce identical output from BOTH encoder streams. Currently, a Rust encoder bug
   could produce packets that decode differently under edge conditions.

3. **No cross-decode testing.** We should test: C-encode → Rust-decode AND Rust-encode → C-decode.
   The current tests only do C-encode → {C-decode, Rust-decode}. Rust-encode → C-decode would
   catch cases where the Rust encoder produces technically valid but unusual packets.

4. **Signal variety.** All torture tests use pure noise. Real audio has silence, transients,
   harmonics, and speech patterns that exercise different encoder paths. The DTX silence detection,
   voice/music classification, and bandwidth detection all depend on signal characteristics that
   noise doesn't exercise realistically.

5. **No FEC recovery verification during torture.** We test FEC/PLC separately but don't exercise
   FEC recovery DURING the torture soak with random config changes. A config change that disables
   FEC mid-stream might leave stale FEC state.

### What We Should NOT Bother With

- **Multi-threaded stress testing:** Opus is single-threaded by design
- **Formal memory profiling:** RSS before/after catches gross leaks; fine-grained tracking would add complexity with little value
- **Fuzzing the Rust-only encoder:** The comparison harness IS the oracle; without it, fuzz findings are ambiguous

### Specific Recommendations

1. **Add Rust-encode → {C-decode, Rust-decode} comparison to torture test.** This is the most valuable addition. When the Rust encoder diverges (which it does), we should verify its output is at least DECODABLE and produces reasonable audio.

2. **Mix signal types in the soak.** Alternate between noise, silence, and sine segments to exercise voice activity detection and bandwidth selection paths more realistically.

3. **Test the DECODER transition path separately.** Create packets with C encoder that switch modes mid-stream, then decode with both decoders. This isolates decoder transition logic.

4. **The 8kHz "addition" was already done.** Brief Section 4 is stale — 8kHz vectors and integration tests already exist. This saved time but also shows the brief wasn't fully current.

## Findings

### Torture test reveals mid-stream transition bugs

The torture test immediately found encode divergences when configs change mid-stream.
This confirms the brief's thesis: **"Real bugs hide in transitions."**

**Transition test results (8 tests):**

| Test | Result | Notes |
|------|--------|-------|
| SILK->CELT->SILK->CELT | FAIL | 68 mismatches, first at step 1 frame 0 |
| BW sweep NB->FB->NB | PASS | 450 frames clean |
| VBR on->off->on->off | PASS | 400 frames clean |
| VOIP-like->AUDIO-like | FAIL | 68 mismatches, first at step 1 frame 0 |
| DTX on->off->on | FAIL | 82 mismatches, first at step 1 frame 0 |
| Stereo force channels | PASS | 400 frames clean |
| Bitrate extremes 6k->510k | FAIL | 213 mismatches, first at step 1 frame 0 |
| FEC off->low->high->off | FAIL | 108 mismatches, first at step 2 frame 0 |

**Key observation:** All failures occur at the FIRST frame after a config change (step N, frame 0).
This strongly suggests the Rust encoder handles some mid-stream CTL changes differently
from the C reference. The existing sweep tests pass because they create FRESH encoders
per config — they never test config changes on live encoders.

**Types of changes that trigger divergence:**
- Force mode changes (SILK_ONLY ↔ CELT_ONLY): encoder internal mode transition
- Large bitrate jumps (6k → 510k): triggers implicit mode change
- DTX toggle: affects internal silence detection state
- FEC with high packet loss: affects redundancy coding state

**Types of changes that work correctly:**
- Bandwidth changes (NB ↔ FB): handled correctly
- VBR toggle: works fine
- Stereo ↔ mono channel forcing: works fine

**Root cause hypothesis:** The Rust encoder's `set_force_mode()` and related CTLs
may not reset/update internal state (mode history, transition flags, fade counters)
the same way the C encoder does. Investigation needed.

## Multiframe Fix (Committed)

Found and fixed a real bug in `src/opus/encoder.rs`: `encode_multiframe()` was calling
`encode_native()` per sub-frame, which re-derived mode, transitions, and redundancy
from scratch each time. The C reference calls `opus_encode_frame_native()` (the inner
function) with per-frame transition flags computed in the outer loop:

```
frame_to_celt = to_celt && i == nb_frames - 1;
frame_redundancy = redundancy && (frame_to_celt || (!to_celt && i == 0));
```

This ensures only the LAST sub-frame gets the mode transition, and redundancy is placed
correctly. The old Rust code let the first sub-frame handle the transition, changing
prev_mode early and causing subsequent sub-frames to encode in the wrong mode.

## Remaining: Single-Frame Mode Transition Bug

The 5 failing transition tests all use 20ms single frames. The multiframe fix doesn't
apply. Detailed diagnostics reveal:

- TOC bytes always match (mode/bandwidth decisions identical)
- First 67-153 bytes match (SILK encoding is correct)
- Divergence is in the CELT redundancy portion appended after SILK data
- Pattern: on SILK→CELT transition, the encoder encodes one more SILK frame with
  `to_celt=true` and appends a 5ms CELT redundancy frame. The CELT redundancy
  output differs between C and Rust.

**Root cause hypothesis:** The CELT encoder state entering the redundancy encode 
differs. Even though `OPUS_RESET_STATE` is called before the redundancy encode,
something about the reset or the subsequent 2.5ms prefill produces different results.
This could be:
1. A subtle difference in what the CELT `ResetState` clears
2. The prefill PCM data being different (offset issue as per investigator Finding #1)
3. A CELT encoder setting (start_band, prediction, etc.) that persists through reset

**Additional findings from investigator agent:**
- Finding #1: CELT prefill uses wrong delay buffer offset (Fs/400 samples too late)
- Finding #2: SILK prefill missing gain_fade onset ramp
- Finding #3: gain_fade guard condition differs (mode check vs celt_mode null check)
All three have been fixed in this session but don't affect the 20ms transition case
because those code paths require `mode != prev_mode` which is false when `to_celt` reverts mode.

**Recommended next step:** Add temporary debug tracing inside `celt_encode_with_ec` for
the redundancy encode path to compare internal CELT state between C and Rust at the
point of divergence.
