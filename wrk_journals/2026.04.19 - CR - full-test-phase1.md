## Verdict
request-changes

Scope, layout, and tests are solid; 20 unit tests all pass and the binary builds and runs cleanly. But the crate ships unused dependencies, a dead code path in `workspace_version()`, its own source fails `cargo fmt --check` (i.e. the exact check Stage 1 enforces), and the HLD's Stage-2-skipping `--quick` contract is mis-implemented for quality.

## Blockers

- **`C:\language\mdopus\full-test\src\main.rs:43`** — `let quality_outcome = if options.skip_quality || options.quick`. HLD § Flags says `--quick` "skip stages 1 and 4". Correct. **But** HLD § Flags also says `--quick`: "stage 2 still runs; if combined with `--skip-coverage`, it downgrades to plain `cargo test`." Currently only quality is wired. That is accurate for Phase 1, so this is not a Phase-1 blocker by itself — keep, but confirm Phase 2 doesn't inherit a `|| options.quick` short-circuit on stage 2. (Keeping under Important, see below.) **Retract: not a blocker.**
- **`C:\language\mdopus\full-test\src\setup.rs:68-76`** — The `workspace_metadata` lookup is dead code. `cargo metadata --format-version 1 --no-deps` exposes root-level keys `packages / workspace_members / workspace_default_members / resolve / target_directory / workspace_root` and a top-level `metadata` field (currently `null`); there is no `workspace_metadata` key. The HLD wants the resolved workspace version; the correct lookup is `packages[*].metadata` on the root crate *or* reading `[workspace.package].version` from the root Cargo.toml. Fix: drop lines 69-76 and keep only the `ropus` crate scan, or parse `workspace_root/Cargo.toml` for `[workspace.package]` via `toml`. Shipping commented-out "supported for when the workspace grows" dead branches is exactly the premature-abstraction anti-pattern CLAUDE.md warns against.
- **`C:\language\mdopus\full-test\src\quality.rs:53-86`** — `cargo fmt --check` exit code handling is too coarse. On the installed toolchain `cargo fmt` returns exit 1 for "code needs formatting" (normal non-fatal failure) **and** exit 101/127 for "rustfmt component missing / couldn't spawn" (tool-broken; should crash the run with a clear message, not be reported as a fmt issue). The current `out.status.success()` collapses both into "passed=false" with the same shape. Task brief explicitly asked about this. Fix: branch on `out.status.code()` — `Some(0)` passes, `Some(1)` is a legitimate fmt failure with issues, anything else (including `None`) surfaces as a setup/infra error distinct from a check failure.
- **Own formatting fails `cargo fmt --check`.** `cargo fmt --check --manifest-path full-test/Cargo.toml` reports four violations: `full-test/src/issues.rs:101`, `full-test/src/quality.rs:70`, `full-test/src/setup.rs:103`, `full-test/src/setup.rs:166`. Stage 1 would flag its own implementation on day one. Run `cargo fmt -p full-test` before merge.

## Important

- **Unused dependencies in `C:\language\mdopus\full-test\Cargo.toml:17,21`.** `chrono` and `clap` appear in `[dependencies]` but neither is `use`-d anywhere (`grep use chrono|use clap` → 0 matches). Task brief flagged this explicitly. `clap` is especially wrong: the comment at `cli.rs:11` says "Clap is available ... but the surface here is tiny enough that a hand parser keeps the binary lean," which is correct reasoning — so the dep should not be listed. `chrono` is needed for Phase 4 (report timestamps) but adding it now with zero usage means a warning under `-W unused-crate-dependencies` and violates CLAUDE.md's "write as little code as possible / don't predict the future." Remove both until the phase that actually uses them.
- **`C:\language\mdopus\full-test\src\issues.rs:42-58`** — Truncation ordering. `MAX_STDERR_BYTES` (1 MiB) is applied by `cap_stderr` **before** `extract` scans for warning/error lines. If a huge dump contains 50 errors where errors 5-50 land past the 1 MiB mark, we silently drop the tail. That is the right call (prevents pathological runs), but there's no race because truncation happens first and `MAX_ISSUES` caps the surviving slice second. Worth a doc comment in `extract`: "called only after `cap_stderr`; the 1 MiB cap is the authoritative truncation point, and `MAX_ISSUES=20` is a summary cap on what survives."
- **`C:\language\mdopus\full-test\src\cli.rs:31-45`** — The hand parser handles only the five boolean flags in their exact spelling. It does **not** handle `--flag=value` (no flags take values here, so fine), `--` separator (would trigger `unknown argument: --` error — wrong, should stop parsing), combined short flags like `-qh` (would error), repeated flags (silently tolerated — fine). The HLD spec is only five booleans, so most of this is out of scope — **but** the `--` separator case is a standard Unix convention and the error message is user-hostile. Either accept `--` as a no-op terminator or document the hand parser's minimal surface in the module docstring.
- **`C:\language\mdopus\full-test\src\main.rs:39`** — `set_low_priority()` is called **before** `setup::capture`. `capture` then spawns `git` and `cargo metadata` subprocesses that inherit the below-normal priority. Stage 0 is supposed to be <1 sec; slowing down `cargo metadata` (which can take 2-5 sec cold) is a real user-visible papercut. Move `set_low_priority()` to after capture, or better, between capture and the first cargo-invoking stage.
- **`C:\language\mdopus\full-test\src\setup.rs:116-135`** — Priority FFI SAFETY comments are accurate but incomplete. `SetPriorityClass` returns `BOOL` (non-zero = success); the return value is discarded. Correct decision (best-effort), but a comment clarifying "failure is non-fatal — we ignore the BOOL return" would match the `nice` treatment and silence a future clippy::must-use lint if one is added. Handle lifetime is fine: `GetCurrentProcess` is a pseudo-handle and must not be closed.
- **`C:\language\mdopus\full-test\src\setup.rs:35-41`** — `git rev-parse` silently returns `"unknown"` if the user runs outside a git checkout (e.g. from a tarball). That is fine for reports, but `capture_does_not_panic_and_fields_populate` (line 153) asserts `!info.commit.is_empty()`, which passes both for a real SHA and for `"unknown"`. The test happy-paths through a real repo; it does not exercise the non-repo fallback path. Add a unit test that injects a failing `run_trimmed` (refactor `git_short_sha` to take a closure, or test via `assert!(matches!(git_short_sha(), Some(_) | None))` — weak but at least covers both arms).
