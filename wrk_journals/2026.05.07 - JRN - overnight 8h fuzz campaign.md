### 00:08 BST (Fri 8 May) — SILK LBRR fix applied + r3 workers relaunched
Diagnosis agent (39 min, 219 tool uses) returned **HIGH-confidence**
root cause:

**Mechanism**: Rust's `silk_encode_indices` takes `&SilkEncoderState`,
so it can't mutate `ec_prev_signal_type`/`ec_prev_lag_index` inside.
The regular-frame path compensates at the end of `silk_encode_frame_fix`
(`encoder.rs:7706-7709`), mirroring C's `encode_indices.c:141,174`
in-function mutation. The **LBRR-data-write loop** (`encoder.rs:8340-
8374`) calls `silk_encode_indices` with `indices_lbrr[i]` and never
reproduces the mutation. So packet N+1's *second* LBRR sub-frame
(`cond_coding == CODE_CONDITIONALLY`) reads stale ec_prev values from
packet N's regular-frame write — 4-bit divergence in the
`delta_lag_index` icdf write.

**Fix applied** (`ropus/src/silk/encoder.rs:8367-8378`, +13 lines):
After the LBRR-write `silk_encode_indices` call, replicate C's
in-function mutation using `indices_lbrr[i]`:
```rust
let lbrr_signal_type = enc.state_fxx[n].s_cmn.indices_lbrr[i].signal_type;
let lbrr_lag_index   = enc.state_fxx[n].s_cmn.indices_lbrr[i].lag_index;
if lbrr_signal_type == TYPE_VOICED as i8 {
    enc.state_fxx[n].s_cmn.ec_prev_lag_index = lbrr_lag_index;
}
enc.state_fxx[n].s_cmn.ec_prev_signal_type = lbrr_signal_type as i32;
```
+ comment cite. Cargo.toml patch-bumped 0.12.11 → 0.12.12.

### Session shipped

1. **PLC short-frame `delay_buffer` contamination** (commit `9b8c552`)
   Rust's PLC short-frame early-return branch in `OpusEncoder::encode_native`

   C's `silk_encode_indices` (`encode_indices.c:141,174`) mutates

   LBRR sub-frame (`cond_coding == CODE_CONDITIONALLY`) read stale
   ec_prev values from packet N's regular frame, producing a 4-bit
   delta-lag-index divergence.

### 23:10 BST — Diagnosis returned, fix applied
Diagnosis agent (538s, 44 tool uses) reported high-confidence root
cause that **invalidated the initial hypothesis**:

- Not a DTX issue. Frame 0 emits 1 byte because bitrate=6000 and
  `3 * frame_rate * 8 = 9600` for 2.5ms frames → falls into the **PLC
  short-frame early-return** branch at `ropus/src/opus/encoder.rs:1708`
  / `reference/src/opus_encoder.c:1340`.
- Frame 1's bitrate (71535) does NOT take the early return — it goes
  through real CELT encode. C produces a tiny 3-byte packet because
  CELT range-codes a near-empty payload at this signal/bitrate combo.

  pcm, frame_size);` — C's matching branch (`opus_encoder.c:1396-1405`)
  has no such write. The spurious write contaminates `delay_buffer`,
  so frame 1's CELT encode sees a different filter history than C.

Side note: line 1777 also sets `self.range_final = 0;` while C in this
branch never touches `st->rangeFinal`. Latent parity deviation; not
the crash trigger; tracking for follow-up but **not bundling** into
this fix.
