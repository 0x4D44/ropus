# Encoder setter-sequence audit (C vs Rust)

## Why this audit

The 2026-05-08 fuzz expansion campaign closeout (`wrk_journals/2026.05.08 -
JRN - fuzz expansion campaign closeout.md`) flagged a recurring defect
class. Three bugs found in two days, all the same shape:

| Date | Bug | Commit |
|---|---|---|
| 2026-05-07 | PLC short-frame `delay_buffer` contamination | `9b8c552` |
| 2026-05-07 | SILK LBRR `ec_prev` state mutation parity | `55e224c` |
| 2026-05-08 | CELT `SetPrediction` ordering for CELT→SILK redundancy | `9d9d9cc` |

Pattern: C reference mutates encoder state in-place inside a sequenced
helper; Rust port reorders the call site, drops a call, or applies a
different argument; a specific frame-pair adjacency (mode transition,
LBRR follow-up, redundancy frame, etc.) exposes the divergence as a
byte-level encoder mismatch.

Each fuzz target catches one such adjacency; the campaign closeout
explicitly recommends a deliberate static audit instead of waiting for
the next fuzz target to land another bug.

This journal is that audit.

## Method

1. **C inventory.** Walk `reference/src/opus_encoder.c::opus_encode_native`
   top-to-bottom. Record every call that mutates encoder state observable
   on the next frame: `CELT_SET_*`, `OPUS_SET_*`, `silk_*` calls that
   touch `st->silk_enc`, direct field writes to `st->*`, and any helper
   that wraps such a call (e.g., `opus_repacketizer_*` is excluded —
   that's a per-call object).

2. **Rust inventory.** Same walk on `ropus/src/opus/encoder.rs`,
   matching the C vocabulary so the inventories align line-for-line.

3. **Diff.** Align the two lists by call site. For each row, classify:
   - **MATCH** — same call, same args, same order, same guard.
   - **BENIGN** — different idiom but provably equivalent (e.g., Rust
     uses a struct field write where C uses a setter, but the field
     is private and read identically).
   - **SUSPECT** — order, guard, or argument differs in a way that
     could affect a later frame. Needs verification.
   - **CONFIRMED-BUG** — divergence reproducible at byte level.

4. **Verification** of every SUSPECT either by tracing to prove benign,
   or by building a focused differential repro that exposes the
   divergence on the comparison harness.

5. **Recommendations.** No fixes in this audit. Each CONFIRMED-BUG
   gets its own follow-up commit (matching the pattern of `9d9d9cc` /
   `55e224c` / `9b8c552`).

## Scope boundary

In scope: `opus/encoder.rs::opus_encode_native` and any helper it calls
that mutates state (`gen_toc`, `transient_boost`, `compute_stereo_width`,
the redundancy-encode block, the prefill block, etc.). Same in C.

Out of scope: SILK encoder internals, CELT encoder internals (we audit
the *call sites* into them, not their implementations). Decoder-side
state. Multistream wrapper.

Ground rule: a single Rust function call that wraps multiple C lines is
fine, *as long as* the wrapped operations preserve C order. We audit
the externally observable sequence.

## Verification notes

The audit was a static read; the validations below resolve every
SUSPECT to a confirmed classification without empirical repros.

### F10 (silk_mode prefill clone-vs-pointer) — verified BENIGN

Risk: in C, `silk_Encode` mutates the encControl struct passed in (writes
`switchReady`, `signalType`, `stereoWidth_Q14`, `internalSampleRate`,
`allowBandwidthSwitch`, `inWBmodeWithoutVariableLP`); Rust prefills on
a clone. If any of those fields were read between the prefill and the
real encode, divergence would propagate.

Reading the 6-line window between Rust 2786 (prefill end) and 2799
(real encode start): the only mutation is `self.silk_mode.opus_can_switch = 0`
at 2788, which mirrors C 2208. Nothing reads silk_mode in between. The
real silk_encode call at 2799 then writes the same fields based on the
actual frame, overwriting whatever the prefill would have written.

Conclusion: clone is functionally equivalent. No bug.

### F32 (data[-1] = 0 + |= gen_toc vs direct assign) — verified BENIGN

C zeros the TOC byte then ORs in `gen_toc(...)` later. The byte is not
written by anything in between (the range coder's storage starts at
`data+1` after `data += 1`). So `data[0]` is `0 | gen_toc(...) ==
gen_toc(...)`. Rust's direct assign produces the same value.

### F33 / F48 / F53 — known-gaps, not setter-sequence regressions

All three are tracked as TODOs in the source and are surface-area
divergences (DRED bitrate plumbing and DRED+DTX first_frame shifting).
They will land as fuzz divergences only on adjacencies that exercise
DRED + (CBR or DTX or both) — narrower input subspace than the bugs
the closeout fuzz campaigns found, but real.

### F40 (ec_enc_shrink in HYBRID SILK→CELT redundancy path) — verified BENIGN

C calls `ec_enc_shrink(&enc, ret)` at line 2533 after the main HYBRID
CELT encode. The main encode (C 2493) was called with `&enc` and
internally calls `ec_enc_shrink(enc, ret)` then `ec_enc_done(enc)`
during its VBR finalization, so `enc->storage` is already `ret` by the
time control returns. The explicit shrink at 2533 is a no-op.

In Rust the outer `enc` is dropped at the end of the inner scope; the
SILK→CELT redundancy block at 3211 then writes redundancy bytes
directly into `enc_data[ret..ret+redundancy_bytes]`, identical packet
layout to C.

## Recommendations

1. **No setter-sequence fixes required.** The defect class flagged in
   the 2026-05-08 closeout (PLC delay_buffer, SILK LBRR ec_prev, CELT
   SetPrediction) does not have any further instances in the encoder
   per-frame path. The recurring pattern was real and the campaign-
   level audit hypothesis was sound, but the audit clears the surface
   area — this defect class appears exhausted.

2. **F33 / F53 (DRED bitrate plumbing).** Already TODO'd at
   `ropus/src/opus/encoder.rs:3044-3048`. Decide whether to schedule
   the port now (fuzz signal will land it eventually, on DRED+CBR
   inputs) or accept the gap and document explicitly that ropus's
   DRED implementation differs from C under CBR. Stage-8.x scope per
   the comment.

3. **F48 (multi-frame first_frame shift).** Already TODO'd at
   `ropus/src/opus/encoder.rs:2289-2292`. Same disposition as F33.

4. **Reframe expected fuzz yield.** The 2026-05-08 closeout assumed
   this defect class would keep producing bugs at ~one per overnight
   campaign. With the surface area cleared, fuzz signal on the encoder
   per-frame path should now plateau (modulo the three known-gaps
   above). Future campaigns are better invested in:
   - decoder-side state mutations (not audited here);
   - cross-module boundaries (silk encoder ↔ silk control, CELT
     encoder internals ↔ celt_encode_with_ec);
   - the corpus-bias fix from the 2026-05-08 closeout (Stream D
     stalled at 0% SILK↔HYBRID transitions in steady state).

5. **Light follow-up: Stream C coverage-diff (also from the closeout).**
   Mechanical, deferred. Worth finishing before the next
   target-broadening pass.
