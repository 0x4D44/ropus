# 2026-04-19 — AVX2 baseline bump (x86-64-v3)

## Context

Earlier today the LTO×PGO bake-off (`2026.04.19 - JRN - lto-pgo-bakeoff.md`)
landed ThinLTO-without-PGO as the release profile. That got the encoder to
~1.19× C reference mean. Arthur observed — correctly — that AVX2 has been
around since Haswell (Q2 2013). We were building for `x86_64` (SSE2 baseline
from 2003) and leaving 13 years of ISA extensions on the floor.

This journal: flip the baseline to `x86-64-v3` (AVX2 + FMA3 + BMI1/2 + F16C
+ LZCNT + MOVBE) and measure. No source changes — purely a `.cargo/config.toml`
addition.

## Change

New file `.cargo/config.toml`:

```toml
[target.'cfg(target_arch = "x86_64")']
rustflags = ["-C", "target-cpu=x86-64-v3"]
```

Scope: applies to all workspace crates built from this tree. The C reference
linked into `harness` via the `cc` crate is unaffected — `cc` sets its own
`-march` and doesn't pick up `RUSTFLAGS` — so the comparison is still
apples-to-apples against xiph/opus built at its own default microarchitecture
level (baseline x86_64 with runtime SSE4.1/AVX2 dispatch inside libopus).

## Methodology

Ran on a git worktree at HEAD `5e37d9c` (clean, unaffected by a concurrent
agent's uncommitted `ropus-fb2k` work):

```
git worktree add /c/tmp/ropus-avx2 HEAD
```

Baseline: the checked-out tree as-is, x86-64 generic target, ThinLTO no-PGO.
Treatment: add the `.cargo/config.toml` above, clean rebuild.

For each config:
1. `cargo build --release --bin ropus-compare` (clean rebuild, 18–20s)
2. `cargo test -p ropus --lib --release` (confirm bit-exact tests still pass)
3. `bash tools/bench_sweep.sh --iters=30` across all 10 vectors

Same binary, same vectors, same machine (AMD Ryzen 9 9950X3D, Zen 5), back-
to-back runs.

## Analysis

### Encode: −17pp mean, catastrophic-scale wins on SIMD-bound vectors

Every single vector improved. Four now encode *faster* than the C reference
(sine 0.94×, sweep 0.96×, SPEECH 0.84×, MUSIC hovers at parity 1.01×). The
−17pp mean is larger than Stages 1+2+3 of indexing work combined delivered.

Biggest wins are on the most SIMD-bound vectors:
- SILK WB 1.38 → 1.14: −24pp. SILK LPC analysis, NSQ, and LTP filters all
  benefit from FMA + wider autovec.
- Hybrid 24k: −22pp (mix of SILK + CELT).
- CELT square: −23pp. Harmonically structured signals are heavy in
  pitch search and quant-bands, both of which autovec nicely at 256-bit.
- SPEECH −35pp: unusually large. Likely a combination of SILK dominating
  (and benefiting from both autovec + FMA) with some favourable instruction
  cache effects that a longer run would average out. Even discounting this
  as a statistical outlier, the trend holds across every vector.

LLVM with AVX2 enabled can now:
- Pair two `i32x4` SIMD ops into one 256-bit AVX2 instruction where data
  flow allows.
- Emit FMA instructions for fixed-point MAC sequences that previously
  compiled as separate multiply-add pairs.
- Use BMI2 `pext`/`pdep` and `mulx` for control-heavy bit-manipulation in
  the range coder.
- Use F16C for any half-float conversions (not used by us but free).

The gain is "free" in the sense that the existing `wide::i32x4` code didn't
change. The wins come from LLVM lowering the same IR to better instructions.

### Decode: −4pp mean, consistent

Decode is dominated by the range decoder and inverse transforms. The range
decoder doesn't vectorize (bit-serial control flow) but does benefit from
BMI2. Inverse transforms (IMDCT) use the same FFT kernels as encode, which
now vectorize at 256-bit.

SILK NB decode went *slightly* backwards (0.672 → 0.694, +2pp). That's within
run-to-run variance at iters=30; not signal.

### Practical consequence

Previously:
- README claimed 1.19× encode / 0.99× decode mean.
- Text described ropus as "at C-parity on decode, encode within ~20%".

After this change:
- 1.02× encode / 0.95× decode mean.
- ropus is now *faster* than C reference on decode across the board and at
  near-parity on encode. Four vectors encode faster than C.
- Several CELT vectors (sine, sweep, SPEECH) encode *and* decode faster
  than C.

This is a qualitatively different story for the README. The codec has moved
from "competitive port" to "equivalent or better" on average.

## Why we were leaving this on the floor

Default `x86_64-pc-windows-msvc` / `x86_64-unknown-linux-gnu` targets have
SSE2 as the ISA baseline because that's what x86_64 the architecture
mandates. Rust inherits that from LLVM. To get anything beyond SSE2 you have
to either:

1. Set `target-cpu` (this change).
2. Use `#[target_feature(enable = "avx2")]` with runtime dispatch.
3. Use `std::arch` intrinsics under `#[cfg(target_feature = "...")]`.

libopus C does (2) — runtime CPU detection with per-ISA code paths. We do
(1), which is simpler, gets most of the benefit, and assumes the user is
on a 2013-or-later machine. Pre-Haswell users can override with
`RUSTFLAGS="-C target-cpu=x86-64-v2"` or `=x86-64`.

AVX-512 is a different story (still not universal — Intel ripped it from
Alder Lake consumer 2021, put it back on Sapphire Rapids server / Zen 4+).
That would need genuine runtime dispatch to be safe. Not in scope here.

## Upper-bound sanity check: `target-cpu=native` on Zen 5

With v3 in hand, Arthur asked the obvious next question: "this is a fancy
machine — what does native give us as an upper bound?" The Ryzen 9
9950X3D has full AVX-512 (Zen 4+), so `target-cpu=native` enables the
512-bit ISA on top of v3.

Same worktree, same methodology — just flipped `target-cpu=native` in
`.cargo/config.toml`, clean rebuild, back-to-back bench sweep at iters=30.

Result: **native is worse than v3 on the mean.** Encode went from 1.020
→ 1.075 (+5pp). Decode is flat. Per-vector:

```
┌──────────────────────────┬────────┬─────────┬──────────┐
│ Vector                   │   v3   │ native  │  Δ v3→n  │
├──────────────────────────┼────────┼─────────┼──────────┤
│ SILK NB 8k mono noise    │ 1.052  │  1.041  │  −1pp    │
│ SILK WB 16k mono noise   │ 1.140  │  1.210  │  +7pp ←  │
│ Hybrid 24k mono noise    │ 1.106  │  1.147  │  +4pp ←  │
│ CELT FB 48k mono noise   │ 1.080  │  1.101  │  +2pp    │
│ CELT FB 48k stereo noise │ 1.042  │  1.040  │   0pp    │
│ CELT 48k sine 1k loud    │ 0.939  │  0.912  │  −3pp    │
│ CELT 48k sweep           │ 0.964  │  0.936  │  −3pp    │
│ CELT 48k square 1k       │ 1.031  │  1.069  │  +4pp ←  │
│ SPEECH 48k mono (TTS)    │ 0.838  │  1.071  │ +23pp ←← │
│ MUSIC 48k stereo         │ 1.009  │  0.923  │  −9pp    │
├──────────────────────────┼────────┼─────────┼──────────┤
│ Mean                     │ 1.020  │  1.075  │  +5pp    │
└──────────────────────────┴────────┴─────────┴──────────┘
```

5 vectors regressed, 4 improved, 1 flat. The SPEECH regression at +23pp is
by far the biggest single swing — big enough that I ran the bench twice to
rule out warm-up variance, and the result held. SILK WB, Hybrid, and CELT
square also consistently regress.

Two plausible causes, both acting together:

1. **AVX-512 frequency downclocking.** Zen 5 is less aggressive than Intel
   about dropping clocks on 512-bit ops, but it still does. If the codec
   can't keep the 512-bit units saturated enough to amortize the clock hit,
   net throughput goes down. The codec's hot paths — pitch search inner
   loops, SILK NSQ, range coder — have short iteration counts and
   data-dependent chains that don't produce 8–16 independent lanes of
   work per cycle.

2. **LLVM over-widens.** With AVX-512 enabled, LLVM will auto-vectorize
   loops to `zmm` registers even when there isn't enough parallel work,
   leaving lanes dead and burning clock cycles anyway. Short loops (which
   Opus has a lot of) are the classic footgun.

MUSIC and CELT sine/sweep are the vectors that benefit — these are the
longest-running kernels with the most uniform data flow, exactly where
512-bit has enough work to fill.

### Implication for `i32x8` widening

The pre-bench plan was: land v3, then widen `xcorr_kernel_simd` from
`i32x4` to `i32x8` for an explicit AVX2 256-bit path. The native
experiment is evidence this isn't worth doing either. Going from 128-bit
to 256-bit SIMD has the same pathology as going from 256-bit to 512-bit,
just scaled down: the kernel has to have enough independent work to
justify the wider register, and if it doesn't, you pay overhead without
throughput gain. LLVM at AVX2 will already widen to 256-bit where that
math works out; forcing it via `i32x8` in places where LLVM judges 128-bit
is better is likely to regress, not help.

Skipping the widening. v3 is the sweet spot for this codec on x86_64.

## Verdict

**Land `.cargo/config.toml` with `target-cpu=x86-64-v3`. Stop there.**

- v3 delivers −17pp encode / −4pp decode from a single config line.
- Native (AVX-512) regresses by +5pp on mean encode — not an upper bound
  worth chasing.
- Explicit `i32x8` widening is likely to behave like AVX-512: neutral to
  negative. Skipping.
- Further performance work, if pursued, would have to come from
  algorithmic changes (pitch search restructuring, different rate
  allocation search) or hand-written architecture-specific intrinsics.
  Neither is in scope for this workstream.
