## Stage 6 — Analysis / tonality — survey and pause

**Survey agent findings.**

1. Port size: `reference/src/analysis.c` (982 lines) + `mlp.c` (131) + `mlp_data.c` (671-line static weights blob). ~5–6 days of careful work.
2. **Bit-exactness against C reference is infeasible.** Analysis depends on `tansig_approx`, `sqrt`, `log`, and FMA-fused MLP matmul. Expected drift: 1e-6 to 1e-4 per frame at the numerical layer. That drift flips classifier boundaries at edge frames → flips mode decisions → diverges bitstream irrecoverably.
3. **No unit-test oracle in reference.** All validation is end-to-end encode comparison — which is exactly the surface that diverges.

**Decision with Arthur.** The original HLD's universal bit-exact oracle was a drafting error for Stages 6–8 (all three are float-heavy DNN-dependent ports). Rather than pretend bit-exactness is achievable or defer the stages entirely, we're revising acceptance to relaxed, layer-appropriate oracles:

- **Numerical layer**: tolerance-based where we care.
- **Decision layer**: agreement-rate on classifier outputs.
- **Audio layer**: SNR against C decoded PCM.
- **Bitstream layer**: abandoned for float-dependent stages; kept bit-exact for fixed-point sub-layers (DRED entropy coder, non-DNN control paths).

Plus a pre-work `mul_add` audit of `ropus/src/dnn/core.rs` — if C auto-fuses and we don't, switching to `f32::mul_add` could close an order of magnitude of drift for free.

**Revised HLD written.** `wrk_docs/2026.04.19 - HLD - stages-6-8-relaxed-oracles.md` supersedes the Stage 6–8 acceptance criteria only. Stages 0–5, 9 are unaffected. Priority order unchanged. Stage 8 still checkpoint-gated.

### Audit finding — revised HLD v2

Before starting Stage 6 I ran the pre-work `mul_add` audit. The result reversed the v1 framing:

- **`ropus/src/dnn/`**: zero uses of `mul_add` across 10,318 lines — Rust is already discrete multiply-then-add throughout.
- **`reference/dnn/vec.h`**: scalar `sparse_sgemv8x4` uses `y[k] += w[k]*xj` — identical pattern to our Rust.
- **`reference/src/mlp.c:38`**: `#define fmadd(a,b,c) ((a)*(b)+(c))` — semantically multiply-then-add, not a fused intrinsic.
- **`harness/build.rs`**: passes no `-mfma` / `-mavx` / `-march=native` / `-ffp-contract=*` flags. MSVC default `/fp:precise` already prohibits contraction; a one-line `-ffp-contract=off` locks gcc/clang down equivalently.
- **`harness/config.h:23`**: `#define DISABLE_FLOAT_API 1` — analysis.c / mlp.c / mlp_data.c compile but are short-circuited today. Stage 0–5 byte-exactness worked precisely because analysis was disabled symmetrically on both sides.
- **Harness does not build any DNN C files** — nnet.c, fargan.c, nnet_default.c absent from build.rs. Stage 7 requires adding them.

Conclusion: the v1 HLD's "bit-exact is infeasible" claim was overstated. It conflated scalar-C with SIMD-C paths. Under scalar-C harness config with `-ffp-contract=off`, bit-exactness for Stages 6–8 is plausible.

**Critical negative finding.** Do NOT add `f32::mul_add` to Rust DNN code. That was v1's pre-work plan; the audit shows it would create drift against the scalar-C path we actually compare against, not close it.

**HLD v2 written.** Same filename (`stages-6-8-relaxed-oracles.md`), full rewrite. New structure:

- Tier 1: bit-exact under scalar-C config (attempted first).
- Tier 2: original tolerance-based oracles (fallback if tier 1 demonstrably fails, with journal justification).
- New pre-work: `-ffp-contract=off` in `harness/build.rs` + unset `DISABLE_FLOAT_API` in `harness/config.h` (1-commit prep step).
- Removed `mul_add` audit from the plan.
- Stage 7 prep adds nnet.c + nnet_default.c + fargan.c to build.rs (scalar only, no vec_avx.h / vec_neon.h).
- Stage 8 DRED entropy coder never leaves tier 1 (fixed-point integer).
