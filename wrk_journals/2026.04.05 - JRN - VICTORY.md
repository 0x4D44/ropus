## Top 3 Insights

### 1. Constants matter more than arithmetic
The single biggest win (+31 tests) was a CONSTANT value wrong by 23 out of 2^31.
The `qconst32` function used f64 precision but C's QCONST32 macro with 'f' suffix 
uses f32. Individual arithmetic functions were all bit-exact — the devil's advocate 
was right to redirect us from arithmetic debugging to constant/initialization issues.

### 2. "Bit-exact in isolation" is insufficient
Multiple functions verified as "bit-exact" individually still produced wrong output 
together. The Burg LPC was bit-exact, the NSQ was bit-exact, the noise shape was 
bit-exact — but `silk_find_lpc_fix` had a wrong slice bound (`2*subfr_length-d` 
instead of `2*subfr_length`) that changed the interpolation search result.

### 3. C-to-Rust porting traps are systematic
- f32 vs f64 constant precision
- Pointer arithmetic → index arithmetic translation errors
- C's implicit i16 truncation in macros (silk_SMLAWB)
- Struct field scoping (res_nrg_Q15 from "last iteration" vs "best iteration")
- memcpy-style state save/restore vs Rust clone semantics
