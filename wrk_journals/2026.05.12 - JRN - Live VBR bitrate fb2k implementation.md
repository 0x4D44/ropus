## Stage A — Rust FFI + reader + tests

### Implementation (agent #1)

- Extended `ropus_fb2k_decode_next` signature with `uint64_t* out_bytes_consumed`.
- `Reader::decode_next` now returns `Result<(i32, u64), ReaderError>`; accumulates `pkt.data.len()` across the seek-pre-roll loop.
- 15 test call-sites updated in `roundtrip.rs` (HLD said 14; minor miscount).
- New TDD test `decode_next_reports_bytes_consumed_correctly`: opens a fixture, drains via FFI summing bytes, cross-checks against `ogg::PacketReader`. Initially red (E0061 arity mismatch), then green.
- Clean build, 67/67 tests green, 68/68 with `--features test-panic`.

### Review

**Code-reviewer (skill):** GREEN. Two nits — ambiguous null-pointer message (skipped: not worth touching); optional post-seek bytes test (skipped: not in HLD §5.1, deferred to smoke test).

**Devil's advocate:** YELLOW. Real finding: **EOF spec/code drift.** HLD §4.2 promises `bytes_consumed == 0` at EOF; implementation returned the accumulator (could be non-zero after pre-roll-then-EOF). The C++ side never reads it on samples==0 so functionally moot, but the spec contradicts the code. Also flagged: test was tautological on the EOF case; `saturating_add` was theatre.

### Apply feedback (agent #3)

- EOF path now forces `bytes_consumed = 0` at `reader.rs:491`. HLD wins.
- Doc comments at `ropus_fb2k.h:105` and `lib.rs:401` rewritten: "On EOF (return 0), `*out_bytes_consumed` is set to 0. On error (negative return), it is not written."
- Test now captures rc==0 call's bytes_consumed into a dedicated local and asserts `== 0` post-loop.
- `saturating_add` → plain `+=` (`reader.rs:489`).
- Build clean, 67/67 + 68/68 (test-panic) green.

### Outcome

Stage A complete. Committed as `ec12e38`.

## Stage B — C++ shim + README

### Implementation (agent #5)

- Two new members: `m_smoothed_bps`, `m_reported_kbps`.
- `decode_initialize` seeds EWMA from `m_info.nominal_bitrate`, resets reported.
- `decode_run` updates EWMA after success (cold-start branch for unseekable streams).
- `decode_get_dynamic_info` override returns rounded kbps on change.
- `decode_next` call site threads `uint64_t bytes_consumed = 0;`.
- README "Verify" section gets one sentence noting the live readout.
- `pwsh -File foo_ropus\build.ps1` clean, `foo_ropus.fb2k-component` produced.

### Review

**Code-reviewer:** GREEN. One non-blocking nit on `std::lround` style — skipped (matches existing rounding at line 217).

**Devil's advocate:** YELLOW with a real bug. Stage A's accumulator counts bytes from packets fully discarded as post-seek pre-roll (`if kept == 0 { continue; }` branch), but Stage B's EWMA divides those bytes by *kept* samples. Result: first `decode_run` after a seek with 80 ms pre-roll produces `inst_bps` ~5× reality; α=0.02 carries the spike for ~1 s. Caught the HLD itself was internally inconsistent — §4.2 parenthetical said "bytes that map to returned samples" (correct), §4.3 prose said "every loop iteration" (the bug). §4.2 wins because the whole purpose of the FFI is the bytes/samples ratio.

### Apply feedback (agent #7)

- `reader.rs`: accumulator moved *after* the `if kept == 0 { continue; }` guard so pre-roll packets contribute zero bytes. Comment notes the ordering is load-bearing.
- FFI doc + Rust wrapper doc rewritten: bytes-consumed is now described as "bytes that produced the returned samples; pre-roll packets do NOT contribute".
- HLD §4.3 rewritten to match §4.2; §4.4 gains a final paragraph noting the EWMA's correctness depends on §4.3.
- New regression test `decode_next_excludes_preroll_bytes_after_seek` — opens a 30-packet fixture, captures no-seek baseline (1 packet), seeks to sample 10000, asserts post-seek `bytes_consumed ≤ 2× baseline`. With the bug present this would be ~5×. PASSES.
- Existing test still passes (pre-skip = 312 of 960 leaves kept > 0 on the first packet, so bytes still count).
- `cargo test`: 68/68 green. `foo_ropus` rebuild clean.

### Outcome

Stage B complete. Committed as `8cdc252`.
