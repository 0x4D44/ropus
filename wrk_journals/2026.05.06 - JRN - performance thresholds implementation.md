# Performance Thresholds Implementation Journal

Date: 2026.05.06

## 00: Initial Read

Read the signed HLD, HLD signoff, remaining-gaps plan, and current
`full-test` benchmark/banner/report code before editing.

Design choice: keep this as testing infrastructure only. No codec code, no
vector expansion, no dependency churn. I am adding a typed benchmark profile and
using a Rust const threshold table so unit tests can exercise the policy with
synthetic rows.

Emotional temperature: focused but cautious. The main risk is accidentally
turning normal local benchmark noise into a hard failure; the signed slice is
clear that only the release-thresholded profile should be blocking.

## 01: Implementation Shape

Added a typed benchmark profile:

- `not-claimed` for quick or local benchmark-skipped runs.
- `observed-warn-only` for default benchmark runs.
- `release-thresholded` for non-quick release preflight, even if
  `--skip-benchmarks` is passed, so disabling the lane becomes a failure rather
  than a silent downgrade.

Added initial per-vector/per-operation release thresholds in Rust const data.
I used README ratios as the initial calibration source and wrote a separate
measurement note because I do not want to pretend these are fresh multi-sweep
machine baselines. That feels honest and keeps the release gate conservative
without broadening the task.

## 03: Bounded Measurement

I ran one direct harness benchmark smoke rather than a broad full-test dry-run,
to avoid depending on unrelated release-preflight assets:

```text
timeout 300 cargo run --release -p ropus-harness --bin ropus-compare -- bench tests/vectors/48k_sine1k_loud.wav --iters 3 --repeats 1 --bitrate 64000 --color=never
```

Result:

```text
C encode    1.511ms
Rust encode 1.623ms
C decode    0.647ms
Rust decode 5.650ms
encode ratio 1.07x slower than C
decode ratio 8.74x slower than C
```

This was a short `--iters 3` smoke, not a calibrated threshold sweep. It proves
the local reference harness and benchmark path can run, but it should not be
used to replace the initial threshold table. It does flag a residual concern:
the current local single-vector release measurement would breach the initial
decode threshold for `CELT 48k mono sine 1k loud`. I am not weakening the
contract around that; the next reviewer should decide whether this is transient
measurement noise, a debug/setup artifact, or a real performance regression
that needs a proper multi-sweep investigation.

Emotional temperature: mildly uneasy about the single-vector decode result, but
the right response is to preserve the gate and surface the risk, not tune the
threshold to silence it.

## 04: Feedback Application Decision

The review was right: the 8.74x decode smoke could not be left as a vague
concern while the release threshold for the same row is 1.26x.

Decision recorded in the measurement note: the new release-thresholded gate
would correctly fail that live measurement. I am treating it as unresolved
release-blocking evidence, not as a reason to widen the threshold. A clean
release-performance pass should not be claimed until a calibrated `--iters 30`
same-machine sweep either clears the threshold or confirms a decode
regression/environment artifact for follow-up.

I also added the two missing pure tests for release-thresholded missing
threshold specs and crashed benchmark rows.

Emotional temperature: more comfortable now. This is exactly the sort of
unpleasant finding a release gate should expose; writing it down plainly is
better than trying to make the first gate green.
