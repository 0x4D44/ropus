## Architecture Decisions

### 1. Differential fuzzing: FFI bindings in fuzz crate
The fuzz crate needs to call both C and Rust implementations. The C reference is
compiled by the main crate's `build.rs` and linked via `cargo:rustc-link-lib`. We
created `tests/fuzz/fuzz_targets/c_reference.rs` with:
- Minimal FFI declarations (public Opus API only, ~20 functions)
- Safe wrapper functions: `c_decode()`, `c_encode()`, `c_packet_get_*()` 
- NO `#[link]` attribute — symbols resolve from the mdopus dependency's link directives

**Key fix:** Initially the infra agent created a separate `build.rs` for the fuzz crate
that compiled a SECOND copy of opus_ref. This would cause duplicate symbol errors.
Removed the fuzz build.rs entirely — the mdopus dependency's build.rs already provides
the library. Verified: only one `libopus_ref.a` exists after clean build.

### 2. Structured fuzzing for config coverage
Each target uses a config header parsed from the first N bytes of fuzzer input:
- **fuzz_decode**: 2 bytes — `[sr_idx, ch_byte]` + packet data
- **fuzz_encode/roundtrip**: 6 bytes — `[sr_idx, ch, app, br_lo, br_hi, cx]` + PCM
- **fuzz_repacketizer**: no header — raw packet bytes
- **fuzz_packet_parse**: no header — raw bytes

Config bytes select from: 5 sample rates, 2 channel configs, 3 applications,
6000-510000 bitrate range, 0-10 complexity.

### 3. Seed corpus: Python generator
Created `tools/gen_fuzz_seeds.py` (Python, uses `wave` module for WAV reading).
Generates seeds with correct structured format per target.

### 4. Platform limitation: cargo-fuzz requires Linux
libFuzzer sanitizer runtime not available on Windows. `cargo check` passes (all types
correct), but actual fuzz execution needs Linux/WSL/macOS. Documented in the automation
script header.

## Devil's Advocate Analysis

### Critical Issue 1: unchecked-indexing and fuzz detectability

**The problem:** 126 unchecked indexing sites in `fft.rs` (90) and `mdct.rs` (36).
When `unchecked-indexing` is enabled (the default), an out-of-bounds access is UB —
not a panic. Under optimized builds (cargo-fuzz uses `-O2`), UB may:
- Silently read/write garbage (caught by differential comparison if output changes)
- Cause a segfault (caught by libFuzzer as crash)
- Get optimized away entirely (invisible)

**Recommendation:** Fuzz primarily with `unchecked-indexing` DISABLED. This makes all
126 sites produce panics on OOB access, which libFuzzer catches as crashes. Differential
comparison provides a secondary detection layer even with unchecked ON.

**Action needed:** The fuzz Cargo.toml should override mdopus's default features:
```toml
mdopus = { path = "../..", default-features = false, features = ["simd"] }
```
This disables `unchecked-indexing` for fuzzing while keeping SIMD (which uses `wide`
crate, safe Rust). Could add a second fuzz profile with default features for production
path testing.

### Critical Issue 2: C reference crash risk

**The problem:** In differential mode, fuzz_decode feeds random bytes to the C decoder
via FFI. If the C decoder has a bug triggered by a specific input, it crashes the
entire fuzzer process (segfault through FFI is unrecoverable).

**Assessment:** xiph/opus is one of the most-fuzzed C projects (Google OSS-Fuzz since
2016). The probability of a C decoder crash on random input is extremely low. The C
code returns error codes for malformed packets.

**Mitigation:** None needed. The risk is theoretical. If it does happen, the crash
artifact is saved by libFuzzer, and we can reproduce and report upstream.

### Issue 3: Missing stateful encoder fuzzing

**The problem:** All encode/roundtrip targets create a fresh encoder per fuzz input.
Real-world usage encodes many sequential frames, building up prediction state. Bugs
in state persistence (carry-over between frames) are invisible to single-frame fuzzing.

**Recommendation:** Future work. Add a `fuzz_encode_multiframe` target that encodes
2-3 sequential frames from the same input. Not blocking for initial deployment.

### Issue 4: Missing multistream target

**The problem:** The repacketizer is tested but multistream encode/decode is not.
Multistream has its own parsing logic and configuration.

**Assessment:** Lower priority than the core codec paths. The multistream layer is
relatively thin (dispatches to single-stream encoder/decoder). Worth adding later.

### Issue 5: Seed corpus effectiveness

**Observation:** The decode seeds are synthetic TOC+random-payload constructions.
LibFuzzer's coverage guidance will find interesting code paths regardless, but
starting from REAL encoded packets (from the existing comparison harness) would give
the fuzzer a better starting point.

**Recommendation:** If sustained fuzzing reveals slow coverage growth, generate seeds
by encoding the WAV vectors through the C reference and saving the packet bytes. This
gives structurally valid packets that exercise deeper codec paths from the start.

### Issue 6: Platform limitation (Windows)

**Observation:** cargo-fuzz requires sanitizer runtime (ASan/coverage instrumentation)
which is only available on Linux and partially on macOS. Windows development environment
cannot do full fuzz builds.

**Mitigation:** Documented in `tools/fuzz_run.sh` header. `cargo check --bins` verifies
correctness on Windows. Actual fuzzing runs on Linux (CI or manual).

### New actionable item from DA: C reference ASAN instrumentation — DONE

When cargo-fuzz enables ASAN for the Rust code, the C reference (compiled by the `cc`
crate in mdopus's build.rs) previously would not get ASAN instrumentation. This meant
C-side memory errors would be raw segfaults rather than sanitizer reports, making them
indistinguishable from Rust bugs in differential fuzzing.

**Fix applied:** `tests/harness/build.rs` now detects cargo-fuzz context via
`CARGO_CFG_FUZZING` env var + confirms ASAN is active via `CARGO_ENCODED_RUSTFLAGS`.
When both are set, adds to the cc build:
- `-fsanitize=address` — C-side AddressSanitizer instrumentation
- `-fno-omit-frame-pointer` — better stack traces
- `-fsanitize-coverage=inline-8bit-counters,pc-table,trace-cmp` — libFuzzer coverage
  guidance from C code paths (fuzzer now learns from C execution, not just Rust)

Uses `flag_if_supported` so MSVC and other compilers without these flags silently skip
them. Normal (non-fuzz) builds are completely unaffected — verified by `cargo check`.

The `rerun-if-env-changed` directives ensure the C library is rebuilt when the fuzz
context changes, avoiding stale cached artifacts.
