## Important

1. **Bench prebuild can fail half the sweep silently** — `C:\language\mdopus\full-test\src\bench.rs:168-177` short-circuits with a `BUILD-FAIL` return when `prebuild_ropus_compare()` errors, so a *failed* prebuild is handled cleanly. **But** if the prebuild claims success (`status.success() == true`) yet `target/release/ropus-compare.exe` isn't actually on disk (stale lock, concurrent delete, exotic `CARGO_TARGET_DIR`), every per-vector `Command::new(bin)` hits `ErrorKind::NotFound` and each row renders as `failed to spawn ropus-compare: …` — 10 identical skip rows with no stage-level signal. Cheap fix: after prebuild, `if !bin.is_file() { return BenchResult::skipped("ropus-compare binary missing after build"); }` before the loop. The corresponding test sits one path-function call away.

2. **Bench parser assumes uncoloured box chars; no explicit guard.** `bench.rs:367` rejects any line not starting with `\u{2502}`. Today `harness/src/main.rs:2761 print_bench_report` uses `println!` with no `colored` dep, and `Stdio::piped()` suppresses cargo's auto-colour, so the box chars land verbatim. **However**, `ropus-compare bench` is spawned as the prebuilt binary, not through cargo — `CARGO_TERM_COLOR` doesn't apply. If a future change to harness adds an ANSI prefix to the table rows, or if a Windows console redirecting via `cmd /c` inserts a BOM/CR mid-line, every line fails the `starts_with('\u{2502}')` check and all four timings become `None` — surfaced as "no data", not as "parser drift". Mirror the Phase 2 defence: set `CARGO_TERM_COLOR=never` on the spawn (cheap, defensive) and/or add an ASCII-fallback detector that logs a warning when the box chars are absent but the line contains `C encode` / `Rust encode` / `C decode` / `Rust decode`. Phase 2 CR already flagged the sibling issue for the cargo test parser (item 6) — same class of bug.

3. **Fixture drift risk on the embedded bench stdout.** `bench.rs:403-425` hand-constructs `SAMPLE_BENCH_STDOUT` with `\u{2502}` escapes instead of copying a real stdout capture. The column widths (`{:<12}`, `{:>6}`, `{:>10.3}ms`, `{:>11.0}`) in `harness/src/main.rs:2766-2772` drive the real output; if that formatter changes (e.g. someone widens the Operation column to 16 chars), the fixture stays valid while real runs break silently. Mitigation: either capture real stdout to a text fixture under `tests/fixtures/bench_sample.txt` and `include_str!` it, or add a harness-side test that snapshots `print_bench_report` output and compares to a golden. Low urgency — the format has been stable — but worth calling out.

4. **`cargo run` (Stage 3) vs `cargo build` + exec (Stage 4) asymmetry.** `ambisonics.rs:81-90` uses `cargo run --release -p ropus-harness --bin projection_roundtrip` while bench prebuilds and execs directly. On a cold cache stage 3 will rebuild `ropus-harness` (~1-3 min); stage 4's prebuild then does almost the same work for `ropus-compare`. On a hot cache both are free. Not a correctness issue and HLD §Stages:66 literally specifies the `cargo run --release` form — so this is "by the spec, possibly worth revisiting." Consider: split Stage 3 into `cargo build --release -p ropus-harness --bin projection_roundtrip` + direct exec, same as bench. Saves the cargo round-trip on every run and makes the two stages uniform. If retained, note in the HLD that the ~15s stage-3 budget assumes a warm cache.

5. **Cargo exit-code 101 for build-failure detection is reliable but underdocumented.** `ambisonics.rs:138` treats `exit_code == Some(101)` as build failure, with `detect_build_failure(&stderr)` as belt-and-braces. Cargo has used 101 for "fatal error" (including build failure) since at least 2018 and it's documented in `cargo/src/bin/cargo/main.rs`. But stage 3's harness binary itself also could theoretically `process::exit(101)` (it doesn't — `projection_roundtrip.rs:575,578` use 0 and 1 only). Worth a one-line comment at `ambisonics.rs:137` pointing at the cargo source or `cargo --help` so a future reader doesn't audit exit codes from scratch. Same for bench — `prebuild_ropus_compare` at `bench.rs:228-252` relies on `status.success()` only, not exit-code disambiguation, which is fine since there's nothing downstream of prebuild that runs iff build-failed.

## Nits

6. **`parse_ms` accepts trailing garbage without `ms`** — `bench.rs:394-397`. `parse_ms("42.1xyz")` strips nothing and `parse().ok()` returns `None`, which is correct; but `parse_ms("42.1ms junk")` also returns `None` because `.strip_suffix("ms")` fails on the non-ms tail. Docstring claims the function "parse[s] a `\"1234.567ms\"`" — accurate, but the `"Trailing garbage without `ms` falls through to `parse()`"` comment at `bench.rs:466-468` is misleading: `parse()` *never* accepts trailing non-numeric text; the fallback does nothing useful. Tighten the comment.

7. **`BenchResult::all_passed() → true` unconditionally** is correct per HLD §PASS/FAIL/WARN:99 ("never blocks a PASS … a WARN is 'look at this, but the codec is fine'"). But `bench.rs:143` deserves a pointer to the HLD section — a future reader may reasonably think this is a bug. The inline doc-comment at 140-142 is close but could cite `HLD § PASS / FAIL / WARN` explicitly.

8. **`OrderOutcome.order: u8` is tight.** `ambisonics.rs:35` uses `u8`. Real fixtures are 1-5; u8 overflow is a non-risk. But `parse_order_line` returns `None` on any `order > 255`, quietly dropping a hypothetical "order 300" line rather than surfacing malformed input. Fine for now. If the harness ever expands to higher orders, revisit.

9. **Zero-ms C baseline handling is right.** `build_vector_row` at `bench.rs:326-333` folds `c > 0.0` into the ratio computation: `ratio = None` iff either side is missing or `c_ms == 0.0`. `None` over `f64::NAN` is the right signal — NaN comparisons are a footgun and the downstream HTML report has to special-case non-finite anyway. Test `build_vector_row_with_zero_c_time_leaves_ratio_none` at `bench.rs:489-502` pins the contract. No change.

10. **Skipped-stage-with-vectors envelope shape.** `BenchResult::skipped()` returns `vectors: Vec::new()`; `main.rs:83-94` always takes the skipped-constructor path and never preserves partial vectors. Cannot have a stage with `skipped: true` and a non-empty `vectors` list. Clean design. The report-layer assertion at `report.rs:528-545` pins this. No change.

11. **Orphaned docstring at `ambisonics.rs:165`** mentions "the `=== SUMMARY ===` block" but `parse_summary` only greps for the `PASS:` marker and per-order lines — it doesn't actually look for the `=== SUMMARY ===` header. Test `only_missing_marker_defaults_to_fail` at 330 confirms. Fine as-is; update the docstring to say "the per-order + overall marker lines in the stdout stream" to match.
