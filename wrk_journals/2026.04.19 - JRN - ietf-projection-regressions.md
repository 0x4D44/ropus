## Piece A ‚Äî implementation

**Start:** 2026-04-19.

Sources inspected:
- `reference/src/opus_demo.c` ‚Äî confirmed `opus_decode24` (single-stream,
  not the multistream alias) is the live code path. The `#if 0` block at
  lines 306-333 that would alias `opus_decode24 -> opus_multistream_decode24`
  is disabled in the checked-in source. HLD's "opus_multistream_decode24
  every decoded packet" is slightly off ‚Äî opus_demo calls the single-stream
  `opus_decode24`. Both wrappers are still in scope per HLD.
- `ropus/src/opus/decoder.rs` ‚Äî `decode24(&mut [i32], ...)` already exists
  as a public method. Single-stream capi wrapper is a trivial delegation.
- `ropus/src/opus/multistream.rs` ‚Äî has `decode(&mut [i16], ...)` but no
  `decode24`. Need to add one (decode streams to i16 buf then route to i32
  output via <<8, mirroring C's `opus_copy_channel_out_int24`).
- `reference/celt/arch.h:148-165` ‚Äî with `FIXED_POINT` on and
  `ENABLE_RES24` off (our config), `opus_res == opus_int16`. But
  `opus_decode24`'s public C signature is explicitly `opus_int32*`, so the
  capi wrapper must take `opus_int32*` regardless of `opus_res`.

**Known deviations from HLD:**
- HLD said `opus_demo.c` calls `opus_multistream_decode24` on every
  packet via the `#define opus_decode24 opus_multistream_decode24`
  alias. That aliasing block (`#if 0 ... #endif` at opus_demo.c:306-333)
  is disabled in the checked-in upstream source, so opus_demo actually
  calls the single-stream `opus_decode24`. Both wrappers are still
  added per HLD (and both have TDD tests), but the multistream wrapper
  is unused by the IETF test path.
- Added stubs for DRED API (not in HLD). opus_demo calls
  `opus_dred_{decoder_create,alloc,parse,free,decoder_destroy}`
  unconditionally (not guarded by `ENABLE_DRED`), so the linker needs
  symbol definitions. Stubs return NULL / 0 / UNIMPLEMENTED and are
  inert on the decode-only path with valid bitstreams.
- Added `opus_encode24` to capi (not in HLD). opus_demo's encode arm
  references it; even though the `-d` path never calls it, the linker
  still needs the symbol. Implemented via the INT24TORES conversion
  the C reference uses (SAT16(PSHR32(a, 8))) rather than stubbed, so it
  round-trips correctly if someone later runs opus_demo with `-e`.
