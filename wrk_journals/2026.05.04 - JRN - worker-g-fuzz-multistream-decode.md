## Root Cause

Harness-side state comparison with `diff_fuzz_decode` showed the decisive mismatch after the PLC subframe:

- C CELT `last_frame_type = 3` (`FRAME_PLC_PERIODIC`)
- Rust CELT `last_frame_type = 4` (`FRAME_PLC_NEURAL`)
- C fuzz reference is fixed-point with `ENABLE_DEEP_PLC` off.
- Rust had embedded DNN weights loaded, and the nested `CeltDecoder` defaulted `complexity = 5`, even though the public `OpusDecoder` default `get_complexity()` is `0`.

So a fresh Rust `OpusDecoder` could take neural CELT PLC by default while the C reference took classical periodic CELT PLC. This is a real decoder parity bug, not a harness issue and not the deferred SILK recovery class.

Fix: initialize the nested `CeltDecoder` complexity to `0` inside `OpusDecoder::new()` so the CELT PLC gate matches the public Opus decoder complexity until callers explicitly set complexity.

Regression handling: copied the exact artifact to `tests/fuzz/seeds/fuzz_multistream/regression-ms-decode-celt-plc-neural-gate.bin` and updated the deterministic multistream asymmetry test to replay it from tracked seeds.
