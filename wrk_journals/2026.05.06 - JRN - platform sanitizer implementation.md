## 00:08 - Generic Profile Proof Probe

Before implementing the generic x86_64 lane, I checked Cargo/rustc behavior
instead of assuming `RUSTFLAGS` overrides `.cargo/config.toml`.

Commands:

```text
timeout 120 env CARGO_TARGET_DIR=target/platform-breadth/probe-default cargo rustc -p ropus --lib -- --print cfg
timeout 120 env CARGO_TARGET_DIR=target/platform-breadth/probe-generic RUSTFLAGS='-C target-cpu=x86-64' cargo rustc -p ropus --lib -- --print cfg
```

Result:

- Default config reported v3 features including `avx`, `avx2`, `bmi1`,
  `bmi2`, `f16c`, `fma`, `lzcnt`, `movbe`, `popcnt`, `sse3`, `sse4.1`,
  `sse4.2`, `ssse3`, and `xsave`.
- The generic `RUSTFLAGS='-C target-cpu=x86-64'` probe reported only `fxsr`,
  `sse`, and `sse2` as CPU features.

This is enough evidence for this slice that the proposed command represents
the intended generic profile on this host. Emotional temperature: focused, with
some healthy suspicion about Cargo flag precedence now resolved by evidence.

## 00:20 - Implementation Shape

Added a narrow `full-test/src/platform.rs` stage with two lanes:

- `generic-x86_64-smoke`
- `asan-fuzz-sanity`

The policy is synthetic-test-first:

- default full-test: both lanes `not_claimed`;
- quick release-preflight: both lanes `not_claimed`, even if fuzz inventory
  passes;
- non-quick release-preflight on `x86_64`: generic smoke runs and is
  release-blocking;
- non-quick release-preflight off `x86_64`: generic lane is `not_applicable`;
- sanitizer is `pass` only when the existing fuzz full-sanity outcome is
  `FullSanity + Pass`.

Wired the stage into main, banner classification, JSON envelope, HTML render,
CLI help, and the preflight claim wording.

Validation so far:

```text

```

Result: pass, 12 tests.
