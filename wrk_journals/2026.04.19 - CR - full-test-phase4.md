## Important

1. **Conformance integration tests land in "Types", not "Conformance"** — `C:\language\mdopus\full-test\src\html.rs:248-262`. Each file under `tests/conformance/tests/*.rs` compiles to its own binary (`api`, `decode`, `encode`, `extensions`, `ietf_vectors`, `padding`, `projection`). `cargo_parse::binary_stem` extracts these as the binary label (e.g. `ietf_vectors`), not `conformance`, and FQNs inside those binaries are flat (`testvector01_mono`, `api`). Neither `binary == "conformance"` nor `fqn.contains("conformance::")` matches, so the entire conformance surface gets bucketed into `Types`. The unit test at `html.rs:1081-1097` only exercises a synthetic `binary="conformance"` case that the real runner never produces. Fix: allow-list the known conformance binaries — `matches!(binary, "conformance" | "api" | "decode" | "encode" | "extensions" | "ietf_vectors" | "padding" | "projection")`.

2. **`module_key` double-prefixes when FQN head already equals binary** — `html.rs:419-442`. `module_key("conformance::decode::test_zeta", "conformance")` produces `"conformance::conformance::decode"`. Cosmetic but visible. Cheap fix: strip the leading FQN segment when it duplicates the binary prefix. Interacts with item 1 — when Stage 2 actually runs, conformance FQNs are flat so each test becomes a singleton module row (~30+ rows).

3. **`bench_ratios_meta` doc/code placeholder mismatch** — `html.rs:601` doc claims "`—` for missing fields"; `html.rs:615, 620` emit `-` (ASCII hyphen-minus). Test at `html.rs:1167` pins the hyphen. Update the doc comment.
