### 22:10 — Round 1: Immediate Findings

Launched all 5 targets. Found 4 bugs within minutes:

#### Bug 1: SILK pulse decode OOB — FIXED
- **File:** `src/silk/decoder.rs:690`
- **Symptom:** `index out of bounds: len is 10 but index is 10`
- **Root cause:** Spurious `use_last` logic added `+1` to `N_RATE_LEVELS - 1` when `n_lshifts == 10`, overflowing `SILK_PULSES_PER_BLOCK_ICDF[10]` (only 10 entries, indices 0-9)
- **Fix:** Removed `use_last` — always index with `N_RATE_LEVELS - 1`, matching encoder and C reference
- **Status:** ✅ Fixed, all 36 SILK decoder tests pass

#### Bug 2: Hybrid SWB decode differential mismatch — INVESTIGATING
- **Target:** fuzz_decode
- **Symptom:** PCM mismatch at sr=48000, ch=1, pkt_len=256, samples=480
- **Details:** TOC byte 0x60 = config 12 = Hybrid SWB, 10ms, mono. Rust output ~[-320,+320], C output ~[-3600,+4400] — wildly different
- **Seed files:** `ones_255b.bin` or `zeros_255b.bin` in fuzz_decode corpus
- **Status:** 🔍 Investigation agent dispatched

#### Bug 3: SILK encoder subtraction overflow — INVESTIGATING
- **Target:** fuzz_encode
- **Symptom:** `attempt to subtract with overflow` at `silk/common.rs:1322`
- **Details:** Arithmetic overflow in fixed-point SILK code, found after ~2000 fuzzer iterations
- **Status:** 🔍 Investigation agent dispatched

#### Bug 4: Repacketizer out() length mismatch — INVESTIGATING
- **Target:** fuzz_repacketizer
- **Symptom:** Rust `out()` returns 30 bytes, C returns -3 (OPUS_BUFFER_TOO_SMALL)
- **Details:** Rust repacketizer produces output where C correctly rejects with buffer-too-small
- **Status:** 🔍 Investigation agent dispatched

#### fuzz_packet_parse — still running (no crash, healthy at 672MB RSS)

### 22:30 — Bug 3 Fixed, Continuing Campaign

- Bug 3 root cause: `(1i32 << 15) as i16 - delta_min_q15[order]` — `as i16` binds tighter than `-` in Rust, so 32768 becomes -32768 before subtraction, causing overflow. C does the subtraction in int then narrows. Fixed: `((1i32 << 15) - delta_min_q15[order] as i32) as i16`
- All 71 silk::common tests pass after fix
- Relaunched fuzz_encode with fix — running clean so far
- 3 investigation agents dispatched (decode mismatch, silk overflow [already fixed], repacketizer mismatch)
- Searched for similar cast-before-arithmetic patterns across codebase — no other instances found
