# 2026.05.04 - JRN - fuzz-vs-mutation-triage

## Context
Arthur asked for thoughts on whether an unchanged third fuzz campaign is better value than mutation testing or targeted fuzzing work.

## Notes
The pasted assessment argues that unchanged fuzzing is saturated: repeated targets, no new root-cause classes, declining artifact yield, and flat targets across campaigns. My read is that this is sound if the campaign statistics are accurate. The useful next work is not more time on the same byte-level setup, but fixing the known-class mask, adding structurally relevant targets, and using the C oracle/mutation tests to tighten correctness around known-risk areas.

## Emotional temperature
Calm and slightly wary. The ranking feels technically defensible, but I would want the numbers behind the 15%, 40%, 80%, and 95% probabilities treated as decision heuristics rather than measured forecasts.

## Follow-up: next-step ordering after unmasking
Arthur asked how the fixed mask changes what we should do next. My view: it materially upgrades fuzzing from "bad unchanged rerun" to "worth a targeted confidence campaign," but it does not make a blind full Cam 3 the top-value option by itself. The unmasking already exposed and closed several adjacent multiframe encoder bugs, and the target then passed a 1-hour smoke, so the immediate cheap payoff has probably been harvested.

Recommended order now: first replay current regression seeds and run a bounded unmasked fuzz smoke to establish the baseline, then prioritize either the C-oracle/mutation harness or one new stateful target such as CTL sequence. A full overnight fuzz run is now defensible if it uses the unmasked target set and captures useful telemetry, but it should come after we decide whether to spend the same overnight on a new target.

Emotional temperature: more optimistic about fuzz than earlier, but not suddenly convinced that "same campaign, more hours" is the best use of compute.

## Follow-up: why IETF vectors are not vendored
Arthur asked why the IETF conformance vectors are fetched rather than checked into git before we start resolving the four remaining failures.

Repo evidence:
- `wrk_docs/2026.04.19 - HLD - conformance-ietf-projection-regressions.md` explicitly says shipping the vectors in-tree is out of scope because the binary fixtures are about 71 MB and belong out-of-repo.
- `tools/fetch_ietf_vectors.sh` downloads the canonical archive from `opus-codec.org`, checks exact size `74624664`, verifies SHA-256 `6b26a22f...fedfa55`, and extracts to gitignored `tests/vectors/ietf/`.
- The HLD notes no known redistribution restriction, but says linking/fetching from the canonical archive avoids a licensing action item.
- `tests/conformance/README.md` documents "fetch once" as the intended workflow.

My read: the original choice was reasonable for keeping the repository smaller and avoiding binary fixture history, but it trades away fresh-clone green conformance. If full conformance is a regular local/CI gate, I would rather make the fetch part of the validation workflow or use Git LFS/release artifacts than keep the current sharp edge.

Emotional temperature: pragmatic. I do not think vendoring 71 MB of binary vectors into ordinary git is a good default, but I also dislike a gate that fails by default without a preflight fetch step.
