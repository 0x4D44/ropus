## Stage 3.4 HLD persisted, then devil's-advocate review

HLD lives at `/home/md/language/ropus/wrk_docs/2026.05.06 - HLD - corpus policy provisioning.md`.

Devil's-advocate found three real blockers and several real risks. Highlights:

- **Blocker A**: `corpus.rs:667-668` `generate_entry` hardcodes `-b:a 32k` and ignores
  the parsed `generation_command`. With a 24k-mono entry plus another rate corner, the
  SHA256 pin would never match because every entry would be re-encoded at 32k.
- **Blocker B**: `opusenc` writes a vendor tag from the linked libopus version with no
  `--no-comments` flag; SHA256 of `opusenc` output is not stable across builds. The
  HLD's path-(a) recipe for the Ogg-edge fixture cannot deliver a reproducible digest.
- **Blocker C**: `corpus.rs` has no `Command`-injectable seam, so HLD tests 8-11
  ("stub Command execution") are speculative until that seam is designed.

Real risks worth recording:
- FFmpeg `-strict -2` is experimental â€” the bitstream is allowed to change across
  versions. The HLD's R1 already documents this and we will live with it knowingly.
- Three counters (`decoded_and_compared`, `compared_required_entries`,
  `compared_pinned_entries`) are near-synonyms; rename for clarity.

## Supervisor decisions resolving the blockers

A. **Per-entry generation parameters.** Add `bitrate_kbps: u32` and `channels: u32` to
   the manifest schema and to `ManifestEntry`. `generate_entry` reads them and uses
   structured-arg `ffmpeg` invocation. Keep `generation_command` as a human-readable
   documentation field (no shell-out, no injection surface).

B. **Drop opusenc entirely.** Replace the `ogg-edge-preskip-or-output-gain` entry with
   a second FFmpeg-native pinned entry that exercises a different rate corner â€” e.g.
   `ffmpeg-native-speech-48k-stereo` using `tests/vectors/speech_48k_mono.wav` (or the
   stereo equivalent if present) at 48 kbps stereo. Same provenance/recipe profile as
   the other two entries, no opus-tools dependency, same R1/R2 risk model. The point
   of the second pinned entry is "catch encoder drift in a different rate corner", not
   "carry an unusual OpusHead field" â€” the latter framing was too clever for the
   actual gate semantics.

C. **CommandRunner seam.** Introduce `trait CommandRunner { fn run(&self, program: &str,
   args: &[String]) -> std::io::Result<Output>; fn version(&self, program: &str) ->
   std::io::Result<Output>; }`. Default impl `RealCommandRunner` shells out via
   `std::process::Command`. Tests inject `FakeCommandRunner` keyed on `(program,
   args)` returning canned `Output`. Threaded through `detect_ffmpeg`, `generate_entry`,
   and the `corpus_diff` invocation. Public API of `corpus.rs` gains a non-breaking
   overload that defaults to `RealCommandRunner` for the existing call sites.

D. **Counter naming.** Rename `compared_pinned_entries` â†’ `pinned_entries_matched` to
   distinguish from `compared_required_entries`. Rename intent: "matched" is the new
   contract; "compared" is the precondition.

## Stage 3.4 code review verdict: NO-PASS

Reviewer surfaced a load-bearing defect plus 5 other Required and 5 Important
findings. Empirical evidence: 10 consecutive runs of the locked FFmpeg recipe
produced 10 distinct SHA256 digests; the bitexact flags do NOT stabilize the
Ogg muxer's randomized stream serial on the static 7.0.2 build the agent
locked against. Pin contract is broken by default.

### Supervisor decisions on the review findings

**Required:**

1. SHA256 pin redesign â€” go with the reviewer's option (a): hash the **Opus
   packet payload bytes only**, after walking Ogg pages and dropping framing
   plus the two header pages (OpusHead + OpusTags). This pins encoder output
   without depending on muxer state or metadata vendor strings. As a side
   effect, drop the `-fflags +bitexact -flags +bitexact` flags entirely
   (their advertised purpose was suppressing vendor strings, but if we don't
   hash them, we don't need to suppress them) and update the comment that
   credits them with stripping page serials (per Suggested #14, that's wrong).
2. Update `generation_command` for entry 1 to match actual argv. Without the
   bitexact flags after step 1, all three `generation_command` strings just
   need to drop them â€” easier alignment.
3. Add a fixture-directory WebM integration test asserting the
   `^  DEFER .*reason=webm-matroska-container-deferred$` line.
4. Add a test that reads the live `corpus_manifest.toml` and asserts the
   exact set of required IDs.
5. Rename `ffmpeg-native-speech-48k-stereo` to `ffmpeg-native-speech-40k-mono`
   in manifest, README, and journal.
6. Gate `ScriptedRunner`-using tests with `#[cfg(unix)]` for now. Portable
   rewrite is out of scope.

**Important:**

7. Add direct SHA256 test vectors (NIST FIPS-180-2 â€” empty, "abc",
   multi-block).
8. **Swap inline SHA256 for `sha2` crate.** The reviewer found `sha2` is
   already a transitive workspace dep via `tools/fetch-assets`, so the
   "zero-dep production" rationale is false. Drop ~150 LOC of locally-
   maintained crypto + the shell-out fallback. Add `sha2` to `full-test`'s
   `[dependencies]`.
9. Tighten `max_size_bytes` for the speech entry from 4_000_000 to
   1_500_000 (still > the ~780KB actual file).
10. Add exit code 3 row to README's exit-code table.
11. Add HTML rendering test for `pinned_entries_matched` + `deferred` strings.

**Suggested:** 12 (comment about scripted-runner dispatch routing) and 13
(`failure_outcome` 10th param) are nice-to-have. 14 (bitexact comment lies)
becomes moot once we drop the flag in step 1. 15 (sha256_of shell-out is
dead code) becomes moot once we switch to `sha2` in step 8.

Knock-on effects of redesigning the pin to hash payload bytes only:
- No re-locking against FFmpeg version drift for muxer/metadata changes; pin
  only fires on actual encoder output drift. This is *strictly stronger*
  than the original contract, not weaker.
- The locked digests change: the apply-feedback agent must regenerate them
  from the same FFmpeg version, since they're computed differently.
- The pin is now portable across maintainer machines that have any FFmpeg
  build that produces the same encoder output (which, for the same FFmpeg
  release, is what we want; cross-version drift is still detected).
