### Reachable Rust state in stage 1 (read-only via existing public surface)

- `enc.silk_encoder() -> Option<&SilkEncoder>` — gives the per-channel SILK state covered by `debug_get_silk_interframe_state` and `debug_get_silk_extended_state`.
- Public getters on `OpusEncoder`: `get_bitrate`, `get_complexity`, `get_vbr`, `get_vbr_constraint`, `get_inband_fec`, `get_packet_loss_perc`, `get_dtx`, `get_signal`, `get_bandwidth`, `get_lsb_depth`. These reveal `use_vbr`, `use_dtx`, `fec_config`, `silk_mode.complexity`, `silk_mode.packet_loss_percentage`.
- `enc.range_final` — public field.
- **Not reachable read-only:** `silk_mode.use_cbr`, `silk_mode.use_in_band_fec`, `silk_mode.use_dtx`, `silk_mode.bit_rate`, `silk_mode.lbrr_coded`.

### How we detect H1/H2 without a Rust-side `silk_mode` accessor

H1 thesis: `ms_set_inband_fec(1)` writes only `fec_config`, missing `silk_mode.use_in_band_fec`. The public `set_inband_fec(1)` writes both. Detection plan:

1. After every setter call, dump the C-side `silk_mode` (new `debug_get_opus_silk_mode_state` accessor).
2. After every setter call, dump every reachable Rust top-level field via the public getters above (`get_inband_fec`, `get_vbr`, `get_dtx`, …).
3. **Asymmetry shows up as a field that C writes but Rust's getter says is unchanged** — but for `silk_mode.use_in_band_fec`, neither side has a public Rust getter. The detection therefore relies on: read `enc_control.bit_rate` / `enc_control.use_in_band_fec` from `silk_encoder()` post-encode (via `state_fxx[0].s_cmn.use_in_band_fec` and `s_cmn.in_band_fec` if mirrored) — these are the *consumed* values. If they differ between C and Rust after an `inband_fec(1)` setter, H1 is confirmed.

Actually, the cleanest signal is even simpler: H1 predicts that **frame N+1's encoded bytes diverge** when `set_inband_fec(1)` is called between frame N and frame N+1. The 4 multistream repros happen to apply the setter shuffle — every byte-divergent repro that contains `set_inband_fec` in its shuffle is a H1 signal. The diagnostic only needs to dump the *post-shuffle* state and the encoded byte difference.

For H2 / H3 / etc. the per-channel SILK state diff is sufficient.

### 2026-05-02 — Hypothesis confirmation

- **H1 (ms_set_inband_fec misses silk_mode.use_in_band_fec) CONFIRMED.**
  Direct evidence in crash-03-cx4-vbr1: top-level `fec_config=1` but
  post-encode `state_fxx[0].s_cmn.use_in_band_fec=0`. The setter wrote to
  `fec_config` only; `silk_mode.use_in_band_fec` stayed at 0; the encode
  consumed the stale 0; the per-channel mirror of the encode-time
  `enc_control.use_in_band_fec` shows it was 0.

- **H2 (ms_set_vbr misses silk_mode.use_cbr) — asymmetry exists but is
  masked.** Confirmed by the C-side dump: after `ms_set_vbr(0)` the C
  `silk_mode.useCBR` flips to 1, but the Rust `OpusMSEncoder::set_vbr` →
  `ms_set_vbr` setter only writes `use_vbr`. **However**, the recompute
  at `ropus/src/opus/encoder.rs:2715` (`silk_mode.use_cbr = 1 - use_vbr`)
  fires inside the SILK-using branch of `encode_native`, which the
  family=0 ch=1 NB encode always reaches. So `silk_mode.use_cbr` is
  effectively the right value at the moment `silk_Encode` runs. The H2
  asymmetry is **latent but does not currently cause divergence on these
  4 repros**. Should still be patched in stage 2 as a defence-in-depth /
  consistency fix — a future bug where `silk_mode.use_cbr` is read
  before the L2715 recompute would surface H2.

- **H3 (n_bits_exceeded / silk_mode.bit_rate drift) — strongly suspected
  for multistream crash-00/01/02.** Pre-encode silk_mode.{useInBandFEC,
  useCBR, useDTX, complexity, packetLossPercentage} match exactly between
  C and Rust at encode time. Post-encode, all 66 NSQ buffer fields
  diverge — meaning the encode produced a different bitstream from the
  same input. The remaining encoder-internal computed input that we
  cannot read pre-encode is `silk_mode.bit_rate` (computed by
  `compute_silk_rate_for_hybrid` / SILK rate logic at
  `encoder.rs:1726-1747` and onwards) and the cross-call
  `n_bits_exceeded` reservoir (which would be 0 on a fresh encoder, but
  one of the 66 fields was `nsq_slf_ar_shp_q14` and the C-side post-encode
  `silk_mode.bitRate=5600` was visible, while Rust's value isn't). Phase
  B should trace `(tell, rng)` at the boundaries listed in HLD §2.3 to
  confirm.

- **H5 (stereo memory) + NEW (side-channel reset gate) — strongly
  suspected for multiframe crash-00..03.** All four show frame-1 ch1
  divergence with frame-0 byte-OK. The `nsq_sltp_buf_idx C=320 vs R=160`
  (= 2× ratio = an extra frame's worth of subframe processing on C side)
  and the crash-02 case where Rust's ch1 has `prev_nlsfq_q15` ALL ZERO
  while C has non-zero values (= Rust re-initialised ch1 between frames,
  C did not — or vice versa) point at the **stereo side-channel reset
  gate at `silk/encoder.rs` mirror of `enc_API.c:453-464`** (the
  `prev_decode_only_middle == 1` path). The HLD did not list this as a
  named hypothesis; closest match is H5 generalised. Calling NEW in the
  report.

- **H4, H6–H12 — not observed as primary divergence vectors in this
  data set.** They may compound (e.g., H6 LPC memory diverges secondarily
  once H5/NEW upstream divergence has fired).

### 2026-05-02 — Recommended fix scope (stage 2)

Two distinct production-code bugs surface from this diagnostic:

1. **Bug A: `ms_set_*` setter asymmetry.** Per the supervisor's Q2 sign-off
   (option b: collapse `ms_*` family into public `set_*`), stage 2 should
   route `OpusMSEncoder::set_inband_fec`, `set_vbr`, `set_dtx`, etc.
   through the public `OpusEncoder::set_*` methods. This closes H1
   directly (crash-03), closes H2 latent (crash-00/01/02 partially), and
   eliminates a class of future drift bugs.
   - Files: `ropus/src/opus/encoder.rs:1340-1463` (delete `ms_*` family),
     `ropus/src/opus/multistream.rs:1316-1414` (re-route).
   - Diff: ~30 lines deleted, ~20 added.

2. **Bug B: stereo side-channel reset / `silk_mode.bit_rate` derivation.**
   The multiframe Findings #7 and the multistream `crash-00/01/02`
   divergence are *not* setter-asymmetry bugs. Bug A's fix won't close
   them. They look like two related but distinct issues:
   - **B1: stereo side-channel state mutation across frames** (multiframe
     crash-00..03). The divergent input to `silk_encode_frame_FIX` for
     ch1 on frame 1 shows up post-frame-0; investigate
     `silk/encoder.rs` mirror of `enc_API.c:453-464` and the
     LR→MS handling around `silk/encoder.rs:8208-8231`.
   - **B2: silk_mode.bit_rate / n_bits_exceeded computation**
     (multistream crash-00/01/02). The encode-time computation of the
     SILK target rate differs between Rust and C even with all
     post-setter input state matched. Investigate
     `compute_silk_rate_for_hybrid` and the bit-reservoir update at
     `silk/encoder.rs:8376-8381`.
   - **Recommendation:** stage 2 fixes Bug A (the clean H1 confirmation)
     first, then re-runs the diagnostic on the multiframe + multistream
     repros. If multiframe crash-00..03 and multistream crash-00/01/02
     STILL diverge (highly likely — Bug A doesn't touch B), proceed to
     **Phase B trace-point instrumentation** per HLD §2.3 to localise
     B1/B2 before any production code touches `silk/encoder.rs`.

### 2026-05-02 — Phase B yes/no

**Yes — Phase B is needed for B1/B2.** Phase A confirms H1 cleanly for
crash-03 only. The other 7 encode-class repros (4 multiframe + 3
multistream non-FEC variants) show post-encode state divergence with
all pre-encode observable state matching. The encode itself produces
a different bitstream — meaning the divergence is *inside* the silk_Encode
flow, in either:
- A computed encoder-internal input (`silk_mode.bit_rate`, `lbrr_coded`,
  `max_bits`) that differs across implementations, or
- A path-divergent gate (stereo side-channel reset, mode selection
  tie-break) that fires on one side but not the other.

Phase B trace-point boundaries from HLD §2.3 are exactly the right tool
to localise this. The 1.4.6 boundary (per-channel post `silk_encode_frame_fix`)
should isolate B1 (stereo side-channel divergence appears at frame 1
ch1, post that frame's call). The 1.4.3 + 1.4.7 boundaries
(`n_bits_target` / `n_bits_exceeded` mismatch) should isolate B2.
