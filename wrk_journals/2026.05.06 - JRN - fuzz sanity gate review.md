# Journal: Fuzz Sanity Gate Review

Date: 2026.05.06

Read the code-reviewer skill and the signed HLD before inspecting the uncommitted source diff. I stayed in review mode and did not edit implementation files.

The shape of the implementation is broadly what I expected from the HLD: shell owns the bounded cargo-fuzz work, full-test owns policy and reporting, and default runs do not become hostile. The known `fuzz_decode` committed-crash replay failure is correctly a failing gate, not a reason to reject the gate.

The part that bothered me was the quick inventory path. It is intentionally cheap, which is right, but it reimplements target discovery and crash inventory in Rust. That reimplementation is close but not identical to the shell source of truth. My gut says this is the kind of small divergence that becomes confusing later because quick and full release-preflight can disagree about whether the same fuzz inventory is valid.

Emotional temperature: calm, slightly wary. The implementation is not reckless, but the duplicated parser is exactly the sort of thing that looks harmless until a malformed manifest or stale nested crash file slips through the quick path. I documented that as an issue rather than a blocker because the full gate still fails correctly and the current committed crash layout is flat.

Validation completed:
- `timeout 10s bash -n tools/fuzz_run.sh`
- `timeout 10s bash tools/fuzz_run.sh --list`
- `timeout 120s cargo test -p full-test fuzz`
- `timeout 120s cargo test -p full-test banner`

Remaining risk: quick release-preflight can pass or warn on malformed/nested inventory states that full sanity rejects. The next clean move is to align quick discovery with `tools/fuzz_targets.sh` or tighten the Rust parser and make crash-bin inventory recursive.
