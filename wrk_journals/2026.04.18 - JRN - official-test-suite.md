### 2026-04-18 — Phase 2 implementation

Phase 2 had significant scope (soft_clip port + decoder CTLs + PLC fix + per-test-file restructure + test_opus_decode.c passing). Work happened in four agent rounds:

**Round 1 — implementer (`a08e769008acac173`):** delivered soft_clip port, CTL plumbing, PLC fix, per-test-file split, but **hit STATUS_HEAP_CORRUPTION on MSVC** when running `test_opus_decode.c`. Minimal-C and minimal-Rust reproducers both passed; only the real test crashed. Agent couldn't isolate and reported blocked.

**Round 2 — debug agent (`af7b7ab2ae758a5f9`):** root-caused at confidence 5/5.
- **Root cause:** our `OpusDecoder` (4888 bytes) holds `Vec<_>` fat pointers (in `SilkDecoderState::exc_q14`, `out_buf`, plus six `Vec<i32>` fields in `CeltDecoder`). The test does `memcpy(decbak, dec, get_size(c))` to snapshot state. This aliases Vec backing allocations between two "copies." On mode transitions `SilkDecoder::init()` drops+realloc Vec contents, freeing heap blocks still referenced by the aliased copy. Use-after-free + double-free → heap corruption.
- **Fix:** introduced POD `OpusDecoderHandle` in `capi/src/decoder.rs` — layout `{ u64 magic, *mut OpusDecoder inner, padding }` sized to `size_of::<OpusDecoder>()`. Handle stored in test-allocated storage; real decoder lives on Rust heap, unreferenced by the bytes the test memcpys. `resolve_handle()` validates magic before every deref. `_destroy` is a no-op leak (intentional — test processes short-lived, avoids CRT/Rust heap-boundary issues).
- Release build passes in 8s, debug in 155s. Both `padding` and `decode` green.

**Round 3 — reviewer (`ac846ed645589f0bb`) — two required fixes + clarifications:**
1. `decoder.rs:62` const assertion used hardcoded `4888` — changed to type-based size check.
2. `encoder.rs:55` dangling rustdoc link — fixed to `state_free`.
3. `soft_clip.rs` ambiguous "returns 0" comment — clarified re: within1 hint vs clamping.
4. `soft_clip.rs:48` dead-code binding — added placeholder comment for future SIMD hook.

Applier agent `aed2bfde66aeec6e6` applied all four.

**Round 4 — devil's advocate (`a1eb8d1ca282f4587`):** one real fix, several phase-3/4 traps.
- **Fixed:** `state_free` dead wiring. Decoder destroy now routes through the same chokepoint as encoder (applier agent `ab10dc47d351bc8e2`).

**Phase-3+ architectural concerns to carry forward:**

1. **Aliased handles share one inner decoder.** `test_opus_decode.c:328-333` memcpys `dec[0] → decbak` expecting a state reset — with our handle indirection, both aliases point at the same real decoder. Test passes today (assertions only check `out_samples >= 20`), but any future byte-exact cross-copy test will fail. If phase 4 test_opus_encode.c does byte-exact copy-and-diverge checks, we need a different strategy (serialize/deserialize state via ropus-side Clone, OR make OpusDecoder POD). **Flag this in the phase-3 brief.**
2. **OpusEncoder has the same Vec-aliasing issue** — will bite in phase 4. Handle-indirection pattern applies mechanically. Phase 4 must replicate.
3. **`opus_packet_has_lbrr` diverges from reference for code-0 packets** — may surface in test_opus_api.c (phase 3).
4. **`opus_decoder_init` on a live handle leaks the old inner Box** — not a bug under "always leak" design, but aliased handles pointing at the detached old Box continue to work silently. Document.
5. **Leak accounting:** ~5-10 kB per decoder × number of creates. Fine for current tests; monitor in phase 3 where loops get tighter.
6. **OPUS_SET_COMPLEXITY** unwired for encoder (phase 4 will need it).
