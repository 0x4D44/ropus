## Findings

The mechanism exists, but the gate does not. `corpus_diff` is a good exact PCM
oracle for supported Ogg Opus family-0 files, but it is not part of full-test
and the checked-in corpus directory contains only the README.

The honest caveats are already written down: the fetch script is a
reference-encoded baseline, non-reference corpus population is open work, WebM
is skipped, and no seek/FEC/PLC behavior is claimed. The problem is that those
caveats are not yet enforced as a machine-readable release/report contract.

The sharpest false-green issue I found is skipped-only coverage. The current
binary exits `2` for no candidate files, but a directory with candidates that
all skip can finish without mismatches or panics. That is fine for exploratory
triage and bad for a named corpus gate.

## Recommendation

Recommend the next implementation slice as:

```text
Corpus gate contract and generated non-reference smoke
```

Keep external media out of git for now. First make the corpus lane impossible
to overstate, then add a tiny generated FFmpeg-native Ogg Opus smoke from an
existing repo WAV fixture when tooling is available. This avoids licensing
ambiguity while proving something meaningfully different from the current
reference-encoded baseline.

## Blockers

No blocker for the assessment. The implementation slice will need a policy
choice: whether non-quick `--release-preflight` should require the corpus gate
immediately, or whether full-test should expose an explicit corpus-claim flag
first. My bias is to fail only when a corpus claim is made, but make the default
report say "not claimed" loudly.

## Emotional Temperature

This felt usefully constrained. The tempting move is to go hunting for
"real-world" samples, but that would tangle the team in rights and provenance
before the gate semantics are trustworthy. The cleaner engineering move is to
make empty and skipped coverage impossible to mistake for signal, then add
small generated signal with no asset baggage.
