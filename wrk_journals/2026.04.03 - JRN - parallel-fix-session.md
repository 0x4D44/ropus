Key finding: At 48kHz, the very first CELT data byte differs for sine. 
This means there's a CELT encoder bug that's separate from SILK.
At 8kHz, the first CELT packet matches but the second doesn't — 
suggesting the CELT codec works for the first frame but state carries wrong.

The 48kHz case is cleaner to debug since it's pure CELT (no SILK involvement).

### CELT MDCT divergence details (48kHz short blocks)
Both C and Rust detect transient (short_blocks=8, is_transient=true).
Input matches (all zeros from overlap).
MDCT frequency output:
- C:  freq[0..4]=[6, -1654539, 3170489, -354501]
- RS: freq[0..4]=[0, -1654541, 1382282, -6369]
freq[0] and [1] are close, but from [2] onwards values are completely different.
This suggests the short-block MDCT interleaving or stride is wrong at 48kHz.
The short block size is 960/8=120 samples, producing 60 freq bins per block.

### CELT stride fix attempt
Fixed compute_mdcts to pass stride directly to clt_mdct_forward instead of
using stride=1 to a temp buffer + copy. The C code passes stride=B (number
of short blocks) directly to the MDCT post-rotation.

However, the fix didn't change the output — the MDCT values are identical before
and after. This means either:
1. The stride was already being handled correctly somewhere else
2. The issue is NOT in the stride but in the per-block MDCT computation itself
3. The freq[0..7] with stride=8 represent interleaved blocks, and the per-block
   values are wrong for a different reason (trig tables, FFT, window)

The freq[0]=0 vs 6 and freq[1]=-1654541 vs -1654539 differences are tiny (rounding).
But freq[2]=1382282 vs 3170489 is a 2x difference — structural bug in one block's MDCT.

Next: need to investigate per-block MDCT output separately to find which block
produces wrong values and why.
