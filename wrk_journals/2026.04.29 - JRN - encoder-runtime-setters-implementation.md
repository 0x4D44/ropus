### 2026-04-29 — Stage 1 — implementation

**Scope delivered (purely additive — no breaking API changes):**

- `ropus/src/api.rs`:
  - New `pub enum InbandFec { Disabled, Enabled, Forced }` mirroring the
    rustdoc / derive set used by `Bitrate`, `Signal`, etc.
    `pub fn as_c_int` (0/1/2) and `pub fn from_c_int` with `debug_assert!`
    fallback to `Disabled`.
  - `Bitrate::from_c_int` (`OPUS_AUTO`→`Auto`, `OPUS_BITRATE_MAX`→`Max`,
    `n>0`→`Bits(n as u32)`, anything else → debug_assert + `Auto`).
  - `Signal::from_c_int`, `ForceChannels::from_c_int`, `Bandwidth::from_c_int`
    — each with `debug_assert!` + documented default for unknown values.
  - `check_ok_runtime(code: i32) -> Result<(), EncodeError>` sibling of
    `check_ok`.
  - Five new unit tests under `#[cfg(test)] mod tests`: one per enum
    asserting `from_c_int(x.as_c_int()) == x` for all variants. Sixth
    test, gated `#[cfg(not(debug_assertions))]`, asserts the
    release-build fallback for `Bitrate::from_c_int(0)` /
    `from_c_int(-42)` returns `Bitrate::Auto`.
- `ropus/src/opus/encoder.rs`:
  - `pub(crate) fn get_user_bitrate_bps(&self) -> i32` adjacent to
    `get_bitrate`, with the rustdoc text from the HLD verbatim.
- `ropus/src/lib.rs`:
  - Added `InbandFec` to the `pub use api::{...}` re-export so the new
    enum is reachable from the crate root. (Without this, the new `pub`
    enum is dead-code from the lint's perspective because `mod api` is
    private — same pattern as `Bitrate`, `Signal`, etc.)

**Deviations from the HLD:**

- Added `#[allow(dead_code)]` on each Stage-1-only-plumbing helper
  (`Bitrate::from_c_int`, `Signal::from_c_int`, `Bandwidth::from_c_int`,
  `ForceChannels::from_c_int`, `OpusEncoder::get_user_bitrate_bps`,
  `check_ok_runtime`). Rationale: clippy with `-D warnings` is in the
  Stage 1 verification gate, and these helpers are intentionally unused
  until Stage 2 wires them through the new getters/setters. The comment
  on each `#[allow]` references Stage 2 for context. Stage 2 will remove
  these attributes when the call sites land.

- HLD's "Stage 1 scope" wording at user-prompt level says
  `pub fn as_c_int / pub fn from_c_int` for `InbandFec` while the
  existing `as_c_int` methods on `Bitrate`/`Signal`/etc. are
  module-private. Followed the literal user prompt for `InbandFec`
  (public methods) and kept the existing pattern for the four added
  `from_c_int` helpers (private). The asymmetry only matters at the
  module-private boundary; both compile. Will revisit visibility in
  Stage 2 if the new getters need to call these helpers from outside the
  `api` module (they will not — getters live in `impl Encoder` inside
  the same module).

### 2026-04-29 — Stage 1 — devil's advocate

DA verdict: **ship Stage 1.** No critical findings. Watch-list items
carried into Stage 2 brief:

- **WL-a (`Bitrate::Bits(0)` round-trip lossy):** `Bits(0).as_c_int() = 0`, `from_c_int(0)` panics in dev / returns `Auto` in release. Encoder rejects `bitrate <= 0` so unreachable from stored state, but worth a one-line caveat on `Bitrate::from_c_int`'s rustdoc when Stage 2 lands.
- **WL-c (consistency vs absolute correctness):** the round-trip tests verify `from_c_int(as_c_int(v)) == v`, which would pass if both directions had cancelling typos. Recommend Stage 2 add at least one literal-value assertion per enum (e.g. `assert_eq!(InbandFec::Forced.as_c_int(), 2)`) to pin to libopus values rather than internal self-consistency.
- **WL-d (`#[allow(dead_code)]` count):** eight attributes by end of Stage 1. Add to Stage 2 acceptance: grep should return zero `#[allow(dead_code)]` newly-introduced in Stage 1.

DA scrutinised and confirmed fine: `OPUS_*` constant values match libopus, `pub(crate)` on `get_user_bitrate_bps` is correct, all five round-trip tests cover all variants exhaustively, no concurrent edits, no reachability bugs in the `from_c_int` helpers.
