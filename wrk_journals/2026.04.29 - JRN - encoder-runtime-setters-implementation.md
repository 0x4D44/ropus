### 2026-04-29 — Stage 2 — DA feedback applied

Devil's advocate pass surfaced two findings (C1, C2) — both about
honesty between rustdoc/HLD claims and the actual encoder behaviour.
Both fixed in `ropus/src/api.rs`; no other files touched, no logic
changes, no test count change (Test 5 just gained two assertions).

- **C1 — Rustdoc lied about `Bits(0)` clamping.**
  `OpusEncoder::set_bitrate` (`ropus/src/opus/encoder.rs:3468-3470`)
  returns `OPUS_BAD_ARG` for `bitrate <= 0` (excluding the `Auto` /
  `Max` sentinels); it does NOT silently clamp `0` up to 500. Two
  rustdoc blocks made the false claim:
  - `OpusEncoder::set_bitrate` rustdoc — rewrote to spell out the
    three regimes accurately: `<= 0` ⇒ `BadArg`, `[1, 499]` ⇒ clamp
    to 500, `> 750_000 * channels` ⇒ clamp to that upper bound.
  - `Bitrate::from_c_int` rustdoc — replaced the "round-trip is
    faithful for any `n >= 1`" wording with a four-bullet round-trip
    table covering: `Bits(0)` (unreachable, `BadArg`), `Bits(1..=499)`
    (unreachable, clamps to 500), `Bits(500..=750_000*ch)` (faithful),
    `Bits(>750_000*ch)` (unreachable, clamps to upper bound).
  Rationale: the previous wording made `Bitrate::Bits(0)` look like a
  valid input that round-trips through clamping, when in fact the
  setter rejects it outright. Misleading docs are worse than missing
  docs.

- **C2 — Tests didn't exercise the BadArg-and-clamp boundary.**
  Test 5 (`runtime_setters_surface_bad_arg`) covered `set_complexity(11)`
  and `set_packet_loss_perc(101)` but said nothing about the bitrate
  edges that C1 documents. Added two assertions on the existing mono
  fixture:
  1. `set_bitrate(Bitrate::Bits(0))` ⇒ `Err(BadArg)` (bottom edge).
  2. `set_bitrate(Bitrate::Bits(2_000_000_000))` succeeds; the
     subsequent `bitrate()` read-back asserts `b <= 750_000` (the
     mono cap), proving the silent top-edge clamp.
  Rationale: the rustdoc now makes specific behavioural claims; tests
  must lock those claims down so future refactors can't quietly
  regress them.
