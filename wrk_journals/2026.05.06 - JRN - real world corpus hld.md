## Findings

The corpus mechanism exists, but it is not yet a release-grade gate.

Important points:

- `corpus_diff` is already the right oracle shape for this slice: decode the same Ogg Opus packets through ropus and the C reference and compare PCM sample-for-sample.
- The corpus directory is not populated in the repo beyond the README.
- The fetch script is currently provenance-safe but practically unusable for provisioning because all SOURCES are `TBD`.
- Even if fetch checksums are pinned, those public samples are reference-encoded. They are a useful baseline, not a non-reference diversity corpus.
- Existing docs point at FFmpeg native `-c:a opus` as the concrete non-reference source. That is a good first gate because it can be generated from repo-owned WAV fixtures.
- `corpus_diff` has an all-skipped false-green risk. Empty directories return 2, but a directory with only skipped candidates can exit 0 because only mismatches and panics trigger exit 1.
- Current harness limitations mean this must not be described as WebM, seek, pre-skip, gain, granule, FEC, PLC, or full playback coverage.

## Design Chosen

I wrote the HLD around a small policy and gate slice:

- Add a tracked manifest for gateable corpus entries.
- Prefer generated non-reference Ogg fixtures from existing WAV files using FFmpeg native `opus`.
- Keep generated `.opus` files ignored by default; record recipe, tool version, and SHA-256 at report time.
- Keep public reference-encoded fetches separate from non-reference coverage.
- Require manifest-aware minimum coverage so skipped-only directories cannot satisfy a corpus claim.
- Add a `full-test` corpus lane that is report-only by default, not claimed in quick release-preflight, and banner-blocking under non-quick release-preflight once signed off.
- Update the oracle matrix to label the corpus lane as exact PCM parity for Ogg family-0 packet decode only.
