Read the three requested targets. `fuzz_roundtrip.rs` is byte-slice based and easy to extend with `data[7]` high bits. `fuzz_encode_multiframe.rs` and `fuzz_decode_plc_seq.rs` are structured `Arbitrary` targets. PLC has a useful opening: `drop_mask` only needs low bits for at most 12 frames, so high bits can select duration without changing parse shape.

The more interesting finding is PLC decode capacity. The target currently passes `5760` to stateful decode, which is fine as a maximum for kept packets but means dropped frames request 120 ms PLC. If Phase 3.3 is about legal frame durations, dropped PLC should request the selected frame duration unless we explicitly want a separate PLC-duration dimension.

Emotional temperature: focused. This is narrow enough to be useful, but only if the implementation resists "while here" expansion into CTL fuzzing or campaign running.
