# Journal: SILK drift cross-precision attribution

Date: 2026-05-10
Workspace: `/home/md/language/ropus`
Task: investigate the "pre-existing SILK encoder drift at fs=16k/40kbps/CBR/FEC=0/loss=30 (24 dB PCM SNR)" follow-up named in the 2026-05-09 DRED bitrate plumbing closeout.

## Conclusion up front

There is no Rust SILK encoder bug. The 24 dB SNR was a comparison artifact: `harness-deep-plc/` builds the C reference in **float mode** (compiles `reference/silk/float/*.FLP.c` because xiph forbids `FIXED_POINT + DEEP_PLC`), while the Rust port mirrors **fixed-point** SILK (`reference/silk/fixed/*.FIX.c`). Cross-precision SILK output cannot be byte-exact, and ~30 dB PCM SNR is the realistic floor for typical configs at this band — adding DRED reconstruction noise on top drops it to the 24 dB we observed.

Through the regular `harness/` C reference (fixed-point, no DEEP_PLC), the Rust SILK encoder is byte-exact at 16 k/{16, 24, 32, 40, 64} kbps mono VOIP CBR for all 5 frames of the same WAV. Verified end-to-end.

## Investigation

### 1. Bisection ruled out FEC, loss, and DRED as the cause

A scratch test in `harness-deep-plc/tests/silk_drift_bisect.rs` walked one knob at a time from the failing config. All variants diverged, including the simplest possible silk-only config (`dred=0, fec=0, loss=0, vbr=0`). Drift exists across every bitrate, every fs, every input (sine/noise/silence). The April 2026 "all bitrates bit-exact" claim from `silk-16kbps-divergence.md` therefore appeared to have regressed.

### 2. `ropus-compare encode` passing was the first clue something was off about the framing

`cargo run --bin ropus-compare encode tests/vectors/16000hz_mono_sine440.wav --bitrate 40000 --complexity 5` reported all 50 frames byte-exact. That's because `harness/src/cli.rs::EncodeConfig::new()` defaults to `OPUS_APPLICATION_AUDIO`, which routes 16 k/40 kbps through CELT-only — bypassing SILK entirely. So the harness-level smoke test wasn't exercising the path under suspicion.

### 3. Phase-C trace through `harness/` showed full byte agreement

Wired up `harness/tests/silk_drift_first_diverge.rs` against the `trace-silk-encode` infrastructure (Rust-side `ropus::silk_trace::*` plus C-side `dbg_silk_trace_*` from `harness/silk_enc_api_traced.c` + `silk_encode_frame_FIX_traced.c`). At 16 k/40 kbps/VOIP/CBR/FEC=0/loss=0/no-DRED, frame 0 was byte-exact, and so were frames 1-4. V2 boundaries 99, 100, 101, 104, 105, 106, 107, 108, 109 had identical payloads.

The two boundaries that "differed" (b102 idx 12 — `gains_unq_q16`; b103 idx 70 — tail of `ar_q13`) both turned out to be **uninitialised stack/scratch memory in the C side** that Rust default-initialises to zero. Frame 1's C value `-1341979680` for `gains_unq_q16[2]` is the giveaway. These fields aren't read downstream; the trace shim should arguably skip them, but it's a diagnostic-cosmetic issue, not a real divergence.

### 4. `harness/` build vs. `harness-deep-plc/` build — the actual delta

Both builds compile the C reference in float mode (neither defines `FIXED_POINT`), but they pick different SILK source sets:

- `harness/build.rs:146-165` — compiles `silk/fixed/*.FIX.c` (this is the regular harness wired up against fixed-point SILK).
- `harness-deep-plc/build.rs:162-189` — compiles `silk/float/*.FLP.c` (because `harness-deep-plc/config.h` defines `ENABLE_DEEP_PLC=1`, and xiph's autoconf at `reference/configure.ac:973` forbids `FIXED_POINT + DEEP_PLC` with `AC_MSG_ERROR`).

`silk/float/noise_shape_analysis_FLP.c` and `silk/fixed/noise_shape_analysis_FIX.c` are independent implementations operating on different precision (`silk_float` vs Q-format integers); they cannot produce byte-identical range-coded output.

The Rust port's function names — `silk_noise_shape_analysis_fix`, `silk_find_pred_coefs_fix`, `silk_burg_modified_fix`, `silk_process_gains_fix` — explicitly carry the `_fix` suffix to make the precision class explicit.

### 5. Cross-bitrate confirmation

`sanity_check_at_other_bitrates_through_fixed_c_ref` in the trace test ran 16 k/{16, 24, 32, 40, 64} kbps mono VOIP CBR through the fixed-point C reference. All bitrates byte-exact across all 5 frames.

## Test changes applied

Both DRED bitrate plumbing tests had attribution comments that pointed at "pre-existing SILK FEC/LBRR drift" or "upstream SILK drift," motivating future work on a non-existent bug. Updated both:

- `harness-deep-plc/tests/dred_bitrate_plumbing_diff.rs` — comment now names the cross-precision boundary explicitly and links the autoconf line.
- `harness-deep-plc/tests/dred_bitrate_plumbing_nonzero_diff.rs` — same comment update; dropped `#[ignore]`; replaced the unrealistic 60 dB Tier-2 SNR floor with an 18 dB cross-precision floor (measurement at this config = 24 dB; floor sits 6 dB below for input-content margin); removed the misleading "f32 op order" debug message that suggested investigating a clean port.

The genuine byte-exactness gate for the f32 DRED ops themselves still lives at `harness-deep-plc/tests/dred_compute_bitrate_ffi_diff.rs`, which links the C `compute_dred_bitrate` / `estimate_dred_bitrate` symbols directly and avoids the SILK precision boundary.

## Open follow-ups

- The exhaustive cross-bitrate / cross-fs / cross-channel sweep through the fixed-point harness has not been run as a checked-in regression. The trace-test sanity sweep was deleted with the rest of the investigation aids; if a "Rust SILK ≡ fixed C SILK byte-exact" gate is wanted long-term, it's a small follow-up.
- `harness/silk_encode_frame_FIX_traced.c` should ideally not include `gains_unq_q16` in the b102 payload, or skip indices known to be uninitialised — to make Phase-C trace diffs less noisy. Cosmetic, not blocking.
- The other "Tier-2 (SNR-bounded)" claims in the project-status README need revisiting against the cross-precision context: lots of harness-deep-plc differentials would land at 30 dB-ish floors, not the 50/60 dB the doc currently advertises. Doc-only sweep, separate from this work.
