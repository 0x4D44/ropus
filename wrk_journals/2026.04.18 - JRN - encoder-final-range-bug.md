# JRN - encoder-final-range-bug

## Date/Goal
- **Date:** 2026-04-18
- **Goal:** Fix `OpusEncoder::get_final_range()` divergence from the produced bitstream on the first WB activation after an intervening non-WB bandwidth in SILK force-mode. Surfaced by the official xiph/opus test suite (test_opus_encode.c:501) via capi FFI shim.

## Symptom
- Force-mode SILK, 6809 bps, 1920-sample 48 kHz frames, bandwidths cycling NB ŌåÆ MB ŌåÆ WB:
  - Iter 0 (NB): encoder range == decoder range == C-ref decoder range
  - Iter 1 (MB): all three match
  - **Iter 2 (first WB): encoder reports `0x0c4c9290`; both decoders see `0x43f05790`**
  - Iter 3-14: match again
- XOR delta = `0x4fbcc500`. Bitstream itself is correct (two independent decoders agree).

## Timeline

### [T+0] Investigation begin
Opened encoder/decoder/ref for reading.

## Diagnosis

### Mapped the three relevant sites
- ropus encoder `range_final`/`redundant_rng` lives at `ropus/src/opus/encoder.rs:1990-2565`.
- ropus decoder mirror at `ropus/src/opus/decoder.rs:835-1014`.
- C reference encoder at `reference/src/opus_encoder.c:1879-2553`.
- C reference decoder at `reference/src/opus_decoder.c:298-700`.

### Root cause (confidence 5/5)
**Missing `redundant_rng = celt.rng;` in ropus encoder's CELTŌåÆSILK redundancy branch.**

- C reference encoder line 2440: `celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng));` captures CELT rng after encoding the 5ms CELTŌåÆSILK redundancy frame, BEFORE `OPUS_RESET_STATE` wipes it.
- ropus encoder `ropus/src/opus/encoder.rs:2393-2409` is missing this capture ŌĆö it calls `celt.ctl(CeltEncoderCtl::ResetState)` (which sets `celt.rng = 0` via `CeltEncoder::reset()` at `ropus/src/celt/encoder.rs:3198`) WITHOUT first saving the rng.
- Downstream at line 2564, `range_final ^= redundant_rng` XORs with 0 instead of the real value.
- Decoder-side (both C and ropus) correctly captures the corresponding decoder rng during redundancy decode and XORs it in ŌĆö hence the divergence.

### Why "first WB activation" specifically
`redundancy && celt_to_silk` fires only when `silk_bw_switch` is set at frame entry. `silk_bw_switch` is set the previous frame when SILK signals `opus_can_switch = switchReady && !nonfinal_frame`. SILK's `switchReady` signal fires only on upward SILK-internal sample-rate changes (12 kHzŌåÆ16 kHz in the observed test). In the NBŌåÆMBŌåÆWB cycle, that step falls on MBŌåÆWB (iteration 1ŌåÆ2). Downward forced bw changes (WBŌåÆNB at iter 2ŌåÆ3) do not trigger `switchReady`. Hence exactly iteration 2 fails, iterations 0, 1, 3-14 match.

## Hypothesis
Single-line fix: add `redundant_rng = celt.rng;` between the CELT encode call and `ResetState` in the CELTŌåÆSILK redundancy block.

### Next-bug notes (for the next agent)
- Remaining failure: `enc_final_range != dec_final_range` at `test_opus_encode.c:501`, typically in `j=7` (first CELT_ONLY iteration after the Hybrid tests, `frame_size=2880` = 60 ms, stereo). Reproduce deterministically with `SEED=1`.
- **Instrumented trace shows:** the failing frame is NOT pure CELT. Due to the mode-transition-undo (`mode=prev_mode`) + bandwidth downgrade (`mode==HYBRID && bw<=WB ŌåÆ mode=SILK_ONLY`), the first j=7 frame is actually encoded as `mode=SILK_ONLY, redundancy=true, celt_to_silk=false, to_celt=true` ŌĆö i.e., a SILK frame with a 5 ms CELT redundancy tail. Trace values:
  - `[ENC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x015658e0`
  - `[DEC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x099b6b00`
  - Base (SILK final rng) matches. **The SILKŌåÆCELT `redundant_rng` differs between encoder and decoder.**
- **Hypothesis for the secondary bug:** the CELT encoder's 2.5 ms prefill (`ropus/src/opus/encoder.rs:2542-2548`) changes `celt_enc.rng` state (which is used as an LCG seed inside `quant_all_bands` ŌåÆ `bands.rs:1657 let mut ctx_seed = *seed;`) before the 5 ms redundancy encode runs. The decoder does not do a prefill, so its `celt_dec.rng` is still 0 at the start of the redundancy decode. This causes different PRNG seeds, different band-noise choices, different RDO decisions, different bitstreams ŌĆö OR the same bitstream but different final range-coder states. **BUT** the C reference has literally the same prefill-vs-no-prefill structure (`reference/src/opus_encoder.c:2536`), and the C test passes. So something else explains the divergence.
- **What I ruled out:**
  - The overwrite at `encode_multiframe` line 1842-1846 is suspicious (overwrites `self.range_final` with `celt.rng` after the multi-frame loop), but removing it didn't change the j=7 failure.   - SILK base range_final matches between enc and dec ŌĆö not a SILK issue.
- **Investigate further:**
  1. Is the CELT encoder producing a different bitstream than what the decoder expects (compare the actual redundancy bytes in the packet against what the decoder reads)? Hex-dump both sides.
  2. Is there a CELT encoder state that the prefill contaminates that AFFECTS THE WRITTEN BITS but does NOT get reset by `ResetState`? (e.g., `spread_decision`, `tonal_average`, `consec_transient`, `intensity`, `delayed_intra` ŌĆö check each.)

  4. Compare ropus's `reset()` field-by-field against C's reset zone for CELT encoder.

Diagnostic instrumentation (removed before commit) is documented above ŌĆö re-apply with `eprintln!` in `encoder.rs::get_final_range`, `encoder.rs::update_state`, the SILKŌåÆCELT redundancy block, and `decoder.rs::decode_frame` redundancy block to trace.

## 2026-04-18 ŌĆö Secondary fix: SILKŌåÆCELT redundancy rng

### Context
Following the primary fix above (`190d4c6a`), the conformance `encode` test
continued to fail but on a NEW divergence: `redundant_rng` mismatch in the
SILKŌåÆCELT redundancy branch (mirror of the CELTŌåÆSILK direction, but opposite
control flow). Prior agent's journal notes pinpointed the failing frame as
`mode=SILK_ONLY, redundancy=true, celt_to_silk=false, to_celt=true` ŌĆö the
`mode-transition-undo + WB-downgrade` path at
`ropus/src/opus/encoder.rs:1487, 1643` that reclassifies a requested
`MODE_CELT_ONLY` (j=7 in `test_opus_encode.c`) as `SILK_ONLY` with a 5 ms CELT
redundancy tail.

### Reproducer
Built a focused standalone regression test at
`harness/tests/c_ref_differential.rs::test_encoder_final_range_matches_decoder_on_silk_to_celt_redundancy`:
3├Ś Hybrid-FB 20 ms stereo warmup frames to set `prev_mode=HYBRID`, then
trigger `MODE_CELT_ONLY` + `WIDEBAND` + 60 ms stereo. Asserts
`enc.get_final_range() == ropus_dec.get_final_range() ==
c_ref_dec.get_final_range()`. Failed at the trigger frame before the fix.

### Diagnosis (confidence 5/5)
Byte-level instrumentation in `celt_encode_core` showed
`nb_compressed_bytes` being SHRUNK from the caller-supplied
`redundancy_bytes=92` down to `18` at the CBR rate-clamping branch
(`ropus/src/celt/encoder.rs:2237-2241`). This happened because `st.bitrate`
was **stale** (e.g. `29200` from a prior Hybrid CELT encode) instead of the
expected `OPUS_BITRATE_MAX` that `OpusEncoder` thought it had set via
`celt.ctl(CeltEncoderCtl::SetBitrate(OPUS_BITRATE_MAX))` at
`ropus/src/opus/encoder.rs:2539`.

Root cause: **two different `OPUS_BITRATE_MAX` constants**:
- `ropus/src/opus/encoder.rs:39`: `OPUS_BITRATE_MAX = -1` (correct, matches
  C `reference/include/opus_defines.h:212`).
- `ropus/src/celt/encoder.rs:58`: `OPUS_BITRATE_MAX = -1000` (WRONG ŌĆö this
  is C's `OPUS_AUTO` value).

`OpusEncoder` called `SetBitrate(-1)` via the opus-level constant. The CELT
ctl handler at `ropus/src/celt/encoder.rs:3280-3283` validates against the
celt-level constant (`-1000`): `-1 <= 500 && -1 != -1000` ŌåÆ `true` ŌåÆ
`OPUS_BAD_ARG`. The ctl SILENTLY failed (no return-code check at the opus
site), leaving `celt.bitrate` stale. The subsequent redundancy encode's
CBR clamp then over-shrunk the storage ŌåÆ `enc_ref.shrink(18)` ŌåÆ 18-byte
encoder storage vs 92-byte decoder storage ŌåÆ final `rng` values diverged
even though the on-wire bytes matched.

The reason the CELTŌåÆSILK direction (primary fix) wasn't also broken: CELT's
`ResetState` runs BEFORE `SetBitrate(OPUS_BITRATE_MAX)` in that branch. The
CeltEncoder's default `bitrate` is `OPUS_BITRATE_MAX` (celt-level `-1000`
before fix). So the stale-bitrate check `st.bitrate != OPUS_BITRATE_MAX`
evaluated `-1000 != -1000 = false` ŌåÆ CBR clamp skipped. The SILKŌåÆCELT
branch, however, runs AFTER a potential main CELT encode that sets
`st.bitrate` to a real positive value (e.g. `bitrate_bps - silk_rate`),
which then persists through `ResetState` (the C `ENCODER_RESET_START`
marker excludes `bitrate` from the reset region) ŌĆö exposing the bug.

### Fix
Single-line constant: `ropus/src/celt/encoder.rs:58` changed from
`pub const OPUS_BITRATE_MAX: i32 = -1000;` to
`pub const OPUS_BITRATE_MAX: i32 = -1;`. Added an explanatory comment
about the numeric coincidence with `OPUS_BAD_ARG` (disjoint value spaces
in the C ABI).
