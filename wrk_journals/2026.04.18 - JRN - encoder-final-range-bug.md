# JRN - encoder-final-range-bug

## Date/Goal
- **Date:** 2026-04-18
- **Goal:** Fix `OpusEncoder::get_final_range()` divergence from the produced bitstream on the first WB activation after an intervening non-WB bandwidth in SILK force-mode. Surfaced by the official xiph/opus test suite (test_opus_encode.c:501) via capi FFI shim.

## Symptom
- Force-mode SILK, 6809 bps, 1920-sample 48 kHz frames, bandwidths cycling NB → MB → WB:
  - Iter 0 (NB): encoder range == decoder range == C-ref decoder range
  - Iter 1 (MB): all three match
  - **Iter 2 (first WB): encoder reports `0x0c4c9290`; both decoders see `0x43f05790`**
  - Iter 3-14: match again
- XOR delta = `0x4fbcc500`. Bitstream itself is correct (two independent decoders agree).

### [T+0] Investigation begin
Opened encoder/decoder/ref for reading.

## Diagnosis

### Mapped the three relevant sites
- ropus encoder `range_final`/`redundant_rng` lives at `ropus/src/opus/encoder.rs:1990-2565`.
- ropus decoder mirror at `ropus/src/opus/decoder.rs:835-1014`.
- C reference encoder at `reference/src/opus_encoder.c:1879-2553`.
- C reference decoder at `reference/src/opus_decoder.c:298-700`.

### Root cause (confidence 5/5)
**Missing `redundant_rng = celt.rng;` in ropus encoder's CELT→SILK redundancy branch.**

- C reference encoder line 2440: `celt_encoder_ctl(celt_enc, OPUS_GET_FINAL_RANGE(&redundant_rng));` captures CELT rng after encoding the 5ms CELT→SILK redundancy frame, BEFORE `OPUS_RESET_STATE` wipes it.
- ropus encoder `ropus/src/opus/encoder.rs:2393-2409` is missing this capture — it calls `celt.ctl(CeltEncoderCtl::ResetState)` (which sets `celt.rng = 0` via `CeltEncoder::reset()` at `ropus/src/celt/encoder.rs:3198`) WITHOUT first saving the rng.
- Downstream at line 2564, `range_final ^= redundant_rng` XORs with 0 instead of the real value.
- Decoder-side (both C and ropus) correctly captures the corresponding decoder rng during redundancy decode and XORs it in — hence the divergence.

### Why "first WB activation" specifically
`redundancy && celt_to_silk` fires only when `silk_bw_switch` is set at frame entry. `silk_bw_switch` is set the previous frame when SILK signals `opus_can_switch = switchReady && !nonfinal_frame`. SILK's `switchReady` signal fires only on upward SILK-internal sample-rate changes (12 kHz→16 kHz in the observed test). In the NB→MB→WB cycle, that step falls on MB→WB (iteration 1→2). Downward forced bw changes (WB→NB at iter 2→3) do not trigger `switchReady`. Hence exactly iteration 2 fails, iterations 0, 1, 3-14 match.

## Hypothesis
Single-line fix: add `redundant_rng = celt.rng;` between the CELT encode call and `ResetState` in the CELT→SILK redundancy block.

## Design doc

## Fix
- Added `redundant_rng = celt.rng;` between the CELT encode and `ResetState` in the CELT→SILK redundancy branch of `OpusEncoder::encode_frame_native` at `ropus/src/opus/encoder.rs:2406-2414`.
- Confirmed the fix with a focused differential test in `harness/tests/c_ref_differential.rs::test_encoder_final_range_matches_decoder_on_celt_to_silk_redundancy` that mirrors the phase-4 agent's reproducer (48 kHz mono, SILK force-mode, 6809 bps, 1920-sample frames, bandwidth cycle NB→MB→WB using a `generate_music`-style synthetic signal).

### Next-bug notes (for the next agent)
- Remaining failure: `enc_final_range != dec_final_range` at `test_opus_encode.c:501`, typically in `j=7` (first CELT_ONLY iteration after the Hybrid tests, `frame_size=2880` = 60 ms, stereo). Reproduce deterministically with `SEED=1`.
- **Instrumented trace shows:** the failing frame is NOT pure CELT. Due to the mode-transition-undo (`mode=prev_mode`) + bandwidth downgrade (`mode==HYBRID && bw<=WB → mode=SILK_ONLY`), the first j=7 frame is actually encoded as `mode=SILK_ONLY, redundancy=true, celt_to_silk=false, to_celt=true` — i.e., a SILK frame with a 5 ms CELT redundancy tail. Trace values:
  - `[ENC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x015658e0`
  - `[DEC_FINALIZE] mode=1000 base_range_final=0x0d72b0ed redundant_rng=0x099b6b00`
  - Base (SILK final rng) matches. **The SILK→CELT `redundant_rng` differs between encoder and decoder.**
- **Hypothesis for the secondary bug:** the CELT encoder's 2.5 ms prefill (`ropus/src/opus/encoder.rs:2542-2548`) changes `celt_enc.rng` state (which is used as an LCG seed inside `quant_all_bands` → `bands.rs:1657 let mut ctx_seed = *seed;`) before the 5 ms redundancy encode runs. The decoder does not do a prefill, so its `celt_dec.rng` is still 0 at the start of the redundancy decode. This causes different PRNG seeds, different band-noise choices, different RDO decisions, different bitstreams — OR the same bitstream but different final range-coder states. **BUT** the C reference has literally the same prefill-vs-no-prefill structure (`reference/src/opus_encoder.c:2536`), and the C test passes. So something else explains the divergence.
- **What I ruled out:**
  - The overwrite at `encode_multiframe` line 1842-1846 is suspicious (overwrites `self.range_final` with `celt.rng` after the multi-frame loop), but removing it didn't change the j=7 failure. The overwrite IS wrong in principle (see HLD §7b) — it clobbers the last sub-frame's XOR-mixed range_final with the raw CELT rng — but it doesn't trigger here because the failing frame goes through single-frame path (frame_size=2880 at SILK_ONLY is max-allowed single-frame, no multi-frame).
  - SILK base range_final matches between enc and dec — not a SILK issue.
- **Investigate further:**
  1. Is the CELT encoder producing a different bitstream than what the decoder expects (compare the actual redundancy bytes in the packet against what the decoder reads)? Hex-dump both sides.
  2. Is there a CELT encoder state that the prefill contaminates that AFFECTS THE WRITTEN BITS but does NOT get reset by `ResetState`? (e.g., `spread_decision`, `tonal_average`, `consec_transient`, `intensity`, `delayed_intra` — check each.)
  3. Alternatively: is ropus's `CeltEncoder::reset()` (encoder.rs:3194-3227) missing a field that C's `OPUS_RESET_STATE` resets? C's CELT reset is `OPUS_CUSTOM_RESET_START` region. Diff against `reference/celt/celt_encoder.c:OPUS_RESET_STATE` handler (probably in the `ctl` switch).
  4. Compare ropus's `reset()` field-by-field against C's reset zone for CELT encoder.

Diagnostic instrumentation (removed before commit) is documented above — re-apply with `eprintln!` in `encoder.rs::get_final_range`, `encoder.rs::update_state`, the SILK→CELT redundancy block, and `decoder.rs::decode_frame` redundancy block to trace.
