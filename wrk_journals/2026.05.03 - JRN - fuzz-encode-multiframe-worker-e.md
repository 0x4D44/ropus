## Triage and fix

I added temporary gated traces to compare CELT-side state after Worker D's
matched SILK result. The important observations:

- Frame 1 CELT control inputs matched: `nb_compressed_bytes`, `tell/tell_frac`,
  range `rng`, `start/end` bands, HYBRID flag, SILK info, VBR reservoir fields,
  transient decision, allocation, final `tell/rng`, and range encoder scalar
  state through `done`.
- The first actual CELT-side input mismatch was PCM amplitude entering CELT
  silence scanning: Rust `sample_max=32279`, C `sample_max=32278`, at PCM index
  283.
- Tracing Opus-layer `pcm_buf` showed the filter output matched before
  `gain_fade`: both sides had index 283 as `32279`. After `gain_fade`, C had
  `32278` and Rust still had `32279`.

Root cause: Rust optimized `gain_fade` by skipping the steady-state tail when
`g2 == Q15ONE`. The C reference does not skip it; it always applies
`MULT16_RES_Q15(g2, in)`. In fixed point, `Q15ONE` is `32767`, not a true
identity multiplier, so positive samples can reduce by one. That one-sample
PCM mismatch perturbed later CELT PVQ/range output while leaving broad
`tell/rng` milestones equal.

Fix: removed the `if g2 != Q15ONE` guard in Rust `gain_fade` and always apply
the tail loop, matching C `opus_encoder.c`.

Validation so far:

- Before fix, exact artifact replay failed with frame `1/2`, equal length
  `118B`, first diff at byte 70.
- After fix, exact artifact replay passed:
  `timeout 300 cargo +nightly fuzz run --fuzz-dir tests/fuzz fuzz_encode_multiframe tests/fuzz/artifacts/fuzz_encode_multiframe/crash-ceb56d54b25a81e37ea51213b118401c6a89b8b4 -- -runs=1`.

I removed the temporary CELT/range/Opus PCM traces after proving the root cause.
