## Sub-stage 6.3b — diagnostic + fix — DONE

**Diagnostic agent.** Used the `#[repr(C)]` struct-layout-equality
trick: run both C and Rust `run_analysis` on one frame, then
byte-compare the full `TonalityAnalysisState`. First differing
field localizes the drift stage.

**Finding.** First divergence at pipeline stage 1
(`downmix_and_resample`). The `inmem` buffer on the Rust side was
**4096x smaller** than on C — exactly `1 << SIG_SHIFT` (SIG_SHIFT=12).
Smoking gun.

**Root cause — NOT the port.** The bug was in the test harness's
caller-supplied `DownmixFunc` callback
(`harness/tests/c_ref_differential.rs::rust_downmix_float`
and the sibling `test_downmix_float` in `analysis.rs`). Both
callbacks used the **non-FIXED_POINT** form of `FLOAT2SIG`:
`x * 32768.0 as i32`. But the harness builds with `FIXED_POINT 1`
(see `harness/config.h:8`), so C uses the FIXED_POINT inline function
at `reference/celt/float_cast.h:166-172`:

```
FLOAT2SIG(x) = { x *= 32768 << SIG_SHIFT; clamp; round-half-even; }
```

Missing the `<<SIG_SHIFT` factor, the clamp, and the rounding mode
produces exactly the observed ~4096x scale gap plus ULP drift.
Cascaded through FFT → band energies → BFCC → MLP = ~60 ULPs of
drift in `music_prob`. The `ropus/src/opus/analysis.rs` port
itself was correct all along.

**Fix.** Both test-harness downmix callbacks now implement the
FIXED_POINT `FLOAT2SIG` chain exactly: multiply by `134_217_728.0`
(= `32768 << 12`), clamp to `+/-(65536 << 12)`, `round_ties_even()
as i32`. Non-production test-only code; zero impact on the shipping
analysis module.

**Validation.**
- `test_analysis_run_matches_c_reference` passes bit-exactly.
- `#[ignore]` attribute removed.
- Full state byte-comparison confirms byte-equality after frame 1,
  excluding only the `arch` metadata tag (0 vs 4 for scalar vs SSE —
  no computational effect on x86 since `opus_fft` only SIMD-dispatches
  on ARM NE10).
- Added two `#[ignore]`d diagnostic helpers for future use.
- `cargo test -p ropus-harness --release --test c_ref_differential
  -- --include-ignored`: 19/19 pass.
- `cargo test -p ropus --lib --release`: 1804/1 pre-existing.

**Implication for Stage 6.4 (wiring).** The encoder's real downmix
callback must apply FIXED_POINT `FLOAT2SIG` to match
`reference/src/opus_encoder.c:748-778`. Exact scaling of downmix
output into `inmem` is load-bearing.

**Outcome.** Stage 6.3 closes tier-1 at module level. Ready for
Stage 6.4 wiring.

## Sub-stage 6.5 — PCM-level drift measurement (post-closure validation)

**Motivation.** Arthur's pushback on tier-2 closure: *"94.9% on music
doesn't sound very good."* Fair point — byte-identity of compressed
output is a blunt metric. The question that actually matters: does the
byte-drift translate to audible audio-quality loss, or do the two
bitstreams decode to perceptually equivalent PCM?

**Experiment.** New binary `harness/src/bin/pcm_drift.rs` runs the full
encode→decode pipeline on both sides and performs a 3-way measurement:

1.  Encode each frame with both the C reference and the Rust port
    (same profile: 64 kbps CBR, complexity 10, 20 ms, AUDIO, FB).
2.  Decode each packet stream with an independent C reference decoder
    (so decoder state evolves separately — we isolate encoder drift
    from decoder drift).
3.  Compare three ways:
    -   direct (C-decoded vs Rust-decoded) — what the byte divergence
        actually translates to at the PCM level.
    -   codec transparency (C-decoded vs input) — baseline lossy-codec
        error of the reference encoder.
    -   codec transparency (Rust-decoded vs input) — whether the Rust
        port is a worse approximation of the source than C is.
    Alignment applies the standard Opus FB 312-sample pre-skip.

**Interpretation.** The differential numbers look alarming in
isolation: 34 dB SNR between C and Rust decoded audio, with 20.7 % peak
amplitude difference. But the 3-way comparison is the perceptually
meaningful one, and it reports the two encoders as **identically
close to the input** (delta below 0.001 dB, i.e. numerical noise — in
the speech case, Rust's RMS error vs input is 0.03 LSB *lower* than C's).

Geometrically: both encoders land on the same-radius sphere around the
input signal, just at different points. The 93-LSB RMS "error" between
the two decoded streams is not a loss of quality — it's two equally
valid paths through the codec's decision space. The low 14 dB absolute
transparency SNR is a feature of lossy transform coding (Opus minimizes
perceptual distortion, not L2 distance); what matters is that both
encoders exhibit the same transparency floor.

**Tier-2 acceptance is confirmed, and the "94.9% byte-exact" framing
was genuinely understating the quality story.** At the decoded-PCM
level the Rust port is audio-quality-equivalent to the C reference on
full-length real content. Tier-1 (byte-exact) remains desirable for
bit-identity with published reference streams, but is not a
perceptual-quality gate.

**Takeaway for future work.** If a tier-1 campaign is ever started,
this binary is the regression harness: any fix should preserve Δ=0.000
dB while reducing byte-drift toward zero. A tier-1 fix must not make
the Rust port a measurably worse approximation of the input than C is.
