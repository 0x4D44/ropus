# 2026-04-18 — CELT encoder: targeted unchecked indexing

## Context

Two earlier journals from today:
- `restore-unchecked-indexing.md` — flipped the `uc!` macros back to
  unchecked on the 95 existing CELT FFT/MDCT call sites. Recovered
  decode parity across all codec modes.
- `silk-encoder-unchecked-indexing.md` — added targeted `uc!` to SILK
  encoder hot inner loops. Every encoder row improved 4–19pp.

CELT encoder **body** (pitch search, quant_bands, alg_quant, rate
allocation, bands) was never touched — only the shared FFT/MDCT
primitives. Explore survey + this journal: extend the same exercise
to the CELT encoder.

Per the SILK-encoder lesson, trivial `for i in 0..n` loops where the
compiler can already hoist the bounds check (e.g. `celt_inner_prod`,
`dual_inner_prod`) are left alone — adding `uc!` there perturbed codegen
without gain in the SILK pass, and there's no reason to expect otherwise
here. Focus: ranks 1–8.

## Changes

`ropus/src/celt/simd.rs`:
- `xcorr_kernel_simd` — main j-loop `x[j]`, `y[j..j+3]` → `uc!`.
- `celt_maxabs32_simd` — 4-wide inner chunk load + tail loop → `uc!`.
- `denormalise_band_simd` — 4-wide load/store + tail → `uc!`/`uc_set!`.
- `mdct_window_simd` — both halves (forward + reverse) → `uc!`/`uc_set!`.

`ropus/src/celt/vq.rs`:
- `op_pvq_search` — sign-strip loop, pre-search L1 projection loop, and
  greedy refinement inner j-loop (1..n) all → `uc!`/`uc_set!`. This is
  the O(n·k)-per-band hot path.
- `exp_rotation1` — forward and backward sweeps `x[i]`, `x[i+stride]`
  → `uc!`/`uc_set!`.
- `normalise_residual` — per-band normalization loop → `uc!`/`uc_set!`.
- `extract_collapse_mask` — nested `iy[i*n0 + j]` → `uc!`.

No new `unsafe` blocks. Every call site has statically-bounded iteration
ranges against slices whose lengths satisfy the bound — documented
inline or via `debug_assert!` preconditions.

### Headlines

1. **CELT encoder gains are modest** (1–3pp across most synthetic
   vectors). This is not surprising — the hot FFT/MDCT was already
   `uc!`-enabled from Stage 1, and the remaining CELT encoder time
   is dominated by the range coder's control-flow-heavy entropy
   paths, which don't benefit from indexing fixes.
2. **Real-content MUSIC sees a meaningful win: −7pp encode.** This is
   a 3+ min stereo track exercising the full CELT encoder under
   realistic audio statistics, and it moves ropus from 1.26× to 1.19×
   slower than C on 48k stereo 128kbps.
3. **SILK/Hybrid rows show decode drift of +3–5pp noise.** My edits
   don't touch SILK decode, so this is run-to-run PGO variance, not
   signal. The CELT-only vectors (which my edits actually affect)
   show decode flat-to-slightly-better.
4. **Code structure unchanged**: no algorithmic rewrites, no new
   `unsafe` blocks outside the existing `uc!`/`uc_set!` macros. Every
   call site has a documented in-bounds argument (inline comment or
   `debug_assert!`).

### Remaining gap

After SILK-encoder Stage 2 + this Stage 3, the encoder is ~1.19×
slower than C on MUSIC. Closing further would require:

- SIMD path rework (the `wide` crate's `i32x4` is fine for pitch
  xcorr but the fixed-point Q-shift sequences in `denormalise_band`
  and `mdct_window` fall back to scalar because they need 64-bit
  multiply-shift; hand-written SSE intrinsics would help).
- Algorithmic work in pitch search / rate allocation, not indexing.

These are out of scope for indexing-only work.

## What didn't work (and why it's not in this commit)

Not tried in this pass:

- `quant_coarse_energy_impl` / `quant_fine_energy` (`idx = i + c*nb_ebands`
  stride). Profiling showed these are dominated by `ec_laplace_encode`
  and `enc.tell()` call costs, not index arithmetic. Indexing `uc!`
  would barely move the needle.
- `anti_collapse` nested `x_[(c*size + ebands<<lm) + (j<<lm) + k]`.
  Only runs when a collapse is detected; not per-frame hot.
- `celt_inner_prod` / `dual_inner_prod` in `pitch.rs`. These are
  trivial `for i in 0..n` loops — per the SILK Stage 2 journal,
  the compiler already hoists the bounds check and adding `uc!`
  perturbed codegen without gain.

## Conclusion

Small but real CELT-encoder win on real content (MUSIC −7pp encode).
Stage 2 (SILK encoder) remains the bigger lever by ~5×. With all
three stages landed:

- CELT FFT/MDCT decode parity (Stage 1)
- SILK encoder hot loops (Stage 2, −10 to −19pp across SILK-using modes)
- CELT encoder hot loops (Stage 3, −7pp on MUSIC, −1 to −3pp elsewhere)

Indexing recovery is essentially complete — further encoder perf
requires profiling-guided algorithmic work or hand-written SIMD.
