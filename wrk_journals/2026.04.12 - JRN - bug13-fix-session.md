## Pre-Investigation Analysis (Supervisor)

After reading the journal, all 5 out-of-scope findings, the C reference, and the Rust implementation, I've formed a strong hypothesis:

### Primary Hypothesis: L2 (silk_nlsf_stabilize fallback saturation)

**C** (`NLSF_stabilize.c:133`):
```c
NLSF_Q15[i] = silk_max_int(NLSF_Q15[i], silk_ADD_SAT16(NLSF_Q15[i-1], NDeltaMin_Q15[i]));
```

**Rust** (`common.rs:1358-1359`):
```rust
let min_val = nlsf_q15[i - 1] as i32 + delta_min_q15[i] as i32;
nlsf_q15[i] = imax(nlsf_q15[i] as i32, min_val) as i16;
```

The C code uses `silk_ADD_SAT16` which saturates the i16 addition at ±32767. The Rust code adds in i32, then truncates via `as i16` — which WRAPS instead of saturating when the sum exceeds i16 range.

**Why this matches all Bug #13 symptoms:**
- **Only WB (order 16):** 16 NLSFs packed into [0, 32768) → tighter spacing → fallback path more likely to trigger with overflow
- **NB/MB clean:** Order 10, fewer NLSFs, less overflow risk even in fallback
- **Only garbage inputs:** Well-formed NLSF indices stabilize within 20 loops (fallback never triggers)
- **Cascading divergence:** Corrupted NLSFs → corrupted LPC coefficients → divergent decode starting at subframe where corrupted coefficients first apply

### Finding L1 (silk_rshift_round wrapping) — NOT a bug
The current `wrapping_add` matches C's undefined-but-practical 2's-complement behavior. Dropped from fix list.

### Findings L3, A, B — Independent CNG/PLC bugs
These only affect PLC (loss_cnt > 0) and CNG paths. Bug #13 triggers on first decode of a garbage packet (loss_cnt = 0), so these cannot be the root cause. Still independently valuable to fix.

| ID | Location | Bug | Impact on #13 |
|----|----------|-----|---------------|
| L2 | `common.rs:1358` | Non-saturating i16 add in nlsf_stabilize fallback | **PRIMARY CANDIDATE** |
| L3 | `decoder.rs:1089` | plc_update searches last subframe only (C searches strongest) | None (PLC only) |
| A  | `decoder.rs:1448` | CNG gain during loss missing PLC-energy subtraction | None (CNG/loss only) |
| B  | `decoder.rs:1438` | CNG threshold comparison operands swapped vs C | None (CNG only) |

### T+45min — ROOT CAUSE FOUND

**Trace agent identified the root cause:**

Missing write-back of `nlsf_interp_coef_q2` in `silk_decode_parameters`.

**C** (`decode_parameters.c:59-60`):
```c
if( psDec->first_frame_after_reset == 1 ) {
    psDec->indices.NLSFInterpCoef_Q2 = 4;  // Write to persistent state
```

**Rust** (`decoder.rs:771-775`):
```rust
let interp_coef = if dec.first_frame_after_reset {
    4i32 // Local variable only — never written back!
} else { ... }
```

Later, `decode_core` reads `dec.indices.nlsf_interp_coef_q2` to compute the NLSF interpolation flag:
```rust
let nlsf_interp = (dec.indices.nlsf_interp_coef_q2 as i32) < 4;
```

In C, this reads 4 (from the write-back) → `nlsf_interp = false`.
In Rust, this reads the stale bitstream value (e.g., 1) → `nlsf_interp = true`.

This incorrect flag causes erroneous re-whitening at subframe k=2, which corrupts the LPC filter state and produces ±1 sample divergence starting at sample 173.

**Why WB-only**: NB/MB use `nb_subfr=2`, so k=2 never exists, and the re-whitening code is never reached. The bug requires WB (4 subframes) + voiced + first_frame_after_reset + decoded interp_coef != 4.

**Fix applied** (one line added to `silk_decode_parameters`):
```rust
dec.indices.nlsf_interp_coef_q2 = 4;
```
