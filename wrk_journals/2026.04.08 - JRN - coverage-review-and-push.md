# Coverage Review And Push

2026-04-08

Started by establishing a real `cargo +nightly llvm-cov --branch` baseline instead of guessing from the existing test count. Stable failed immediately because branch coverage still needs nightly. Nightly was already installed, so that part was easy.

The first real blocker was more annoying than hard: coverage died on the placeholder `src/main.rs` binary on Windows. Normal `cargo test` worked, but `cargo llvm-cov` did not. I handed that manifest fix to a worker and reviewed it locally.

Baseline result is rough:

- Lines: 48.95%
- Functions: 64.31%
- Regions: 50.36%
- Branches: 35.10%

The shape of the repo is clear now. Small math/helper modules are mostly fine; the big stateful codec paths are not. The ugly holes are exactly where I expected the pain to be: SILK encoder/decoder, Opus encoder/decoder, multistream, plus a few completely under-tested utility modules.

Emotional temperature: focused, slightly irritated by the harness blocker, not surprised by the low branch number. The 98% ask is aggressive enough that I need to stay honest about the math. We are not polishing the last few percent here; we are trying to drag a large codec through a serious test expansion campaign.

Next step is to turn the existing `mdopus-compare` harness from a CLI pile into actual executed tests. That should give the biggest first jump without inventing new infrastructure.
