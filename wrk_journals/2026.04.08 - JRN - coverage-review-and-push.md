# Coverage Review And Push

2026-04-08

Started by establishing a real `cargo +nightly llvm-cov --branch` baseline instead of guessing from the existing test count. Stable failed immediately because branch coverage still needs nightly. Nightly was already installed, so that part was easy.

The first real blocker was more annoying than hard: coverage died on the placeholder `src/main.rs` binary on Windows. Normal `cargo test` worked, but `cargo llvm-cov` did not. I handed that manifest fix to a worker and reviewed it locally.

Baseline result is rough:

- Lines: 48.95%
- Functions: 64.31%
- Regions: 50.36%
- Branches: 35.10%

The shape of the repo is clear now. Small math/helper modules are mostly fine; the big stateful codec paths are not. The ugly holes are exactly where I expected the pain to be: SILK encoder/decoder, Opus encoder/decoder, multistream, plus a few completely under-tested utility modules.

Emotional temperature: focused, slightly irritated by the harness blocker, not surprised by the low branch number. The 98% ask is aggressive enough that I need to stay honest about the math. We are not polishing the last few percent here; we are trying to drag a large codec through a serious test expansion campaign.

Next step is to turn the existing `mdopus-compare` harness from a CLI pile into actual executed tests. That should give the biggest first jump without inventing new infrastructure.

Later update:

Stage 1 worked much better than expected. The harness smoke tests were stable and fast, which is not the usual outcome with this kind of integration-heavy coverage push. The new overall numbers are:

- Lines: 77.41%
- Functions: 79.32%
- Regions: 78.63%
- Branches: 65.69%

That is a real jump, not noise. The source-side additions in `celt::lpc`, `celt::ec_ctx`, `celt::rate`, `silk::common`, and `opus::encoder` all paid off, and the harness tests pulled a lot of useful execution through Opus/SILK/multistream code without blowing up runtime.

Current feeling: more optimistic, but still wary. The repo is no longer in embarrassing coverage territory, but 98% remains brutal. The easy structural wins are mostly gone now. The next ugly truth is that DNN coverage is terrible, and multistream still has a lot of cold code.

Next move is to push on DNN (`core`, `lpcnet`, `fargan`) and then come back to multistream branch gaps. That feels like the only honest route left if the target stays at 98.

Later update:

Stage 2 landed cleanly. Full library tests passed, the harness smoke tests still passed, and the next `cargo +nightly llvm-cov --branch` run came back at:

- Lines: 85.48%
- Functions: 85.95%
- Regions: 86.35%
- Branches: 70.35%

That is enough movement that I do not need to second-guess the direction. The DNN work was worth it. `core` and `fargan` are no longer embarrassing, `lpcnet` at least looks salvageable now, and the multistream tests finally reached the real parser and roundtrip paths instead of just skimming constructors.

Current feeling: relieved, sharper, still skeptical about the last twelve points. The repo has crossed out of "coverage emergency" territory, but the remaining gap is the hard kind: large stateful encoders, decoders, and packet machinery. That means slower gains per test and more chances to write brittle garbage if I get lazy.

The next honest push is obvious now: `silk::encoder`, then `opus::encoder`, then `opus::decoder` and `opus::repacketizer`, with another pass over `lpcnet` if the branch curve still refuses to move. The work is getting narrower and less glamorous, which usually means it is finally real.

Later update:

Stage 3 is in. The good news is that everything passed together: full library tests, harness smoke tests, and the next branch-coverage run. The less comforting news is that the gains were modest:

- Lines: 86.81%
- Functions: 88.06%
- Regions: 87.57%
- Branches: 71.83%

That confirms what the file-level tables were already hinting at. I am no longer dragging coverage upward with obvious missing tests. I am paying for every extra point now, mostly in the big state machines and packet parsers where each branch needs bespoke setup.

Current feeling: steady, less optimistic than after Stage 2, but not discouraged. The slope flattening is annoying, though it is also honest. If I keep pretending 98% is around the corner, I will start writing low-value tests just to soothe the number, and that would be garbage work.

The next pass should probably split into two different kinds of work instead of pretending everything is the same problem:

- line-heavy direct tests in `dnn::lpcnet` and maybe `dnn::pitchdnn`
- branch-heavy cleanup in `silk::encoder`, `opus::repacketizer`, `silk::decoder`, and `opus::decoder`

That is slower, but at least it matches the actual shape of the remaining debt.
