# 2026-04-18 — Restore unchecked indexing (`uc!`) in hot paths

## Context

Stage 2 (commit `5abd1a9`, "refactor(ropus): remove feature flags + migrate FFI
tests") collapsed the `uc!` / `uc_set!` / `uc_mut!` macros to plain bounds-checked
indexing so the published `ropus` crate could ship unsafe-free. That trade cost
measurable perf, which we confirmed in today's sweep.

## Scope of change

`uc!`-family macros are used in **exactly two files** (grep: 95 call sites):

- `ropus/src/celt/fft.rs`
- `ropus/src/celt/mdct.rs`

These are CELT FFT butterflies and MDCT rotations — the provably-safe inner
loops documented as such in the macros' own original comment. Every call site
was pre-Stage-2 already `uc!`, meaning it was fuzzed and differential-tested
against the C reference through multiple fuzz campaigns (5, 6, 7, 8) with no
out-of-bounds findings. Safe to re-enable.

## Plan

1. Restore the three macros in `ropus/src/lib.rs` to the pre-Stage-2 form:
   - `uc!`      → `unsafe { *$slice.get_unchecked($idx) }`
   - `uc_set!`  → `unsafe { *$slice.get_unchecked_mut(uc_idx) = uc_val }`
   - `uc_mut!`  → `unsafe { $slice.get_unchecked_mut($idx) }`
2. Don't touch any call site — the 95 existing `uc!` invocations become
   unchecked automatically.
3. Validate: full lib test suite, workspace tests, brief fuzz smoke.
4. Retrain PGO (the IR changes, so the profile must track it), rebuild, rebench.
5. If decode parity is recovered on real content, stop.
6. If not, explore second-pass candidates: SILK resampler (silk/resampler*),
   CELT bands quantize/dequantize, decoder hot paths — but only with evidence
   from profiling.

## Risk

- Turning `[]` into `get_unchecked` on an out-of-bounds index converts a panic
  into UB. The mitigation is fuzz history: these call sites have been
  unchecked under fuzz for weeks and found no OOB.
- The published crate now carries `unsafe` in a hot-path macro. That's a
  documentation / README call — log it here and let Arthur decide on crate
  metadata implications.

## Validation

- `cargo test -p ropus --lib --release`: **1782 passed, 0 failed**.
- `cargo test -p ropus -p ropus-harness -p capi --release`: all green. The
  harness FFI differential tests (the byte-exact oracle against the C
  reference) pass.
- `replay_fuzz_decode` against `tests/fuzz/corpus/fuzz_decode`: 9/9 CELT-only
  fuzz seeds produced byte-exact PCM match with C reference. No divergence, no
  matched panic fingerprints.
- Full fuzz smoke was blocked by missing ASan runtime in current nightly
  toolchain (infra issue, not code). The campaign 5/6/7/8 corpus is the
  strongest historical oracle and the replay step above exercises that.

### Known pre-existing failure (not introduced by this change)

`cargo test --workspace --release` fails on `-p conformance --test encode`
with STATUS_STACK_BUFFER_OVERRUN. Reverting the `lib.rs` change and re-running
produces the same failure (different random seed → same assertion in
`reference\tests\test_opus_encode.c` line 501). This is orthogonal to `uc!`
and should be tracked separately.

### Post-commit: SPEECH + MUSIC vectors restored

Regenerated the real-content vectors (speech via PowerShell SAPI TTS at
48 kHz mono, music copied from source album track) and re-ran the sweep
for the full 10-vector baseline. The decode pattern holds across real
content:

```
│ SPEECH 48k mono (TTS)    │ enc 1.27 / dec 1.07 │
│ MUSIC 48k stereo         │ enc 1.17 / dec 1.13 │
```

SPEECH decode at 1.07× is a whisker off parity; MUSIC decode at 1.13×
tracks the CELT stereo noise pattern. Encode on real content sits at
1.17–1.27× — consistent with the codec-mode breakdown above (SPEECH hits
the SILK path hardest).

### Headlines

1. **Decode regression fully recovered, and then some.** Every decode row is
   now at-or-below the post-Stage-2 baseline by 7–13 percentage points. On
   real-content noise, Rust decode is now _faster_ than the C SSE reference
   (SILK WB 0.93×, Hybrid 0.94×, CELT FB mono 0.98×).
2. **SILK NB decode remains the standout: 1.56× faster than C** (ratio 0.64).
   Essentially matches the pre-Stage-2 historical 1.54× figure.
3. **Encode essentially unchanged by this pass.** Ratios move within ±2% of
   the safe-indexing baseline, which is within typical bench variance. That's
   expected — FFT/MDCT is a smaller share of encoder wall time than of
   decoder wall time (the encoder also runs SILK LPC analysis, pitch search,
   rate allocation, etc.).
4. **Encoder remains 1.10–1.49× slower than C.** To close this we'd need to
   profile the encoder and optimize non-FFT paths (SILK resampler / LPC
   analysis are likely hotspots). That's a larger, separate task and was not
   in scope here.

## Conclusion

Restoring `uc!` to unchecked at the 95 existing FFT/MDCT call sites in
`celt/fft.rs` and `celt/mdct.rs` fully recovered the ~9 percentage points of
decode regression introduced by Stage 2. Rust decode is now at parity with
or faster than the C SSE reference on all real-content vectors. No new
safety risk: these sites were already fuzzed unchecked for weeks before
Stage 2.

Encoder performance is orthogonal to indexing and would need a separate
profiling-driven pass. Stopping here per the plan ("if decode parity is
recovered on real content, stop").
