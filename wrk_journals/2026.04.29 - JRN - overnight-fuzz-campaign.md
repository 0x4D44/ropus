## Known caveats from Campaign 9 (2026-04-19)

The fuzz harness compiles the C reference with `DISABLE_FLOAT_API` *off*,
so the C encoder runs `analysis.c`. ropus does not yet have an analysis port
(Stage 6 still in flight). At `complexity ≥ 10` and `Fs ≥ 16000` with
`app != RESTRICTED_SILK`, the C side gets a `valid` AnalysisInfo and Rust
does not — they diverge by design. Campaign 9 logged 4 panics from that
exact source within 5 minutes of launch and triaged them all as
"expected-divergence, no fix".

**Mitigation for tonight**: keep the same target list, but bias the corpus
toward inputs that don't touch the analysis path (lower complexity, or
`RESTRICTED_SILK` mode), and triage any encoder-side divergence panic
against the Campaign 9 fingerprint before opening a fix loop. Anything
NOT matching that pattern (decoder panic, safety panic, OOB, sample-count
contract violation) is a real bug.

## Triage workflow per crash

1. Capture `crash_<hash>.bin` location and minimised input via `cargo fuzz tmin`.
2. Spawn an investigation agent (read-only) with: minimised bytes, target name,
   stack trace from a `cargo +nightly fuzz run … <crash>.bin` replay.
3. Investigation agent produces a hypothesis and points at a file:line.
4. Spawn a fix agent (write-permitted) with the diagnosis. It applies the
   minimal fix and a regression test that encodes the input by value (per
   the `tests/fuzz/README.md` rule: tests don't read from `tests/fuzz/crashes/`).
5. Spawn a review agent on the diff. If it flags issues, a markup agent
   addresses them.
6. Re-run the crash file once to confirm; commit only if Arthur says so —
   default tonight is leave the fix on disk uncommitted, mark "ready for
   morning review".

## Interpretation

**The good.** No crashes is the right outcome for a maturing codec at
this stage. Campaign 9 (April 19) flagged 4 panics within 5 minutes,
all triaged as "expected divergence from the missing analysis port" —
not real defects. Tonight's run on the same fuzz harness, same pinned
reference, same complexity ceiling produced **zero** such panics.
That's consistent with two improvements since Campaign 9:

1. The corpus carry-over: targets started with 2-3 thousand seeds
   already covering the analysis-divergent paths, so libFuzzer's
   mutator landed elsewhere.
2. The ten days of API-side work (commits `6928557…b5193f9`) were on
   `Encoder` runtime setters — coupling to the existing encode path,
   no new analysis call sites — so the divergent C-side analyzer
   wasn't exercised any more aggressively than before.

**The neutral.** Differential targets are now corpus-saturated against
this seed set. `fuzz_decode` plateaued at cov 5501 from minute 78
onwards — the nine new branches found over 8 h were all in
`fuzz_decode_safety` / `fuzz_encode` / `fuzz_encode_safety` /
`fuzz_roundtrip` / `fuzz_roundtrip_safety` (low double digits each).
Future campaigns should reseed corpora from richer real-world streams
or run with `-len_control=0` to push libFuzzer past the saturation
plateau.

**The dull.** `fuzz_packet_parse` ran 16.5 billion executions and added
0 new units. Its reachable code is fully covered by the 312 seeds.
Either retire it, drop its slice of CPU into a more productive target,
or graduate it to a property-test instead of a fuzz target.

`fuzz_encode_multiframe` is similar (5.7 B execs, 2 new units, 2
corpus files). Its 5–10-frame protocol is too narrow to mutate
productively without TOC-aware seed structure. Worth investing in
`tools/gen_fuzz_seeds.py` extensions before next campaign.
