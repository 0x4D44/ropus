# 2026.04.11 — Decoder PLC/LPC Precision Fix

## Objective

Fix the remaining 7 decoder PCM mismatches across 4 transition tests.
All are decoder-side: encoder packets match, but C and Rust decoders
produce different PCM at mode transition frames.

## Root cause (from previous session)

The SILK PLC synthesis produces matching 16-bit output but slightly
different 32-bit internal LPC state (`s_lpc_q14_buf`). This state
cascades through `outBuf` to affect the crossfade at mode transitions.

The divergence is in the LPC synthesis loop — a rounding/accumulation
difference in `silk_SMLAWB` multiply-accumulate producing different
32-bit results that truncate to the same 16-bit value.

## Previous attempt (reverted)

A major PLC rewrite (~350 lines) improved structural C-parity but
produced no improvement in mismatch count and broke 2 unit tests.
Reverted; only the `loss_cnt` fix was kept.

## Strategy

Targeted LPC synthesis trace: add per-sample 32-bit comparison at the
exact point where `s_lpc_q14_buf` first diverges between C and Rust.
Then find the single multiply-accumulate operation that differs.

### T+3 — Critical realization: SILK PLC is the wrong target

Traced the Opus decoder transition path (`src/opus/decoder.rs`):

**For CELT→SILK transitions** (which is what the 3 single-mismatch tests do):
- Line 665-668: `decode_frame(None, ...)` uses `self.prev_mode` for PLC mode
- `prev_mode = MODE_CELT_ONLY` → the transition buffer is generated by **CELT PLC**, not SILK PLC
- All SILK PLC fixes are irrelevant for these mismatches

**For the 4 FEC mismatches** (step 1→2):
- Steps 1 and 2 are BOTH Hybrid FB (same mode)
- No mode transition occurs → no PLC crossfade → no PLC involvement at all
- These 4 mismatches are decoder divergence on identical HYBRID packets WITHOUT any mode transition

**This explains why**:
- The SILK PLC rewrite had no effect (wrong decoder path)
- 5 targeted SILK PLC fixes had no effect (wrong decoder path)
- The mismatches persist regardless of PLC code quality

### T+6 — celt_fir saturation fix: ALL TESTS PASS

Agent traced the last 2 mismatches to a 1-LSB saturation difference in
`celt_fir`. Rust used `sround16(sum, SIG_SHIFT)` which clamps to
[-32767, 32767] (symmetric). C reference compiled with SSE4.1 uses
`packssdw` / `SATURATE16` which clamps to [-32768, 32767] (full i16).

The 1-LSB difference at -32768 cascaded through 8 stages:
1. celt_fir output: -32768 vs -32767
2. celt_maxabs16: 32768 vs 32767
3. celt_zlog2: 15 vs 14
4. Energy shift: 10 vs 8
5. Energy accumulation: completely different
6. frac_div32 → celt_sqrt → decay: 30393 vs 30394
7. Extrapolation: 4096-unit differences
8. IIR/deemphasis: 107 of 960 output samples differ by 1-3

**Fix:** Replace `sround16(sum, SIG_SHIFT)` with `sat16(pshr32(sum, SIG_SHIFT)) as i32`
in `celt_fir` (5 lines in `src/celt/lpc.rs`).
