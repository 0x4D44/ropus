## Notes, blockers, dead ends

### 1. Section 6 landed first

- Added 16 new `OPUS_GET_*_REQUEST` constants to `tests/harness/bindings.rs`.
- `state_checkpoint_diffs()` compares 16 encoder getters and 7 decoder
  getters between C and Rust. All i32 except `final_range` (u32) and
  decoder `phase_inversion_disabled` (bool on Rust side).
- `get_in_dtx` deliberately skipped — no Rust getter, and the HLD Section 1
  forbids `src/` changes.
- Wired `--state-check-interval N` and `--state-check-loose` flags.
  Strict-by-default (Decision 5). Defaults: interval 1000, strict on.
- A final checkpoint runs unconditionally at end-of-test to catch drift
  that never crossed a periodic boundary.

### 2. Section 3 landed second

- Added a `Burst`/`BurstStep` structure with 12 named bursts in a
  `static BURSTS: &[Burst]`. Each burst mutates a neutral baseline config,
  so bursts do not inherit state from the random walk.
- The scheduler selects bursts deterministically via
  `(frame_idx / burst_interval) % usable_bursts.len()`. `usable_bursts`
  filters out bursts that need more channels than the test has — so the
  `stereo_mono_*` bursts are skipped in mono runs without breaking the
  rotation.
- Burst state machine in `cmd_torture`:
  1. If currently in a burst, increment the frame-in-step counter. When
     the step hold count is reached, advance to the next step or (after
     the last step) restore the pre-burst random-walk config.
  2. Otherwise, if at a `burst_interval` boundary, save the current cfg
     and start a new burst.
  3. Otherwise, fall through to the random-walk `change_interval` check.
- Added `CurrentCfgSource { Random, Burst { name, step_idx, step_label } }`
  so every diagnostic line labels whether the failure came from a burst
  or a random config (Decision 8).
- Wired `--burst-interval N` flag. Default 200 (≈4 s between bursts at
  20 ms frames), 0 disables.

### 3. Section 2 landed third (also in cmd_transitions)

- New helpers `c_decode_packet` / `rust_decode_packet` that each take one
  decoder instance, run a packet, and return `(nsamples, final_range)`.
  Shared by both `cmd_torture` and `run_transition_sequence`.
- `cmd_torture` now allocates up to 4 decoders: a C and Rust decoder
  dedicated to the C-encoded stream (always), plus a separate C and Rust
  decoder dedicated to the Rust-encoded stream (only in `--cross-decode`).
- 2x2 decode matrix with strong/weak pass logic per HLD:
  - Strong (packets byte-equal): A == B == C == D required.
  - Weak (packets diverge): A == B (existing check) and C == D required.
- `run_transition_sequence` got the same treatment and now takes a
  `cross_decode: bool` parameter. `cmd_transitions(cross_decode: bool)`
  propagates the flag from the CLI to all 8 scripted sequences.
- CLI: `torture --cross-decode` and `transitions --cross-decode`.

### 4. tools/torture_test.sh wiring

- Added `torture_crossdec` (10 min cap, seed 44) after the existing
  `torture_stereo`.
- Added `mode_transitions_xdec` (`transitions --cross-decode`) after

### 7. Cross-decode finds extra signal

On failing transition tests, `transitions --cross-decode` reports
"N primary + M cross-decode mismatches" where M is 17–40 additional
failures that the encoder-only check did not catch — i.e. frames where
one of the two encoders produced a valid-looking packet that the
opposite decoder interpreted differently. This is precisely the bug
class Section 2 is designed to expose. On passing tests (e.g.
`Stereo->ForceMono->Auto->Stereo`), cross-decode cleanly reports
"cross-decode ok" with zero additional mismatches — no false positives.

### 8. Mid-session fix: strong cross-decode criterion is history-sensitive

First integration run surfaced a false-positive mode: cross-decode
reported "A vs C, packets equal" and "B vs D, packets equal" mismatches
on frames *after* the encoders had first diverged. 40 spurious failures
in one run.

Root cause: the HLD's "strong" criterion (A==B==C==D when packets are
byte-equal) is only valid when the four decoder instances share an
identical input history. Once the encoders have diverged at frame N,
`c_dec_cs` has seen the C-encoded stream from 0..N+K while `c_dec_rs`
has seen the Rust-encoded stream — so at any later frame K > N the
two C decoder instances have different internal state, and even if
`c_pkt == rust_pkt` at frame K the decoded PCMs will legitimately
differ. That is a property of decoder history divergence, not a
decoder bug.

The HLD author acknowledged this: "If the encoders already agree
byte-for-byte (which is the project's hard requirement), A == C and
B == D trivially". The qualifier "already agree byte-for-byte" means
"on every frame in history", not "on this frame". But that qualifier
is load-bearing and wasn't reflected in the implementation.

Fix: track `prior_divergence_seen` in both `cmd_torture` and
`run_transition_sequence`. The strong A==C / B==D check only runs
while this flag is false. The weak C==D check still runs on every
divergent frame (because `c_dec_rs` and `rust_dec_rs` both see the
Rust-encoded stream from frame 0 — their histories match regardless
of encoder drift). Result:

- Cross-decode smoke run (seed 200, 10s, bursts + state check):
  40 X-decode errors → 4 X-decode errors. Remaining 4 are all
  "C vs D, packets diverged" — the real Section 2 signal.
- `transitions --cross-decode`: per-test X-decode count dropped from
  17–40 to 1–4 on failing tests. Passing tests (`BW sweep`, `VBR`,
  `Stereo->ForceMono`) still cleanly report "cross-decode ok" with
  zero additional mismatches.

This should probably be folded back into the HLD as a correction to
Section 2. Noting here so the next HLD iteration can pick it up.

**Findings that applied and were addressed this session:**

- **#5** (cross-decode counter unbounded after divergence) — **already
  fixed mid-session** via `prior_divergence_seen` gate. DA agreed this
  was the right fix. Nice independent validation.
- **#9** (burst 8 "CELT 60ms" silently degrades) — **fixed**. Renamed
  to `celt_framesize_2p5_10_20` using CELT-valid sizes (2.5/10/20 ms)
  and added a companion `silk_framesize_60_40_20` burst (burst index
  13) for the SILK side. 60 ms is invalid for CELT (encoder rejects)
  so my original "force CELT on all three steps" was wrong on step 1.
  DA's suggestion of splitting into CELT-only and SILK-only bursts is
  what I did.
- **#15** (5-line print budget starves burst diagnostics) — **fixed**.
  Added a separate `burst_mismatches_printed` counter with its own
  5-line budget. Computed `in_burst` and `burst_print_budget_ok`
  once per frame, then gated each of the 9 mismatch-print sites on
  `count <= 5 || (in_burst && burst_print_budget_ok)`. Burst failures
  are now much more visible — verified empirically by a 30 s burst
  run showing `burst=hybrid_celt_hybrid step=1:CELT 20ms` prints that
  would previously have been suppressed after 5 random-walk mismatches.
- **#4** (state checkpoint undefined in cross-decode 2x2) — **fixed**
  by documenting: the state checkpoint deliberately only reads the
  column-1 pair (`c_dec_cs` vs `rust_dec_cs`). That is the only
  comparison with a bit-exact invariant (both decoders consumed the
  C-encoded stream from frame 0). The row-axis or rust-stream-column
  comparisons would false-positive on history divergence. Added a
  comment block at the state checkpoint call site.

### 11. Known follow-ups for a future session

1. The pre-existing encoder bugs that the torture suite now catches
   need to be diagnosed and fixed. Not in scope for this HLD (Section
   10 explicitly bars `src/` changes), but they block `torture_test.sh`
   from green.
2. The HLD Section 2 "trivial" claim should be corrected to note the
   history-sensitivity of the strong criterion. A future revision of
   the HLD can explicitly state that the strong check is only valid
   while `prior_divergence_seen == false`.
3. Sections 4 (stereo→mono targeted test) and 5 (PLC during mode
   transitions) remain deferred. The scaffolding (`c_decode_packet`,
   `rust_decode_packet`, `CurrentCfgSource`, burst table) is now in
   place and should make those sections cheaper to implement. DA
   finding #10 suggests a cheap mitigation for the Section 4 gap: a
   latched `saw_stereo_to_mono_transition` assertion that fires on
   `stream_channels_out` transitions observed via
   `debug_get_encoder_hp_state` during burst 5/6.
4. `OPUS_GET_IN_DTX` is still not exposed on the Rust encoder. DA
   finding #2 makes a strong case for fixing this: DTX is exercised
   in ~50% of torture runs (`cfg.dtx = rng%2`), and a drifted
   `noSpeechCounter` would be exactly the kind of latent bug Section
   6 was designed to detect. Adding a 10-line getter to
   `src/opus/encoder.rs` would close this gap. Considered out of
   scope for this HLD ("no src/ changes") but should be revisited
   in the next iteration.
5. DA finding #14 — the simpler alternative of a per-frame
   `debug_get_encoder_hp_state` comparison. The existing shim
   already exposes 5 fields of cross-frame encoder state
   (`hp_mem`, `variable_hp_smth2`, `mode`, `stream_channels`,
   `bandwidth`). A 30-line per-frame comparison might catch
   drift-shaped bugs faster than Section 6's 1000-frame checkpoint,
   without any of the 2x2 decoder bookkeeping. Worth measuring
   before committing to Sections 4/5.
6. `apply_config_to_c_encoder` and `apply_config_to_rust_encoder`
   ignore CTL return values (DA finding #3 sub-issue). A quiet
   failure mode — if any future burst or test sets an invalid
   parameter, the test will "pass" while actually testing nothing.
   Worth adding return-value checks in a future cleanup pass.
