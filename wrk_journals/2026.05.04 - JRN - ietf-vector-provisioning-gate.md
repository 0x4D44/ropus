# Journal: IETF vector provisioning gate HLD

## 2026-05-04

Read the root `AGENTS.md`, `full-test`, the vector fetch scripts, the
conformance README, and the top-level README. Current state is clear:
`full-test` intentionally skips `::testvector` when the IETF vectors are
missing, which is fine for local iteration but not enough for a conformance
gate.

Wrote the HLD for option #1: keep the 71 MB vectors unvendored, use the
existing pinned fetch scripts for provisioning, and make missing-after-fetch a
FAIL-level validation result. I am mildly annoyed by the current sentinel
check in the fetch scripts because it can accept partial state; that is exactly
the kind of small convenience that turns into a false green gate later. The
clean fix is still small: validate the complete expected file set before
skipping download and after staging extraction.

Emotional temperature: focused, slightly wary of overcomplicating `full-test`.
The design deliberately avoids adding env vars or a broad offline mode because
the request is about making the gate honest, not adding more ways around it.

## Implementation

Arthur signed off the HLD, so I implemented the gate. Added an isolated
`full-test/src/ietf_vectors.rs` module for complete-set detection, mockable
fetch execution, and present/provisioned/unavailable status. `full-test` now
attempts provisioning in Stage 0, passes the structured status into Stage 2,
and when vectors remain unavailable it filters only the 24 IETF vector probes
while appending a synthetic `conformance::ietf_vectors::provisioning` failure.
That preserves non-IETF test signal without allowing a green gate.

Hardened both fetch scripts so a single sentinel file no longer counts as
complete. They now validate all 12 `.bit` files plus `.dec` or `m.dec` per
vector before skipping, and validate extracted staging before publishing into
the gitignored vector directory.
