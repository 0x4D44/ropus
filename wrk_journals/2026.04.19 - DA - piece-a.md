## 2. `opus_encode24` implementation — **REAL issue (zero oracle coverage)**

**Concern:** the implementer chose to implement rather than stub (`capi/src/encoder.rs:240-276`), admitting it's "primarily to resolve the linker reference." But:

a. **No differential test.** I grepped `harness/tests/` — the 4 `test_24bit_*` tests are all DECODE. `opus_encode24` has no `test_encode24_*` and no `test_encode24_roundtrip`. 60 LOC of new public ABI with no oracle.

b. **The conversion is mathematically different from the C reference.** C's `INT24TORES` at `reference/celt/arch.h:171` is `SAT16(PSHR32(a, 8))`. `PSHR32(a, 8) = SHR32(a + (1<<7), 8) = (a + 128) >> 8` — in C, this is signed `opus_int32` arithmetic with **UB on overflow**. Rust uses `(s as i64 + 128) >> 8` then clamps — safe, but **will disagree with C on pathological inputs** (`a > i32::MAX - 128`). Reference C overflows to a large negative, clamps to i16::MIN = -32768. Rust computes the correct saturated value and clamps to i16::MAX = 32767. Opposite signs.

c. **The encode path uses the wrong downmix.** C `opus_encode24` (`reference/src/opus_encoder.c:2724-2725`) calls `opus_encode_native` with `downmix_int24` as the downmix callback and passes the ORIGINAL int24 `pcm` as `analysis_pcm`. The Rust wrapper calls `enc.encode(&in_i16, ...)` which dispatches through `encode_native_with_analysis` with `downmix_int` and the *degraded* int16 as analysis. The analysis tonality classifier sees truncated samples. For typical 16-bit-grade test inputs this is identical; for genuine 24-bit content it diverges.

d. **Documented as "round-trips with opus_decode24" — where's the test?** No differential round-trip test exists. The journal's "implemented rather than stubbed so it round-trips correctly if someone later runs opus_demo with -e" is unverified.

**Is this blocking Piece A?** Arguably no — IETF vectors use `-d` only, encode24 is never called. But the HLD says "no new Rust codec code" and this is new public ABI with no test. If we're shipping it, it needs an oracle. Otherwise stub it.

**Verification step:** either (a) revert to a stub that returns `OPUS_UNIMPLEMENTED` (clean; documents what we support), or (b) add a `test_encode24_matches_c_reference_mono/stereo` differential test and a `test_encode24_decode24_roundtrip` invariant. Preferred: stub it. The implementation-as-written is untested and subtly diverges from C.

## 3. DRED stub — **POSSIBILITY worth checking, specifically around the `dred_input > 0` guard**

**Concern:** the stub's safety argument is "IETF vectors contain no packet loss, so `lost_count` stays 0, so `dred_input = 0`, so the `opus_decoder_dred_decode24` call at `opus_demo.c:1149` is guarded." I walked the code path:

- Line 1122: `if (len == 0) lost = 1;` — ZERO-LENGTH PACKETS FORCE LOSS. A .bit file with a zero-length frame (valid per format — it's just `int32 len=0` followed by no payload) triggers `lost_count++`.
- Line 1132: `if (!lost && lost_count > 0)` — on the NEXT successful packet, triggers `opus_dred_parse` (line 1137). Our stub returns 0 → `dred_input = 0`.
- Line 1148: `if (dred_input > 0)` — false, falls through to `opus_decode24(dec, NULL, 0, ...)` PLC. Safe.

**So the stub IS correct on current IETF vectors.** But: this invariant is "the DRED stub is inert so long as `opus_dred_parse` returns 0." That's fine forever on valid vectors. But the moment anyone runs opus_demo against a file where the real DRED parse would return >0 (a stream encoded by a 1.5+ encoder with DRED enabled), the stub silently returns 0 — packet loss concealment falls back to PLC instead of DRED, and the decode is LOWER quality than the reference claims. That wouldn't surface as a crash; it would surface as a failing `opus_compare`.

This is not a Piece A blocker (RFC 8251 vectors don't have DRED payload). But the stub description says "inert at runtime" without this caveat. Worth adding: **"inert only when the stream has no DRED extensions and no packet loss with a DRED-carrying neighbour."**

**Separately:** could a malformed .bit file that gets past `fread` produce a path-corrupted `dred_input > 0`? I don't see one — `dred_parse` is the only writer, it returns 0 in the stub. Clean.

**Linker-flag alternative:** MSVC's `/FORCE:UNRESOLVED` would let unresolved symbols through at link time but produces a warning and is discouraged. Not a real alternative. 122 LOC of stubs is on the heavy side for what's logically ~10 symbol resolutions, but each needs a correct type signature. Could be tightened to ~40 LOC if `OpusDREDDecoder` and `OpusDRED` were typedefs to `void` and every arg was the same type — but the current shape mirrors upstream headers, which is defensible. Leave as-is.

**Verification step:** add a sentence to `dred_stub.c`'s header comment clarifying the "no-DRED-extensions AND no-loss" precondition for inertness. Optional.

## 4. 24-bit decode TDD tests — **POSSIBILITY (coverage gaps)**

The 4 passing tests in `harness/tests/c_ref_differential.rs:1505-1690` cover:
- Mono VOIP packet encoded by Rust, decoded by both Rust and C, compared.
- Stereo VOIP packet, same.
- `decode24 == (decode16 << 8)` invariant.
- MS decode24 stereo.

**Gaps:**
- **No CELT-mode decode24 test.** Both mono/stereo tests use `OPUS_APPLICATION_VOIP` at 48 kHz, 960 samples — likely lands in HYBRID or SILK-only mode. The failing IETF vectors are stereo across multiple bandwidths — quite possibly CELT-mode ones. If the bug is in CELT stereo decode, the TDD tests wouldn't catch it by construction.
- **No DTX / silence test.** The decoder behaves differently on silence (SILK DTX emits a repeat frame). No coverage for `(all-zero input) -> decode24 -> output should also be all-zero`.
- **No PLC-with-decode24 test.** `opus_decode24(dec, NULL, 0, ...)` is called by opus_demo on every lost packet. No test covers the PLC-to-int32 output path specifically.
- **No "entire buffer" assertion.** `assert_eq!(&rust_pcm24[..n_samples], &c_pcm24[..n_samples])` DOES cover all decoded samples, so this one is OK — false alarm from my concern list.
- **No overflow test.** `(i16 as i32) << 8` is always safe; the decoder output is bounded; not a real gap.

**Verification step:** add three targeted tests before Piece A closes: `test_24bit_decode_plc_matches_c`, `test_24bit_decode_celt_mode_stereo`, `test_24bit_decode_dtx_silence_matches_c`. At minimum, the PLC one, since opus_demo hits that path on every zero-length packet.

## 5. File-scope globals in opus_demo.c — **POSSIBILITY, not yet verified**

`--test-threads=1` serialises calls. BUT opus_demo.c:
- `fopen`s `fin` and `fout`, closes them at the `failure:` label (lines towards EOF). Does it `fclose` on SUCCESS paths too? I skimmed — yes, the normal exit path at the end of `main` calls `fclose(fin)` and `fclose(fout)`. OK.
- `malloc`s `data`, `in`, `out`, `fbytes` — all `free`'d at success AND failure paths (checked via grep).
- RNG state `iseed`/`Rw`/`Rz` in `test_opus_common.h` — set per-call if needed.

Test 24 calls `main()` back-to-back. If any malloc'd buffer leaks on any path, the process RSS grows linearly. On Windows with 24 × 2 MB of leaked PCM buffers that's ~50 MB — survivable. But if opus_compare is also leaking (it allocates float buffers for signal-accumulation), we could be pushing 100 MB by test 24. Not a correctness bug, just bloat.

**The real risk:** static `variable_duration` and similar — if one vector mutates a static to a bad value and the next vector expects a fresh default, tests get order-dependent flakiness. I didn't trace all statics. The journal asserts 22/24 pass; re-running in a different test order would reveal whether any flake exists.

**Verification step:** add a test that re-runs the same passing vector 3× in a row and asserts identical output each time, to catch state leakage across `main()` invocations.

## 6. SHA-256 verification timing — **REAL, minor**

Read `tools/fetch_ietf_vectors.sh`:
- Download → `tmp_dir` (mktemp). Size check first, SHA next.
- If SHA fails: `exit 3` BEFORE `mv ... "${dest_dir}/"`. `${dest_dir}` stays empty. Good.
- If extraction fails: script exits; `dest_dir` may have partial contents. `rm -rf "${tmp_dir}"` on EXIT cleans the tmp archive but not the partial extraction in dest.
- Resume check: `if [[ -f "${dest_dir}/testvector12m.dec" ]]; then ... skip.` — so if a prior run failed mid-extraction (e.g., after `mv` started but before completing), re-running sees an incomplete dest_dir BUT the sentinel file `testvector12m.dec` will be missing → it tries to mkdir (no-op, dir exists) → download → extract → `mv ... "${dest_dir}/"`.

**Problem:** the `mv "${extracted_dir}"/* "${dest_dir}/"` will FAIL if `dest_dir` already has files from a prior partial move (mv won't overwrite without `-f`). Script aborts on `set -e`. User has to manually `rm -rf tests/vectors/ietf/` to recover. Not graceful.

**Verification step:** change line 103 from `mv "${extracted_dir}"/* "${dest_dir}/"` to `mv -f "${extracted_dir}"/* "${dest_dir}/"`. One-word fix. Alternatively: check sentinel file existence but also validate a second file exists, and on mismatch, `rm -rf "${dest_dir}"` before re-extracting.

## 8. Panic guards under malformed input — **POSSIBILITY**

`ffi_guard!` wraps in `catch_unwind`. But `catch_unwind` doesn't catch:
- Aborts from `panic=abort` (our release profile uses this — check `Cargo.toml`).
- Stack overflow.
- SIGSEGV / AV on Windows.

The Cargo.toml has `panic = "abort"` (common in ropus). Under `cargo test` the panic strategy for the `test` profile typically reverts to `unwind` — but the IETF vector harness is running in release-mode builds if invoked via the full `cargo test --release`. Check that conformance tests actually run under `panic=unwind`.

**More concrete:** the 24-bit decode wrappers use `.checked_mul` for `n_samples` — good. They use `std::slice::from_raw_parts_mut` — which will happily construct a slice of any length if the caller lied about the pointer. Not guaranteed to panic. If the C caller passes `frame_size=5760` but allocates only 960 samples, we write past the buffer. No bounds check catches this — the slice has the claimed length, and `dec.decode24(...)` writes up to that length. Buffer overflow, not panic.

This is a pre-existing issue with all the capi functions, not specific to 24-bit. But worth flagging: `ffi_guard` doesn't protect against wrong-sized output buffers. IETF vectors won't exercise this, but malformed callers could.

**Verification step:** accept as-is for Piece A (consistent with existing wrappers). File a separate issue: "capi 24-bit wrappers trust caller-provided buffer sizes."
