### Triage — not a bug ("bug L" retracted)

Root cause: **all four panics are the expected-divergence state
introduced by commit `3768181`**, which removed
`#define DISABLE_FLOAT_API 1` from `harness/config.h` as pre-work for
Stage 6 (the analysis.c port). The fuzz build uses the same config.h
via `tests/fuzz/build.rs` `.include(&harness_dir)`, so the fuzz C
reference now runs tonality analysis at complexity 10, Fs in [16k, 48k].
Rust has no analysis module yet, so `AnalysisInfo.valid` is always
`false`, and every input that trips the C analyzer produces different
bytes.

The task brief's "prime suspect" — commit `621105f` MLP port —
is **not** the cause. `ropus/src/opus/mlp.rs` and `mlp_data.rs`
are dead code with zero external call sites; they cannot influence
encoder output. MLP port quality is irrelevant to these panics.

All four crashes trigger analysis (`cx >= 10`, `Fs >= 16000`,
`app != RESTRICTED_SILK`). The analysis output feeds:

- Mode decision (via `voice_ratio` / `detected_bandwidth`) — explains
  the AUDIO crashes where Rust picks CELT-only (TOC 188) while C picks
  Hybrid (TOC 79).
- `CELT_SET_ANALYSIS` → `alloc_trim_analysis` tonality-slope adjustment
  — explains the LOWDELAY and multiframe crashes where TOC matches but
  CELT payload bytes diverge.

Detailed write-up: `notes/campaign9_bug_L.md`.

### Disposition

- **No code fix applied.** There is no bug to fix in this codebase.
  The divergence is the documented, intended Stage 6 red-signal state.
- **No commit, no push.** Per task guardrails: "don't commit speculative
  changes."
- **5 non-affected fuzzers left running** as instructed.
- **Recommendations for future campaigns** (not implemented — needs
  coordination):
  1. Gate the differential assertion on
     `!(cx >= 10 && sr in [16k,48k] && app != RESTRICTED_SILK)` in the
     four affected targets until Stage 6 lands.
  2. Alternatively fork a fuzz-only `tests/fuzz/config.h` that keeps
     `DISABLE_FLOAT_API 1` (preserving Stage 6's harness red-signal
     while un-breaking differential fuzzing).

### Decision — wait for Stage 6 (2026-04-19)

Evaluated the two mitigation options vs. doing nothing:

- **Fork fuzz-only `tests/fuzz/config.h`**: creates a coordination
  burden — someone has to remember to remove the fork the moment Stage 6
  lands, otherwise the fuzz C reference permanently diverges from the
  harness config and we silently miss future analysis-side regressions.
- **Gate assertions**: scatters dead conditionals across four fuzz
  targets, noisy to revert.
- **Wait**: 5 live fuzzers already cover decode, decode_safety,
  encode_safety, repacketizer, packet_parse — most non-analysis paths.
  When Stage 6 lands, the four dormant differential encoders snap
  back to bit-exact coverage of the brand-new analysis port — exactly
  the work you want fuzzing to vet.

Chose **wait**. Reassess when Stage 6's analysis port merges.

### Bug L — analysis subtract overflow

Fix: `285d6c5` (2026-04-19 ~11:05).

Within minutes of the 10:43 relaunch, five targets panicked at the
same source location:

```
thread '<unnamed>' panicked at ropus/src/opus/analysis.rs:1247:45:
attempt to subtract with overflow
```

Tripped by `fuzz_encode`, `fuzz_roundtrip`, `fuzz_encode_safety`,
`fuzz_roundtrip_safety`, and `fuzz_encode_multiframe`. The presence
of `fuzz_encode_safety` (pure-Rust, no differential) confirms this
is a genuine Rust bug, not a Stage-6 analysis-off mismatch.

Root cause: the tonality loop computes
`band_tonality[b - NB_TBANDS + NB_TONAL_SKIP_BANDS]` under the guard
`b >= NB_TBANDS - NB_TONAL_SKIP_BANDS`. In the C reference (signed
`int`) this evaluates to `b - 9`, range 0..=8. The Rust port uses
`usize` and evaluates left-to-right, so `b - NB_TBANDS` underflows
for every `b` admitted by the guard (since `b < NB_TBANDS = 18`).
cargo-fuzz builds with overflow-checks on and caught the underflow
at the subtraction; release builds without overflow-checks would
have wrapped to a huge usize and OOB-panicked on the array access.

Fix: reorder to `b + NB_TONAL_SKIP_BANDS - NB_TBANDS` — bit-exact to
C, non-negative in usize under the existing guard. See
`notes/campaign9_bug_L.md` for full writeup (this note file replaces
the earlier retracted "not-a-bug" triage under the same letter).

Repros: `C:\language\mdopus\fuzz_crashes\crash_{24da1ad627e2a5a7,
44ecced56f288382, 8606db8d3057c654, cde9a65042489ae3,
f727a37351a8ab5a}.bin`. All five exit cleanly on the rebuilt
