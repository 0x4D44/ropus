## Challenge 4 — `opus_projection_*_float` returns OPUS_UNIMPLEMENTED

**Concern.** Agent's stated rationale: "matching the multistream float pattern" (`capi/src/projection.rs:358`). I'm pushing back.

**Verification.**
- `capi/src/ms_encoder.rs:437-446`: `opus_multistream_encode_float` → `OPUS_UNIMPLEMENTED`. Matches the projection encoder float behavior.
- `capi/src/ms_decoder.rs:369-415`: `opus_multistream_decode_float` is **fully implemented** — calls the i16 decode and scales by `1/32768`. Does NOT match the projection decoder float behavior which returns `OPUS_UNIMPLEMENTED`.

So the claim "matching multistream's float pattern" is **false for the decoder**. The projection decoder *could* have used the same i16-then-scale trick with ~8 lines of code.

**Test-exposure.** `test_opus_projection.c` never calls `opus_projection_encode_float` or `opus_projection_decode_float` — compile-time linkage only. So the test passing doesn't validate anything about the float path. This is a silent gap, not a surfaced failure.

**Verdict.** REAL issue. Not a correctness bug (test doesn't exercise it) but the **justification in the code comment and journal is wrong**. Either: (a) actually match ms_decoder's implemented pattern — trivial, ~10 LOC; or (b) change the comment to "float paths not in scope for this conformance test; UNIMPLEMENTED surfaces future callers rather than silently corrupting." **Verification:** Edit `capi/src/projection.rs:358` comment and decide which option.

## Challenge 7 — `os_support.h` drift risk

**Verification.** `diff -q tests/conformance/include/os_support.h reference/celt/os_support.h` returns nothing — vendored VERBATIM. So it isn't a stub, it's a copy. If upstream changes `opus_alloc` semantics (e.g. aligned allocation), the copy silently diverges — same risk as any vendored C file. `mapping_matrix.c` only uses 4-byte alignment (`align()` macro) for `i16` data placement, which any `malloc` trivially satisfies.

**Verdict.** POSSIBILITY at the scale of any vendored file. Same mitigation as elsewhere: `da86a6a` now triggers rebuild on header changes, which surfaces drift at CI time rather than silently. **Verification:** Consider a periodic `diff` check against `reference/celt/os_support.h` in CI (one-liner).

## Challenge 8 — Cherry-picked vs. verbatim headers

**Verification.** Of five vendored headers: three verbatim (`opus_projection.h`, `mapping_matrix.h`, `os_support.h`), two cherry-picked (`arch.h` = 80 LOC vs reference 400; `float_cast.h` = 21 LOC vs 176; `mathops.h` = 21 LOC vs 662). Stubs claim the real guard (`ARCH_H`, `FLOAT_CAST_H`, `MATHOPS_H`) so they dominate any transitive include of the real files.

**Verdict.** REAL maintenance risk. If upstream adds a new macro to `arch.h` (e.g. a new saturation helper) that `mapping_matrix.c` starts using, our stub will silently miss it and the compile will error — that's actually a good failure mode. More subtle: if a macro's *behavior* changes (e.g. `SAT16` becomes saturating-plus-assert under a new debug flag), our stub's old definition wins and we silently run the old semantics. **Verification:** Document the drift expectation in each stub's header comment; consider a CI job that verifies `mapping_matrix.c`'s symbol set is unchanged in upstream.
