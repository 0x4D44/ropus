### 23:58 BST — Phase 1 task #4 (NaN fix) DEFERRED
Investigated `float2int16` NaN handling. The README's "one-line clamp
to -32768" doesn't actually match C semantics:
- C: `MAX32(-32768, MIN32(32767, NaN * CELT_SIG_SCALE))` propagates NaN
  through both comparisons (IEEE 754 returns false for both `<` and `>`
  with NaN), so the clamps don't bite.
- Then `float2int(NaN)` is implementation-defined: `lrintf(NaN)` returns
  0 with FE_INVALID raised, but `_mm_cvttps_epi32(NaN)` returns
  `0x80000000` (INT_MIN); cast to `opus_int16` is 0 either way.
- So the C side likely produces 0 too — not -32768 as the README claimed.
  The fuzz-finding's actual byte sequence must include a value that
  *isn't* exactly NaN, or the divergence comes from elsewhere in the
  encode path under `+/-Inf` inputs.

A correct fix needs a side-by-side trace, not a one-liner. The Stream
A workaround (sanitise NaN/Inf to 0.0 before `encode_float`) preserves
codec coverage. Deferred per "Severity: low" classification.

Phase 1 closes with 3 of 4 tasks landed and 1 deferred-with-rationale.
Net effect for the campaign: P0 OOB fixed, complexity capped, oracle
gate tightened. Ready to launch Phase 2.
