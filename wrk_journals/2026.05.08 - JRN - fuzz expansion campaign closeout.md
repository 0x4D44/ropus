## What was the bug?

`fuzz_encode_mode_transitions` (Stream D's new target) found a
divergence on a configuration where the encoder transitions from
CELT_ONLY to HYBRID across two consecutive frames in a multi-frame
packet. The HYBRID frame fires the CELT->SILK 5 ms redundancy block.

C reference: calls `CELT_SET_PREDICTION(celt_pred)` BEFORE the
redundancy encode (`opus_encoder.c:2288-2295`). This clobbers the
stale `disable_pf=1, force_intra=1` left behind by the previous
frame's prefill `SetPrediction(0)`.

Rust port: was calling it AFTER the redundancy encode (encoder.rs
old lines 3097-3107). The 5 ms redundancy frame therefore ran with
the wrong prediction state, producing 26 different bytes in the
redundancy region of the packet. Main CELT output and frames 3+
re-converge because the redundancy block's `ResetState` plus the
prefill block's `SetPrediction(0)` reset state on both sides.

Fix: move the SetPrediction call up to mirror the C order. Net
diff: +18 / -13 in `ropus/src/opus/encoder.rs`. Crash artifact
preserved at
`tests/fuzz/known_failures/mode-transitions-stereo-hybrid-state-divergence/`.

## Per-stream summary

### Stream A — broaden fuzz_encode_multiframe grammar (commit `57c391e`)

Added 5 new per-frame CTL dimensions: `vbr_constraint` (promoted
from hardcoded 0), `max_bandwidth`, `signal`, `force_channels`,
`prediction_disabled`. Setters fire on both Rust and C in a shared
canonical 11-step order.

`force_channels(2)` on a mono encoder verified clean-reject on both
sides (Rust `encoder.rs:3510-3516`, C `opus_encoder.c:2843-2851` →
`bad_arg:` 3357-3359). `force_ch_idx` left un-clamped.

5 setters added; 3 dropped: `force_mode` → Stream D's scope,
`lsb_depth` and `phase_inversion_disabled` → low signal-per-byte.
Complexity 0..=9 cap preserved (analysis.c divergence dodge).

### Stream B — triage and fix fuzz_decode_safety slow-units (commit `9ed51d4`)

Single root cause across all three slow-unit artifacts: at low API
sample rates, `MAX_FRAME=5760` requested for the FEC `decode_fec=true`
call caused 240–720 ms of phantom prefix-PLC, which the deep-PLC
LPCNet/FARGAN pipeline ran for ~50 ms per chunk. Speedups:

| Artifact | Before | After |
|---|---|---|
| `8728aaa0` (27 B) | 6505 ms | 1052 ms |
| `75d8b2d8` (55 B) | 4647 ms | 753 ms |
| `46a6f8fa` (114 B) | 3415 ms | 778 ms |

10-min synchronous re-fuzz post-fix: 8045 runs, 0 new artifacts.
Audit confirmed only `fuzz_decode_safety` had the
`MAX_FRAME + decode_fec=true` shape; no fan-out required.

This is a fuzz-harness fix, not a codec change. Coverage delta on
the prefix-PLC + FEC composition path acknowledged in the triage
doc (`wrk_docs/2026.05.08 - TRIAGE - fuzz_decode_safety slow-units.md`).

### Stream C — stereo-biased multiframe fuzz target (commit `bb2693d`)

New target `fuzz_encode_multiframe_stereo`. Pins `channels=2`,
narrows sample rates to {16, 24, 48} kHz, biases application toward
AUDIO (4:3:1), bitrate toward the 16-32 kbps ms_pred decision zone.
Per-frame `force_channels` / `max_bandwidth` / `signal` with an
INHERIT slot for cross-frame coercion semantics. Sentinel values
(`i32::MIN`) sit unambiguously outside every legal CTL value range
(-1000..=3002).

Sibling helper `apply_c_encoder_config_with_inherit` and wrapper
`c_encode_multiframe_inherit` reuse Stream A's widened
`CEncodeConfig` rather than forking a parallel struct.

Phase 1 commit gate (build + smoke) green at 820 exec/sec.
Phase 2 coverage-diff (HLD's `compute_stereo_width` and
`force_channels != OPUS_AUTO` branch coverage) explicitly **deferred**
to a follow-up — coverage-diff infra needs a separate llvm-cov build.

### Stream D — mode-transition fuzz target (commit `ba64896`)

New target `fuzz_encode_mode_transitions`. Boundary-biased grammar:
70 % of bitrates within ±6 kbps of the per-config mode-decision
threshold centre (mono voice 60k / stereo voice 44k / mono+stereo
music 14k, with VOIP +8k boost), 20 % in the WB↔SWB SILK/HYBRID
seam (9-15 kbps), 10 % uniform safety net. Per-frame
`set_max_bandwidth`, `set_signal`, and `set_force_mode`. Frame
duration restricted to ≥10 ms (shorter forces CELT_ONLY).
Application held constant per sequence (legal-CTL constraint —
`encoder.rs:3849-3851`).

`MODE_SILK_ONLY` / `_HYBRID` / `_CELT_ONLY` were already `pub` in
`ropus::opus::decoder` — fallback hardcode policy from HLD wasn't
needed.

Boundary-biased smoke validation (one-time, env-var-guarded):
58.29 % of sequences had a mode flip (HLD bar 30 %), 25.74 % had
SILK↔HYBRID transitions (HLD bar 5 %), all three modes observed.

The new target found the bug Stream E fixed in ~55 seconds /
~2933 runs. **The campaign's primary deliverable.**

### Stream E — fix CELT prediction sequencing (commit `9d9d9cc`)

Diagnosis (`wrk_journals/2026.05.08 - JRN - mode-transition crash diagnosis.md`)
HIGH confidence with empirical patch verification — gated test
fix made the crash artifact replay clean; reverting reproduced.
Packet-layout math conclusive: 1 (TOC) + 41 (main CELT) + 26
(redundancy) = 68 bytes; divergence at byte 42 = exact start of
redundancy region.

Surgical fix: moved the `if mode != MODE_SILK_ONLY {
SetPrediction(celt_pred) }` block from after the redundancy frame
(old encoder.rs:3097-3107) to before it (new encoder.rs:2891),
mirroring `opus_encoder.c:2288-2295` exactly. The orphaned
post-redundancy block was deleted (clean C mirror; the prefill
block at encoder.rs:3133-3143 still does its own `ResetState +
SetPrediction(0)` for the main CELT encode).

Verification:
- Crash artifact replays clean (59 ms).
- 2026-05-07 known-failure replays clean (24 / 33 ms — no regression).
- 1868/1868 lib tests, 17/17 param-grid.
- Code-reviewer LGTM, no required changes.

## Open items for Arthur

These are surface-area follow-ups, not blockers. None were triaged
overnight per the autonomous-operation guardrails.

1. **Stream A acceptance gate is on the floor.** 300 NEW units in
   the 15-min validation window — exactly meets the HLD's absolute
   floor. The HLD also allowed a "≥2× rate-ratio vs same-machine
   pre-broadening baseline" alternative; the rate-ratio was not
   measured (the agent shortcut). Saturation visible — ft growth
   has flattened. Recommendation: optional 1-2 h extended run to
   confirm, or just accept the 300 floor as the formal pass. The
   broadening did expose 729 REDUCE events (corpus shrinking),
   which is healthy.

2. **Stream D's mode-flip / SILK↔HYBRID HLD floors are unreachable
   on the current corpus.** Post-fix probe data:
   - 17.47 % sequences with a mode flip (HLD ≥30 %)
   - **0 SILK↔HYBRID transitions** (HLD ≥5 %)
   - Cause: 4853/4941 frames at sr=8 000 Hz, where HYBRID is
     invalid by spec (HYBRID requires ≥16 kHz). All 221 actual
     `set_force_mode(HYBRID)` requests landed at 8 kHz and were
     silently coerced to SILK_ONLY.

   Note this is **not a codec defect** — the encoder is correctly
   following HYBRID admissibility rules. It's a corpus-bias issue:
   libFuzzer's coverage-guided corpus selection has converged on
   8 kHz seeds during the run because they reach SILK-only edges
   that higher rates don't.

   The original Stream D smoke validation (synthetic, pre-corpus,
   2000 runs) hit 58.29 % flips and 25.74 % SILK↔HYBRID — the
   *grammar* does what it's supposed to; libFuzzer's corpus drift
   is the issue in steady state.

   Possible follow-ups (HLD-class change, not overnight scope):
   - Bias the Arbitrary impl to skip 8 kHz when forced_mode is
     HYBRID, OR favour higher rates when HYBRID is interesting.
   - Pre-seed the corpus with hand-curated 24/48 kHz inputs.

   Either way: the target's value is not in question — it found
   the actual bug we cared about within minutes of cold start.

3. **Stream C Phase 2 coverage-diff is pending.** HLD requires a
   campaign-end blocking gate: stereo target's coverage on
   `compute_stereo_width` (encoder.rs:1133) and the `force_channels
   != OPUS_AUTO` branch (encoder.rs:1816) must be strictly greater
   than the parent target's. Coverage-diff infra requires a
   separate `cargo +nightly fuzz coverage` build + llvm-cov report
   pass. Deferred per the kickoff plan.

4. **Stream G's first invocation backgrounded its work and exited
   without reporting.** Stream G2 picked up the batons cleanly, so
   no campaign data was lost — but this is the second time today
   an agent has done this (Stream B did the same earlier). Process
   note for future campaigns: brief agents explicitly that
   `run_in_background=true` on Bash within an Agent does not give
   the agent a way to receive the background task's notification —
   they have to run synchronously or accept incomplete reports.

## Lessons / observations

1. **Targeted fuzzing dominates broadening.** Stream D's
   boundary-biased target found a bug in ~55 seconds. Stream A's
   broadening (which would also have eventually exposed this same
   defect surface in some form) reached only the HLD's floor in
   the same wall-time. When you have a strong hypothesis about
   where bugs live, building a target that biases input-space
   *toward* that hypothesis is far cheaper than waiting for a
   uniform target to randomly land there.

2. **Same defect class, third time.** Each of the 2026-05-07 PLC
   delay_buffer bug, the 2026-05-07 SILK LBRR ec_prev bug, and
   today's CELT SetPrediction bug followed the same pattern: C
   mutates state in-place inside a sequenced helper; the Rust port
   reorders the call site or misses one of the calls; a specific
   frame-pair adjacency exposes the divergence. **There may be
   more.** Worth a deliberate audit of every per-frame setter
   sequence in `ropus/src/opus/encoder.rs` against
   `reference/src/opus_encoder.c` ordering. Out of overnight scope.

3. **Empirical patch verification beats theoretical diagnosis.**
   Stream E's diagnosis was already strong (packet-layout math +
   line-pair citations), but applying a guarded test-fix and
   confirming the artifact replays clean turned a HIGH-confidence
   diagnosis into an unambiguous one before any commit landed.

4. **HLD-first multi-agent flow worked well**, with one caveat:
   agents that background a long-running validation step and exit
   need explicit briefing not to do that. Both Stream B and Stream
   G hit this. Easy fix for next campaign.
