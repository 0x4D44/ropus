### Key Insight: SILK Codec Has Systematic Numerical Divergence

The SILK encoder and decoder diverge from C at ALL tested sample rates (8/12/16/48 kHz) when processing random/garbage data. The divergences range from small accumulating drift (decoder) to complete output divergence (encoder).

However, **all 210 real encode-decode sweep configurations remain bit-exact**. The divergences only occur with inputs that no real audio stream would produce.

**Root cause hypothesis:** The Rust SILK implementation has subtle numerical differences in fixed-point arithmetic that are masked by the statistical properties of real audio but revealed by random/garbage inputs. Likely candidates:
1. LPC coefficient computation — rounding/saturation differences
2. Gain decoding — different handling of extreme values
3. Range coder state — accumulated bit-level differences
4. Division/shift operations — Rust vs C integer division rounding

### Bug #12: silk_lpc_fit conditional write-back — FIXED (resolves Bugs #2/7/9)

**File:** `src/silk/common.rs:577` (silk_lpc_fit)
**Root cause:** Rust unconditionally wrote back rounded/saturated values to `a_q_from` after the BWE loop. C has two paths:
- `i == 10` (max iterations): SAT16 + write-back to `a_QIN`
- `i < 10` (early convergence): plain `RSHIFT_ROUND`, **NO write-back**, **NO SAT16**

The unconditional write-back clobbered QA+1 precision in `a32_qa1`, causing the `silk_nlsf2a` stability loop to diverge from C when processing garbage NLSFs. Different LPC coefficients cascaded into wildly different PCM output.

**Fix:** Added `reached_max_iter` flag to track whether BWE loop exhausted all 10 iterations. Only write back to `a_q_from` (with SAT16) when max iterations reached.

**Result:** 4.2M differential decode iterations on full corpus with ZERO assertion failures. Bugs #2, #7, #9 are resolved.

### Bug #8: SILK NSQ del_dec encode divergence (documented, not yet fixed)

**Root cause (identified by investigation agent):** The SILK delayed-decision quantizer (`silk_nsq_del_dec`) produces different pulse outputs than C, leading to different encoded byte counts. This causes the CBR padding path to differ (code 3 vs code 0). The Opus-level padding logic is correct.

**Next steps:** Fix must be in `silk_nsq_del_dec` (src/silk/encoder.rs:4221). Check delayed-decision state management, winner selection tie-breaking, and partial memcpy emulation.
