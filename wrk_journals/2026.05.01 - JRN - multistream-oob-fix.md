# 2026-05-01 — Multistream sub-packet OOB fix

## Goal
Fix the P0 DoS reported by Stream C of the fuzz-coverage-expansion campaign:

- Site: `ropus/src/opus/multistream.rs:624`
- Symptom: 20-byte attacker-controlled multistream packet panics
  `OpusMSDecoder::decode` with
  `range end index 1278 out of range for slice of length 20`.
- Root cause hypothesis: `parse_multistream_subpacket` returns the
  self-delimited length encoded in the TOC without bounds-checking it
  against the remaining buffer; `multistream_packet_validate` then slices
  with that length and panics.
- Fix shape (per finding README): bounds-check inside
  `parse_multistream_subpacket`, return `OPUS_INVALID_PACKET` if the
  encoded length exceeds the remaining buffer. Mirrors the C reference
  which returns `OPUS_INVALID_PACKET` on the same input.

Also: add a unit test that drives the panic path with the existing
known-failure crash file and asserts a graceful `OPUS_INVALID_PACKET`
instead of a panic.

## Plan

### 14:15 — Context analysis (Task #1 ✓)
Read `parse_multistream_subpacket` in detail. The bug is clear:

- Line 743: `pos as i32 + remaining` where `remaining` gets reassigned from TOC-encoded size
- Lines 722-736: self-delimited path sets `remaining = sz` without bounds-checking `sz` against actual buffer
- `multistream_packet_validate:624` then slices `&sub_data[..packet_offset as usize]` with unchecked length

The fix belongs in `parse_multistream_subpacket` — add bounds check before reassigning `remaining`.

### 14:20 — Panic reproduced (Task #2 ✓)
Confirmed: `range end index 1278 out of range for slice of length 20` at exactly multistream.rs:624.

### 14:25 — First fix attempt (Task #3)
Added bounds check in `parse_multistream_subpacket` after line 741, but this broke an existing test case `[0x03, 0x02, 0x01]` which expects to return 4 for a 3-byte input. The issue is semantic: in code=3 VBR, `remaining` represents leftover bytes after parsing, not total payload size.

### 14:30 — Refined fix
Moved the bounds check to `multistream_packet_validate:624` right before the slice operation. This catches the OOB at the exact point where it would panic, without breaking the legitimate case where `parse_multistream_subpacket` returns a theoretical packet size larger than the input buffer.
