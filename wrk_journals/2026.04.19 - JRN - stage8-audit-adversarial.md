# 2026-04-19 — Stage 8 devil's-advocate audit

**Role.** Adversarial auditor. The sibling row-by-row audit is assumed cooperative; this report targets what both implementers and the row-by-row auditor are most likely to overlook.

**Scope.** 9 commits `fe8c97e` → `c0a92f3` (stages 8.1 through 8.9).

**Method.** Read-only. No code changes. Test runs are observational only.

---

## Summary

The Stage 8 port is substantively delivered against the HLD. The tier-1 claims for 8.1, 8.4, 8.5, 8.7 hold up to inspection — the tests genuinely assert `f32::to_bits()` equality, not some softer SNR-tolerance sleight of hand. The biggest concerns are (a) the 8.8 integrated test has a markedly weaker oracle than the sub-stage tests before it, letting real drift slip past; (b) a latent stereo-DRED bug guarded only by `debug_assert!`; (c) a stale-state bug in `set_dred_duration` when called multiple times; and (d) the CTL dispatch seam is wider than acknowledged — the entire Stage 8.8 wiring is unreachable through the C ABI today.

## Concerns

### 1. Stage 8.8 integrated test oracle is structurally weaker than Stages 8.4–8.7

- **Concern.** `dred_integrated_encode.rs` (3 tests) only asserts that *at least one* frame produces a packet parseable by the other side's parser (`assert!(found)`). It never compares Rust-encoded bytes to C-encoded bytes on matching frames, never compares `nb_latents` between C and Rust on matching frames, and tolerates any number of silently-different packets as long as one parses.
- **Evidence.** `C:\language\mdopus\harness-deep-plc\tests\dred_integrated_encode.rs:236-241` (`rust_encoded_packets_parse_on_c_reference`) asserts only `assert!(found, ...)`. Compare to the 8.7 test `C:\language\mdopus\harness-deep-plc\tests\dred_decode_payload_diff.rs:152-157` which asserts `c_bytes == r_bytes`.
- **Material impact.** Medium. Given the documented upstream LPCNet drift, Rust-encoded packets and C-encoded packets on live WAV input will necessarily differ. The test is weakened to hide that drift behind a "parseable by the other side" bar, but that bar is satisfied by any payload with a valid range-coded prefix — it tells us almost nothing about encoder correctness beyond "the emission wiring isn't totally broken." A silent regression that worsens drift would not be caught.
- **What resolves it.** When 8.6 LPCNet drift is fixed, strengthen the 8.8 test to assert byte-equality on at least one matching frame, or at minimum assert `c_nb_latents == rust_nb_latents` per frame across the whole sequence. Until then, explicitly document in 8.8's journal that the tests are format-level plausibility checks, not correctness checks.

### 2. Stereo-DRED silently runs mono pointer arithmetic in release builds

- **Concern.** `DREDEnc::compute_latents` guards channel count with `debug_assert!(self.channels == 1, "TODO(stage-8.x): verify DRED stereo against C ref")`, compiled away in release builds. `OpusEncoder::set_dred_duration` does **not** reject stereo configurations; it accepts any channel count and lazily constructs `DREDEnc::new(self.fs, self.channels)`. A stereo encode with DRED enabled would (a) panic on debug, (b) silently emit incorrect DRED payload bytes on release.
- **Evidence.** `C:\language\mdopus\ropus\src\dnn\dred.rs:1982-1985` (debug-only assert); `C:\language\mdopus\ropus\src\opus\encoder.rs:3648-3671` (no channel gate in `set_dred_duration`).
- **Material impact.** Medium. Release-build user who constructs a stereo encoder and calls `set_dred_duration(100)` will produce garbage DRED payloads without any runtime error. Not caught by any current test — all DRED tests drive 1-channel paths.
- **What resolves it.** Either promote the `debug_assert!` to a runtime `return OPUS_BAD_ARG` in `set_dred_duration` when `self.channels != 1`, or port the channel-stride-aware path from C and enable it. The current state creates a user-visible failure mode with no error signal.

### 3. `dred_target_chunks` becomes stale on repeated `set_dred_duration` calls

- **Concern.** `set_dred_duration` sets `dred_target_chunks = (duration + 5) / 4` **only** inside the `if duration != 0 && self.dred_encoder.is_none()` allocation branch. Subsequent `set_dred_duration` calls update `self.dred_duration` but leave `dred_target_chunks` at its first-ever value. The emission branch at `encoder.rs:3331-3333` uses this stale value to cap chunk count in VBR mode: `if self.use_vbr != 0 { dred_chunks = imin(dred_chunks, self.dred_target_chunks); }`.
- **Evidence.** `C:\language\mdopus\ropus\src\opus\encoder.rs:3656-3669`. Compare `dred_duration` update at line 3652 (unconditional) vs `dred_target_chunks` at line 3667 (conditional on `is_none()`).
- **Material impact.** Low-to-medium. Affects VBR-mode DRED consumers who change duration after first enabling. The effective chunk count silently clamps to the old value. None of the integrated tests exercise re-configuration.
- **What resolves it.** Move the `dred_q0 / dred_d_q / dred_qmax / dred_target_chunks` assignments out of the `is_none()` block. These are independent of the `DREDEnc` allocation.

### 4. No CTL dispatch wiring in capi — 8.8 is unreachable through the C ABI

- **Concern.** `OPUS_SET_DRED_DURATION_REQUEST` / `OPUS_GET_DRED_DURATION_REQUEST` are not routed in `capi/src/ctl.rs` nor `capi/src/ctl_shim.c`. Only `OPUS_SET_DNN_BLOB_REQUEST` (4052) is defined. A downstream C application linking against the libopus ABI has no way to enable DRED encoding.
- **Evidence.** `C:\language\mdopus\capi\src\ctl.rs` (grep `dred_duration|DRED_DURATION` → no matches). The journal acknowledges this as "CTL-dispatch not wired" in the Known gaps section of 8.8, but frames it as a Stage 7 responsibility.
- **Material impact.** Low-to-medium. The HLD's Stage 7 coordination section does hand off CTL routing, but the consequence is the entire 8.8 wiring + 8.9 stub path is exercised only by Rust-native callers. No part of Stage 8's encoder integration is reachable from the only ABI ropus is actually supposed to ship behind. If Stage 7 never finishes the CTL routing, Stage 8 is dead code to any real consumer.
- **What resolves it.** Either explicitly schedule the CTL wiring under Stage 8.x (rather than Stage 7), or surface this in the Stage 8.10 close-out as a blocker for release rather than a completion detail.

### 5. `embedded_blob_initialises_rdovae_enc_and_dec` only confirms structural loadability, not byte content

- **Concern.** The 8.3 "exit criterion" test parses the embedded blob, calls `init_rdovaeenc` / `init_rdovaedec`, and asserts they return Ok. It does not verify that any specific layer's bytes match what `dred_rdovae_enc_data.c` encodes — it's load-without-panic only. If `gen_weights_blob.c` mis-serialized a layer's size or type, *and* Rust's `linear_init` lookup happened to accept the mis-serialized variant (e.g., because the expected `nb_inputs × nb_outputs` still multiplied to the same total block size), this test would pass.
- **Evidence.** `C:\language\mdopus\ropus\src\dnn\dred.rs:2523-2546`. The only checks are `parse_weights(..).expect()` + `!arrays.is_empty()` + both `init_*` calls returning Ok.
- **Material impact.** Low. The 8.4 and 8.5 differential tests *do* effectively exercise the blob content by running the RDOVAE forward pass bit-for-bit against the C reference. So a corrupted blob would cause those tests to fail. But this leaves an edge case: a correlated error in `gen_weights_blob.c` emission *and* in Rust's `linear_init` ordering could let 8.4/8.5 pass on the shared shuffle. The 8.3 journal claims 8.3 is a "skeleton ready for 8.4" — in practice 8.3's own test is not a strong independent oracle.
- **What resolves it.** Optional — add a targeted byte-content test that checks one known weight's first 64 bytes against the corresponding `.c` source. Given the 8.4/8.5 bit-exact forward-pass tests, this is belt-and-braces rather than strictly necessary.

### 6. `#[ignore]`d 8.6 WAV payload test has a clear un-ignore path but uses a brittle criterion

- **Concern.** `dred_encode_silk_frame_bytes_match_c_reference` carries `#[ignore = "Stage 7 LPCNet drift..."]` pointing at upstream Ex[]/FFT bit-exactness as the gate. The un-ignore criterion is "when `features[0..20]` match C bit-for-bit on a reference WAV frame." This is correct as a necessary condition but doesn't mean DRED will actually pass — it's possible LPCNet could be made bit-exact and then 8.6 still fails because of some secondary drift in the DRED pipeline that the layered diagnostics happen not to catch.
- **Evidence.** `C:\language\mdopus\harness-deep-plc\tests\dred_encode_payload_diff.rs:154`.
- **Material impact.** Low. The test body includes inline drift-isolation diagnostics (`resample_mem`, `input_buffer`, LPCNet `features`, `latents/state`, payload), so a secondary drift would surface clearly when the test is un-ignored. But the journal treats LPCNet fix as if it guarantees 8.6 will flip green; that's an assumption, not a proof.
- **What resolves it.** When un-ignoring, do so incrementally: first assert LPCNet features match; then assert RDOVAE outputs match; only finally assert payload bytes match. Don't flip the whole gate at once.

### 7. 8.6 LPCNet edits are narrow and localised but untested on the audio reconstruction path

- **Concern.** The three LPCNet edits in 8.6 change the numerical precision of `dct()`, `compute_frame_features` cepstrum, and `celt_log10_f32`. These are *upstream* of both the DRED feature path and the PLC concealment path. The existing 63 `dnn::lpcnet::tests::` pass, and `tier2_snr.rs` still clears 50 dB, but neither is a per-sample-bit-exact oracle against the C reference on LPCNet output — they're internal regression guards.
- **Evidence.** `git diff 9b09bcd~1..9b09bcd -- ropus/src/dnn/lpcnet.rs` (+34/-7 lines). Three edits: f32→f64 multiply in `dct`/`idct`, cepstrum log10 emulation, `celt_log10_f32` emulation.
- **Material impact.** Low. The edits are strictly more precise (match C's unconditional double-precision intermediate), so in the worst case they slightly improve bit-exactness against C rather than hurt it. But they narrow a known drift without closing it, meaning they changed the numerical output of a cross-cutting DNN primitive in service of one caller (DRED) while leaving the other caller (PLC) unverified at the same precision level. If LPCNet PLC output byte-exactness was ever a forward goal, 8.6 moved that goalpost without consulting Stage 7.
- **What resolves it.** When Stage 7b.3/7b.4 does a bit-exact LPCNet feature comparison, re-verify these three edits against the same ground truth. Currently they're "probably correct" by inspection, not by test.

### 8. TODO density in Stage 8 code: 4 live markers, all deferred

- **Concern.** Stage 8 leaves 4 `TODO(stage-8.x)` markers in production code: channel stride (`dred.rs:1977-1985`), CELT-budget steal (`encoder.rs:3051-3055`), DTX first-frame shift (`encoder.rs:2350-2353`), and hardcoded quantiser params (implicit at `encoder.rs:3664-3669` — no inline TODO but documented in journal). One `TODO(stage-8.3)` is still live in `dred_stats.rs:10` on `#![allow(dead_code)]`. Plus pre-existing `TODO(stage-8)` markers in `opus/decoder.rs:666,673`. These aren't individually concerning, but the journal's "8.8 known gaps" section lists 4 items that collectively mean the Rust DRED encoder does not match C behaviour on low bitrates, DTX+multi-frame packets, or rate-adaptive quantisation.
- **Evidence.** Grep `TODO\(stage-8` across the tree → 5 matches in source, all real. Journal §"Known gaps / TODOs" at line 1131-1135 of execution journal enumerates the four encoder-side gaps.
- **Material impact.** Medium. The hardcoded `q0=9, d_q=3, qmax=15` in particular is a substitute for C's `compute_dred_bitrate` — a significant missing helper. The integrated test runs at 32 kbps; at <= 20 kbps or on bitrate-adaptive scenarios, behaviour diverges from C in ways no current test catches. The journal is honest about this, but the HLD's staging table exit criterion ("packets C can parse") is satisfied while the underlying behaviour is not C-equivalent.
- **What resolves it.** Schedule a follow-up sub-stage (call it 8.11 or Stage 8.x cleanup) specifically for the four gaps, with an acceptance oracle at each of 20 kbps, DTX mode, and per-frame q0 variation. Or explicitly classify them as "won't fix in DRED-experimental phase" and document which production call-sites are affected.

## What I did not find

- No out-of-scope files edited in any of the 9 Stage 8 commits. The only "non-HLD" touch is `ropus/src/dnn/lpcnet.rs` at 8.6, and that's documented as a collateral bit-exactness fix with explicit rationale.

- No silent tier-1 → tier-2 downgrades. 8.4, 8.5, 8.7 genuinely achieve bit-exact; 8.6 honestly escalates and uses `#[ignore]` rather than weakening the oracle.
- No hidden `#[ignore]` graveyards. The only `#[ignore]` in Stage 8 files is the 8.6 WAV test, with a documented un-ignore criterion and layered diagnostics preserved.
- Weight blob integrity appears intact: both C-side DRED tests and Rust-side `init_rdovae*` paths trace back to the same `reference/dnn/dred_rdovae_{enc,dec}_data.c` source, via `gen_weights_blob.c` for Rust and direct linkage for C.
- API ergonomics for `OpusDREDDecoder` are reasonable: `new` / `load_model` / `parse(&self, &[u8], ..) -> Result<OpusDred, i32>` / `process(&self, &mut OpusDred)` compose cleanly, and the `usable_samples(sr)` helper restores the C return value. A downstream consumer can express the full flow in safe Rust.

## Cross-cutting observation

The nine Stage 8 sub-commits are unusually clean in structure — each has a scope-matched commit message, a journal entry with test output, and a commit size proportional to the HLD's staging row. The concerns above are about the *breadth* of what's been validated (integrated test weakness, stereo gap, CTL-dispatch gap), not about the *depth* of what's claimed within each sub-stage. Stage 8 is a well-executed port with a few frontier-boundary leaks that the row-by-row audit is unlikely to surface on its own.
