## Bug Found: CELT Code-3 PLC/DTX Decode Divergence

**Known since:** Campaign 5 (2026-04-11) — deferred as "requires dynamic tracing"

**Trigger:** Code-3 (multi-frame) CELT packet containing a size-0 or size-1
sub-frame (PLC/DTX within the packet). The sub-frame after the PLC diverges
from C reference.

**Root cause:** Three interrelated bugs:

### Bug A: Swapped window coefficients in prefilter_and_fold()
**File:** `src/celt/decoder.rs:626-628`  
**Issue:** `w_rise` and `w_fall` were applied to the wrong TDAC fold operands:
```rust
// WRONG: w_fall * etmp_reversed + w_rise * etmp_forward
// RIGHT: w_rise * etmp_reversed + w_fall * etmp_forward
```
**Impact:** Corrupted overlap buffer after PLC → all subsequent samples wrong.

### Bug B: Missing conditional skip_plc reset
**File:** `src/celt/decoder.rs:1549-1551`  
**Issue:** C has `if (loss_duration == 0) skip_plc = 0;` at start of normal
decode. Rust was missing this entirely.

### Bug C: Erroneous unconditional skip_plc = 0 at end of decode
**File:** `src/celt/decoder.rs:~2048`  
**Issue:** Rust cleared `skip_plc` at end of every normal decode. C only clears
it conditionally at the START of the next decode (when loss_duration == 0).
Wrong state machine: after PLC, C keeps skip_plc=1 through the first recovery
frame; Rust was clearing it immediately.

### Bug D (opus layer): Empty sub-frame passed as Some(&[]) instead of None
**File:** `src/opus/decoder.rs:1231-1238`  
**Issue:** Multi-frame loop passed `Some(frame_data)` for all sub-frames including
size-0 and size-1 frames. C passes `data=NULL` for these, triggering PLC directly.

**Verification:** All 4 crash files now produce "PCM exactly equal" via
diff_fuzz_decode. All 1315+ tests pass.

**Secondary issue noted:** `decode_with_ec()` sets `self.error = 1` on bitstream
overread while C returns `OPUS_INTERNAL_ERROR`. Not yet fixed.

## Finding #3: CELT Extended PLC State Drift (post-fix, NEW)

**Time:** 10:56 (captured by panic hook in fuzz_decode_fixed R2)  
**Crash file:** `fuzz_crashes/crash_6f23c804c2e8d388.bin` (196B)  
**Trigger:** CELT NB 2.5ms, mono, code-3 with 29 sub-frames, heavy PLC.  
**Severity:** 1-sample off-by-1 (Rust=-1, C=0) at sample index 509/580.

### Packet anatomy
29 sub-frames, mostly DTX/PLC (size-0) with sparse real data:
```
[0-1] PLC → [2] 39B → [3] PLC → [4] 63B → [5-16] 12×PLC
→ [17] 7B → [18-24] 7×PLC → [25] 9B → [26-27] PLC → [28] 45B
```

Divergence occurs at sub-frame [25] (9B real data) after 7 consecutive PLC frames,
which themselves follow 12 consecutive PLC frames. The extended PLC sequence causes
a state drift that produces a 1-sample rounding difference in the recovery frame.

### Root cause (found by main context, 11:15)

**File:** `src/celt/decoder.rs:600` (`prefilter_and_fold` inline comb filter)

**Issue:** The inline comb filter guard condition was `neg_g0 != 0 || neg_g1 != 0`.
When `postfilter_gain_old != 0` but `postfilter_gain == 0` (old filter active, new
filter inactive), the code entered the filter block even though all NEW filter
coefficients were zero. This applied the fixed-point bias `-1` to every overlap
sample (120 samples at internal rate) without applying any actual filtering.

C's `comb_filter()` with `window=NULL, overlap=0` has an early return at
`if (g1==0) { OPUS_MOVE(y, x, N); return; }` — no filter, no bias. Our code
was missing this early-out.

**How it triggers:** Frame [4] (63B) signals a postfilter → gain non-zero. Frame [17]
(7B, too small for postfilter) → gain=0. After state update: `gain_old != 0`,
`gain == 0`. PLC frames [19]-[24] each call prefilter_and_fold with this mismatch,
accumulating the erroneous -1 bias 6 times through the TDAC fold into decode_mem.
When frame [25] decodes, the accumulated bias crosses a rounding boundary in one
sample after deemphasis/downsampling.

**Fix:** Changed guard from `neg_g0 != 0 || neg_g1 != 0` to `neg_g1 != 0`.
Removed unused `_old` parameters. This matches C's comb_filter behavior: when the
NEW gain is zero, just copy without filtering or bias.

**Verification:** 
- Crash file `crash_6f23c804c2e8d388.bin` now produces "PCM exactly equal"
- All 4 previous crash files still pass
- All 1317 tests pass

**Status:** **FIXED**

## Finding #4: Encoder Mode Selection Divergence (fuzz_roundtrip R3, NEW)

**Time:** 11:36 (captured by panic hook in fuzz_roundtrip_r3)  
**Crash file:** `fuzz_crashes/crash_be6cdf45c14bcb42.bin` (3155B)  
**Reproduces:** YES — confirmed with current (fully-patched) code.

**Parameters:** sr=24000, ch=2, app=VOIP(2048), br=71279, cx=3, VBR off.  
**Input:** Mostly-silent PCM with a few non-zero samples at start.

**Divergence:** Both produce 178 bytes but completely different content:
- **Rust:** TOC=0x6C → Hybrid SWB 10ms, stereo
- **C:** TOC=0xDC → CELT-only SWB 20ms, stereo

This is a dramatic mode selection divergence — not just different samples, but
different codec modes AND frame sizes. The encoder's mode/bandwidth/frame-size
decision pipeline diverges between Rust and C at this parameter boundary.

**Root cause (confirmed 2026-04-14 ~14:30):**  
The Rust `mode_voice` / `mode_music` threshold computation used an affine form:
```rust
let mode_voice = MODE_THRESHOLDS[0][0]
    + (stereo_width as i64 * (MODE_THRESHOLDS[1][0] - MODE_THRESHOLDS[0][0]) as i64
        / Q15ONE as i64) as i32;
let mode_music = MODE_THRESHOLDS[1][1]; // constant, no interpolation
```

C uses a Q15 lerp with per-term truncation:
```c
mode_voice = MULT16_32_Q15(Q15ONE-stereo_width, mode_thresholds[0][0])
           + MULT16_32_Q15(stereo_width, mode_thresholds[1][0]);
mode_music = MULT16_32_Q15(Q15ONE-stereo_width, mode_thresholds[1][1])
           + MULT16_32_Q15(stereo_width, mode_thresholds[1][1]);
```

For `stereo_width=0`: C `mode_voice=63998` vs Rust `64000` (diff=2); C `mode_music=9999` vs Rust `10000` (diff=1). The MULT16_32_Q15 truncation (`(a*b)>>15` loses 1-2 LSBs) propagates through the voice_est² interpolation, shifting the final threshold by ~2 units. With non-zero stereo_width from real audio data, the divergence grows further.

**Fix:** Changed to `mult16_32_q15(Q15ONE - stereo_width, A) + mult16_32_q15(stereo_width, B)` matching C exactly.  
**Verification:** Crash file matches, all 1317 tests pass, full bitrate sweep (5k–99k, 1k steps) clean.

**Status:** FIXED ✓

  Bug E's crash file was captured by the hook; libFuzzer's native crash writer would
  have been bypassed by __fastfail on Windows.
- **Secondary issue noted (not fixed):** `decode_with_ec()` sets `self.error = 1` on
  bitstream overread while C returns `OPUS_INTERNAL_ERROR`. Low priority.
- **Potential issue:** `celt_fir()` in `src/celt/lpc.rs` uses `sround16()` which
  clips to [-32767, 32767]. The C SSE4.1 path (`_mm_packs_epi32`) clips to
  [-32768, 32767]. This 1-LSB asymmetry at the negative extreme could cause rare
  divergences on inputs where the FIR output hits exactly -32768. Needs separate
  investigation — not the cause of Bug E's crash file (that was the prefilter_and_fold
  guard condition).
