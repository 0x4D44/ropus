## Design Direction

I wrote the HLD around a deliberately narrow policy:

- periodic audit only,
- no release-preflight integration,
- no full-test gate,
- `ropus` crate only,
- unit-test-only first,
- strict baseline by default,
- skip-degraded mode only if explicitly stamped into the report.

The strongest design call is to not attempt a C-reference mutation wrapper in
this slice. That feels right. The C oracle is essential for codec confidence,
but forcing it into this final infrastructure cleanup would make the work much
larger and easier to get wrong.

## Emotional Temperature

I am mildly wary of mutation testing as a metric. It can become a vanity score
fast, especially in fixed-point DSP code where equivalent mutants and static
tables are common. The useful thing here is not a percentage; it is a short,
auditable survivor list that says where the tests are weak. Keeping it
non-release-blocking makes me more comfortable with the proposal.

I am also a little irritated by the stale path issue because it is small and
obvious once noticed, but that is exactly why this HLD should force zero-mutant
resolved files to be non-green. A silent no-op audit is worse than no audit.
