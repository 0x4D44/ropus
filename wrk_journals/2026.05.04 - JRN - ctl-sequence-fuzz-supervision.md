## Worker A Implementation Notes
I am taking the Worker A slice directly in this sandbox: implement `fuzz_ctl_sequence` without committing. I re-read the top-level `AGENTS.md`, inspected `tests/fuzz/Cargo.toml`, `fuzz_encode.rs`, `fuzz_encode_multiframe.rs`, `fuzz_encode_safety.rs`, and the shared `c_reference.rs` FFI wrapper before editing.

Initial design feeling: the obvious trap is cloning `fuzz_encode_multiframe` and adding more knobs. That would look productive but mostly duplicate existing coverage. The useful shape is an operation-sequence target: compare Rust and C setter return codes for each CTL, keep both encoders alive across frames, and only then assert byte-exact packets. I am a little wary of `set_phase_inversion_disabled` because Rust currently does less validation than C for invalid values; I will keep that target path to valid values rather than bake in an immediate known mismatch.

Implemented the first version of `fuzz_ctl_sequence`. It uses structured `Arbitrary` input, 1-8 frames, 1-6 CTL/reset ops before each frame, exact-sized PCM per selected sample rate/channel count, and a small stateful C encoder wrapper in the target that calls the real C `opus_encoder_ctl`/`opus_encode` functions. I added C CTL request constants only; I did not fake C behavior.

Coverage includes bitrate, complexity capped at 9 to avoid the documented analysis divergence class, VBR, VBR constraint, FEC, loss, DTX, force_channels, bandwidth, max_bandwidth, signal, lsb_depth, expert_frame_duration, prediction_disabled, phase_inversion_disabled valid values, application before the first frame, and reset_state. The target compares setter return codes, selected getter parity, packet bytes, final range, and packet parseability/sample bounds.

Validation notes: I accidentally let `cargo fmt --manifest-path tests/fuzz/Cargo.toml` format the full fuzz crate. That was too broad for Worker A's scope, so I backed out the unrelated formatting deltas and kept only the intended files.
