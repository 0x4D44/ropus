## Investigation Notes

### Finding #1: fuzz_decode SILK PCM mismatch (22:34)

**Crash details:**
- Both workers (Job 0 and Job 1) panicked on seed corpus data
- `fuzz_decode.rs:60` assertion failed
- Config: sr=24000, ch=1 (mono SILK), pkt_len=31 bytes, 960 samples (2 frames)
- Exit code: -1073740791 (0xC0000409 = Windows ASan/abort)

**Divergence pattern:**
- First ~290 samples: exact match between Rust and C reference
- Sample ~291 onward: dramatic divergence (Rust produces different SILK decode output)
- This is consistent with SILK subframe state corruption — 24kHz uses 120-sample subframes, divergence starts around subframe 2-3 boundary of the second frame

**Impact on campaign:**
- fuzz_decode could not complete its 53-minute run (crashed on seed loading)
- Campaign script (`set +e`) continued to fuzz_encode successfully
- Effective campaign time reduced by ~53 minutes for this target

**Relation to known issues:**
- This is likely the known SILK decoder divergence previously documented
- The `fuzz_decode_safety` target handles this by only doing differential on CELT-only modes

**Action:** Investigation agent dispatched to identify root cause and propose fix

**Resolution (22:50):**
Investigation agent identified:
1. **Crashing seed**: `rich_sr3_ch1_toc79_c1.bin` (33 bytes) - Hybrid Fullband 20ms (config 15), NOT pure SILK
2. **Secondary bug**: `is_celt_only_packet()` in `fuzz_decode_safety.rs` used bandwidth instead of TOC mode bit, incorrectly classifying Hybrid packets as CELT-only
3. **Divergence onset**: Sample ~290 = mid-subframe-2 at 24kHz Hybrid (internal SILK rate 16kHz, subframes at 0/120/240/360/480)

**Fixes applied:**
- `fuzz_decode.rs`: Added `is_celt_only_packet()` using `packet[0] & 0x80 != 0` (TOC bit 7). PCM differential now only runs for CELT-only packets; sample count + error agreement checked for all modes.
- `fuzz_decode_safety.rs`: Replaced incorrect `opus_packet_get_bandwidth` check with `packet[0] & 0x80 != 0`. Removed unused imports.
- Both targets build clean, fuzz_decode verified: 46K iterations across full seed corpus, 7235 edge coverage, 0 crashes.
- **fuzz_decode relaunched** for full 3200s run separately (logs/fuzz-decode-rerun.log)

### Finding #2: `is_celt_only_packet` bug in fuzz_decode_safety (22:42)

**Bug**: `is_celt_only_packet()` checked `opus_packet_get_bandwidth()` for SWB/FB, but Hybrid mode (config 12-15) also has SWB/FB bandwidth. This meant Hybrid packets with SILK components would pass the "CELT-only" check and trigger differential PCM comparison, leading to false-positive crashes.

**Fix**: Use `packet[0] & 0x80 != 0` which directly checks TOC config >= 16 (true CELT-only modes).

### Finding #3: Multi-frame Hybrid encoder state divergence (~07:30)

**Crash details:**
- Both workers panicked on seed corpus data (within first 32 iterations)
- `fuzz_encode_multiframe.rs:124` assertion failed
- Frame 1/5 byte mismatch: sr=24000, ch=2 (stereo), app=AUDIO(2048), br=24000, cx=7
- TOC byte 0x6C = config 13 = Hybrid Wideband
- 60-byte packets; first 3 bytes match, diverge at byte 4 (Rust: 0xA5, C: 0xA6 — off by 1)
- Exit code: -1073740791 (0xC0000409 = Windows ASan/abort)

**Crashing seeds:** 8 files matching `mf5_24k_stereo_*_br24000.bin` in `tests/fuzz/corpus/fuzz_encode_multiframe/` (each 9607 bytes)

**Root cause analysis (investigation agent, 70% confidence):**

The bug is in SILK encoder inter-frame state. Frame 0 is bit-exact, so the core encoding path is correct. The divergence manifests when encoder state accumulated during frame 0 is used by frame 1.

**Most likely cause — gain quantization chain:**
- `SilkShapeStateFix::last_gain_index` (`silk/encoder.rs:274`) used as `prev_ind` in `silk_gains_quant` (line 2377). Controls hysteresis rounding at lines 2387-2388 and delta gain computation at lines 2401-2430.
- Off-by-1 at byte 4 is characteristic of gain quantization hysteresis error — byte 4 of 60 SILK bytes falls in the gain/NLSF encoding region.
- `sum_log_gain_q7` (line 454) accumulates through `silk_quant_ltp_gains`, affects max LTP gain constraint.
- `NsqState::prev_gain_q16` (line 248) — gain adjustment in NSQ at lines 3552-3583.

**Secondary suspect — HP variable cutoff activation:**
- `silk_hp_variable_cutoff` (`silk/encoder.rs:1152-1201`) is a **NO-OP for frame 0** (guarded by `prev_signal_type != TYPE_VOICED`). After frame 0 sets `prev_signal_type = TYPE_VOICED`, it activates for frame 1. Any error in the smoothing computation at lines 1189-1193 would be invisible in frame 0 but manifest in frame 1.

**Code paths to audit:**
1. `silk_hp_variable_cutoff` (lines 1152-1201) — compare against `reference/silk/HP_variable_cutoff.c`
2. `silk_gains_quant` (lines 2373-2440) — `double_step_size_threshold` logic at lines 2406-2427
3. `silk_encode_frame_fix` post-loop state update (lines 7382-7398) — verify `ec_prev_lag_index` and `ec_prev_signal_type` update order matches C

**Why single-frame fuzzing missed this:**
- `fuzz_encode` creates a fresh encoder per input → no state accumulation
- `fuzz_roundtrip` encodes one frame → same issue
- Only `fuzz_encode_multiframe` reuses the encoder across sequential frames, exposing state drift

**Note:** The staged `silk/common.rs` change in git status is from the previous commit (pinning tests only) — not related to this bug.
