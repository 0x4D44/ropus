# 2026-04-03 - CELT Short-Block MDCT Divergence

## Problem
48kHz mono sine @ 64kbps (CELT fullband): FAIL at offset 3 (first CELT data byte).
The MDCT frequency output is wrong for short blocks (transient detection).

Both C and Rust detect transient (short_blocks=8), input matches (all zeros from overlap).
But MDCT output diverges dramatically from freq[2] onwards:
- C:  freq[0..4] = [6, -1654539, 3170489, -354501]
- RS: freq[0..4] = [0, -1654541, 1382282, -6369]

freq[0] and [1] are close (rounding), but [2] onwards is completely different.
This suggests per-block MDCT computation is wrong for some blocks.

## Configuration
- 48kHz, 960 samples/frame, 8 short blocks of 120 samples each
- MDCT size per block: N=120, N/2=60 freq bins
- With stride=8, output is interleaved: freq[k*8+blk] = block blk's bin k
- Overlap = 120 (mode->overlap for 48kHz)

## Investigation results

### Per-block MDCT trace
Added traces to both C and Rust. Parameters match (N=120, ov=120, shift=3, stride=8).

**KEY FINDING: The MDCT INPUT data differs!**
Block 0: C has in[120]=-3, Rust has in[120]=0
Block 1: C has in[60]=-3, Rust has in[60]=0
Block 3: C has in[60]=2969058, Rust has in[60]=4239439 (40% different!)

The MDCT function itself is probably correct. The issue is UPSTREAM — the
pre-emphasis/overlap-add buffer that feeds compute_mdcts has different content.

### pcm_buf trace
Frame 1: PCM buffers IDENTICAL between C and Rust ✓
Frame 2: PCM buffers DIFFER — delay buffer has wrong values
- C: pcm_buf[0]=6583, Rust: pcm_buf[0]=6519 (diff=64)
- But even frame 1's MDCT input differs (C has -3, Rust has 0)
- Since pcm_buf is identical for frame 1, the bug must be in:
  1. celt_preemphasis processing (unlikely, produces same output for same input+state)
  2. The overlap memory (prefilter_mem) — might be initialized differently
  3. Something inside the CELT encoder between pre-emphasis and MDCT

### Pre-emphasis trace
Frame 1 pcm_buf: IDENTICAL between C and Rust ✓
Frame 1 pre-emphasis output (before prefilter): IDENTICAL ✓
Frame 1 pre-emphasis output (AFTER prefilter):
- C:  in[ov..ov+2] = [-3, -3]  (modified by prefilter)
- RS: in[ov..ov+2] = [0, 0]    (NOT modified)

## ROOT CAUSE: CELT prefilter (comb filter)
The C encoder runs `run_prefilter()` which applies a comb filter to the input signal,
producing small residuals (-3) even for near-silent input. The prefilter is enabled 
(complexity >= 1, sufficient bytes available, not hybrid).

The Rust `run_prefilter()` function exists (encoder.rs:1424) and IS called (line 1872),
but it appears to NOT modify the input buffer correctly. Either:
1. The pitch search finds no pitch → gain=0 → no filtering applied
2. The comb filter application has a bug
3. The prefilter memory state differs

### Next steps
1. Add trace inside run_prefilter to check gain, pitch_index
2. Compare the comb filter application between C and Rust
3. Check if the Rust prefilter returns pf_on=0 when C returns pf_on=1
4. This explains both the 48kHz CELT divergence AND the frame-2 delay buffer divergence

## DETAILED ROOT CAUSE (C vs Rust run_prefilter comparison)

C run_prefilter (celt_encoder.c:1411-1608):
1. Copies pre-emphasis output to `pre[]` work buffer  
2. Does pitch search on `pre[]`
3. Copies overlap from `st->in_mem[]` to `in[]` (NOT prefilter_mem!)
4. ALWAYS applies comb filter with NEGATIVE gain: `comb_filter(in, pre, -old_gain, -new_gain, ...)`
   This is the analysis filter (removes pitch). Applied even when gain=0 for crossfade.
5. If filter made signal worse: reverts by copying `pre[]` back and crossfading to gain=0
6. Updates `st->in_mem[]` with tail of filtered `in[]`
7. Updates `prefilter_mem[]` with `pre[]` (UN-filtered data)

Rust run_prefilter (encoder.rs:1424-1615):
1. Copies input + prefilter_mem to work buffer ✓
2. Does pitch search ✓
3. Copies overlap from `prefilter_mem[]` (WRONG — should be `in_mem[]`)
4. Only applies comb filter when gain1 > 0 (WRONG — should always apply for crossfade)
5. No cancel_pitch check (MISSING)
6. No `in_mem[]` memory at all (MISSING)
7. Updates prefilter_mem from input (should be from unfiltered `pre[]`)

Key missing state: `st->in_mem` (overlap memory for filtered signal, separate from prefilter_mem)

### Impact
- ALL non-silence CELT frames are affected
- The overlap between frames uses wrong (unfiltered) data
- The comb filter crossfade is skipped, causing discontinuities
- The prefilter_mem gets corrupted (updated with filtered instead of unfiltered data)
