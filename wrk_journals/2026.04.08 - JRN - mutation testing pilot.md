# Journal: Mutation Testing Pilot

Date: 2026-04-08

## 00 — Kickoff

Read the brief. Objective: pilot `cargo-mutants` on Phase 1 modules (types.rs, math_ops.rs, range_coder.rs, ec_ctx.rs) to assess test suite sensitivity.

Key insight from brief: the C reference oracle should be an exceptionally strong mutation detector. Any mutation that changes numerical output gets caught by differential comparison. So surviving mutants are either equivalent mutants, dead code, or genuinely weak tests.

**Agent team:**
- Scout: explore codebase, check tooling, gather facts
- Implementer: configure cargo-mutants, run phases, write tests
- Devil's Advocate: question assumptions, challenge approach, spot blind spots

**Plan:**
1. Check cargo-mutants availability and configure
2. Run Phase 1 mutation testing
3. Analyze and triage surviving mutants
4. Write targeted tests for genuine gaps
5. Document baseline and decide on Phase 2

---

## 01 — Devil's Advocate Review (critical findings)

The DA raised five issues before we even ran the tool. Two are blockers:

**Blocker 1: C reference oracle not in the loop.**  
`cargo-mutants` runs `cargo test`. The 280-config differential comparison harness is a separate `[[bin]]`, not a `#[test]`. The brief's "exceptionally strong mutation detector" claim is hollow unless we pipe those tests through. **Decision**: accept this limitation for the pilot. The 744 unit tests are still valuable — they use exact-value assertions (`assert_eq!`), which are strong oracles for the paths they cover. The gap is coverage, not assertion quality.

**Blocker 2: Two failing tests break the baseline.**  
`dnn::lpcnet::tests::test_compute_frame_features_updates_state_before_pitchdnn_panic` and `test_plc_load_model_invalid_blob_returns_error` fail. These are DNN tests we didn't author. **Fix**: exclude via `additional_cargo_test_args` in mutants.toml.

**Adjustments based on DA feedback:**
- Drop `ec_ctx.rs` from Phase 1 (pure delegation, 4 tests, minimal logic)
- Start with `math_ops.rs` alone as proof-of-concept before committing to full Phase 1
- Expect noise from `types.rs` trivial wrappers — may need `--exclude-re` filtering
- Acknowledge: mutation testing here mostly confirms what branch coverage shows. Value is in finding specific *which* mutations survive, not aggregate scores.

## 03 — math_ops.rs Results (proof-of-concept)

**271 mutants generated. Results:**
| Outcome | Count | % |
|---------|-------|---|
| Caught  | 186   | 68.6% |
| Missed  | 69    | 25.5% |
| Timeout | 16    | 5.9% |
| Kill rate (caught+timeout) | 202/271 | **74.5%** |

Far below the expected 95%. The DA was right that the unit tests alone (without the C reference oracle) aren't as strong as assumed.

**Triage summary of 69 survivors:**

| Category | Functions | Survivors | Notes |
|----------|-----------|-----------|-------|
| Dead code | celt_atan2p, celt_cos_norm32, opus_limit2_checkwithin1 | 17 | No external callers at all |
| No test (live) | celt_div, celt_log2_db, celt_exp2_db, celt_exp2_db_frac, celt_sqrt32, celt_rcp_norm32, celt_atan2p_norm, celt_rsqrt_norm32 | 24 | Live code with zero test coverage |
| Weak test | celt_cos_norm, celt_log2, celt_exp2, celt_sqrt, celt_rcp, celt_rsqrt_norm, frac_div32 | 23 | Tests exist but too few inputs / loose tolerances |
| Indirect only | celt_cos_pi_2, celt_rcp_norm16 | 5 | Only tested via callers |

**Key insight**: The problem isn't assertion quality — it's coverage. Functions with tests mostly catch mutations. Functions without tests catch nothing. The "weak test" category is mostly about insufficient input diversity (1-2 test points with loose tolerances).

**Action plan**: Write targeted tests for the highest-impact gaps: celt_div, celt_sqrt32, celt_rcp_norm32, celt_log2_db, celt_exp2_db, and strengthen celt_cos_norm/celt_log2.

## 04 — Writing mutation-killing tests

Added 34 new tests to math_ops.rs (31 → 65 total). Targeting:
- **No-test functions (live)**: celt_div (4), celt_sqrt32 (4), celt_rcp_norm32 (3), celt_log2_db (2), celt_exp2_db (2), celt_exp2_db_frac (1), celt_atan2p_norm (4), celt_rsqrt_norm32 (1)
- **Strengthened existing**: celt_cos_norm (4), celt_log2 (2), celt_exp2 (3), celt_sqrt (1), frac_div32 (2), celt_rsqrt_norm (1)

Initial failure: 2 celt_div tests had wrong Q-format expectations. Fixed by computing correct expected values (celt_div uses `mult32_32_q31(a, celt_rcp(b))` which means the result is NOT simply a/b).

## 05 — V2 Results (after new tests)

**271 mutants. Results:**
| Outcome | Count | % | Change |
|---------|-------|---|--------|
| Caught  | 215   | 79.3% | +29 |
| Missed  | 40    | 14.8% | -29 |
| Timeout | 16    | 5.9% | 0 |
| Kill rate | 231/271 | **85.2%** | **+10.7pp** |

**Improvement: 69 → 40 survivors (42% reduction)**

**Breakdown of 40 remaining survivors:**

| Category | Survivors | Functions |
|----------|-----------|-----------|
| Dead code (no callers) | 17 | celt_atan2p (13), celt_cos_norm32 (2), opus_limit2_checkwithin1 (2) |
| Equivalent mutants | ~8 | Boundary comparisons (< vs <=, > vs >=) where both branches produce same result |
| Newton iteration signs | 6 | celt_rsqrt_norm (1), celt_rsqrt_norm32 (1), celt_rcp_norm16 (2), celt_rcp_norm32 (1), celt_rcp (1) |
| Polynomial internals | 5 | celt_log2 (5) — line 373 combining integer+fractional parts |
| Other weak tests | 4 | celt_exp2 boundary (3), celt_cos_norm quadrant (1) |

**Excluding dead code (17 survivors from 3 dead functions):**
- Live-function kill rate: ~90.4% (significantly above the 74.5% baseline)

**Key categories of remaining survivors:**

1. **Dead code (17/40)**: celt_atan2p has 13 survivors and zero external callers. Should be removed.

2. **Equivalent mutants (~8/40)**: Boundary comparisons where the exact boundary value is handled by a different code path (e.g., `k < 12` vs `k <= 12` in celt_sqrt32 where k=12 gives identical result either way).

3. **Newton iteration signs (6/40)**: These are in the correction/refinement steps of iterative algorithms. The sign change produces a slightly wrong correction, but the result is still within tolerance for the limited test inputs. Would need exact C reference values or many more test inputs to catch these.

4. **celt_log2 line 373 (5/40)**: The integer+fractional combining logic `shl32(i - 13, 10) + shr32(frac, 14 - 10)` has 5 surviving mutations. These are hard to catch because the fractional part is small relative to the integer part for exact powers of 2.

## 06 — Devil's Advocate Final Review

DA challenged the report on 5 points. Key corrections incorporated:

1. **"90.4% excluding dead code" is defensible but should not be the headline.** The dead functions are `pub` and might gain callers during integration. Lead with 85.2%, report 90.4% as a secondary metric.

2. **Phase 2 recommendation is conditional, not unconditional.** Prerequisites (dead-code cleanup, branch coverage improvements, Newton FFI tests) must be done first. Phase 2 on low-coverage code produces "no test" noise, not actionable "weak test" insights.

3. **Framing matters.** The real finding is "8 functions with no fast-feedback safety net," not "kill rate improved 10.7pp." The kill rate is a vanity metric. The actionable output is the prioritized gap list.

4. **"Need exact C reference values" for Newton iteration is a cop-out.** The `debug_helper.c` FFI infrastructure already exists. Adding differential tests for rcp/rsqrt is ~30 minutes of work, not a research project. We chose not to do it in pilot scope — say that honestly.

5. **Was it worth it?** Mostly yes. Mutation testing was a forcing function for a systematic function-by-function audit. The 34 tests are real value as fast regression guards. But the same gaps could have been found by carefully reading the coverage report. Mutation testing's value is prioritization, not discovery.

## 07 — range_coder.rs (supplementary data)

Ran mutation testing on range_coder.rs (849 mutants, still in progress). Early results: ~33% kill rate.

Pattern: massive noise from debug/accessor functions (debug_state tuple replacements, get_val, error, restore_snapshot). Confirms DA's prediction that mutation testing on code with sparse unit tests produces mostly "no test" noise rather than actionable "weak test" insights.

**Decision**: range_coder.rs confirms the pilot's conclusion but doesn't add much signal beyond what math_ops.rs already showed. Not worth deep-triaging for the pilot.
