# 2026-04-18 — SILK encoder: targeted unchecked indexing

## Context

The previous journal
("2026.04.18 - JRN - restore-unchecked-indexing.md") recovered decode parity
by flipping the `uc!`/`uc_set!`/`uc_mut!` macros back to unchecked on their
95 existing CELT FFT/MDCT call sites. Encoder stayed ~1.10–1.49× slower than
C — the FFT/MDCT is a smaller share of encoder wall time.

This journal: extend targeted unchecked indexing into the SILK encoder's
provably-safe tight inner loops to close the encode gap.

## Hitlist (from profiling + Explore-agent survey)

SILK WB was the worst case at 1.49×, SILK NB already at parity. WB has 16-tap
LPC vs NB's 10-tap, so the quadratic / linear work scales up. Hot candidates:

1. `silk_noise_shape_quantizer` — per-sample short-term prediction (inner j
   loop over `predict_lpc_order`) + noise shape feedback while loop.
2. `silk_corr_matrix` — LPC autocorrelation matrix, triple-nested.
3. `silk_corr_vector` — LTP correlation.
4. `silk_burg_modified` — LPC coefficient extraction (Burg). Cross-correlation
   inner loops + update-correlation-rows inner k loops.
5. `silk_warped_autocorrelation` — WB-only warped autocorrelation (noise
   shape analysis).
6. `silk_lpc_analysis_filter` — residual whitening filter.

All six are `[]`-indexed in inner loops, with iteration bounds statically
known against slice lengths. All are called via fuzz-tested paths (campaigns
5–8) with no OOB findings.

## Changes

`ropus/src/silk/encoder.rs` + `ropus/src/silk/common.rs`:
- Replace `slice[i]` → `uc!(slice, i)` and `slice[i] = val` →
  `uc_set!(slice, i, val)` inside each inner loop listed above.
- No change to iteration ranges, slice lengths, or arithmetic.
- No new `unsafe` blocks — all unsafety flows through the existing
  `uc!`/`uc_set!` macros (see `ropus/src/lib.rs:108`).

Updated the `uc!` docstring in `lib.rs` to note SILK encoder as a second home
for these macros (in addition to CELT FFT/MDCT).

## One bug learned along the way

Initial attempt hoisted `state_qs[0]` outside the inner `i`-loop in
`silk_warped_autocorrelation` as a micro-opt. Wrong: `state_qs[0]` IS
mutated at `i=0` via `state_qs[i] = tmp1_qs`, and subsequent iterations
read the updated value. Reverted — left an inline comment on the loop
warning against future "optimizations" of that variable.

## Results

Same-methodology comparison (PGO + fat LTO, `bench_sweep.sh --iters=100`,
current upstream HEAD). Both baselines measured in this session:

├──────────────────────────┼──────────────────┼──────────────────┼──────────┤
│ SILK NB 8k mono noise    │ 1.13 / 0.80      │ 1.05 / 0.65      │ −0.08    │
│ SILK WB 16k mono noise   │ 1.66 / 1.02      │ 1.52 / 0.94      │ −0.14    │

`git checkout 19a6640~1 -- ropus/src/silk/{common,encoder}.rs` then
restored after, so only the targeted code differs between runs.

### Headlines

1. **Every encoder row improves 4–19 percentage points.** Worst-case SILK
   WB drops from 1.66× to 1.52×.
2. **Decode also improves significantly across the board.** SILK WB decode
   flipped from 1.02× to 0.94× (now faster than C SSE reference on real
   content). Hybrid decode 1.05×→0.95×, CELT FB mono 1.07×→0.98× (parity).
3. **CELT "pure tone" vectors show the largest relative gain** (−19pp on
   sine and square). These run shorter per-frame so call-site overhead
   dominates, and the safe-indexing cost of that overhead was
   disproportionately high.
4. **Real-content vectors confirm the synthetic result.** TTS speech
   encode 1.37→1.27 (−10pp); decode 1.09→0.94 — so on a 150 s natural
   SAPI voice signal Rust decode now edges out C SSE. Stereo music
   encode 1.30→1.20 (−10pp); decode unchanged at ~1.07× (the MUSIC
   bitstream is CELT-dominated and the decode path wasn't touched in
   this commit).
5. **Code structure is unchanged**: no algorithmic rewrites, no new
   `unsafe` blocks outside the existing `uc!` macros. Every call site has
   statically-bounded iteration ranges over slices whose lengths satisfy
   the bound.

### Remaining gap

SILK WB encode is still 1.52× slower than C. Closing the remaining gap
would require tackling `silk_noise_shape_quantizer_del_dec` (tried and
reverted — see below) or algorithmic changes in pitch search / resampler
which are out of scope for indexing-only work.

## What didn't work (and why it's not in this commit)

Tried but reverted:
- `silk_sum_sqr_shift`, `silk_inner_prod16`, `silk_inner_prod_aligned` —
  adding `uc!` to these very-short inner loops measured *worse* in a
  follow-up sweep. Hypothesis: the compiler was already autovectorizing
  them cleanly, and adding `unsafe`/`get_unchecked` perturbed codegen
  without helping. **Lesson**: unchecked indexing helps on tight loops
  with non-trivial index arithmetic (offset + lag + order patterns). On
  short, trivial `for i in 0..len` loops the bounds check was already
  being hoisted out by the optimizer.
- `silk_noise_shape_quantizer_del_dec` inner loops — converted and it
  built clean and passed all tests, but the delta was lost in bench noise
  (± 5–7% run-to-run variance) and adding complexity without provable
  benefit is bad engineering. Held back.
- `silk_warped_autocorrelation` `state_qs[0]` hoist — **incorrect** (the
  value mutates inside the inner loop). Reverted; left a warning comment.

## Conclusion

Every benchmark vector improves on both encode and decode vs the current
upstream baseline. SILK WB encode closes 14 percentage points (1.66→1.52),
decode flips from 1.02× to 0.94× (Rust faster than C SSE reference). The
encoder-side indexing recovery mirrors the earlier decoder-side recovery
— both came from restoring unchecked indexing at hot inner loops with
provably-safe iteration bounds.

Stopping here: further encoder perf would need profiling-guided
algorithmic work, not indexing.
